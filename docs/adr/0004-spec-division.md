# ADR-0004. 명세 분업 — G9 게이트 신설 · 기능 명세(D9) / 백엔드 명세(D10) 분리

## 맥락

Admin 화면(로그인·대시보드·회원 관리)이 컨펌됐다. 그런데 **컨펌된 화면을 백엔드 개발자와 QA에게
넘길 문서가 없다.**

- SCR(D2, 화면정의서)은 **기획 단계 문서**다. 화면을 만들기 위해 쓰였고, 화면이 만들어지는 순간
  역할이 끝난다. SCR에는 확정된 화면의 **요소 좌표계**(어느 버튼을 무슨 번호로 부를 것인가)도,
  **예외 전수**도, **백엔드 연동 계약**도 없다. 있을 이유도 없다 — 화면을 그리기 전에 쓰는 문서이므로.
- 프론트는 `data-source.ts` 어댑터로 **연동 지점만 뚫어둔 상태**다. 백엔드는 구현하지 않는다(사용자 방침).
  `// TODO(backend): GET /api/members?tier=&keyword=&page=&size=` 같은 주석과 함수 시그니처가
  현재로선 프론트↔백엔드 계약의 유일한 흔적이다.

즉 백엔드 개발자가 받을 수 있는 것은 (a) 기획 단계의 SCR, (b) 프론트 소스뿐이다. 이 상태로 넘기면
백엔드는 화면을 역공학하며 명세를 스스로 지어내게 되고, QA는 무엇을 테스트해야 하는지 합의할 근거가 없다.

## 결정

### (1) 신규 게이트 G9 (명세 확정)

G8(릴리스)과 **병렬**로 돈다. G6(React 구현) 완료가 선행 조건이지만 릴리스를 기다리지 않는다 —
백엔드 개발자가 명세를 먼저 받아야 병목이 안 생긴다.

- 승인자: 명세 리뷰 · 생산자: 기능 명세, 백엔드 명세 · SLA 8시간 · 체크리스트 `docs/_templates/checklists/G9.md`
- 진입 조건: 화면이 **컨펌**됨 (사용자 또는 오케스트레이터 승인) + SCR과 구현이 존재
- 병목 완화: 기능 명세(FS) 완료 즉시 백엔드 명세(BE)가 착수한다. FS 전체 승인을 기다리지 않고 **요소 단위로 흘린다.**

### (2) 명세 역할 3종 추가

| 역할 | 정의 | 소유 경로 |
|---|---|---|
| **기능 명세** Functional Spec Writer | 컨펌된 화면을 요소별로 전수 넘버링해 FS-NNN 작성 | `docs/spec/functional/**` |
| **백엔드 명세** Backend Spec Writer | FS와 프론트 data-source를 근거로 BE-NNN 작성 | `docs/spec/backend/**` |
| **명세 리뷰** Spec Reviewer | G9 기준으로 FS/BE 검수 · APPROVED/CHANGES_REQUESTED/BLOCKED 판정 | `docs/spec/review/**` |

기능 명세는 백엔드를 설계하지 않고(`[서버]` 표시까지만), 백엔드 명세는 백엔드를 **구현하지 않는다**(API 계약까지만 —
서버 코드·DB 스키마 금지). 명세 리뷰는 생산하지 않고 판정만 한다(자가 검수 금지, P3).

### (3) 표준 문서 2종 추가

- **D9 기능 명세서** (`docs/_templates/D9-functional-spec.md`) → `docs/spec/functional/FS-NNN-<slug>.md`
- **D10 백엔드 기능 명세** (`docs/_templates/D10-backend-spec.md`) → `docs/spec/backend/BE-NNN-<slug>.md`

### (4) 요소 넘버링 규칙

```
FS-NNN-EL-nnn
    │      └─ 요소 일련번호 (좌→우 · 위→아래. 재사용·재배정 금지)
    └──────── 화면 번호 (대응 SCR 번호와 동일 — SCR-002 → FS-002 → BE-002)
```

번호는 **영구 불변**이다. 요소가 삭제되면 번호를 비우고 `(삭제됨, vX.Y)` 로 표기한다.
영역은 `FS-NNN-SEC-nn`, 엔드포인트는 `BE-NNN-EP-nn` — 엔드포인트는 FS 요소 번호를 **역참조**한다.

### (5) 예외 전수 원칙

- **FS 7축**: 빈 상태 / 로딩 / 실패 / 유효성 / 권한없음 / 경합 / 대량
- **BE 9축**: 400 / 401 / 403 vs 404 / 404 / 409 / 422 / 429 / 500 / 타임아웃

**빈칸 금지.** 해당 없으면 `N/A — <사유>`. 빈칸은 "검토했으나 해당 없음"과 "검토 자체를 안 함"을
구분하지 못한다. G9에서 빈칸 1건 = blocker.

## 근거

- **넘버링이 없으면 같은 요소를 서로 다른 이름으로 부른다.** "로그인 버튼"이라고 하면 사람마다 다른 걸
  떠올리지만 `FS-001-EL-014`는 하나만 가리킨다. 프론트·백엔드·QA·기획이 **같은 좌표계**를 쓰는 것이
  명세서의 존재 이유다. 티켓·테스트·BE 엔드포인트가 전부 이 번호를 물고 있으므로 번호 재사용은
  과거 산출물이 엉뚱한 요소를 가리키게 만든다 — 그래서 불변이다.
- **예외를 표로 강제하지 않으면 "정상 동작만 적힌 명세"가 나온다.** 그리고 그 빈칸이 백엔드 개발자를
  멈춰 세운다. "회원이 0건이면?", "이미 탈퇴한 회원에게 알림을 보내면?", "권한 없는 회원 상세는 403인가
  404인가?" — 이 질문들은 구현 시점에 반드시 나오고, 명세에 없으면 개발자가 즉흥적으로 답을 만든다.
  그럴듯한 문장 100줄보다 채워지지 않은 예외 한 칸이 더 비싸다.
- **BE 명세를 프론트의 data-source 시그니처에서 출발시키면 백지에서 시작하지 않는다.**
  `fetchMembers(query, signal): Promise<MemberListResult>` 는 이미 **합의된 모양**이다. 백엔드 명세는 그것을
  확정만 하면 된다. 백엔드가 백지에서 설계하면 프론트가 기대하는 필드명·타입과 어긋나고, 그 불일치는
  통합 시점에 발견된다 — 가장 비싼 시점이다.

## 대안

1. **SCR(D2)을 확장해 겸용** — 기획 문서와 확정 명세는 **생명주기가 다르다.** SCR은 화면이 확정되는
   순간 얼어붙어야 하고(더 고치면 기획 이력이 사라진다), FS는 화면 확정 **후에** 갱신된다(구현이 바뀌면
   따라 바뀐다). 한 문서에 담으면 둘 다 썩는다 — 기획 이력은 오염되고, 명세는 기획 잔재에 파묻힌다. 기각.
2. **백엔드 개발자가 직접 명세 작성** — 프론트가 이미 기대하는 모양(`types.ts` 도메인 타입,
   `data-source.ts` 시그니처)이 존재하는데 백엔드가 백지에서 설계하면 **불일치가 구조적으로 발생한다.**
   명세의 목적이 합의인데, 합의 전에 각자 쓰면 명세가 아니라 두 개의 제안서가 된다. 기각.
3. **자동 생성 (구현 → 명세)** — 요소 목록은 AST로 뽑을 수 있을지 몰라도 **예외 축과 권한 정책은 코드에서
   유도되지 않는다.** "권한 없는 회원 상세를 404로 숨긴다"는 결정이지 사실이 아니고, "0건일 때 무엇을
   보여줄지"는 코드에 있어도 "0건일 때 무엇을 보여줘야 하는지"는 코드에 없다. 기각.

- 스킬 3종: `skills/functional-spec-writer/`, `skills/backend-spec-writer/`, `skills/spec-reviewer/`

  이 조직의 경계 밖이다.
