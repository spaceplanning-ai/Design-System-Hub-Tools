# ADR-0007. 프론트엔드 시니어리티 분리

## 맥락

프론트 작업이 성격상 **두 갈래**로 갈렸다.

- **(a) 화면을 새로 만드는 일** — 계약과 Screen Spec을 받아 동작하는 화면을 낸다. 산출물의 성패는 "명세대로 동작하는가"다.
- **(b) 만들어진 코드를 정리하고 도구를 도입하는 일** — 중복을 걷어내고, 결합을 끊고, 라이브러리·OpenAPI를 앱 전반에 적용한다. 산출물의 성패는 "동작을 안 바꾸고 구조를 개선했는가"다.

두 일은 **성공 기준이 정반대다.** (a)는 동작을 만들고, (b)는 동작을 **보존**한다. 그런데 지금까지 이 둘은
`apps/*/src/**`라는 하나의 소유 경로 위에서 사실상 한 몸으로 굴러갔고, 예측 가능한 결과가 나왔다.

**결함 1 — 페이지가 다른 페이지의 컴포넌트를 import 하는 구조가 리뷰 없이 자랐다** (ADR-0006 맥락 표).

| import 하는 쪽 | import 되는 쪽 |
|---|---|
| `pages/admins/components/AdminsSearchCard.tsx` | `pages/members/components/Card`, `pages/members/styles` |
| `pages/permissions/components/RolePanel.tsx` | `pages/members/components/Button`, `Modal`, `Card` |
| `pages/customer-settings/components/TierPolicyCard.tsx` | `pages/members/components/Card`, `Alert`, `pages/members/types` |

`members`를 지우면 `admins`·`permissions`·`customer-settings`가 함께 죽는다. 각 PR은 "기존 Card를 재사용했다"로 보였다.

**결함 2 — 수제 훅이 페이지마다 복제됐다.** `useAsyncData` 같은 비동기 훅이 페이지별 사본으로 존재한다.
이것은 "라이브러리를 도입할지 판단하고, 도입했다면 전부 갈아끼우는" 주체가 없었기 때문이다.
아무도 도입을 결정하지 않으니 각자가 각자의 30줄을 짰다.

**한 사람이 둘 다 하면 만들면서 정리까지 하려다 범위가 흐려지고, 결국 둘 다 어중간해진다.**
구현 중에는 "이참에 정리하자"는 충동이 오고, 정리 중에는 "이참에 이 버그도 고치자"는 충동이 온다.
전자는 화면을 늦추고, 후자는 리팩터의 **동작 보존 증거**(기존 테스트 무수정 통과)를 파괴한다.

## 결정

**프론트 구현을 중급(구현), 프론트 리팩터를 고급(검수·리팩터 + 라이브러리·OpenAPI 집행)으로 명시적으로 분리한다.**

| | **프론트 구현 — Frontend Engineer (Mid)** | **프론트 리팩터 — Frontend Engineer (Senior)** |
|---|---|---|
| 한 문장 | **화면을 만든다** | **정리하고 집행한다** |
| 성공 기준 | 동작하고 · 계약을 지키고 · 명세대로다 | 동작 회귀 0 · 클린코드 점검 위반 0 · 잔해 0 |
| 하는 일 | 계약 + Screen Spec → React 화면 | (1) 프론트 구현 코드 검수 → 클린코드 (SOLID·DRY·KISS·YAGNI)<br>(2) 라이브러리·OpenAPI 조사 → 앱 전반 일괄 적용 |
| 안 하는 일 | 대규모 리팩터 · **의존성 임의 추가** | **동작 변경** · 신규 기능 · **도입 결정** |
| 완벽함 요구 | **없다** — 클린코드는 프론트 리팩터의 몫 | 있다 — 취향이 아니라 **측정된 위반**에 대해 |
| 게이트 | G6 (reviewer: 코드 리뷰) | G6 (reviewer: 코드 리뷰) — **자가 승인 없음** |

**소유 경로는 `apps/*/src/**` 공동 소유를 유지하되, 순차 배타(`sequentialExclusiveWith`)로 동시성을 막는다.**
둘은 같은 파일을 다른 시점에 만진다. 착수 전 상대의 작업 상태를 확인한다.
(상호 `sequentialExclusiveWith` 쌍에 한해 경로 공동 소유를 허용한다 — P1 단일 소유권의 유일한 예외.)

**프론트 리팩터이 두 역할을 한 몸으로 갖는 이유: 무엇을 쓸지 아는 사람이 어떻게 정리할지도 안다.**
라이브러리를 도입하면 기존 수제 코드(`useAsyncData`, 수동 검증, 손으로 쓴 API 타입)를 **걷어내야** 하는데, 그것이 곧 리팩터다.
도입과 정리가 다른 사람에게 나뉘면 **도입만 하고 잔해가 남는다.**

**인접 역할과의 경계** (ADR-0006에서 신설된 의존성 관리/클린코드 점검, 모듈 추출 포함):

| 질문 | 주체 | 산출물 |
|---|---|---|
| **무엇을 만드는가** (구현) | **프론트 구현** | `apps/*/src` 화면 |
| **무엇을 도입하는가** (정책) | **의존성 관리** Dependency Manager | 라이브러리 선정 ADR · lockfile |
| **어떻게 집행하는가** (리팩터·적용) | **프론트 리팩터** | 클린코드 리팩터 · 라이브러리/OpenAPI 일괄 적용 |
| **얼마나 나쁜가** (측정) | **클린코드 점검** Clean Code Inspector | `reports/code-quality/` (6축 수치) |
| **무엇을 모듈로 뽑는가** (계획) | **모듈 추출** Module Extraction Architect | 모듈화 계획(MP-NNN) · 리팩터 지시서 |
| **통과인가** (G6 승인) | **코드 리뷰** Code Reviewer | RR-G6-* |

즉 **의존성 관리가 정책(무엇을) · 프론트 리팩터가 집행(어떻게) · 클린코드 점검이 측정(얼마나 나쁜가) · 코드 리뷰가 승인(G6)** 이다.

## 근거

**1. 재는 자와 고치는 자를 분리해야 측정이 정직해진다.**
자기 코드를 자기가 재면 기준이 무뎌진다. 임계값을 넘으면 임계값을 의심하게 되고, "이번 건은 예외"가 쌓인다.
클린코드 점검은 측정만 하고 **코드를 고치지 않으며**, 프론트 리팩터는 수정만 하고 **리포트를 고치지 않는다**.
이는 조직이 경계 담당·계약 테스트(4자 일치)·네이밍 가드에서 이미 확립한 패턴 — **측정 주체와 생산 주체의 분리** — 를 코드 품질에 적용한 것이다.

**2. 도입과 정리를 같은 사람이 해야 잔해가 안 남는다.**
`react-query`를 넣고 수제 `useAsyncData`를 남기면 두 벌이 공존한다. **두 벌은 한 벌보다 나쁘다** —
다음 사람은 어느 쪽을 써야 할지 모르고, 버그는 두 벌 중 한쪽에만 고쳐진다.
도입(설치)과 정리(제거)를 다른 역할에 나누면 정리는 항상 뒤로 밀린다. 밀린 정리는 오지 않는다.
그래서 **집행은 한 손이 한다** — 넣는 사람이 빼는 사람이다.
(단 **결정**은 프론트 리팩터의 것이 아니다. 무엇을 넣을지는 의존성 관리의 ADR이 정한다 — 집행자가 결정까지 하면 "쓰고 싶은 것을 쓰는" 도입이 된다.)

**3. 중급에게 완벽을 요구하면 화면이 안 나온다.**
"동작하고, 계약을 지키고, 명세대로여야 한다" — 이것이 프론트 구현의 전부다.
클린코드까지 프론트 구현의 DoD에 넣으면 두 가지가 벌어진다: (a) 화면이 느려지고, (b) **자기 코드를 자기가 검수하게 된다**(P3 자가 승인 금지 위반과 같은 성질의 문제).
검수는 다른 눈이 한다. 그 눈이 프론트 리팩터다.

**4. 성공 기준이 정반대인 두 작업을 한 커밋에 섞으면 검증이 불가능해진다.**
리팩터의 증거는 "기존 테스트를 한 글자도 고치지 않고 통과했다"이다.
같은 커밋에 기능 변경이 섞이면 테스트가 바뀌고, 그 순간 **동작 보존을 증명할 방법이 사라진다.**
분리는 도덕적 요구가 아니라 **검증 가능성의 요구**다.

## 대안

1. **한 명이 전부 한다** (현행).
   지금까지의 방식이고, **그래서 결함이 자랐다** — 페이지 간 결합 4건, 수제 훅 복제.
   구현자가 리팩터를 겸하면 리팩터는 항상 "다음 스프린트"로 밀리고, 밀린 부채는 클린코드 점검 도입 시 즉시 PR 차단 사유가 된다.
   무엇보다 **자기 코드를 자기가 검수하는 구조**여서 기준이 무뎌진다. **기각.**

2. **고급/중급 대신 '구현/리팩터'만 분리한다** (프론트 리팩터를 리팩터 전담으로 두고 라이브러리 도입은 아무도 안 맡음).
   그러면 **라이브러리 도입의 집행 주체가 없어진다.** 의존성 관리가 ADR을 써도 그것을 앱 코드에 적용할 사람이 없으므로,
   ADR은 `accepted` 상태로 방치되고 코드는 그대로 수제 구현에 머문다 — **정책만 있고 집행이 없는 상태**가 ADR-0006 이전과 실질적으로 같아진다.
   또한 리팩터 전담자는 "수제 `useAsyncData` 3벌"을 보고도 통합만 할 뿐, **그것을 라이브러리로 대체할 권한이 없어** 더 나은 해법을 못 낸다. **기각.**

3. **코드 리뷰(Code Reviewer)가 검수와 리팩터를 겸한다.**
   **리뷰어는 지적하고 직접 수정하지 않는다(P1).** 리뷰어가 코드를 고치면 그 코드를 누가 승인하는가 — 자가 승인이다.
   G6 승인권자가 생산까지 하면 게이트가 무의미해진다. **기각.**

4. **클린코드 점검(Clean Code Inspector)이 위반을 발견하고 직접 고친다.**
   측정자가 수정하면 **재는 기준이 고칠 수 있는 만큼으로 수렴한다** — 고치기 어려운 위반은 조용히 임계값 밖으로 밀려난다.
   측정은 정직해야 하고, 정직하려면 측정자에게 수정 부담이 없어야 한다. **기각.**

## 결과

- **역할 정의 갱신**:
  - **프론트 구현** `title`: `React Front-End Engineer` → **`Frontend Engineer (Mid)`**. `outputs`: 화면 구현(React + Vite + TS). `notes`: 산출물은 프론트 리팩터의 검수를 거친다 · **의존성 임의 추가 금지**(필요 시 의존성 관리에 요청) · `packages/ui`는 public entry(`@tds/ui`)로만 import · 프론트 리팩터와 순차 배타.
  - **프론트 리팩터** `title`: `React Refactoring Agent` → **`Frontend Engineer (Senior)`**. `outputs`: **(1)** 중급 구현물 검수 → 클린코드 리팩터 **(2)** 라이브러리·OpenAPI 조사 및 앱 전반 일괄 적용. `notes`: **동작 변경 금지** · 도입 **정책은 의존성 관리** / **집행은 프론트 리팩터** · 위반 **측정은 클린코드 점검** / **수정은 프론트 리팩터** · 프론트 구현과 순차 배타.
  - `sequentialExclusiveWith` 상호 대칭(프론트 구현 ↔ 프론트 리팩터) 유지 — `apps/*/src/**` 공동 소유의 유일한 근거.
- **SKILL 2종 재작성**:
  - `skills/react-engineer/SKILL.md` — 정체성("화면을 만든다. 완벽할 필요는 없다"), 절대 금지에 **의존성 임의 추가 금지 · 대규모 리팩터 금지 · 프론트 리팩터 순차 배타 확인** 추가.
  - `skills/react-refactorer/SKILL.md` — 두 갈래 절차 **(A) 검수·클린코드** / **(B) 라이브러리·OpenAPI 일괄 적용**. DoD: **동작 회귀 0 · 클린코드 점검 위반 0 · 잔해 0 · 코드 리뷰 G6 승인**.
  - `skills/react-refactorer/references/refactor-scope-rules.md` — 리팩터 경계표(바꿔도 되는 것/안 되는 것) + **§6 라이브러리 교체 시 잔해 제거 체크리스트**(수제 훅 · 수동 검증 · 손으로 쓴 타입 · pass-through 어댑터 등 10종).
- **인접 역할과의 경계 명문화** — 의존성 관리(정책) / 프론트 리팩터(집행) / 클린코드 점검(측정) / 모듈 추출(계획) / 코드 리뷰(승인). 각 SKILL에 경계표로 박아 넣어 중복 소유·중복 판단을 막는다.
- **후속 (별건 PR)** — ADR-0006 §결과에서 프론트 구현/프론트 리팩터에 할당된 **페이지 간 결합 부채 상환**은 이 분리에 따라 **프론트 리팩터가 집행**한다:
  `members`에 얹힌 공통 요소(`Card`/`Button`/`Modal`/`Alert`/`HelpTip`/`styles`/`format`)를 `shared/ui`로 승격하고,
  `MemberTier`·`TIER_LABEL` 같은 도메인 값은 **prop 주입**으로 전환한다. 클린코드 점검 게이트 편입 **전에** 상환한다.
