# ADR-0010. 조직 감사 후속 조치

## 맥락

`docs/architecture/org-audit-2026-07-15.md`(감사 리포트)는 조직도가 아니라 **리포에 남은 파일**을 셌다. 그 결론은 하나다.

> **조직은 실패한 게 아니라 아직 시작되지 않았다.**

- `git log` = **커밋 1개**(`Initial commit`), 그 안에는 **`README.md` 하나뿐**이다. 나머지 **886개 파일 전부가 untracked**다.
- `pr-gates.yml`은 `on: pull_request`인데 **PR이 0건**이다. `.husky/`는 pre-commit 훅인데 **commit이 없다.**
- 결과: **경계 담당 · 네이밍 가드 · CI·CD · 클린코드 점검의 차단권(`blocks: PR/commit`)이 한 번도 발동하지 않았다.** 차단할 대상이 없었다.
- **검수 리포트(RR-\*)는 리포 전체에 0건**이다. 11개 게이트 전부 승인 기록이 없고, 리뷰어 소유 디렉터리 10개가 전부 빈 폴더다.

**게이트는 우회된 것이 아니라 물리적으로 연결된 적이 없다.** 산출물(계약 15 · 컴포넌트 15 · 화면 7 · 명세 8)은 훌륭하지만, 그것을 승인한 기록은 한 건도 없다.

그 위에 감사는 **구조적 결함 3건**을 찾았다. 이것들은 커밋을 하면 즉시 사고가 되는 것들이다.

| # | 결함 | 실측 |
|---|---|---|
| 1 | **테스트 파일에 소유자가 없다** | 47명의 `owns` 중 `**/*.test.*`·`e2e/**`에 매칭되는 것이 **0개**. → **경계 담당가 테스트 파일 생성을 '미소유'로 판정해 PR을 차단한다.** "아무도 테스트를 안 썼다"가 아니라 **"테스트를 쓸 수 있는 사람이 조직에 없다."** 프론트 구현의 SKILL(`:77`)은 렌더 테스트를 지시하는데 **쓸 경로가 없다** — SKILL과 레지스트리가 모순된다 |
| 2 | **보안 판정의 주체가 없다** | 보안 판정 **8건**(계정 열거 오라클 · 계정 탈취 벡터 · CSRF · 레이트리밋 · 멱등키 이중 지급 · 403↔404 은닉 · 세션 무효화 · 프론트 권한이 보안 경계가 아님)이 **전부 백엔드 명세(백엔드 문서 작가)에게서 우연히 나왔고 아키텍처에게 떠넘겨졌다.** `skills/` 47개 중 보안 검수 절차를 가진 스킬은 **0개** |
| 3 | **`reads` 스코프가 통제되지 않는다** | `reads: ["**/*"]` 보유 **7명** = **887파일 / 14MB** 스코프. `contracts/**` 읽기 권한 33/47명, `tokens/tokens.json` 21/47명. 같은 원본을 여러 에이전트가 처음부터 반복 정독한다 |

## 결정

### 1. 테스트 경로 소유권 부여 (P0 — 나머지의 선행 조건)

| 에이전트 | 추가되는 `owns` |
|---|---|
| **프론트 구현** (Frontend Engineer, Mid) | `apps/*/src/**/*.test.*` · `apps/*/src/**/*.spec.*` |
| **프론트 리팩터** (Frontend Engineer, Senior) | 동일 (프론트 구현과 `sequentialExclusiveWith` 쌍이므로 공동 소유가 허용된다 — 이 쌍만 예외로 통과된다) |
| **컴포넌트 엔지니어** (Storybook Component Engineer) | `packages/ui/src/**/*.test.*` |
| **E2E 테스트** (신설) | `e2e/**` · `apps/*/e2e/**` |

**이것이 다른 모든 조치의 선행 조건이다.** 이 한 줄이 없으면 테스트 커버리지도 E2E 테스트도 자기 임무를 수행할 수 없다 — 쓸 수 없는 파일의 커버리지를 재는 일은 성립하지 않는다.

### 2. 에이전트 3종 신설 (47 → 50)

| ID | 이름 | 계층 · 부서 | type | 소유 | 차단권 |
|---|---|---|---|---|---|
| **테스트 커버리지** | Test Coverage Guard | L3 verification | verifier | `tools/test-coverage/**` · `reports/test-coverage/**` | **blocks: G5 · G6** |
| **E2E 테스트** | E2E / Integration Test Engineer | L2 react | producer | `e2e/**` · `apps/*/e2e/**` | — (gate G6, reviewer 코드 리뷰) |
| **보안 리뷰** | Security Reviewer | L0 governance | reviewer | `docs/security/**` · `reports/security/**` | **blocks: G6 · G9** |

**보안 리뷰는 approver가 아니라 blocker다.** G6의 approver는 코드 리뷰, G9의 approver는 명세 리뷰로 **유지**한다 — **게이트 하나에 승인자가 둘이면 상태머신이 깨진다.** 대신 보안 리뷰는 `approves: []` · `blocks: [G6, G9]`로 두어, **코드 리뷰/명세 리뷰의 승인 *전에* 보안 blocker를 걸 수 있게** 한다. 판정은 PASS / BLOCKED 두 가지뿐이며, 보안 결함 1건이면 BLOCKED다 (우회 가능한 인증은 "조금 우회 가능"할 수 없다).

**부서 구성**: governance · planning · design · contract · token · storybook · react · figma · docs · verification.

### 3. 폐지 · 통합은 **보류한다** (오너 결정)

감사는 **9명의 폐지·통합**을 제안했다. **오너가 보류를 결정했다. 기존 47명은 그대로 둔다.** 상세와 재검토 조건은 §보류 참조.

### 4. 토큰 최적화 — Context Optimizer 신설 **없이**

**Context Optimizer 에이전트를 만들지 않는다.** 대신 4가지를 한다.

| # | 조치 | 내용 |
|---|---|---|
| **T1** | `docs/architecture/context-digest.md` 생성 | 자주 정독되는 파일의 요약본 — 토큰 목록(**경로만, 값 제외**) · 계약 15종(**이름 · level · props 키만**) · 게이트 요약 · 리포 구조 · 방어선 12규칙. **원본을 대체하지 않는다** — 정확한 값·타입·제약이 필요하면 원본을 읽으라고 문서 첫머리에 명시했다. **§7에 생성 방법(원천 · 추출 규칙 · 갱신 트리거)을 적어 드리프트를 구조로 막는다.** |
| **T3** | 레지스트리 `reads` 축소 — **7명 → 4명** | 아래 표 |
| **경계 담당 확장** | `reads` 스코프 검증 | 레지스트리 검증에 전역 읽기(`**/*`) **허용 목록 + 상한** 검사 추가. 목록 밖 보유자 또는 상한 초과 시 **경고**. `skills/boundary-enforcer/SKILL.md`에도 반영 |
| **T4** | 보고서 길이 상한 | **에이전트 산출 보고는 150행 이내** — 판정 + 지적사항 목록 + evidence 경로만. 근거 서술은 링크로 |

**T3 — `reads` 축소 결과 (7 → 4):**

| 에이전트 | 이전 | 이후 | 근거 |
|---|---|---|---|
| 오케스트레이터 Orchestrator | `**/*` | **유지** | 오케스트레이션은 전역 판단이 본질 |
| 아키텍처 Architecture | `**/*` | **유지** | 설계 거부권은 전역 판단이 본질 |
| 경계 담당 Boundary Enforcer | `**/*` | **유지** | 경계 감시는 **전수 검사**가 본질 |
| **보안 리뷰 Security Reviewer** (신설) | — | **`**/*`** | **비밀 노출은 어디에나 있을 수 있다.** 보안 검수 표면을 미리 열거하면 열거되지 않은 경로가 사각지대가 된다 |
| 네이밍 가드 Naming Guard | `**/*` | `apps/**` · `packages/ui/**` · `contracts/**` · `tokens/**` · `assets/**` · `docs/figma/specs/**` | 네이밍 판정 대상(Folder/Component/Variant/Variable/Token/Icon/Story/Page)만 |
| 의존성 관리 Dependency Manager | `**/*` | `**/package.json` · `pnpm-workspace.yaml` · `pnpm-lock.yaml` · `docs/adr/**` · `reports/**` | `MembersPage.tsx`(16.7KB)를 읽을 이유가 없다 — 의존성 판정에 필요한 것은 매니페스트와 lockfile이다 |
| 린트·포맷 Lint & Format | `**/*` | `*.config.*` · `**/*.ts` · `**/*.tsx` · `reports/code-quality/**` · `docs/adr/**` | 위반 목록은 **도구 출력**이지 소스 전문이 아니다 |
| CI·CD | `**/*` | `.github/**` · `.husky/**` · `**/package.json` · `reports/**` | 워크플로우 · 훅 · 스크립트 · 게이트 정의 |

**감사 처방은 3명(오케스트레이터/아키텍처/경계 담당)이었다. 보안 리뷰 신설로 4명이 됐다** — 이 확대는 위 표의 근거로 정당화되며, **더 늘리려면 ADR이 필요하다.** 레지스트리 검증의 `GLOBAL_READ_ALLOWLIST`가 이 목록을 코드로 고정한다.

**T6 (원칙, 도구 후속)**: 네이밍 가드 · 린트·포맷 · 클린코드 점검은 **소스를 읽지 말고 도구 리포트를 읽는다.** 위반이 0건이면 소스를 열 이유가 없다. 이 원칙을 각 SKILL의 절차에 반영하는 것은 후속 작업이다.

## 근거

- **테스트 경로 (결정 1)** — 감사 발견 2: *"47개 역할의 `owns` 배열 전수 검색 결과 테스트 경로에 매칭되는 것이 0개다."* 경계 담당의 판정 로직은 소유자 없는 파일을 '미소유'로 기록하고 PR을 차단한다. **지금 이 조직은 테스트를 쓰는 것이 경계 위반이다.** 그리고 프론트 리팩터의 리팩터 규율 전체(`skills/react-refactorer/SKILL.md:46` — *"기존 테스트를 한 글자도 고치지 않고 전부 통과하는가?"*)가 그 위에 서 있다. 프론트 리팩터가 방금 끝낸 페이지 결합 해소 리팩터(ADR-0007)는 **동작 보존을 증명할 방법 없이** 병합됐다.

- **테스트 커버리지 (결정 2)** — 감사 발견 2: `pnpm test` = `vitest run --passWithNoTests`이고 리포 전체 `*.test.*` **0건**인데, **G5 체크리스트 #6과 G6 체크리스트 #7이 바로 이 명령을 자동 증거로 인용한다.** 테스트 0건 위에서 *"계약의 모든 상태에 렌더 테스트가 있다"*는 **공집합 위에서 참인 명제**다. 그 참은 아무것도 보증하지 않는다. **"통과했다"와 "검증되지 않았다"를 구분하는 주체가 없었다.** 그리고 커버리지 목표는 라인 %가 아니다 — **"계약이 정의한 상태 전부 + FS가 정의한 예외 축 전부"**다. 90% 라인 커버리지는 계약의 상태 하나가 빠져 있어도 나온다.

- **E2E 테스트 (결정 2)** — ADR-0004: *"프론트·백엔드·**QA**·기획이 같은 좌표계를 쓰는 것이 명세서의 존재 이유"* — **그런데 QA가 조직에 없었다.** FS-NNN은 이미 실행 가능한 시나리오다: 요소 전수 넘버링(`FS-001-EL-016`)과 예외 7축(빈·로딩·실패·유효성·권한없음·경합·대량)이 표로 채워져 있다. **표의 한 칸이 테스트 케이스 하나다.** 명세가 적발한 구현 결함 29건 중 **회귀를 막을 장치는 0개**였다 — 멱등키 이중 지급(*"적립금은 돈이다"*) 같은 결함은 다음 배포에서 되살아난다. E2E 테스트는 시나리오를 **지어내지 않고 번호를 인용한다** — 테스트 이름에 `FS-NNN-EL-nnn`을 박아 테스트 커버리지가 FS 예외 표와 기계 대조할 수 있게 한다.

- **보안 리뷰 (결정 2)** — 감사 발견 4: 보안 판정 8건이 **전부 백엔드 명세에게서 나왔다.** 백엔드 명세는 백엔드를 구현하지 않는 문서 작가다. **그가 사실상의 보안 아키텍트였다.** 그중 아키텍처에게 에스컬레이션된 2건(step-up 재인증 · 세션 자격 증명 전달 방식)은 **아직 미응답**이다. **아키텍처는 보안 판정자가 아니다** — 아키텍처의 SKILL·레지스트리 어디에도 보안 권한·절차·체크리스트가 없다. verification 부서는 8명이 8개 축을 재지만 **보안 축이 없었고**, 이 제품은 **관리자 백오피스**다: 개인정보를 보고, 비밀번호를 바꾸고, **적립금을 지급한다.** 다음 화면(결제·주문·쿠폰)에서 같은 판단이 또 나온다.

- **Context Optimizer를 만들지 않는 이유 (결정 4)** — 감사 §3.3의 논증을 채택한다.
  1. **이 조직에서 에이전트는 소유 경로 + 판정 권한을 가진 주체다.** Context Optimizer는 **무엇을 소유하는가?** `context-digest.md`는 아키텍처의 `docs/architecture/**` 안이며, **요약본과 원본이 다른 소유자에게 갈라지면 즉시 드리프트한다** (ADR-0001 결정 2: *"소유권 매트릭스를 두 곳에 수기 유지하면 반드시 드리프트한다"*). **요약은 원본 소유자가 쓰거나, 원본에서 생성해야 한다.**
  2. **차단권 없는 감시자는 읽히지 않는 리포트를 쓴다.** 이 리포에 이미 증거가 있다 — 디자인 드리프트의 드리프트 경고(**미사용 토큰 27.1%**, SLO의 5배)에 **응답 0건**이다.
  3. **문제의 성격이 조직이 아니라 데이터다.** 낭비의 근원은 `reads` 글롭(레지스트리 필드)과 요약본 부재(파일 1개)이며, **둘 다 기계적으로 해결된다.**
  → 대신 **아키텍처(요약본 · reads 축소)와 경계 담당(스코프 검증)에 할당한다. 신설 0명, 도구 1개 확장.**

## 대안

1. **테스트 경로에 별도 소유자(예: QA 역할)를 두고 프론트 구현/컴포넌트 엔지니어에서 분리한다.**
   테스트와 구현이 다른 소유자에게 갈리면 **프론트 구현이 구현을 바꿀 때 테스트를 고칠 수 없다** — 매번 변경 요청을 거쳐야 한다. 그리고 프론트 리팩터의 리트머스 테스트(*"기존 테스트를 한 글자도 고치지 않고 통과하는가?"*)는 **테스트가 구현자와 같은 손에 있을 때만** 의미를 갖는다. 단위 테스트는 **구현의 일부**다. **기각.**
   다만 **화면 단위 시나리오는 다르다** — 그것은 구현이 아니라 **명세의 집행**이므로 별도 소유자(E2E 테스트)가 옳다.

2. **보안 리뷰을 G6·G9의 공동 approver로 둔다.**
   게이트 하나에 승인자가 둘이면 **상태머신이 깨진다** — 누구의 APPROVED가 게이트를 여는가? 한쪽이 승인하고 한쪽이 반려하면? 게이트 정의의 `stateMachine`은 게이트당 단일 approver를 전제한다. **차단권만 주면 이 문제가 사라진다**: 코드 리뷰/명세 리뷰가 여전히 유일한 승인자이고, 보안 리뷰는 그들이 승인하기 **전에** 막는다. **기각(차단권 모델 채택).**

3. **보안을 아키텍처(아키텍트)가 계속 겸임한다.**
   이것이 **현 상태**이며, 실측 결과는 **미응답 2건**이다. 아키텍처는 이미 10개 부서 중 7개의 에스컬레이션 종착지이고, 여기에 보안 판정·도메인 경계 판정·번호 체계 판정이 얹혀 있다 (감사 §2.3: *"진짜 1인 병목은 계약 엔지니어가 아니라 아키텍처다"*). **과부하가 미응답의 원인이다.** 그리고 아키텍처의 SKILL에는 보안 체크리스트가 **없다** — 절차 없이 판정하면 판정은 매번 달라진다. **기각.**

4. **테스트 커버리지 없이 `--passWithNoTests`만 제거한다.**
   그러면 CI가 빨개지고 **누군가 다시 켠다.** 임계값을 지키는 주체가 없으면 임계값은 위반이 생길 때마다 조용히 완화된다 (ADR-0006 대안 2가 이미 내린 판단). 그리고 `--passWithNoTests` 제거는 "테스트가 1건이라도 있으면 통과"를 만들 뿐, **계약의 states가 덮였는지는 여전히 아무도 재지 않는다.** **기각 — 도구 수정은 필요하지만 충분하지 않다.**

### 보류의 근거 — 그리고 보류의 대가

**보류가 옳을 수 있는 이유**: 감사의 판정은 **"산출물 0건"**에 근거하는데, 이 조직은 **아직 시작되지 않았다**(커밋 1개, PR 0건). **한 번도 돌지 않은 조직에서 "안 돌았다"를 근거로 폐지하는 것은 순환 논증에 가깝다.** 게이트를 CI에 물린 뒤 **한 사이클을 실제로 돌려보고** 판단하는 것이 정직하다.

**보류의 대가**: 조율 비용은 인원 수에 비선형으로 는다. 50명 중 **27명이 파일 한 개도 남기지 않았다.** 다음 사이클에도 같은 결과가 나오면, 그때는 **"역할이 없었다"가 아니라 "역할을 강제하는 장치가 없었다"는 변명이 소진된다.**

**따라서 공통 재검토 시점을 못박는다: 2026-Q4 사이클 종료 시.** 그때 RR-\* 리포트가 여전히 0건이면, **폐지 논의는 개별 에이전트가 아니라 리뷰어 11명 전체를 대상으로 다시 연다** — 두 번 안 도는 역할은 역할이 아니다.

## 추가 결정 — 커버리지 게이트의 하한 설계 (테스트 커버리지 구현 중 발견, 아키텍처 판정)

테스트 커버리지가 도구를 구현하면서 **자기 도구의 공허 통과 경로 5건을 스스로 신고**했다. 그중 2건이 하한 설계의 결함이었고, 아래와 같이 판정한다.

### 결정 1 — 축1(테스트 존재)을 **워크스페이스 스코프별로 분리**한다

**문제**: 축1이 리포 전역 카운트였다. `packages/ui`에 테스트가 생기면 **`apps/**`가 0건이어도 축1이 초록으로 바뀐다.**

**이것은 가설이 아니었다.** 테스트 커버리지가 도구를 고치는 동안 컴포넌트 엔지니어가 `packages/ui`에 테스트 9건을 착지시켰다. 구 버전이었다면 축1은 그 순간 GREEN이 됐고, `apps/admin`의 **0건은 그 초록에 가려졌다.** 예측한 사고가 같은 날 실현될 뻔했다.

**결정**: 스코프별로 독립 판정한다. 어느 한쪽이 0건이면 **그 스코프가 blocker**다. **한쪽의 초록이 다른 쪽의 빈칸을 가리지 못한다.**

**스코프 목록은 `pnpm-workspace.yaml`에서 파생한다 — 하드코딩 금지.** 지금 손으로 두 개를 박아 두면, 세 번째 패키지가 추가되는 날 같은 구멍이 다시 열린다. (`agent-registry.v1.json`의 `minItems`를 에이전트 수의 정본으로 삼은 것과 같은 원리다.)

**`tools/*`는 축1 대상에서 제외한다.** 검증 도구의 테스트 요구는 축5(골든 픽스처)가 담당한다. 넣으면 **레지스트리가 인가하지 않은 blocker 11건을 도구가 발명**하게 된다 — 테스트 커버리지가 스스로 세운 원칙("도구가 blocker를 발명하지 않는다")의 정면 위반이다.

### 결정 2 — 축4(FS 예외축)는 major 유지 + **래칫(후퇴 금지)**

**기각한 대안 — blocker 승격**: 현재 713칸이 미커버다. 승격하면 리포 전체가 **무기한 RED**가 되고, RED가 상시화되면 사람은 게이트를 우회하기 시작한다. **항상 빨간 신호등은 항상 초록인 신호등과 똑같이 무용하다.** (`bundle-size` job을 되살리지 않은 것과 같은 논리 — ADR-0009.)

**기각한 대안 — 현상 유지**: "코드 리뷰가 사람 눈으로 본다"는 감사가 **이미 실패로 판정한 방식**이다.

**결정**: 직전 리포트의 커버 칸 수를 기준선으로 삼고, **줄어들면 blocker**. 새 테스트를 요구하지는 않되 **있던 커버리지를 잃는 것은 차단**한다. 713칸은 0에서 시작해 **단조 증가만** 한다. 채워 넣는 속도는 조직의 속도지 게이트의 문제가 아니다.

**래칫은 규율의 기계 집행자이기도 하다** — 테스트가 지워지든, 이름에서 요소 번호가 빠지든, 단언이 제거되든 커버 칸이 줄어 걸린다.

**⚠ 미완의 전제 (CI·CD 인계)**: 래칫의 기준선은 **리포트 파일**이다. **CI 캐시가 초기화되면 기준선이 0으로 떨어져 후퇴가 은폐된다.** `reports/`를 커밋 또는 아티팩트로 **영속화하기 전까지 이 결정은 반쪽이다.**

### 결정 3 — `.prettierignore`에 `reports/**` 추가

도구는 `JSON.stringify(r, null, 2)`로 쓰고 Prettier는 짧은 배열을 접는다(`["G5","G6"]`). 그 결과 `format:check`가 **도구가 방금 쓴 리포트를 반려**했다. **포매터를 통과시키려고 도구 출력 포맷을 Prettier에 맞추는 것은 꼬리가 개를 흔드는 일이다.** 린트·포맷의 원칙 그대로다 — 사람이 손으로 쓰지 않는 파일은 포매터가 건드리지 않는다.

## 이 사이클에서 확인된 가장 큰 사실

**G5 게이트는 한 번도 통과한 적이 없다. 통과했다고 보고됐을 뿐이다.**

| | 실측 |
|---|---|
| Storybook play function | **62건** |
| 그중 `expect`를 가진 것 | **0건** |
| 스파이(`fn()`) 주입 | **0건** |

play들은 전부 `userEvent.hover/tab/pointer`로 **상태를 만들기만** 하고 아무것도 단언하지 않았다. **실패할 수 없는 검사였다.** 그런데 G5의 exit 조건 *"Play Function으로 `events.blockedWhen` 전수 검증"*이 **이 62개의 초록불로 통과돼 왔다.**

그리고 `blockedWhen` 검증은 없는 것이 아니라 **물리적으로 불가능했다** — 콜백의 *비*발생을 관찰할 스파이가 리포 전체에 하나도 주입된 적이 없기 때문이다.

**교훈은 `--passWithNoTests`와 정확히 같다: 검사의 존재와 검사의 작동은 다른 것이며, 게이트는 후자만을 증거로 받아야 한다.** 테스트 커버리지의 축3이 단언의 **종류**까지 검사하는 이유가 이것이다 — `expect(button).toBeDisabled()`는 `onClick`이 발화하지 **않음**을 증명하지 못한다. `disabled` 속성 없이 CSS로만 흐리게 처리하고 핸들러를 물려도 통과하기 때문이다.
