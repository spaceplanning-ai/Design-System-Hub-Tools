# ADR-0013. 엔타이틀먼트(플랜) 층

## 맥락

이 어드민은 지금까지 **판정 축이 하나**였다 — 권한(RBAC). 어떤 화면을 못 보면 이유도 하나였다:
"권한이 없다". 그래서 거절 화면도 한 종류였다(`ForbiddenScreen` — '접근 권한이 없습니다').

그런데 제품이 플랜(구독 등급)으로 팔리기 시작하면 **못 보는 이유가 둘**이 된다.

| 이유 | 운영자가 해야 할 일 | 누가 풀어 줄 수 있나 |
|---|---|---|
| 권한이 없다 | 관리자에게 권한을 요청한다 | 이 테넌트의 최상위 관리자 |
| **사지 않았다** | 플랜을 올린다(= 계약을 바꾼다) | **아무도 이 앱 안에서는 못 한다** |

두 번째를 첫 번째의 문구로 말하면 **운영자는 실행 불가능한 지시를 받는다.** 관리자에게 권한을 요청하고,
관리자는 권한 화면을 열고, 거기에 그 항목이 없거나 켜도 안 열린다. 그다음은 지원 티켓이다.
"권한이 없습니다" 한 문장이 두 사람의 시간을 쓰고 결국 아무것도 해결하지 못한다.

그래서 축을 하나 더 만들되 — **권한 축에 끼워 넣지 않는다.**
끼워 넣으면 권한 매트릭스가 "관리자가 켤 수 있는 것"과 "계약이 정하는 것"을 같은 체크박스로 그리게 되고,
그 순간 관리자는 자기가 켤 수 없는 스위치를 보게 된다.

## 결정

### 1. 판정은 네 층이고, 순서는 인증 → **플랜** → 권한 → 설정이다

```
① 인증        없으면 /login              RequireAuth
② 엔타이틀먼트 없으면 숨김 또는 잠금       RequireEntitlement
③ 권한(RBAC)  없으면 403                 RequirePermission
④ 도메인 설정  off 면 화면 안 문맥 전환    pgLock() · resolvePriceDisplay() 등
```

이 순서는 스케줄러가 아니라 **JSX 중첩이 그대로 표현한다** — `AppShell` 에서
`<RequireEntitlement>` 가 `<RequirePermission>` **바깥**에 있다. 정본 주석은
`shared/entitlements/plan.ts` 머리말과 `RequireEntitlement.tsx` 머리말 두 곳에 같은 상자로 있다.

**왜 플랜이 권한보다 먼저인가**: 위 표의 두 번째 행 때문이다. 사지 않은 기능에 403 을 내면
운영자는 고칠 수 없는 방향으로 움직인다. 반대로 산 기능에 잠금 화면을 내면 그건 그냥 버그이고,
그 버그는 순서를 뒤집었을 때만 생긴다.

**사이드바에서는 순서가 아니라 결과가 겹친다** — 권한이 없으면 잎이 사라지고, 플랜이 `absent` 여도
사라지고(모듈 전체가 `absent` 면 가지 제목까지), 플랜이 `locked` 면 **잎이 남고 라벨에 `' · 잠금'` 이 붙는다.
살 수 있는 것은 보여야 팔린다.
순서가 의미를 갖는 것은 **URL 로 직접 들어왔을 때** — 그때 무슨 화면을 보여 줄지가 순서로 갈린다.

### 2. 판정 결과는 3상태다 — `granted | locked | absent`

```ts
export type EntitlementState =
  | { readonly kind: 'granted' }
  | { readonly kind: 'locked'; readonly reason: string; readonly upgradeTo: PlanTier }
  | { readonly kind: 'absent' };
```

불리언 두 개(`allowed` + `sellable`)로 쪼개지 않은 이유는 `locked` 가 **혼자서는 쓸모없기 때문**이다.
"잠겨 있다"는 화면에 그릴 수 없다 — **무슨 플랜을 사면 열리는지**를 같이 말해야 문장이 된다.
그래서 `reason` 과 `upgradeTo` 를 상태와 한 벌로 낸다. 호출부가 따로 조회하게 두면 언젠가 한쪽만 갱신된다.

`absent` 는 `locked` 의 특수한 경우가 아니다. **살 수 없는 모듈**이다(카탈로그의 `minTier` 가 `null`).
살 수 없는 것에 '업그레이드하세요'를 보이면 그건 거짓말이다. 그래서 `absent` 는 화면도 문구도 없다 —
사이드바에서 빠지고, 라우트는 `/dashboard` 로 `replace` 하고, 플랜 화면의 표에도 행이 올라가지 않는다.

### 3. 거절 문구는 **네 가지**이고 절대 섞지 않는다

| # | 상황 | 표면 | 운영자에게 남는 다음 행동 |
|---|---|---|---|
| 1 | 권한 없음 | `ForbiddenScreen` — '접근 권한이 없습니다' | **관리자에게 권한 요청** |
| 2 | 플랜에 없음(`locked`) | `UpgradeScreen` — '{플랜} 플랜에서 사용할 수 있습니다' + `플랜 보기`·`현재 플랜 확인` | **결제/계약** (앱 밖) |
| 3 | 살 수 없음(`absent`) | **없음** — 메뉴에서 빠지고 라우트는 대시보드로 | 없음(존재를 알릴 이유가 없다) |
| 4 | 테넌트가 그렇게 운영하기로 함 | `PgLockNotice` 등 **info 톤** 안내 + `결제 설정 열기` | **설정 변경** (같은 앱 안) |

네 번째가 `tone="info"` 인 것이 중요하다 — **잠금은 오류가 아니라 설정의 결과**다.
danger 톤으로 그리면 운영자는 자기가 켠 설정을 장애로 읽는다(자세한 것은 ADR-0014).

이 넷을 섞지 않는다는 것은 **한 화면이 두 문구를 동시에 보이지 않는다**는 뜻이기도 하다.
판정 순서(①②③④)가 그것을 구조로 보장한다 — 바깥 층에서 막히면 안쪽 층은 렌더되지 않는다.

### 4. 실패 방향이 축마다 **정반대**다

| 축 | 모를 때 | 근거 |
|---|---|---|
| 권한 | **닫는다**(fail-closed) | 모르는 권한을 열어 주면 그것이 사고다. 새 역할은 전 권한 OFF 로 시작하고, 저장값이 모순이면 더 좁은 쪽으로 되돌린다 |
| 엔타이틀먼트 | **연다**(fail-open) | 고객이 **이미 산** 기능을 우리 조회 실패로 멈추면 그것이 사고다 |

fail-open 이 코드에서 수렴하는 지점은 하나가 아니라 여럿이고, 전부 같은 방향이다:

- 카탈로그에 없는 키 → `granted`
- 저장된 값이 `undefined` → `granted`
- localStorage 접근 불가 / JSON 파손 → `DEFAULT_PLAN_STATE`(전 기능 on)
- 개별 키의 형태가 깨짐 → **그 키를 없는 것으로** 둔다(없으면 `granted` 로 읽힌다)
- 버전 불일치 → `DEFAULT_PLAN_STATE`
- 라우트에 매핑이 없음 → `granted`

**"파손이 차단이 되지 않는다"** 가 이 축의 한 줄 규칙이다.

권한 축에도 좁은 fail-open 이 하나 있다 — 저장된 JSON 에 **없는** 리소스는 `read: true` 로 본다
(`unseenResourceGrant`). 화면이 새로 생겼는데 저장된 역할이 그것을 몰라 아무도 못 보는 사고를 막는 장치이며,
읽기 아닌 액션은 알려진 리소스 전체가 만장일치일 때만 준다. 엔타이틀먼트 축은 **그 선례를 일반화한 것**이지
새로 발명한 것이 아니다.

### 5. **플랜 변경 UI 를 만들지 않는다**

구독·결제·계약은 사내 어드민과 홈페이지 소관이다. 여기에 변경 수단을 두면 **실제 계약과 어긋나는
두 번째 정본**이 생긴다 — 화면에서 pro 로 올렸는데 청구는 basic 인 상태가 만들어지고, 그때 어느 쪽이
참인지 판정할 근거가 이 앱에는 없다.

그래서:

- `/settings/plan` 은 **조회 전용**이다. 폼도 저장 버튼도 없고 쓰기 엔드포인트도 없다
  (`TODO(backend): GET /api/tenant/entitlements` 하나뿐).
- 나가는 길은 상수 URL 하나(`PLAN_PORTAL_URL`)이고 화면이 그렇게 말한다 —
  '플랜을 바꾸려면 사내 홈페이지의 요금제 안내에서 진행해 주세요. 이 화면에서는 바꿀 수 없습니다.'
- 상태를 들여오는 함수(`receivePlan`)는 **인바운드 전용**이고 호출부가 0건이다 — UI 에서 닿는 경로가 없다.
- 등급을 바꾸는 유일한 손잡이 `PlanDevPanel` 은 `import.meta.env.DEV` 분기 안에서만 마운트된다
  (번들러가 운영 빌드에서 상수 `false` 로 접는다).

`UpgradeScreen` 의 버튼이 앱 안이 아니라 **밖으로** 나가는 것도 같은 결정의 결과다.

### 6. 키는 **모듈 단위**이고 권한 좌표계와 분리된 파일이 잇는다

`ENTITLEMENT_KEYS` 는 `{도메인}.{모듈}` 11개다(`commerce.orders`·`commerce.products`·`commerce.coupons`·
`commerce.points`·`commerce.shipping`·`sales.pipeline`·`cms.pages`·`marketing.email`·`marketing.sms`·
`ai.agent`·`stats.advanced`).

**화면 단위가 아니다.** 플랜은 "주문 관리를 살 수 있다"를 팔지 "주문 목록 화면"을 팔지 않는다.
화면이 늘거나 쪼개져도 판매 단위는 그대로여야 한다.

키를 화면에 잇는 것은 **별도 파일**(`module-resources.ts`)이고, 그 파일은 권한 축이 이미 쓰는
좌표계(`page:{to}` / `group:{basePath}`)를 그대로 재사용한다. 두 축이 좌표계를 공유하되 **정책은 공유하지 않는다** —
같은 경로에 대해 서로 다른 답을 낼 수 있어야 두 축을 나눈 의미가 있다.

엔타이틀먼트 값은 세 종류다: `switch`(on/off) · `quota`(상한 + 사용량) · `level`(등급 문자열).
`quota` 는 **생성만** 막고 조회를 막지 않는다 — 상한을 넘겼다고 이미 만든 것을 못 보게 하면
그건 데이터를 인질로 잡는 것이다. 같은 논리로 청구 미납(`past_due`·`suspended`)은 **조회 전용으로 내릴 뿐**
기능을 `locked` 로 만들지 않는다.

## 결과

- 못 보는 이유가 **화면에서 구별된다.** 403 을 받은 운영자는 관리자에게 가고, 잠금을 받은 운영자는 계약으로 간다.
- 엔타이틀먼트 조회가 깨져도 **앱이 멈추지 않는다.** 최악의 경우 안 산 기능이 잠깐 열리는데,
  그 반대(산 기능이 전부 잠기는 것)보다 회복이 싸다.
- 플랜을 바꾸는 정본이 **하나로 유지된다.**

## 이 결정이 만드는 부채

1. **엔타이틀먼트 축에 컴포넌트 테스트가 0건이다.** `plan.ts` 의 순수 함수는
   `plan.test.ts`·`route-entitlement.test.ts` 가 잘 덮는다(fail-open 전용 describe 까지 있다).
   그러나 `UpgradeScreen` 을 렌더하는 테스트, `RequireEntitlement` 의 중첩 순서를 고정하는 테스트,
   사이드바의 `LOCKED_NAV_SUFFIX` 를 단언하는 테스트가 **리포 전체에 없다** — 즉
   **§1 의 판정 순서와 §3 의 네 문구는 지금 기계가 지키지 않는다.** e2e 도 이 축을 건드리지 않는다.
2. `module-resources.ts` 가 인용하는 `module-resources.test.ts` 는 **실재하지 않는다.**
   그 단언은 `route-entitlement.test.ts` 에 있다 — 주석이 없는 파일을 가리킨다.
3. 사용량 조회기는 `commerce.products` 하나만 배선돼 있다. 나머지 quota 키가 생기면 각각 배선해야 하고,
   미배선은 `null`(= 저장된 값 사용)로 조용히 떨어진다 — fail-open 이라 안전하지만 **말없이** 틀린다.
4. `receivePlan` 은 호출부가 0건이다. 사내 어드민에서 상태를 들여오는 실제 경로가 붙기 전까지
   운영 환경의 플랜은 **언제나 `DEFAULT_PLAN_STATE`(enterprise)** 다 — 즉 지금 이 축은 실질적으로 항상 열려 있다.
