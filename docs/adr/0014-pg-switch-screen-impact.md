# ADR-0014. PG 스위치가 바꾸는 화면들

## 맥락

이 어드민은 **결제(PG)를 쓰는 사이트와 쓰지 않는 사이트를 같은 코드로** 운영한다.
PG 를 쓰지 않는 사이트에서는 구매 버튼이 '문의하기'가 되고, 실질적인 '결제'는 영업 쪽의 청구·입금이 된다.

이 하나의 사실이 **어드민 화면 여러 곳을 동시에 바꾼다.** 상품 폼의 적립·쿠폰·배송 구획, 재고 수량,
쿠폰 화면, 적립금 정책, 매출·주문 통계, 대시보드, 그리고 사이드바의 '문의' 메뉴까지.

바꾸는 방식을 화면마다 각자 정하게 두면 세 가지가 곧바로 어긋난다 — **무엇을 잠글지**, **왜 잠겼다고
말할지**, 그리고 가장 위험한 **잠글 때 값을 어떻게 할지**.

## 결정

### 1. 축은 둘이다 — 하나로 묶지 않는다

| | 축 A `pgSellable` | 축 B `priceDisplay` |
|---|---|---|
| 범위 | 사이트 전역 (`/settings/payment`) | **상품별** (상품 폼) |
| 값 | 파생 불리언 | `'amount' \| 'inquiry'` + 대체 문구(`inquiryText`) |
| 실패 방향 | **fail-closed** | — (열거형이라 미지 상태가 없다) |
| 무엇을 정하나 | 결제 자체가 가능한가 | 이 상품의 **금액 자리에** 무엇을 그리나 |

`pgSellable` 은 저장된 값이 아니라 **파생값**이다: `usePg && merchantId.trim() !== ''`.
'쓴다'고 켜 두고 상점 ID 가 비어 있으면 결제창을 띄울 수 없으므로, 그 상태는 **켠 것으로 치지 않는다.**
애매하면 닫는 쪽으로 수렴한다 — 켜져 있다고 잘못 믿으면 **결제되지 않는 '구매하기' 버튼**이 고객에게 나간다.
출하 기본값도 같은 이유로 off 다.

**하나로 묶지 않는 이유**: 묶으면 *"PG 는 쓰지만 이 상품만 가격문의"* 를 표현할 수 없다.
이것은 예외 사례가 아니라 흔한 운영이다(견적이 필요한 대형 품목, 옵션이 복잡한 맞춤 제작).
전역 스위치 하나로 축약하면 그 사이트는 상품 하나 때문에 전체를 문의 모드로 내려야 한다.

**전역이 개별을 이긴다.** `resolvePriceDisplay(settings, policy)` 가 축 A 를 먼저 보고,
`pgSellable` 이 false 면 상품이 무엇으로 설정돼 있든 문의로 확정한다. 반대 방향은 성립하지 않는다 —
PG 를 켜 두어도 상품이 `'inquiry'` 면 **그 상품만** 문구로 대체된다. 두 축의 관계는 비대칭이고,
그 비대칭이 한 함수 안에 있어 화면들이 각자 판단하지 않는다.

### 2. 잠금은 **입력만** 막는다 — 저장된 값을 지우지 않는다

이것이 이 ADR 의 가장 중요한 규칙이다.

PG 를 끄면 여섯 구획이 잠긴다(`PG_LOCK_SECTIONS`):
`product-points` · `product-coupons` · `product-stock` · `product-shipping` · `coupon-admin` · `points-policy`.

잠금의 구현은 어디서나 `disabled || lock.locked` 한 줄이고, **어디에서도 `reset`·`setValue('')` 를 하지 않는다.**
결제를 다시 켜면 적립률·쿠폰 설정·배송비·재고가 **저장해 둔 그대로** 살아난다.

값을 지우는 구현이 왜 위험한가: 운영자가 결제 설정을 잠깐 껐다 켜는 것은 흔한 일이고
(연동 교체, 심사 대기, 테스트), 그때마다 상품 수백 건의 적립률이 0 이 된다면 그건 설정 변경이 아니라
**데이터 손실**이다. 그리고 손실은 되돌린 뒤에야 발견된다.

**옵션은 잠기지 않는다.** 잠기는 것은 그 옆의 **재고 수량**뿐이다.
결제가 없어도 옵션(색상·사이즈)은 계속 편집할 수 있어야 한다 — 문의로 받은 요청의 **품목 명세**로 쓰이고,
그 명세가 그대로 견적의 품목 라인이 된다. 상품 등록·수정 자체도 잠기지 않는다.

### 3. 잠금 문구는 **info 톤**이고, 사유와 보존 약속을 함께 말한다

`PgLockNotice` 는 `tone="info"` 다 — **잠금은 오류가 아니라 설정의 결과**다.
danger 로 그리면 운영자는 자기가 켠 설정을 장애로 읽고 지원 티켓을 연다.

각 구획의 사유 문자열은 `pg-lock.ts` 한 곳에 모여 있고, 전부 같은 문형을 쓴다 —
**왜 잠겼는가 + 무엇이 보존되는가**. 예:

> 결제가 없어 적립이 발생하지 않습니다. 저장된 적립 설정은 그대로 보존됩니다.
> 결제가 없어 재고를 차감할 주문이 들어오지 않습니다. 옵션 구성은 계속 편집할 수 있습니다 — 문의로 받은 요청의 품목 명세가 됩니다.

이 문구는 ADR-0013 §3 의 **네 번째** 거절 표면이다(설정 안내). 403 문구도 업그레이드 문구도 쓰지 않는다 —
이건 권한 문제도 계약 문제도 아니고, **이 테넌트가 고른 운영 방식**이다.

### 4. 통계는 0 을 그리지 않고 **지표를 치환한다**

결제가 꺼져 있으면 매출도 주문도 구조적으로 0 이다. 그 0 을 차트로 그리면 운영자는 **장애로 읽는다.**

그래서 매출 통계·주문 통계는 화면 전체를 `InquiryStatsPanel` 로 바꾼다 —
문의 건수 · 평균 응답시간 · 견적 발행 · 견적 수락률. 대시보드도 같은 술어로 갈린다.

두 가지를 함께 지킨다:

- **모름과 0 을 구분한다.** 집계를 못 불러왔으면 0 이 아니라 '(아직 집계를 불러오지 못했습니다)' 를 붙인다.
- **되돌아온다고 약속한다.** '결제를 켜면 이 화면은 다시 {매출 통계}로 돌아옵니다. 지난 기록은 사라지지 않습니다.'

### 5. '문의' 메뉴는 **조건부로 보이되, 라우트는 살아 있다**

상품·프로그램의 '문의' 잎은 `visibleWhen: 'pg-off'` 를 달고 있다. 조건을 렌더 코드의 `if` 가 아니라
**메뉴 데이터**에 적은 이유: 사이드바 렌더 함수에 심으면 '어느 메뉴가 조건부인가'를 알기 위해
렌더 함수를 읽어야 한다. 메뉴는 데이터고, 조건도 메뉴의 성질이다.

판정은 `inquiryMenuState(settings, backlog)` 3상태다:

| 상황 | 결과 | 화면 |
|---|---|---|
| PG 를 못 쓴다 | `open` | 평소대로 — 문의는 지금도 들어온다 |
| PG 를 쓰고 잔여 0건 | `hidden` | 메뉴에서 사라진다 |
| PG 를 쓰고 잔여 있음 | `archive` | 라벨에 `' · 읽기 전용'` 이 붙는다 |
| 잔여를 **모른다**(미배선) | `archive` | 남긴다 — fail-open |

`total` 은 종결까지 포함한 **전체** 문의 수다. '처리 안 된 것만' 으로 세면 다 처리한 순간 과거 기록이 사라진다.

그리고 **`collectNavRoutes()` 는 이 필터를 타지 않는다.** 라우트는 메뉴와 무관하게 등록되므로
북마크·이메일 링크로 들어온 과거 문의 화면은 계속 열린다. **감추는 것과 없애는 것은 다른 결정이다.**
`findCoveringLeaf` 도 같은 목록을 걷기 때문에, 감춰진 잎으로 deep-link 해도 권한·엔타이틀먼트 판정은 정상 작동한다.

## 결과

- PG 를 껐다 켜도 **아무 값도 사라지지 않는다.**
- 잠긴 화면이 "권한이 없습니다"라고 말하지 않는다 — 사유와 복구 경로(`결제 설정 열기`)를 말한다.
- 전역과 개별이 **동시에 존재**하므로 "PG 는 쓰되 이 상품만 가격문의"가 표현된다.
- 결제를 안 쓰는 사이트의 통계 화면이 **빈 차트가 아니라 자기 지표**를 보여 준다.

## 이 결정이 만드는 부채

1. **결제 설정은 백엔드가 없다.** `PaymentSettings` 는 모듈 스코프 변수에 산다 —
   새로고침하면 기본값(off)으로 되돌아간다. 즉 지금 **개발 중에는 항상 PG-off 경로가 기본**이다.
2. **적립금 정책 화면의 잠금은 '저장 금지'를 간접적으로만 강제한다.** 사유 문구는 '정책은 열어 두되
   저장은 결제를 켠 뒤에 반영하세요' 라고 말하고, 실제로는 필드 전부가 `disabled` 라 dirty 가 될 수 없어
   저장이 열리지 않는다. 즉 **문구는 권고인데 동작은 금지**다 — 문구가 동작보다 무르다.
3. **문의 문구가 두 개다.** 전역 `inquiryGuide`(결제 설정 — 구매 CTA 아래 안내문)와 상품별
   `inquiryText`(금액 자리 대체 문구)는 쓰임이 다른 값인데 이름이 비슷하다. 상속 관계는 없고
   `inquiryText` 만 '비어 있으면 상수 기본값(`가격문의`)' 규칙을 갖는다. 둘을 혼동한 설정이 나오면
   화면에서만 드러난다 — 폼이 서로를 설명하지 않는다.
4. `pgLock` 이 잠근 구획의 데이터 조회를 막는 곳이 한 군데 있다(쿠폰 옵션은 잠기면 아예 조회하지 않는다).
   성능상 합리적이나, '잠금은 입력만 막는다'는 §2 의 문장과 표면적으로 어긋나 보인다 — 문서로만 구분돼 있다.
