# SSOT 파이프라인 — 작업 순서와 인수인계 규약

이 문서는 **무엇을 만들지**가 아니라 **어떤 순서로 만들지**를 정한다.

## 왜 이 문서가 생겼나

2026-07 작업에서 산출물이 세 번 비어 있었다. 원인은 능력이 아니라 순서였다.

| 실제로 벌어진 일 | 어긴 규칙 |
|---|---|
| 화면 계약이 아직 움직이는 중에 Figma 를 생성했다 | 상류가 굳기 전에 하류를 만들었다 |
| 명세를 **명세의 인용**을 근거로 갱신했다 | 정본이 아니라 사본을 읽었다 |
| Storybook 을 건너뛰고 Admin → Figma 로 갔다 | 파이프라인의 한 단을 통째로 생략했다 |
| `packages/ui/pages/**` 가 DS 를 하나도 import 하지 않는 채로 남았다 | 상류가 플레이스홀더인데 하류를 정확히 반영했다 |
| 어댑터에 검사가 없어 런타임 결함 6건이 전부 빠져나갔다 | 검사 없는 단을 통과시켰다 |

**결론: 단을 건너뛰지 않고, 상류가 굳은 뒤에 하류를 만들고, 각 단에 검사를 둔다.**

---

## 1. 단(Stage) 순서

```
① 감사        →  ② 토큰·계약   →  ③ DS 구현     →  ④ Storybook
                                                        ↓
⑨ 커밋·PR   ←  ⑧ 검증        ←  ⑦ 문서화     ←  ⑤ Admin 적용
                                                   ↓
                                                ⑥ Figma
```

각 단은 **바로 앞 단의 인수인계 보고서를 읽고 시작한다.**

### ① 감사 (읽기 전용)
현재 상태를 숫자로 확정한다. 코드를 만들지 않는다.
- 산출: 대조표 · 누락 목록 · 사이즈 분포 · 우선순위
- **완료 조건**: 모든 숫자에 근거 파일 경로가 있다. 추측 0. 확인 못 한 것은 "확인 불가" 로 표시.
- **반박 패스 필수** — 1·2차와 **다른 방법**으로 자기 결과를 반증 시도한다. 같은 도구로 세 번 세면 같은 것을 세 번 놓친다.

### ② 토큰 · 계약
`tokens/tokens.json` 과 `contracts/*.contract.json` 이 SSOT 의 뿌리다. 여기가 바뀌면 하류가 전부 재생성된다.
- **하류 작업 중에 여기를 바꾸지 않는다.** 바꿔야 하면 하류를 멈추고 재생성 후 재개한다.
- `pnpm codegen` → `codegen:check` · `validate:contracts` · `drift:check` 통과가 이 단의 문.

### ③ DS 구현 (`packages/ui`)
계약이 선언한 것을 React 로 구현한다.
- 계약에 없는 prop 을 만들지 않는다. Figma 표현만 필요하면 `figmaVariant`/`figmaToggle`/`figmaText` 같은 **Figma 전용 플래그**를 쓴다 — React 타입을 오염시키지 않는다.
- `contract-test` 의 양방향 parity 가 이 단의 문.

### ④ Storybook
**여기가 SSOT 의 표시면이다. 건너뛰지 않는다.**
- 컴포넌트마다 Controls · Docs · Variant · States · Playground · A11y · Dark · Responsive.
- 스토리가 기본형 하나뿐이면 `존재` 가 아니라 **`부분 구현`** 이다.
- `packages/ui/pages/**` 는 **DS 컴포넌트 조합으로만** 그린다. 자체 마크업은 플레이스홀더이며 완료가 아니다.

### ⑤ Admin 적용
화면은 DS 를 소비한다.
- 화면이 UI 를 자체 제작했다면 **그건 DS 의 결손 신호다.** 즉시 ①에 기록하고 승격 대상으로 올린다.
- 화면끼리 직접 import 하지 않는다(축1 page-coupling blocker). 도메인을 이어야 하면 `apps/admin/src/wiring.ts` 의 등록기 패턴을 쓴다.

### ⑥ Figma
계약 · 토큰 · Storybook 이 **굳은 뒤에** 생성한다.
- 규칙은 `docs/figma/specs/plugin-build-rules.md` 가 정본이다.
- 실행 순서 ① 토큰 Variables → ② Component Set → ③ 문서 생성. **폰트 로드 상태와 자기 검사 기록은 메시지를 넘어가지 않는다** — 각 실행이 스스로 갖춰야 한다.
- 자기 검사 실패 0 이 아니면 완료가 아니다.

### ⑦ 문서화 (`specs/`)
**소스를 읽고 쓴다. 명세의 인용을 근거로 명세를 쓰지 않는다.**
- 해소된 결함은 **지우지 않는다.** "해소(사유)" + 근거 경로로 남긴다. 지우면 그 판단이 있었다는 사실이 사라진다.
- 확인 못 한 수치는 코드에 박지 말고 명세에 "미확인" 으로 남긴다.

### ⑧ 검증
`pnpm verify:all` · `pnpm e2e` · VRT.
- **"테스트가 통과했다" 와 "실제로 동작한다" 는 다른 사실이다.** 목킹만 초록인 상태를 완료로 보고하지 않는다.
- 화면을 실제로 열어 눈으로 본다. 이메일은 HTML 로 렌더해서 본다. Figma 는 플러그인을 돌려서 본다.

### ⑨ 커밋 · PR
직접 push 금지. 의미 단위로 쪼개 커밋 → Issue → PR → 리뷰 → 머지.

## 2. 인수인계 보고서 규약

**각 단이 끝나면 다음 단이 읽을 보고서를 남긴다.** 형식은 아래 여섯 항목 고정.

```markdown
## [단 이름] 인수인계 — YYYY-MM-DD

### 1. 무엇이 바뀌었나
변경 파일과 그 이유. 경로 포함.

### 2. 다음 단이 반드시 알아야 할 것
하류의 판단을 바꾸는 사실만. "참고" 수준은 적지 않는다.

### 3. 숫자
[대상 / 처리 / 남음]. 근거 경로 포함. 어림 금지.

### 4. 내가 확인한 것 · 확인 못 한 것
셋으로 나눈다: 실제로 돌려서 확인 / 테스트로만 확인 / 미확인.
**미확인 칸을 비우지 않는다.** 비어 있으면 검증을 안 한 것이다.

### 5. 판단하고 넘긴 것
지시와 다르게 한 것, 그 이유, 되돌리는 방법.

### 6. 다음 단이 밟을 지뢰
이번에 밟은 함정. 다음 사람이 같은 데 빠지지 않도록.
```

### 규약의 근거

- **2항이 없으면 하류가 상류의 전제를 모른 채 만든다.** 이번에 명세 담당이 소스가 아니라 명세를 읽고 갱신해 죽은 근거를 옮겨 적었다.
- **4항의 미확인 칸이 이 문서에서 가장 중요하다.** "44/44 · 실패 0" 이 목킹 기준이었는데 실제로는 27건이 실패했다. 미확인을 미확인이라고 적었기 때문에 그다음에 실제 실행으로 원인을 잡을 수 있었다.
- **6항이 없으면 같은 함정에 반복해서 빠진다.** `repeat` 이 표본을 공유해 18곳을 망친 것, 폰트 스타일 이름이 `SemiBold`(붙임) 인 것, 미로드 폰트에 변수를 바인딩하면 노드 생성이 실패하는 것 — 전부 문서가 없어 두 번 이상 밟았다.

## 3. 결함 처리 순서

결함을 발견하면 **문서 → 검사 → 수정** 순으로 한다. 이 순서를 뒤집으면 같은 결함이 다시 들어온다.

1. `docs/` 또는 `specs/` 에 **증상과 원인을 먼저 적는다.**
2. 그 증상을 재현하는 **검사를 추가한다.** 검사가 실패하는 것을 확인한다.
3. **고친다.** 검사가 통과하는 것을 확인한다.
4. 수정을 되돌려 검사가 다시 실패하는지 본다 — **검사가 헛돌지 않는다는 증명.**

4번을 실제로 해서 105건 실패를 재현한 사례가 있다. 그게 없으면 "검사도 통과, 화면도 초록, 그런데 실제로는 비어 있음" 이 된다.

## 4. 동시 작업 규칙

여러 단이 병렬로 돌 때:

- **파일 단위로 소유권을 나눈다.** 같은 파일을 두 곳이 쓰지 않는다.
- **`git checkout -- <파일>` 금지.** 커밋 안 된 남의 작업이 사라진다. 실제로 168줄을 날린 적이 있다.
- 남의 영역에서 결함을 발견하면 **고치지 말고 기록해서 전달한다.**
- 상류(② 토큰·계약)가 재생성되면 **하류는 잠시 멈춘다.** 중간 상태에서 게이트가 빨간 것은 정상이며, 그 사실을 보고서 4항에 적는다.
