# 작업 사이클과 체크리스트 원장

`ssot-pipeline.md` 가 **어떤 순서로** 만들지를 정한다면, 이 문서는 **누락을 어떻게 검출할지**를 정한다.

## 왜 이 문서가 생겼나

기존 9단 파이프라인에는 순서는 있었으나 **누락 검출 장치가 없었다.** 2026-07-20 감사에서 드러난 것:

| 실제로 벌어진 일 | 없었던 장치 |
|---|---|
| 계약 `a11y.role` 과 구현이 4건 어긋났는데 아무도 몰랐다 | a11y role 대조 게이트 |
| `packages/ui/pages/_data/pages.ts` 가 정본에서 표류해 없는 화면 5개를 광고했다 | 수동 복사본 동기화 검사 |
| Storybook 8축 중 4축이 0/91 인데 "스토리 91개 있음" 으로 읽혔다 | 축별 커버리지 원장 |
| 감사가 결손 90건을 찾았는데 산출물이 **코드 주석으로만** 남았다 (`CrudListShell.tsx:110`) | 감사 결과의 문서화 의무 |

그리고 **게이트가 있던 곳에서는 결함이 나오지 않았다.** 감사가 anatomy 별칭 8건·계약외 prop 6건을 결함으로 올렸으나 반박 패스에서 **둘 다 가짜**로 판명됐다 — 전자는 `spec/__tests__/contracts.test.ts:128` 이 44개 전수를 이미 검사 중이었고, 후자는 계약 description 이 명시적으로 허용한 설계였다. 게이트가 없던 a11y role 축에서만 진짜 결함 4건이 나왔다.

**결론: 진행률을 서술이 아니라 셀 카운트로 만든다. 빈 셀이 곧 미완료다.**

## 1. 사이클

한 사이클 = `착수 → 실행 → 게이트 → 자체 검증 → 오너 검수 요청`.

사이클은 **단(stage)보다 작다.** 한 단이 여러 사이클로 쪼개질 수 있다. 사이클마다 오너에게 넘길 검수 항목을 뽑는다 — 오너가 볼 것이 없으면 그 사이클은 자체 종료한다.

### 사이클 종료 조건

네 가지를 모두 만족해야 종료다. 하나라도 못 채우면 **완료로 선언하지 않는다.**

1. 원장의 담당 셀이 전부 채워졌다 (빈 셀 0)
2. 해당 게이트가 초록이다 — 명령과 출력을 보고서에 적는다
3. 인수인계 보고서 6항을 남겼고 **4항 미확인 칸이 비어 있지 않다**
4. 오너 검수가 필요한 항목을 분리해 제시했다 (없으면 "없음" 을 명시)

### 오너만 할 수 있는 일

에이전트가 **원리적으로 확인 불가능한 것**은 사이클 보고에서 반드시 분리한다.

- **Figma 플러그인 실행** — 변형 조합 생성, Variable 바인딩률, light/dark 2모드, 폰트 로드. 자기 검사 기록은 `figma.root` 플러그인 데이터에만 남아 리포에 나올 수 없다.
- 실제 브라우저에서의 시각 판정 (VRT 기준 이미지 승인 포함)
- 외부 계정·플랜에 걸린 것 (무료 Figma 플랜은 컬렉션당 1모드 — `addMode` 가 throw 한다)
- 제품 판단 (어느 쪽이 정본인지 계약이 자기모순일 때)

## 2. 체크리스트 원장

원장은 **행 × 열의 격자**다. 서술형 진척 보고를 금지하고 셀로만 말한다.

### 원장 A — 컴포넌트 (44행 × 13열)

열: 계약 · 구현 · argTypes · 스토리 · **Docs(autodocs)** · **MDX** · **Playground** · Variant · States · Dark · **RTL** · **Responsive** · **A11y파라미터** · Figma

굵은 열이 2026-07-20 기준 결손 축이다. 나머지는 이미 채워져 있다.

### 원장 B — 페이지 (132행 × 5열)

열: Admin 라우트 · Storybook Page · **DS 조립 여부** · 플레이스홀더 아님 · 인라인 HTML 없음

2026-07-20 기준 3행만 Page 가 있고 **DS 조립은 0행이다.**

### 원장 C — 결함 (건별)

열: 증상 · 원인 · **문서화** · **검사 추가** · 검사 실패 확인 · 수정 · 검사 통과 확인 · **되돌려 재실패 확인**

마지막 열이 핵심이다. `ssot-pipeline.md` §3 의 4번 — 검사가 헛돌지 않는다는 증명.

### 원장 D — 분류표 커버리지 (286행)

`packages/ui/generated/taxonomy.ts` 의 23모듈 286항목이 행이다. `component: null` 이 곧 빈 셀이다.

2026-07-20 기준 **41 / 286 = 14.3%**.

**이 원장이 진척의 정본이다.** 계약 44개끼리의 1:1 동기화율(44/45 = 97.8%)과 혼동하면 안 된다 — 분모가 다르고 재는 것이 다르다.

| 지표 | 분모 | 재는 것 |
|---|---|---|
| 44/45 = 97.8% | 계약에 등재된 것 | 등재된 것끼리 어긋나지 않는가 |
| **41/286 = 14.3%** | 분류표 전체 | **만들어야 할 것 중 얼마나 만들었는가** |

**진척 보고에는 후자를 쓴다.** 2026-07-20 사이클이 전자를 진척으로 보고해 실제 14.3% 를 97.8% 로 **7배 부풀렸다.** 오너가 카탈로그의 "미구현" 표시를 보고 지적할 때까지 드러나지 않았다.

### 원장 운용 규칙

- **셀은 근거 경로와 함께 채운다.** 근거 없는 O 는 빈 셀과 같다.
- 셀을 채운 뒤 **다른 방법으로 한 번 더 센다.** 두 결과가 다르면 원인을 찾을 때까지 그 셀은 미완이다.
- **"해당 없음" 은 빈 셀이 아니다.** 사유를 적어 채운다 (예: TableReorder 는 훅 API 라 계약 비대상).

## 3. 게이트

게이트 없는 단은 통과시키지 않는다. 검사가 없다면 **검사를 만드는 것이 그 사이클의 일**이다.

기존 축은 `tools/contract-test/src/axes/` 에 있다 — `figma.ts` · `react.ts` · `storybook.ts` · `token.ts`. 새 검사는 신규 도구가 아니라 **이 축에 추가**한다.

| 검사 | 자리 | 상태 |
|---|---|---|
| anatomy style 별칭 해소 | `figma-plugin/src/spec/__tests__/contracts.test.ts:128` | **이미 있음** — 44 전수, 63 tests passed |
| a11y role 대조 | `axes/react.ts` | **없음 → 신설 대상** |
| 수동 복사본 동기화 | 신규 스크립트 | **없음 → 신설 대상** |

**신설 전에 기존 축을 먼저 읽어라.** anatomy 게이트를 "없다" 고 판단해 새로 만들 뻔했다 — 이미 있었고 통과 중이었다. `tools/contract-test/src/axes/`(figma·react·storybook·token) 와 `tools/figma-plugin/src/spec/__tests__/` 를 확인한 뒤에 없을 때만 만든다.

`pages.ts` 동기화 검사는 기존 축에 자리가 없다. eslint `banApps` 로 import 가 구조적으로 금지돼 있어 **CI 가 원리적으로 잡지 못한다** — 값 대조 스크립트가 따로 필요하다.

### a11y role 축의 성격 — 일괄 처리 금지

`a11y.role` 은 어떤 게이트도 강제하지 않고 소비자는 `generate-docs.ts:129` 문서 표 하나뿐이다. 스키마상 자유 문자열이라 산문이 들어가기도 한다(`StatusBadge.contract.json:52` 는 role 필드에 한 문단을 담는다). 즉 이 축의 불일치는 **런타임 접근성 결함이라기보다 계약/문서 정합성 결함**이다.

고치는 방향은 **건별 판단**이다. FormField 는 계약이 자기모순(`role:"group"` vs `aria["no-container-role"]`)이라 고칠 곳이 구현이 아니라 계약이다. **일괄로 `role="group"` 을 다는 것은 FormField 에서 명백히 틀린 처방이다.**

## 4. 반박 패스

각 사이클의 조사 결과는 **다른 방법으로 반증을 시도한 뒤에만** 확정한다. 기본 자세는 "이 결과는 가짜다" 이고, 반증에 실패했을 때만 진짜로 인정한다.

2026-07-20 감사에서 반박 패스가 실제로 잡아낸 것:

- `generated/*.figma.json` 의 `properties` 를 배열이 아닌 객체로 취급 → **43/44 가짜 불일치**
- `samples` 가 repeat 노드가 아니라 하위 text 노드에 있음을 오인 → **8건 가짜 위반**
- `variantTokens` 3단 중첩(`prop → value → key`)을 2단으로 순회 → Toast/ToggleSwitch **가짜 오탐**
- 1차 "84 distinct 토큰 참조" → `variantTokens` 누락이 원인, 2차 **103 으로 정정**

그리고 반박 패스가 **감사 결함 18건 중 14건을 가짜로 걷어냈다** — anatomy 별칭 8건, 계약외 prop 6건. 진짜는 a11y role 4건뿐이었다.

**오판 패턴은 매번 같았다: 소비자 코드를 읽지 않고 계약 JSON 만 보고 규칙을 추측했다.**

- anatomy 별칭 — `figma-plugin/src/spec/tokens.ts:23-32` 의 `tokenKeyCandidates` 가 접미형(`surfaceDanger`)과 **접두형(`neutralSurface`)** 후보를 둘 다 생성한다. 이 함수를 읽었으면 즉시 걸러졌다.
- 계약외 prop — 계약 description 본문이 "[imperative props — 계약 밖 컴포넌트 경계]" 로 명시 허용하고 있었다. 스키마만 보고 본문을 안 읽었다.

**규칙: 계약이 무엇을 뜻하는지 판정하기 전에 그 계약을 소비하는 코드를 먼저 읽는다.** JSON 은 규칙을 담지 않는다. 규칙은 소비자에 있다.

**같은 도구로 세 번 세면 같은 것을 세 번 놓친다.** 방법을 바꾸지 않은 재확인은 반박 패스가 아니다.

## 5. 인수인계

`ssot-pipeline.md` §2 의 6항 보고서를 그대로 쓴다. 이 문서가 더하는 것은 **착수 조건**이다.

**다음 단은 직전 보고서를 읽었다는 확인 없이 착수하지 않는다.** 보고서가 없으면 그 단은 시작할 수 없고, 앞 단으로 돌려보낸다.

병렬 작업 시:

- 파일 단위로 소유권을 나눈다. 같은 파일을 두 곳이 쓰지 않는다.
- **읽기 전용 감사가 도는 동안 그 영역에 쓰지 않는다.** 감사는 기준 숫자를 뽑는 단인데 밑에서 파일이 바뀌면 움직이는 과녁이 된다.
- 남의 영역 결함은 고치지 말고 기록해 전달한다.
- `git checkout -- <파일>` 금지.

## 6. 알려진 함정

작업 전에 읽는다. 전부 실제로 밟은 것들이다.

- **`.claude/worktrees/` 아래 `packages/ui/.storybook` 사본이 13벌 있다.** 전역 grep 이 이걸 잡아 숫자가 14배로 부풀거나 엉뚱한 사본을 편집하게 된다. 경로 필터 필수.
- **Storybook 설정은 리포 루트가 아니라 `packages/ui/.storybook/` 이다.**
- **`generated/` 는 산출물이다.** argtypes/types 를 직접 고치면 다음 `pnpm codegen` 에 날아간다. 계약을 고쳐야 한다.
- **`pnpm codegen` 없이 Storybook 을 띄우면 토큰 CSS import 가 실패한다.** 이때 `preview.ts` 의 import 줄을 지우고 싶어지는데, 정답은 codegen 실행이다.
- **`tags: ['autodocs']` 일괄 추가 금지.** Button 115 · Icon 50 · TextField 30 캔버스가 단일 Docs 에 렌더돼 브라우저가 멎는다. `docs: { page }` 나 `excludeStories` 를 병행한다.
- **다크모드를 컴포넌트 CSS 에 추가하면 아키텍처 위반.** `[data-theme='dark']` 토큰 계층이 유일한 스위치다.
- **`vitest css:false` 로 바꾸면 TokenGuard 가 빈 문자열 위에서 전부 통과한다.**
- **`forwardRef(function X(...))` 패턴** — Alert/TextField/PasswordField/SelectField. `export function X(` 만 찾는 파서는 이 넷을 놓치고 "prop 전부 누락" 거짓 경보를 낸다.
- **미사용 토큰 판정은 `packages/ui` 만 보면 오탐.** `color.channel.kakao.*` 와 `print.*` 는 `apps/admin` 에서만 쓰인다. 또 `primitive.*` 는 semantic 이 alias 로 소비하므로 alias 그래프를 먼저 펼쳐야 한다.
- **계약 산문의 "현행 구현은 …" 주장은 고칠 때 함께 늙는다 — 그리고 어떤 게이트도 이걸 잡지 못한다.** `Alert.contract.json` 은 "현행 구현(Alert.tsx:52)이 `<p>` 를 쓰고 있다 — 정정 대상" 을, `DataTable.contract.json` 은 "본문 셀에 withUnit=false 를 하드코딩해(DataTable.tsx:84) 단위가 사라진다" 를 각각 **구현이 이미 고쳐진 뒤에도** 계속 주장하고 있었다(2026-07-20 확인). 그리고 MDX 담당이 그 산문을 읽고 `Alert.mdx`·`DataTable.mdx` 에 그대로 옮겨 적어 **사용자 문서가 존재하지 않는 결함을 광고했다** — `ssot-pipeline.md` §7 의 "명세의 인용을 근거로 명세를 쓰지 않는다" 를 어긴 그 패턴이다.
  - **자동화 불가다. 가짜 게이트를 만들지 마라.** 산문 주장은 자연어라 기계적으로 검증되지 않는다. 그럴싸한 대안인 "인용된 `파일:줄` 이 실제로 존재하는가" 검사도 **이 두 건을 못 잡는다** — 실측 결과 `Alert.tsx:52` 는 `) : (`, `DataTable.tsx:84` 는 `<td` 로 둘 다 멀쩡히 존재하는 줄이었다. 줄 번호는 살아 있고 주장만 죽는다.
  - **처방은 사람의 규율이다: 구현을 고칠 때 그 결함을 서술한 계약 산문을 같은 커밋에서 함께 고친다.** 그리고 계약 산문의 구현 주장을 문서로 옮기기 전에 **반드시 구현을 직접 읽어 확인한다** (work-cycle.md §4 — "계약이 무엇을 뜻하는지 판정하기 전에 그 계약을 소비하는 코드를 먼저 읽는다").
  - **해소된 주장은 지우지 말고 해소로 표기한다** (`ssot-pipeline.md` §7). 선례: `FormField.contract.json` 의 `a11y.role`, `Alert.contract.json` 의 루트 요소 항, `DataTable.contract.json` 의 `columns.unitInBody` 항.
- **`networkAccess: none`** — Figma 플러그인은 리포를 읽을 수 없다. 오퍼레이터가 계약 파일을 수동 업로드해야 하며, 업로드 누락이 최대 운영 리스크다.
- **`codegen:check` 가 초록이어도 파생물이 최신이라는 뜻이 아니다.** `tools/figma-plugin/generated/tds-pages.json` 은 `pnpm codegen` 이 아니라 **`tools/figma-plugin/build.mjs` 가** `nav-config.ts` 에서 만든다. 그래서 codegen 이 세는 199건 밖에 있다. 2026-07-20 에 nav 라벨을 바꾼 뒤 이 파일이 옛 라벨을 그대로 들고 있었는데 **`codegen:check` 는 "생성물 199건 모두 최신 ✔" 을 냈다.**
  - **`nav-config.ts` 를 고쳤으면 `node tools/figma-plugin/build.mjs` 를 함께 돌려라.**
  - 이것은 `packages/ui/pages/_data/pages.ts` 표류와 **같은 계열**이다 — 정본에서 파생됐는데 정본과의 동기화를 아무 게이트도 검사하지 않는다. 파생물을 만들 때는 "누가 이 파일이 낡았음을 알려 주는가" 를 먼저 답해야 한다.

### `apps/admin/**/icons.tsx` 를 지우면 DS 가 줄어든다 (2026-07-20 실측)

**의존 방향이 여기서만 뒤집혀 있다.** `tools/codegen/src/extract-icons.ts:184` 가 `apps/admin/src` 를 걸어 파일명이 `icons.tsx` 인 모든 파일에서 아이콘 기하를 추출해 `Icon` 계약의 name enum 을 만든다.

실측: 계약 enum **59종 중 `assets/icons/ds-icon-geometry.json` 에 저작된 것은 11종뿐이고 48종이 앱 파일에서 온다.**

즉 그 파일들은 앱의 지역 컴포넌트가 아니라 **DS 의 저작 원천**이다. 호출부를 전부 `<Icon name="…" />` 으로 옮긴 뒤 "이제 import 하는 곳이 없으니 죽은 코드" 라고 판단해 지우면 **계약 enum 이 무너지고 DS 가 줄어든다.**

- 파일명이 판정 기준이다 — `components/glyphs.tsx` 는 이름이 안 걸려 추출 대상이 **아니고**, 그래서 거기 있던 중복 정의만 실제로 지울 수 있었다.
- 이 뒤집힘 자체가 부채다. 아이콘 저작을 `assets/` 로 모으는 것이 정방향이지만, 그 전에는 앱 파일을 **원천으로 취급**해야 한다.

### 테스트 실행에 관한 함정 (2026-07-20 실측)

- **루트 `pnpm test` 는 `@tds/ui` 에서 멈춘다.** turbo 가 거기서 중단하면 `@tds/admin#test` 를 **아예 실행하지 않는다.** 루트 결과만 보고 "admin 회귀 없음" 이라고 읽으면 안 된다. `cd apps/admin && pnpm run test` 를 따로 돌려라.
- **테스트 기준선이 안정적이지 않다.** 같은 트리에서 두 번 돌려 실패 9건 → 22건으로 갈렸고, 실패 대부분이 `waitFor` 타임아웃이다. **"실패 N건" 을 회귀 판정에 쓰지 말고 실패한 파일 이름 집합으로 비교하라.**
- **기준선을 재는 동안에는 소스를 쓰지 마라.** vitest 가 편집 중인 파일을 읽어 collect 단계에서 유령 실패를 낸다. 실제로 착수 기준선에서 "3 files failed" 를 봤는데 그중 2건이 collect 실패였고, 편집을 멈추고 다시 돌리자 **119/119** 였다.
- **`toBe(0)` 형 단언은 카운터가 고장 나도 통과한다 — 실측으로 재현했다.** 스켈레톤 헬퍼를 레거시 클래스로 좁히자 같은 파일의 `toBeGreaterThan(0)` 은 실패했는데 `toBe(0)` 2건은 그대로 통과했다. **"없음" 을 단언하는 검사 옆에는 반드시 "있음" 앵커를 둔다.** 현재 스켈레톤 검사 10파일 전부 그 앵커를 갖고 있으니 그 규율을 깨지 마라.
- **일부 실패는 부하 의존이다 — 격리 실행에서 재현되지 않는다.** `RichTextField.test.tsx` 는 격리 실행 **3회 연속 35/35 통과**하지만 전체 스위트에서는 1건 실패한다(2026-07-20 실측, 해당 파일은 미변경 커밋 상태). 실패 지점은 `:209` 의 `waitFor` — **지연 로드(`Suspense`) 청크를 기다리는 곳**이고, 병렬 부하에서 기본 타임아웃 안에 동적 import 가 풀리지 않는다.
  - **실패를 보면 먼저 그 파일만 단독 실행하라.** 남의 변경 탓으로 돌리기 전에 부하 의존인지 가른다. 실제로 한 작업자는 병렬 4개가 도는 중에 "격리에서도 34/35 실패" 를 보고했고 다른 관측은 35/35 였다 — **두 관측 다 사실이었고 변수는 머신 부하였다.**
  - **다만 부하에서 깨지는 검사는 깨진 검사다.** 지연 로드를 기다리는 `waitFor` 에 넉넉한 타임아웃을 주거나 청크를 테스트에서 미리 해소하는 것이 정답이며, "flaky 니까 무시" 는 답이 아니다. 미해결 부채로 남아 있다.
- **`toBeInTheDocument()` 는 `pnpm typecheck` 에서 TS2339 로 죽는다.** jest-dom 매처가 setup 에는 등록돼 있으나 tsconfig `types` 에 배선돼 있지 않다. 리포 관례대로 `not.toBeNull()` 을 쓴다.
- **`usePermissionStore` 는 모듈 싱글턴이고 localStorage 에 저장한다.** 권한 테스트에 `beforeEach`/`afterEach` 복구가 없으면 역할이 테스트 사이로 샌다. 그리고 매트릭스를 손으로 조립하지 마라 — `enforceMatrix` 의 read 의존 불변식을 우회해 실제로 존재할 수 없는 권한 상태가 된다. 기본 역할 `VIEWER_ROLE_ID` 를 쓰면 된다.
- **`CrudListShell` 은 `firstLoading`/`error` 와 무관하게 `toolbar` 를 항상 렌더한다**(`CrudListShell.tsx:132`, 분기 바깥). toolbar 를 검사하는 테스트에 `waitFor` 를 붙이면 위의 타임아웃 불안정에 스스로 걸어 들어간다.
- **병렬 작업 중에는 `git diff` 가 자기 변경보다 훨씬 크게 보인다.** 미커밋 선행 변경이 같은 파일에 이미 들어 있을 수 있다. 자기 몫을 보고할 때는 diff 크기가 아니라 **무엇을 넣었는지**로 말한다.
