## 3. 오늘 뚫은 병목 — typography 결손은 "손이 부족" 이 아니었다

`cssVar()` 이관이 654건에서 멈춰 있던 이유는 인력이 아니라 **codegen 이 표현할 수 없었기**
때문이다. 그리고 파 보니 결손이 셋이었다:

1. **합성 토큰은 자기 이름의 CSS 변수를 갖지 않는다.** `typography.label.md` 는 서브 변수 4개로만
   전개되는데 `tokenVars` 에는 합성 경로가 그대로 올라 있었다 → `cssVar('typography.label.md')`
   는 **타입 검사를 통과하면서 정의된 적 없는 변수를 참조**했다.
2. **죽은 별칭이 실제로 하나 있었다.** `--tds-component-button-typography: var(--tds-typography-label-md)`
   — 오른쪽이 존재하지 않는다. CSS 는 조용히 삼키므로 아무도 몰랐다.
3. **가드에 사각지대가 있었다.** 토큰 가드는 `tokens.css` 를 **정의의 출처**로만 읽고
   그 파일 자신의 참조는 보지 않았다. HEAD 두 커밋(`80c6983`·`1c06e51`)이 세운 가드가
   못 잡은 이유가 이것이다.

**조치**: 생성기가 별칭도 서브 변수로 전개하게 고치고, `tokenVars` 에는 **CSS 에 실제로 나온
이름만** 올리고, `typography('label.md')` 헬퍼를 생성하게 했다. 가드에는 *생성물 자신의 참조*
축을 더했고 **되돌려 재실패를 확인**했다(`tokens.css:232 → --tds-typography-label-md` 로 정확히 실패).

### 부수 소득 — 같은 코드가 리포에 여섯 벌 있었다

서브 변수 4개를 손으로 조립하는 코드가 `CatalogTable`·`foundations/_shared`·pages 스토리 4벌에
**독립적으로 여섯 번** 구현돼 있었다. 그중 넷은 `typography` 라는 **같은 이름을 로컬로 선언**해
서로를 가리고 있었다. 전부 지우고 생성된 헬퍼 한 곳으로 모았다. 예전 시그니처는 인자가
`TokenPath` 라 **타이포그래피가 아닌 토큰도 통과**시켰다 — 이제 `TypographyPath` 로 좁다.

### 이관에서 실제로 위험했던 지점

dry-run 이 아니었으면 **화면이 바뀔 뻔했다.** 376개 묶음 중 **완전한 4속성은 22개뿐**이고
354개가 부분 묶음이었다. 부분을 `typography()` 로 접으면 없던 `fontWeight`·`fontFamily` 가
새로 붙는다. 실제 사례:

```
fontFamily: label.md    fontSize: label.md    lineHeight: label.md
fontWeight: primitive.typography.font-weight.**bold**   ← label.md 는 medium 이다
```

접었으면 이 제목이 **bold → medium 으로 조용히 얇아졌다.** 그래서 완전 묶음만 접고(22건)
부분은 낱개 `cssVar('typography.label.md.font-size')` 로만 바꿨다 — **렌더 결과 무변경**이다.

사후 조건도 한 번 일했다: 1차 실행에서 임포트 정규식의 `\s*$` 가 개행을 먹어 **128개 파일의
빈 줄을 조용히 지웠다.** var 개수 검사만으로는 안 잡혀서, 되돌리고 "줄 수는 접힌 묶음만큼만
줄어야 한다" 는 조건을 추가했다(net −66 = 22×3 로 정확히 일치).

## 4. 표(table) — "다 따로따로 논다" 는 진단은 정확했다

실측: **DS `Table` 을 쓰는 admin 파일은 `CrudTable` 단 하나**다. 나머지는 전부 손으로 쓴
`<table>` 이다.

| 구분 | 화면 수 | 줄 수 |
|---|---|---|
| 공용 스택(`CrudListShell` → `CrudTable` → DS `Table`) | **23** | — |
| 손으로 쓴 표 | **17** | **3,846** |

### 진짜 문제는 마크업 중복이 아니라 **이름이 없다는 것**이었다

`onEdit(item)` 하나가 세 가지 뜻을 겸하고 있었다:

- 17개 화면 → `/:id/edit` **수정 폼** (이름 그대로)
- 2개 화면 → `/:id` **상세** (Review · MessageTemplate)
- 4개 화면 → **모달** (카테고리 3종 · LogoList)

그 결과가 전부 사용자에게 샜다:

1. **캡션이 거짓말했다.** 23개 화면 전부에 *"행을 누르면 해당 항목으로 이동합니다"* 를 읽어 줬다 —
   모달 4개는 이동하지 않는다.
2. **조용한 무반응.** 발송 끝난 캠페인(Email·Newsletter·SMS)은 `onEdit` 에 빈 콜백이 갔다.
   커서는 pointer, 캡션은 이동 약속, 눌러도 아무 일 없음 = 사용자에겐 고장 난 화면.
3. **형제 화면끼리 어긋났다.** 공지·FAQ 는 상세로 가는데 팝업·배너는 행 클릭이 죽어 있다.

### 오늘 세운 것 — `rowTarget`

`apps/admin/src/shared/crud/rowTarget.ts` — 행 목적지를 **판별 유니온**으로 만들었다:

```ts
type RowTarget<T> =
  | { kind: 'detail'; href: (item: T) => string; disabled?: (item: T) => string | false }
  | { kind: 'edit';   href: (item: T) => string; disabled?: … }
  | { kind: 'modal';  open: (item: T) => void;   disabled?: … }
  | { kind: 'none' };
```

- **종류를 밝히지 않고는 만들 수 없다** → 세 가지가 다시 한 이름 뒤에 숨지 못한다.
- **캡션이 여기서 파생된다** → 화면이 하는 일과 낭독이 갈라질 수 **없다**.
- **`disabled` 는 조용한 무반응을 없앤다** → 갈 수 없으면 `onActivate` 를 아예 안 걸어
  커서가 pointer 가 되지 않는다. 빈 콜백으로 표를 속이지 않는다.

기존 23개 화면은 **하나도 안 깨진다** — `rowTarget` 을 안 주면 예전 동작 그대로다(점진 이관 사다리).
오늘은 어긋나 있던 **5개 화면**에 이름을 붙였다: Review·MessageTemplate(상세),
Email·Newsletter·SMS(발송 게이팅). 캡션 파생과 "갈 수 없는 행은 활성화하지 않는다" 를
테스트로 못 박았다(`rowTarget.test.ts`).

## 6. 오너 판단이 필요한 것 (막혀 있다)

1. **팝업·배너 행 클릭** — 상세 라우트가 없고 `/:id/edit` 만 있다. 공지·FAQ 처럼 행 클릭을 열 것인가,

4. **미구현 245건 생성 범위** — 원지침은 *"Admin 에서 안 쓰더라도 무조건 생성"* 이다.

## 7. 미확인 — 비워 두지 않는다

- **브라우저로 아무것도 보지 않았다.** 오늘 바꾼 것(typography 이관 128파일 · calc 130파일 ·
  rowTarget 5화면)은 **타입·테스트·게이트로만** 확인했다. 픽셀 미확인.
- **`pnpm vrt` · `a11y` · `e2e` 미실행.** `verify:full` 이 아니라 `verify:all` 까지다.
- **부분 묶음 354건이 "렌더 무변경" 이라는 것은 문자열 수준의 논증**이다 — 각 호출부를 눈으로
  본 것이 아니라, 낱개 치환이 값에 손대지 않는다는 성질에 기댄다.
- **클러스터 1~4 의 줄 수는 파일 전체 줄 수**다. 그중 몇 줄이 실제로 표 마크업인지는 세지 않았다.
- **`styles.ts` 해체 비용을 계산하지 않았다** — 242개 파일이 이 배럴을 import 한다.
- 페이지 로컬 127건 중 **정독은 12건**이다. "나머지는 도메인 조립체" 는 여전히 **표본 판정**이다.
