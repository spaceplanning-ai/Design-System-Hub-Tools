# 재활용 판정 가이드 — "이걸 새로 만들어야 하나, 있는 걸 써야 하나"

> **작성**: 모듈 추출 · **정식 편입 대상**: `docs/tds/guidelines/` (TDS 문서 소유 — 후속 변경 요청)
> **목적**: 화면을 만들다 UI 조각이 필요해졌을 때 **5분 안에** 판정한다.
> **사례는 전부 실제다** — MP-001 (확정 8화면 전수 조사)에서 내린 판정들이다.

---

## 0. 왜 이 문서가 있는가

MP-001 실측:

```
packages/ui 가 만들어 둔 모듈 ...................... 15종
관리자 앱이 그중 실제로 import 한 컴포넌트 .......... 0개
그래서 페이지 안에 다시 태어난 사본 ................ 18개
```

**"이미 있는 걸 또 만드는 것"이 디자인 시스템을 죽이는 가장 흔한 방법이다.** 그리고 그것은 악의로 일어나지 않는다. 위 18건은 전부 *"@tds/ui 는 아직 비어 있다"* 는 **한때 참이었던 주석**을 근거로 태어났다. 모듈이 승격된 뒤에도 아무도 그 주석을 지우지 않았다.

> **교훈 1: "아직 없다"를 주석에 쓰지 마라.** 그 문장은 반드시 낡고, 사본이 사는 근거가 되어 영구화된다.
> 정 써야 한다면 `// TODO(G5): Button 승격 시 교체 — 추적 TASK-xxx` 처럼 **만료 조건**을 함께 써라.

## 1. 5분 판정 절차

```
                    UI 조각이 필요하다
                            │
            ┌───────────────▼───────────────┐
      [1분] │ packages/ui/src/index.ts 를 열어 본다  │   ← 반드시 첫 단계
            │ (또는 pnpm reuse:check --name X)      │
            └───────────────┬───────────────┘
                            │
           ┌────────────────┴────────────────┐
        있다                               없다
           │                                 │
    ┌──────▼──────┐                   ┌──────▼──────┐
    │ prop 이     │              [2분]│ 이 화면 말고 │
    │ 전부 있나?  │                   │ 다른 화면도  │
    └──┬───────┬──┘                   │ 쓰나?        │
       │       │                      └──┬────────┬──┘
      예      아니오                    2개+      1개
       │       │                         │        │
   ┌───▼───┐ ┌─▼──────────┐        ┌─────▼────┐ ┌─▼──────┐
   │ 재활용 │ │ 재활용     │        │ 도메인을 │ │ 유지    │
   │       │ │ (EXTEND 후)│        │ 아나?    │ │ 페이지에│
   │ 지금  │ │ 계약부터   │        └──┬────┬──┘ │ 둔다    │
   │ import│ │ 요청 발행   │          아니오  예  └────────┘
   └───────┘ └────────────┘           │      │
                                  ┌───▼──┐ ┌─▼──────┐
                                  │ 승격 │ │ 유지    │
                                  └──────┘ └────────┘
```

**절대 하지 말 것: 1단계를 건너뛰고 새로 만들기.** 18건이 그렇게 태어났다.

## 2. 판정별 기준과 실제 사례

### 재활용 — `packages/ui` 에 이미 있다

**조치: 페이지가 그것을 쓰게 고친다. 새로 만들지 마라.**

| 실제 사례 | 무슨 일이 있었나 |
|---|---|
| `dashboard/components/TabBar.tsx` | `Tabs` 계약이 **TabBar.tsx 를 출처로 명시**하고 있었다. 승격이 끝났는데 원본이 안 지워졌다. shipped 는 화살표키 로빙까지 얹은 **상위집합**이다 |
| `dashboard/components/VisitorChart.tsx` | `LineAreaChart` 와 기하 상수·경로 함수가 **바이트 단위로 동일**하다. 문자 그대로의 복사본 |
| `dashboard/components/Card.tsx#CountBadge` | `Badge` 의 `if (count <= 0) return null` 과 **로직이 같다** |
| `MembersTable`·`AdminsTable` 의 `SelectAllCheckbox` | **같은 앱의 `shared/ui/TriStateCheckbox`** 를 손으로 다시 짰다. 게다가 두 사본은 `aria-checked="mixed"` 를 **빠뜨렸다** — 중복이 단순 낭비가 아니라 **a11y 퇴행**이 된 사례 |

> **교훈 2: 중복은 복사본이 원본보다 나빠지는 방식으로 썩는다.** 두 벌이 되는 순간 한 벌은 반드시 뒤처진다.

#### ⚠ 재활용의 함정 — "있다"고 "지금 쓸 수 있다"는 아니다

MP-001 의 18건 중 **7건만 즉시 교체 가능**했고 **11건은 계약 EXTEND 가 선행 조건**이었다. 실측된 파괴 사례:

| 순진하게 바꿨다면 | 벌어졌을 일 |
|---|---|
| `Button` → shipped | shipped 는 `type="button"` **하드코딩**. `type` prop 이 없다 → **폼 5개가 제출을 멈춘다** |
| `Button` → shipped | 기본 `variant` 가 `secondary` → `primary`. `variant` 없이 쓴 **앱 전역 버튼이 조용히 파랗게** 바뀐다 |
| `ListCard`/`TodoCard` → shipped | 클릭 콜백이 `MouseEvent` 를 안 넘긴다 → `preventDefault()` 불가 → SPA 내비가 **전체 새로고침**이 된다 |
| `TextField` → shipped | `required` 일 때 라벨에 `*` 를 주입 → `getByLabelText('이메일')` **E2E 셀렉터가 깨진다** |

> **교훈 3: 재활용 판정과 재활용 실행은 다른 순간이다.**
> `packages/ui` 에 있다 = 판정은 '재활용'이다. 하지만 **prop 이 부족하면 계약(계약 엔지니어) → 구현(컴포넌트 엔지니어) → 페이지(프론트 리팩터) 순서를 반드시 지켜야 한다.**
> 순서를 뒤집고 "일단 바꿔 보는" 것은 **동작을 바꾸는 것**이고, E2E 가 그것을 잡는다.

**즉시 교체 가능한지 확인하는 법 — prop diff 3줄:**
```
1. 로컬 컴포넌트의 prop 목록을 적는다
2. contracts/<Name>.contract.json 의 props 를 적는다
3. 로컬에만 있는 prop = EXTEND 목록. 비어 있으면 오늘 바꿔도 된다
```

### 승격 — 2개 이상 화면이 쓰고, 도메인을 모른다

**두 조건 모두**를 만족해야 한다.

| 실제 사례 | 소비 페이지 | 판정 근거 |
|---|---|---|
| `useToast` 트리오 | **6** | 앱 전역 쓰기-결과 피드백 규약. 회원·권한·등급을 모른다 |
| `ConfirmDialog` | **4** | "무엇을 삭제하는지" 모른다 — `intent` 와 문구만 받는다 |
| `SearchField` | **3** | Members·LoginHistory·Admins 툴바에 **3벌이 손으로 복사**돼 있었다 |
| `FilterGroup` | **4** | `filterHeadingStyle` 승격(프론트 구현)은 **스타일만** 올렸고 **마크업은 5벌 그대로** 남았다 |

> **교훈 4: 스타일만 승격하면 마크업이 남는다.** `filterHeadingStyle`·`filterListStyle`·`filterItemStyle` 3종은 올라갔는데, 그것을 감싸는 `<li><button aria-pressed><span>{label}</span><span>{count}</span></button></li>` 는 **다섯 벌 그대로**다. 승격은 **변경의 축 전체**를 옮겨야 끝난다.

#### 승격의 순서 함정 — 먼저 재활용부터

`TriStateCheckbox` 는 소비자가 1페이지라 규칙상 '유지'였다. 그런데 `MembersTable`·`AdminsTable` 이 **같은 것을 손으로 다시 짜고 있었다.**

- 틀린 순서: 소비자 1개니까 유지 → 두 테이블은 계속 사본을 쓴다 → 중복 영구화
- **맞는 순서**: 두 테이블이 **기존 것을 재활용** → 소비자 3페이지 → **그때 승격이 정당해진다**

> **교훈 5: "소비자 수"를 셀 때 사본을 빼먹지 마라.** 손으로 베낀 사본도 소비자다. 사본을 원본으로 되돌린 뒤 다시 세라.

#### 이미 일반화된 것이 있으면 그것을 올려라

`FilterGroup<T>` 는 **이미 `LoginHistoryFilters.tsx:74-122` 에 일반화되어 존재한다.** 그 파일의 주석은 *"소비자가 하나다"* 라며 로컬 유지를 정당화하는데, 그 주석은 낡았다 — 동일 마크업이 다른 4페이지에 살아 있다. **새로 설계하지 말고 그것을 올려라.**

### 유지 — 소비자가 1개이거나, 도메인을 안다

**'유지'는 패배가 아니다. 판정표에 '유지'가 있는 이유다.**
소비자가 1개인데 "나중에 쓸지도 모르니" 올리는 것은 **부채다** — 추상화 비용만 치르고 이득이 없다.

| 실제 사례 | 유지 근거 |
|---|---|
| `MembersTable`·`AdminsTable` | shipped `DataTable` 은 **데이터 전용**이다. 셀이 `string \| number` 이고 `<tr>`/`<td>` 가 prop 을 하나도 안 받는다. 행 선택·tri-state 헤더·ReactNode 셀·액션 메뉴는 **표현 자체가 불가능**하다. 소비자도 각 1개 |
| `LoginHistoryTable` | 가장 DataTable 에 가깝다. 그래도 **유지**했다 — 셀 렌더러 + 행 톤 + 조건부 행 내비를 넣으려면 **파괴적 DataTable v2** 가 필요하고, 그 수혜자는 **1페이지뿐**이다. 이게 과잉 추상화의 정의다 |
| `RowIcon` | `'order' \| 'tag' \| 'question' \| 'contract'` → 아이콘 매핑. **도메인 어휘다.** `icon` 은 이미 `ReactNode` 슬롯이라 그대로 주입하면 된다 |
| `ActionMenu` | 소비자 1개(회원) |
| 내보내기 버튼 | 소비자는 2개지만 `Button` + 이미 승격된 `DownloadIcon` + 라벨 삼항일 뿐이다. **모듈이 아니라 3줄이다** |
| `Toast` (단독) | 페이지 import **0건** — `ToastProvider` 가 렌더한다. 승격 단위는 트리오지 `Toast` 단독이 아니다 |

> **교훈 6: "닮았다"는 합치는 근거가 아니다.** 합치는 근거는 **같은 이유로 변할 것**이어야 한다 (형태가 아니라 **변경 축**이 같아야 한다).
> `AdminGroupPanel.noticeStyle`(하단 공지, `border-top`)과 `RolePanel.actionsStyle`(상단 액션바, `border-bottom`)은 토큰 모양이 같아서 중복 검출기에 걸렸지만 **다른 것**이다. 합치면 한쪽이 바뀔 때마다 다른 쪽이 깨진다.

## 3. 계약을 쓸 때 — "계약은 소망이 아니라 현실의 기술이다"

승격이 새 계약을 요구하면 `states` 와 `events[].blockedWhen` 을 **정직하게** 채워라.

**전례 (선행 변경 요청)**: `DataTable`·`LineAreaChart` 계약이 `states: ["default","loading"]` 을 선언했는데 **loading 으로 진입할 prop 이 없었다.** 그 결과:
- 커버리지 축2 가 "테스트할 수 없는 칸"을 요구하며 **blocker** 로 남았다
- 판정은 prop 추가가 아니라 **states 에서 loading 제거**였다 — 구현에 존재한 적 없는 상태였기 때문이다
- prop 을 추가했다면 **아무도 호출하지 않는 죽은 표면**이 생겼을 것이다

> **교훈 7: 도달할 수 없는 상태를 계약에 쓰지 마라.** 그것은 게이트를 빨간불로 만들고, 그걸 끄려고 빈 테스트를 쓰게 만든다(초록불 위조).
> **prop 으로 진입할 수 없는 state 는 state 가 아니다.**

같은 함정이 MP-001 에서 또 나왔다: `StatsCard` 계약의 `action.hiddenWhen: ["loading"]` 은 **loading 중 액션을 언마운트**한다. 그런데 대시보드에서 그 액션은 **기간 토글 자신**이고 loading 은 `isFetching` 이다 → **토글을 누를 때마다 토글이 사라진다.** 계약이 현실을 잘못 기술한 것이다. → `disabledWhen` 으로 고쳐야 한다 (`SegmentedControl` 은 이미 `disabled` 를 지원한다).

## 4. 경계 — 판정과 집행은 다른 사람이 한다 (P1 단일 소유권)

| 경로 | 소유 | 당신이 못 쓴다면 |
|---|---|---|
| `contracts/**` | **계약 엔지니어** | 변경 요청 발행 |
| `packages/ui/src/**` | **컴포넌트 엔지니어** | 변경 요청 발행 |
| `apps/*/src/**` | **프론트 구현/리팩터** | 변경 요청 발행 |
| `docs/tds/guidelines/**` | **TDS 문서** | 변경 요청 발행 |

`pnpm boundary:check --agent <역할>` 가 **exit 1 로 차단**한다. 몰래 넘지 마라 — 경계 담당이 잡는다.

> **교훈 8: 판정을 내린 사람이 고치는 사람일 필요는 없다.** 오히려 그게 정상이다.
> 클린코드 점검도 같은 원칙이다: *"클린코드 점검은 측정만 한다 — 수정은 소유자의 일이다."*
