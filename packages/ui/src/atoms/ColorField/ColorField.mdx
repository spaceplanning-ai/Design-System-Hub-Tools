# ColorField

<code>atom</code> · <code>Inputs</code> · <code>taxonomyItem: color-picker</code> · <code>v1.0.0</code> · <code>status: beta</code>

색 스와치 + hex 텍스트 — **같은 값을 보는 두 개의 창.** 출처:
`apps/admin/src/pages/marketing/message-templates/email/controls/ColorField.tsx`
(이메일 빌더 STYLE/INSPECT 패널의 배경·캔버스·테두리·글자 색 등 12곳).

### 왜 hex 텍스트도 같이 두나

`<input type="color">` 만 두면 값을 눈으로 확인할 수 없고, 브랜드 색을 코드로 받아 적는 실무 흐름
(디자이너가 hex 를 준다)이 막힌다. 반대로 텍스트만 두면 색을 고를 수 없다.
**어느 쪽을 바꿔도 같은 `onChange` 로 나간다.**

|---|---|---|---|
| `value` | `string` | — (필수) | 현재 색 — `#RGB` · `#RRGGBB` · `#RRGGBBAA` 를 받는다. 제어 값이며 **유효한 hex 로 확정된 `onChange` 로만** 바뀐다. |

| `id` | `string` | `''` | hex 텍스트 입력의 DOM id. 비우면 id 속성을 렌더하지 않는다. |
| `disabled` | `boolean` | `false` | 비활성 — 스와치와 텍스트를 **함께** 잠그고 흐리게 표시한다. `onChange` 발화 없음. |

### 이벤트 — `onChange` · 입력 도중에는 올리지 않는다

payload `string` · **`blockedWhen: ["disabled"]`**.

계약 `description`: `'#6B4'` 까지 친 순간을 그대로 올리면 캔버스가 엉뚱한 색으로 한 번 깜빡인다.
**유효한 hex 가 됐을 때만** `onChange` 를 발화하고 그 전까지는 지역 초안(draft)으로 들고 있는다.
유효하지 않은 채로 포커스를 떠나면 마지막 유효값으로 되돌린다 — **반쯤 친 값이 화면에 남지 않는다.**

## 접근성 계약

- **role**: `group`
- **keyboard**: `Tab` · `Enter` · `Space`
- **focusVisible**: `true`
- **contrastMin**: 4.5

계약 `a11y.aria`:

- `label`: hex 텍스트 입력이 `label` prop 을 `aria-label` 로 갖고, 스와치는 `'<label> swatch'` 를 갖는다 —
  **한 값을 두 컨트롤이 편집하므로 이름이 갈려야 어느 쪽에 있는지 알 수 있다.**
- `locking`: `disabled` 이면 두 `<input>` 모두 disabled 로 잠근다 — `onChange` 발화가 **네이티브로** 차단된다.

### `swatchFill` 은 **Figma 표현 전용 표본**이다 — React 는 이 색을 칠하지 않는다

브라우저에서 스와치는 `<input type="color">` 이고 **현재 값을 네이티브로 칠한다.** 그래서
`ColorField.css` 의 스와치 규칙은 `background: transparent` 이고, 화면에 보이는 색은 언제나
`value` 다 — `swatchFill` 이 아니다.

Figma 레이어에는 그 네이티브 페인터가 없다. 계약이 `fill` 을 지시하지 않으면 프레임이 Figma
기본값(흰색)으로 태어나고, 실제로 그렇게 나왔다(2026-07-20 오너 실행 — 스와치가 빈 흰 사각형).
`value` 는 런타임 값이고 Figma 컴포넌트 셋은 정적이라 **바인딩할 방법이 없으므로**, 어떤 토큰을
걸든 표본이다. 남는 질문은 '어느 표본이 옳은가' 뿐이고 답은 이 컴포넌트의 정의가 준다 —
**같은 값을 보는 두 개의 창.** 다른 한 창(`Hex Value` 텍스트)이 이미 정적 표본을 들고 있고
그 표본은 `primitive.color.blue.600` 의 값이므로, 스와치도 같은 색이어야 두 창이 같은 값을 본다.

`color.chart.series-1` 이 light 에서 정확히 그 `primitive.color.blue.600` 으로 풀리고,
동시에 **UI 크롬의 의미가 붙지 않은 유일한 색 계열**이다 — 그 계열의 존재 이유가 '임의의 값을
대신하는 색' 이며 그것이 스와치가 하는 일이다. `color.action.primary.default` 도 light 값은
같지만 계약 18건이 체크박스 채움·토글 트랙·탭 인디케이터에 쓰고 있어, 스와치가 그 색을 쓰면
라이브러리 안에서 '체크된 체크박스' 와 같은 그림이 되어 **표본이 아니라 상태로 오독된다.**

디자이너는 Figma 에서 이 채움을 시안의 실제 색으로 덮어써서 쓴다.
