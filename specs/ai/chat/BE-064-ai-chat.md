# BE-064. AI 에이전트 새 채팅 백엔드 기능 명세

## 1. 개요

| 항목 | 내용 |
|---|---|
| 대상 화면 | FS-064 AI 에이전트 새 채팅 (`/ai/chat`) |
| 범위 | 대화 목록·상세 조회, 질문 1건 처리(파싱 → 조회 → 답변), 대화 삭제 |
| 범위 밖 | 언어 모델 추론 자체(§7.1) · 멘션 대상 도메인의 원본 CRUD(BE-003 · BE-026 · BE-041 이 소유) |
| 전제 | 인증은 세션 쿠키 기반. 모든 경로는 `/api` 프리픽스. 응답 본문은 `application/json; charset=utf-8`. 시각은 ISO 8601(오프셋 포함) |
| 프론트 어댑터 | `apps/admin/src/pages/ai/data-source.ts` |
| 도메인 타입 | `apps/admin/src/pages/ai/_shared/conversations.ts` · `_shared/answer.ts` · `_shared/execute.ts` |

> **현재 상태**: 서버 구현은 없다. 어댑터가 픽스처와 결정적 파서로 답한다. 이 문서는 **연동 시 지켜야 할 계약**을 적는다.

## 2. 공통 에러 봉투

모든 4xx / 5xx 응답은 아래 봉투를 따른다. 예외 없다. 각 엔드포인트(§4)는 **에러코드 목록만** 나열하며 자체 응답 형식을 정의하지 않는다.

```json
{
  "error": {
    "code": "CONVERSATION_NOT_FOUND",
    "message": "대화를 찾을 수 없습니다.",
    "fields": null,
    "traceId": "01J8X4K2M9P7Q3R5S6T8V0W2Y4"
  }
}
```

| 필드 | 타입 | 필수 | 설명 |
|---|---|---|---|
| `error.code` | string | O | 기계 판정용 에러코드(SCREAMING_SNAKE_CASE). 프론트 분기의 유일한 근거 |
| `error.message` | string | O | 사용자에게 그대로 보여줄 수 있는 문구. **내부 정보(스택·쿼리·경로·프롬프트) 노출 금지** |
| `error.fields` | array \| null | X | 400 / 422 검증 실패 시에만. 질문 길이 초과 등 |
| `error.traceId` | string | O | 로그 상관관계 ID |

### 2.1 공통 에러코드

| HTTP | 에러코드 | 발생 조건 | 프론트 처리 |
|---|---|---|---|
| 400 | `VALIDATION_FAILED` | 질문이 비었거나 길이 상한 초과 | 입력 옆 인라인 오류 |
| 401 | `UNAUTHENTICATED` | 세션 없음·만료 | `/login?returnUrl=` 로 이동 |
| 403 | `FORBIDDEN` | `/ai/chat` read 권한 없음 | `ForbiddenScreen` |
| 404 | `CONVERSATION_NOT_FOUND` | 대화가 없거나 타인 소유 | FS-064-EL-022 삭제 안내 |
| 409 | `CONFLICT` | 같은 대화에 동시 append 충돌 | 재조회 후 재시도 |
| 422 | `UNPROCESSABLE` | 멘션 도메인에 대한 조회 권한 없음 | 답변 kind='guidance' 로 안내 |
| 429 | `RATE_LIMITED` | 질문 과다 | 실패 답변 + `Retry-After` 안내 |
| 500 | `INTERNAL_ERROR` | 서버 오류 | FS-064-EL-016 실패 답변 |
| 504 | `REQUEST_TIMEOUT` | 모델·조회 지연 | FS-064-EL-016 실패 답변 |

## 3. 인증 · 권한 모델

### 3.1 역할

| 역할 | 설명 | 비고 |
|---|---|---|
| `admin` | 전체 도메인 조회 가능 | |
| `operator` | 자신이 `read` 권한을 가진 도메인만 조회 가능 | §3.5 |

### 3.2 403 vs 404 은닉 정책

| 리소스 | 권한 부족 시 응답 | 근거 |
|---|---|---|
| 대화 컬렉션 `/api/ai/conversations` | **403** | 컬렉션 존재는 비밀이 아니다 |
| 대화 1건 `/api/ai/conversations/{id}` | **404** | 타인의 대화는 **존재 자체를 숨긴다** — 403 이면 id 를 훑어 남의 대화 존재를 알 수 있다 |

### 3.3 CSRF

쓰기(POST · DELETE)는 `X-CSRF-Token` 헤더를 요구한다. 누락·불일치면 403 `CSRF_TOKEN_INVALID`.

### 3.4 타임아웃 상한

| 계열 | 서버 처리 상한 | 초과 시 | 프론트 상한(권고) |
|---|---|---|---|
| 조회(대화 목록·상세) | 3초 | 504 | 10초 |
| 질문 처리(규칙 기반) | 5초 | 504 | 15초 |
| 질문 처리(모델 연동 후) | 60초 | 504 | 90초 · 스트리밍 권고 (§7.1) |

### 3.5 멘션 도메인의 권한 승계 — **이 화면의 핵심 보안 판정**

에이전트는 회원·상품·문의를 대신 읽어 준다. 그래서 **이 화면이 권한 우회 통로가 될 수 있다**: `/users/members` read 가 꺼진 운영자가 `@회원목록` 으로 회원 명단을 받아 가면, 권한 모델이 무의미해진다.

| 규칙 | 내용 |
|---|---|
| 승계 | 질문이 도메인 D 를 멘션하면 서버는 **D 의 목록 조회와 동일한 권한 검사**를 수행한다 (회원 → BE-003 §3, 문의 → BE-026 §3, 상품 → BE-041 §3) |
| 거부 시 | 422 `UNPROCESSABLE`. 프론트는 kind='guidance' 로 '조회 권한이 없습니다' 를 답한다 — **행을 0건으로 내려보내지 않는다**(0건과 권한 없음은 다른 사실이다) |
| 감사 | 에이전트를 통한 조회도 원본 도메인의 조회 로그에 남긴다. 경로만 다를 뿐 같은 열람이다 |

**요청 — 본문**

| 이름 | 타입 | 필수 | 제약 | 설명 |
|---|---|---|---|---|
| `question` | string | O | 1–1000자 | 질문 원문. 서버가 파싱한다 |
| `mode` | string | X | `rules` 고정 | 응답 모드. 현재 `rules` 외 값은 400 (§7.1) |

**응답 201** — `AskResult`

```json
{
  "conversation": { "id": "CV-0001", "title": "이번달 구매 VIP 뽑아줘", "updatedAt": "…" },
  "answer": {
    "kind": "result",
    "notices": [
      { "code": "period-unsupported",
        "message": "'누적 구매액' 에는 기간 정보가 없어 '이번달' 로 거를 수 없습니다. …",
        "suggestion": "@회원목록 이번달 가입한 회원 보여줘" }
    ],
    "outcome": {
      "domainId": "members", "domainLabel": "회원 목록",
      "columns": ["닉네임", "계정", "등급", "가입일", "누적 구매액"],
      "rows": [{ "id": "M-00056", "cells": ["윤도현", "…", "VIP", "2026-06-19", "150,000원"], "href": "/users/members/M-00056" }],
      "total": 9, "conditionSummary": "등급 VIP · 누적 구매액 있음", "listUrl": "/users/members?tier=vip"
    },
    "followUps": [{ "label": "이번달 가입한 회원 보여줘", "query": "@회원목록 이번달 가입한 회원 보여줘" }]
  }
}
```

**에러 목록**

| HTTP | 에러코드 | 조건 |
|---|---|---|
| 400 | `VALIDATION_FAILED` | 질문 공백·1000자 초과·알 수 없는 `mode` |
| 401 | `UNAUTHENTICATED` | 세션 없음 |
| 403 | `CSRF_TOKEN_INVALID` | CSRF 토큰 누락·불일치 |
| 404 | `CONVERSATION_NOT_FOUND` | 이어 물으려는 대화가 없다 (§3.2) |
| 422 | `UNPROCESSABLE` | 멘션 도메인 조회 권한 없음 (§3.5) |
| 429 | `RATE_LIMITED` | 분당 한도 초과 |
| 504 | `REQUEST_TIMEOUT` | 조회·모델 지연 (§3.4) |

> **파싱 실패는 에러가 아니다.** 멘션 없음·모르는 도메인·지원하지 않는 의도는 **200/201 로 `kind:"guidance"` 답변**을 돌려준다. 4xx 로 내리면 프론트가 '실패' 로 그리고, 사용자는 시스템이 고장 났다고 읽는다 — 실제로는 시스템이 정확히 알아듣고 거절한 것이다.

## 5. 예외 매트릭스

| 엔드포인트 | 400 검증 | 401 인증 | 403 vs 404 | 404 대상없음 | 409 충돌 | 422 상태위반 | 429 과부하 | 500 오류 | 타임아웃 |
|---|---|---|---|---|---|---|---|---|---|
| BE-064-EP-01 목록 조회 | `limit` 범위 밖·손상된 `cursor` → `VALIDATION_FAILED` | 401 `UNAUTHENTICATED` | **403** — 컬렉션 존재는 비밀이 아니다 (§3.2) | N/A — 컬렉션은 항상 존재한다. 0건은 빈 배열이다 | N/A — 읽기 전용이라 동시 수정 충돌이 없다 | N/A — 상태 전이가 없는 조회다 | 429 + `Retry-After`. 관리자당 분당 120회 | 500 `INTERNAL_ERROR` + `traceId`. 좌측 기록만 실패하고 본문 대화는 유지된다 | 서버 3초 → 504. 프론트 권고 10초 |
| BE-064-EP-02 대화 조회 | 형식이 아닌 `id` → `VALIDATION_FAILED` | 401 `UNAUTHENTICATED` | **404** — 타인 대화는 존재를 숨긴다. 403 이면 id 훑기로 존재를 알 수 있다 (§3.2) | 404 `CONVERSATION_NOT_FOUND` → FS-064-EL-022 삭제 안내 | N/A — 읽기 전용이다 | N/A — 상태 전이가 없다 | 429 + `Retry-After`. 분당 120회 | 500 + `traceId`. '없음'과 구분해 실패로 그린다(삭제 안내를 띄우지 않는다) | 서버 3초 → 504 |
| BE-064-EP-03 질문 처리 | 공백·1000자 초과·알 수 없는 `mode` → `VALIDATION_FAILED` + `fields` | 401 `UNAUTHENTICATED` | **404** — 대상 대화 은닉은 EP-02 와 같은 규칙 (§3.2) | 404 `CONVERSATION_NOT_FOUND` — 이어 물으려던 대화가 사라졌다 | 409 `CONFLICT` — 같은 대화에 동시 append. 프론트는 재조회 후 재시도 | 422 — 멘션 도메인 조회 권한 없음 (§3.5). **파싱 실패는 여기가 아니라 200 답변이다** | 429 + `Retry-After`. 규칙 기반 분당 20회 · 모델 연동 후 6회 | 500 + `traceId`. 실패 답변 1건으로 남고 **대화는 유지된다** | 서버 5초(모델 60초) → 504. 프론트 권고 15초(90초) |
| BE-064-EP-04 삭제 | `ids` 빈 배열·100건 초과 → `VALIDATION_FAILED` | 401 `UNAUTHENTICATED` | **404 를 쓰지 않는다** — 멱등 삭제라 없는 id 도 204 다. 권한 없음만 403 | N/A — 멱등이라 대상 없음이 오류가 아니다 | N/A — 삭제는 마지막 상태로 수렴한다 | N/A — 삭제에 상태 선행 조건이 없다 | 429 + `Retry-After`. 분당 60회 | 500 + `traceId`. 토스트로 실패를 알리고 목록은 유지한다 | 서버 3초 → 504 |

| 요구사항 | 대상 함수 | 내용 |
|---|---|---|
| 파싱 위치 이동 | `askAgent` | 현재는 클라이언트가 파싱한다(`buildAnswer`). 서버 연동 시 질문 원문을 올리고 `AgentAnswer` 를 받는다 — **파서는 서버로 옮겨간다**. 프론트에 두 벌이 남으면 판정이 갈린다 |
| 유니온 유지 | `askAgent` | 서버 응답을 `AgentAnswer` 로 좁히지 못하면 실패 답변으로 떨어뜨린다. 알 수 없는 `kind` 를 결과인 것처럼 그리지 않는다 |
| 없음 vs 실패 구분 | `fetchConversation` | 404 는 `null`(삭제 안내), 그 외 오류는 throw(실패 표시). 둘을 섞으면 서버 장애가 '대화가 지워졌다' 로 보인다 |
| 취소 전파 | 전부 | `signal` 을 그대로 fetch 에 넘긴다. 새 질문이 이전 요청을 abort 하는 경로가 화면에 이미 있다 |

## 7. 핵심 판정 (근거를 남긴다)

### 7.1 언어 모델은 `askAgent` 안쪽만 바꾼다 — 계약은 바뀌지 않는다

모델이 붙어도 아래 셋은 유지되어야 한다. 하나라도 무너지면 이 화면은 신뢰할 수 없어진다.

| # | 계약 | 무너지면 |
|---|---|---|
| 1 | 답변은 `AgentAnswer` 유니온을 벗어나지 않는다 | 자유 문장이 데이터 답변인 것처럼 그려진다 |
| 2 | `kind:"result"` 의 `rows`·`total` 은 **조회 결과**다. 모델이 만들어 낸 값이 들어갈 수 없다 | 관리자가 존재하지 않는 회원을 보게 된다 |
| 3 | 알아듣지 못한 요청은 `kind:"guidance"` 로 남는다 | 모델이 아무 답이나 지어내고, 거절 경로가 사라진다 |

모델 응답은 **조회 계획(어느 도메인·어떤 조건)** 으로만 쓰고, 값은 언제나 저장소에서 읽는다.

### 7.2 `AgentAnswer` 는 서버가 소유하는 형태다

프론트가 종류를 판단해 그리므로 서버가 종류를 정해 내려야 한다. 종류: `result` · `guidance` · `not-wired` · `error`.
`guidance` 는 **정상 응답**이다(§4 EP-03 각주).

### 7.3 대화 메시지 상한 200건 — 잘라내지 않고 거절한다

상한을 넘으면 EP-03 이 422 로 거절하고 새 대화를 권한다. 오래된 메시지를 조용히 버리면 관리자가 앞서 받은 답을 다시 볼 수 없게 된다.

### 7.7 후속 이관 (프론트 구현 · UI 기획 · 아키텍처)

| # | 내용 | 이관 대상 |
|---|---|---|
| 1 | 모델 연동 시 스트리밍(SSE) 여부 — 60초 응답을 통째로 기다리면 화면이 멈춘 것처럼 보인다 | 아키텍처 |
| 2 | 파서를 서버로 옮길 때 프론트 사본 제거 (§6.1) | 프론트 구현 |
| 3 | 에이전트 경유 조회의 감사 로그 형식 (§3.5) | 백엔드 구현 · FS-061 회원 활동 로그 |
