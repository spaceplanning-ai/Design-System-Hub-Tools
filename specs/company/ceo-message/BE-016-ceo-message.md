# BE-016. CEO 인사말 백엔드 기능 명세

## 1. 개요

| 항목 | 내용 |
|---|---|
| 대상 화면 | FS-016 CEO 인사말 (`/company/ceo-message` — 단일 문서 편집 폼. 목록·상세·`:id` 라우트 없음) |
| 범위 | CEO 인사말 문서 1건의 조회 · 저장(전체 치환) |
| 범위 밖 | **사진 파일 업로드** — 프론트에 심이 없다(§4 BE-016-EP-03 · §7.6 판정). **고객 인사말 페이지 렌더** — §7.1 XSS·개행 판정만 다룬다. **문서 생성/삭제** — 회사당 1건이 항상 존재한다고 가정한다(§7.3) |
| 전제 | 인증은 세션 쿠키 기반. 모든 경로는 `/api` 프리픽스. 응답 본문은 `application/json; charset=utf-8` |
| 프론트 어댑터 | `apps/admin/src/pages/company/ceo-message/data-source.ts` (`ceoMessageStore` = `createDocumentStore<CeoMessage>('ceo-message', CEO_MESSAGE_SEED)`) |
| 도메인 타입 | `apps/admin/src/pages/company/ceo-message/types.ts` |
| 검증 정본 | `apps/admin/src/pages/company/ceo-message/validation.ts` (`ceoMessageSchema`) — 프론트 규칙. **서버는 이를 신뢰하지 않고 재검증한다** |
| 형제 명세 | BE-015 (회사 정보) — **같은 단일 문서형 골격**을 쓴다. §7.2(스킴)·§7.4(동시성)·§7.5(403/404)·§7.6(업로드) 판정이 **같은 근거로 같은 결론**에 이른다 |

> **에러 봉투·권한 모델 상속**: BE-003 §2(공통 에러 봉투 · §2.1 공통 에러코드)·§3(인증 · 권한 모델 · 403 vs 404 은닉 · CSRF · 타임아웃)을 **그대로 상속한다**. 아래는 CEO 인사말 도메인 고유 차이만 기술한다.

## 2. 공통 (상속)

- **에러 봉투**: BE-003 §2. `error.code` · `error.message` · `error.fields` · `error.traceId`. 공통 에러코드 동일.
- **권한**: `admin` = 조회 + 저장. `operator` = **조회만**, 저장(`PUT`) → 403. 기업 관리 읽기 권한이 없는 인증 관리자 → **403**(컬렉션 성격 — §7.5).
- **CSRF**: 쓰기(`PUT`)에 `X-CSRF-Token` 헤더. 누락·불일치 → 403 `CSRF_TOKEN_INVALID`.
- **타임아웃**: 조회·쓰기 5초 → 504 `REQUEST_TIMEOUT`.
- **프론트 역할 분기 없음** (FS-016 §4.1): 화면이 `can(resource,'update')` 를 보지 않아 저장 버튼이 권한과 무관하게 렌더된다(FS-016 §7 #3). **권한 강제는 전적으로 서버 책임**이며, 프론트는 403 을 다른 실패와 같은 문구로 표시한다(FS-016-EL-004).

## 3. 데이터 계약 (types.ts 대조)

| 타입 | 필드 | 비고 |
|---|---|---|
| `CeoMessage` | `title`(string) · `body`(string) · `photoUrl`(string) | **3필드 전부 `readonly` · 전부 non-nullable string.** 옵셔널 필드가 없다 — 사진 '없음' 은 `null` 이 아니라 **빈 문자열**로 표현된다 |

**제약 상수** (`types.ts`) — 프론트 `maxLength` 와 zod 스키마가 공유한다. 서버는 같은 값으로 재검증한다.

| 상수 | 값 | 적용 필드 | 프론트 검증 |
|---|---|---|---|
| `TITLE_MAX_LENGTH` | 120 | `title` | `requiredText('제목', 120)` — trim 후 1–120자 |
| `BODY_MAX_LENGTH` | 5000 | `body` | trim 후 비어있지 않을 것 + **`value.length <= 5000`(trim 없이 원문 길이)** — §7.3 |
| (상한 없음) | `z.string()` | `photoUrl` | **형식 무제약**(§7.2) |

> **`CeoMessage` 에 `id`·`updatedAt`·`version` 이 없다** — BE-015 §3 과 같다. 이것이 §7.4(동시성) 판정의 출발점이다: 프론트가 보낼 수 있는 낙관적 동시성 토큰이 타입에 존재하지 않는다.
>
> **`body` 는 서식 없는 plain text 다.** 줄바꿈(`\n`)만으로 문단을 나눈다 — 픽스처 seed(`data-source.ts:10-13`)가 `\n\n` 로 문단을 구분한다. HTML 도 마크다운도 아니다(§7.1).

| 레이트리밋 | 분당 60회 |

**쿼리**: 없다. 식별자도 없다 — 세션의 회사가 곧 대상이다.

**응답 200** — `CeoMessage`(§3 의 3필드 전부). **누락 필드를 두지 않는다**: 프론트가 `reset(data)` 로 폼을 통째로 덮으므로(`CeoMessagePage.tsx:57`) 필드가 빠지면 그 입력이 `undefined` 가 된다 — 특히 `body`·`photoUrl` 은 **제어 컴포넌트**(`TextareaField`·`ImageUploadField`)에 `value` 로 들어가므로 `undefined` 는 React 경고와 uncontrolled 전환을 일으킨다. 값이 없으면 **빈 문자열**을 보낸다.

**`body` 의 개행**: 저장된 그대로(`\n` 포함) 돌려준다 — 관리자 화면이 `<textarea>` 에 원문을 넣기 때문이다. 서버가 `\r\n` 을 `\n` 으로 정규화한다(§7.3).

**에러**: 401 `UNAUTHENTICATED` · 403 `FORBIDDEN` · 404 `CEO_MESSAGE_NOT_FOUND`(§7.3) · 429 · 500 · 504. 프론트는 이 전부를 `error !== null` 하나로 뭉개 '내용을 불러오지 못했습니다.' + '다시 시도'(FS-016-EL-012)로 그린다.

### BE-016-EP-02 · CEO 인사말 저장
| 항목 | 내용 |
|---|---|
| 근거 (FS) | FS-016-EL-011, EL-015, EL-016 |
| 근거 (심) | `data-source.ts:19` — `// TODO(backend): PUT /api/company/ceo-message` |
| 메서드·경로 | `PUT /api/company/ceo-message` |
| 권한 | **`admin` 만.** `operator` → 403 |
| 멱등성 | **멱등**(PUT 전체 치환) — 같은 바디 재요청은 같은 최종 상태. 그러나 이 멱등성이 중복 제출을 무해하게 만들지는 **않는다**(§7.4) |
| 페이징 | N/A |
| 레이트리밋 | 분당 20회 |
| 요청 본문 상한 | **≥ 64KB** — `body` 5000자가 UTF-8 로 최대 약 15KB(한글 3바이트)이고 여유를 둔다. 상한 초과는 413 이 아니라 **400 `VALIDATION_FAILED`**(`fields[].name='body'`)로 준다 — 프론트에 413 표면이 없다 |

**바디**: `CeoMessage` **전체**(§3). 부분 갱신(PATCH) 이 아니다 — 프론트가 폼 값 3개를 통째로 보낸다(`save.mutate({ input: values, ... })`).

**서버 검증** (프론트 스키마와 같은 규칙 + 프론트에 없는 규칙):

| 필드 | 규칙 | 프론트 대응 |
|---|---|---|
| `title` | trim 후 1–120자 | 동일(`requiredText`) |
| `body` | trim 후 1자 이상 · **5000자 이하(계수 기준 §7.3)** · `\r\n` → `\n` 정규화 · **저장 시 정제(§7.1)** | 프론트는 원문 길이만(`value.length`), 정제 없음 |
| `photoUrl` | 빈 문자열 **또는** `https://` 스킴 + 허용 호스트. **`blob:`·`data:`·`javascript:`·`file:` 은 400 거절**(§7.2) | 프론트는 무제약(`z.string()`) |

**응답 200/204**. 프론트 `save(input, signal): Promise<void>` — **응답 본문을 읽지 않는다.** **이것이 이 화면에서 특히 중요하다**: 서버가 `body` 를 정제(§7.1)하거나 개행을 정규화해도 화면은 제출한 원본으로 `reset` 한다(FS-016-EL-016) — 정제 결과는 뒤이은 EP-01 재조회(`invalidateQueries`)로만 도달한다. 즉 **관리자는 자기가 쓴 태그가 제거된 사실을 즉시 보지 못한다.**

**에러**: 400 `VALIDATION_FAILED`(`error.fields`: `title`·`body`·`photoUrl`) · 401 · 403 `FORBIDDEN` · 403 `CSRF_TOKEN_INVALID` · 404 `CEO_MESSAGE_NOT_FOUND` · 409 `CONFLICT`(§7.4) · 429 · 500 · 504. 프론트는 전부 '저장하지 못했습니다. 잠시 후 다시 시도해 주세요.'(FS-016-EL-004) 한 문구로 그린다 — `error.fields` 를 쓰지 않는다(§7.7 이관 #2).

### BE-016-EP-03 · 사진 이미지 업로드 — **심 없음 (미정)**

**엔드포인트를 정의하지 않는다.** `data-source.ts` 의 `// TODO(backend)` 주석은 **`GET`·`PUT /api/company/ceo-message` 2건뿐**이며 업로드 심이 없다. 계약의 유일한 근거가 없으므로 경로·메서드·응답을 지어내지 않는다.

**현재 동작** (FS-016-EL-008.7): `ImageUploadField` 가 `URL.createObjectURL(file)` 로 만든 **`blob:` URL** 을 `photoUrl` 에 넣고, 그 문자열이 EP-02 바디로 나간다. 파일 바이트는 **어디에도 전송되지 않는다.** 저장은 성공하고 토스트도 뜨지만 그 URL 은 해당 브라우저 세션에서만 유효해, 새로고침·다른 기기·고객 화면에서 **사진이 깨진다.**

**BE-015-EP-03 과 같은 결정 대상이다** — 회사 로고와 CEO 사진은 **같은 업로드 계약을 공유해야 한다**(둘 다 `ImageUploadField` 를 쓰고 둘 다 `blob:` 을 저장한다). 화면마다 다른 업로드 방식을 만들면 `ImageUploadField` 계약이 화면별로 갈린다.

| # | 결정 사항 | 선택지 |
|---|---|---|
| 1 | 업로드 방식 | (a) 공용 `POST /api/uploads` multipart → `{ url }` (b) presigned URL (c) 도메인별 엔드포인트(`/api/company/logo`·`/api/company/ceo-photo` — 중복이라 권장하지 않음) |
| 2 | 업로드 시점 | (a) 파일 선택 즉시 (b) 저장 제출 시 |
| 3 | 고아 파일 회수 | 업로드 후 저장하지 않고 이탈하면 파일이 남는다 — TTL 또는 참조 카운트 |
| 4 | 프론트 계약 | `ImageUploadField` 는 값 콜백(`onChange(url)`)만 노출한다 — **BE-015 와 공유하는 결정** |

**임시 방어 (엔드포인트 결정 전에도 적용한다 — §7.2)**: EP-02 가 `photoUrl` 의 스킴을 화이트리스트로 검사해 `blob:`·`data:` 를 **400 `VALIDATION_FAILED`** 로 거절한다.

## 7. 핵심 판정

### 7.1 인사말 본문·제목의 저장형 XSS — 서버가 저장 시 정제한다 【보안 판정】

CEO 인사말 3필드는 **관리자 입력이며 고객 인사말 페이지에 그대로 노출된다**(FS-016-EL-002 안내문이 그 계약을 명시한다: '저장하면 사용자 화면의 인사말 페이지에 반영됩니다'). 관리자 화면은 `<input>`/`<textarea>` 로만 그려 안전하지만, **고객 렌더가 HTML 로 해석하면 저장형 XSS 가 된다.**

**이 화면은 BE-015 보다 위험이 크다.** 본문이 **5000자 자유 텍스트**이고 문단·강조가 필요한 콘텐츠라, 고객 화면이 개행을 살리려고 `dangerouslySetInnerHTML` + `replace(/\n/g,'<br>')` 같은 구현을 택할 유인이 실재한다 — 그 순간 5000자 전체가 주입 벡터가 된다.

**판정** (BE-009 §7.3 · BE-010 §7.2 와 같은 결): **서버가 저장 시(EP-02) 정제**한다. 현재 `body` 는 **서식 없는 plain text 이므로**(§3) 정제는 **모든 마크업 제거**다 — 태그·`javascript:` 스킴·이벤트 핸들러를 전부 제거하고 텍스트와 개행만 남긴다. `title` 도 동일. 계약은 '저장된 값에 실행 가능한 스크립트가 없다' 는 관측 동작만 정한다.

**개행 규약** (프론트에 명시되지 않아 여기서 확정한다 — FS-016 §7 #8): 저장된 `body` 의 `\n` 은 **의미 있는 문단 구분**이다. 고객 화면은 **`white-space: pre-wrap` 으로 렌더한다** — HTML 변환(`<br>`/`<p>`)을 하지 않는다. 관리자 상세 화면들이 답변·본문을 pre-wrap 으로 그리는 것과 같은 규약이다(FS-010-EL-020 선례).

> **로드맵 연계**: HTML 에디터가 이 필드에 도입되면(FS-016 §7 #8) 이 판정이 **'전체 마크업 제거' → '허용 태그 화이트리스트'** 로 바뀐다. 그때 `body` 의 계약(plain text)이 바뀌므로 **이 절과 §3 을 함께 개정**한다.

### 7.2 `photoUrl` 스킴 화이트리스트 — 서버가 `blob:`/`data:` 를 거절한다 【보안 판정】

**BE-015 §7.2 와 동일한 판정이며 같은 근거다.** 두 문제가 한 필드에 겹쳐 있다.

**(a) 보안**: `photoUrl` 은 고객 화면의 `<img src>` 로 나간다. 프론트 검증이 `z.string()`(`validation.ts:16`) 이라 **어떤 문자열도 통과한다**. `shared/crud/validation.ts:22` 의 `optionalHttpUrl()` 헬퍼가 **이미 있고** `company/logo-list` 는 그것을 쓴다 — 이 화면과 회사 정보만 무방비다.

**(b) 정합**: 업로드 심이 없어(§4 BE-016-EP-03) 저장되는 값이 `blob:` 인 경우가 **정상 경로에서 발생한다.** **⚠ 이것은 프론트의 결함이 아니라 '알려진 빚(known debt)'이며, 서버 계약을 세울 때 그렇게 읽어야 한다** — `ImageUploadField` 에는 URL 을 칠 입력이 없어 사용자 조작으로 도달 가능한 값이 `blob:…` 과 빈 문자열뿐이고, 프론트가 http(s) 를 강제하면 **회사 관리 5개 폼이 제출 불가**가 된다(근거 전문: `shared/crud/validation.ts` 의 `requiredImage` 주석). 그래서 **깨질 것을 아는 채로 통과시키고** 테스트가 그 상태를 못 박는다 — `pages/company/certificates/certificates.test.ts:106-107` · `logo-list/logo-list.test.ts:98-99` · `profile/profile.test.ts:71` 이 전부 `TODO(backend): POST /api/uploads 가 붙으면 이 단언은 뒤집힌다(blob: 거절)` 를 달고 있다. **즉 프론트는 EP-03 이 붙는 순간 조일 준비가 되어 있고, 아래 화이트리스트 계약과 충돌하지 않는다.** (이 화면의 `photoUrl` 은 `ceoMessageSchema` 가 형식을 강제하지 않아 `data:` 도 통과한다 — `ceo-message.test.ts:42-46`.) `ceo-message.test.ts:42-46` 이 `data:image/png;base64,xx` 통과를 **테스트로 고정**하고 있어 프론트만 고쳐서는 회귀가 난다.

**판정**: EP-02 가 `photoUrl` 을 **스킴 화이트리스트로 검사**한다 — 빈 문자열 또는 `https://` + 허용 호스트만 통과, 그 외(`blob:`·`data:`·`javascript:`·`file:`·`http:`)는 **400 `VALIDATION_FAILED`**(`error.fields[].name = 'photoUrl'`). 업로드 엔드포인트가 정해지기 전에도 적용하는 **임시 방어**다: 조용히 저장됐다가 나중에 깨지는 실패를 **저장 시점의 명시적 거절**로 바꾼다. **서버 검사가 정본**이다(§7.7 이관 #1).

### 7.3 본문 길이 계수 기준 · 개행 정규화 · 문서 미생성 【정합 판정】

**(a) 계수 기준**: 프론트는 `value.length <= 5000`(`validation.ts:11-13`) — **UTF-16 code unit** 기준이라 이모지·일부 한자가 2로 세어진다. `TextareaField` 의 `maxLength=5000` 도 같은 기준으로 입력을 **먼저 막으므로**, 스키마의 초과 문구는 붙여넣기 절단으로만 도달한다. **서버가 다른 기준(바이트·code point)을 쓰면 프론트 통과/서버 거절이 경계에서 갈린다.**
**판정**: 서버도 **UTF-16 code unit(= JS `String.length`)** 을 기준으로 5000 을 강제한다 — 프론트가 정본 UI 이므로 그 기준에 맞춘다. 저장 용량 상한은 별도로 요청 본문 상한(EP-02)이 담당한다.

**(b) trim 불일치**: `title` 은 `requiredText` 가 **trim 후** 길이를 보는데(`shared/crud/validation.ts:15`), `body` 는 **trim 없이 원문** 길이를 본다(`validation.ts:11`) — 같은 화면 두 필드의 기준이 다르다. **판정**: 서버는 두 필드 모두 **trim 후 저장**하고, 길이 검증도 **trim 후** 수행한다. 프론트의 `body` 원문 계수는 카운터 표시용으로만 허용한다(사용자가 입력한 만큼 보여주는 것이 자연스럽다).

**(c) 개행 정규화**: 붙여넣기 출처에 따라 `\r\n`(Windows)이 들어온다 — code unit 계수에서 2자를 먹고 저장값이 출처마다 달라진다. **판정**: 서버가 저장 시 `\r\n`·`\r` → `\n` 으로 정규화한다(길이 검증은 정규화 후).

**(d) 문서 미생성(404)**: 회사당 1건이 '항상 존재' 한다고 가정하는 것은 **프론트의 가정**이다(FS-016 §1 — 등록 라우트가 없다). **권장**: 서버가 **빈 문서(3필드 전부 빈 문자열)를 200 으로** 돌려 이 상태를 없앤다 — 프론트에 '아직 만들어지지 않음' 표면이 없으므로 404 를 만들지 않는 쪽이 계약과 맞다. **BE-015 §7.3 과 같은 판정.**

### 7.4 저장 동시성 — 프론트에 토큰이 없다. 마지막 쓰기가 이긴다 【판정 + 이관】

**BE-015 §7.4 와 동일한 구조이며, 이 화면에서 손실이 더 크다.**

**낙관적 잠금을 도입할 수 없다.** `CeoMessage` 타입에 `updatedAt`·`version`·`id` 가 **없고**(§3), `createDocumentStore` 의 `save(input, signal)` 시그니처에 토큰을 실을 자리가 없다. F2 가 `useCrudForm` 에 넣은 `If-Match` 배선(EXC-04)은 **이 화면에 적용되지 않는다** — 이 화면은 `useCrudForm` 을 우회하고 `useSaveDocument` 를 직접 쓴다.

따라서 현재 계약은 **마지막 쓰기 승리**다. **본문이 5000자 자유 텍스트라 덮어쓰기의 실질 손실이 회사 정보(짧은 필드 6개)와 비교되지 않는다** — 두 관리자가 인사말을 동시에 다듬으면 한쪽의 수십 분 작업이 조용히 사라지고, 프론트에 충돌 다이얼로그가 없어 서버가 409 를 줘도 같은 '저장하지 못했습니다' 배너만 뜬다.

**여기에 중복 제출이 겹친다**(FS-016 §7 #2): 이 화면은 `submitLockRef`(동기 잠금)와 **제출 시도 단위 멱등키가 없다.** PUT 이 멱등이라 같은 바디의 중복 요청은 최종 상태를 바꾸지 않지만, **① 429 한도를 소모하고(본문 15KB 요청이 두 번 나간다) ② 두 요청 사이에 다른 관리자의 저장이 끼면 그 값을 두 번 덮으며 ③ 감사 로그에 유령 변경이 두 줄 남는다.**

**이관** (§7.7 #1·#2): 낙관적 동시성을 도입하려면 **타입·어댑터 시그니처·`useSaveDocument` 를 함께 바꿔야 한다** — 그리고 그 수정은 **BE-015 와 공유된다**(같은 `document.ts`). 그 전까지 서버는 **감사 로그(누가·언제·무엇을 + 이전 본문)를 남겨** 덮어쓰기를 사후 복구 가능하게 한다 — **이 화면에서는 감사 로그가 유일한 복구 수단이므로 이전 본문 보관이 필수다.**

### 7.5 403 vs 404 은닉 — CEO 인사말은 은닉 대상이 아니다

BE-003 §3.2 의 두 원칙 적용. CEO 인사말은 **컬렉션 성격**(식별자 없는 단일 문서)이고 **그 존재가 비밀이 아니다** — 인사말은 고객에게 공개되는 콘텐츠다. 따라서 권한 부족은 **403 `FORBIDDEN`** 이며 404 은닉을 적용하지 않는다. **BE-015 §7.5 와 같은 판정.**

라우트 read 권한(`page:/company/ceo-message`)은 프론트 `RequirePermission` 이 403 화면으로 먼저 막지만(`AppShell.tsx:490`), 그것은 **UX 이지 보안 경계가 아니다.** 서버 403 이 정본이다.

### 7.6 사진 업로드 — 엔드포인트를 지어내지 않는다 【판정】

`// TODO(backend)` 주석이 계약의 유일한 근거이고, 업로드 심은 **없다**. 따라서 BE-016-EP-03 은 '심 없음(미정)' 으로 남긴다(§4). **BE-015 §7.6 과 같은 판정이며, 두 화면은 같은 결정을 공유해야 한다** — 회사 로고와 CEO 사진이 서로 다른 업로드 계약을 갖는 것은 `ImageUploadField` 를 화면별로 갈라놓는다.

**연쇄**: 업로드가 정해지기 전까지 §7.2 의 스킴 검사가 **유일한 방어선**이고, 그 결과 '사진 업로드' 는 **저장 시 400 으로 거절되는 기능**이 된다. 이 상태를 운영자에게 숨기지 않는 것이 조용히 깨진 이미지를 배포하는 것보다 낫다.
