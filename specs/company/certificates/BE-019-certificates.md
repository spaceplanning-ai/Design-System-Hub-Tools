# BE-019. 인증서/특허 관리 백엔드 기능 명세

## 1. 개요

| 항목 | 내용 |
|---|---|
| 대상 화면 | FS-019 인증서/특허 관리 (`/company/certificates` · `/new` · `/:id/edit`) |
| 범위 | 인증서/특허 목록 조회, 단건 조회, 등록, 수정, 삭제(단건·일괄) |
| 범위 밖 | 고객 노출 렌더(§7.2 XSS 판정만). 일괄 전용 엔드포인트(§7.1). 구분 필터·정렬·페이징 파라미터(§7.3 — 프론트가 클라이언트에서 처리한다). **이미지 업로드 — 심이 없어 명세할 수 없다(§7.6 · BE-019-EP-06)** |
| 프론트 어댑터 | `apps/admin/src/pages/company/certificates/data-source.ts` |
| 도메인 타입 | `apps/admin/src/pages/company/certificates/types.ts` |
| 검증 정본 | `apps/admin/src/pages/company/certificates/validation.ts` (`certSchema`) |
| 공용 어댑터 | `apps/admin/src/shared/crud/crud.ts` (`createCrudAdapter`) |

> **에러 봉투·권한 모델 상속**: BE-003 §2·§3 을 그대로 상속한다. 아래는 인증서/특허 고유 차이만 기술한다.

## 2. 공통 (상속)

- **에러 봉투**: BE-003 §2. 공통 에러코드 동일. `error.traceId` 는 프론트의 '오류 코드' 표시(FS-019-EL-014)와 대조된다.
- **권한**: `admin` = 전체. `operator` = 조회 계열만(목록·단건), 쓰기(등록·수정·삭제) → 403. 기업 관리 읽기 권한 없는 관리자 → 컬렉션 403 / 개별 404 은닉(BE-003 §3.2). BE-017 §7.5 와 같은 근거로 **공개 정보라도 은닉 규칙을 통일**한다.
- **CSRF**: 쓰기(POST·PUT·DELETE)에 `X-CSRF-Token`.
- **타임아웃**: 조회·쓰기 5초 → 504. **이미지 업로드는 별도 상한이 필요하다(§7.6)**.
- **프론트 역할 분기 없음**(FS-019 §4.1) — 프론트의 쓰기 버튼이 권한과 무관하게 렌더되므로(FS-019 §7 #6) **서버 403 이 유일한 방벽이다**.

## 3. 데이터 계약 (types.ts 대조)

| 타입 | 필드 | 비고 |
|---|---|---|
| `CertItem`(목록 행 = 상세) | `id`(string) · `name`(string) · `issuer`(string) · `issuedOn`(string, 'YYYY-MM-DD') · `kind`(`'certificate' \| 'patent'`) · `imageUrl`(string) | **목록과 상세가 같은 타입이다** — 부분 표현이 없다 |
| `CertInput`(쓰기 입력) | `name` · `issuer` · `issuedOn` · `kind` · `imageUrl` | 폼이 `trim()` 으로 정규화해 보낸다 |
| `CertKind` | `'certificate' \| 'patent'` | 두 값 유니온. 표시명·톤은 프론트가 파생(`certKindLabel`·`certKindTone`) |

**제약 상수**(`types.ts`): `NAME_MAX_LENGTH = 100` · `ISSUER_MAX_LENGTH = 100`. **`issuedOn` 형식**: `/^\d{4}-\d{2}-\d{2}$/`(`validation.ts:7`).

**정렬**: `sortCertificates` 가 **프론트에서** 발급일 내림차순(같은 날짜는 id 역순)으로 정렬한다 — §7.3.

> **`updatedAt`·`version` 필드가 없다** — §7.4. **`imageUrl` 의 값 형식이 계약에 없다** — §7.5·§7.6 의 전제다.

### BE-019-EP-04 · 인증서/특허 수정
| 항목 | 내용 |

### BE-019-EP-06 · 이미지 업로드 — **심 없음 (미정)**

**이 엔드포인트는 존재하지 않는다. 지어내지 않는다.**

`data-source.ts:38` 의 `// TODO(backend)` 주석에 이미지 업로드가 **없다**:
```
// TODO(backend): GET/POST /api/company/certificates · GET/PUT/DELETE /api/company/certificates/:id
```

현재 프론트는 `ImageUploadField` 가 `URL.createObjectURL(file)` 로 만든 **`blob:` URL 을 `imageUrl` 에 그대로 저장**한다(`ImageUploadField.tsx:178-181`). 파일은 어디로도 전송되지 않는다. 그 결과:

1. 저장된 `imageUrl` 은 **그 탭·그 세션에서만** 유효하다.
2. 컴포넌트가 언마운트되며 `URL.revokeObjectURL` 로 그 URL 을 **스스로 무효화한다**(`ImageUploadField.tsx:156-161`) — 저장 후 목록으로 돌아가는 순간 이미 깨져 있다.
3. `certSchema` 의 `requiredImage`(`shared/crud/validation.ts:34-38`)는 **등록 여부만 검사하고 형식을 강제하지 않는다.** 단위 테스트가 이 사실을 계약으로 못 박았다 — `certificates.test.ts:95` 가 `imageUrl: 'blob:abc-123'` 의 `success === true` 를 단언한다. 즉 **`blob:` 통과는 버그가 아니라 현재의 명시적 계약이다** — 업로드 심이 없기 때문에.

**따라서 백엔드 개발자가 정해야 하는 것**(§7.6 에서 판정):
- 업로드 방식(멀티파트 직접 업로드 vs presigned URL 2단계)
- 응답 URL 의 형식·수명·CDN 경로
- 크기·타입 상한의 서버 측 정본(프론트는 `image/*`·5MB 를 클라이언트에서만 막는다 — 우회 가능)
- 고아 파일 회수(등록을 취소하거나 EP-05 로 삭제했을 때)

**이 계약이 정해지기 전까지 EP-03·EP-04 의 `imageUrl` 은 §7.5 의 스킴 화이트리스트만 강제한다.**

## 7. 핵심 판정

### 7.1 일괄 삭제 — 전용 엔드포인트를 만들지 않는다
프론트가 `settleAll`(`Promise.allSettled`)로 단건 DELETE(EP-05)를 항목마다 병렬 호출하고 부분 실패를 건수로 집계한다(FS-019-EL-011.1). BE-010 §7.1 · BE-017 §7.1 과 같은 근거다.

**대가**: 실패 id 를 돌려받지 못해 '실패분만 재시도'가 불가능하고, 재시도가 성공분까지 다시 보내 409 를 유발한다(FS-019 §7 #4). 페이지네이션이 없어 전 행 선택이 가능하므로 항목 수만큼의 동시 요청이 분당 60회 상한을 넘길 수 있다. **인증서가 60건을 넘어서면 벌크 승격을 재검토한다.**

### 7.2 명칭·발급기관 XSS — 서버가 저장 시 정제한다 【보안 판정】
`name`·`issuer` 는 **관리자 입력이며 회사 소개 페이지의 인증 현황 섹션을 통해 고객에게 노출된다**. 관리자 화면은 표 셀에 텍스트로만 렌더하지만(FS-019-EL-006.5/.6), 고객 렌더가 HTML 로 해석하면 **저장형 XSS** 가 된다.

BE-010 §7.2 · BE-017 §7.2 와 동일 판정 — **서버가 저장 시**(EP-03·EP-04) 허용 태그 화이트리스트 밖의 마크업·`javascript:`/`data:` 스킴·이벤트 핸들러 속성을 제거한다. 계약은 '저장된 `name`·`issuer` 에 실행 가능한 스크립트가 없다'는 관측 동작만 정한다. 정제는 100자 상한 검사 **이전**에 수행한다.

### 7.3 구분 필터·정렬·페이징을 계약에 넣지 않는다 (지금은)
프론트 어댑터가 `fetchAll(signal)` 하나뿐이고, **구분 필터마저 클라이언트에서 건다**(`filterCertificates`). 서버가 페이징을 강제하면 프론트가 2페이지 이후를 영영 보지 못하므로 EP-01 은 전량 반환이다.

이것은 프론트의 미비(FS-019 §7 #2·#3)를 계약이 따라간 것이지 옳은 종착지가 아니다. **IA-04/IA-13 이 해소되어 프론트가 `page`/`size`/`kind` 를 URL 과 함께 보내기 시작하면 EP-01 에 offset 페이징(`page` 1-base, `size` 기본 20·상한 100)과 `kind` 필터·`sort=issuedOn,desc` 를 추가한다.** 그때까지 서버는 전량을 내려주되 응답 크기(이미지 URL 이 행마다 붙는다)를 관측한다.

### 7.4 동시 수정 — 지금은 '대상 없음'만 잡고, 낙관적 잠금은 유보한다 【정합 판정】
BE-017 §7.4 와 동일 판정이다. `CertItem` 에 `updatedAt`·`version` 이 없어(§3) 서버가 '이 클라이언트가 본 판본'을 알 수 없고, 두 관리자가 같은 인증서를 동시에 수정하면 나중 요청이 조용히 덮는다(last-write-wins). 편집 빈도가 낮아 지금은 도입하지 않는다.

**다만 이 도메인에는 연혁에 없는 위험이 하나 더 있다**: 동시 수정이 `imageUrl` 을 덮으면 **이전 이미지 자산이 고아가 된다**(참조가 사라진 파일이 저장소에 남는다). 업로드 계약(§7.6)을 정할 때 자산 수명주기를 함께 정해야 하며, **낙관적 잠금은 그때 함께 도입하는 편이 자연스럽다** — 자산 회수는 '누가 마지막에 이겼는가'를 알아야 하기 때문이다.

도입 시 계약: `CertItem` 에 불투명 `version` → EP-02 응답이 `ETag` → EP-04 가 `If-Match` 요구, 불일치 시 **412** → 프론트는 코드 변경 없이 같은 충돌 다이얼로그를 띄운다(`isConflict` 가 409·412 를 함께 본다).

### 7.5 이미지 URL 스킴 화이트리스트 — 서버가 정본이다 【보안 판정】
`imageUrl` 은 고객 화면과 관리자 화면 양쪽에서 **`<img src>` 로 렌더된다**(FS-019-EL-006.4 · `ImageThumb.tsx:52`). 그런데 **프론트는 이 값의 형식을 전혀 검증하지 않는다**:

- `requiredImage`(`shared/crud/validation.ts:34-38`)는 `value.trim() !== ''` 만 본다. 주석이 그 이유를 명시한다 — '업로드 결과(object/data URL 또는 업로드 응답 URL)라 형식은 강제하지 않고 등록 여부만 본다'.
- 단위 테스트가 이를 계약으로 고정했다 — `certificates.test.ts:95` 가 `'blob:abc-123'` 통과를 단언한다.
- `optionalHttpUrl`(같은 파일 22-28행)이 `^https?:\/\/\S+$` 검사를 갖고 있으나 **이 스키마는 그것을 쓰지 않는다.**

**판정**: 서버가 EP-03·EP-04 에서 `imageUrl` 의 스킴을 화이트리스트로 강제한다 — **`https:` 만 허용**(운영 CDN 도메인으로 한정하는 것을 권고). 거절 시 **422 `INVALID_IMAGE_URL`**.

**차단 대상과 그 이유**:
| 스킴 | 왜 막는가 |
|---|---|
| `javascript:` | `<img src>` 에서는 대부분의 최신 브라우저가 실행하지 않지만, 같은 값이 `<a href>`·CSS `url()`·서버 사이드 템플릿으로 옮겨지는 순간 실행 경로가 된다. 값 자체를 저장소에 들이지 않는 것이 유일하게 안정적인 방어다 |
| `data:` | `data:image/svg+xml` 은 **SVG 안의 `<script>` 를 실행할 수 있다**. 이미지처럼 보이지만 이미지가 아니다 |
| `blob:` | 현재 프론트가 실제로 저장하는 값이다(§7.6). 다른 오리진·다른 세션에서 무의미하며 **저장소에 영구 보관될 가치가 없는 쓰레기 데이터**다. 서버가 이것을 받아들이면 깨진 이미지를 DB 에 영구 기록하게 된다 |
| `http:` | 혼합 콘텐츠(mixed content) — HTTPS 고객 페이지에서 차단되거나 경고를 낸다 |
| 상대 경로·프로토콜 상대(`//host/x.png`) | 렌더 컨텍스트에 따라 해석이 달라져 예측 불가능하다 |

**이 판정은 §7.6 이 해소될 때까지 임시 방벽이 아니다 — 영구 계약이다.** 업로드 엔드포인트가 생겨 서버가 URL 을 직접 발급하게 되더라도, 클라이언트가 보낸 `imageUrl` 을 그대로 신뢰해서는 안 된다(요청은 위조된다). **서버는 자신이 발급한 URL 인지 대조하는 편이 가장 안전하다.**

> **프론트 후속(§7.8 이관)**: `requiredImage` 를 `https:` 강제로 좁히고 `certificates.test.ts:95` 의 `blob:` 통과 단언을 뒤집는다. **단 그것은 업로드 심이 생긴 뒤여야 한다** — 지금 좁히면 이미지를 아예 등록할 수 없게 된다(현재 유일한 값 생성 경로가 `blob:` 이다).

### 7.6 이미지 업로드 — 심이 없다. 계약을 지어내지 않고 결정 목록을 남긴다 【미정】
§4 BE-019-EP-06 이 사실관계를 적었다. 여기서는 **무엇이 결정돼야 이 화면이 실제로 동작하는가**를 정한다. **이것은 이 도메인에서 가장 큰 미결이다 — 지금 이 화면은 이미지를 저장할 수 없다.**

| # | 결정 사항 | 선택지 | 판정 근거 |
|---|---|---|---|
| 1 | 업로드 방식 | (a) `POST /api/uploads` 멀티파트 → `{ url }` (b) presigned URL 2단계(`POST /api/uploads/sign` → 스토리지 직접 PUT → `url` 회신) | (b)가 앱 서버를 파일 트래픽에서 뺀다. 다만 `ImageUploadField` 가 2단계를 알아야 해 **DS 계약 변경 폭이 커진다**. 인증서 이미지는 5MB·저빈도라 **(a) 로 시작하고 필요 시 (b) 로 승격**을 권고 |
| 2 | 응답 URL 의 형식 | `https://<cdn>/company/certificates/<uuid>.<ext>` | §7.5 의 화이트리스트와 정합해야 한다 — 서버가 발급한 도메인만 통과시킬 수 있다 |
| 3 | 크기·타입 상한의 정본 | 서버가 정본. 프론트의 `image/*`·5MB(`ImageUploadField.tsx:37-43`)는 **UX 편의이며 우회 가능**하다 | 클라이언트 검증은 보안 경계가 아니다. 서버가 실제 바이트를 검사해야 한다(확장자·Content-Type 이 아니라 **매직 넘버**로) |
| 4 | SVG 허용 여부 | **불허 권고** | SVG 는 스크립트를 담을 수 있다(§7.5 `data:` 항목과 같은 이유). PNG·JPG·GIF 만 — 프론트 안내 문구도 이미 'PNG · JPG · GIF'다(`ImageUploadField.tsx:239`) |
| 5 | 고아 파일 회수 | (a) 업로드 즉시 영구 저장 + 주기적 GC(참조 없는 자산 삭제) (b) 임시 업로드 → EP-03/EP-04 성공 시 커밋 | 등록을 취소하거나(FS-019-EL-020) 이탈 가드로 나가면 업로드된 파일이 참조 없이 남는다. **EP-05 삭제 시 연결 자산 처리도 여기서 함께 정한다**(§7.4 의 덮어쓰기 고아 포함) |
| 6 | 업로드 타임아웃 | 조회·쓰기 5초와 **별도 상한**(예: 30초) | 5MB 업로드는 5초 안에 끝나지 않을 수 있다. BE-003 §3.4 의 내보내기(30초)와 같은 결 |
| 7 | 진행률·취소 | 계약에 포함할지 | 현재 프론트에 진행률·cancel 이 없다(`createObjectURL` 이 동기라 필요가 없었다). 실제 업로드가 생기면 EXC-15 가 요구하는 progress/cancel 경로가 **프론트에도 새로 필요하다** |

**이 결정 전까지의 상태**: 이 화면은 이미지를 **저장할 수 없다**. 등록·수정은 성공하지만 저장된 `imageUrl` 은 무효한 `blob:` 이며, 새로고침하면 목록 썸네일이 전부 placeholder 가 된다(FS-019 §7 #1). **§7.5 의 서버 스킴 검사가 켜지면 `blob:` 이 422 로 거절되어 등록 자체가 실패한다** — 즉 **§7.5 와 §7.6 은 반드시 같은 배치에서 함께 켜야 한다.** 순서를 어기면 화면이 멈춘다.

### 7.7 발급일 — 형식은 프론트, 유효성은 서버
`certSchema` 는 `/^\d{4}-\d{2}-\d{2}$/` 만 본다(`validation.ts:7,14-16`) — **'2023-99-99' 가 통과한다**. `<input type="date">` 가 실사용 경로에서는 유효한 날짜만 만들지만, 폼은 `noValidate` 이고 요청은 위조 가능하므로 **서버가 실제 달력 유효성을 검사한다**(400 `VALIDATION_FAILED`, `error.fields[].name = 'issuedOn'`).

**미래 발급일**: 프론트에 상·하한이 없다(FS-019 §7 #12). **미래 날짜를 허용할지 백엔드 명세 이 정한다** — 인증서는 발급 후 등록하는 것이 자연스러워 미래 발급일은 오타일 가능성이 높으나, 발급 예정일을 미리 넣는 운영이 있을 수 있다. 막기로 하면 400 `VALIDATION_FAILED`, `message` = '발급일은 오늘 이후일 수 없습니다.' 하한은 두지 않는다(오래된 특허가 정당하다).
