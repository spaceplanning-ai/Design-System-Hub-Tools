# BE-018. 오시는 길 백엔드 기능 명세

## 1. 개요

| 항목 | 내용 |
|---|---|
| 대상 화면 | FS-018 오시는 길 (`/company/directions` — 단일 편집 폼 1개. 하위 라우트 없음) |
| 범위 | 오시는 길 문서(회사당 1건) 조회 · 저장(전체 치환) |
| 범위 밖 | **지도 임베드**(§4 EP-03 · §7.5 — 프론트에 심이 없다). 고객 화면 렌더(§7.2 XSS 판정만). 주소 → 좌표 지오코딩(§7.6 — 프론트에 심이 없다) |
| 프론트 어댑터 | `apps/admin/src/pages/company/directions/data-source.ts` (`directionsStore`) |
| 도메인 타입 | `apps/admin/src/pages/company/directions/types.ts` (`Directions`) |
| 검증 정본 | `apps/admin/src/pages/company/directions/validation.ts` (`directionsSchema`) — **클라이언트 검증은 UX 다. 서버가 정본이다**(§7.3) |
| 리소스 형태 | **싱글턴** — id 가 없다. 컬렉션도 페이징도 없다 |

> **에러 봉투·권한 모델 상속**: BE-003 §2·§3 을 그대로 상속한다. 아래는 오시는 길 고유 차이만 기술한다.

## 2. 공통 (상속)

- **에러 봉투**: BE-003 §2. 공통 에러코드 동일(`error.code` · `error.message` · `error.fields` · `error.traceId`).
- **권한**: `admin` = 조회 + 저장. `operator` = 조회만, 저장 → 403. 기업 도메인 읽기 권한 없는 관리자 → **403**(싱글턴은 컬렉션 성격이라 존재를 숨기지 않는다 — BE-003 §3.2 원칙 1, §7.4).
- **CSRF**: 쓰기(PUT)에 `X-CSRF-Token`. 누락·불일치 → 403 `CSRF_TOKEN_INVALID`.
- **타임아웃**: 조회·쓰기 5초 → 504 `REQUEST_TIMEOUT`. **프론트 상한은 없다**(FS-018 §7 #12).
- **프론트 역할 분기 없음**(FS-018 §4.1) — 권한 강제는 전적으로 서버 책임이다. 저장 버튼은 권한과 무관하게 렌더되고 눌린다(FS-018 §7 #8).

## 3. 데이터 계약 (types.ts 대조)

| 타입 | 필드 | 비고 |
|---|---|---|
| `Directions` | `address: string` · `addressDetail: string` · `latitude: string` · `longitude: string` · `transit: string` | **전 필드 필수 존재**(옵셔널 없음). 값이 없으면 `null` 이 아니라 **빈 문자열** |
| — | id · createdAt · updatedAt · version | **없다**. 싱글턴이며 낙관적 잠금 토큰이 타입에 존재하지 않는다(§7.4) |

**좌표가 `number` 가 아니라 `string` 인 이유**(`types.ts:7-10` 주석): 입력 원문을 그대로 보관한다. `'37.500000'` 의 후행 0, `'-0'`, 지수 표기가 파싱으로 소실되지 않게 하기 위함이다. **응답도 문자열로 준다** — 서버가 `number` 로 내리면 `37.5000` → `37.5` 가 되어 관리자가 저장한 표기가 화면에서 바뀐다.

| 상수 (types.ts) | 값 | 강제 지점 |
|---|---|---|
| `ADDRESS_MAX_LENGTH` | 200 | EP-02 `address` |
| `ADDRESS_DETAIL_MAX_LENGTH` | 100 | EP-02 `addressDetail` |
| `TRANSIT_MAX_LENGTH` | 1000 | EP-02 `transit` |
| 위도 범위 | `-90 ~ 90` (절댓값) | EP-02 `latitude` (`validation.ts:34`) |
| 경도 범위 | `-180 ~ 180` (절댓값) | EP-02 `longitude` (`validation.ts:35`) |
| 좌표 형식 | `/^-?\d+(\.\d+)?$/` (앞뒤 공백 제거 후) | EP-02 (`validation.ts:11`) — **지수 표기·`Infinity`·`NaN`·`+` 부호를 허용하지 않는다** |

## 4. 엔드포인트 명세

### BE-018-EP-01 · 오시는 길 조회

| 항목 | 내용 |
|---|---|
| 근거 (FS) | FS-018-EL-004, EL-005, EL-006, EL-007.1, EL-007.2, EL-010, EL-014, EL-016 |
| 메서드·경로 | `GET /api/company/directions` |
| 권한 | `admin`, `operator` |
| 멱등성 | 멱등(GET) |
| 페이징 | **없음 — 싱글턴** |
| 레이트리밋 | 분당 120회 |

**쿼리**: 없음.

**응답 200** — `Directions` (`{ address, addressDetail, latitude, longitude, transit }`). **전 필드 필수**이며 값이 없으면 빈 문자열(`""`)을 준다 — `null`/필드 누락이면 프론트의 `reset(data)`(FS-018-EL-016)가 RHF 를 uncontrolled 로 떨어뜨린다. 좌표는 **문자열**(§3).

**문서가 아직 만들어지지 않았을 때**: 404 가 아니라 **200 + 전 필드 빈 문자열**을 준다. 싱글턴이므로 '없음' 은 '아직 안 채움' 이고, 프론트에는 404 화면이 없다(FS-018 §7 #2 — `DocumentFormShell` 은 `loadFailed` boolean 하나뿐이라 404 를 '다시 시도' 로 오안내한다). 이 판정을 §7.1 에 남긴다.

**에러**: 401 `UNAUTHENTICATED` · 403 `FORBIDDEN` · 429 `RATE_LIMITED` · 500 `INTERNAL_ERROR` · 504 `REQUEST_TIMEOUT`.

### BE-018-EP-03 · 지도 임베드 — **심 없음 (미정)**

`DirectionsPage.tsx:214-215` 에 `// TODO(backend): 지도 임베드 — 좌표를 지도 SDK(예: 카카오/구글 맵) 로 렌더한다.` 주석이 있으나 **`<METHOD> /api/...` 형태의 엔드포인트 심이 아니다.** 어댑터에도 대응 함수가 없다. 화면에 있는 것은 좌표를 텍스트로 보여 주는 `aria-hidden` placeholder(FS-018-EL-009)뿐이다.

**따라서 이 문서는 지도 관련 엔드포인트를 정의하지 않는다.** 지도는 대체로 백엔드 API 가 아니라 **프론트가 로드하는 외부 SDK**(카카오/구글 맵 JS)로 붙으며, 그렇다면 백엔드의 몫은 '좌표를 정확히 저장·반환하는 것'(EP-01/EP-02)에서 끝난다. 서버 몫이 생기는 경우(SDK 키를 서버가 발급·프록시, 지오코딩 대행, 정적 지도 이미지 생성)는 **아직 결정된 바 없다** — §7.5 로 이관한다.

### BE-018-EP-04 · 주소 → 좌표 지오코딩 — **심 없음 (미정)**

관리자는 주소와 좌표를 **각각 손으로** 입력한다(FS-018-EL-005 · EL-007.1 · EL-007.2). 주소에서 좌표를 유도하는 어댑터 함수도, `// TODO(backend)` 주석도 없다. 두 값이 어긋나도 아무것도 막지 않는다(§7.6). 엔드포인트를 지어내지 않는다.

## 7. 핵심 판정

### 7.1 싱글턴 리소스 — id 없는 PUT upsert, 404 를 쓰지 않는다
오시는 길은 회사당 1건이며 `Directions` 에 id 가 없다(§3). 컬렉션(`GET /api/company/directions?page=`)도 생성(`POST`)도 만들지 않는다 — 프론트에 목록·등록 라우트가 없다(FS-018 §1). **미생성 상태를 404 로 표현하지 않는다**: `DocumentFormShell` 은 실패를 `loadFailed` boolean 하나로만 받아(FS-018 §7 #2) 404 를 '내용을 불러오지 못했습니다. + 다시 시도' 로 그린다 — 아직 아무도 채우지 않은 문서에 '다시 시도' 를 권하는 셈이다. 대신 **200 + 전 필드 빈 문자열**을 주면 관리자가 곧바로 채워 넣는 화면을 본다. 같은 이유로 EP-02 는 upsert 다.

### 7.2 주소 · 교통편 XSS — 서버가 저장 시 정제한다 【보안 판정】
`address` · `addressDetail` · `transit` 은 관리자 자유 입력이며 **고객 '오시는 길' 화면에 그대로 노출된다**(이 화면의 존재 이유다). 관리자 화면은 `<input>`/`<textarea>` 값으로만 다루므로 여기서는 실행되지 않지만, 고객 렌더가 HTML 로 해석하면 **저장형 XSS** 가 된다 — `transit` 은 1000자 자유 텍스트에 줄바꿈까지 보존하는, 이 도메인에서 가장 넓은 입력면이다. BE-010 §7.2 와 동일 판정: **서버가 저장 시 허용 태그 화이트리스트 밖의 마크업 · `javascript:`/`data:` 스킴 · 이벤트 핸들러 속성을 제거**하고, 계약은 '저장된 값에 실행 가능한 스크립트가 없다' 는 관측 동작만 정한다. 정제로 값이 바뀌면 EP-02 성공 후 프론트가 재조회하므로(§6) 관리자가 정제 결과를 즉시 본다.

### 7.3 좌표 범위·형식 — 서버가 정본이다 【보안·정합 판정】
`directionsSchema` 의 좌표 검증(`validation.ts:8-22`)은 **브라우저 안에서만** 돈다. 관리자가 폼을 우회해 요청을 보내면(또는 프론트 버그로 검증이 건너뛰이면) 아무것도 막지 않는다. 서버는 **저장 트랜잭션 안에서 같은 세 판정을 다시** 한다:

1. **형식** — 앞뒤 공백 제거 후 `/^-?\d+(\.\d+)?$/`. 이 정규식은 의도적으로 좁다: `1e2`(지수) · `Infinity` · `NaN` · `+37.5`(선행 `+`) · `37.5.1` 을 전부 거절한다. 서버가 `parseFloat`/`Number()` 로 느슨하게 받으면 프론트가 막던 값이 통과해 **두 계층의 진실이 갈린다** — `Number('1e2')` 는 100 이고 `Number('  37.5  ')` 는 37.5 다. 서버도 **정규식 우선, 파싱은 그 다음**이어야 한다.
2. **범위** — `|latitude| ≤ 90` · `|longitude| ≤ 180`. 범위 밖은 지구상에 없는 좌표이며, 지도 SDK 가 붙으면 렌더 예외 또는 엉뚱한 위치가 된다.
3. **위반은 400 `VALIDATION_FAILED`** 다(422 아님) — 형식·범위 위반이지 상태 위반이 아니다.

정규화(공백 제거)는 서버가 하고, **후행 0 등 표기는 보존한다**(§3) — `'37.500000'` 을 `'37.5'` 로 바꿔 돌려주면 관리자가 저장한 값과 다음 조회에서 본 값이 달라진다.

### 7.4 동시 수정 — 마지막 쓰기 승리(409 를 정의하지 않는다) 【정합 판정】
`Directions` 에 `updatedAt`/`version` 이 없고 `pages/company/**` 어디에도 `If-Match` 가 없다(전수 grep 0건). 프론트의 저장 경로(`useSaveDocument` → `document.ts:58-63`)는 조건부 요청을 보내지 않으며, `DocumentFormShell` 에는 충돌 다이얼로그를 받을 prop 자체가 없다(목록형 폼의 `FormPageShell` 에는 `conflict` 가 있다 — 단일 문서 폼에는 없다). **서버가 409 를 올려도 프론트는 '저장하지 못했습니다…' 로 뭉개고 입력만 남는다** — 사용자가 무엇이 충돌했는지도, 무엇을 눌러야 하는지도 알 수 없다.

그래서 이 계약은 **409 를 정의하지 않는다**: 두 관리자가 동시에 편집하면 마지막 PUT 이 이기고 앞선 변경은 통지 없이 사라진다. 이것은 **의도된 타협이 아니라 기록된 결손**이다 — 오시는 길은 변경 빈도가 낮아 위험이 상대적으로 작다는 것이 유일한 근거다. 낙관적 잠금을 도입하려면 세 가지가 함께 와야 한다: ① `Directions` 에 `version`(또는 `updatedAt`) 추가 ② EP-01 이 `ETag` 를, EP-02 가 `If-Match` 를 다룸 ③ `DocumentFormShell` 에 `conflict` prop + `FormConflictDialog` 배선(`FormPageShell.tsx:195` 가 이미 갖고 있는 것과 같은 것). §7.9 로 이관한다.

### 7.5 지도 임베드 — 심이 없다, 계약을 만들지 않는다
FS-018 §5 · §4 EP-03 대로 프론트에 지도 관련 어댑터 함수도 엔드포인트 심도 없다. **여기서 `GET /api/company/map-config` 같은 것을 지어내면 백엔드가 아무도 부르지 않는 것을 만든다.** 결정이 필요한 것은 계약이 아니라 **선행 설계**다: (a) 지도 제공자(카카오/구글/네이버) (b) SDK 키를 클라이언트 번들에 넣을지 서버가 발급/프록시할지 (c) 도메인 화이트리스트로 키를 묶을지 (d) 좌표만 넘길지 마커·줌·경로 안내까지 계약할지. (b)·(c) 가 서버 몫으로 정해지는 경우에만 이 문서에 엔드포인트가 추가된다.

### 7.6 주소와 좌표의 불일치 — 서버가 검사하지 않는다
관리자가 주소를 서울로, 좌표를 부산으로 넣어도 아무것도 막지 않는다 — 지오코딩 심이 없기 때문이다(§4 EP-04). 서버가 저장 시 주소를 지오코딩해 좌표와 대조하는 것은 **가능하지만 이 계약의 범위 밖**이다: 외부 지오코딩 API 의존·레이트리밋·실패 시 저장을 막을지 경고만 할지가 전부 미정이고, 프론트에 경고를 표시할 자리도 없다(FS-018 에 그런 요소가 없다). 지금은 **관리자 입력을 그대로 믿는다**고 못 박는다.

### 7.7 중복 저장 — PUT 멱등성이 유일한 방어다 【정합 판정】
프론트에 동기 제출 잠금이 없다(FS-018 §7 #9 — `DirectionsPage.tsx:100-118` 의 `onValid` 는 `useCrudForm.ts:102` 의 `submitLockRef` 에 해당하는 것이 없다). 저장 버튼의 `disabled={saving}` 은 RHF 의 **비동기 검증이 끝난 뒤에야** 켜지므로, 그 틈의 두 번째 Enter 가 두 번째 PUT 을 만든다. 서버는 이 중복을 **막을 수 없고 막을 필요도 없다** — PUT 전체 치환이 멱등이므로 같은 바디 두 번은 한 번과 같다. `Idempotency-Key` 를 요구하지 않는 이유도 이것이다(금액·생성·발송이 아니라 문서 덮어쓰기다). **단, 이 멱등성이 방어인 것은 두 요청의 바디가 같을 때뿐이다** — 프론트 잠금은 여전히 고쳐야 한다(중복 요청은 레이트리밋과 서버 부하로 남는다).

### 7.8 프론트가 서버의 구분을 버린다 — 계약이 아니라 결손이다
서버는 400(필드별 `error.fields`) · 403 · 429(`Retry-After`) · 500(`traceId`) · 504 를 구분해 내리지만, **이 화면은 전부 같은 두 문구로 그린다**: 조회 실패 → '내용을 불러오지 못했습니다.'(FS-018-EL-014) · 저장 실패 → '저장하지 못했습니다. 잠시 후 다시 시도해 주세요.'(FS-018-EL-003). `error.fields` 를 인라인 에러로 꽂지 않고(`useCrudForm.ts:176-186` 이 하는 일을 이 화면은 하지 않는다), `traceId`/오류 참조 코드를 보여 주지 않는다(`FormFeedback.tsx:44` 의 `errorReference` 표시가 `DocumentFormShell` 에는 없다). **서버는 계약대로 구분해 내려야 한다** — 프론트가 지금 그것을 쓰지 못한다는 사실이 계약을 축소시킬 이유는 아니다. §7.9 로 이관한다.
