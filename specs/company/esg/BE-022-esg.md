# BE-022. ESG 활동 관리 백엔드 기능 명세

## 1. 개요

| 항목 | 내용 |
|---|---|
| 대상 화면 | FS-022 ESG 활동 관리 (`/company/esg` · `/company/esg/new` · `/company/esg/:id/edit`) |
| 범위 | ESG 활동 목록 조회 · 상세 조회 · 등록 · 수정 · 삭제(단건·일괄) |
| 범위 밖 | **이미지 업로드**(§4 EP-06 · §7.6 — 프론트에 심이 없다). **분류(카테고리) CRUD**(§7.3 — 코드 상수다). 고객 화면 렌더(§7.2 XSS 판정만). 일괄 전용 엔드포인트(§7.1) |
| 프론트 어댑터 | `apps/admin/src/pages/company/esg/data-source.ts` (`esgAdapter`) |
| 도메인 타입 | `apps/admin/src/pages/company/esg/types.ts` (`EsgItem` · `EsgInput` · `EsgCategory`) |
| 검증 정본 | `apps/admin/src/pages/company/esg/validation.ts` (`esgSchema`) — **클라이언트 검증은 UX 다. 서버가 정본이다**(§7.4) |

> **에러 봉투·권한 모델 상속**: BE-003 §2·§3 을 그대로 상속한다(BE-010 §2 와 동일 선언). 아래는 ESG 고유 차이만 기술한다.

## 2. 공통 (상속)

- **에러 봉투**: BE-003 §2. 공통 에러코드 동일(`error.code` · `error.message` · `error.fields` · `error.traceId`).
- **권한**: `admin` = 전체. `operator` = 조회 계열만(목록·상세), 쓰기(등록·수정·삭제) → 403. 기업 도메인 읽기 권한 없는 관리자 → **컬렉션 403 / 개별 404 은닉**(BE-003 §3.2 · §7.7).
- **CSRF**: 쓰기(POST·PUT·DELETE)에 `X-CSRF-Token`. 누락·불일치 → 403 `CSRF_TOKEN_INVALID`.
- **타임아웃**: 조회·쓰기 5초 → 504 `REQUEST_TIMEOUT`. **프론트 상한은 없다**(FS-022 §4.1).
- **프론트 역할 분기 없음**(FS-022 §4.1) — 권한 강제는 전적으로 서버 책임이다. 등록·수정·삭제 버튼은 권한과 무관하게 렌더되고 눌린다(FS-022 §7 #2).

## 3. 데이터 계약 (types.ts 대조)

| 타입 | 필드 | 비고 |
|---|---|---|
| `EsgItem`(목록 행 = 상세) | `id: string` · `category: EsgCategory` · `title: string` · `summary: string` · `date: string` · `imageUrls: readonly string[]` | **목록과 상세가 같은 타입이다** — 목록 응답에 `summary`·`imageUrls` 까지 전부 들어온다(§7.5) |
| `EsgInput`(등록·수정 바디) | `category` · `title` · `summary` · `date` · `imageUrls` | `EsgItem` 에서 `id` 만 뺀 것 |
| `EsgCategory` | `'environment' \| 'social' \| 'governance'` | **코드 상수 3종 고정**(§7.3). 서버는 이 세 값만 저장·반환한다 |
| — | createdAt · updatedAt · version | **없다**. 낙관적 잠금 토큰도 감사 필드도 타입에 존재하지 않는다(§7.8 · §7.9) |

| 상수 (types.ts) | 값 | 강제 지점 |
|---|---|---|
| `TITLE_MAX_LENGTH` | 120 | EP-03/EP-04 `title` |
| `SUMMARY_MAX_LENGTH` | 1000 | EP-03/EP-04 `summary` |
| `MAX_ESG_IMAGES` | 10 | EP-03/EP-04 `imageUrls` 배열 길이 |
| 일자 형식 | `/^\d{4}-\d{2}-\d{2}$/` (앞뒤 공백 제거 후) | EP-03/EP-04 `date` (`validation.ts:7,24-26`) |

**`date` 가 `Date`/ISO datetime 이 아니라 `'YYYY-MM-DD'` 문자열인 이유**: 화면이 `<input type="date">`(FS-022-EL-012.2)의 값을 그대로 다루고 그대로 렌더한다(FS-022-EL-006.7 — `shared/format` 을 거치지 않는다). **응답도 `'YYYY-MM-DD'` 로 준다** — 서버가 `2024-03-05T00:00:00+09:00` 을 내리면 `<input type="date">` 가 값을 받지 못하고 정렬(`sortEsg` 는 문자열 비교다 — `types.ts:52-57`)도 깨진다. ESG 활동 일자는 시각·타임존이 없는 **달력 날짜**이므로 이 표현이 도메인상 옳다.

**`imageUrls` 는 URL 문자열 배열이다** — 파일도 id 도 아니다. 서버는 업로드된 이미지의 **접근 가능한 URL** 을 그대로 저장·반환한다(§7.6).

## 4. 엔드포인트 명세

### BE-022-EP-01 · ESG 활동 목록 조회

| 항목 | 내용 |
|---|---|
| 근거 (FS) | FS-022-EL-001.2, EL-003, EL-004, EL-006, EL-007, EL-008, EL-009 |
| 메서드·경로 | `GET /api/company/esg` |
| 권한 | `admin`, `operator` |
| 멱등성 | 멱등(GET) |
| 페이징 | **없음 — 전량 반환**(§7.5 · FS-022 §7 #3) |
| 레이트리밋 | 분당 120회 |

**쿼리**: **없다.** 어댑터 시그니처가 `fetchAll(signal)`(`crud.ts:17`)로 파라미터를 받지 않는다 — 분류 필터도 정렬도 클라이언트에서 일어난다(FS-022 §4.1 '필터 적용 지점'). `?category=` 를 지어내지 않는다: 프론트가 보내지 않으므로 서버가 구현해도 죽은 표면이다.

**응답 200** — `readonly EsgItem[]`. **전 필드 포함**(`summary`·`imageUrls` 까지 — §3, §7.5). 서버 정렬은 요구하지 않는다: 프론트가 `sortEsg`(일자 내림차순, 동일 일자는 id 역순 — `types.ts:52-57`)로 **항상 다시 정렬**하므로(`crud.ts:39`) 서버 순서는 관측되지 않는다. 0건이면 `[]`(404 아님).

**에러**: 401 `UNAUTHENTICATED` · 403 `FORBIDDEN` · 429 `RATE_LIMITED` · 500 `INTERNAL_ERROR` · 504 `REQUEST_TIMEOUT`.

### BE-022-EP-02 · ESG 활동 상세 조회

| 항목 | 내용 |
|---|---|
| 근거 (FS) | FS-022-EL-014, EL-014.1 |
| 메서드·경로 | `GET /api/company/esg/:id` |
| 권한 | `admin`, `operator`. 기업 도메인 읽기 권한 없음 → **404 은닉**(§7.7) |
| 멱등성 | 멱등(GET) |

**응답 200** — `EsgItem`. 프론트는 수정 폼을 채우는 데만 쓴다(읽기 전용 상세 화면이 없다 — FS-022 §1).

**404 의 UX 가 다르다**: 어댑터가 `HttpError(404)` 를 던지면(`crud.ts:54-57`) `useCrudForm` 이 `loadFailure = 'not-found'` 로 갈라(`useCrudForm.ts:144-149`) **'다시 시도' 를 주지 않고** '목록으로' 만 제공한다(`FormPageShell.tsx:120-142`). 그러므로 **일시적 장애를 404 로 내리면 안 된다** — 그것은 영구 부재의 신호다.

**에러**: 400 `VALIDATION_FAILED`(id 형식) · 401 · 404 `NOT_FOUND` · 429 · 500 · 504.

### BE-022-EP-03 · ESG 활동 등록

| 항목 | 내용 |
|---|---|
| 근거 (FS) | FS-022-EL-013.1(등록) |
| 메서드·경로 | `POST /api/company/esg` |
| 권한 | `admin` 만. `operator` → 403 |
| 멱등성 | **비멱등**. 그러나 프론트가 `Idempotency-Key` 를 보낸다 — §7.10 |
| 레이트리밋 | 분당 30회 |

**바디**(`EsgInput`):

| 필드 | 제약 | 위반 시 `error.fields[].name` |
|---|---|---|
| `category` | `environment` \| `social` \| `governance` 중 하나(§7.3) | `category` |
| `title` | 앞뒤 공백 제거 후 1–120자. **서버 정제 §7.2** | `title` |
| `summary` | 앞뒤 공백 제거 후 1–1000자. 줄바꿈 보존. **서버 정제 §7.2** | `summary` |
| `date` | `/^\d{4}-\d{2}-\d{2}$/` + **실재하는 날짜**(§7.4) | `date` |
| `imageUrls` | 문자열 배열, 길이 0–10. **스킴 화이트리스트 §7.6** | `imageUrls` |

**응답 201** — 생성 `EsgItem`. 프론트 `create(input, signal): Promise<void>` 는 **응답 본문을 읽지 않는다**(`crud.ts:19`) — 성공 후 목록을 무효화해 EP-01 로 재조회한다(`crud.ts:179-181`).

**헤더**: `X-CSRF-Token`(필수) · `Idempotency-Key`(§7.10).

**에러**: 400 `VALIDATION_FAILED`(`error.fields`) · 401 · 403 `FORBIDDEN` · 403 `CSRF_TOKEN_INVALID` · 422 `UNPROCESSABLE`(§7.4 — 필드 단위 거절을 프론트가 인라인으로 꽂는다) · 429 · 500 · 504.

### BE-022-EP-04 · ESG 활동 수정

| 항목 | 내용 |
|---|---|
| 근거 (FS) | FS-022-EL-013.1(수정), EL-014.4 |
| 메서드·경로 | `PUT /api/company/esg/:id` |
| 권한 | `admin` 만. `operator` → 403 |
| 멱등성 | **멱등**(PUT 전체 치환). 부분 갱신 없음 |
| 레이트리밋 | 분당 30회 |

**바디**: `EsgInput`(EP-03 동일 제약). **응답 200/204**. 프론트 `update(id, input, signal): Promise<void>`.

**409 를 실제로 쓴다** — 이 도메인에서 **유일하게 충돌 UI 가 배선된 경로**다:

| 조건 | 응답 | 프론트 동작 |
|---|---|---|
| 대상 id 가 이미 삭제됨 | **409 `CONFLICT`** · `message` = '다른 사용자가 먼저 삭제한 항목입니다.' | `useCrudForm.ts:159-172` → FS-022-EL-014.4 충돌 다이얼로그. **입력 보존 · 성공 토스트 없음 · 목록 이동 없음** |
| (§7.8 도입 시) `If-Match` 불일치 | **412 `PRECONDITION_FAILED`** | 위와 동일 — `isConflict`(`http-error.ts:105-107`)가 409·412 를 같은 UX 로 수렴시킨다 |

**404 를 쓰지 않는다**(수정 경로에서): 어댑터가 없는 id 에 대해 409 를 던지고(`crud.ts:71-74`) 프론트가 그것을 충돌로 그린다. 서버가 404 를 내리면 `isConflict` 가 false 라 generic '저장하지 못했습니다' 배너로 떨어져 **'다른 관리자가 지웠다' 는 사실이 사라진다** — §7.11 로 이관한다.

**에러**: 400 · 401 · 403 `FORBIDDEN` · 403 `CSRF_TOKEN_INVALID` · **409 `CONFLICT`** · 412(§7.8 도입 시) · 422 · 429 · 500 · 504.

### BE-022-EP-05 · ESG 활동 삭제 (단건·일괄 공용)

| 항목 | 내용 |
|---|---|
| 근거 (FS) | FS-022-EL-006.8, EL-015, EL-005.1, EL-016 |
| 메서드·경로 | `DELETE /api/company/esg/:id` |
| 권한 | `admin` 만. `operator` → 403 |
| 멱등성 | **비멱등으로 계약한다 — 어댑터가 그렇게 동작하기 때문이다**(§7.11) |
| 레이트리밋 | 분당 60회(일괄이 항목당 개별 요청) |

**응답 204**. 일괄 삭제는 프론트가 `settleAll` 로 항목마다 **병렬 DELETE** 한다(§7.1).

**이미 삭제된 id 재요청** → **409 `CONFLICT`** · `message` = '이미 삭제된 항목입니다.'. 어댑터가 그렇게 던진다(`crud.ts:82-85`). 이것은 **BE-010-EP-07(FAQ 삭제, 멱등 204)과 갈리는 판정**이며 의도된 것이 아니다 — §7.11 참조.

**에러**: 400 · 401 · 403 `FORBIDDEN` · 403 `CSRF_TOKEN_INVALID` · **409 `CONFLICT`**(이미 삭제) · 429 · 500 · 504.

### BE-022-EP-06 · 이미지 업로드 — **심 없음 (미정)**

`EsgFormPage.tsx:161-169` 의 `ImageGalleryField` 는 선택한 파일을 **서버로 보내지 않는다**. `packages/ui/src/molecules/ImageGalleryField/ImageGalleryField.tsx:149` 가 `URL.createObjectURL(file)` 로 만든 `blob:` URL 을 값으로 넣고, 그것이 `imageUrls` 에 그대로 담겨 EP-03/EP-04 로 간다. **업로드 어댑터 함수도 `// TODO(backend): POST /api/...` 주석도 없다**(`data-source.ts` 의 유일한 주석은 `:48` 의 CRUD 5종이다).

**따라서 이 문서는 업로드 엔드포인트를 정의하지 않는다.** 지어내면 백엔드가 아무도 부르지 않는 것을 만든다. 먼저 결정돼야 하는 것(§7.6):

1. 업로드 시점 — 파일 선택 즉시 업로드(URL 을 받아 폼 값에 넣는다) vs 폼 제출 시 multipart 동봉
2. 저장 위치·URL 형태 — 오브젝트 스토리지 공개 URL vs 서명 URL vs `/api/files/:id`
3. 고아 파일 회수 — 업로드 후 폼을 버리면 파일이 남는다(즉시 업로드 방식의 필연)
4. 제약 — 서버측 MIME·용량·해상도 상한(현재 클라이언트만 `image/*` · 5MB · 10장을 본다)

그 전까지 EP-03/EP-04 의 `imageUrls` 계약은 **§7.6 의 스킴 화이트리스트로만** 방어한다.

---

### BE-022-EP-07 · 분류(카테고리) CRUD — **해당 없음 (심 없음)**

분류는 서버 데이터가 아니라 **코드 상수**다 — `ESG_CATEGORY_OPTIONS`(`types.ts:33-37`) 3종 고정. 조회·생성·삭제 어댑터 함수도 `// TODO(backend)` 주석도 없다. FAQ(BE-010-EP-02·EP-03·EP-11·EP-12)가 카테고리 엔드포인트 4종을 갖는 것과 대비되며, 이는 **의도된 차이**다 — §7.3.

## 7. 핵심 판정

### 7.1 일괄 삭제 — 전용 엔드포인트를 만들지 않는다
프론트가 `settleAll`(`bulk.ts:15-21`)로 단건 DELETE(EP-05)를 병렬 호출하고 취소가 아닌 실패의 **건수**를 집계한다(`useCrudList.tsx:135-148`). BE-010 §7.1 과 같은 근거로 전용 벌크 계약을 두지 않는다. **다만 ESG 에는 페이징이 없어 선택 상한도 없다**(FS-022 §7 #3, #11) — FAQ 는 페이지당 10건이 곧 선택 상한이었지만 여기서는 '전체 선택'이 조회된 전량을 고른다. 그래서 **EP-05 의 레이트리밋(분당 60)이 실제로 걸릴 수 있다**: 100건 선택 → 100개 병렬 DELETE → 40건이 429 로 실패해 '100건 중 40건을 삭제하지 못했습니다' 가 뜨고, 재시도는 **성공분까지 다시 요청한다**(FS-022 §7 #16). 벌크 엔드포인트가 아니라 **선택 상한 또는 페이징**이 옳은 해법이며 §7.14 로 이관한다.

### 7.2 제목 · 내용 XSS — 서버가 저장 시 정제한다 【보안 판정】
`title`(120자) · `summary`(1000자)는 관리자 자유 입력이며 **고객 ESG 화면에 그대로 노출된다**(이 화면의 존재 이유다). 관리자 화면은 텍스트 노드로만 렌더하지만(FS-022-EL-006.5/.6), 고객 렌더가 HTML 로 해석하면 **저장형 XSS** 가 된다. BE-010 §7.2 · BE-018 §7.2 와 동일 판정: **서버가 저장 시 허용 태그 화이트리스트 밖의 마크업 · `javascript:`/`data:` 스킴 · 이벤트 핸들러 속성을 제거**하고, 계약은 '저장된 값에 실행 가능한 스크립트가 없다' 는 관측 동작만 정한다. 정제로 값이 바뀌면 EP-03/EP-04 성공 후 프론트가 목록을 무효화해 재조회하므로(§6) 관리자가 결과를 즉시 본다.

### 7.3 분류는 서버 데이터가 아니다 — 코드 상수 3종 고정
FAQ 는 카테고리를 서버에서 받고 관리자가 만들고 지운다(BE-010-EP-02·EP-11·EP-12). **ESG 는 그렇지 않다**: `ESG_CATEGORY_OPTIONS`(`types.ts:33-37`)가 환경·사회·지배구조 3종을 코드에 박아 두고, `EsgCategory` 가 그 세 문자열의 union 이며(`types.ts:4`), 필터(`EsgCategoryFilter.tsx:23-26`)·폼 select(`EsgFormPage.tsx:110-114`)·배지 색(`types.ts:45-49`)이 전부 그것에서 파생된다. 카테고리 조회·생성·삭제 어댑터 함수도 `// TODO(backend)` 주석도 없다.

**이것은 의도된 차이다.** ESG 의 E/S/G 는 국제적으로 확립된 3분류이며 관리자가 늘리고 줄일 성질이 아니다 — FAQ 카테고리('결제'·'배송'…)와 달리 도메인이 값을 고정한다. 그러므로 서버도 **이 세 값만 저장·반환하고 그 밖의 값은 400 으로 거절한다**(§7.4). 4번째 분류를 추가하려면 코드 변경(상수 + tone 매핑 + 라벨)이 필요하고, 그것이 옳다 — 배지 색과 라벨이 코드에 있기 때문에 서버가 임의 분류를 내려도 화면이 `neutral` + 원문 문자열로 떨어질 뿐이다(`types.ts:41,48`).

### 7.4 검증 — 서버가 정본이다 【보안·정합 판정】
`esgSchema`(`validation.ts:9-34`)는 **브라우저 안에서만** 돈다. 폼을 우회한 요청은 아무것도 막지 않는다. 서버는 저장 트랜잭션 안에서 같은 판정을 다시 하되, **두 곳에서 클라이언트보다 엄격해야 한다**:

1. **`date` 의 실재성** — 클라이언트는 `/^\d{4}-\d{2}-\d{2}$/` **형식만** 본다(`validation.ts:24-26`). `2024-13-45` 가 통과한다(`<input type="date">` 가 UI 상 막지만 계약은 아니다). 서버는 **파싱해 실재하는 날짜인지** 확인하고 아니면 400 `error.fields[].name = 'date'`.
2. **`category` 의 union** — 세 값 밖이면 400. 클라이언트 select 에는 3옵션뿐이라 폼으로는 위반이 불가능하지만(FS-022-EL-012.1), 그것이 서버가 검사하지 않을 이유는 아니다.

**나머지는 클라이언트와 같게** 한다 — `title` 1–120 · `summary` 1–1000(공백 제거 후 비어 있지 않을 것) · `imageUrls` 길이 0–10. **위반은 400 이 아니라 422 로 내리는 것이 프론트에 유리하다** — §7.13.

**미래 일자·과거 하한은 정하지 않는다**: ESG '활동' 일자가 미래여도 되는지(예고된 활동)가 도메인상 미정이다(FS-022 §7 #13). 지금은 제약을 두지 않는다고 못 박고 §7.14 로 이관한다.

### 7.5 목록이 전량 · 전 필드다 — 계약이 아니라 현재 형태다
EP-01 은 페이징 파라미터가 없고(`fetchAll(signal)`) `EsgItem` 전 필드를 준다 — `summary`(1000자)와 `imageUrls`(최대 10개 URL)까지 포함해서다. FAQ 가 목록(`FaqSummary`)과 상세(`Faq` = `FaqSummary` + `answer`)를 나눈 것(BE-010 §3)과 대비된다.

**지금은 이것이 옳다**: 목록 화면이 `summary` 를 실제로 렌더하고(FS-022-EL-006.6) 시드가 4건이며 페이징이 없다. **그러나 이것은 확장되지 않는다** — ESG 활동이 수백 건이 되면 응답이 수백 KB 가 되고(내용 1000자 × N + URL 배열), 전량이 DOM 에 그려진다(FS-022 §7 #3). 페이징이 도입되면 **그때 목록 타입을 `summary` 없이 좁히는 것을 함께 검토**해야 한다(계약 축소이므로 프론트 변경을 수반한다 — 목록 내용 열을 지우거나 별도 조회가 필요하다). §7.14 로 이관한다.

### 7.6 이미지 — 업로드 심이 없다. 계약을 만들지 않되 스킴은 막는다 【보안 판정】
§4 EP-06 대로 업로드 엔드포인트를 정의하지 않는다. **그러나 `imageUrls` 는 이미 EP-03/EP-04 의 바디에 들어오고 고객 화면의 `<img src>` 가 된다** — 그러므로 **서버는 지금도 이 배열을 검증해야 한다**:

1. **스킴 화이트리스트** — `https:` 만 허용한다(운영 도메인이 정해지면 origin 까지 좁힌다). `javascript:` 는 명백하고, **`data:` 도 거절한다**(SVG data URL 은 스크립트를 실행한다). **`blob:` 도 거절한다** — 지금 프론트가 보내는 것이 정확히 그것이며(FS-022-EL-012.6), 그 URL 은 만든 브라우저 탭 안에서만 유효하고 폼을 떠나는 순간 죽는다. 서버가 그것을 저장하면 **DB 에 영원히 깨진 URL 이 남는다**. 400 `error.fields[].name = 'imageUrls'` 로 거절하는 것이 옳다.
2. **길이** — 배열 0–10(`MAX_ESG_IMAGES`). URL 문자열 자체의 길이 상한도 둔다(예: 2048자).

**이 판정의 귀결**: 서버가 스킴을 제대로 막으면 **업로드 심이 붙기 전까지 ESG 폼에서 이미지를 올릴 수 없다**(전부 400). 그것이 지금의 조용한 데이터 손상(FS-022 §7 #9)보다 낫다 — 관리자가 이미지를 올렸는데 목록에서 깨져 보이는 것보다, 저장이 거절되고 '아직 지원하지 않는다' 는 사실이 드러나는 편이 정직하다. **업로드 심(EP-06)과 이 검증은 함께 와야 한다** — §7.14.

### 7.7 403 vs 404 은닉 — BE-003 §3.2 를 그대로 상속하되 ESG 는 개인정보가 아니다
| 리소스 | 권한 부족 시 | 근거 |
|---|---|---|
| `GET /api/company/esg`(컬렉션) | **403** | 컬렉션의 존재는 비밀이 아니다(BE-003 §3.2 원칙 1). 개별 id 가 노출되지 않는다 |
| `GET /api/company/esg/:id` | 기업 도메인 읽기 권한 없음 → **404** / `operator`(읽기 가능) → 200 | 원칙 2 의 형식적 적용 |
| `PUT`·`DELETE /api/company/esg/:id` | `operator` → **403** / 읽기 권한 없음 → **404** | `operator` 는 이미 목록에서 이 항목을 본다 — 존재를 숨기는 것이 무의미하다 |

**단, ESG 는 회원 데이터와 다르다**: ESG 활동은 **고객에게 공개될 목적으로 존재하는 정보**이며 id 열거가 개인정보 유출이 아니다. 그러므로 상세 404 은닉은 **일관성을 위한 것이지 실질 보호가 아니다**. 이 판정을 기록하는 이유: 나중에 '왜 여기만 404 인가' 를 묻지 않기 위해서다. 은닉을 걷어내고 403 으로 통일해도 이 도메인에서는 잃는 것이 없다 — BE-003 §3.2 의 원칙 2 를 도메인별로 재평가하는 것은 별도 일이다.

### 7.8 동시 수정 — 낙관적 잠금이 없다. 그러나 충돌 UI 는 이미 있다 【정합 판정】
`EsgItem` 에 `updatedAt`/`version` 이 없고 `pages/company/**` 어디에도 `If-Match` 가 없다(전수 grep 0건). 그래서 두 관리자가 **같은 ESG 활동을 동시에 편집하면 마지막 PUT 이 이기고** 앞선 변경은 통지 없이 사라진다. 어댑터가 409 를 던지는 것은 **대상이 삭제된 경우뿐**이다(`crud.ts:71-74`).

**그런데 이 화면은 특이하게도 충돌 UI 가 이미 완비돼 있다**: `useCrudForm.ts:159-172` → `FormPageShell.tsx:195` → `FormFeedback.tsx:58-74` 의 경로가 409·412 를 받아 입력을 보존한 채 다이얼로그를 띄운다(단일 문서 폼에는 이것이 없다 — BE-018 §7.4 와 대비). **즉 서버가 토큰을 도입하면 프론트 변경 없이 즉시 성립한다.** 필요한 것은 셋뿐이다: ① `EsgItem`/`EsgInput` 에 `version`(또는 `updatedAt`) 추가 ② EP-02 가 `ETag`, EP-04 가 `If-Match` 를 다룸 ③ 어댑터 본문이 그 헤더를 싣고 412 를 `HttpError` 로 변환. **③ 은 시그니처를 바꾸지 않는다.** 이것이 이 도메인에서 EXC-04 를 완성하는 가장 값싼 경로이며 §7.14 로 이관한다.

### 7.9 감사 필드가 없다 — 공개 정보인데 변경 이력이 남지 않는다
`EsgItem` 에 `createdAt`/`updatedAt`/`createdBy`/`updatedBy` 가 없다(§3). ESG 활동은 **고객에게 공개되는 대외 발표**이며 누가 언제 무엇을 바꿨는지가 남아야 하는 성질이다(잘못된 ESG 공시는 기업 리스크다). 지금은 화면이 그 필드를 요구하지 않으므로 **계약에 넣지 않되**, 서버측 감사 로그(요청 단위)는 이 계약과 별개로 반드시 필요하다고 기록한다. 화면에 '최종 수정' 열/필드를 노출하려면 FS 변경이 선행된다 — §7.14.

### 7.10 멱등키 — 프론트가 만들지만 어댑터가 받지 않는다 【정합 판정】
`useCrudForm.ts:118-123` 의 `idempotencyKeyRef` 는 **제출 시도 단위로** 키를 만들고(mutationFn **밖**에서 — 재시도가 같은 키를 재사용하도록), 성공하면 폐기한다(`:214`). 주석(`:110`)이 `TODO(backend): 이 키는 Idempotency-Key 헤더로 나간다` 라고 명시한다.

**그러나 `takeIdempotencyKey()`(`:204`)의 반환값이 어디에도 쓰이지 않는다** — `CrudAdapter.create/update`(`crud.ts:19-20`)의 시그니처가 `(input, signal?)` 라 키를 받을 자리가 없다. 즉 **키는 만들어지고 버려진다.** 이것은 죽은 코드가 아니라 **미완의 배선**이다.

계약 판정: **EP-03 은 `Idempotency-Key` 헤더를 받아 최초 응답을 재생한다.** 등록은 비멱등이고(같은 ESG 활동이 두 건 생긴다) 프론트에 재시도 경로가 있으므로(실패 배너 → 재클릭) 이 헤더가 필요하다. EP-04 는 PUT 전체 치환이라 멱등이므로 헤더가 없어도 무해하다. **어댑터 시그니처 확장(`CrudAdapter` 에 `idempotencyKey?` 추가)이 선행돼야 하며** §7.14 로 이관한다.

### 7.11 삭제 멱등성 — 어댑터가 FAQ 와 반대로 동작한다 【정합 판정】
BE-010-EP-07(FAQ 삭제)은 **멱등**이다 — 이미 삭제된 id 재요청에 204 를 준다. **ESG 는 그렇지 않다**: `crud.ts:82-85` 가 `HttpError(409, '이미 삭제된 항목입니다.')` 를 던진다. 같은 이유로 `update`(`crud.ts:71-74`)도 없는 id 에 409 를 준다.

**이 비대칭은 의도된 설계가 아니라 두 어댑터의 역사다.** `createCrudAdapter` 의 409 는 **유령 저장(ghost saved)을 막기 위해** 들어온 것이고(`crud.ts:68-70` 주석 — 예전에는 `map`/`filter` 가 없는 id 를 조용히 통과시키고 '저장했습니다' 를 띄웠다) 그 목적에서는 옳다. 문제는 **삭제**다:

- **수정의 409 는 옳다** — 사용자가 편집 중인 것이 사라졌다는 사실을 알려야 하고, 충돌 다이얼로그가 그것을 제대로 한다(§7.8).
- **삭제의 409 는 나쁘다** — 사용자가 원한 결과(그 항목이 없어지는 것)가 **이미 달성됐는데** '삭제하지 못했습니다' 를 보여 준다. 일괄 삭제에서는 더 나쁘다: 다른 관리자가 먼저 지운 3건이 '10건 중 3건을 삭제하지 못했습니다' 로 세어지고, 재시도하면 또 3건이 실패한다 — **영원히 성공하지 않는 재시도**다.

**판정: EP-05 는 멱등이어야 한다** — 이미 삭제된 id 재요청에 **204**(BE-010-EP-07 과 동일). 그러려면 어댑터(`crud.ts:82-85`)가 바뀌어야 하며 그것은 앱 공용 프레임워크 변경이다(연혁·인증서·ESG·조직도가 함께 영향). **이 문서는 계약을 옳게 적고(204) 코드가 그것과 다르다는 사실을 §7.14 로 이관한다** — 계약을 코드의 현재 형태에 맞춰 굽히지 않는다.

### 7.12 프론트가 서버의 구분을 부분적으로 버린다
서버는 400/403/404/409/422/429/500 을 구분해 내리고, **이 화면은 그중 404·409·422 를 제대로 쓴다**(BE-018 의 단일 문서 폼이 전부 버리는 것과 대비): 404 → not-found 화면 · 409/412 → 충돌 다이얼로그 · 422 → 인라인 필드 에러 + 포커스 · 5xx → 참조 코드. **버려지는 것은 두 가지다**: ① **목록 조회 실패**가 403/404/500 을 뭉갠다(`CrudListShell.tsx:156-165` 가 `error !== null` 하나만 본다) ② **400 이 인라인으로 꽂히지 않는다**(§7.13). 전자는 `CrudListShell` 소관이며 §7.14 로 이관한다.

### 7.13 필드 단위 거절은 422 로 내린다 — 400 이 아니다
`useCrudForm.ts:176` 은 `isUnprocessable(cause)`(= **422**)일 때만 `error.fields` 를 RHF `setError` 로 꽂는다. **400 은 보지 않는다** — generic '저장하지 못했습니다' 배너로 떨어진다.

BE-003 §2.1 은 400 을 `VALIDATION_FAILED`(파라미터 제약 위반), 422 를 `UNPROCESSABLE`(형식은 맞으나 상태 위반)로 나눈다. 형식적으로는 `title` 121자가 400 이다. **그러나 그렇게 내리면 관리자는 어느 필드가 틀렸는지 화면에서 볼 수 없다.**

**판정: EP-03/EP-04 의 필드 단위 거절은 422 `UNPROCESSABLE` + `error.fields` 로 내린다.** 근거: (a) 이 필드들은 클라이언트가 이미 검증하므로(§7.4) 서버 거절이 발생하는 것은 **폼 우회·경합·서버가 더 엄격한 경우**(§7.4 의 날짜 실재성)뿐이고, 후자는 '형식은 맞으나 받아들일 수 없다' 는 422 의 정의에 정확히 들어맞는다 (b) 프론트가 그것을 **이미 옳게 처리한다** — 422 경로가 배선돼 있는데 400 으로 내려 그것을 죽일 이유가 없다. 400 은 **필드로 환원되지 않는 요청 오류**(잘못된 JSON, id 형식, 헤더 누락)에만 쓴다. `useCrudForm` 이 400 도 보도록 넓히는 것이 대안이며 §7.14 에 함께 적는다.
