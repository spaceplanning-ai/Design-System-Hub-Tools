# BE-017. 연혁 관리 백엔드 기능 명세

## 1. 개요

| 항목 | 내용 |
|---|---|
| 대상 화면 | FS-017 연혁 관리 (`/company/history` · `/new` · `/:id/edit`) |
| 범위 | 연혁 목록 조회, 단건 조회, 등록, 수정, 삭제(단건·일괄) |
| 범위 밖 | 고객 노출 렌더(§7.2 XSS 판정만). 일괄 전용 엔드포인트(§7.1). 정렬·필터·페이징 파라미터(§7.3 — 프론트가 요구하지 않는다) |
| 프론트 어댑터 | `apps/admin/src/pages/company/history/data-source.ts` |
| 도메인 타입 | `apps/admin/src/pages/company/history/types.ts` |
| 검증 정본 | `apps/admin/src/pages/company/history/validation.ts` (`historySchema`) |
| 공용 어댑터 | `apps/admin/src/shared/crud/crud.ts` (`createCrudAdapter`) — 5개 함수 시그니처의 원천 |

> **에러 봉투·권한 모델 상속**: BE-003 §2·§3 을 그대로 상속한다. 아래는 연혁 고유 차이만 기술한다.

## 2. 공통 (상속)

- **에러 봉투**: BE-003 §2. 공통 에러코드(`VALIDATION_FAILED`·`UNAUTHENTICATED`·`FORBIDDEN`·`CSRF_TOKEN_INVALID`·`NOT_FOUND`·`CONFLICT`·`UNPROCESSABLE`·`RATE_LIMITED`·`INTERNAL_ERROR`·`REQUEST_TIMEOUT`) 동일. `error.traceId` 는 프론트의 '오류 코드' 표시(FS-017-EL-013)와 대조된다 — §7.6.
- **권한**: `admin` = 전체. `operator` = 조회 계열만(목록·단건), 쓰기(등록·수정·삭제) → 403. 기업 관리 읽기 권한 없는 관리자 → 컬렉션 403 / 개별 404 은닉(BE-003 §3.2). **연혁은 고객 웹사이트에 공개되는 정보라 개별 리소스의 존재 자체는 개인정보가 아니다** — 그러나 은닉 규칙의 일관성을 위해 BE-003 §3.2 를 그대로 따른다(§7.5).
- **CSRF**: 쓰기(POST·PUT·DELETE)에 `X-CSRF-Token`.
- **타임아웃**: 조회·쓰기 5초 → 504.
- **프론트 역할 분기 없음**(FS-017 §4.1) — 권한 강제는 서버 책임. 프론트의 쓰기 버튼은 권한과 무관하게 렌더되므로(FS-017 §7 #5) **서버 403 이 유일한 방벽이다**.

## 3. 데이터 계약 (types.ts 대조)

| 타입 | 필드 | 비고 |
|---|---|---|
| `HistoryItem`(목록 행 = 상세) | `id`(string) · `year`(number) · `month`(number) · `content`(string) | **목록과 상세가 같은 타입이다** — 부분 표현이 없다 |
| `HistoryInput`(쓰기 입력) | `year`(number) · `month`(number) · `content`(string) | 폼이 `Number(...)`·`trim()` 으로 정규화해 보낸다 |

**제약 상수**(`types.ts`): `YEAR_MIN = 1900` · `YEAR_MAX = 2200` · `CONTENT_MAX_LENGTH = 300`.

**정렬**: `sortHistory` 가 **프론트에서** 연·월 내림차순(같은 연·월은 id 역순)으로 정렬한다 — §7.3.

> **`updatedAt`·`version` 필드가 없다.** 이것이 §7.4 낙관적 동시성 판정의 전제다.

## 4. 엔드포인트 명세

### BE-017-EP-01 · 연혁 목록 조회

|---|---|
| 근거 (FS) | FS-017-EL-001, EL-003, EL-005, EL-006, EL-007, EL-008 |
| 메서드·경로 | `GET /api/company/history` |

| 멱등성 | 멱등(GET) |

**응답 200** — `readonly HistoryItem[]`. 정렬은 서버가 보장하지 않아도 되지만(프론트가 `sortHistory` 로 재정렬한다) **연·월 내림차순으로 내려주는 것을 권고**한다 — 프론트 정렬이 제거될 때 계약이 깨지지 않는다.

**에러**: 401 · 403 · 429 · 500 · 504.

**에러**: 400(id 형식) · 401 · 404 `NOT_FOUND` · 429 · 500 · 504.

### BE-017-EP-04 · 연혁 수정
| 항목 | 내용 |
|---|---|
| 근거 (FS) | FS-017-EL-018(수정) |
| 메서드·경로 | `PUT /api/company/history/:id` |
| 권한 | `admin` 만. `operator` → 403 |
| 멱등성 | 멱등(PUT 전체 치환) |
| 레이트리밋 | 분당 30회 |

**바디**: `HistoryInput`(EP-03 동일). **응답 200/204**.

**409 가 계약의 핵심이다**: 대상 id 가 **없으면**(다른 관리자가 먼저 삭제) 200 이 아니라 **409 `CONFLICT`** 로 거절한다. 프론트는 `isConflict` 로 이를 잡아 입력을 보존한 채 충돌 다이얼로그를 띄운다(FS-017-EL-022) — 성공 토스트도 목록 이동도 없다. **없는 id 에 200 을 주면 '저장했습니다' 토스트가 뜨고 목록으로 돌아가지만 저장된 것은 없다(유령 저장).** `message` = '다른 사용자가 먼저 삭제한 항목입니다.'

**에러**: 400 `VALIDATION_FAILED` · 401 · 403 · 403 CSRF · **409 `CONFLICT`**(대상 없음 · 동시 수정 — §7.4) · 412 `PRECONDITION_FAILED`(낙관적 잠금 도입 시 — §7.4) · 429 · 500 · 504.

> 프론트는 409 와 412 를 **같은 UX 로 수렴**시킨다(`isConflict`) — 둘 중 무엇을 쓸지는 서버가 정해도 화면이 갈리지 않는다.

## 7. 핵심 판정

### 7.1 일괄 삭제 — 전용 엔드포인트를 만들지 않는다
프론트가 `settleAll`(`Promise.allSettled`)로 단건 DELETE(EP-05)를 항목마다 병렬 호출하고 부분 실패를 **건수로** 집계한다(FS-017-EL-010.1). BE-010 §7.1 과 같은 근거로 벌크 계약을 두지 않는다.

**단, 이 결정에는 지금 대가가 붙어 있다** — 프론트가 실패 id 를 돌려받지 못해 '실패분만 재시도'를 할 수 없고, 재시도가 성공분까지 다시 보내 409 를 유발한다(FS-017 §7 #3). 전 행 선택이 가능하므로(페이지네이션 없음) 항목 수만큼의 동시 요청이 EP-05 의 분당 60회 레이트리밋을 넘길 수 있다. **연혁이 60건을 넘어서면 벌크 엔드포인트 승격을 재검토한다** — 그 전까지는 단건 병렬이 계약이다.

### 7.2 연혁 내용 XSS — 서버가 저장 시 정제한다 【보안 판정】
연혁 `content` 는 **관리자 입력이며 회사 소개 페이지를 통해 고객에게 노출된다**. 관리자 화면은 표 셀에 텍스트로만 렌더하지만(FS-017-EL-005.6), 고객 렌더가 HTML 로 해석하면 **저장형 XSS** 가 된다. 관리자 계정 하나가 탈취되면 전 방문자에게 스크립트가 배포된다.

BE-010 §7.2 와 동일 판정 — **서버가 저장 시**(EP-03·EP-04) 허용 태그 화이트리스트 밖의 마크업·`javascript:`/`data:` 스킴·이벤트 핸들러 속성을 제거한다. 계약은 '저장된 `content` 에 실행 가능한 스크립트가 없다'는 관측 동작만 정한다. 출력 시 이스케이프는 정제를 대체하지 않는다(두 층 모두 둔다). 정제는 300자 상한 검사 **이전**에 수행해 정제 후 길이가 상한을 넘지 않게 한다.

### 7.3 페이징·정렬·필터를 계약에 넣지 않는다 (지금은)
프론트 어댑터가 `fetchAll(signal)` 하나뿐이라 보낼 파라미터가 없다. 서버가 페이징을 강제하면 **프론트가 2페이지 이후를 영영 보지 못한다** — 계약이 화면을 깨뜨린다. 그래서 EP-01 은 전량 반환이다.

이것은 프론트의 미비(FS-017 §7 #2 — 페이지네이션 없음)를 계약이 따라간 것이지 옳은 종착지가 아니다. 연혁은 해마다 단조 증가하는 누적 데이터다. **IA-04/ERP-15 가 해소되어 프론트가 `page`/`size` 를 보내기 시작하면 EP-01 에 offset 페이징(`page` 1-base, `size` 기본 20·상한 100)과 `sort=year,desc` 를 추가한다** — 그때까지 서버는 전량을 내려주되 응답 크기를 관측한다.

### 7.4 동시 수정 — 지금은 '대상 없음'만 잡고, 낙관적 잠금은 유보한다 【정합 판정】
현재 계약이 잡는 충돌은 **한 갈래뿐**이다: 대상 id 가 사라진 경우(EP-04 → 409). 이것이 유령 저장을 없앤다.

**잡지 못하는 갈래**: 두 관리자가 **같은 연혁을 동시에 수정**하면 나중 요청이 먼저 요청을 조용히 덮는다(last-write-wins). `HistoryItem` 에 `updatedAt`·`version` 이 없어(§3) 서버가 '이 클라이언트가 본 판본'을 알 방법이 없기 때문이다.

**판정**: 지금은 도입하지 않는다. 연혁은 편집 빈도가 낮고(연 몇 건) 동시 편집 확률이 낮으며, 프론트는 이미 409/412 를 **입력 보존 다이얼로그로 수렴**시킬 준비가 되어 있다(`isConflict` 가 두 status 를 함께 본다). 즉 **도입 비용은 서버 쪽에만 있다**.

도입할 때의 계약은 이미 정해져 있다: `HistoryItem` 에 불투명 `version` 문자열을 추가 → EP-02 응답이 `ETag` 로 실어 보냄 → EP-04 가 `If-Match` 를 요구하고 불일치 시 **412** → 프론트는 코드 변경 없이 같은 충돌 다이얼로그를 띄운다. **연혁이 여러 편집자에게 열리면 이 승격을 먼저 한다.**

### 7.5 403 vs 404 — 공개 정보라도 은닉 규칙은 통일한다
연혁은 고객 웹사이트에 공개되므로 '개별 리소스의 존재가 비밀'이라는 BE-003 §3.2 의 전제는 이 도메인에 성립하지 않는다. 그럼에도 **읽기 권한 없는 주체에게 개별 404 은닉**을 유지한다: 도메인마다 은닉 규칙이 다르면 프론트가 404 를 '삭제됨'으로만 해석하는 현재 구현(FS-017-EL-020)이 도메인별로 갈리고, 백엔드는 두 벌의 인가 분기를 구현하게 된다. 일관성의 이득이 이 도메인의 은닉 이득보다 크다.

### 7.6 멱등키 — 프론트가 이미 만든다. 서버가 받기만 하면 된다
`useCrudForm` 은 **제출 시도 단위**로 `crypto.randomUUID()` 키를 ref 에 잡고(재시도가 같은 키를 재사용하도록 `mutationFn` **밖**에서 만든다) 성공 시 버린다. 주석(`useCrudForm.ts:110`)이 `TODO(backend): 이 키는 Idempotency-Key 헤더로 나간다` 로 연동 지점을 못 박았다.

**판정**: EP-03(등록)이 `Idempotency-Key` 헤더를 받아 최초 응답을 재생한다. 연혁 등록은 금전 거래가 아니라 중복 생성의 손해가 작지만, 키가 이미 프론트에 있고 어댑터 본문 한 줄이면 실리므로 **받지 않을 이유가 없다**. EP-04·EP-05 는 멱등(PUT)이거나 409 로 갈리므로 키가 필요 없다.

### 7.7 이미 삭제된 항목의 DELETE — 204(멱등)가 아니라 409 다
BE-010 §EP-07 은 DELETE 를 멱등으로 정의해 이미 삭제된 id 에 204 를 준다. **이 도메인은 반대로 간다** — 현재 어댑터(`crud.ts:82-84`)가 409 '이미 삭제된 항목입니다.' 를 던지고, 그것이 의도된 동작이다: 예전에는 `filter` 가 없는 id 를 조용히 통과시켜 **삭제되지 않았는데 삭제 성공**이었다.

**대가**: 일괄 삭제 부분 실패 후 재시도하면 성공분이 409 로 떨어져 실패 건수가 부풀려진다(FS-017 §7 #3). **이 대가는 프론트가 실패 id 를 받아 실패분만 재시도하도록 고치면 사라진다** — 계약을 멱등으로 되돌려 유령 삭제를 부활시키는 것은 잘못된 해법이다. §7.1 의 벌크 승격 시 `{ deleted: string[], failed: {id, code}[] }` 응답으로 함께 해소한다.

### 7.8 422 를 쓰지 않는다
연혁에는 상태 머신이 없다(발행/승인/보관 같은 lifecycle 이 없고 `HistoryItem` 에 status 필드가 없다). 형식은 맞으나 상태 때문에 거절할 조건이 존재하지 않으므로 **422 는 이 도메인에서 발생하지 않는다**. 필드 거절은 전부 400 `VALIDATION_FAILED` + `error.fields` 다.

> 단 프론트는 422 를 받으면 처리할 수 있다 — `useCrudForm.handleWriteError` 가 `isUnprocessable` + `violations` 를 RHF `setError` 로 각 입력에 꽂고 첫 필드로 포커스를 옮긴다(EXC-07). **서버가 필드 거절을 400 대신 422 로 주기로 하면 프론트 코드 변경 없이 인라인 에러가 붙는다.** 400 으로 주면 폼 레벨 배너(FS-017-EL-013)로만 떨어진다 — `error.fields` 가 있어도 프론트가 읽지 않는다. **필드 단위 거절은 422 로 주는 편이 화면에 유리하다**(§7.9 이관).
