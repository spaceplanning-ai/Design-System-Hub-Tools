# BE-015. 회사 정보 백엔드 기능 명세

## 1. 개요

| 항목 | 내용 |
|---|---|
| 대상 화면 | FS-015 회사 정보 (`/company/profile` — 단일 문서 편집 폼. 목록·상세·`:id` 라우트 없음) |
| 범위 | 회사 정보 문서 1건의 조회 · 저장(전체 치환) |
| 범위 밖 | **로고 파일 업로드** — 프론트에 심이 없다(§4 BE-015-EP-03 · §7.6 판정). **고객 회사 소개 렌더** — §7.1 XSS 판정만 다룬다. **문서 생성/삭제** — 회사당 1건이 항상 존재한다고 가정한다(§7.3) |
| 전제 | 인증은 세션 쿠키 기반. 모든 경로는 `/api` 프리픽스. 응답 본문은 `application/json; charset=utf-8` |
| 프론트 어댑터 | `apps/admin/src/pages/company/profile/data-source.ts` (`companyProfileStore` = `createDocumentStore<CompanyProfile>('profile', PROFILE_SEED)`) |
| 도메인 타입 | `apps/admin/src/pages/company/profile/types.ts` |
| 검증 정본 | `apps/admin/src/pages/company/profile/validation.ts` (`companyProfileSchema`) — 프론트 규칙. **서버는 이를 신뢰하지 않고 재검증한다** |

> **에러 봉투·권한 모델 상속**: BE-003 §2(공통 에러 봉투 · §2.1 공통 에러코드)·§3(인증 · 권한 모델 · 403 vs 404 은닉 · CSRF · 타임아웃)을 **그대로 상속한다**. 아래는 회사 정보 도메인 고유 차이만 기술한다.

## 2. 공통 (상속)

- **에러 봉투**: BE-003 §2. `error.code` · `error.message` · `error.fields` · `error.traceId`. 공통 에러코드 동일.
- **권한**: `admin` = 조회 + 저장. `operator` = **조회만**, 저장(`PUT`) → 403. 기업 관리 읽기 권한이 없는 인증 관리자 → **403**(컬렉션 성격 — §7.5).
- **CSRF**: 쓰기(`PUT`)에 `X-CSRF-Token` 헤더. 누락·불일치 → 403 `CSRF_TOKEN_INVALID`.
- **타임아웃**: 조회·쓰기 5초 → 504 `REQUEST_TIMEOUT`.
- **프론트 역할 분기 없음** (FS-015 §4.1): 화면이 `can(resource,'update')` 를 보지 않아 저장 버튼이 권한과 무관하게 렌더된다(FS-015 §7 #3). **권한 강제는 전적으로 서버 책임**이며, 프론트는 403 을 다른 실패와 같은 문구로 표시한다(FS-015-EL-004).

## 3. 데이터 계약 (types.ts 대조)

| 타입 | 필드 | 비고 |
|---|---|---|
| `CompanyProfile` | `companyName`(string) · `businessNumber`(string, `'XXX-XX-XXXXX'`) · `address`(string) · `ceoName`(string) · `contact`(string) · `logoUrl`(string) | **6필드 전부 `readonly` · 전부 non-nullable string.** 옵셔널 필드가 없다 — 로고 '없음' 은 `null` 이 아니라 **빈 문자열**로 표현된다 |

**제약 상수** (`types.ts`) — 프론트 `maxLength` 와 zod 스키마가 공유한다. 서버는 같은 값으로 재검증한다.

| 상수 | 값 | 적용 필드 |
|---|---|---|
| `COMPANY_NAME_MAX_LENGTH` | 100 | `companyName` |
| `ADDRESS_MAX_LENGTH` | 200 | `address` |
| `NAME_MAX_LENGTH` | 50 | `ceoName` |
| `CONTACT_MAX_LENGTH` | 40 | `contact` |
| (상수 없음) | 정규식 `^\d{3}-\d{2}-\d{5}$` | `businessNumber` — 자리수만. 체크섬 없음(§7.3) |
| (상한 없음) | `z.string()` | `logoUrl` — **형식 무제약**(§7.2) |

> **`CompanyProfile` 에 `id`·`updatedAt`·`version` 이 없다.** 이것이 §7.4(동시성) 판정의 출발점이다 — 프론트가 보낼 수 있는 낙관적 동시성 토큰이 타입에 존재하지 않는다.

### BE-015-EP-02 · 회사 정보 저장
| 항목 | 내용 |
|---|---|
| 근거 (FS) | FS-015-EL-014, EL-018, EL-019 |
| 근거 (심) | `data-source.ts:21` — `// TODO(backend): PUT /api/company/profile` |
| 메서드·경로 | `PUT /api/company/profile` |
| 권한 | **`admin` 만.** `operator` → 403 |
| 멱등성 | **멱등**(PUT 전체 치환) — 같은 바디 재요청은 같은 최종 상태. 그러나 이 멱등성이 중복 제출을 무해하게 만들지는 **않는다**(§7.4) |
| 페이징 | N/A |
| 레이트리밋 | 분당 20회 |

**바디**: `CompanyProfile` **전체**(§3). 부분 갱신(PATCH) 이 아니다 — 프론트가 폼 값 6개를 통째로 보낸다(`save.mutate({ input: values, ... })`).

**서버 검증** (프론트 스키마와 같은 규칙 + 프론트에 없는 규칙):

| 필드 | 규칙 | 프론트 대응 |
|---|---|---|
| `companyName` | trim 후 1–100자 | 동일(`requiredText`) |
| `businessNumber` | trim 후 `^\d{3}-\d{2}-\d{5}$` **+ 체크섬 검증**(§7.3) | 프론트는 정규식만 |
| `address` | trim 후 1–200자 | 동일 |
| `ceoName` | trim 후 1–50자 | 동일 |
| `contact` | trim 후 1–40자 **+ 전화번호 형식**(§7.3) | 프론트는 길이만 |
| `logoUrl` | 빈 문자열 **또는** `https://` 스킴 + 허용 호스트. **`blob:`·`data:`·`javascript:`·`file:` 은 400 거절**(§7.2) | 프론트는 무제약(`z.string()`) |

**응답 200/204**. 프론트 `save(input, signal): Promise<void>` — **응답 본문을 읽지 않는다.** 서버가 값을 정규화(trim 등)해 응답에 실어도 화면은 제출한 원본으로 `reset` 한다(FS-015-EL-019). 정규화 결과는 뒤이은 EP-01 재조회(`invalidateQueries`)로만 화면에 도달한다.

**에러**: 400 `VALIDATION_FAILED`(`error.fields`: `companyName`·`businessNumber`·`address`·`ceoName`·`contact`·`logoUrl`) · 401 · 403 `FORBIDDEN` · 403 `CSRF_TOKEN_INVALID` · 404 `PROFILE_NOT_FOUND` · 409 `CONFLICT`(§7.4) · 429 · 500 · 504. 프론트는 전부 '저장하지 못했습니다. 잠시 후 다시 시도해 주세요.'(FS-015-EL-004) 한 문구로 그린다 — `error.fields` 를 쓰지 않는다(§7.7 이관 #2).

### BE-015-EP-03 · 로고 이미지 업로드 — **심 없음 (미정)**

**엔드포인트를 정의하지 않는다.** `data-source.ts` 의 `// TODO(backend)` 주석은 **`GET`·`PUT /api/company/profile` 2건뿐**이며 업로드 심이 없다. 계약의 유일한 근거가 없으므로 경로·메서드·응답을 지어내지 않는다.

**현재 동작** (FS-015-EL-011.7): `ImageUploadField` 가 `URL.createObjectURL(file)` 로 만든 **`blob:` URL** 을 `logoUrl` 에 넣고, 그 문자열이 EP-02 바디로 나간다. 파일 바이트는 **어디에도 전송되지 않는다.** 저장은 성공하고 토스트도 뜨지만 그 URL 은 해당 브라우저 세션에서만 유효해, 새로고침·다른 기기·고객 화면에서 **로고가 깨진다.**

**필요한 결정** (백엔드 명세 · UI 기획 · 프론트 구현 합의 대상):

| # | 결정 사항 | 선택지 |
|---|---|---|
| 1 | 업로드 방식 | (a) `POST /api/company/logo` multipart → `{ url }` 응답 (b) presigned URL 발급 후 스토리지 직접 업로드 (c) EP-02 바디에 base64 인라인(권장하지 않음 — 5MB 상한이 요청 본문에 들어온다) |
| 2 | 업로드 시점 | (a) 파일 선택 즉시(프론트 `onChange` 에서) (b) 저장 제출 시 |
| 3 | 고아 파일 회수 | 업로드 후 저장하지 않고 이탈하면 파일이 남는다 — TTL 또는 참조 카운트 |
| 4 | 프론트 계약 | `ImageUploadField` 는 값 콜백(`onChange(url)`)만 노출한다. 업로드 진행률·취소 경로를 넣으려면 이 계약을 바꿔야 한다(P1 EXC-15) |

**임시 방어 (엔드포인트 결정 전에도 적용한다 — §7.2)**: EP-02 가 `logoUrl` 의 스킴을 화이트리스트로 검사해 `blob:`·`data:` 를 **400 `VALIDATION_FAILED`** 로 거절한다. 그래야 '조용히 저장되고 나중에 깨지는' 지금의 실패가 **저장 시점의 명시적 거절**로 바뀌어 운영자가 원인을 즉시 안다.

## 7. 핵심 판정

### 7.1 회사명·대표자명·주소의 저장형 XSS — 서버가 저장 시 정제한다 【보안 판정】

회사 정보 6필드는 **관리자 입력이며 고객 회사 소개 화면에 그대로 노출된다**(FS-015-EL-002 안내문이 그 계약을 명시한다: '저장하면 사용자 화면의 회사 소개에 반영됩니다'). 관리자 화면은 값을 `<input value>` 로만 그려 안전하지만, **고객 렌더가 HTML 로 해석하면 저장형 XSS 가 된다** — 회사명·대표자명·주소는 회사 소개의 헤더·푸터·구조화 데이터(JSON-LD)·`<title>` 등 **이스케이프가 누락되기 쉬운 자리**에 동시에 들어간다.

BE-009 §7.3 · BE-010 §7.2 와 동일 판정: **서버가 저장 시(EP-02) 정제**한다 — 허용 태그 화이트리스트 밖의 마크업, `javascript:` 스킴, 이벤트 핸들러 속성을 제거한다. 이 6필드는 **서식이 필요 없는 순수 텍스트**이므로 FAQ 답변보다 강하게 — **모든 마크업을 제거**하고 텍스트만 남긴다. 계약은 '저장된 값에 실행 가능한 스크립트가 없다' 는 관측 동작만 정한다(정제 라이브러리는 구현 재량).

### 7.2 `logoUrl` 스킴 화이트리스트 — 서버가 `blob:`/`data:` 를 거절한다 【보안 판정】

두 문제가 한 필드에 겹쳐 있다.

**(a) 보안**: `logoUrl` 은 고객 화면의 `<img src>` 로 나간다. 프론트 검증이 `z.string()`(`validation.ts:27`) 이라 **어떤 문자열도 통과한다** — `javascript:...` 도, 외부 추적 픽셀 URL 도 막히지 않는다. `shared/crud/validation.ts:22` 에 `optionalHttpUrl()` 헬퍼가 **이미 있고** `company/logo-list` 는 그것을 쓴다 — 이 화면만 무방비다.

**(b) 정합**: 업로드 심이 없어(§4 BE-015-EP-03) 저장되는 값이 `blob:` 인 경우가 **정상 경로에서 발생한다.** `profile.test.ts:71-75` 이 `blob:abc-123` 통과를 **테스트로 고정**하고 있어 프론트만 고쳐서는 회귀가 난다.

**판정**: EP-02 가 `logoUrl` 을 **스킴 화이트리스트로 검사**한다 — 빈 문자열 또는 `https://` + 허용 호스트만 통과, 그 외(`blob:`·`data:`·`javascript:`·`file:`·`http:`)는 **400 `VALIDATION_FAILED`**(`error.fields[].name = 'logoUrl'`). 이것은 업로드 엔드포인트가 정해지기 전에도 적용하는 **임시 방어**다: 지금은 로고가 조용히 저장됐다가 나중에 깨지지만, 이 검사가 있으면 **저장 시점에 명시적으로 거절**되어 운영자가 원인을 즉시 안다. 프론트가 `optionalHttpUrl()` 로 바꾸면 요청 전에 걸리지만(§7.7 이관 #1), **서버 검사가 정본**이다.

### 7.3 사업자등록번호 체크섬 · 연락처 형식 — 서버가 정본이다 【정합 판정】

프론트 정규식 `^\d{3}-\d{2}-\d{5}$`(`validation.ts:13`)는 **자리수만** 본다 — `000-00-00000` 같은 존재하지 않는 번호가 통과한다. 연락처(`contact`)는 **형식 검증이 아예 없다**(`requiredText` 로 공백·길이만) — `'abc'` 가 통과한다. 사업자등록번호는 세금계산서·견적서(ERP-10)의 근거 데이터이므로 자리수 통과만으로는 부족하다.

**판정**: 서버가 EP-02 에서 **국세청 체크섬 알고리즘**으로 사업자등록번호를 검증하고, 실패 시 400 `VALIDATION_FAILED`(`fields[].name='businessNumber'`, `code='CHECKSUM_INVALID'`). 연락처는 **한국 전화번호 형식**(지역번호·휴대폰·대표번호)으로 검증한다. 프론트는 이 거절을 필드에 매핑하지 못하고 배너 한 문구로만 표시하므로(§7.7 이관 #2), `error.message` 를 **그 자체로 이해 가능한 문구**로 준다.

> **문서 미생성(404)**: 회사당 1건이 '항상 존재' 한다고 가정하는 것은 **프론트의 가정**이다(FS-015 §1.1 — 등록 라우트가 없다). 서버가 문서를 아직 만들지 않았다면 EP-01 이 404 를 줄 수밖에 없고, 프론트는 그것을 '불러오지 못했습니다' 로 그린다(FS-015 §7 #5). **권장**: 서버가 **빈 문서(6필드 전부 빈 문자열)를 200 으로** 돌려 이 상태를 없앤다 — 프론트에 '아직 만들어지지 않음' 표면이 없으므로 404 를 만들지 않는 쪽이 계약과 맞다.

### 7.4 저장 동시성 — 프론트에 토큰이 없다. 마지막 쓰기가 이긴다 【판정 + 이관】

**낙관적 잠금을 도입할 수 없다.** `CompanyProfile` 타입에 `updatedAt`·`version`·`id` 가 **없고**(§3), `createDocumentStore` 의 `save(input, signal)` 시그니처에 토큰을 실을 자리가 없다. F2 가 `useCrudForm` 에 넣은 `If-Match` 배선(EXC-04)은 **이 화면에 적용되지 않는다** — 이 화면은 `useCrudForm` 을 우회하고 `useSaveDocument` 를 직접 쓴다.

따라서 현재 계약은 **마지막 쓰기 승리**다. 두 관리자가 같은 문서를 편집하면 나중 저장이 앞선 저장을 조용히 덮는다 — 프론트에 충돌 다이얼로그가 없어 서버가 409 를 올려도 같은 '저장하지 못했습니다' 배너가 뜨고, 재제출하면 그대로 덮어쓴다.

**여기에 중복 제출이 겹친다**(FS-015 §7 #2): 이 화면은 `submitLockRef`(동기 잠금)와 **제출 시도 단위 멱등키가 없다.** PUT 이 멱등이라 같은 바디의 중복 요청은 최종 상태를 바꾸지 않지만, **① 429 한도를 소모하고 ② 두 요청 사이에 다른 관리자의 저장이 끼면 그 값을 두 번 덮으며 ③ 서버 감사 로그에 유령 변경이 두 줄 남는다.** '멱등이니 괜찮다' 는 성립하지 않는다.

**이관** (§7.7 #1·#2): 낙관적 동시성을 도입하려면 **타입·어댑터 시그니처·`useSaveDocument` 를 함께 바꿔야 한다**(`CompanyProfile` 에 `updatedAt` 추가 → `save(input, { ifMatch }, signal)` → EP-02 가 `If-Match` 헤더 요구 → 불일치 시 412). 그 전까지 서버는 **감사 로그(누가·언제·무엇을)를 남겨** 덮어쓰기를 사후 추적 가능하게 한다.

### 7.5 403 vs 404 은닉 — 회사 정보는 은닉 대상이 아니다

BE-003 §3.2 의 두 원칙을 이 도메인에 적용한다. 회사 정보는 **컬렉션 성격**(식별자 없는 단일 문서, 열거할 개별 id 가 없음)이고 **그 존재가 비밀이 아니다** — 회사 소개는 고객에게 공개되는 정보다. 따라서 권한 부족은 **403 `FORBIDDEN`** 이며, 404 은닉을 적용하지 않는다. (회원 개별 리소스가 404 로 은닉되는 이유 — '존재 자체가 개인정보' — 가 여기엔 성립하지 않는다.)

라우트 read 권한(`page:/company/profile`)은 프론트 `RequirePermission` 이 403 화면으로 먼저 막지만(`AppShell.tsx:490`), 그것은 **UX 이지 보안 경계가 아니다**(위조된 localStorage 로 우회된다 — `RequirePermission.tsx` 헤더가 명시). 서버 403 이 정본이다.

### 7.6 로고 업로드 — 엔드포인트를 지어내지 않는다 【판정】

`// TODO(backend)` 주석이 계약의 유일한 근거이고, 업로드 심은 **없다**. 따라서 BE-015-EP-03 은 '심 없음(미정)' 으로 남긴다(§4). 이것은 문서화의 누락이 아니라 **구현의 누락을 그대로 기록한 것**이다 — 경로를 지어내면 백엔드가 프론트가 호출하지도 않는 엔드포인트를 만든다.

**연쇄**: 업로드가 정해지기 전까지 §7.2 의 스킴 검사가 **유일한 방어선**이고, 그 결과 '로고 업로드' 는 **저장 시 400 으로 거절되는 기능**이 된다. 즉 이 화면의 로고 기능은 백엔드 결정 전까지 **사실상 사용 불가**임을 계약이 명시한다. 이 상태를 운영자에게 숨기지 않는 것이 조용히 깨진 이미지를 배포하는 것보다 낫다.
