# FS-009. 공지사항 관리 (목록·상세·등록/수정)

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 목적 | 공지사항을 분류·상태·키워드로 좁혀 찾고, 개별 공지의 상세를 보거나 등록·수정·삭제(단건·일괄)한다 |
| 역할(주 사용자) | 관리자 (구현에 역할 분기 없음 — §4.1 참조) |
| 진입 경로 | 좌측 GNB > 콘텐츠 관리 > 공지사항 (`nav-config.ts`, `to: '/content/notices'`) |
| 포함 화면 | 목록 `/content/notices` · 상세 `/content/notices/:id` · 등록 `/content/notices/new` · 수정 `/content/notices/:id/edit` |
| 구현 경로 | `apps/admin/src/pages/content/notices/**` (`NoticesPage.tsx` · `NoticeDetailPage.tsx` · `NoticeFormPage.tsx` · `components/**`) |
| 대응 SCR | SCR-009 (미작성 — §7) |
| 공통 컴포넌트 | `shared/ui/{SelectionBar,useRowSelection,SelectField,ConfirmDialog,Pagination,SearchField,StatusBadge,RowActions,RowSelectCell,SelectAllHeaderCell,TextareaField,FormField}` · `shared/bulk.ts`(`settleAll`) |

### 1.1 이 화면군에 없는 기능

| 없는 것 | 이유 |
|---|---|
| 공지 본문의 리치 텍스트(HTML) 에디터 | 본문은 순수 텍스트(`textarea`)이며 상세는 `white-space: pre-wrap` 로 렌더한다. 서식·이미지 삽입이 없다 |
| 목록에서의 상태(게시/예약) 직접 전환 | 공지 목록에는 토글이 없다 — 상태 변경은 수정 폼에서만 한다. (팝업·배너 목록의 ON/OFF 토글과 다르다) |
| 정렬 열 토글 | 목록 정렬은 '고정 우선 → 최신순' 고정이다(서버 정렬). 열 헤더 클릭 정렬이 없다 |

| 세션 만료 | 세션 만료를 감지해 로그인으로 보내는 처리가 구현에 없다. 조회는 인라인 배너로, 쓰기는 각 요소 실패 처리로 떨어진다 — §7 미결 |
| 요청 타임아웃 | 프론트 타임아웃 상한이 구현에 없다. abort 는 조회 조건 변경·화면 이탈·다이얼로그 닫기 시 발생한다 — §7 미결(상한 미정) |
| 중복 제출 | 등록·수정(FS-009-EL-031)·단건 삭제(FS-009-EL-013)·일괄 삭제(FS-009-EL-014)는 요청 중 해당 버튼이 비활성되어 차단된다 |
| 실패 통지의 자리 | ① 조회 실패는 **인라인 배너**(FS-009-EL-011 / FS-009-EL-020) ② 저장 실패는 **폼 카드 안 배너**(FS-009-EL-023) ③ 삭제 실패는 **다이얼로그 안 배너**(FS-009-EL-013.1) ④ 삭제·저장 **성공**은 **토스트**(`useToast`, 자동 소멸) — 사용자가 지금 무엇을 보고 있는가로 자리를 정한다(`shared/ui/README.md`) |
| 동시 조회 | 목록/상세 조회는 동시에 1건만 유지된다(react-query). 조건이 바뀌면 진행 중 요청을 취소하므로 늦게 도착한 이전 응답이 최신 결과를 덮어쓰지 않는다 |
| 권한 없음 | 이 화면군 구현에는 역할 기반 분기가 없다. 요소가 역할에 따라 숨거나 비활성되지 않으며 전용 403 화면도 없다. 서버가 권한 부족을 응답하면 조회는 인라인 배너로, 쓰기는 각 요소 실패 처리로 떨어진다. 은닉 정책(403 vs 404)은 BE-009 가 확정한다 |
| 행 선택의 수명 | 선택(FS-009-EL-008.1 · FS-009-EL-008.2)은 화면 로컬 상태이며 서버로 전송되지 않는다. 분류·상태·키워드·페이지 중 하나라도 바뀌면 전부 해제된다 |

## 5. 서버 연동 지점

| 요소번호 | 이름 | 읽기/쓰기 | 필요 데이터 | 프론트 어댑터 (data-source.ts) | 비고 |
|---|---|---|---|---|---|
| FS-009-EL-001 / EL-002 | 분류·상태 필터 (+건수) | R | 필터가 반영된 공지 목록 + 분류별·상태별 전체 건수 | `fetchNotices(query, signal)` | 건수는 검색과 무관한 전체 기준 |
| FS-009-EL-003 | 분류 × 상태 AND 조합 | R | 두 조건 AND 만족 목록 + 총 건수 | `fetchNotices(query, signal)` | 서버 쿼리로 처리 |
| FS-009-EL-004 | 검색 입력 | R | 제목 부분 일치 목록 + 총 건수 | `fetchNotices(query, signal)` | 250ms 디바운스, 대소문자 무시 |
| FS-009-EL-006 | 조회 요약 텍스트 | R | 조건에 걸린 총 건수 | `fetchNotices(query, signal)` | 응답 `total` |
| FS-009-EL-008 | 공지 목록 표 | R | 공지 행 배열(id·제목·분류·상태·고정·작성자·게시일시·조회수) + 건수 + 총 건수 | `fetchNotices(query, signal)` | 목록 행은 본문(body)을 싣지 않는다 |
| FS-009-EL-008.10 / EL-013 / EL-021 | 삭제(단건·상세) | W | 공지 id | `deleteNotice(id, signal)` | 성공 후 목록 무효화 |
| FS-009-EL-007.2 / EL-014 | 일괄 삭제 | W | 선택된 공지 id 배열 | `deleteNotice(id, signal)` × N (`settleAll`) | 전원 성공일 때만 목록 무효화 |
| FS-009-EL-011 / EL-012 | 조회 실패 재시도 · 페이지네이션 | R | 마지막 조회 조건 재전송 · 요청 페이지 | `fetchNotices(query, signal)` | 페이지 크기 10 고정 |
| FS-009-EL-018 / EL-020 | 상세 조회 (+실패) | R | 공지 전문(목록 행 + body) | `fetchNotice(id, signal)` | 404 는 '공지를 찾을 수 없습니다' 로 변환 |
| FS-009-EL-031 (등록) | 공지 등록 | W | 제목·분류·상태·고정·게시일·본문 | `createNotice(input, signal)` | 성공 후 목록으로 이동 |
| FS-009-EL-031 (수정) | 공지 수정 | W | 공지 id + 위 입력 | `updateNotice(id, input, signal)` | 성공 후 상세로 이동 |
| FS-009-EL-032 | 수정 폼 초기 로드 | R | 공지 전문 | `fetchNotice(id, signal)` | 폼 `reset` 에 사용 |

> **현재 구현 상태 (백엔드 명세 참고)**: `data-source.ts` 의 쓰기 함수(`createNotice`·`updateNotice`·`deleteNotice`)는 지연 후 resolve 만 하고 **아무것도 저장하지 않는다**(백엔드 미구현, `// TODO(backend)` 주석이 연동 지점). 화면은 이 resolve 를 성공으로 간주한다. 위 표는 백엔드 연결 후의 의도된 동작이다.
