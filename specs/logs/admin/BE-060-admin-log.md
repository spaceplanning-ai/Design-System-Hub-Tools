# BE-060. 관리자 로그 (감사 로그) 백엔드 기능 명세

## 1. 개요

| 항목 | 내용 |
|---|---|
| 대상 화면 | FS-060 관리자 로그 (`/logs/admin`) |
| 범위 | 관리자 행위 로그 조회(결과·액션·기간·키워드 필터 · 정렬 · 페이징 · 기간 안 건수 · **요청 페이로드**), 내보내기 조회. 그리고 이 도메인의 핵심 — **감사 불변성과 페이로드 민감정보의 서버 책임**(§3·§7) |
| 범위 밖 | **로그의 생성** — 관리자 행위는 각 도메인이 처리 중에 append-only 로 남긴다. 이 화면은 그 기록을 **읽기만** 한다. **로그의 삭제·수정** — 감사 불변성 원칙상 그런 엔드포인트를 만들지 않는다(§7.5). **CSV 직렬화** — 프론트 `toCsv` 가 브라우저에서 한다(§7.4) |
| 전제 | 인증은 세션 쿠키 기반. 모든 경로는 `/api` 프리픽스. **읽기 전용이므로 CSRF 대상 쓰기가 없다.** 응답 본문은 `application/json; charset=utf-8`. 시각은 **오프셋을 가진 ISO 8601** |
| 프론트 어댑터 | `apps/admin/src/pages/logs/admin/data-source.ts` |
| 도메인 타입 | `apps/admin/src/pages/logs/admin/types.ts` (`AdminLogEntry` · `AdminAction` · `AdminOutcome`) + 섹션 공용 `logs/types.ts` (`LogQuery` · `LogResult` · `PageSize`) |
| 검증 정본 | `logs/validation.ts` (zod/mini — 기간 '직접 지정'만. 나머지 조건은 버튼·URL 화이트리스트라 자유 입력이 아니다) |
| 형제 명세 | BE-061(회원 활동) · BE-062(API) · BE-063(오류) — **같은 셸·같은 엔진·같은 계약 골격**을 공유한다. 화면마다 다른 것은 축의 어휘·컬럼·보존기간뿐이다 |

> **에러 봉투·권한 모델 상속**: BE-003 §2·§3 을 그대로 상속한다. 아래 §2 는 재정의가 아니라 **이 도메인의 코드 목록**이다.
>
> **감사 로그는 이 앱이 바꾸지 않는다 — 오직 읽는다.** 이 문서에 쓰기 엔드포인트가 없는 것은 누락이 아니라 **설계**다(§7.5). 정본 선례는 BE-008(로그인 이력)이며 그 불변성 패턴을 그대로 잇는다.

### 1.1 코드 대조 근거표

이 문서가 정의하는 것과 **정의하지 않는 것**의 근거는 추정이 아니라 코드다.

| 사실 | 근거 |
|---|---|
| 심이 **정확히 2개**다 | `admin/data-source.ts:55` `// TODO(backend): GET /api/logs/admin?from=&to=&outcome=&action=&keyword=&sort=&dir=&page=&size=` · `:66` `// TODO(backend): GET /api/logs/admin/export?from=&to=&outcome=&action=&keyword=&sort=&dir=` |
| 심이 응답 의미까지 못 박는다 | `data-source.ts:56-57` — '응답의 `total` 은 필터·검색까지 적용한 뒤의 건수여야 한다(페이지네이션의 모수). `axisCounts` 는 **기간 안에서만** 센 값이다 — 축·검색과 무관하다' |
| **쓰기 함수가 0건이다** | `admin/data-source.ts` 의 export = `adminLogSpec` · `fetchAdminLogs` · `fetchAdminLogsForExport` · `toCsv`. `logs.test.ts:59-72` 가 이 목록을 **전수 단언**하고 `/^(delete|remove|update|patch|save|create|purge|clear|resolve|archive)/i` 로 시작하는 export 가 0건임을 단언한다 |
| 쓰기 라우트가 0건이다 | `App.tsx:328` 주석 — '쓰기 라우트(등록/수정)는 **존재하지 않는다**: 감사 기록은 불변이다'. `:329` 가 `/logs/admin` 하나만 등록 |
| 필터·정렬·페이징이 **지금은** 클라이언트 파생이다 | `query-engine.ts:142-156` `runLogQuery` 가 픽스처 배열을 훑는다. `:10-11` — '백엔드가 붙으면 이 파일은 통째로 서버의 WHERE · ORDER BY · LIMIT 로 대체된다' |
| 페이로드가 **날것**으로 온다 | `admin/types.ts:59-63` — '요청 본문 또는 변경 전/후. **날것 그대로** 담는다 — 마스킹은 화면에 그리는 순간 masking.ts 가 한다' |
| **보존기간·마스킹·열람 감사에 심이 없다** | `admin/types.ts:100-103`(`ADMIN_LOG_RETENTION`)은 클라이언트 상수다. `masking.ts` 는 프론트 순수 함수다. 열람을 기록하는 호출이 어디에도 없다 → **전부 '심 없음(미정)'**(§7.3 · §7.1 · §7.6) |

**결론**: 이 문서는 **엔드포인트 2개만** 정의한다. 보존기간 강제·서버 마스킹·열람 감사·로그 생성은 계약상 있어야 할 법해 보여도 **심이 없으므로 '심 없음(미정)'** 으로 남기고 §7 에서 판정한다.

## 2. 공통 (상속)

- **에러 봉투**: BE-003 §2 를 그대로 상속한다(`{ error: { code, message, fields?, traceId } }`). 이 화면은 조회 실패를 status 별로 가르지 않고 한 배너로 표시하므로(FS-060-EL-015) 봉투의 `code` 를 소비하지 않는다 — 그래도 `traceId` 는 서버가 실어야 한다(§7.6 #4: 화면이 아직 표시하지 않는다).
- **권한**: BE-003 §3 을 상속한다. 이 도메인의 리소스는 `page:/logs/admin` 이고 액션은 **`read` 와 `export` 둘**이다(§3.1).
- **CSRF**: **해당 없음** — 쓰기 요청이 없다.
- **프론트 역할 분기 없음**(FS-060 §4.1). 권한 강제는 전적으로 서버의 책임이다 — 프론트 가드는 UX 층이다(`RequirePermission.tsx:8-11`: '위조된 localStorage 로 이 가드는 우회된다').

### 2.1 이 도메인의 에러코드

| HTTP | 에러코드 | 발생 조건 | 프론트 처리 |
|---|---|---|---|
| 400 | `VALIDATION_FAILED` | `outcome`·`action`·`sort`·`dir` enum 밖, `size` 가 20\|50\|100 밖, `page` < 1, `keyword` 길이 초과 | 조회 실패 배너(프론트 선검증·화이트리스트가 대부분 막는다 — §7.7) |
| 400 | `INVALID_DATE_RANGE` | `from`·`to` 형식·미래·역전·90일 초과 | 프론트가 선검증하므로 정상 흐름에서 미발생(§7.7) |
| 401 | `UNAUTHENTICATED` | 세션 없음·만료 | 쿼리 계층이 재인증 경로로 |
| 403 | `FORBIDDEN` | 관리자 로그 `read`(또는 `export`) 권한 부족 (§3.2) | 조회는 FS-060-EL-015, 내보내기는 FS-060-EL-017 |
| 429 | `RATE_LIMITED` | 레이트리밋 초과. `Retry-After` 동반 | 재시도 안내 |
| 500 | `INTERNAL_ERROR` | 서버 내부 오류 | 일반 문구 + `traceId` |
| 504 | `REQUEST_TIMEOUT` | 서버 처리 상한 초과 | 500 과 동일 |

## 3. 인증 · 권한 모델 · 감사 원칙

### 3.1 역할

| 역할 | 설명 | 비고 |
|---|---|---|
| `page:/logs/admin` **read** 보유 | 로그를 조회한다 — **행·페이로드 전부** | 프론트 가드가 `<RequirePermission>` 으로 확인하고 **없으면 조회 요청 자체를 보내지 않는다**(`LogListShell.test.tsx:136-142`) |
| `page:/logs/admin` **export** 보유 | 내보내기 버튼이 보인다 | `useRouteCan('export')`(`LogListShell.tsx:115`) → `LogToolbar.tsx:117` 이 버튼을 렌더하지 않는다 |
| 위 권한 없음 | 403 화면 / 버튼 부재 | 서버도 같은 판정을 독립적으로 강제해야 한다 |

> **`read` 와 `export` 를 서버가 갈라야 한다.** 프론트는 이미 갈라 놨다 — 내보내기 버튼이 `export` 권한에 걸린다. 그런데 **버튼을 숨기는 것은 보안이 아니다**: `GET /api/logs/admin/export` 를 직접 호출하면 그만이다. 서버가 EP-02 에서 `export` 를 독립적으로 확인하지 않으면 **프론트의 그 분기는 장식이 된다.** 반대로 서버가 `read` 만 보면 '조회는 되지만 반출은 안 되는 역할'이라는 개념 자체가 성립하지 않는다 — 그리고 그 역할은 **감사에서 가장 중요한 구분**이다(반출은 조회가 아니라 유출 경로다. `admin/fixtures.ts:184` 가 그 사건을 '조회'가 아니라 '반출'로 부른다).

### 3.2 403 vs 404 은닉

| 리소스 | 권한 부족 시 응답 | 근거 |
|---|---|---|
| `GET /api/logs/admin` (컬렉션) | **403** `FORBIDDEN` | 로그 컬렉션의 존재는 비밀이 아니다. 열거할 개별 id 가 노출되지 않는다 |
| `GET /api/logs/admin/export` (컬렉션) | **403** `FORBIDDEN` | 위와 동일 |

**개별 로그 조회 엔드포인트가 없으므로 404 은닉 문제가 이 도메인에 성립하지 않는다** — 상세는 목록 응답의 `payload` 로 그려진다(§7.2). 존재하지 않는 엔드포인트에 은닉 정책을 발명하지 않는다.

### 3.3 감사 불변성 (append-only)

- 로그는 **추가만 되고 수정·삭제되지 않는다.** 이 계약에 로그를 바꾸는 엔드포인트가 없고(§7.5), 저장소도 append-only 로 다뤄야 한다 — **권한 있는 최상위 관리자도 개별 로그를 지울 수 없다.** FS-060-EL-005 안내문('관리자도 수정하거나 삭제할 수 없으며')이 이 계약을 사용자에게 고지한다.
- **로그 열람 자체도 감사 대상이다** — 누가 언제 어떤 조건으로 조회·내보냈는지 별도로 남긴다(§7.6 #1 — 심 없음).

### 3.4 타임아웃 상한

| 계열 | 서버 처리 상한 | 초과 시 | 프론트 상한(권고) |
|---|---|---|---|
| 조회 | 5초 | 504 `REQUEST_TIMEOUT` | 10초 (**현재 상한 없음** — §7.6 #5) |
| 내보내기 조회 | 30초 | 504 `REQUEST_TIMEOUT` | 60초 (**현재 상한 없음**. 단 사용자가 취소할 수는 있다 — FS-060-EL-009.1) |

## 4. 엔드포인트 명세

### BE-060-EP-01 · 관리자 로그 조회

| 항목 | 내용 |
|---|---|
| 근거 (FS) | FS-060-EL-001, EL-002, EL-003, EL-003.1, EL-003.2, EL-007, EL-008, EL-010, EL-010.1, EL-011, EL-011.2, EL-014, EL-015, EL-015.1, EL-016, EL-016.4 |
| 메서드 · 경로 | `GET /api/logs/admin` — 근거: `data-source.ts:55` |
| 권한 | `page:/logs/admin` **read** |
| 멱등성 | 멱등 (GET). 재시도 안전 |
| 페이징 | offset 방식. `page` 1-base, `size` ∈ **{20, 50, 100}** (프론트 `PAGE_SIZE_OPTIONS` — `logs/types.ts:184`) |
| 레이트리밋 | 관리자당 분당 120회 |

**요청 — 쿼리** (전부 `data-source.ts:55` 심에 열거된 것뿐이다)

| 이름 | 타입 | 필수 | 제약 | 설명 |
|---|---|---|---|---|
| `from` | string | O | `YYYY-MM-DD`. 실재 날짜. **KST 달력일** | 조회 구간 시작(양 끝 포함) |
| `to` | string | O | `YYYY-MM-DD`. `to >= from`, `to <= 오늘(KST)`, 구간 ≤ 90일 | 조회 구간 끝(양 끝 포함) |
| `outcome` | string | X | `all` \| `success` \| `failure` (기본 `all`) | 결과 필터 |
| `action` | string | X | `all` \| `login` \| `logout` \| `create` \| `update` \| `delete` \| `export` \| `permission` (기본 `all`) | 액션 필터 |
| `keyword` | string | X | 앞뒤 공백 제거. **길이 상한 미정** (§7.6 #3) | 행위자 계정·이름·대상 유형·대상·IP 부분 일치(대소문자 무시) |
| `sort` | string | X | `occurredAt` \| `actor` \| `action` \| `target` \| `outcome` \| `ip` (기본 `occurredAt`) | 정렬 키 |
| `dir` | string | X | `asc` \| `desc` (기본 `desc`) | 정렬 방향 |
| `page` | number | X | 정수 ≥ 1 (기본 1) | 페이지 번호 |
| `size` | number | X | **20 \| 50 \| 100** (기본 20) | 페이지 크기 |

- `outcome`·`action`·`keyword`·기간은 **AND** 로 결합한다(`query-engine.ts:58-75`).
- **정렬이 페이지 자르기보다 먼저다**(`query-engine.ts:139-141`). 동점은 `id` 로 끊어 순서를 확정한다(`:132`) — 그러지 않으면 같은 초에 일어난 두 행이 페이지 경계에서 두 번 나오거나 사라진다.
- 정렬 값의 의미는 **프론트 표시값과 같아야 한다**: `action`·`outcome` 은 **한국어 라벨**로 정렬한다(`data-source.ts:48,50` — `ADMIN_ACTION_LABEL[entry.action]`). 서버가 enum 원문(`login`·`permission`)으로 정렬하면 **화면의 정렬이 사용자가 보는 글자 순서와 어긋난다**(§7.7).
- 한국어 비교는 로케일 사전순이다(`query-engine.ts:109` — `localeCompare(…, 'ko')`).

### 6.1 어댑터 본문 요구사항

1. **`axisCounts` 를 서버가 계산해 내려준다.** 지금은 `countAxes`(`query-engine.ts:86-102`)가 클라이언트에서 센다. 서버가 페이징된 응답만 주면 **좌측 배지는 '이 페이지의 건수'가 되어 완전히 틀린다** — 배지의 모수는 **기간 안 전체**다(`data-source.ts:57` 이 명시). 이것이 이 어댑터의 가장 깨지기 쉬운 계약이다.
2. **`total` 은 필터·검색까지 적용한 뒤의 건수다** — `entries.length` 가 아니다(`data-source.ts:56`). 페이지네이션과 범위 요약의 모수다.
3. **정렬을 서버가 하되 표시값 기준으로 한다** — `action`·`outcome` 은 한국어 라벨 순이다(§4 EP-01). 그리고 **동점은 `id` 로 끊어야 한다**(`query-engine.ts:117-118,132`) — 그러지 않으면 페이지 경계의 행이 두 페이지에 나타나거나 어느 페이지에도 나타나지 않는다.
4. **`occurredAtIso` 에 오프셋을 반드시 싣는다.** 순진한 로컬 문자열을 보내면 `seoulTimeParts` 가 브라우저 로컬로 해석해 **타임존 버그가 조용히 숨는다**(`fixture-lib.ts:14-19`).
5. **응답이 이미 `sort`·`dir` 순이어야 한다** — 프론트는 재정렬하지 않는다(백엔드 연동 후 `query-engine` 이 사라지므로).

## 7. 핵심 판정

### 7.1 【보안 판정】 마스킹은 **표시 통제**이지 보안 통제가 아니다 — 원본은 이미 브라우저에 도달해 있다

**이것이 이 섹션에서 가장 중요한 판정이다.**

**사실 (코드 대조)**:

| 사실 | 근거 |
|---|---|
| 페이로드는 **날것으로** 서버에서 온다 | `admin/types.ts:59-63` — '요청 본문 또는 변경 전/후. **날것 그대로** 담는다' |
| 마스킹은 **클라이언트 순수 함수**다 | `masking.ts:168-170` `maskPayload` · `:176-182` `formatMaskedPayload` — React 도 네트워크도 모른다 |
| 마스킹은 **렌더 순간**에 일어난다 | `LogPayloadDialog.tsx:96` `<pre …>{formatMaskedPayload(detail.payload)}</pre>` |
| 원본은 **바뀌지 않는다** | `masking.ts:165-166` — '원본은 바꾸지 않는다(새 객체를 만든다) — 마스킹은 *표시*의 일이지 *기록*의 일이 아니다' |
| **서버 마스킹 심이 없다** | BE-060-EP-05 — `data-source.ts` 의 심 2개는 쿼리만 열거한다 |

**따라서**: `password: 'Sup3rSecret!2026'` 은 **네트워크를 타고 브라우저 메모리에 도착한 뒤** 화면에 그려지기 직전에만 가려진다. 그 값은 다음 어디에나 남아 있다 — 응답 본문, react-query 캐시(`logs.list` 키 아래 `entries[].payload`), React 컴포넌트 props, 브라우저 devtools 의 Network 탭. **`maskPayload` 를 우회하는 데 필요한 것은 devtools 를 여는 일뿐이다.**

**그러므로 이 마스킹은 다음을 막는다**: 어깨너머로 보는 사람, 화면 캡처, 화면 공유, 실수로 붙여넣는 스크린샷 — **그리고 그것으로 충분히 가치가 있다.** 이 화면을 여는 모든 운영자가 모든 회원의 비밀번호를 **눈으로 보는** 것을 막는 것이 목적이고(`masking.ts:7-8`), 그 목적은 달성된다.

**그러나 다음은 막지 못한다**: 권한 있는 운영자가 devtools 로 원본을 꺼내는 것. **즉 마스킹은 신뢰 경계가 아니다.** `masking.ts:18-22` 가 이 사실을 **이미 정직하게 적어 두었다** — '클라이언트 마스킹은 **두 번째 방어선**이다. 진짜 방어선은 애초에 비밀을 로그에 쓰지 않는 것이고 그 다음이 서버가 저장 시점에 가리는 것이다.' **이 문서는 그 자기 판정에 동의하며 계약으로 승격한다.**

**계약이 확정하는 것**:

1. **서버가 저장 시점에 가린다 — 그것이 1차 방어선이다.** 비밀번호·토큰·자격증명은 **애초에 로그에 기록되지 않아야** 하고(프레임워크가 요청 본문을 통째로 찍는 것이 문제의 근원이다 — `admin/fixtures.ts:155-158`), 기록됐다면 **저장 시점에** 가려야 한다. `GET /api/logs/admin` 응답의 `payload` 에 **평문 비밀번호가 들어 있어서는 안 된다.**
2. **클라이언트 마스킹은 사라지지 않고 이중으로 남는다**(`masking.ts:21-22`). 서버가 새 키로 민감값을 싣기 시작하면 그 사이의 유일한 방어선이 그것이다.
3. **⚠ 키 기반 마스킹의 구멍을 계약이 인정한다.** `masking.ts:49-68` 의 `KEY_RULES` 는 **키 이름 정규식**이고 값을 보지 않는다. `masking.ts:47` 이 '모르는 키는 가리지 않는다'를 의도로 밝힌다(다 가리면 감사가 불가능해지므로 — 근거 있는 선택이다). **그 결과 규칙에 없는 키의 민감값은 그대로 렌더된다 — 그리고 이것은 이론이 아니라 형제 화면에서 오늘 발현하고 있다**:
   - FS-061 `member-activity/fixtures.ts:91` — `shipping.address: '서울특별시 중구 세종대로 000'` → **키 `address` 가 어느 규칙에도 걸리지 않아 배송지 전체가 그대로 그려진다.** 같은 객체의 `phone` 은 가려지는데 `address` 는 안 가려진다.
   - FS-063 `errors/fixtures.ts:143` — `context.recipient: 'user1099@example.com'` → **키 `recipient` 가 `/e[-_]?mail|이메일/` 에 걸리지 않아 회원 이메일이 그대로 그려진다.** 같은 값이 `email` 키였다면 가려졌다.
   - `masking.ts:56-61` 이 `^card$` 를 넣은 사연('예전 규칙은 cardNumber 는 잡았지만 감싼 card 는 못 잡았고 카드번호가 그대로 화면에 찍혔다')은 **이 구멍이 이미 한 번 실제로 뚫렸었다는 증거**다. 규칙을 한 줄씩 늘리는 방식은 다음 키에서 또 뚫린다.
   → **판정: 키 기반 마스킹은 값 기반 탐지로 보강되거나(카드번호·JWT·이메일 패턴을 *값*에서 찾기), 서버가 화이트리스트로 페이로드를 정제해야 한다.** 지금 구조는 '아는 이름만 가린다'이고 감사 로그의 페이로드는 정의상 **모르는 모양**이다.
4. **화면의 마스킹 안내가 지키지 못할 약속을 한다.** FS-060-EL-016.3 은 '민감한 값은 **자동으로** 가려집니다'라고 단언하는데, 실제로는 '**우리가 아는 이름의** 값은 가려집니다'다. 운영자가 그 문구를 믿고 화면을 공유하면 위 #3 의 값들이 함께 나간다. **문구를 정확하게 하거나 규칙을 문구에 맞추거나 둘 중 하나여야 한다.**

**이관**: **A63(서버 마스킹·저장 정제) · A11(값 기반 탐지·문구) · A01(로그에 무엇을 남길 것인가 — 정책)** — §7.6 #2.

### 7.2 【보안 판정】 목록 응답이 **모든 행의 페이로드를 싣는다** — 그리고 상세에 별도 권한이 없다

**상세 다이얼로그는 조회를 하지 않는다.** `LogPayloadDialog` 는 `spec.detailOf(detailEntry)` 의 결과만 받고(`LogListShell.tsx:297-302`), `detailOf`(`AdminLogPage.tsx:79-105`)는 **이미 손에 있는 행 객체**의 `entry.payload` 를 그대로 넘긴다. 즉 **`GET /api/logs/admin` 한 번이 그 페이지 모든 행의 요청 본문 전체를 내려준다.**

세 가지가 따라온다.

**(a) 보이지 않는 데이터가 이미 브라우저에 있다.** 운영자가 상세를 **한 건도 열지 않아도** 그 페이지 20~100건의 페이로드가 전부 도착해 있다. 화면에 그려지는 것은 6개 컬럼뿐인데 전송되는 것은 그 전부다. `page`·`size`·필터를 바꿀 때마다 또 온다. **'무엇을 보았는가'와 '무엇을 받았는가'가 어긋난다** — 그리고 §7.1 대로 마스킹은 표시 층에만 있으므로 **받은 것은 가려지지 않은 원본이다.**

**(b) 페이로드에 별도 권한이 없다.** `page:/logs/admin` 의 `read` 하나가 '목록을 본다'와 '요청 본문을 본다'를 **동시에** 연다. 그런데 이 둘은 민감도가 다르다 — 목록은 '누가 무엇을 했다'이고 페이로드는 '무엇을 보냈다'(비밀번호·토큰 포함)이다. **'로그는 봐도 되지만 요청 본문은 안 되는 역할'을 만들 수 없다.**

**(c) 내보내기가 이 비대칭을 드러낸다.** CSV 에는 **페이로드 열이 없다**(`data-source.ts:81-83`: '페이로드에는 비밀번호·토큰이 들어 있고, CSV 는 마스킹을 거치지 않은 채 메일로 오간다 — 파일로 반출되는 순간 통제할 수 없는 것을 파일에 싣지 않는다'. `logs.test.ts:555-559` 가 고정). **판단은 옳다.** 그러나 **같은 데이터가 EP-02 응답 배열에는 실려 온다** — `readonly AdminLogEntry[]` 는 `payload` 를 포함하는 타입이고, `toCsv` 가 그 열을 **버릴 뿐**이다. 즉 조건 전체의 페이로드가 브라우저에 도착한 뒤 CSV 에서만 빠진다. **파일에 안 싣기로 한 것을 네트워크로는 전량 내려받는다.**

**판정 — 계약이 확정하는 것**:

1. **EP-01 의 `payload` 를 목록 응답에서 뺄지 여부는 프론트 구조가 정한다.** 지금 구조(상세 = 목록 행 객체)에서는 뺄 수 없다 — 빼면 상세가 빈다. **뺀다면 개별 조회 엔드포인트(`GET /api/logs/admin/:id`)가 선행돼야 하고, 그때 비로소 (a)(b)가 함께 풀린다**: 페이로드는 실제로 연 건에만 흐르고, 그 엔드포인트에 별도 권한(예: `page:/logs/admin` 의 `read` 와 구분되는 payload 권한)을 걸 수 있으며, **열람 감사가 '이 운영자가 이 건의 본문을 봤다'로 좁혀져 감사 가치가 생긴다**(§7.6 #1). **심이 없으므로 그 엔드포인트를 여기서 정의하지 않는다** — 결정이 나면 FS 의 상세 표면과 함께 설계한다.
2. **EP-02(내보내기)는 `payload` 를 응답에서 뺄 수 있다 — 그리고 빼야 한다.** CSV 가 그것을 쓰지 않는다는 것이 **코드로 이미 확정**돼 있고(`CSV_COLUMNS` 10열에 페이로드가 없다), 뺀다고 화면이 깨지지 않는다. 이것은 §7.2(a)의 '보이지 않는 데이터'를 **구조 변경 없이** 없애는 유일한 지점이다. **다만 `readonly AdminLogEntry[]` 라는 반환 타입이 `payload` 를 필수 필드로 갖고 있어**(`admin/types.ts:63`) 서버가 그것을 빼면 타입이 어긋난다 — 내보내기 전용 projection 타입이 필요하다. **그 타입은 아직 없다** — A11 이관(§7.6 #3).
3. **어느 경우에도 `read` 없이는 아무것도 주지 않는다.** 프론트 가드는 UX 층이고(`RequirePermission.tsx:8-11`) 실제 차단은 서버다.

**이관**: **A63 · A11 (보안)** — §7.6 #3.

### 7.3 보존기간(3년)은 화면이 약속하고 아무도 이행하지 않는다 — 심 없음 (미정)

| 값 | 어디서 | 심 |
|---|---|---|
| 보존기간 '3년' | `admin/types.ts:100-103` (`ADMIN_LOG_RETENTION`) — **클라이언트 상수** | **없음** |
| '보존기간이 지나면 자동 폐기됩니다' | 같은 곳 — 화면에 **단언으로** 표시된다(`LogFilterPanel.tsx:169-170`) | **없음** |

**BE-060-EP-04 는 '심 없음(미정)'이다.** 지어내지 않는다.

**그러나 이 부재의 결과는 계약 문제다.** 화면이 운영자에게 **정책을 단언**하는데(FS-060-EL-004) 그 정책의 이행 주체가 없다:

- 실제로 3년보다 **짧게** 지워지면 → 운영자는 화면을 믿고 **없는 기록을 찾아 헤맨다.** 이것이 `logs/types.ts:265-268` 이 보존기간을 화면에 적기로 한 이유('없는 것'과 '폐기된 것'은 다른 사건이다)인데, 표시된 기간이 틀리면 그 장치가 **거꾸로 작동한다.**
- 실제로 3년보다 **길게** 남으면 → 개인정보 최소보관 원칙에 어긋난다. 이 로그에는 행위자 계정·IP·대상 회원 이메일·요청 본문이 있다.
- 4화면의 기간이 전부 다르다(관리자 3년 · 회원 활동 1년 · 오류 180일 · API 90일). **그 차이에는 근거가 적혀 있고**(각 `types.ts` 의 `basis`) 근거가 서로 다른 종류다 — 관리자 로그는 '내부 통제', 회원 활동은 '개인정보 최소 보관', API 는 '부피', 오류는 '재발 추적'. **즉 이것은 기술 상수가 아니라 도메인 정책이다.**

**판정**: **보존기간은 서버가 소유하고 강제해야 한다.** 그리고 그때 화면의 상수는 **서버가 내려준 값을 표시**하도록 바뀌어야 한다 — 두 곳에 적히는 순간 둘은 갈라지고, 갈라지면 화면이 거짓말한다. **심이 없으므로 여기서 확정하지 않고 이관한다**(§7.6 #6 · FS-060 §7 #4).

### 7.4 내보내기 — 상한이 없다. CSV 는 프론트에 남긴다

- **CSV 직렬화는 프론트에 남긴다.** 서버가 CSV 를 내려주면 `toCsv` 가 사라지고 Content-Type·다운로드 코드가 바뀐다. 열 구성이 이미 FS-060-EL-009 에 고정돼 있고 프론트에 구현돼 있어(`data-source.ts:85-96`) 서버로 옮길 이득이 없다. BE-008 §7.4 와 같은 판정이다.
- **⚠ 행 수 상한이 없다 — BE-008 과 갈리는 지점이다.** BE-008 §7.4 는 `422 EXPORT_TOO_LARGE` 를 두어 대량을 거절하기로 했다('조용히 잘린 감사 CSV 는 잘못된 판단으로 이어진다'). **이 도메인에는 그 심이 없고 프론트에 그 코드를 아는 분기도 없다** — 내보내기 실패는 전부 '내보내기에 실패했습니다. 잠시 후 다시 시도해 주세요.' 한 문구로 붕괴한다(`LogListShell.tsx:194`). 즉 **지금 422 를 내려도 운영자는 '기간을 좁히라'는 안내를 받지 못하고 그냥 재시도한다.**
- 90일 상한(`types.ts:89`)이 규모를 **1차로만** 제한한다. 관리자 로그는 부피가 작은 편이지만(형제 화면 중 API 로그가 '관리자 로그의 수천 배' — `api/types.ts:101`) 상한의 근거가 '부피가 작을 것이다'라는 **추정**이다.
- **도입 시 조건**(지금 정하지 않고 조건만 남긴다): ① 잘라서 일부만 주지 않는다 — 조용히 잘린 감사 CSV 는 잘못된 판단으로 이어진다. ② 거절한다면 프론트가 **그 사유를 구분해 표시**할 수 있어야 한다(지금은 못 한다 — 위). ③ 상한 값은 4화면이 공유할지 화면마다 다를지 정해야 한다(보존기간처럼 부피가 다르다).

**이관**: A63 · A01 (§7.6 #7).

### 7.5 쓰기 엔드포인트가 없다 — 감사 불변성의 계약

이 도메인에 로그를 **생성·수정·삭제하는 엔드포인트를 만들지 않는다.**

근거: **지울 수 있는 감사 로그는 감사 로그가 아니다** — 침입자가 가장 먼저 지우는 것이 자기 흔적이다(`logs/types.ts:8-10`). 로그의 **생성**은 각 도메인이 자기 처리 중에 append-only 로 남기고(BE-060-EP-03 — 이 화면의 범위 밖), 이 화면은 **읽기만** 한다.

**삭제·수정 API 는 최상위 관리자에게도 제공하지 않는다.** FS-060-EL-005 안내문('관리자도 수정하거나 삭제할 수 없으며, 조회와 내보내기만 제공합니다')이 이 계약을 사용자에게 고지한다. 저장소도 append-only 로 다뤄 물리 삭제를 막는다 — **유일한 예외가 보존기간 만료 폐기이고, 그것은 정책에 따른 일괄 폐기이지 개별 삭제가 아니다**(§7.3).

**이 계약은 프론트에서 이미 구조적으로 강제돼 있다** — 그리고 그것이 이 판정의 근거다:

| 층 | 강제 방식 |
|---|---|
| 타입 | `logs/types.ts:6-12` — 쓰기 payload 타입이 0건 |
| 어댑터 | `admin/data-source.ts` 의 export 4개에 쓰기 함수 0건. **없는 것을 화면이 부를 수 없다** |
| 라우트 | `App.tsx:328` — 쓰기 라우트 0건 |
| 표 | `LogTable.tsx:6-9` — 체크박스 열·⋯ 액션 열 0건 |
| 상세 | `LogPayloadDialog.tsx:66-70` — 푸터가 '닫기' 하나 |
| 쿼리 | `queries.ts:6-9` — `invalidateQueries` 0건 ('무효화할 쓰기가 없기 때문이다') |
| 테스트 | `logs.test.ts:45-73` — 4개 어댑터의 export 목록을 **전수 단언**. `deleteAdminLog` 가 추가되면 빨개진다 |

**서버가 이 계약을 깨면 프론트는 그것을 모른다** — 화면에는 삭제 버튼이 없으므로. 그래서 **불변성은 서버 저장소에서 강제돼야 하며**, 이 문서에 쓰기 엔드포인트가 없는 것은 그 선언이다.
