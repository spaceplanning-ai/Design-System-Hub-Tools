# BE-062. API 로그 (감사 로그) 백엔드 기능 명세

## 1. 개요

| 항목 | 내용 |
|---|---|
| 대상 화면 | FS-062 API 로그 (`/logs/api`) |
| 범위 | API 호출 로그 조회(상태 계열·메서드·기간·키워드 필터 · 정렬 · 페이징 · 기간 안 건수 · **API 키** · **요청·응답 페이로드**), 내보내기 조회. 그리고 이 도메인의 핵심 — **자격증명(API 키)의 서버 책임과 상태 계열 판정의 소유**(§3·§7) |
| 범위 밖 | **로그의 생성** — API 게이트웨이·미들웨어가 요청 처리 중에 append-only 로 남긴다. **로그의 삭제·수정** — 감사 불변성 원칙상 만들지 않는다(§7.5). **CSV 직렬화** — 프론트가 한다(§7.4) |
| 전제 | 세션 쿠키 인증. `/api` 프리픽스. **읽기 전용이므로 CSRF 대상 쓰기가 없다.** `application/json; charset=utf-8`. 시각은 **오프셋을 가진 ISO 8601** |
| 프론트 어댑터 | `apps/admin/src/pages/logs/api/data-source.ts` |
| 도메인 타입 | `api/types.ts` (`ApiLogEntry` · `HttpMethod` · `StatusClass`) + 섹션 공용 `logs/types.ts` |
| 형제 명세 | **BE-060(관리자 로그)이 이 섹션의 골격 정본이다** — 같은 셸·같은 엔진·같은 계약 구조. 이 문서는 **겹치는 것을 반복하지 않고 이 화면 고유의 것만** 적는다 |

> ⚠ **용어 주의**: 이 문서의 주제는 **HTTP 로그**이고 이 문서 자신도 **HTTP 계약**이다. 둘을 혼동하지 않기 위해 아래에서 **'로그의 값'**(응답 데이터로서의 상태 코드·메서드)과 **'이 엔드포인트의 계약'**(이 조회 API 가 내는 상태 코드)을 명시적으로 구분한다.
>
> **에러 봉투·권한 모델 상속**: BE-003 §2·§3 을 그대로 상속한다.
>
> **계약 골격 상속**: **BE-060 §2·§2.1·§3.2·§3.3·§3.4·§5·§7.5 를 그대로 상속한다.** 두 화면은 `LogListShell`·`query-engine`·`adapter`·`queries`·`masking` 한 벌을 공유하므로 에러코드 목록·은닉 정책·불변성·타임아웃·예외 매트릭스의 **구조**가 동일하다. 아래에는 **값이 다르거나 이 화면에만 있는 것**을 적는다.

### 1.1 코드 대조 근거표

| 사실 | 근거 |
|---|---|
| 심이 **정확히 2개**다 | `api/data-source.ts:39` `// TODO(backend): GET /api/logs/api?from=&to=&status=&method=&keyword=&sort=&dir=&page=&size=` · `:47` `// TODO(backend): GET /api/logs/api/export?from=&to=&status=&method=&keyword=&sort=&dir=` |
| **쓰기 함수가 0건이다** | export = `apiLogSpec` · `fetchApiLogs` · `fetchApiLogsForExport` · `toCsv`. `logs.test.ts:55,59-72` 가 전수 단언 |
| 쓰기 라우트가 0건이다 | `App.tsx:328` 주석 + `:331` 이 `/logs/api` 하나만 등록 |
| **`apiKey` 가 날것으로 온다** | `api/types.ts:52-56` — '클라이언트가 쓴 API 키. **날것 그대로 담는다** — 표시할 때 뒤 4자만 남기고 가린다' |
| **상태 계열이 클라이언트 파생이다** | `types.ts:35-39` `statusClassOf(status)` — `>= 500` → 5xx, `>= 400` → 4xx, 나머지 → 2xx. `data-source.ts:22` 가 축의 `valueOf` 로 그것을 쓴다 → **서버는 계열을 모른다**(§3.1) |
| **느림 임계가 클라이언트 상수다** | `types.ts:91-95` `SLOW_THRESHOLD_MS = 1000` — 서버가 정한다는 심이 없다(§7.6 #6) |
| **API 키가 검색 대상이 아니다** | `data-source.ts:25-26` — '키 전문을 입력해야 걸리는 검색은 키를 화면에 붙여넣게 만든다. 그 대신 요청 ID 로 찾는다' |
| **CSV 에 API 키가 마스킹된 채 나간다** | `data-source.ts:69` `{ header: 'API 키', cell: (entry) => maskTail(entry.apiKey) }` — **4화면 중 유일하게 민감 열을 내보낸다**(`:58-60` 이 근거) |
| **보존기간·마스킹·열람 감사에 심이 없다** | `types.ts:104-107`(`API_LOG_RETENTION`)은 클라이언트 상수. `masking.ts` 는 프론트 순수 함수. 열람 기록 호출 없음 → **전부 '심 없음(미정)'** |

**결론**: 이 문서는 **엔드포인트 2개만** 정의한다. 생성·보존기간 강제·서버 마스킹·열람 감사·상태 계열 집계·느림 임계는 **'심 없음(미정)'** 으로 남기고 §7 에서 판정한다.

## 2. 공통 (상속)

**BE-060 §2·§2.1 을 그대로 상속한다** — 에러 봉투(BE-003 §2), 에러코드 목록, CSRF 해당 없음, 프론트 역할 분기 없음. **재정의하지 않는다.**

이 화면 고유의 차이:

- **권한 리소스가 `page:/logs/api` 다** — 형제들과 다른 행이다. **그리고 이 화면은 개인정보가 아니라 자격증명에 닿는다**(§3.1).
- `error.fields[].name` 의 가능한 값: `from` · `to` · `status` · `method` · `page` · `size` · `sort` · `dir`.
- ⚠ **`status` 라는 이름이 두 뜻으로 쓰인다** — 쿼리 파라미터 `status` 는 **로그의 상태 계열 필터**(2xx/4xx/5xx)이고, 이 엔드포인트가 내는 HTTP status 는 별개다. 심(`data-source.ts:39`)이 그 이름을 정했으므로 따른다.

## 3. 인증 · 권한 모델 · 감사 원칙

### 3.1 역할

| 역할 | 설명 | 비고 |
|---|---|---|
| `page:/logs/api` **read** | 로그를 조회한다 — **행·API 키 뒷자리·요청/응답 페이로드 전부** | 없으면 403 화면 + **조회 요청 미발행**(`LogListShell.test.tsx:136-142`) |
| `page:/logs/api` **export** | 내보내기 버튼이 보인다 | `useRouteCan('export')`(`LogListShell.tsx:115`) → `LogToolbar.tsx:117` |

> **이 화면의 권한은 개인정보가 아니라 자격증명에 걸린다 — 다른 종류의 위험이다.** 형제 화면(FS-061)이 회원의 배송지·카드번호에 닿는다면 이 화면은 **파트너사의 API 키와 그 키가 무엇을 호출했는지**에 닿는다. `read` 를 가진 사람은 ① 어느 키가 살아 있는지 ② 그 키가 어느 엔드포인트를 부르는지 ③ 그 요청·응답 본문이 무엇인지를 본다. **키 뒷자리만으로 남의 키를 쓸 수는 없지만**(`types.ts:53-55`) **그 정보의 조합은 공격 표면 지도다.**
>
> **`read` 와 `export` 를 서버가 갈라야 한다** — BE-060 §3.1 과 같은 이유. **버튼을 숨기는 것은 보안이 아니다**: `GET /api/logs/api/export` 를 직접 호출하면 그만이고, 그 응답에는 **키 원본이 실려 있다**(§7.2).

### 3.2 상태 계열은 **서버 계약이 아니다** — 지금은 클라이언트가 정한다

**이 화면의 축 값(2xx/4xx/5xx)은 서버가 내려주는 필드가 아니다.** `data-source.ts:22` 가 `{ key: 'status', valueOf: (entry) => statusClassOf(entry.status) }` 로 **낱개 코드에서 파생**한다(`types.ts:35-39`). 즉:

| 화면이 보이는 값 | 어디서 오는가 | 서버 계약인가 |
|---|---|---|
| `entry.status`(200·401·500) | 서버 응답 필드 | **O** |
| 축의 계열(2xx/4xx/5xx) | `statusClassOf(entry.status)` — **클라이언트 파생** | **✕** |
| 계열별 배지 건수 | `countAxes` — **클라이언트 집계**(현재) | **✕** (연동 후에는 `axisCounts` 로 서버가 준다 — §6.1) |
| 셀의 '느림' 판정 | `durationMs >= SLOW_THRESHOLD_MS` — **클라이언트 상수 비교** | **✕** (§7.6 #6) |
| 행의 톤(danger/warning/neutral) | `toneOf` — **클라이언트 판정** | **✕** |

**그런데 심(`data-source.ts:39`)은 `status=` 를 쿼리로 보낸다.** 즉 **백엔드가 붙으면 서버가 그 계열 개념을 알아야 한다** — 프론트가 `?status=5xx` 를 보내면 서버가 그것을 `status >= 500` 으로 해석해야 하고, `axisCounts.status` 도 같은 경계로 세야 한다. **§7.4 가 그 경계를 계약으로 못 박는다** — 두 곳이 다르게 자르면 배지 숫자와 목록이 어긋난다.

### 3.3 403 vs 404 은닉 · 3.4 감사 불변성 · 3.5 타임아웃

**BE-060 §3.2 · §3.3 · §3.4 를 그대로 상속한다.** 컬렉션은 403, 개별 조회 엔드포인트가 없어 404 은닉 문제가 성립하지 않는다, append-only, 조회 5초 / 내보내기 30초.

이 화면 고유의 차이:

- **부피가 4화면 중 압도적이다**(`types.ts:99-102`: '관리자 로그의 수천 배'). **타임아웃 상한이 실제로 물릴 화면은 이곳이다** — 형제들의 5초/30초는 여유롭지만 여기서는 90일치 전량 스캔이 30초를 넘길 수 있다(§7.4).

## 4. 엔드포인트 명세

### BE-062-EP-01 · API 로그 조회

| 항목 | 내용 |
|---|---|
| 근거 (FS) | FS-062-EL-001, EL-002, EL-003, EL-003.1, EL-003.2, EL-007, EL-008, EL-010, EL-010.1, EL-011, EL-011.2, EL-014, EL-015, EL-015.1, EL-016, EL-016.1, EL-016.4 |
| 메서드 · 경로 | `GET /api/logs/api` — 근거: `data-source.ts:39` |
| 권한 | `page:/logs/api` **read** |
| 멱등성 | 멱등 (GET) |
| 페이징 | offset. `page` 1-base, `size` ∈ **{20, 50, 100}** |
| 레이트리밋 | 관리자당 분당 120회 |

**요청 — 쿼리** (전부 `data-source.ts:39` 심에 열거된 것뿐이다)

| 이름 | 타입 | 필수 | 제약 | 설명 |
|---|---|---|---|---|
| `from` · `to` | string | O | `YYYY-MM-DD`, KST 달력일, `to >= from`, `to <= 오늘`, ≤ 90일 | 조회 구간(양 끝 포함) |
| `status` | string | X | `all` \| `2xx` \| `4xx` \| `5xx` | **상태 계열 필터 — 낱개 코드가 아니다.** 경계는 §7.4 |
| `method` | string | X | `all` \| `GET` \| `POST` \| `PATCH` \| `PUT` \| `DELETE` | 메서드 필터 |
| `keyword` | string | X | 앞뒤 공백 제거. **길이 상한 미정** | **경로·클라이언트·요청 ID·클라이언트 IP** 부분 일치 — ⚠ **`apiKey` 는 제외한다. 의도된 것이며 계약이다**(§7.3) |
| `sort` | string | X | `occurredAt` \| `method` \| `path` \| `status` \| `durationMs` \| `client` (기본 `occurredAt`) — **6종** | 정렬 키 |
| `dir` | string | X | `asc` \| `desc` (기본 `desc`) | 정렬 방향 |
| `page` · `size` | number | X | ≥1 / {20,50,100} | 페이징 |

- **AND 결합 · 정렬이 자르기보다 먼저 · 동점은 id 로** — BE-060 §4 EP-01 과 동일(`query-engine.ts`). **이 화면에서 동점 처리가 가장 중요하다** — 같은 초에 여러 호출이 몰린다(FS-062 §1.3).
- 정렬 값의 의미: `status` 와 `durationMs` 는 **숫자로** 비교한다(`data-source.ts:31,34` — '문자열로 비교하면 90 이 1400 보다 크다'). `method`·`path`·`client` 는 문자열 사전순이다. ⚠ **`method`·`status` 는 라벨 맵을 거치지 않는다** — 형제 화면들이 한국어 라벨로 정렬하는 것과 다르다(그 값들이 이미 표시값이기 때문이다).

### BE-062-EP-02 · API 로그 내보내기 조회

| 항목 | 내용 |
|---|---|
| 근거 (FS) | FS-062-EL-009, EL-009.1, EL-017 |
| 메서드 · 경로 | `GET /api/logs/api/export` — 근거: `data-source.ts:47` |
| 권한 | `page:/logs/api` **export** — ⚠ **`read` 로 갈음하지 않는다**(§3.1) |
| 멱등성 | 멱등 (GET). 요청 시점 스냅샷 |
| 페이징 | **없음 — 조건 전체.** `page`·`size` 를 받지 않는다(심에 없다) |
| 레이트리밋 | 관리자당 **분당 3회** — ⚠ **이 화면에서 그 제한의 근거가 가장 크다**(부피가 수천 배) |

**요청 — 쿼리**: `from` · `to` · `status` · `method` · `keyword` · `sort` · `dir`.

**응답 200** — `readonly ApiLogEntry[]` (봉투 없이 배열)

- **화면과 같은 순서여야 한다**(`runLogExport` — `query-engine.ts:159-164`).
- 0건이면 **빈 배열**.
- **CSV 는 서버가 만들지 않는다.** 프론트 `toCsv`(`data-source.ts:74-76`)가 **9열**(시각(KST)·메서드·경로·상태·응답시간(ms)·클라이언트·**API 키**·요청 ID·클라이언트 IP)로 직렬화한다.

### 6.1 어댑터 본문 요구사항

**BE-060 §6.1 의 5개 요구를 그대로 상속한다** — ① `axisCounts` 를 서버가 계산(기간 안 모수) ② `total` 은 필터·검색 적용 후 ③ 정렬을 표시값 기준 + 동점은 id ④ `occurredAtIso` 에 오프셋 필수 ⑤ 응답이 이미 정렬돼 있을 것.

이 화면 고유의 추가 요구 **셋**:

1. **`status`·`durationMs` 를 숫자로 보낼 것.** 문자열로 보내면 정렬이 사전순이 되어 **`'90'` 이 `'1400'` 보다 크다**(`data-source.ts:33` 이 그 함정을 명시). `durationMs` 는 `formatNumber` 로 그려지므로 숫자여야 한다.
2. **계열 경계를 §7.4 의 것과 같게 자를 것.** `?status=5xx` 의 필터와 `axisCounts.status['5xx']` 의 집계가 **같은 경계**여야 한다 — 다르면 '배지엔 7건인데 목록엔 5건'이 된다.
3. **⚠ `apiKey` 를 응답에 실을지 재검토할 것** — §7.2. 지금 구조에서는 상세가 그것을 쓰므로 필요하지만, **뒤 4자만 내려주면 화면과 CSV 가 그대로 동작한다**(둘 다 `maskTail` 의 결과만 쓴다). 그때 `maskTail` 호출부 2곳이 사라지고 §7.1 의 위험이 근본적으로 없어진다.

## 7. 핵심 판정

### 7.1 【보안 판정】 API 키가 **`masking.ts` 의 보호 밖에서** 두 호출부에 의해 각자 가려진다

**BE-060 §7.1 을 상속한다**(마스킹은 클라이언트에서 일어나는 **표시 통제**이지 보안 통제가 아니다 — 원본이 브라우저에 도달한 뒤 렌더 직전에만 가려지고 devtools 로 우회된다).

**그리고 이 화면에는 형제들에 없는 구조적 취약점이 하나 더 있다.**

`masking.ts` 의 설계 원칙은 **'화면에 닿는 유일한 경로'** 다:

> `masking.ts:14-16` — '**마스킹은 화면이 아니라 여기서 한다.** 컴포넌트가 각자 마스킹하면 **한 곳만 빠뜨려도 유출이다.** 페이로드는 **이 파일을 통과한 뒤에만** 화면에 닿는다'
> `LogPayloadDialog.tsx:12` — '**화면에 닿는 유일한 경로가 `formatMaskedPayload` 다** — 날것을 그리는 분기는 이 파일에 없다'

**그 보호는 `payload` 트리에만 적용된다.** 그런데 `apiKey` 는 **entry 의 최상위 필드**이고(`types.ts:56`) `maskPayload` 는 그것을 훑지 않는다 — 그것은 `detail.payload` 만 받는다. 그래서 **이 화면은 `maskTail` 을 직접 두 번 부른다**:

| 호출부 | 근거 |
|---|---|
| 화면 상세 필드 | `ApiLogPage.tsx:100` — `{ label: 'API 키', value: maskTail(entry.apiKey) }` |
| CSV 열 | `data-source.ts:69` — `{ header: 'API 키', cell: (entry) => maskTail(entry.apiKey) }` |

**즉 `masking.ts` 가 없애려던 바로 그 상황이 이 필드에 대해 성립한다 — '컴포넌트가 각자 마스킹'하고 있다.** 세 가지가 따라온다:

**(a) 세 번째 호출부가 생기면 그 자리에서 잊을 수 있다.** 새 컬럼(예: 목록에 키 뒷자리를 보이기)이나 새 CSV 열, 새 상세 필드를 추가하는 사람은 **`maskTail` 을 스스로 기억해야 한다.** 타입은 `apiKey: string` 이라 아무 경고도 없다.

**(b) 어떤 테스트도 그것을 막지 않는다.** `logs.test.ts:304-311` 은 **페이로드의** `sk_live_` 만 단언한다(`formatMaskedPayload(entry.payload)`). **`entry.apiKey` 필드 자체가 화면에 어떻게 그려지는지는 단언하지 않는다.** `logs.test.ts:203-207` 은 `maskTail` 함수 자체와 `maskPayload({ apiKey })` 를 단언하지만 — **후자는 `apiKey` 가 *페이로드 안에* 있을 때의 이야기이지 entry 필드일 때가 아니다.** 즉 **두 호출부 중 하나를 지워도 이 테스트 스위트는 초록색이다.**

**(c) 그리고 §7.2 대로 키 원본이 이미 브라우저에 있다.**

**판정 — 계약이 확정하는 것**:

1. **가장 좋은 것은 서버가 뒤 4자만 내려주는 것이다**(§6.1 #3). 화면과 CSV 가 둘 다 `maskTail` 의 **결과만** 쓰므로 **응답이 이미 `'●●●●b6a5'` 이면 두 호출부가 사라지고 (a)(b)(c)가 한꺼번에 없어진다.** 상세가 키 원본을 필요로 하는 곳이 **어디에도 없다** — 목적이 '어느 키였나 대조'이지 '키가 무엇인가'가 아니기 때문이다(`types.ts:53-55`).
   ⚠ **다만 그러면 CSV 의 '대조' 목적이 서버 마스킹 방식에 종속된다** — 서버가 매번 다른 방식으로 가리면 파일 간 대조가 깨진다. 계약이 **`maskTail` 과 같은 규칙**(뒤 4자, 4자 이하는 전체 가림 — `masking.ts:79-82`)임을 못 박아야 한다.
2. **키 원본을 계속 내려준다면 `apiKey` 를 `maskPayload` 와 같은 단일 게이트에 넣어야 한다** — 예컨대 `detailOf` 가 entry 를 통째로 정제 함수에 통과시키는 구조. **'각자 기억하기'로 두지 않는다.**
3. **`payload.headers.authorization`·`x-api-key` 는 지금도 잘 가려진다** — `masking.ts:51`(`/token|authorization|bearer|jwt|cookie|session[-_]?id|refresh/i` → redact)과 `:54`(`/(api|access|client)[-_]?(key|secret)|signature/i` → tail)에 정확히 걸린다. **이 화면의 페이로드는 마스킹 규칙에 가장 잘 걸리는 편이다**(형제 FS-061 의 `address` 누락과 대조된다 — BE-061 §7.1). `logs.test.ts:304-311` 이 그것을 단언한다.
4. **`read` 없이는 아무것도 주지 않는다.**

**이관**: **A63(서버가 뒤 4자만) · A11(단일 게이트·테스트)** — §7.6 #1.

### 7.2 【보안 판정】 목록 응답이 전 행의 **키 원본과 요청·응답 본문**을 싣는다

**BE-060 §7.2 를 상속한다**(상세는 조회하지 않고 목록 행 객체로 그린다 → 한 번의 목록 조회가 그 페이지 모든 행의 본문을 내려준다. 상세에 별도 권한이 없다).

**이 화면에서 그 판정의 성격이 다르다 — 새는 것이 개인정보가 아니라 자격증명이다.**

**(a) 상세를 한 건도 열지 않아도 100건의 API 키 원본이 브라우저에 있다.** 목록에 그려지는 것은 6개 컬럼이고 **API 키는 그중에 없다**(상세에만 있다 — FS-062-EL-011.7). 그런데 `apiKey` 필드는 **모든 행에 실려 온다.** 즉 **화면이 의도적으로 감춘 값이 네트워크로는 전량 온다.** 그리고 `payload.headers` 에 **같은 키가 또** 있다(`fixtures.ts:110-113`).

**(b) 내보내기가 그 비대칭을 드러낸다.** CSV 는 키를 **가려서** 내보낸다(`maskTail` — 그것이 이 화면의 결정이다). 그런데 **EP-02 응답 배열에는 키 원본이 조건 전체만큼 실려 온다.** 90일치를 내보내면 그 기간 모든 파트너사 키가 브라우저 메모리에 들어온 뒤 CSV 에서만 가려진다 — **파일은 가리고 네트워크는 안 가린다.**

**(c) 그런데 이 화면에는 형제들에 없는 해결책이 있다.** FS-061 은 상세가 페이로드를 **보여줘야 해서** `payload` 를 뺄 수 없었다(BE-061 §7.2). **여기서는 `apiKey` 를 뺄 수 있다** — 화면이 원본을 쓰지 않기 때문이다(§7.1 #1). **키 원본을 응답에서 없애는 것이 구조 변경 없이 가능한 유일한 필드다.**

**판정**:

1. **`entries[].apiKey` 를 뒤 4자로 내려준다**(§7.1 #1 · §6.1 #3). 이것이 (a)(b)를 한꺼번에 없앤다.
2. **`payload.headers` 의 키는 서버가 저장 시점에 가린다**(BE-060 §7.1 의 1차 방어선). 지금은 클라이언트 마스킹이 그것을 가리지만 **원본이 이미 왔다.**
3. **EP-02 응답에서 `payload` 를 뺀다** — CSV 가 쓰지 않는다(9열에 페이로드가 없다). **⚠ `ApiLogEntry.payload` 가 필수 필드라**(`types.ts:60`) 내보내기 전용 projection 타입이 필요하다 — **아직 없다**(§7.6 #3).
4. **EP-01 의 `payload` 는 지금 구조에서 뺄 수 없다** — 상세가 그것으로 그린다. 빼려면 개별 조회 엔드포인트가 선행돼야 한다. **심이 없으므로 정의하지 않는다.**

**이관**: **A63 · A11 (보안)** — §7.6 #1 · #3.

### 7.3 API 키를 검색 대상에서 뺀 것은 **계약이다** — 되돌리지 않는다

`data-source.ts:25-26` 이 그 결정과 이유를 적었다:

> '**API 키로는 검색하지 않는다** — 키 전문을 입력해야 걸리는 검색은 **키를 화면에 붙여넣게 만든다.** 그 대신 요청 ID 로 찾는다(서버 로그와 대조하는 열쇠이자, **유출돼도 무해한 식별자**다).'

**판정: 이것을 계약으로 승격한다.** 근거가 코드 주석보다 강하다 — **이 앱에서 검색어는 URL 에 실린다**(`list-state.ts:213-226` — `?q=`). 즉 API 키를 검색 대상으로 만들면:

- 키가 **주소창에** 남는다.
- 키가 **브라우저 history** 에 남는다(검색어는 `replace` 라 항목이 쌓이진 않지만 현재 URL 에는 있다).
- 키가 **공유 링크**에 붙어 나간다 — 그리고 이 섹션은 링크 공유를 **핵심 운영 루프로 설계했다**(`list-state.ts:4-10`: '"이거 좀 봐주세요" 하며 링크를 붙여넣는다').
- 키가 **서버 액세스 로그**에 남는다 — 그리고 그것이 곧 이 화면이 보여주는 로그다. **API 키 검색은 자기 자신에게 키를 기록하게 만든다.**

**따라서 `keyword` 는 `apiKey` 를 보지 않는다**(§4 EP-01). 서버가 그 필드를 검색 대상에 넣으면 **프론트가 안 보내도 다른 호출부가 그것을 쓸 수 있게 된다** — 계약이 그것을 금지한다.

**대체 경로가 이미 있다**: `requestId`(유출돼도 무해) · `client`(이름) · `clientIp`. '이 키가 무엇을 했나'는 **키 뒷자리를 상세에서 확인한 뒤 `client` 로 좁히면** 답할 수 있다.

### 7.4 상태 계열의 경계는 계약이어야 한다 — 그리고 3xx 전제가 강제되지 않는다 【정합 판정】

**계열 판정은 지금 클라이언트에 있다**(§3.2 — `statusClassOf`). **그런데 심(`data-source.ts:39`)은 `status=` 를 서버로 보낸다.** 즉 **백엔드가 붙는 순간 그 판정이 서버로 옮겨간다.** 계약이 경계를 못 박지 않으면 두 곳이 갈라진다.

**계약이 확정하는 경계** (`types.ts:35-39` 를 그대로 승격한다):

| 계열 | 경계 | 근거 |
|---|---|---|
| `5xx` | `status >= 500` | `types.ts:36` |
| `4xx` | `400 <= status < 500` | `types.ts:37` |
| `2xx` | `status < 400` — **나머지 전부** | `types.ts:38` |

**그리고 `axisCounts.status` 도 같은 경계로 세야 한다**(§6.1 #2) — 다르면 '배지엔 7건인데 목록엔 5건'이 된다.

**⚠ 그러나 `2xx = 나머지 전부` 라는 정의에 문제가 있다.** 주석이 전제를 밝힌다:

> `types.ts:34` — '3xx 는 이 API 에 없어 2xx 로 접는다(**리다이렉트를 쓰지 않는 JSON API 다**)'

**전제는 명시적이고 합리적이다. 그러나 강제되지 않는다.** 결과:

1. **302 가 오면 '성공 (2xx)' 배지에 집계되고** 표에서도 **강조 없이 `302`** 로만 보인다(`ApiLogPage.tsx:26` — `kind === '2xx'` 면 코드만 그린다). 운영자는 **리다이렉트 루프를 성공으로 읽는다.**
2. **1xx 도 2xx 다.**
3. **알 수 없는 값(0·-1·999)도 2xx 다** — `< 400` 이면 무조건. 즉 **게이트웨이가 타임아웃을 `0` 으로 기록하면 그것이 '성공'이 된다.**
4. **`status` 축으로 좁힐 때도 같다** — `?status=2xx` 가 302·0 을 함께 데려온다.

**판정**:

1. **서버 계약이 그 전제를 보증하거나**(응답의 `status` 는 2xx·4xx·5xx 만이다 — 그러면 지금 코드가 정확하다), **화면이 예외를 드러내야 한다**(3xx·1xx·알 수 없는 값을 위한 계열을 추가하거나, 최소한 `2xx` 를 `200 <= status < 300` 으로 좁히고 나머지를 '기타'로).
2. **둘 중 하나를 고르기 전까지는 '성공'이라 부르는 칸에 성공이 아닌 것이 섞일 수 있다** — 감사 화면에서 그것은 조용한 거짓말이다.
3. **심이 없으므로 여기서 확정하지 않고 이관한다** — 다만 **경계 자체(위 표)는 계약으로 확정한다.** 프론트가 이미 그렇게 자르고 있고 서버가 다르게 자르면 즉시 어긋나기 때문이다.

**이관**: **A63 · A11** — §7.6 #5 · FS-062 §7 #7.

### 7.5 보존기간(90일)과 조회 상한(90일)이 서로를 모른다 — 심 없음 (미정)

| 값 | 어디서 | 심 |
|---|---|---|
| 보존기간 '90일' | `api/types.ts:104-107` — **이 화면의 상수** | **없음** |
| 조회 상한 '90일' | `logs/types.ts:89` `MAX_RANGE_DAYS = 90` — **4화면 공용 상수** | **없음** |

**BE-062-EP-04 는 '심 없음(미정)'이다.** 지어내지 않는다.

**그러나 이 화면에만 있는 정합 문제가 있다**: **두 90 이 우연히 같다.** 하나는 이 화면의 부피 판단('트래픽 부피가 커 단기 보존')이고 다른 하나는 4화면 공용 조회 상한이다 — **서로를 참조하지 않는다.** 결과:

- **이 화면은 '전 보존기간을 한 번에 조회·내보내기 할 수 있는' 유일한 화면이다**(형제들은 보존 3년·1년·180일 > 조회 90일). **그것이 의도인지 우연인지 코드에 근거가 없다.**
- 보존기간이 **60일**로 바뀌면 90일 조회는 언제나 **빈 앞 30일**을 갖는다 — 운영자는 그 빈칸이 '없었던 것'인지 '폐기된 것'인지 알 수 없고, **그것이 정확히 보존기간을 화면에 적기로 한 이유였다**(`logs/types.ts:265-268`).
- 보존기간이 **180일**로 늘면 앞 90일은 **화면으로 볼 수 없다** — 데이터는 있는데 도달 경로가 없다.

**판정**: **보존기간은 서버가 소유하고 강제한다**(BE-060 §7.3 상속). **그리고 조회 상한은 보존기간을 알아야 한다** — 최소한 '보존기간을 넘는 구간은 요청할 수 없다'가 성립해야 한다. **심이 없으므로 확정하지 않고 이관한다.**

**이관**: **A63 · A01** — §7.6 #6.
