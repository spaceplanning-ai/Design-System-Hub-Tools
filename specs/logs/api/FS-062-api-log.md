### 1.1 앞의 두 화면과 무엇이 다른가 — 이 화면 고유의 것

**4화면은 축의 어휘와 컬럼만 다르고 나머지가 전부 같다**(`logs/types.ts:229-232`). 그래서 여기에는 **다른 것만** 적는다.

| 항목 | 이 화면 | FS-060 관리자 로그 / FS-061 회원 활동 로그 |
|---|---|---|
| **행위자가 누구인가** | **사람이 아니라 키(client)** 다(`types.ts:7`). 그래서 검색이 경로·클라이언트·요청 ID 로 가고, 축이 '결과'가 아니라 **상태 코드 계열**이며, **응답 시간이라는 숫자 컬럼이 처음 등장한다** | 운영자 / 회원 |
| **보존기간** | **90일 — 4화면 중 가장 짧다.** '트래픽 부피가 커 단기 보존합니다.'(`types.ts:104-107`) | 3년 / 1년 |
| 보존기간이 짧은 **이유** | **'부피가 압도적이고(관리자 로그의 수천 배) 가치는 빨리 식는다 — 석 달 전의 응답 시간은 아무도 묻지 않는다. 오래 남길수록 비용만 남는 로그다'**(`types.ts:99-102`). **근거가 개인정보가 아니라 부피·가치다** — FS-061 과 다른 종류의 근거다 | 내부 통제 / 개인정보 최소 보관 |
| 축 | **상태 계열**(2xx/4xx/5xx) · **메서드**(5종) | 결과 · 액션/활동 |
| 컬럼 | 시각 · 메서드 · 경로 · **상태** · **응답시간(ms)** · 클라이언트 | 시각 · 행위자/회원 · 액션/활동 · 대상/내용 · 결과 · IP/접속 |
| **숫자 컬럼** | **있다 — 응답시간(ms).** 우측 정렬 + tabular-nums(FS-062-EL-011.6). **4화면 중 이 화면과 FS-063 에만 있다** | 없다 |
| **결과 셀** | **없다** — `outcome`/`failureReason` 필드 자체가 없다. 대신 **상태 코드**가 그 일을 한다(`OutcomeCell` 미사용) | `OutcomeCell`(✕ 아이콘 + '실패' + 사유) |
| **CSV 의 민감 열** | **있다 — API 키가 마스킹된 채로 나간다**(§1.2). **4화면 중 유일하다** | 없다(민감 열을 통째로 뺀다) |
| 경고 강조 | **5xx** > 4xx | 권한 변경 / 탈퇴 > 실패 |
| 톤 | 5xx=danger · 4xx=warning · **느림=warning** · 나머지 neutral(3단계) | 실패=danger · 나머지 neutral(2단계) |

### 1.2 이 화면의 설계 원칙 (요구사항 — 설계 근거가 곧 명세다)

| 원칙 | 내용과 이유 |
|---|---|
| **읽기 전용 감사 로그 — 삭제·수정·행 ⋯ 메뉴·체크박스가 없다** | `api/data-source.ts` 의 공개 표면은 `apiLogSpec`·`fetchApiLogs`·`fetchApiLogsForExport`·`toCsv` **넷뿐**이고 `logs.test.ts:55,59-72` 가 전수 단언한다 — **없는 것을 화면이 부를 수 없다** |
| **상태 코드를 낱개가 아니라 계열로 나눈다** | **'운영자가 묻는 것은 404 가 몇 건인가가 아니라 깨진 게 있나(5xx) · 누가 잘못 부르나(4xx) 다'**(`types.ts:22-24`). 그래서 축이 2xx/4xx/5xx 3칸이다. ⚠ **3xx 는 2xx 로 접힌다** — '리다이렉트를 쓰지 않는 JSON API 다'(`types.ts:34`)가 그 전제이며, **그 전제가 깨지면 리다이렉트가 성공으로 집계된다**(§7 #7) |
| **5xx 는 우리 잘못이고 4xx 는 부르는 쪽의 잘못이다 — 같은 색으로 섞지 않는다** | `ApiLogPage.tsx:81-87` — 5xx=danger, 4xx=warning. 섞으면 '누가 고쳐야 하는가'가 사라진다 |
| **느린 호출은 숫자만으로 보이지 않는다 — 임계를 넘으면 글자가 스스로 말한다** | 1초 이상이면 `'1,400 느림'`(`ApiLogPage.tsx:41-55`). **'1초는 사람이 멈췄다고 느끼기 시작하는 지점이고, API 에서는 이미 사고다'**(`types.ts:92-95`). 색 위에 **글자**가 있다 |
| **단위(ms)는 숫자에 붙이지 않고 헤더로 올린다** | **'값마다 ms 가 따라다니면 그것이 자릿수 정렬을 다시 깨뜨린다'**(`ApiLogPage.tsx:7-10`). 등폭 숫자가 아니면 1400 과 980 의 자릿수가 어긋나 '어느 쪽이 큰가'를 눈으로 못 읽는다 |
| **⚠ API 키로는 검색하지 않는다 — 의도된 부재다** | **'키 전문을 입력해야 걸리는 검색은 키를 화면에 붙여넣게 만든다. 그 대신 요청 ID 로 찾는다(서버 로그와 대조하는 열쇠이자, 유출돼도 무해한 식별자다)'**(`data-source.ts:25-26`). **검색창에 비밀을 치게 하지 않는 것이 이 결정의 요점이다** — 검색어는 URL(`?q=`)에 실리고 history·로그·공유 링크에 남는다 |
| **⚠ API 키는 뒤 4자만 남긴다 — 그리고 그것이 CSV 에도 나간다** | 화면(`ApiLogPage.tsx:100`)과 CSV(`data-source.ts:69`)가 둘 다 `maskTail`. **'어느 키였나를 대조해야 그 키를 폐기할 수 있고, 그것으로 남의 키를 쓸 수는 없다'**(`types.ts:53-55`). **다른 화면의 CSV 는 민감 열을 아예 빼지만 여기서는 '어느 키가 401 을 맞고 있나'가 곧 그 파일을 만드는 이유다**(`data-source.ts:58-60`). ⚠ **`apiKey` 는 entry 에 날것으로 온다** — 마스킹은 표시·CSV 시점이다(§7 #2 · BE-062 §7.1) |
| 나머지 원칙 | **URL 이 곧 view** · **시각은 KST 초까지**(이 화면에서 특히 중요하다 — §1.3) · **보존기간을 화면에 적는다** · **페이로드는 화면에 닿기 전에 가려진다** — 전부 FS-060 §1.1 과 동일하며 같은 공용 모듈이 결정한다 |

### 1.3 초 정밀도가 이 화면의 존재 이유와 맞닿는다

`time.ts:14-18` 이 이 섹션만 초를 보여주는 이유를 밝히는데, **그 문장이 지목하는 것이 정확히 이 화면이다**:

> '로그인 이력은 분까지로 충분했지만 **감사 로그는 아니다. 1초에 40번 두드리는 API 호출은 분 단위로 보면 전부 같은 시각이 되어 순서가 사라진다. 무차별 대입인지 정상 트래픽인지를 가르는 것이 바로 그 순서다.**'

그리고 이 화면의 픽스처가 그 시나리오를 담는다 — **'폐기된 키로 30분마다 6번 막히는 배치'**(`fixtures.ts:86-91`: '파트너사가 키 교체를 안 했거나, 그 키가 유출돼 남이 쓰고 있거나 둘 중 하나다'). **동점 정렬이 id 로 끊기는 것**(`query-engine.ts:113-118`)도 같은 이유다: '같은 초에 일어난 두 API 호출의 순서가 실행마다 달라지면 페이지 경계의 행이 두 페이지에 나타나거나 어느 페이지에도 나타나지 않는다'.
