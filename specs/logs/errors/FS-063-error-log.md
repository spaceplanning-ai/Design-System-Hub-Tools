### 1.1 형제 세 화면과 무엇이 다른가 — 이 화면 고유의 것

| 항목 | 이 화면 | FS-060 / FS-061 / FS-062 |
|---|---|---|
| **행위자가 없다 — 유일하다** | **'앞의 세 화면이 누가 무엇을 했나라면 이 화면은 무엇이 깨졌나다. 그래서 유일하게 행위자가 없다 — 아무도 하지 않았는데 일어난 일이다'**(`types.ts:3-4` · `ErrorLogPage.tsx:3-4`). 그래서 `ip` 필드도 없다 | 운영자 / 회원 / 클라이언트 키 |
| **보존기간** | **180일** — 'API 로그(90일)보다 길고 관리자 로그(3년)보다 짧다'(`types.ts:99`) | 3년 / 1년 / 90일 |
| 보존기간의 **근거** | **'부피는 크지만 작년에도 이랬나(재발 판단)를 물을 일이 실제로 있고, 반기면 대개 답이 된다'**(`types.ts:99-101`). **근거가 통제도 개인정보도 부피도 아닌 '재발 추적'이다** — 4화면의 근거가 전부 다른 종류다 | 내부 통제 / 개인정보 최소 보관 / 트래픽 부피 |
| 축 | **심각도**(3단계) · **발생 위치**(5종 고정) | 결과·액션 / 결과·활동 / 상태·메서드 |
| 컬럼 | **5개** — 시각 · 심각도 · 오류 · 발생 위치 · **발생 횟수** | 6개 |
| **숫자 컬럼** | **있다 — 발생 횟수.** 우측 정렬 + tabular-nums(FS-063-EL-011.6). **FS-062 와 이 화면에만 있다** | FS-062 만 |
| **결과 셀** | **없다** — `outcome`/`failureReason` 필드가 없다. **오류 로그의 모든 행은 이미 실패다**(`OutcomeCell` 미사용) | 있다(FS-060·FS-061) |
| **'해결됨' 토글이 없다** | **'오류 추적 도구(Sentry 등)에는 해결됨 토글이 있다. 여기에는 없다. 이것은 이슈 트래커가 아니라 감사 로그다 — 일어난 일은 일어난 일이고, 그 사실을 나중에 해결됨으로 덧칠할 수 있다면 그것은 이미 불변 기록이 아니다'**(`types.ts:6-10`). §1.2 | 해당 없음(원래 상태가 없다) |
| 경고 강조 | **치명** > 오류 | 권한 변경 / 탈퇴 / 5xx > … |
| 톤 | **warning → warning, 나머지(치명·오류) → danger** — **2단계**(§7 #6) | 2단계 / 2단계 / 3단계 |

### 1.2 이 화면의 설계 원칙 (요구사항 — 설계 근거가 곧 명세다)

| 원칙 | 내용과 이유 |
|---|---|
| **읽기 전용 감사 로그 — 그리고 여기서는 '해결됨' 토글조차 없다** | 형제들은 '삭제·수정이 없다'인데 **이 화면은 한 걸음 더 간다.** 오류 추적 도구의 표준 기능인 '해결됨' 상태를 **의도적으로 만들지 않았다**: '**일어난 일은 일어난 일이고, 그 사실을 나중에 해결됨으로 덧칠할 수 있다면 그것은 이미 불변 기록이 아니다. 고치는 일은 이슈 트래커에서 하고, 여기에는 일어났다는 사실만 남는다**'(`types.ts:6-10`). `errors/data-source.ts` 의 공개 표면은 `errorLogSpec`·`fetchErrorLogs`·`fetchErrorLogsForExport`·`toCsv` **넷뿐**이고 `logs.test.ts:56,59-72` 가 전수 단언하며 **그 정규식에 `resolve`·`archive` 가 포함돼 있다**(`:68`) — **`resolveError` 를 추가하는 순간 빨개진다** |
| **경고를 오류와 섞지 않는다 — 섞으면 진짜 사고가 잡음에 묻힌다** | 심각도가 3단계인 이유다(`types.ts:16-20`). **'그것이 오류 로그를 아무도 안 보는 가장 흔한 이유다'** — 매일 뜨는 이미지 업로드 경고 사이에 결제 게이트웨이 타임아웃이 묻히면 그 화면은 무용지물이다. 픽스처가 그 대조군을 의도적으로 담는다(`fixtures.ts:93`: '일상 경고 — 매일 조금씩 있고 아무 일도 아니다. **이것이 신호와 잡음을 가르는 대조군이다**') |
| **정렬은 사전순이 아니라 심각한 순이다** | `ERROR_SEVERITY_RANK`(`types.ts:29-34`) — critical 0 · error 1 · warning 2. **'사전순이면 경고가 맨 위로 온다'** — '경고' < '오류' < '치명' 이 한국어 사전순이다. `data-source.ts:27-28` 이 그것을 정렬 값으로 쓰고 `logs.test.ts:519-531` 이 고정한다 |
| **발생 횟수가 사건의 크기다** | **'1회와 340회는 완전히 다른 사건이다 — 이 숫자가 없으면 표에서 둘이 똑같이 한 줄로 보인다'**(`types.ts:48-51` · `ErrorLogPage.tsx:65`). 픽스처의 결제 게이트웨이 타임아웃이 **340회**다: '**1회였다면 재시도로 끝났을 것이고, 340회는 사고다. 그 차이를 말하는 것이 occurrences 다**'(`fixtures.ts:26-28`) |
| **추적 ID 가 이 화면의 출구다** | **'운영자가 이것을 복사해 개발자에게 준다 — 없으면 "오류 났어요"로 끝나 아무도 못 찾는다'**(`types.ts:52-55`). 상세에 있고(FS-063-EL-016.1) **CSV 에도 나간다**(`data-source.ts:65`) — 스택 대신 그것을 싣는다(§1.2 아래 항목) |
| **⚠ CSV 에 스택·컨텍스트를 싣지 않는다 — 대신 추적 ID 를 싣는다** | **'스택에는 내부 경로·커넥션 문자열·토큰이 들어 있다 — 그것이 파일로 나가 메일로 오가면 내부 구조를 통째로 흘리는 것이다. 대신 추적 ID 를 싣는다: 개발자는 그것 하나로 서버 로그에서 전문을 찾는다'**(`data-source.ts:52-56`). **판단이 명시적이고 대체 경로까지 설계했다** |
| **⚠ 그러나 스택·컨텍스트는 화면에 나온다 — 그리고 마스킹이 완전하지 않다** | 페이로드에는 **커넥션 문자열·API 키·토큰이 실제로 들어 있고**(`fixtures.ts:12`: '마스킹 확인용이다') 그 셋은 규칙에 정확히 걸린다. **`context.recipient` 의 회원 이메일도 이제 걸린다** — `masking.ts:73` 이 `recipient|수신자` 를 규칙에 넣었다(기준 `a5c2639`, 테스트 `logs.test.ts:330`). **그러나 `stack` 문자열은 여전히 통째로 통과한다** — 키가 하나뿐이라 그 안에 무엇이 들어 있든 마스킹이 닿지 않는다(§7 #2 · BE-063 §7.1) |
| **한 사건은 여러 로그에 자국을 남긴다** | 픽스처가 그것을 의도적으로 설계했다 — 결제 게이트웨이 타임아웃(치명, 340회)이 **API 로그의 5xx 스파이크와 같은 시각**이다(`fixtures.ts:6-9`): '두 화면을 나란히 놓으면 사건이 재구성된다: 게이트웨이가 죽었고 → 주문 API 가 500 을 뱉었고 → 회원의 결제가 실패했다. **감사 화면이 여러 개인 이유가 이것이다**'. ⚠ **다만 화면에 그 연결을 건너는 경로가 없다**(§7 #11) |
| 나머지 원칙 | **URL 이 곧 view** · **시각은 KST 초까지** · **보존기간을 화면에 적는다** — 전부 FS-060 §1.1 과 동일하며 같은 공용 모듈이 결정한다 |

### 4.1 공통 예외 규칙

> **FS-060 §4.1 과 동일하다** — 같은 공용 모듈이 결정한다. 아래는 **이 화면 고유의 것과 값이 다른 것만** 적는다.

|---|---|
| **감사 불변성 — 그리고 '해결됨'조차 없다** | 쓰기 경로가 **없다** — `errors/data-source.ts` 에 삭제·수정 함수가 없고 하위 라우트도 없다. **그리고 이 화면은 상태 변경 자체를 만들지 않았다**('해결됨' 토글 — §1.2). `logs.test.ts:67-72` 의 금지 정규식에 **`resolve`·`archive` 가 포함돼 있어**(`:68`) `resolveError` 를 추가하면 테스트가 빨개진다 |
