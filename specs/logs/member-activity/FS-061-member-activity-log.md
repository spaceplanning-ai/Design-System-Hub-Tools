# FS-061. 회원 활동 로그 (읽기 전용 감사 로그)

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 목적 | **"이 회원이 정말 그 주문을 했나요?" · "탈퇴 전에 무슨 일이 있었나요?"에 답한다.** 회원이 서비스에서 한 일(가입·로그인·주문·결제·리뷰·적립금·정보 수정·탈퇴)을 결과·활동·기간·키워드로 좁혀 조회하고, 한 건의 요청 페이로드를 열어보고, 필요하면 CSV 로 내보낸다 |
| 역할(주 사용자) | 관리자 (구현에 역할 분기 없음 — §4.1) |
| 진입 경로 | 좌측 GNB > 로그 관리 > 회원 활동 로그 (`nav-config.ts:209` — `['회원 활동 로그', '/logs/member-activity']`) |

### 1.1 관리자 로그(FS-060)와 무엇이 다른가 — 이 화면 고유의 것

**4화면은 축의 어휘와 컬럼만 다르고 나머지(URL 상태·IME 검색·정렬·페이지네이션·빈 상태·에러·권한·내보내기·상세)가 전부 같다**(`logs/types.ts:229-232`). 그래서 여기에는 **다른 것만** 적는다.

| 항목 | 이 화면 | 관리자 로그(FS-060) |
|---|---|---|
| **행위자가 누구인가** | **회원** — 그래서 **개인정보 밀도가 훨씬 높다.** 주문 페이로드에는 배송지와 결제수단이 들어 있다(`types.ts:6-7`) | 운영자 |
| **보존기간** | **1년** — '개인정보 최소 보관 원칙. 보존기간이 지나면 자동 폐기됩니다.'(`types.ts:92-95`) | 3년 |
| 보존기간이 짧은 **이유** | **관리자 로그는 '우리 직원이 무엇을 했나'라서 오래 들고 있는 것이 통제이지만, 이 로그는 남의 개인정보다. 오래 들고 있는 것 자체가 위험이고 비용이다 — 개인정보는 목적을 다하면 지우는 것이 원칙이지 자산이 아니다**(`types.ts:85-90`) | 사고는 몇 달 뒤에 발견된다 |
| 축 | 결과 · **활동**(8종) | 결과 · 액션(7종) |
| 컬럼 | 시각 · 회원 · 활동 · 내용 · 결과 · **접속**(IP+기기) | 시각 · 행위자 · 액션 · 대상 · 결과 · IP |
| **정렬 가능 컬럼** | **6개 중 5개** — '접속'은 정렬 불가(§7 #6) | 6개 전부 |
| **레코드 링크** | **있다** — 계정을 누르면 회원 상세로 간다(FS-061-EL-011.4) | 없다(FS-060 §7 #12) |
| 경고 강조 | **탈퇴** > 실패 | 권한 변경 > 실패 |
| 상세 필드 | + **회원 상태**(탈퇴/유지) · 기기 | + 당시 역할 |

### 1.2 이 화면의 설계 원칙 (요구사항 — 설계 근거가 곧 명세다)

| 원칙 | 내용과 이유 |
|---|---|
| **읽기 전용 감사 로그 — 삭제·수정·행 ⋯ 메뉴·체크박스가 없다** | 감사 기록은 **불변**이어야 한다(`logs/types.ts:6-12`). `member-activity/data-source.ts` 의 공개 표면은 `memberActivitySpec`·`fetchMemberActivityLogs`·`fetchMemberActivityLogsForExport`·`toCsv` **넷뿐**이고 `logs.test.ts:49-54,59-72` 가 그것을 전수 단언한다 — **없는 것을 화면이 부를 수 없다** |
| **탈퇴해도 로그는 남는다 — 계정이 아니라 사건을 기록한다** | 탈퇴 회원은 `memberId` 가 `null` 이라 가리킬 레코드가 없다(`types.ts:42`). **그래도 로그는 남는다** — 탈퇴했다고 그 사람이 한 주문의 기록까지 사라지면 **정산도 분쟁 대응도 불가능하다**(`fixtures.ts:181-183`). 그래서 링크만 없어지고 행은 남는다 |
| **없는 계정을 있는 것처럼 보이게 하지 않는다** | 탈퇴 회원의 계정 셀은 **링크가 아니라 글자**이고 보조 줄에 '탈퇴'가 붙는다(`MemberActivityPage.tsx:37-39`). 억지로 경로를 만들면 죽은 링크가 된다(FS-008-EL-008.2 와 같은 규칙) |
| **두 개의 목적지가 헷갈리지 않는다** | **행**을 누르면 **페이로드**가 열리고(이 화면의 본업 — 무엇을 보냈는가), **계정 링크**를 누르면 **회원 상세**로 간다. `useRowNavigation` 의 인터랙티브 가드가 링크 클릭을 행 활성화에서 제외하므로 **계정을 누른 사람에게 페이로드가 뜨는 일은 없다**(`MemberActivityPage.tsx:7-12`) |
| **⚠ 페이로드의 민감정보는 화면에 닿기 전에 가려진다 — 그러나 완전하지 않다** | 이 화면의 페이로드에는 **카드번호·전화번호·주소·비밀번호가 실제로 들어 있다**(`fixtures.ts:10`: '마스킹이 도는지 보기 위해서다'). `LogPayloadDialog.tsx:96` 이 `formatMaskedPayload` 의 결과만 그린다. **그러나 마스킹은 키 이름 기반이라 값을 보지 않는다** — `card`·`phone`·`password`·`email` 은 가려지지만 **`address` 와 `receiver` 는 규칙에 없어 그대로 나온다**(§7 #2 · BE-061 §7.1). **이 화면이 4화면 중 그 구멍이 가장 아픈 곳이다** — 개인정보 밀도가 가장 높기 때문이다 |
| **탈퇴를 실패보다 먼저 말한다** | 요약 줄의 경고(`MemberActivityPage.tsx:122-130`)는 탈퇴를 실패보다 앞세운다 — **탈퇴는 되돌릴 수 없는 사건이다** |
| **연쇄로만 답할 수 있는 질문이 있다** | 픽스처가 '**결제 실패 → 재시도 → 성공**'을 의도적으로 담는다(`fixtures.ts:114-119`): 'CS 가 가장 자주 받는 문의가 **결제가 안 됐는데 돈이 나갔어요** 다. 개별 행으로는 답할 수 없고 **연쇄로만** 답할 수 있다'. ⚠ **다만 화면에 연쇄를 묶는 개념이 없다**(§7 #10) |
| 나머지 원칙 | **URL 이 곧 view** · **시각은 KST 초까지** · **보존기간을 화면에 적는다** — 전부 FS-060 §1.1 과 동일하며 같은 공용 모듈이 결정한다 |
