# BE-035. 이메일 발송 백엔드 기능 명세

## 1. 개요

| 항목 | 내용 |
|---|---|
| 대상 화면 | FS-035 이메일 발송 (`/marketing/email` · `/new` · `/:id/edit`) |
| 범위 | 이메일 발송 캠페인 목록·단건 조회, 등록·수정, 삭제(단건·일괄), **발송 실행 트리거**(심 근거 — §4 EP-06 · §7.3) |
| 범위 밖 | 실제 메일 전송(MTA·ESP 연동) — 이 계약은 '발송을 지시받았다'까지만 정한다. 예약 실행(스케줄러) · 오픈/클릭 추적 픽셀·링크 래핑 구현 — 서버 내부 관심사. 발신자·세그먼트·템플릿 레지스트리 — **심 없음(§4 EP-07/08/09)** |

| 도메인 타입 | `apps/admin/src/pages/marketing/email/types.ts` · 공용 `apps/admin/src/pages/marketing/_shared/messaging.ts` |
| 검증 정본 | `apps/admin/src/pages/marketing/email/validation.ts` (`emailSchema`) — **단, 프론트 검증은 UX 층이고 법적 강제는 서버가 정본이다(§7.4~§7.7)** |
| BE-034(SMS)와의 관계 | **형제 계약이다.** 상태 머신(§3.1) · 동시성(§7.2) · 발송 멱등(§7.3) · 수신 동의 TOCTOU(§7.6) · 규모 상한(§7.7) · 변수 화이트리스트(§7.8) · 감사 로그(§7.9) 판정이 **동일하다** — 아래에서는 이메일 고유 차이만 논증하고 나머지는 BE-034 를 참조한다. **채널 차이**: 제목이 있고 · (광고) 표기가 제목에 걸리며 · 야간 제한이 없고(§7.5) · 바이트/유형 개념이 없으며 · 수신거부 링크와 오픈/클릭 추적이 있다 |

> **에러 봉투·권한 모델 상속**: BE-003 §2·§3 을 그대로 상속한다. 아래는 이메일 발송 고유 차이만 기술한다.

## 2. 공통 (상속)

- **에러 봉투**: BE-003 §2. 공통 에러코드 동일. `error.traceId` 는 프론트가 '오류 코드'로 노출한다(FS-035-EL-016).
- **권한**: `admin` = 전체. `operator` = 조회 계열만(목록·단건). 쓰기(등록·수정·삭제) → 403. **발송 실행(EP-06)은 `admin` 전용이며 별도 `marketing:send` 권한 분리 권고**(§7.3). 마케팅 도메인 읽기 권한 없는 관리자 → 컬렉션 403 / 개별 404 은닉(BE-003 §3.2 — 수신자 세그먼트 구성이 영업 정보다).
- **CSRF**: 쓰기(POST·PUT·DELETE)에 `X-CSRF-Token`.
- **타임아웃**: 조회·쓰기 5초 → 504. **발송 실행(EP-06)은 트리거만 하고 즉시 202 로 반환**한다.
- **프론트 역할 분기 없음**(FS-035 §4.1) — 권한 강제는 서버 책임. **프론트는 등록·수정·삭제 버튼을 권한과 무관하게 렌더한다**(FS-035 §7 #13).

## 3. 데이터 계약 (types.ts 대조)

| 타입 | 필드 | 비고 |
|---|---|---|
| `EmailCampaign` | `id` · `name` · **`subject`** · `senderId` · `senderEmail`(비정규화) · `senderName`(비정규화) · `segmentIds[]` · `recipientCount`(저장 시점 스냅샷) · `isAd` · `body` · **`includeUnsubscribe`** · `status` · `scheduledAt` · `stats` | 목록·단건이 같은 타입 — **상세 전용 필드가 없다**. SMS 대비 `subject`·`includeUnsubscribe` 추가, `hasImage`·`kind` 없음 |
| `EmailCampaignInput` | `EmailCampaign` − `id` · `senderEmail` · `senderName` · `recipientCount` · `stats` | **파생값 4개는 서버가 계산한다** |
| `MailStats` (`_shared`) | `SendStats`(`total`·`success`·`failed`) + **`opened`** · **`clicked`** | **오픈율·클릭율의 분모는 `total` 이 아니라 `success`(전달) 다**(`messaging.ts:396-405`) — 서버 집계도 같은 기준이어야 화면 수치가 맞는다 |
| `SendStatus` (`_shared`) | `'draft' \| 'scheduled' \| 'sending' \| 'sent' \| 'canceled'` | **입력은 `draft`·`scheduled` 만**(`emailSchema` 가 `z.enum(['draft','scheduled'])`) |
| `SenderEmail` (`_shared`) | `id` · `email` · `name` · `verified` | 도메인 인증(SPF/DKIM) 완료 발신자만 발송 가능 |
| `Segment` (`_shared`) | `id` · `label` · `recipientCount` · `description` | **SMS 와 같은 레지스트리를 공유한다** |

**서버가 계산해야 하는 파생값** (프론트 어댑터가 픽스처로 흉내 내는 것 — `data-source.ts:74-95`):
`senderEmail`·`senderName` = `senderId` 로 조회 · `recipientCount` = 세그먼트 수신자 **중복 제거 후** 합(§7.6) · `stats` = 신규 생성 시 `{0,0,0,0,0}`, 이후 전송·오픈·클릭 누적.

**계약에 없는데 필요한 것 — 수신거부 URL**: `includeUnsubscribe` 는 **불리언뿐**이다(`types.ts:22`). 실제 수신거부 URL·토큰이 도메인 모델에 없다. **서버가 발송 시 수신자별 수신거부 링크를 주입해야 한다**(§7.7) — 프론트 미리보기의 '수신거부'는 밑줄 친 텍스트일 뿐 링크가 아니다(`EmailPreview.tsx:113-116`).

### 3.1 상태 머신 (`_shared/messaging.ts:322-364`)

**BE-034 §3.1 과 동일하다** — `draft→scheduled→sending→sent`, 예약 취소 `→canceled`. 편집 가능 상태는 `draft`·`scheduled` 만(`isEditableSend`)이며 그 밖의 상태 수정은 **422 `CAMPAIGN_NOT_EDITABLE`**(§5 의 '422 상태위반' 축). 상태 입력은 `draft`\|`scheduled` 만 허용하고 `sending`·`sent`·`canceled` 를 바디로 보내면 400. `scheduled→sending→sent` 전이의 주체는 **서버**다.

**이메일 고유 — `sent` 이후에도 통계가 움직인다**: SMS 의 `stats` 는 전송 완료 시점에 확정되지만, 이메일의 `opened`·`clicked` 는 **발송 후 며칠에 걸쳐 계속 누적된다**. 즉 `sent` 는 '전송 종료'이지 '집계 종료'가 아니다. 프론트는 재조회 시점 값만 보이며(FS-035-EL-007.8) 이 사실이 계약에 명시돼야 한다 — **`sent` 캠페인의 GET 응답이 호출마다 달라질 수 있다**.

**실제 전송은 하지 않는다**: 이 화면은 메일을 보내지 않는다. 저장은 레코드 생성일 뿐이다(FS-035 §5).

### BE-035-EP-04 · 이메일 발송 수정
| 항목 | 내용 |
|---|---|
| 근거 (FS) | FS-035-EL-032(수정), EL-033 |
| 메서드·경로 | `PUT /api/marketing/email/:id` |
| 권한 | `admin` 만 |
| 멱등성 | 멱등(PUT 전체 치환) |
| 레이트리밋 | 분당 30회 |

**바디**: `EmailCampaignInput`(EP-03 동일 제약). **추가 제약 — 상태 위반(§3.1)**: 대상 캠페인의 현재 `status` 가 `sending`·`sent`·`canceled` 면 **422 `CAMPAIGN_NOT_EDITABLE`**. 발송된 캠페인의 제목·본문·대상을 사후 수정하면 발송 이력이 거짓이 되고, **오픈율/클릭율이 어떤 콘텐츠의 성과인지 알 수 없게 된다**(§7.2).

**낙관적 동시성**: `If-Match: <etag>`(또는 `updatedAt`) 필요. 불일치 → **412 `PRECONDITION_FAILED`**. 이미 삭제된 id → **409 `CONFLICT`**. 프론트는 둘을 같은 충돌 다이얼로그로 처리하고 **입력을 보존한다**(FS-035-EL-033). **단 현재 `EmailCampaign` 타입에 `updatedAt`/`version` 이 없고 `If-Match` 를 보내는 코드도 없다 — §7.11 이관.**

**응답 200/204**.

**에러**: 400 · 401 · 403 · 403 CSRF · 404 `EMAIL_CAMPAIGN_NOT_FOUND` · 409 `CONFLICT` · 412 `PRECONDITION_FAILED` · 422 `CAMPAIGN_NOT_EDITABLE` + EP-03 의 422 전부 · 429 · 500 · 504.

| 권한 | `admin` 만 |
| 멱등성 | **멱등이 아니다** — 프론트 어댑터가 이미 삭제된 id 에 409 를 던진다(`crud.ts:82-84`). BE-034 §7.2 와 같은 근거 |

### BE-035-EP-06 · 이메일 발송 실행 (트리거) — **프론트 호출부 없음**
| 항목 | 내용 |
|---|---|
| 근거 (심) | `data-source.ts:70` — `// 발송 실행: POST /api/marketing/email/:id/send (예약은 스케줄러가 실행) — 프론트는 트리거만.` |
| 근거 (FS) | **없다** — FS-035-EL-035 가 '미구현'으로 기록한 자리다 |
| 메서드·경로 | `POST /api/marketing/email/:id/send` |
| 권한 | `admin` 만. **별도 `marketing:send` 권한으로 분리 권고**(§7.3) |
| 멱등성 | **`Idempotency-Key` 헤더 필수**(§7.3) |
| 레이트리밋 | 분당 5회 + 일일 발송량 상한(BE-034 §7.7) |

**헤더**: `Idempotency-Key: <UUID v4>`(보존 24시간, 누락 시 400). **바디 없음.**

**응답 202 Accepted** — `{ campaignId, status: 'sending', acceptedAt }`. 전송 완료를 기다리지 않는다.

**서버가 발송 시점에 **반드시 재검증**하는 것**:
1. 캠페인 `status` 가 `draft`\|`scheduled` 인가 → 아니면 **422 `CAMPAIGN_NOT_SENDABLE`**(이미 `sending`/`sent` 면 중복 발송)
2. `senderId` 의 도메인 인증(SPF/DKIM)이 **여전히** 유효한가(§7.4)
3. 대상 수신자 각각의 **마케팅 수신 동의가 발송 시점에 유효한가**(§7.6)
4. 광고성이면 제목이 여전히 `(광고)` 로 시작하는가(§7.5)
5. 대상 규모가 상한 이내인가(BE-034 §7.7)
6. 본문·**제목**의 `#{...}` 토큰이 전부 화이트리스트 안에 있는가(§7.8)
7. **수신거부 링크를 수신자별로 주입했는가**(§7.7) — 프론트는 불리언만 준다

## 7. 핵심 판정

### 7.1 목록·일괄 계약 — BE-034 §7.1 과 동일
전량 반환 + 클라이언트 필터(`EmailListPage.tsx:116-119`), 전용 벌크 엔드포인트 없이 단건 DELETE 의 `settleAll` 병렬 호출. **프론트가 `useListState` 로 옮기는 변경과 동시에** 쿼리를 열 것을 권고한다.

### 7.2 동시성 — 토큰이 **없다**. 이메일은 근거가 더 강하다
BE-034 §7.2 와 같은 상태다 — 409(삭제 충돌)는 잡히지만 `updatedAt`/`version` 이 없어 **동시 수정은 마지막 쓰기가 조용히 이긴다**.

**이메일에서 이 문제가 더 무거운 이유**: `sent` 캠페인의 오픈율/클릭율은 **어떤 제목·본문이 그 성과를 냈는가**를 말한다. 발송 후 제목을 고칠 수 있으면 그 수치는 무엇의 성과인지 알 수 없는 값이 된다. §4 EP-04 의 422 `CAMPAIGN_NOT_EDITABLE` 이 1차 방어선이고, 낙관적 잠금이 2차다. **결정: `EmailCampaign` 에 `updatedAt` 을 추가하고 EP-04 가 `If-Match` 를 요구한다 — §7.11 이관.**

### 7.3 발송 실행 — 심은 있고 UI 는 없다. 멱등키는 **UI 가 붙는 순간 필수** 【보안 판정】
**BE-034 §7.3 과 동일 판정이다.** `/send` 심은 `data-source.ts:70` 에 실재하므로 EP-06 을 정식 명세했다. **그러나 프론트 호출부가 0건이다**(§6).

**따라서 '중복 발송 경합'은 현재 성립하지 않는다** — 두 번 누를 버튼이 없다. 이것을 지금 P0 위험으로 올리는 것은 사실이 아니다.

**그러나 발송 트리거를 구현하는 순간** 이것이 가장 위험한 쓰기가 된다. 메일은 회수되지 않고, 발송량 과금이 있으며, 법적 책임이 따른다. UI 구현 시 다음이 **동시에** 와야 한다: ① **`Idempotency-Key` 를 요청 함수 밖(ref)에서 생성**(`mutationFn` 안에서 만들면 재시도마다 새 키 = 전 수신자 재발송. `useCrudForm.ts:112-117` 이 이미 그 패턴이다) ② 서버 상태 재검사(2차 방어선) ③ 동기 제출 락 + 대상 수를 echo 하는 강화 confirm ④ `marketing:send` 권한 분리.

**이메일 고유 추가 권고 — 테스트 발송**: 이메일은 렌더가 클라이언트마다 다르고 오타의 대가가 크다(수만 명이 본다). '내게 보내보기'(`POST /api/marketing/email/:id/test-send`)를 두면 실발송 전 검증이 가능하다. **다만 이것은 심이 없으므로 이 계약에 넣지 않는다** — 발송 UI 설계 시 함께 논의할 후보로만 기록한다(§7.11 #8).

### 7.4 발신자 도메인 인증 — 서버가 정본 【보안 판정】
프론트는 미검증 발신자를 `disabled` 옵션으로 잠그고 스키마로도 막는다(`validation.ts:44-51` · `EmailFormPage.tsx:317`). **그러나 그 판정 근거인 `verified` 플래그가 브라우저 안 픽스처 배열이다**(`_shared/store.ts:69-83`).

**이메일은 SMS 보다 이 문제가 명확하다**: SMS 의 발신번호 사전등록은 '등록했는가'라는 **상태**지만, 이메일의 `verified` 는 **SPF/DKIM DNS 레코드가 지금 유효한가**라는 **런타임 사실**이다 — 브라우저가 알 수 있는 값이 아니다. DNS 레코드는 언제든 만료·변경된다.

**서버가 정본이다.** EP-03/EP-04 는 저장 시점에, **EP-06 은 발송 시점에 다시** 인증 상태를 확인하고 유효하지 않으면 422 `SENDER_NOT_VERIFIED`. 미인증 도메인으로 발송하면 스팸 처리되어 **도메인 평판이 훼손**되고, 그 피해는 이후 모든 메일에 미친다.

### 7.5 광고 규제 — (광고) 제목 표기는 서버 강제. **야간 제한 부재는 명시적 결정이 필요하다** 【보안 판정】
정보통신망법 제50조 제4항은 광고성 정보에 `(광고)` 표기를 강제한다. 프론트는 **제목** 앞에 그것을 요구한다(`validation.ts:63-73` — `hasAdPrefix(subject)`). **서버가 독립적으로 다시 강제한다** — 프론트 스키마는 브라우저 코드이고 우회된다. EP-06 발송 시점에도 재검사한다(그 사이 제목이 바뀌었을 수 있다).

**SMS 와 비대칭인 두 지점을 기록한다:**

1. **야간 전송 제한이 없다.** `emailSchema` 에 `isNightAt` 검사가 없다(`validation.ts:85-105`) — SMS 는 있다(`sms/validation.ts:106-114`). 공용 규칙(`messaging.ts:252-267`)이 존재하는데 이메일만 소비하지 않는다. **의도된 것일 수 있다**: 제50조 제3항의 야간 제한은 '전자적 전송매체'를 넓게 규정하지만, 실무상 수신자를 깨우는 채널(SMS·앱푸시)에 적용하고 이메일은 제외하는 해석이 일반적이다. **그러나 그 판단 근거가 코드 주석에도, 어떤 문서에도 없다.** 코드는 그저 그 검사를 빠뜨린 것처럼 보인다. **결정 요청: 이메일 야간 제한 적용 여부를 법무 확인 후 명시하라** — 적용한다면 서버가 `Asia/Seoul` 기준으로 강제하고 프론트 스키마에도 추가한다. 적용하지 않는다면 **그 근거를 `emailSchema` 주석과 이 계약에 남긴다**(§7.11 #4).
2. **무료수신거부 문구 검사가 없다.** SMS 는 본문에 '수신거부' 문자열을 요구하지만(`messaging.ts:243-245`) 이메일은 대신 **수신거부 링크 토글**을 쓴다(§7.7). 채널 관례상 타당하다 — 이메일의 수신거부는 문구가 아니라 링크다.

**시각 기준**: 야간 제한을 적용하지 않더라도 **과거 예약 검사는 남는다**. 프론트는 `Date.now()` 로 판정하지만(`validation.ts:17-21`) 검증 시점과 저장 시점 사이에 시간이 흐른다 — **서버가 재검증**하고 `scheduledAt` 을 오프셋 포함 ISO 8601 로 주고받아야 한다(현재 프론트는 `datetime-local` 의 오프셋 없는 문자열을 저장한다 — §7.11 #4).

### 7.6 수신 동의(opt-in) — **발송 시점에 서버가 재확인한다** 【보안 판정 · TOCTOU】
**BE-034 §7.6 과 동일한 판정이며, BE-010 §7.7(카테고리 참조 무결성)의 논증 구조를 따른다.** 두 화면이 **같은 세그먼트 레지스트리를 공유**하므로 판정도 하나여야 한다.

프론트는 `SegmentPicker` 로 세그먼트를 고르고 `recipientCount` 를 합산해 대상 수를 보인다. **그러나 세그먼트 선택은 UX 편의이고 서버가 정본이다** — 프론트가 세그먼트를 조회한 시점(`EmailFormPage.tsx:188` 의 `useMemo` 로 **1회 고정**)과 저장 시점, 그리고 **실제 발송 시점** 사이에 시간이 흐른다. 그 사이에 수신자가 수신 동의를 **철회**하거나 수신거부 링크를 **클릭**할 수 있다. 예약이면 그 간격이 며칠이다(픽스처 예약이 3주 뒤다).

**이메일 고유 — 수신거부 링크가 이 창을 스스로 넓힌다**: 이전 캠페인의 수신거부 링크를 클릭한 사람은 **그 순간부터** 수신 거부자다. 그 클릭은 이 캠페인을 저장한 뒤에도 계속 일어난다. **즉 이메일은 자기가 만든 opt-out 을 발송 전에 반드시 반영해야 하는 채널이다.**

**결정: 서버는 발송 트랜잭션 안에서 수신자별 동의·수신거부 상태를 재조회하고, 유효하지 않은 수신자를 발송 대상에서 제외한다.** 세그먼트가 그 사람을 포함하더라도 그렇다. 제외 건수는 감사 로그에 남긴다(§7.9). 프론트의 대상 수는 **예상치**이며 화면 문구도 이미 그것을 인정한다('발송 시 1회로 합산됩니다' — `SegmentPicker.tsx:142-145`).

**중복 제거도 서버 몫**: `totalRecipients` 는 세그먼트별 인원의 **단순 합**이다(`messaging.ts:411-419`). 서버는 **합집합**으로 대상을 정하고 `recipientCount` 를 그 값으로 돌려준다.

### 7.7 수신거부 링크 — 불리언이 아니라 **서버가 주입하는 URL** 이다 【보안 판정】
`includeUnsubscribe` 는 **불리언뿐**이다(`types.ts:22`). 미리보기의 '수신거부'는 밑줄 친 텍스트일 뿐 링크가 아니다(`EmailPreview.tsx:113-116`). 즉 **프론트는 '수신거부 링크를 넣겠다'는 의사만 표현하고 실제 링크는 존재하지 않는다.**

**서버가 발송 시 수신자별 수신거부 URL 을 본문에 주입한다.** 요구사항:
- **수신자마다 다른 토큰** — 서명된 불투명 토큰(수신자 id 를 유추할 수 없어야 한다). 공용 링크는 남이 남의 수신을 끊을 수 있게 만든다.
- **원클릭 처리** — 클릭 하나로 수신거부가 완료돼야 한다(로그인 요구 금지). `List-Unsubscribe` 헤더(RFC 8058)도 함께 실을 것을 권고한다.
- **즉시 반영** — 그 처리가 §7.6 의 발송 시점 재확인에 곧바로 반영돼야 한다.
- **토큰 만료 정책** — 만료된 토큰의 처리를 정의한다.

**`includeUnsubscribe: false` 는 광고성 여부와 무관하게 거절해야 하는가?** 현재 프론트는 **항상 거절한다**(`validation.ts:74-84` — `isAd` 를 보지 않는다). 그래서 정보성 메일(주문 확인·배송 안내)도 수신거부 링크 없이는 저장이 안 된다. **법적으로 수신거부 의무는 광고성 정보에 붙는다** — 거래 관계에서 발생하는 정보성 메일은 그 대상이 아니다. 즉 **현재 규칙은 법이 요구하는 것보다 강하다.** 안전한 기본값이므로 그대로 둬도 무방하나, 이 화면이 정보성 메일도 다루게 되면(픽스처의 '장바구니 리마인드'가 이미 `isAd: false` 다) 재검토가 필요하다. **서버는 최소한 `isAd === true` 에서 `UNSUBSCRIBE_REQUIRED` 를 강제한다** — 그 위의 정책은 협의 대상이다(§7.11 #6).

### 7.8 치환변수 화이트리스트 · HTML 정제 【보안 판정】
**변수 화이트리스트는 BE-034 §7.8 과 동일 판정이다.** 본문은 자유 텍스트라 관리자가 `#{잔액}` 을 타이핑하면 프론트가 막지 않고, 미리보기에 토큰이 그대로 보일 뿐이다(`applyVariableSamples` 는 모르는 변수를 그대로 둔다 — `messaging.ts:130-136`). **서버가 저장 시(EP-03/04) 모든 `#{...}` 를 화이트리스트와 대조해 목록 밖이면 422 `UNRESOLVED_VARIABLE`, 발송 시(EP-06) 재검사한다.** 치환은 서버가 수행하고 값을 **이스케이프**한다.

**이메일 고유 — 제목도 검사 대상이다.** 미리보기가 제목을 치환하고(`EmailPreview.tsx:97`) 픽스처 템플릿 제목이 실제로 `#{이름}` 을 쓴다(`store.ts:120`). 그런데 **제목에는 변수 삽입 바가 없다**(`VariableInsertBar` 가 본문 카드에만 — `EmailBodyCard.tsx:91-96`) — 즉 제목 변수는 **지원되지만 손으로 타이핑해야 한다**. 오타 확률이 높은 경로다. `error.fields[].name = 'subject'` 로 거절해야 인라인에 붙는다.

**저장형 XSS — 지금은 성립하지 않지만 HTML 본문이 오면 즉시 필수가 된다** 【보안 판정】:
현재 본문은 **평문**이고 미리보기도 `pre-wrap` 텍스트로만 렌더한다(`EmailPreview.tsx:60,110`) — `dangerouslySetInnerHTML` 이 이 화면 어디에도 없다. **관리자 화면에서는 XSS 가 성립하지 않는다.** 그러나 두 갈래로 위험이 실재한다:
1. **메일 클라이언트가 정본이다** — 관리자가 평문에 `<img src=x onerror=...>` 를 넣고 서버가 그것을 HTML 파트로 발송하면, **수신자의 메일 클라이언트**가 그것을 해석한다. 관리자 화면이 안전한 것은 관리자만 안전하다는 뜻이지 수신자가 안전하다는 뜻이 아니다.
2. **HTML 에디터가 로드맵에 있다**(FS-035 §7 #11). 그 순간 본문은 마크업이 되고 미리보기는 그것을 렌더해야 한다.

**결정: 서버가 저장 시 본문을 정제한다** — 허용 태그 화이트리스트 밖의 마크업 · `javascript:`/`data:` 스킴 · 이벤트 핸들러 속성을 제거한다(BE-009 §7.3 · BE-010 §7.2 와 같은 판정). 계약은 '저장된 본문에 실행 가능한 스크립트가 없다'는 관측 동작만 정한다. **제목도 정제한다** — 제목은 헤더로 나가므로 **개행 문자 주입(헤더 인젝션)** 을 반드시 막는다: `subject` 에 `\r`·`\n` 이 있으면 400 으로 거절한다. 이것은 HTML 본문 도입 전에도 **지금 필요한 방어선**이다.

### 7.9 발송 감사 로그 — 법적 증빙 【보안 판정】
**BE-034 §7.9 와 동일 판정이다.** 캠페인 레코드에 작성자·발송 실행자·실행 시각 필드가 아예 없다(`types.ts:7-26`).

**서버가 남긴다**: 발송 실행 감사(실행 주체·시각·캠페인 스냅샷·최종 대상 수·**동의 검증 결과와 제외 사유**·멱등키) · 수신자별 전송 기록(동의 근거 + 결과) · 캠페인 변경 감사 · **캠페인 삭제와 독립적인 보존**.

**이메일 고유 — 추적 데이터가 개인정보다**: `opened`·`clicked` 는 집계 숫자로만 화면에 오지만(`MailStats`), 그것을 만들려면 서버가 **수신자별 오픈·클릭 이벤트**를 저장한다. 그 데이터는 개인정보이며 별도 보존 기간·목적 제한이 필요하다. **관리자 화면에 수신자별 오픈/클릭을 노출하지 않는 현재 설계가 옳다** — 집계만 내린다. 추적 픽셀·링크 래핑을 도입할 때 그 사실이 개인정보 처리방침에 반영돼야 한다.

**`sent` 삭제는 soft-delete 여야 한다** — 오픈/클릭 이벤트가 발송 후에도 계속 도착하는데(§3.1) 레코드를 지우면 귀속시킬 곳이 없다. 현재 프론트는 상태와 무관하게 hard-delete 를 호출한다(`crud.ts:86`).

### 7.10 발신자·세그먼트·템플릿 — 심 없음. 지어내지 않는다
셋 다 프론트가 **동기 픽스처 함수를 직접 호출**한다 — 어댑터도 `// TODO(backend)` 주석도 없다(§6). 그래서 **엔드포인트를 명세하지 않는다**(EP-07/08/09 = 심 없음).

**우선순위는 BE-034 §7.10 과 같고, 발신자는 이메일 쪽이 더 급하다** — `verified` 가 SPF/DKIM 런타임 사실이라 브라우저 배열로는 애초에 표현될 수 없는 값이다(§7.4). **세그먼트는 SMS 와 공유하므로 심을 뚫으면 두 화면이 함께 산다.**

### 7.12 세션 만료와 작성 중 입력
401 은 쿼리 계층 한 곳에서 가로채 재인증 경로로 보낸다(`queryClient.ts:36-38`). **작성 중이던 폼 입력은 보존되지 않는다** — 앱이 스스로 이동하므로 미저장 가드(FS-035-EL-034)가 발화하지 않는다. 이메일 폼은 제목·본문(최대 5,000자)·세그먼트를 모으는 긴 작업이라 SMS 보다 손실이 크다. 서버 계약의 문제는 아니나(EXC-19 · 앱 전역) 이 화면이 그 피해자임을 기록한다.
