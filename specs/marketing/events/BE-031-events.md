# BE-031. 이벤트 백엔드 기능 명세

## 1. 개요

| 항목 | 내용 |
|---|---|
| 대상 화면 | FS-031 이벤트 (`/marketing/events` · `/new` · `/:id/edit`) |
| 범위 | 이벤트 목록 조회, 단건 조회, 등록, 수정, 삭제 |
| 범위 밖 | **일괄 삭제 전용 경로** — 프론트가 N 개의 단건 DELETE 를 병렬로 낸다(§7.5 판정). **서버측 필터·검색·페이징** — 심에 파라미터가 없다(§7.4 판정). **쿠폰·적립 발급** — 혜택 상세는 자유 텍스트이며 이 계약은 어떤 혜택도 발급하지 않는다(§7.2). **배너 조회** — 연동 배너명은 문자열 사본이고 배너 도메인 API 를 호출하지 않는다(§7.2). **이미지 업로드** — 이 화면에 이미지 필드가 없다(§7.8) |
| 전제 | 인증은 세션 쿠키 기반. 모든 경로는 `/api` 프리픽스. 응답 본문은 `application/json; charset=utf-8` |
| 프론트 어댑터 | `apps/admin/src/pages/marketing/events/data-source.ts` (`eventAdapter`) |
| 도메인 타입 | `apps/admin/src/pages/marketing/events/types.ts` · 공용 enum `apps/admin/src/pages/marketing/_shared/campaign.ts` |
| 검증 정본 | `apps/admin/src/pages/marketing/events/validation.ts` (`eventSchema`) — 서버는 이 규칙을 **재구현**한다(클라이언트 검증은 UX 이지 강제가 아니다) |
| 자매 문서 | BE-032 (프로모션) — 같은 캠페인 골격. 기간·상태 규칙이 동일하다 |

> **에러 봉투·권한 모델 상속**: BE-003 §2(공통 에러 봉투 · 공통 에러코드)·§3(인증 · 권한 모델 · CSRF · 타임아웃 상한)을 **그대로 상속한다.** 아래 §2 는 이 도메인의 고유 차이만 기술한다.

## 2. 공통 (상속)

| 항목 | 내용 |
|---|---|
| 에러 봉투 | BE-003 §2 그대로 — `{ error: { code, message, fields?, traceId } }`. 이 도메인은 자체 응답 형식을 정의하지 않는다 |
| 공통 에러코드 | BE-003 §2.1 그대로 — `VALIDATION_FAILED`(400) · `UNAUTHENTICATED`(401) · `FORBIDDEN`(403) · `CSRF_TOKEN_INVALID`(403) · `NOT_FOUND`(404) · `CONFLICT`(409) · `UNPROCESSABLE`(422) · `RATE_LIMITED`(429) · `INTERNAL_ERROR`(500) · `REQUEST_TIMEOUT`(504) |
| 도메인 고유 에러코드 | `EVENT_NOT_FOUND`(404) — 이벤트 id 가 없거나 삭제됨. `EVENT_CONFLICT`(409) — 동시 수정·이미 삭제된 대상(§7.3) |
| 권한 역할 | `admin` = 조회 + 등록 + 수정 + 삭제. `operator` = **조회 계열만**(목록·단건). 등록·수정·삭제는 **403**. 마케팅 도메인 권한이 없는 인증 관리자 = §3.2 |
| CSRF | BE-003 §3.3 그대로 — POST·PUT·DELETE 는 `X-CSRF-Token` 헤더 요구. 누락·불일치 시 403 `CSRF_TOKEN_INVALID`. **어댑터 함수 시그니처는 바뀌지 않는다**(본문이 헤더를 싣는다) |
| 타임아웃 상한 | 조회 5초 · 쓰기 5초, 초과 시 504 `REQUEST_TIMEOUT`. **프론트에는 상한이 없다**(`AbortSignal.timeout` 0건 — FS-031 §7 #19). 서버 상한이 유일한 천장이다 |
| 프론트 역할 분기 | **없다.** 이 화면은 역할에 따라 요소를 숨기거나 비활성하지 않는다(FS-031 §4.1). 쓰기 컨트롤이 `operator` 에게도 보이므로 **403 은 정상 경로로 자주 발생한다** — 서버는 이를 예외가 아닌 기대 응답으로 다룬다(§7.7) |

### 2.1 403 vs 404 은닉 정책 (§3.2 위치)

BE-003 §3.2 의 두 원칙을 이 도메인에 적용한다.

| 리소스 | 권한 부족 시 응답 | 근거 |
|---|---|---|
| `GET /api/marketing/events` (컬렉션) | 403 `FORBIDDEN` | 이벤트 컬렉션의 존재는 비밀이 아니다 |
| `POST /api/marketing/events` | 403 `FORBIDDEN` | 컬렉션 쓰기. 은닉할 개별 리소스가 없다 |
| `GET /api/marketing/events/:id` | 마케팅 읽기 권한 없음 → 403 `FORBIDDEN` | **회원과 다르다** — 이벤트는 개인정보가 아니라 **곧 고객에게 공개될 마케팅 자산**이다. 존재를 숨길 이유가 없고, 404 로 은닉하면 정당한 운영자가 '삭제됐다' 고 오판한다(FS-031-EL-028 이 404 를 '이미 삭제되었을 수 있습니다' 로 렌더한다) |
| `PUT` · `DELETE /api/marketing/events/:id` | `operator` → 403 `FORBIDDEN` / 마케팅 읽기 권한 없음 → 403 `FORBIDDEN` | 위와 동일 |

> **FS-031 §7 #8 을 이 표로 확정한다.** 이 도메인에서 **404 는 오직 '정말 없다'** 를 뜻한다 — 권한 은닉에 쓰지 않는다. 프론트가 404 를 '이미 삭제되었을 수 있습니다' 로 번역하므로(§6 대조), 404 를 은닉에 재사용하면 그 문구가 거짓말이 된다.

## 3. 데이터 계약

`types.ts` 의 `MarketingEvent` 와 `_shared/campaign.ts` 의 enum 을 그대로 옮긴다.

| 필드 | 타입 | 필수 | 제약 (검증 정본 = `eventSchema`) | 비고 |
|---|---|---|---|---|
| `id` | string | O (응답) | 서버 배정. 픽스처는 `ev-<seq>` | 요청 본문에 없다(`MarketingEventInput = Omit<MarketingEvent,'id'>`) |
| `title` | string | O | `trim()` 후 1자 이상 · **80자 이하** | 고객 노출 — §7.1 |
| `startAt` | string | O | `YYYY-MM-DD`, 실재 날짜 | 시각·타임존 성분이 없다 — §7.6 |
| `endAt` | string | O | `YYYY-MM-DD`, 실재 날짜, **`endAt >= startAt`** | 같은 날 허용(하루짜리 이벤트) |
| `phase` | `'upcoming' \| 'ongoing' \| 'ended'` | O | 이 3값만 | 관리자 지정값. 기간과 어긋나도 저장된다 — §7.6 |
| `target` | string | O | `trim()` 후 1자 이상 · **60자 이하** | 자유 텍스트. 구조화된 세그먼트가 아니다 |
| `benefitType` | `'none' \| 'coupon' \| 'points'` | O | 이 3값만 | 기본값 `'none'` |
| `benefitDetail` | string | O (필드 자체) | `benefitType !== 'none'` 이면 `trim()` 후 1자 이상. `'none'` 이면 프론트가 `''` 로 비워 보낸다. **길이 상한 없음** — §7.9 | 자유 텍스트. 쿠폰 id·금액이 아니다 — §7.2 |
| `bannerLinked` | boolean | O | — | — |
| `bannerLabel` | string | O (필드 자체) | `bannerLinked === true` 면 `trim()` 후 1자 이상. `false` 면 프론트가 `''` 로 비워 보낸다. **길이 상한 없음** — §7.9 | 배너 id 참조가 아니라 **문자열 사본** — §7.2 |
| `description` | string | O (필드 자체) | `trim()` 후 **1000자 이하**. 빈 문자열 허용 | 고객 노출 — §7.1 |

> **모든 필드가 요청 본문에 항상 존재한다.** 프론트 `toInput` 이 폼 값 전량을 매번 조립하므로 PATCH 시맨틱(부분 수정)이 없다 — §7.10 판정.
>
> **응답에 없는 것**: `createdAt` · `updatedAt` · `version` · `createdBy`. 그래서 낙관적 동시성이 불가능하다 — §7.3.

### BE-031-EP-06 · 이벤트 일괄 삭제 — **심 없음 (미정)**

`data-source.ts` 에 일괄 삭제 심이 없다. 프론트는 `settleAll` 로 **BE-031-EP-05 를 N 번 병렬 호출**한다(`useCrudList.tsx:132-148` · `crud.ts:235-236`). 따라서 이 계약에 일괄 엔드포인트는 **존재하지 않는다**. 도입 여부·트랜잭션 경계·부분 실패 응답 형태는 §7.5 판정으로 이관한다.

### BE-031-EP-07 · 목록 필터·검색·페이징 — **심 없음 (미정)**

`fetchAll(signal)` 이 인자를 받지 않는다. 상태 필터·키워드 검색·페이지는 전부 브라우저 메모리 연산이다(`filterEvents` · `searchEvents`). 서버 쿼리 파라미터 계약은 §7.4 판정으로 이관한다.

### BE-031-EP-08 · 배너 목록 조회 / 쿠폰·적립 발급 — **심 없음 (미정)**

`bannerLabel` · `benefitDetail` 은 자유 텍스트이며 어떤 도메인 API 도 호출하지 않는다. §7.2 판정으로 이관한다.

## 7. 핵심 판정

### 7.1 【보안 판정】 저장형 XSS — 이벤트 본문은 고객 화면에 그대로 나간다

`title` · `description` · `target` · `benefitDetail` 은 관리자가 자유롭게 입력하고, 이벤트의 목적상 **고객 대면 화면(이벤트 상세·배너·목록)에 그대로 노출된다.** 현재 프론트는 어떤 정제도 하지 않는다 — `eventSchema` 가 검사하는 것은 **길이와 공백뿐**이다(`validation.ts:12,21-25`). 관리자 계정 하나가 탈취되거나 내부자가 악의를 가지면 `description` 에 `<script>` · `<img onerror>` · `javascript:` 링크를 심어 **전체 고객에게 배포**할 수 있다.

**판정**:
1. **서버가 저장 시점에 정제한다.** 클라이언트 검증은 UX 이지 강제가 아니다 — 어댑터를 우회한 직접 호출을 막지 못한다.
2. **정책은 '허용 목록'이다.** `description` 은 서식이 필요할 수 있으므로 태그 화이트리스트(`<b> <i> <br> <p> <ul> <li> <a>`)를 두고 그 밖은 이스케이프한다. `title` · `target` · `benefitDetail` 은 **평문 전용** — 태그를 허용할 이유가 없으므로 전량 이스케이프한다.
3. **`<a href>` 는 스킴 화이트리스트**(`http:` · `https:` · `mailto:` 만)를 적용한다. `javascript:` · `data:` · `vbscript:` 는 거절한다.
4. **저장값과 표시값을 분리한다** — 원문을 저장하고 렌더 시점에 이스케이프하되, **정제 실패 시 저장을 거절**(400 `VALIDATION_FAILED`)한다. 조용히 태그를 지우면 관리자는 자기가 쓴 것이 왜 사라졌는지 모른다.
5. 이 판정은 **BE-032 §7.1(프로모션)과 동일**하게 적용한다 — 두 도메인의 본문이 같은 고객 화면 계열로 흐른다.

### 7.2 【보안 판정】 참조가 아니라 문자열이다 — 배너 연동·혜택의 무결성 부재

`bannerLabel` 은 hint 로 '배너 관리에서 노출 중인 배너와 연결됩니다.' 라고 **약속하지만**(FS-031-EL-023), 실제로는 검증 없는 자유 텍스트 사본이다. `benefitDetail`('3,000 적립금') 도 쿠폰·적립 레코드를 참조하지 않는다. 배너 도메인 API 호출이 코드에 **0건**이다.

**판정**:
1. **현재 계약은 문자열을 그대로 받는다** — 이 문서는 존재하지 않는 배너 API 를 지어내지 않는다.
2. 그러나 이 상태의 결과를 명시한다: 배너가 개명·삭제돼도 이벤트는 옛 문자열을 들고 있고, **끊긴 참조가 조용히 남는다.** 고객 화면이 이 문자열로 배너를 찾으면 아무것도 못 찾는다.
3. **후속에서 `bannerId: string | null` 로 승격**하고 저장 시 **참조 무결성**(존재·노출 상태)을 검증할 것을 이관한다. 그때 **TOCTOU** 를 유의한다 — 검증과 저장 사이에 배너가 삭제될 수 있으므로 같은 트랜잭션 안에서 FK 제약으로 강제한다. 배너 삭제 측에서도 참조 중인 이벤트가 있으면 409 로 막는다(FAQ 카테고리의 사용 중 삭제 차단과 같은 결 — BE-010).
4. 마찬가지로 `benefitType !== 'none'` 이면 **`couponId`/`pointAmount` 로 승격**해 실제 발급과 잇는다. 지금은 **어떤 혜택도 발급되지 않는다** — 관리자가 '3,000 적립금' 이라 써도 아무 일도 일어나지 않는다. 이 사실이 화면에 고지되지 않는 것이 더 큰 위험이다.

### 7.3 【판정】 낙관적 동시성 토큰 부재 — 마지막 저장이 이긴다

`MarketingEvent` 에 `updatedAt` · `version` · `ETag` 가 **없다**(§3). 그래서 `If-Match` 를 실을 수단 자체가 없고, 프론트에도 `If-Match` 배선이 없다(`apps/admin/src` 에서 이 도메인은 0건). 어댑터가 잡는 409 는 **'대상이 사라진 경우'뿐**이다(`crud.ts:71-73`).

재현: 두 탭에서 같은 이벤트의 수정 폼을 연다 → 탭 A 가 기간을 바꿔 저장 → 탭 B 가 대상을 바꿔 저장 → **탭 B 가 탭 A 의 기간 변경을 조용히 되돌린다.** 경고도, 충돌 다이얼로그도 없다. PUT 이 전 필드를 치환하므로(§7.10) 손실은 **필드 단위가 아니라 레코드 전체**다.

**판정**:
1. **서버가 `version: number`(또는 `updatedAt` 기반 ETag)를 응답에 포함**한다. GET(EP-01·EP-02) 응답의 각 이벤트에 실린다.
2. **PUT(EP-04)·DELETE(EP-05)는 `If-Match: <version>` 헤더를 요구**한다. 누락 시 400 `VALIDATION_FAILED`(`error.fields[].name = "If-Match"`), 불일치 시 **412 `PRECONDITION_FAILED`**.
3. 프론트는 **이미 준비돼 있다** — `isConflict` 가 409 와 412 를 **같은 UX 로 수렴**시키고(`http-error.ts:102` 주석), `useCrudForm.ts:158-172` 가 충돌 다이얼로그를 연다. 즉 **서버가 412 를 주기 시작하면 프론트 화면 코드는 그대로 두고 어댑터만 헤더를 실으면 된다.**
4. 남는 프론트 작업은 ① `types.ts` 에 `version` 추가 ② 어댑터가 조회 시 보관하고 쓰기 시 헤더로 싣기 — 회원 메모(BE-004 §7.4)가 이 패턴의 정본이다.
5. FS-031 §7 #15 를 이 판정으로 확정한다.

### 7.4 【판정】 목록이 전량 조회다 — 필터·검색·페이징이 서버 계약에 없다

심(`fetchAll(signal)`)에 파라미터가 없다. 상태 필터·키워드는 **브라우저 메모리 연산**이고(`filterEvents` · `searchEvents`), 페이지네이션은 아예 없다(FS-031 §7 #5).

**판정**:
1. **현재 계약은 전량 반환이다** — 심에 없는 쿼리 파라미터를 지어내지 않는다.
2. 이 상태의 천장을 명시한다: 이벤트가 수천 건이 되면 **매 진입마다 전량이 네트워크를 타고 전 행이 DOM 에 올라간다.** 검색은 서버에 도달하지 못하므로 인덱스의 이득이 없다.
3. **후속 계약(권고)**: `?phase=` · `?q=` · `?page=` · `?size=` 를 도입하고 응답을 `{ items, total, page, size }` 로 감싼다. 이때 **IA-13(URL 상태) · IA-04(페이지네이션) · COMP-10(디바운스) 이 함께 해소돼야** 의미가 있다 — 서버 검색만 붙이고 디바운스가 없으면 자모마다 요청이 나간다(현재는 클라이언트 필터라 그 위험이 가려져 있다).
4. 그때까지 서버는 **정렬(시작일 내림차순)을 보장**하되, 프론트가 수신 후 `sortEvents` 로 재정렬하므로 이를 신뢰 경계로 삼지 않는다.

### 7.5 【판정】 일괄 삭제 심이 없다 — N 개의 병렬 DELETE 다

`settleAll` 이 선택 전량에 대해 `remove(id)` 를 **동시에** 발사한다(`crud.ts:235-236`). 일괄 전용 엔드포인트도, 선택 상한도, 진행률도, 취소도 없다(FS-031 §7 #10).

**판정**:
1. **일괄 엔드포인트는 심에 없다 — 미정.** 이 계약에 추가하지 않는다.
2. **DELETE(EP-05)는 원자적이어야 한다** — 부분 삭제 상태를 남기지 않는다. 클라이언트 abort 로 연결이 끊겨도 그렇다.
3. **`Retry-After` 를 반드시 실어라.** 세션 분당 30회 한도에서 31건을 선택하면 **일부가 429 로 실패**하고, 프론트는 이를 'N건 중 M건을 삭제하지 못했습니다' 로만 보고한다 — 운영자는 그것이 과부하 때문인지 권한 때문인지 알 수 없다. 이 UX 결함은 프론트 몫이지만(EXC-10), 서버는 최소한 구분 가능한 코드를 줘야 한다.
4. **멱등하지 않게 설계한다** — 이미 없는 id 에 204 를 주면 '지웠다' 와 '이미 없었다' 가 구분되지 않는다. 409 를 준다(픽스처가 이미 그렇게 한다).
5. 후속에서 `POST /api/marketing/events/bulk-delete` 를 도입한다면 **실패 id 배열을 응답에 실어야** EXC-10('실패분만 재시도')이 성립한다.

### 7.6 【판정】 상태(phase)의 정본이 정해지지 않았다

`phase` 는 관리자가 **지정**하는 값이고, `derivePhase(startAt, endAt, today)` 는 기간에서 **파생**하는 값이다. 둘이 어긋나면 목록이 '기간상 XX' warning 배지로 알릴 뿐, **폼은 저장을 막지 않는다**(FS-031-EL-018). 즉 '2026-03-01 ~ 2026-03-31' 기간에 '진행' 상태인 이벤트가 저장될 수 있고, 실제 픽스처에도 정합한 데이터만 들어 있다.

**판정**:
1. **서버가 정본을 정해야 한다.** 세 선택지:
   - (a) **파생이 정본** — `phase` 를 저장하지 않고 조회 시 계산한다. 그러면 지정 필드와 배지가 모두 사라진다(화면 변경 수반).
   - (b) **지정이 정본** — 지금 동작. 관리자가 기간과 무관하게 노출을 통제한다(예: 기간은 남았지만 조기 종료). 이 경우 **'기간상 XX' 배지는 결함이 아니라 기능**이다.
   - (c) **지정 + 서버 강제** — 저장 시 `phase` 와 기간의 정합을 422 로 강제한다. 그러면 조기 종료가 불가능해진다.
2. **권고는 (b)** — 마케팅 운영에서 '기간은 남았지만 지금 멈춘다' 는 실재하는 요구다. 다만 그렇다면 **파생 상태의 타임존을 서버가 정해야 한다**(§7.7 아래 ERP-09 판정).
3. A01 과 함께 확정할 것. FS-031 §7 #13 을 이관한다.

### 7.7 【판정】 403·429 에 전용 문구가 없다 — 프론트가 모든 실패를 한 문구로 뭉갠다

서버가 403 `FORBIDDEN` 을 정확히 주고 `HttpError` 가 status 를 싣는데도, 이 화면은 **그 status 를 읽지 않는다**. 목록은 '이벤트 목록을 불러오지 못했습니다.'(`CrudListShell.tsx:159`), 폼은 '저장하지 못했습니다.'(`useCrudForm.ts:188`) 하나로 수렴한다. `operator` 는 삭제 버튼을 **보고 누를 수 있으므로**(§2 · EXC-03 gap) 이 403 은 드문 예외가 아니라 **일상 경로**다.

**판정**:
1. **서버 책임 아님 — 프론트 결함이다.** 이 문서는 서버가 줄 것을 정확히 규정하고(`FORBIDDEN` · `RATE_LIMITED` + `Retry-After`), 문구 분기를 A11 로 이관한다.
2. 다만 **서버가 도울 수 있는 것**: `error.message` 를 사용자에게 그대로 보여줄 수 있는 문구로 쓴다(BE-003 §2). 프론트가 status 분기를 붙이기 전이라도, 최소한 충돌 다이얼로그는 이미 `error.message` 를 그대로 표시한다(`useCrudForm.ts:161-163`).
3. **동시에 쓰기 컨트롤 권한 게이팅(EXC-03)이 붙으면 403 빈도가 급감한다** — 근본 해결은 그쪽이다.

### 7.8 【판정】 이미지 업로드 심이 없다 — 이 화면에 이미지 필드가 없기 때문이다

**이벤트 등록/수정 폼에 이미지 업로드 필드가 없다.** `ImageUploadField` 를 import 하지 않고, 목록에도 이미지 열이 없다(`EventListPage.tsx:4` 주석이 명시). '배너 연동' 은 배너 **이미지**가 아니라 배너 **이름 문자열**이다(§7.2). 따라서 `data-source.ts` 에 업로드 심이 없는 것은 누락이 아니라 **정합한 상태**다.

**판정**:
1. **업로드 엔드포인트 — 심 없음(미정).** 지어내지 않는다.
2. 그러나 **후속에서 이벤트 배너 이미지를 도입하면** 반드시 다음을 함께 계약한다:
   - `ImageUploadField` 는 파일을 서버로 보내지 않고 `URL.createObjectURL` 로 **`blob:` URL 을 만든다** — 업로드 심 없는 화면이 이 컴포넌트를 붙이면 저장값이 `blob:` 이 되고 **새로고침 시 깨진다.** 이벤트 이미지는 고객 화면에 나가므로 이 결함은 고객에게 노출된다.
   - **서버는 이미지 URL 의 스킴을 화이트리스트로 강제한다** — `https:` 와 자체 CDN 오리진만 허용하고, **`blob:` · `data:` · `javascript:` · `file:` 은 400 `VALIDATION_FAILED` 로 거절**한다. 이는 §7.1 의 XSS 방어와 같은 축이다(`data:text/html` 은 스크립트를 실을 수 있다).
   - 업로드는 별도 엔드포인트(`POST /api/uploads`)로 분리하고 이벤트 저장은 **그 결과 URL 만** 받는다.

### 7.9 【판정】 길이 상한 없는 필드 2개 — 서버가 상한을 정해야 한다

`benefitDetail` 과 `bannerLabel` 에 **길이 상한이 없다**(`validation.ts:18-19` 는 `z.string()` 일 뿐이고, refine 은 공백 여부만 본다). `title`(80) · `target`(60) · `description`(1000) 과 대조적이다. 폼에도 `maxLength` 속성이 없다.

**판정**: 서버가 상한을 정하고 400 `VALIDATION_FAILED` 로 강제한다. **권고: `benefitDetail` 100자 · `bannerLabel` 100자.** 상한 없는 텍스트 필드는 저장소·인덱스·고객 화면 레이아웃을 모두 위협한다. 프론트 스키마·`maxLength` 동기화를 A11 로 이관한다.

### 7.10 【판정】 PUT 은 전체 치환이다 — PATCH 시맨틱이 없다

`toInput(values)` 가 폼 값 **전량**을 매번 조립하므로(`EventFormPage.tsx:55-69`) 부분 수정이 존재하지 않는다. 또한 `create`/`update` 의 반환형이 `Promise<void>` 라 **프론트가 응답 본문을 버린다** — 성공 후 목록·상세를 무효화해 다시 조회한다(`crud.ts:179-181,198-201`).

**판정**:
1. **PUT(전체 치환)이 맞다** — 심의 메서드와 프론트 동작이 일치한다. PATCH 를 도입하지 않는다.
2. **응답 본문은 규정하되(201/200 + 엔티티) 프론트가 안 쓰는 것을 허용한다.** 이유: ① `version`(§7.3)이 도입되면 프론트가 응답에서 그것을 받아야 한다 ② 서버가 배정한 `id`·정규화한 값을 확인할 수 있어야 한다 ③ 무효화 후 재조회는 한 번의 왕복을 더 쓰지만, `staleTime` 30초와 함께 현재 규모에서 수용 가능하다.
3. 후속에서 `setQueryData` 로 응답을 직접 반영하면 왕복 1회가 준다(STATE-06 최적화) — A11 로 이관.

### 7.11 【판정】 멱등키가 만들어지지만 전송되지 않는다

`useCrudForm.ts:112-117` 이 제출 **시도** 단위 멱등키를 ref 에 만들고 `takeIdempotencyKey()` 로 잡는다. 주석도 `TODO(backend): 이 키는 Idempotency-Key 헤더로 나간다` 라고 명시한다. **그러나 `CrudAdapter.create(input, signal)` 시그니처에 키 자리가 없어 실제로는 어디에도 전달되지 않는다.**

**판정**:
1. **서버는 `POST /api/marketing/events` 에서 `Idempotency-Key` 헤더를 받도록 설계한다**(BE-004-EP-03 의 적립금 지급이 이 패턴의 정본). 같은 키의 재요청에는 최초 응답을 재생한다. 보관 기간 권고 24시간.
2. **프론트 어댑터 시그니처 확장이 선행돼야 한다** — `create(input, signal, idempotencyKey?)`. 이는 공용 `CrudAdapter` 인터페이스 변경이므로 A41 스코프다. A11 로 이관한다.
3. **현재 위험도는 낮다** — 동기 제출 락(`submitLockRef`)이 이중 발사를 이미 막고 있고(EXC-08 pass), 이벤트 등록은 금전 거래가 아니다. 다만 §7.2 대로 혜택이 **실제 발급**과 이어지는 순간 이 키는 필수가 된다 — 이중 등록 = 이중 쿠폰 발급이다.

### 7.12 【판정】 조회·삭제 실패에 참조 코드가 없다

`HttpError.reference` 가 모든 오류에 존재하고 저장 실패는 이를 표시하지만(`useCrudForm.ts:189` → FS-031-EL-015), **목록 조회 실패 배너**(`CrudListShell.tsx:157-164`)와 **삭제 실패 배너**(`useCrudList.tsx:111`)는 참조 코드를 버린다. 운영자가 '이벤트 목록이 안 떠요' 라고 신고하면 개발자가 로그와 맞출 수단이 없다.

**판정**: 서버는 모든 5xx 에 `traceId` 를 싣는다(BE-003 §2 상속 — 이미 계약됨). **표시 누락은 프론트 결함**이므로 A11 로 이관한다(EXC-20).

### 7.13 【판정】 400 과 422 를 갈라야 프론트 인라인 매핑이 작동한다

프론트는 **422 만** 필드 인라인 에러로 매핑한다(`useCrudForm.ts:176-186` — `isUnprocessable(cause) && cause.violations.length > 0`). **400 은 매핑하지 않아** '저장하지 못했습니다' 배너로 뭉개진다.

**판정**: 서버는 **필드 단위로 되돌릴 수 있는 거절은 전부 422 `UNPROCESSABLE` + `error.fields`** 로 준다 — 형식 위반이든 상태 위반이든. **400 은 필드에 귀속되지 않는 요청 자체의 파손**(JSON 파싱 실패 · 헤더 누락)에만 쓴다. 이 구분이 없으면 관리자는 **어느 필드가 틀렸는지 영영 알 수 없다.** BE-003 §2.1 의 `VALIDATION_FAILED`(400) 정의와 이 도메인의 운용이 다른 지점이므로 명시한다.

### 7.14 【판정】 중복 이벤트명을 막지 않는다

`eventSchema` 에 유니크 제약이 없고 서버 계약에도 없다. 같은 이름의 이벤트를 여럿 만들 수 있다.

**판정**: **의도된 것으로 확정한다.** 이벤트명은 식별자가 아니라 표시 문구이고, '여름맞이 리뷰 이벤트' 를 해마다 다시 여는 것은 정상 운영이다. 따라서 **409 중복 충돌은 이 도메인에 없다**(§5 EP-03 의 409 열이 N/A 인 이유). 다만 §7.2 대로 `bannerId`·`couponId` 가 도입되면 **그 참조에는 유니크·무결성 제약이 필요**하다.
