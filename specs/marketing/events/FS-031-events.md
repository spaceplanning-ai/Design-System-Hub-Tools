# FS-031. 이벤트 (목록 · 등록/수정)

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 목적 | 마케팅 이벤트를 상태·키워드로 좁혀 찾고, 기간·대상·혜택(쿠폰/적립)·배너 연동을 등록·수정하며, 단건·일괄로 삭제한다. 관리자가 지정한 상태가 기간과 어긋나면 목록에서 '기간상 XX' 힌트로 알린다 |
| 역할(주 사용자) | 관리자 (구현에 역할 분기 없음 — §4.1) |
| 진입 경로 | 좌측 GNB > 마케팅 관리 > 이벤트 (`/marketing/events`) |
| 포함 화면 | 목록 `/marketing/events` · 등록 `/marketing/events/new` · 수정 `/marketing/events/:id/edit`. **읽기 전용 상세 화면은 없다** — 행을 누르면 곧바로 수정 폼으로 간다 |
| 구현 경로 | `apps/admin/src/pages/marketing/events/**` · 공용 도메인 `apps/admin/src/pages/marketing/_shared/campaign.ts` |
| 대응 SCR | SCR-031 (미작성 — §7 #1) |
| 공통 컴포넌트 | `shared/crud/{CrudListShell,CrudTable,useCrudList,useCrudForm,FormPageShell,useListState,useDebouncedSearch,parseFilter,createCrudAdapter,requiredText}` · `shared/ui/{SearchField,SelectField,StatusBadge,Button,FormField,TextareaField,ToggleSwitch,SelectionBar,ConfirmDialog,RowActions,RowSelectCell,SeqCell,Empty,useRowSelection,useToast,useUnsavedChangesDialog}` · `@tds/ui DateRangeField` · `shared/useRowNavigation` · `shared/format.{formatDate,objectParticle}` · `shared/async.isAbort` · `shared/bulk.settleAll` |

> **캠페인형 2종**: 이벤트(FS-031)와 프로모션(FS-032)은 `_shared/campaign.ts`(상태 enum·라벨·tone·기간→상태 파생)를 공유하고 **같은 골격**(목록 = `CrudListShell` + `useCrudList`, 폼 = `FormPageShell` + `useCrudForm`)을 쓴다. 이 문서가 그 골격의 정본이며, FS-032 는 필드 차이(할인·최소주문금액·쿠폰)만 기술하고 골격 동작은 이 문서를 참조한다.
>
> **이 화면에 없는 것**(지어내지 않기 위해 명시): 이미지 업로드 필드 · 페이지네이션 · 기간 범위 **필터**(폼의 기간 **입력**은 있다) · 정렬 가능 헤더 · 읽기 전용 상세 · 재정렬 · 인라인 토글. 목록에 이미지 열이 없다(`EventListPage.tsx:4` 주석이 명시).

### 4.1 공통 예외 규칙

| 규칙 | 내용 |
|---|---|
| 조회 상태 머신 | 목록은 `{최초로드, 데이터보유-재조회, 빈, 실패}` 중 정확히 하나만 그린다. 스켈레톤은 **`data === undefined` 일 때만**. 재조회 중에는 이전 행이 유지되고 요약에 '새로고침 중…' 만 붙는다 |

| 중복 제출 | 저장은 `disabled={saving}` **와** 동기 제출 락(`useCrudForm.ts:103`)을 함께 쓴다(비동기 검증이 도는 사이의 두 번째 Enter 를 막는다). 그 위에 제출 시도 단위 **멱등키**가 어댑터 ledger 까지 전달돼(`useCrudForm.ts:211,228,235` → `crud.ts:114,121`) 재시도가 두 건을 만들지 않는다. 삭제·일괄 삭제는 확인 버튼 `busy` 로 막는다 |
| 실패 통지의 자리 | ① 목록 조회 실패 = 인라인 배너 ② 저장 실패 = 폼 카드 배너(+참조 코드) ③ 삭제·일괄 삭제 실패 = **다이얼로그 안** 배너 ④ 저장 충돌(409/412) = 전용 다이얼로그 ⑤ 서버 필드 거절(422) = 그 필드 인라인 에러 ⑥ 삭제·저장 **성공** = 토스트 |

| 동시 조회 | 목록·상세 조회는 react-query 가 키(`['marketing-events','list']` · `[…,'detail',id]`)마다 1건만 유지한다. `staleTime` 30초, 자동 재시도 없음, 창 포커스 재조회 없음 |
| 권한 없음 | **역할 기반 분기가 이 화면에 없다.** 읽기 권한은 셸의 라우트 가드(`AppShell.tsx:490-492` `RequirePermission` → `route-resource.ts:32-35` `findCoveringLeaf`)가 403 화면으로 막지만, **쓰기 컨트롤(등록·수정·삭제)은 권한과 무관하게 렌더된다**(§7 #4). 서버 403 은 조회=인라인 배너, 쓰기=각 요소의 실패로 떨어진다. 은닉 정책은 BE-031 §3.2 가 확정 |

| 필터·검색의 자리 | 둘 다 **클라이언트 필터**다 — 서버 쿼리 파라미터가 아니다. 조회는 언제나 전량이고, 거르기는 브라우저가 한다 |
| URL 반영 | **있다.** 상태 필터(`?phase=`)·키워드(`?q=`)의 단일 원천이 URL 쿼리스트링이다(`EventListPage.tsx:84` `useListState`). `replace: true` 로 갱신하므로(`useListState.ts:125`) 필터 한 번에 history 가 쌓이지 않고, 수정 폼에서 Back 하면 그 조건이 그대로 복원된다. 기본값과 같은 값은 URL 에서 지워진다. 링크 공유로도 재현된다 |
