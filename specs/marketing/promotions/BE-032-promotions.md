# BE-032. 프로모션 백엔드 기능 명세

## 1. 개요

| 항목 | 내용 |
|---|---|
| 대상 화면 | FS-032 프로모션 (`/marketing/promotions` · `/new` · `/:id/edit`) |
| 범위 | 프로모션 목록 조회, 단건 조회, 등록, 수정, 삭제 |
| 범위 밖 | **일괄 삭제 전용 경로** — 프론트가 N 개의 단건 DELETE 를 병렬로 낸다(§7.5). **서버측 필터·검색·페이징** — 심에 파라미터가 없다(§7.4). **쿠폰 조회·발급·검증** — 쿠폰코드는 자유 텍스트이며 이 계약은 쿠폰 도메인 API 를 호출하지 않는다(§7.2). **할인 계산·주문 적용** — 이 계약은 프로모션 **정의**만 저장한다. 실제 주문에 할인을 적용하는 것은 주문 도메인의 몫이다(§7.3). **이미지 업로드** — 이 화면에 이미지 필드가 없다(§7.8) |
| 전제 | 인증은 세션 쿠키 기반. 모든 경로는 `/api` 프리픽스. 응답 본문은 `application/json; charset=utf-8` |
| 프론트 어댑터 | `apps/admin/src/pages/marketing/promotions/data-source.ts` (`promotionAdapter`) |
| 도메인 타입 | `apps/admin/src/pages/marketing/promotions/types.ts` · 공용 enum `apps/admin/src/pages/marketing/_shared/campaign.ts` |
| 검증 정본 | `apps/admin/src/pages/marketing/promotions/validation.ts` (`promotionSchema`) — 서버는 이 규칙을 **재구현**한다(클라이언트 검증은 UX 이지 강제가 아니다 — §7.3) |
| 자매 문서 | BE-031 (이벤트) — 같은 캠페인 골격. **기간·상태·403/404 은닉·동시성·일괄 삭제·필터 판정이 동일**하므로 이 문서는 그것을 상속하고 **할인·금액 고유 판정**을 상세화한다 |

> **에러 봉투·권한 모델 상속**: BE-003 §2(공통 에러 봉투 · 공통 에러코드)·§3(인증 · 권한 모델 · CSRF · 타임아웃 상한)을 **그대로 상속한다.** 아래 §2 는 이 도메인의 고유 차이만 기술한다.

## 2. 공통 (상속)

| 항목 | 내용 |
|---|---|
| 에러 봉투 | BE-003 §2 그대로 — `{ error: { code, message, fields?, traceId } }` |
| 공통 에러코드 | BE-003 §2.1 그대로 — `VALIDATION_FAILED`(400) · `UNAUTHENTICATED`(401) · `FORBIDDEN`(403) · `CSRF_TOKEN_INVALID`(403) · `NOT_FOUND`(404) · `CONFLICT`(409) · `UNPROCESSABLE`(422) · `RATE_LIMITED`(429) · `INTERNAL_ERROR`(500) · `REQUEST_TIMEOUT`(504) |
| 도메인 고유 에러코드 | `PROMOTION_NOT_FOUND`(404) · `PROMOTION_CONFLICT`(409 — 동시 수정·이미 삭제된 대상) · `DISCOUNT_OUT_OF_RANGE`(422 — 할인값 범위 위반, §7.3) |
| 권한 역할 | `admin` = 조회 + 등록 + 수정 + 삭제. `operator` = **조회 계열만**. 등록·수정·삭제는 **403**. **할인율·금액을 정하는 것은 금전 권한이므로 `operator` 에게 쓰기를 열지 않는다**(§7.7) |
| CSRF | BE-003 §3.3 그대로 — POST·PUT·DELETE 는 `X-CSRF-Token` 요구 |
| 타임아웃 상한 | 조회 5초 · 쓰기 5초, 초과 시 504. **프론트에는 상한이 없다**(FS-032 §7 #19) |
| 프론트 역할 분기 | **없다.** 쓰기 컨트롤이 `operator` 에게도 보이므로 **403 은 일상 경로다**(§7.9) |

### 2.1 403 vs 404 은닉 정책 (§3.2 위치)

**BE-031 §2.1 을 그대로 상속한다.** 프로모션도 이벤트와 같이 **곧 고객에게 공개될 마케팅 자산**이지 개인정보가 아니다.

| 리소스 | 권한 부족 시 응답 | 근거 |
|---|---|---|
| `GET`·`POST /api/marketing/promotions` (컬렉션) | 403 `FORBIDDEN` | 컬렉션의 존재는 비밀이 아니다 |
| `GET`·`PUT`·`DELETE /api/marketing/promotions/:id` | 403 `FORBIDDEN` | 존재를 숨길 이유가 없다. 404 로 은닉하면 정당한 운영자가 '삭제됐다' 고 오판한다 — 프론트가 404 를 '이미 삭제되었을 수 있습니다' 로 렌더한다(FS-032-EL-029) |

> **이 도메인에서 404 는 오직 '정말 없다'** 를 뜻한다. **FS-032 §7 #8 을 이 표로 확정한다.**
>
> **다만 금액 열람은 별개 축이다** — §7.7 참조.

## 3. 데이터 계약

`types.ts` 의 `Promotion` 과 `_shared/campaign.ts` 의 enum 을 그대로 옮긴다.

| 필드 | 타입 | 필수 | 제약 (검증 정본 = `promotionSchema`) | 비고 |
|---|---|---|---|---|
| `id` | string | O (응답) | 서버 배정. 픽스처는 `pr-<seq>` | 요청 본문에 없다 |
| `title` | string | O | `trim()` 후 1자 이상 · **80자 이하** | 고객 노출 — §7.1 |
| `startAt` | string | O | `YYYY-MM-DD`, 실재 날짜 | 시각·타임존 성분 없음 — §7.6 |
| `endAt` | string | O | `YYYY-MM-DD`, 실재 날짜, **`endAt >= startAt`** | 같은 날 허용 |
| `phase` | `'upcoming' \| 'ongoing' \| 'ended'` | O | 이 3값만 | 관리자 지정값. 기간과 어긋나도 저장된다 — §7.6 |
| `target` | string | O | `trim()` 후 1자 이상 · **60자 이하** | 자유 텍스트 |
| `discountType` | `'rate' \| 'amount'` | O | 이 2값만 | `'rate'` = 정률(%), `'amount'` = 정액(원) |
| **`discountValue`** | **number** | **O** | **정수 · `> 0` · `discountType === 'rate'` 이면 `<= 100`. 정액은 상한 없음** | **`rate` 면 %, `amount` 면 원.** 프론트 폼은 문자열로 들고 제출 시 `digitsToNumber` 로 변환 — §7.3 · §7.4 |
| **`minOrderAmount`** | **number** | **O** | **정수 · `>= 0`(0 = 조건 없음) · 상한 없음** | 원 단위. 프론트에서 빈 문자열이면 0 — §7.3 |
| `couponLinked` | boolean | O | — | — |
| `couponCode` | string | O (필드 자체) | `couponLinked === true` 면 `trim()` 후 1자 이상. `false` 면 프론트가 `''` 로 비워 보낸다. **형식·길이 규칙 없음** | 쿠폰 id 참조가 아니라 **문자열 사본** — §7.2 |
| `description` | string | O (필드 자체) | `trim()` 후 **1000자 이하**. 빈 문자열 허용 | 고객 노출 — §7.1 |

> **모든 필드가 요청 본문에 항상 존재한다** — PATCH 시맨틱 없음(BE-031 §7.10 상속).
>
> **응답에 없는 것**: `createdAt` · `updatedAt` · `version` · `createdBy`. 낙관적 동시성 불가 — §7.5.
>
> **`discountValue` 의 의미가 `discountType` 에 의존한다** — 이 결합이 §7.3 · §7.4 판정의 근거다.

## 7. 핵심 판정

### 7.1 【보안 판정】 저장형 XSS — 프로모션 본문은 고객 화면에 그대로 나간다

**BE-031 §7.1 을 그대로 상속한다.** `title` · `description` · `target` · `couponCode` 는 관리자 자유 입력이고, 프로모션의 목적상 **고객 대면 화면에 그대로 노출된다.** `promotionSchema` 가 검사하는 것은 **길이·공백·숫자 형식뿐**이며 어떤 정제도 하지 않는다.

**판정** (BE-031 §7.1 과 동일하게 적용):
1. **서버가 저장 시점에 정제한다** — 클라이언트 검증은 어댑터를 우회한 호출을 막지 못한다.
2. **허용 목록 정책** — `description` 은 태그 화이트리스트(`<b> <i> <br> <p> <ul> <li> <a>`), 그 밖은 이스케이프. `title` · `target` · `couponCode` 는 **평문 전용** 전량 이스케이프.
3. **`<a href>` 는 스킴 화이트리스트**(`http:` · `https:` · `mailto:` 만). `javascript:` · `data:` · `vbscript:` 거절.
4. **정제 실패 시 저장 거절**(400) — 조용한 태그 제거 금지.
5. **`couponCode` 에 특히 주의한다** — 이 값이 고객 화면에 '쿠폰코드: SUMMER20' 으로 표시되거나 URL 쿼리(`?coupon=`)에 실릴 개연성이 높다. **형식 화이트리스트**(`^[A-Za-z0-9_-]{1,32}$`)를 적용하면 XSS 와 §7.2 의 무결성 문제를 동시에 좁힌다 — **권고**.

### 7.2 【보안 판정】 쿠폰코드가 참조가 아니라 문자열이다 — 검증 없는 할인 약속

`couponCode` 는 hint 로 '상품 관리 쿠폰과 연결됩니다.' 라고 **약속하지만**(FS-032-EL-024), 실제로는 **아무 검증 없는 자유 텍스트**다. 형식 규칙도, 실재 여부 확인도, 참조 무결성도 없다. 쿠폰 도메인 API 호출이 코드에 **0건**이다.

이 화면에서 이것이 이벤트의 배너명보다 **위험한 이유**: 배너명이 틀리면 배너가 안 보일 뿐이지만, **쿠폰코드가 틀리면 고객이 프로모션을 믿고 주문했는데 할인이 적용되지 않는다** — 고객 분쟁과 보상 비용이 된다. 오타 하나(`SUMMER20` vs `SUMER20`)가 그것을 만든다.

**판정**:
1. **현재 계약은 문자열을 그대로 받는다** — 존재하지 않는 쿠폰 API 를 지어내지 않는다.
2. **최소 방어선(즉시 적용 권고)**: §7.1 의 형식 화이트리스트(`^[A-Za-z0-9_-]{1,32}$`)를 400 으로 강제한다. 오타를 못 잡지만 **형식 파손과 XSS 는 잡는다.**
3. **후속에서 `couponId: string | null` 로 승격**하고 저장 시 **참조 무결성**(존재·유효기간·활성 상태)을 검증한다. **TOCTOU 를 유의한다** — 검증과 저장 사이에 쿠폰이 비활성화될 수 있으므로 같은 트랜잭션 안에서 FK 로 강제한다. 쿠폰 삭제·만료 측에서도 **참조 중인 진행 프로모션이 있으면 409 로 막는다.**
4. **프로모션 기간과 쿠폰 유효기간의 정합**도 그때 함께 검증한다 — 프로모션은 진행 중인데 쿠폰이 만료된 조합이 지금은 자유롭게 저장된다.
5. **`couponCode` 유니크 제약은 두지 않는다** — 같은 쿠폰을 여러 프로모션이 참조하는 것은 정상이다(BE-031 §7.14 의 이름 유니크 부재와 같은 결). 무결성은 **참조 방향**에서 강제한다.

### 7.3 【보안 판정】 할인값 서버 측 범위 검증 — **필수**

**`promotionSchema` 만이 할인 범위를 막고 있다.** 이 화면에서:
- `validation.ts:65-72` — `value <= 0` 거절 ('할인값은 0보다 커야 합니다.')
- `validation.ts:74-81` — `discountType === 'rate' && value > 100` 거절 ('정률 할인은 100%를 넘을 수 없습니다.')
- `validation.ts:55-62` — `/^\d+$/` 위반 거절 (**음수 부호가 여기서 걸린다** — 범위 검사가 아니라 형식 검사가 음수를 막는 구조다)
- `promotions.test.ts:78-91` 이 이 세 규칙을 고정한다

**이것은 브라우저 안의 규칙이다.** 어댑터를 우회한 직접 호출(`fetch`)·조작된 클라이언트·미래의 다른 클라이언트(모바일 앱·배치 스크립트)는 **아무것도 막지 않는다.** 그리고 이 필드는 **돈이다** — `discountValue: -100` 이 통과하면 할인이 **가산**이 되고, `discountType: 'rate', discountValue: 1000` 이 통과하면 **1000% 할인**이 된다.

**판정** — 서버는 다음을 **전부** 강제한다. 클라이언트 규칙의 재구현이 아니라 **정본**이다:

| 규칙 | 값 | 위반 시 |
|---|---|---|
| `discountValue` 타입 | 정수(integer). 소수·문자열·`NaN`·`Infinity` 거절 | 400 `VALIDATION_FAILED` |
| `discountValue` 하한 | **`> 0`** — **음수·0 거절.** 음수 할인은 가산이다 | 422 `DISCOUNT_OUT_OF_RANGE` |
| `discountValue` 상한 (정률) | **`<= 100`** — 100% 초과 거절 | 422 `DISCOUNT_OUT_OF_RANGE` |
| `discountValue` 상한 (정액) | **상한을 정해야 한다 — 현재 프론트에 없다**(FS-032 §7 #24). **권고: 1,000,000원**. 그 이상은 별도 승인 절차 대상 | 422 `DISCOUNT_OUT_OF_RANGE` |
| `minOrderAmount` 타입·범위 | 정수 · **`>= 0`** · **상한 권고 100,000,000원** | 400 / 422 `DISCOUNT_OUT_OF_RANGE` |
| 조합 검증 | **정액 할인 > 최소 주문금액(> 0)** 이면 거절 — '3만원 이상 주문 시 5만원 할인' 은 주문마다 손실이다. **현재 프론트에 이 검사가 없다** | 422 `DISCOUNT_OUT_OF_RANGE` |

**부가 판정**:
1. **`discountType` 과 `discountValue` 를 함께 검증한다** — `discountValue` 의 의미가 `discountType` 에 의존하므로(§3) 단독 검증은 불가능하다. 유형만 바꾸는 수정(정액 5000 → 정률)이 **5000% 할인**을 만들 수 있다.
2. **정수만 받는다** — `0.5%` 같은 소수 할인은 현재 프론트가 표현하지 못하고(`/^\d+$/`) 반올림 정책도 없다. 도입하려면 **금액 계산의 반올림 규칙**(내림/반올림/원 단위 절사)을 먼저 확정해야 한다 — §7.4.
3. **422 를 쓴다 (400 이 아니라)** — 프론트가 422 만 필드 인라인 에러로 매핑하기 때문이다(§7.12). 400 을 주면 관리자는 '저장하지 못했습니다' 만 보고 **어느 값이 왜 거절됐는지 모른 채** 재시도한다.
4. **이 검증은 프로모션 정의 저장 시점의 것이다.** 실제 주문에 할인을 적용할 때 **다시 한 번** 상한을 확인해야 한다 — 저장 후 정책이 바뀔 수 있고, 주문 금액과의 조합은 주문 시점에만 알 수 있다(§7.4).

### 7.4 【판정】 할인의 의미론 — 이 계약은 정의만 저장한다

`discountValue` 가 `discountType` 에 따라 %(정률)이거나 원(정액)이다. 그러나 **이 계약 어디에도 '할인이 어떻게 계산되는가' 가 없다** — 프론트는 정의를 저장할 뿐이고 주문에 적용하는 코드는 이 앱에 존재하지 않는다.

**판정** — 주문 도메인이 붙기 전에 확정해야 할 것을 이관한다:
1. **정률 할인의 기준액** — 상품가? 주문 총액? 배송비 포함? 다른 할인 적용 전/후?
2. **반올림 정책** — '20% of 3,333원' = 666.6원. 내림·반올림·원 단위 절사 중 무엇인가. **정의 시점에 정하지 않으면 프로모션마다 다르게 계산된다.**
3. **중첩 정책** — 같은 기간에 두 프로모션이 걸린 주문. 합산? 최대 하나? 우선순위?
4. **`minOrderAmount` 의 비교 기준** — 할인 전 금액인가 할인 후인가. 다른 프로모션 적용 후인가.
5. **정액 할인 > 주문 금액** 인 경우 — 0원 처리인가 거절인가.
6. **주문 시점 재검증** — §7.3 #4. 프로모션이 그 사이 삭제·수정됐을 수 있다. **주문에는 적용된 할인의 스냅샷이 남아야 한다** — 프로모션을 참조만 하면 나중에 프로모션이 바뀔 때 과거 주문의 할인이 소급 변경된다.
7. 이 항목들은 **이 화면의 범위 밖**이지만, 여기서 저장한 값이 그 계산의 입력이므로 명시한다. A01 · A63 이관.

### 7.5 【판정】 낙관적 동시성 토큰 부재 — 할인율이 조용히 덮인다

**BE-031 §7.3 을 그대로 상속한다.** `Promotion` 에 `updatedAt`/`version`/`ETag` 가 **없다**(§3). `If-Match` 를 실을 수단이 없고 프론트에도 배선이 없다. 어댑터가 잡는 409 는 **'대상이 사라진 경우'뿐**이다.

**이 도메인에서 더 심각한 이유**: 두 탭에서 같은 프로모션을 열고 A 가 할인율을 20%→10% 로 낮춰 저장 → B 가 대상만 바꿔 저장 → **B 가 A 의 인하를 되돌려 20% 로 복구한다.** PUT 이 전 필드 치환이므로(§3) 손실은 레코드 전체다. **경고도 없고, 그 프로모션은 진행 중일 수 있다** — 즉 **의도하지 않은 할인이 고객에게 즉시 나간다.**

**판정** (BE-031 §7.3 과 동일):
1. **서버가 `version: number`(또는 `updatedAt` 기반 ETag)를 응답에 포함**한다.
2. **PUT(EP-04)·DELETE(EP-05)는 `If-Match: <version>` 을 요구**한다. 누락 시 400(`error.fields[].name = "If-Match"`), 불일치 시 **412 `PRECONDITION_FAILED`**.
3. **프론트는 이미 준비돼 있다** — `isConflict` 가 409·412 를 같은 UX 로 수렴시키고(`http-error.ts:102`), `useCrudForm.ts:158-172` 가 충돌 다이얼로그를 연다. **서버가 412 를 주기 시작하면 화면 코드는 그대로 두고 어댑터만 헤더를 실으면 된다.**
4. 남는 프론트 작업: `types.ts` 에 `version` 추가 + 어댑터가 조회 시 보관하고 쓰기 시 헤더로 싣기(회원 메모 BE-004 §7.4 가 정본).
5. **우선순위를 이벤트보다 높게 둔다** — 이벤트의 last-write-wins 는 표시 오류지만 프로모션의 그것은 **금전 손실**이다.
6. FS-032 §7 #15 를 이 판정으로 확정한다.

### 7.6 【판정】 상태(phase) 정본 · 진행 중 프로모션의 삭제

**상태 정본은 BE-031 §7.6 을 그대로 상속한다** — 권고는 (b) '지정이 정본'(조기 종료 허용).

**프로모션 고유 논점 — 진행 중 삭제**: 현재 계약은 `phase === 'ongoing'` 인 프로모션의 DELETE 를 **막지 않는다**(§5 EP-05 의 422 열이 N/A 인 이유). 고객이 보고 있는 할인이 예고 없이 사라질 수 있다.

**판정**:
1. **막지 않는 것으로 현재 계약을 확정한다** — 프론트에 그 개념이 없고(삭제 다이얼로그가 상태를 보지 않는다), 지어내지 않는다.
2. 그러나 이 상태의 결과를 명시한다: **진행 중 프로모션의 하드 삭제는 되돌릴 수 없고**(soft-delete·undo 없음 — FS-032 §4.1) 감사 로그도 없다(§7.14). 누가 어떤 할인을 언제 지웠는지 추적 불가능하다.
3. **후속 권고**: `phase === 'ongoing'` 삭제 시 **422 `PROMOTION_IN_PROGRESS`** 로 막고 '먼저 종료 처리하세요' 를 안내하거나, 최소한 확인 다이얼로그에서 강화 경고를 띄운다. **주문에 적용된 이력이 있으면 soft-delete 만 허용**한다(§7.4 #6 의 스냅샷과 연결).

### 7.7 【판정】 금액 마스킹 · 열람 권한 — ERP-14 축

할인액·최소 주문금액이 **목록(FS-032-EL-007.7)과 폼에서 권한과 무관하게 전부 노출된다.** 마스킹·열람 권한 분리가 없다. `operator` 는 쓰기가 막히지만(§2) **읽기는 열려 있어 모든 금액을 본다.**

**판정**:
1. **현재 계약은 마스킹하지 않는다 — 그리고 그것이 맞다.** 프로모션의 할인액은 **곧 고객에게 공개될 값**이다(§2.1 의 '마케팅 자산' 논리와 같다). 회원 개인정보나 정산 금액과 달리 비밀이 아니다.
2. 따라서 **ERP-14(금액 마스킹) 축은 이 도메인에 적용되지 않는다** — 숨길 이유가 없는 값에 마스킹을 붙이면 운영만 느려진다.
3. **다만 열람 권한은 컬렉션 read 권한과 동일 축임을 명시한다**(§4 EP-01) — 마케팅 도메인 read 가 있으면 금액도 본다. 별도 축을 만들지 않는다.
4. **`minOrderAmount` 는 재검토 여지가 있다** — 이것은 고객에게 '3만원 이상 구매 시' 로 공개되므로 역시 비밀이 아니다. 결론 동일.
5. FS-032 §7 #26 을 이 판정으로 **'적용하지 않음(사유 있음)'** 으로 확정한다.

### 7.8 【판정】 이미지 업로드 심이 없다 — 이 화면에 이미지 필드가 없기 때문이다

**BE-031 §7.8 을 그대로 상속한다.** 프로모션 등록/수정 폼에 이미지 업로드 필드가 없다(`ImageUploadField` import 0건, 목록에 이미지 열 없음 — `PromotionListPage.tsx:4` 주석). 따라서 `data-source.ts` 에 업로드 심이 없는 것은 누락이 아니라 **정합한 상태**다.

**판정**: **업로드 엔드포인트 — 심 없음(미정).** 후속에서 프로모션 배너 이미지를 도입하면 BE-031 §7.8 의 계약을 그대로 적용한다 — 업로드 엔드포인트 분리 + **서버측 스킴 화이트리스트**(`https:`·자체 CDN 오리진만 허용, **`blob:` · `data:` · `javascript:` · `file:` 은 400 거절**). `ImageUploadField` 가 `URL.createObjectURL` 로 `blob:` URL 을 만들기 때문에, 업로드 심 없이 그 컴포넌트를 붙이면 **저장값이 `blob:` 이 되어 새로고침 시 깨진다.**

### 7.9 【판정】 403·429 에 전용 문구가 없다

**BE-031 §7.7 을 그대로 상속한다.** 서버가 403 을 정확히 주고 `HttpError` 가 status 를 싣는데도 이 화면은 status 를 읽지 않아 '불러오지 못했습니다' / '저장하지 못했습니다' 하나로 수렴한다. `operator` 가 쓰기 버튼을 보고 누를 수 있으므로 이 403 은 일상 경로다.

**판정**: 서버 책임 아님 — 프론트 결함(A11 이관). 서버는 `error.message` 를 사용자에게 그대로 보여줄 수 있는 문구로 쓴다. **쓰기 컨트롤 권한 게이팅(EXC-03)이 붙으면 403 빈도가 급감한다** — 근본 해결은 그쪽이다.

### 7.10 【판정】 멱등키가 만들어지지만 전송되지 않는다

**BE-031 §7.11 을 그대로 상속한다.** `useCrudForm.ts:112-117` 이 제출 시도 단위 멱등키를 만들지만 `CrudAdapter.create` 시그니처에 자리가 없어 전송되지 않는다.

**판정**: 서버는 `Idempotency-Key` 헤더를 받도록 설계한다(보관 24시간 권고). 프론트 어댑터 시그니처 확장은 A41 스코프 — A11 이관. **현재 위험도는 낮다**(동기 제출 락이 이중 발사를 막고, 프로모션 등록은 그 자체로 금전 이동이 아니다). 다만 §7.2 대로 쿠폰 발급과 이어지는 순간 필수가 된다.

### 7.11 【판정】 조회·삭제 실패에 참조 코드가 없다

**BE-031 §7.12 를 그대로 상속한다.** 저장 실패만 참조 코드를 표시하고 목록 조회 실패·삭제 실패 배너는 이를 버린다.

**판정**: 서버는 모든 5xx 에 `traceId` 를 싣는다(BE-003 §2 — 이미 계약됨). 표시 누락은 프론트 결함 — A11 이관(EXC-20).

### 7.12 【판정】 400 과 422 를 갈라야 인라인 매핑이 작동한다

**BE-031 §7.13 을 그대로 상속한다.** 프론트는 **422 만** 필드 인라인 에러로 매핑한다(`useCrudForm.ts:176` — `isUnprocessable && violations.length > 0`).

**판정**: **필드 단위로 되돌릴 수 있는 거절은 전부 422 + `error.fields`.** 400 은 요청 자체의 파손(JSON 파싱 실패·헤더 누락)에만 쓴다. **이 도메인에서 특히 중요하다** — §7.3 의 할인 범위 위반이 400 으로 나가면 관리자는 '저장하지 못했습니다' 만 보고 **할인율이 왜 거절됐는지 모른 채** 같은 값으로 재시도한다.

### 7.13 【판정 · A11Y】 `minOrderAmount` 의 422 를 받아도 화면이 무효를 알리지 못한다

**서버가 §7.3 대로 `minOrderAmount` 를 422 로 거절해도**, 프론트의 `setError('minOrderAmount')` 는 `FormField` 에 `<p id="promo-min-order-error" role="alert">` 를 그릴 뿐 — **`PromotionFormPage.tsx:233-242` 의 입력에 `aria-invalid` 도 `aria-describedby` 도 없다.** 이 폼의 다른 입력 4종(프로모션명 `:154-155` · 대상 `:189-190` · 할인값 `:219-223` · 쿠폰코드 `:274-277`)은 전부 배선하는데 이 하나만 빠졌다. 남는 무효 표시는 `controlStyle(true)` 의 **붉은 테두리뿐 = 색상 단독 표기**(WCAG 1.4.1 위반). hint('0 이면 조건 없음')도 연결되지 않는다.

**판정**:
1. **서버 계약과 무관한 프론트 결함이다** — 그러나 **서버가 이 필드를 422 로 거절하기 시작하면 결함이 비로소 사용자에게 도달**하므로 여기 기록한다. 지금은 클라이언트 검증이 대부분 먼저 잡아 가려져 있고, `?status=save:422` 는 `violations` 가 비어 있어(§6) **재현조차 되지 않는다.**
2. **A11Y-11 은 P0 다** — quality-bar 기준 배치 실패 사유. A11 change_request 로 이관한다(NFR-032 §2 · §5 #7).
3. 수정은 3줄이다 — 다른 4개 입력과 같은 `aria-invalid={errors.minOrderAmount !== undefined}` + `aria-describedby={errors.minOrderAmount !== undefined ? errorIdOf('promo-min-order') : undefined}`.

### 7.14 【판정】 감사 로그 — 할인 변경 이력이 없다

`Promotion` 에 `createdAt`·`updatedAt`·`createdBy`·`updatedBy` 가 없고(§3) 감사 로그 계약도 없다. **누가 언제 할인율을 20%에서 50%로 바꿨는지 추적할 수 없다.** 하드 삭제라 지워진 프로모션은 흔적도 남지 않는다(§7.6).

**판정**:
1. **이벤트(BE-031)보다 강한 요구다** — 프로모션의 변경은 곧 매출 영향이고, 사후에 '왜 이 할인이 나갔는가' 를 묻는 상황이 반드시 온다.
2. **최소 요구**: `createdAt`·`updatedAt`·`updatedBy` 를 응답에 포함한다. **이는 §7.5 의 `version` 도입과 같은 작업이므로 함께 처리한다.**
3. **권고**: 할인 관련 필드(`discountType`·`discountValue`·`minOrderAmount`·`phase`)의 변경은 **별도 감사 테이블에 이전/이후 값과 함께** 남긴다. A01 · A63 이관.
