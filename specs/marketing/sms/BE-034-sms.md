# BE-034. SMS 발송 백엔드 기능 명세

## 1. 개요

| 항목 | 내용 |
|---|---|
| 대상 화면 | FS-034 SMS 발송 (`/marketing/sms` · `/new` · `/:id/edit`) |
| 범위 | SMS/LMS/MMS 발송 캠페인 목록·단건 조회, 등록·수정, 삭제(단건·일괄), **발송 실행 트리거**(심 근거 — §4 EP-06 · §7.3) |
| 범위 밖 | 실제 문자 전송(통신사·중계사 연동) — 이 계약은 '발송을 지시받았다'까지만 정한다. 예약 실행(스케줄러) — 서버 내부 관심사이며 프론트 계약이 아니다. 발신번호·세그먼트·템플릿 레지스트리 — **심 없음(§4 EP-07/08/09)**. 발송 결과 통계 집계 로직 — 응답 형태만 정한다 |
| 전제 | BE-003 을 상속한다: 세션 쿠키 인증, `/api` 프리픽스, `application/json; charset=utf-8`, ISO 8601 |
| 프론트 어댑터 | `apps/admin/src/pages/marketing/sms/data-source.ts` (`smsAdapter` = `createCrudAdapter`) |
| 도메인 타입 | `apps/admin/src/pages/marketing/sms/types.ts` · 공용 `apps/admin/src/pages/marketing/_shared/messaging.ts` |
| 검증 정본 | `apps/admin/src/pages/marketing/sms/validation.ts` (`smsSchema`) — **단, 프론트 검증은 UX 층이고 법적 강제는 서버가 정본이다(§7.4~§7.7)** |

> **에러 봉투·권한 모델 상속**: BE-003 §2·§3 을 그대로 상속한다. 아래는 SMS 발송 고유 차이만 기술한다.

## 2. 공통 (상속)

- **에러 봉투**: BE-003 §2. 공통 에러코드(`VALIDATION_FAILED`·`UNAUTHENTICATED`·`FORBIDDEN`·`CSRF_TOKEN_INVALID`·`NOT_FOUND`·`CONFLICT`·`UNPROCESSABLE`·`RATE_LIMITED`·`INTERNAL_ERROR`·`REQUEST_TIMEOUT`) 동일. `error.traceId` 는 프론트가 '오류 코드'로 노출한다(FS-034-EL-016).
- **권한**: `admin` = 전체. `operator` = 조회 계열만(목록·단건). 쓰기(등록·수정·삭제) → 403. **발송 실행(EP-06)은 `admin` 전용이며 별도 권한으로 분리할 것을 권고한다(§7.3)**. 마케팅 도메인 읽기 권한 없는 관리자 → 컬렉션 403 / 개별 404 은닉(BE-003 §3.2 — 캠페인 id 는 개인정보가 아니나 **수신자 세그먼트 구성이 영업 정보**이므로 같은 정책을 적용한다).
- **CSRF**: 쓰기(POST·PUT·DELETE)에 `X-CSRF-Token`.
- **타임아웃**: 조회·쓰기 5초 → 504. **발송 실행(EP-06)은 트리거만 하고 즉시 202 로 반환한다** — 전송 완료를 기다리지 않는다(§7.3).
- **프론트 역할 분기 없음**(FS-034 §4.1) — 권한 강제는 서버 책임. **프론트는 등록·수정·삭제 버튼을 권한과 무관하게 렌더한다**(FS-034 §7 #10).

## 3. 데이터 계약 (types.ts 대조)

| 타입 | 필드 | 비고 |
|---|---|---|
| `SmsCampaign` | `id` · `name` · `senderId` · `senderNumber`(비정규화) · `segmentIds[]` · `recipientCount`(저장 시점 스냅샷) · `isAd` · `body` · `hasImage` · `kind` · `status` · `scheduledAt` · `stats` | 목록·단건이 같은 타입 — **상세 전용 필드가 없다** |
| `SmsCampaignInput` | `SmsCampaign` − `id` · `senderNumber` · `recipientCount` · `kind` · `stats` | **파생값 5개는 서버가 계산한다** — 클라이언트가 보내지 않는다 |
| `SmsKind` (`_shared`) | `'sms' \| 'lms' \| 'mms'` | 이미지 있으면 mms, 없으면 90byte 이하 sms · 초과 lms |
| `SendStatus` (`_shared`) | `'draft' \| 'scheduled' \| 'sending' \| 'sent' \| 'canceled'` | **입력은 `draft`·`scheduled` 만**(`smsSchema` 가 `z.enum(['draft','scheduled'])`) |
| `SendStats` (`_shared`) | `total` · `success` · `failed` | 성공률 = `success/total` 반올림(전체 0이면 0) |
| `SenderNumber` (`_shared`) | `id` · `number` · `label` · `verified` | 사전등록 검증 완료 번호만 발신 가능 |
| `Segment` (`_shared`) | `id` · `label` · `recipientCount` · `description` | 수신자 그룹 |

**서버가 계산해야 하는 파생값** (프론트 어댑터가 현재 픽스처로 흉내 내는 것 — `data-source.ts:75-92`):
`senderNumber` = `senderId` 로 조회한 번호 · `recipientCount` = 세그먼트 수신자 수 **중복 제거 후** 합(§7.6) · `kind` = 본문 바이트 + 이미지 여부 판정 · `stats` = 신규 생성 시 `{0,0,0}`, 이후 발송 결과 누적.

### 3.1 상태 머신 (`_shared/messaging.ts:322-364`)

```
draft ──저장(status=scheduled)──▶ scheduled ──스케줄러/발송 트리거──▶ sending ──▶ sent
  │                                   │
  └──────발송 트리거(즉시)────────────┴──예약 취소──▶ canceled
```

| 규칙 | 계약 |
|---|---|
| 편집 가능 상태 | `draft` · `scheduled` 만 (`isEditableSend`). `sending`·`sent`·`canceled` 는 **수정 거절(422)** — §5 의 '422 상태위반' 축이 이것이다 |
| 상태 입력 | 등록·수정 바디의 `status` 는 `draft`\|`scheduled` **만** 허용. `sending`·`sent`·`canceled` 를 바디로 보내면 400 |
| 상태 전이의 주체 | `draft→scheduled` = 관리자 저장. `scheduled→sending→sent` = **서버**(스케줄러 또는 EP-06). `→canceled` = 예약 취소(**프론트 UI 미구현** — §7.8) |
| 실제 전송 | **이 화면은 문자를 보내지 않는다.** 저장은 레코드 생성일 뿐이다(FS-034 §5) |

**에러**: 401 · 403 · 429 · 500 · 504.

### BE-034-EP-03 · SMS 발송 등록
| 항목 | 내용 |
|---|---|
| 근거 (FS) | FS-034-EL-033(등록) |
| 메서드·경로 | `POST /api/marketing/sms` |
| 권한 | `admin` 만. `operator` → 403 |
| 멱등성 | 비멱등. **`Idempotency-Key` 헤더를 권고**(§7.3) |
| 레이트리밋 | 분당 30회 |

**바디**(`SmsCampaignInput`):

| 필드 | 제약 (서버 강제 — 프론트 스키마와 **일치해야 한다**) |
|---|---|
| `name` | 앞뒤 공백 제거 후 1–60자 |
| `senderId` | 존재하는 발신번호 id **AND `verified === true`**(§7.4) |
| `segmentIds` | 1개 이상. 각 id 가 존재해야 한다 |
| `isAd` | boolean |
| `body` | 1자 이상 · **≤ 2,000 byte(EUC-KR 기준: 코드포인트 > 0x7F = 2byte, 그 외 1byte)**. `isAd === true` 면 `(광고)` 로 시작 **AND** '무료수신거부'\|'무료거부'\|'수신거부' 중 하나 포함(§7.5) |
| `hasImage` | boolean. **파일이 아니다** — MMS 판정 플래그뿐(FS-034 §7 #6) |
| `status` | `draft` \| `scheduled` **만** |
| `scheduledAt` | `status === 'scheduled'` 면 필수 · **미래 시각** · `isAd === true` 면 **21:00–08:00(Asia/Seoul) 금지**(§7.5). `status === 'draft'` 면 `''` |

**응답 201** — 생성 `SmsCampaign`(파생값 5개 포함). 프론트 어댑터 반환 타입은 `Promise<void>` 라 본문을 읽지 않지만, 목록을 무효화해 재조회한다.

**에러**: 400 `VALIDATION_FAILED`(`error.fields`: `name`·`senderId`·`segmentIds`·`body`·`scheduledAt`·`status`) · 401 · 403 · 403 `CSRF_TOKEN_INVALID` · 422 `SENDER_NOT_VERIFIED`(§7.4) · 422 `SEGMENT_NOT_FOUND` · 422 `AD_REQUIREMENTS_UNMET`(§7.5) · 422 `NIGHT_AD_FORBIDDEN`(§7.5) · 422 `RECIPIENT_LIMIT_EXCEEDED`(§7.7) · 429 · 500 · 504.

> **400 vs 422 의 경계**: 형식 위반(빈 발송명·61자·모르는 status)은 **400 + `error.fields`** → 프론트가 인라인 에러로 꽂는다(EXC-07 경로). 형식은 맞으나 **도메인 상태가 거절**하는 것(미검증 발신번호·야간 광고·대상 초과)은 **422** → 현재 프론트는 `violations` 가 있으면 인라인, 없으면 폼 배너로 떨어뜨린다(`useCrudForm.ts:176-186`). **422 에도 `error.fields` 를 실어라** — 그래야 '발신번호' 필드에 정확히 붙는다.

### BE-034-EP-04 · SMS 발송 수정
| 항목 | 내용 |
|---|---|
| 근거 (FS) | FS-034-EL-033(수정), EL-034 |
| 메서드·경로 | `PUT /api/marketing/sms/:id` |
| 권한 | `admin` 만 |
| 멱등성 | 멱등(PUT 전체 치환) |
| 레이트리밋 | 분당 30회 |

**바디**: `SmsCampaignInput`(EP-03 동일 제약). **추가 제약 — 상태 위반(§3.1)**: 대상 캠페인의 현재 `status` 가 `sending`·`sent`·`canceled` 면 **422 `CAMPAIGN_NOT_EDITABLE`** 로 거절한다. 발송된 캠페인의 본문·대상을 사후 수정하면 발송 이력이 거짓이 된다(§7.2 판정).

**낙관적 동시성**: 요청에 `If-Match: <etag>`(또는 `updatedAt`)를 실어야 한다. 불일치 → **412 `PRECONDITION_FAILED`**. 이미 삭제된 id → **409 `CONFLICT`**. 프론트는 409·412 를 같은 충돌 다이얼로그로 처리하고 **입력을 보존한다**(FS-034-EL-034 · `useCrudForm.ts:158-172`). **단 현재 `SmsCampaign` 타입에 `updatedAt`/`version` 필드가 없고 `If-Match` 를 보내는 코드도 없다 — §7.2 이관.**

**응답 200/204**.

**에러**: 400 · 401 · 403 · 403 CSRF · 404 `SMS_CAMPAIGN_NOT_FOUND` · 409 `CONFLICT`(삭제됨) · 412 `PRECONDITION_FAILED`(동시 수정) · 422 `CAMPAIGN_NOT_EDITABLE` · 422 `SENDER_NOT_VERIFIED` · 422 `SEGMENT_NOT_FOUND` · 422 `AD_REQUIREMENTS_UNMET` · 422 `NIGHT_AD_FORBIDDEN` · 429 · 500 · 504.

### BE-034-EP-05 · SMS 발송 삭제 (단건·일괄 공용)
| 항목 | 내용 |
|---|---|
| 근거 (FS) | FS-034-EL-007.9, EL-011, EL-006.2, EL-012 |
| 메서드·경로 | `DELETE /api/marketing/sms/:id` |
| 권한 | `admin` 만 |
| 멱등성 | **멱등이 아니다** — 프론트 어댑터가 이미 삭제된 id 에 409 를 던진다(`crud.ts:139-141`). §7.2 판정 |
| 레이트리밋 | 분당 60회(일괄이 항목당 개별 요청) |

**응답 204**. 일괄 삭제는 프론트가 `settleAll` 로 항목마다 **병렬 DELETE** 한다(§7.1).

**상태 위반**: `sending` 캠페인 삭제는 **422 `CAMPAIGN_SENDING`** 으로 거절한다 — 전송 중인 캠페인의 레코드를 지우면 결과를 기록할 곳이 사라지고, **발송 감사 로그가 끊긴다**(§7.9). `sent` 캠페인은 삭제 대신 **soft-delete** 를 권고한다(§7.9 — 법적 증빙).

**에러**: 400 · 401 · 403 · 403 CSRF · 404 `SMS_CAMPAIGN_NOT_FOUND` · 409 `CONFLICT`(이미 삭제) · 422 `CAMPAIGN_SENDING` · 429 · 500 · 504.

### BE-034-EP-06 · SMS 발송 실행 (트리거) — **프론트 호출부 없음**
| 항목 | 내용 |
|---|---|
| 근거 (심) | `data-source.ts:71` — `// 발송 실행: POST /api/marketing/sms/:id/send (예약은 스케줄러가 시각에 실행) — 프론트는 트리거만.` |
| 근거 (FS) | **없다** — FS-034-EL-036 이 '미구현'으로 기록한 자리다 |
| 메서드·경로 | `POST /api/marketing/sms/:id/send` |
| 권한 | `admin` 만. **별도 `marketing:send` 권한으로 분리 권고**(§7.3) |
| 멱등성 | **`Idempotency-Key` 헤더 필수** — 이 계약에서 가장 중요한 한 줄(§7.3) |
| 레이트리밋 | 분당 5회 + 일일 발송량 상한(§7.7) |

**헤더**: `Idempotency-Key: <UUID v4>`(보존 24시간, 누락 시 400). **바디 없음.**

**응답 202 Accepted** — `{ campaignId, status: 'sending', acceptedAt }`. 전송 완료를 기다리지 않는다. 상태는 EP-01/EP-02 재조회로 확인한다.

**서버가 발송 시점에 **반드시 재검증**하는 것** (§7.4~§7.7 — 저장 시점 검증만으로는 불충분하다):
1. 캠페인 `status` 가 `draft`\|`scheduled` 인가 → 아니면 **422 `CAMPAIGN_NOT_SENDABLE`**(이미 `sending`/`sent` 면 중복 발송이다)
2. `senderId` 가 **여전히** 사전등록 검증 상태인가(§7.4)
3. 대상 수신자 각각의 **마케팅 수신 동의가 발송 시점에 유효한가**(§7.6) — 저장 이후 철회했을 수 있다
4. 광고성이면 **현재 시각**이 야간(21:00–08:00 Asia/Seoul)이 아닌가(§7.5)
5. 대상 규모가 상한 이내인가(§7.7)
6. 본문의 `#{...}` 토큰이 **전부 화이트리스트 안에** 있는가(§7.8)

**에러**: 400 `VALIDATION_FAILED`(`Idempotency-Key` 누락) · 401 · 403 · 403 CSRF · 404 `SMS_CAMPAIGN_NOT_FOUND` · 409 `IDEMPOTENCY_CONFLICT`(같은 키 · 다른 바디) · 422 `CAMPAIGN_NOT_SENDABLE` · 422 `SENDER_NOT_VERIFIED` · 422 `NIGHT_AD_FORBIDDEN` · 422 `RECIPIENT_LIMIT_EXCEEDED` · 422 `UNRESOLVED_VARIABLE` · 429 `RATE_LIMITED` · 500 · 504.

> **⚠ 이 엔드포인트는 심(주석)이 근거이지 화면이 근거가 아니다.** 프론트에 이것을 부르는 코드가 **한 줄도 없다**(FS-034-EL-036 · §6). 백엔드가 구현해도 **UI 가 함께 오지 않으면 아무도 호출하지 않는다.** 반대로 UI 만 먼저 붙이고 위 6개 재검증을 서버가 하지 않으면 그것이 이 도메인 최악의 사고 경로다(§7.3).

## 7. 핵심 판정

### 7.1 목록 계약 — 전량 반환을 계약으로 굳히지 않는다
프론트가 전량을 받아 **클라이언트에서** 필터·검색·정렬한다(`SmsListPage.tsx:118-121`). 캠페인이 수백 건을 넘기면 이 계약은 무너진다. **지금은 EP-01 을 '쿼리 없는 전량 반환'으로 두되, 프론트가 `useListState`(URL 직렬화)로 옮기는 변경과 **동시에** `status`·`keyword`·`page`·`size` 쿼리를 여는 것을 권고한다.** 서버가 먼저 쿼리를 열어도 프론트가 보내지 않으므로 아무 일도 일어나지 않는다. 일괄 삭제도 같은 결이다 — 전용 벌크 엔드포인트를 만들지 않고 프론트가 단건 DELETE 를 `settleAll` 로 병렬 호출한다(BE-010 §7.1 과 같은 근거).

### 7.2 동시성 — 토큰이 **없다**, 그래서 마지막 쓰기가 이긴다
프론트는 409/412 를 받으면 입력을 보존한 채 충돌 다이얼로그를 띄울 **준비가 돼 있다**(`useCrudForm.ts:158-172` · FS-034-EL-034). 어댑터도 이미 삭제된 id 에 409 를 던져 '유령 저장'을 막는다(`crud.ts:71-74`). **그러나 `SmsCampaign` 타입에 `updatedAt`/`version` 이 없고 `If-Match` 를 보내는 코드가 앱 전체에 없다** — 즉 **삭제 충돌은 잡히지만 동시 수정 충돌은 잡히지 않는다**(두 관리자가 같은 캠페인을 동시에 고치면 마지막 저장이 조용히 이긴다). 발송 캠페인의 본문·대상은 **법적 책임이 걸린 값**이므로 이 도메인은 낙관적 잠금을 도입할 가치가 있다. **결정: `SmsCampaign` 에 `updatedAt` 을 추가하고 EP-04 가 `If-Match` 를 요구한다 — 타입 변경이 선행돼야 하므로 §7.12 로 이관한다.**

한편 **DELETE 는 멱등이 아니다**(이미 삭제 → 409, `crud.ts:82-84`). 일반적으로 DELETE 는 멱등이 권장이지만, 여기서는 **의도된 선택**이다: 일괄 삭제가 `settleAll` 로 병렬 실행되므로 '이미 없음'을 204 로 삼키면 다른 관리자가 방금 지운 것을 내가 지웠다고 보고하게 된다. 서버도 이 동작을 따라 409 를 준다.

### 7.3 발송 실행 — 심은 있고 UI 는 없다. 멱등키는 **UI 가 붙는 순간 필수**다 【보안 판정】
`/send` 심 주석은 `data-source.ts:71` 에 실재하므로 EP-06 을 정식 명세했다. **그러나 프론트 호출부가 0건이다**(§6) — 어댑터 함수도, 발송 버튼도, 확인 다이얼로그도 없다.

이 사실이 판정을 바꾼다: **발송 UI 가 없으므로 '중복 발송 경합'은 현재 성립하지 않는다.** 두 번 누를 버튼 자체가 없다. 그러므로 이 항목을 지금 '중복 발송 위험 P0' 으로 올리는 것은 사실이 아니다.

**그러나 발송 트리거를 구현하는 순간, 이것이 이 앱 전체에서 가장 위험한 쓰기가 된다** — 되돌릴 수 없고(문자는 회수되지 않는다), 돈이 들고(건당 과금), 법적 책임이 따른다. 그래서 UI 구현 시 다음이 **동시에** 와야 한다:

1. **`Idempotency-Key` 필수** — 제출 시도 단위로 **요청 함수 밖**(ref)에서 생성해 재시도가 같은 키를 재사용한다. `mutationFn` 안에서 만들면 재시도마다 새 키가 생겨 서버가 두 요청을 별개 발송으로 본다 = **전 수신자에게 두 번 발송**. `useCrudForm.ts:118-123` 이 이미 이 패턴을 구현해 두었으니 발송 뮤테이션도 그것을 따른다. **F3b 로 그 키가 실제로 어댑터까지 도달하게 됐다** — `:211,228,235`(variables) → `crud.ts:288-289,310-311` → `WriteContext.idempotencyKey`(`crud.ts:41,47-48`) → 픽스처 ledger(`:91,114,121`). 즉 **발송 뮤테이션이 붙을 때 배선을 새로 만들 필요가 없다 — 같은 경로를 그대로 탄다.** 남은 것은 실 HTTP 어댑터가 그 값을 헤더로 옮기는 한 줄이며 그 자리를 `crud.ts:39` 의 `TODO(backend)` 가 표시한다. **ledger 의 순서 계약도 서버가 지켜야 한다** — 키는 **적용에 성공한 뒤에만** 기록한다(`crud.ts:54-60,116,131`). 미리 태우면 실패한 첫 시도가 재시도를 영원히 no-op 으로 만들어 **'발송했습니다'를 보여주고 아무것도 보내지 않는다.**
   또 하나: 멱등키 선례가 이 앱에 이미 둘 있다 — `pages/members/components/PointsCard.tsx:103,162-173`(`idempotencyKeyRef` — **성공해야 키를 버린다**)와 `pages/members/data-source.ts:243-253`(심: `Idempotency-Key: <uuid>`, 24h). 발송은 그보다 비가역이므로 최소 그 수준을 따른다.
2. **서버 상태 재검사** — `status` 가 `draft`\|`scheduled` 가 아니면 422. 멱등키가 뚫려도 이것이 2차 방어선이다.
3. **동기 제출 락 + 확인 다이얼로그** — 대상 수를 정확히 echo 하는 강화 confirm(EXC-18 결).
4. **`marketing:send` 권한 분리 권고** — '캠페인을 편집할 수 있다'와 '12,840명에게 실제로 보낼 수 있다'는 다른 권한이다.

### 7.4 발신번호 사전등록 — 서버가 정본 【보안 판정】
발신번호 사전등록제(전기통신사업법 제84조의2)는 **법적 요건**이다. 프론트는 미검증 번호를 `disabled` 옵션으로 잠그고 스키마로도 막는다(`validation.ts:43-50` · `SmsFormPage.tsx:306`). **그러나 그 판정 근거인 `verified` 플래그가 브라우저 안 픽스처 배열이다**(`_shared/store.ts:63-67`) — 사용자가 고칠 수 있는 값으로 법적 요건을 판정하고 있다는 뜻이다.

**서버가 정본이다.** EP-03/EP-04 는 저장 시점에, **EP-06 은 발송 시점에 다시** `senderId` 의 검증 상태를 확인하고 미검증이면 422 `SENDER_NOT_VERIFIED` 로 거절한다. 저장 시점 검사만으로 부족한 이유는 §7.6 과 같다 — **저장과 발송 사이에 번호 등록이 해지될 수 있다.**

### 7.5 광고 규제((광고)·수신거부·야간) — 서버가 정본, **시각 기준은 Asia/Seoul** 【보안 판정】
정보통신망법 제50조는 광고성 정보에 ①`(광고)` 표기 ②무료수신거부 안내 ③야간(21:00–08:00) 전송 제한을 강제한다. 프론트는 셋 다 스키마로 막는다(`validation.ts:74-115` · `messaging.ts:227-262`).

**두 가지가 서버 강제를 요구한다:**

1. **프론트 검증은 우회된다** — 스키마는 브라우저 코드다. 서버가 같은 규칙을 독립적으로 강제하지 않으면 법 위반 발송이 저장·실행된다.
2. **시각 기준이 어긋난다 — 2026-07-17 재확인, F3b 이후 오히려 선명해졌다.** 프론트의 야간 판정은 `new Date(value).getHours()`(`messaging.ts:258-262` `isNightAt`)로 **브라우저 로컬 타임존**을 쓰고, 과거 판정 `isPastLocal`(`validation.ts:17-21`)도 오프셋 없는 `datetime-local` 문자열을 로컬로 파싱한다. 해외 체류 중인 관리자의 브라우저가 UTC 면 한국 시각 23시를 14시로 읽어 야간 검사를 그냥 통과한다.
   **⚠ F3b 가 표시 축만 고쳤다**: `shared/format.ts:63` 이 `DISPLAY_TIME_ZONE = 'Asia/Seoul'` 을 고정하고 `:76-85,171-175` 가 `formatDateTime` 을 KST 로 찍는다. 그래서 **목록의 예약 시각은 KST 로 표시되는데 폼의 검증은 로컬 기준**이다 — 같은 값에 두 기준이 걸린다. UTC 브라우저에서 `2026-07-20T14:00` 으로 광고를 예약하면 **검증은 통과하고 목록에는 '23:00'** 으로 뜬다. (`format.ts:39-47` 의 UTC 정오 앵커 수렴은 달력 **일** 산술용이라 이 시각 축에 닿지 않는다.)
   **서버는 반드시 `Asia/Seoul` 고정 타임존으로 판정한다** — 법정 기준이 그것이기 때문이다. `scheduledAt` 은 오프셋 포함 ISO 8601 로 주고받아 해석 차이를 없앤다(현재 프론트는 `datetime-local` 의 오프셋 없는 문자열을 그대로 저장한다 — §7.12). **프론트도 `isNightAt`/`isPastLocal` 을 `shared/format` 의 KST 포매터 위로 옮겨야 표시와 검증이 한 기준이 된다** — 그 수렴은 이미 `format.ts` 에 있다.

**수신거부 문구 판정도 서버가 다시 한다.** 프론트의 `hasOptOut` 은 '무료수신거부'·'무료거부'·'수신거부' 문자열 **포함** 여부만 본다(`messaging.ts:232,238-240`) — '수신거부 불가' 같은 문구도 통과한다. 서버는 실제 무료 회신 번호(080 등)의 존재까지 확인하는 것이 옳다.

### 7.6 수신 동의(opt-in) — **발송 시점에 서버가 재확인한다** 【보안 판정 · TOCTOU】
이 도메인에서 가장 중요한 판정이다. BE-010 §7.7(카테고리 참조 무결성)과 **같은 논증 구조**를 따른다.

프론트는 `SegmentPicker` 로 세그먼트를 고르고(`SegmentPicker.tsx`) 그 세그먼트의 `recipientCount` 를 합산해 대상 수를 보여 준다(`messaging.ts:411-419`). 픽스처의 세그먼트 이름이 '전체 수신동의 회원'인 것에서 보이듯, **세그먼트 정의 자체가 수신 동의를 전제한다**.

**그러나 세그먼트 선택은 UX 편의이고 서버가 정본이다** — 프론트가 세그먼트를 조회한 시점(폼 마운트, `SmsFormPage.tsx:195` 의 `useMemo` 로 **1회 고정**)과 캠페인을 저장하는 시점, 그리고 **실제 발송되는 시점** 사이에는 시간이 흐른다. 그 사이에:

- 수신자가 **마케팅 수신 동의를 철회**할 수 있다. 철회 후 발송하면 **정보통신망법 위반**이다.
- 수신자가 **수신거부(080) 처리**를 했을 수 있다.
- 예약 발송이면 그 간격이 **며칠**일 수도 있다(픽스처 예약이 2주 뒤다).
- `recipientCount` 는 **저장 시점 스냅샷**으로 비정규화돼 있어(`types.ts:17` · `data-source.ts:81`) 시간이 지날수록 거짓이 된다.

**결정: 서버는 발송 트랜잭션 안에서 수신자별 동의 상태를 재조회하고, 동의가 유효하지 않은 수신자를 발송 대상에서 제외한다.** 세그먼트가 그 사람을 포함하더라도 그렇다. 제외 건수는 발송 결과(`stats`)에 반영하고 감사 로그에 남긴다(§7.9). 프론트의 세그먼트 선택과 대상 수 표시는 **예상치**이며 서버 검사가 정본이다 — 이 사실을 화면 문구도 이미 인정한다('중복 수신자는 발송 시 1회로 합산됩니다', `SegmentPicker.tsx:142-145`).

**중복 제거도 서버 몫이다**: `totalRecipients` 는 세그먼트별 인원을 **단순 합**한다(`messaging.ts:411-419`) — 두 세그먼트에 모두 속한 사람이 두 번 세어진다. 서버는 수신자 집합의 **합집합**으로 대상을 정하고, `recipientCount` 를 그 값으로 돌려줘야 한다.

### 7.7 발송 규모 상한 · 레이트리밋 — 대량 발송 남용 방지 【보안 판정】
현재 프론트에는 **대상 규모 상한이 없다**. 픽스처 세그먼트를 전부 고르면 21,345명이고, 실서비스에서는 그 자릿수가 더 크다. 발송 트리거 UI 가 붙는 순간 **오클릭 한 번이 수만 건의 비가역 발송과 그만큼의 과금**이 된다. 화면에는 비용 표시조차 없다(FS-034 §7 #14).

**서버가 정한다:**
- **캠페인당 대상 수 상한** — 초과 시 422 `RECIPIENT_LIMIT_EXCEEDED`. EP-03/EP-04(저장 시)와 EP-06(발송 시) **양쪽**에서 검사한다.
- **일일·시간당 발송 건수 상한**(계정 단위) — 초과 시 429 `RATE_LIMITED` + `Retry-After`.
- **EP-06 레이트리밋 분당 5회** — 발송 트리거는 드물게 일어나는 조작이다. 초당 여러 번 들어오면 그것은 사고이거나 공격이다.

구체적 수치는 계약 협의 대상이며 이 문서가 정하지 않는다 — **상한이 존재해야 한다는 것**이 판정이다.

### 7.8 치환변수 화이트리스트 — 미해석 변수 발송 방지 【보안 판정】
본문은 `#{이름}` 문법의 치환변수를 담는다(`messaging.ts:97-136`). 프론트는 **알려진 5개 토큰만** 표본값으로 치환해 미리보기하고, **모르는 변수는 그대로 둔다**(`applyVariableSamples` — `messaging.ts:130-136`). 삽입 바도 그 5개만 제공한다.

**그러나 본문은 자유 텍스트다.** 관리자가 `#{잔액}` 을 직접 타이핑하면 프론트는 막지 않는다 — 미리보기에 `#{잔액}` 이 그대로 보일 뿐이고, 그것을 눈치채지 못하면 **수신자에게 `#{잔액}` 이라는 날 토큰이 그대로 발송된다**.

**결정: 서버는 저장 시(EP-03/04) 본문의 모든 `#{...}` 토큰을 화이트리스트와 대조하고, 목록 밖 토큰이 있으면 422 `UNRESOLVED_VARIABLE`(`error.fields[].name = 'body'`)로 거절한다.** 발송 시(EP-06)에도 다시 검사한다 — 화이트리스트가 그 사이 줄었을 수 있다. **치환 자체도 서버가 수행한다**: 치환 값은 수신자의 개인정보이므로 관리자 브라우저가 실제 값을 알아서는 안 된다(프론트가 **표본값**만 쓰는 현재 설계가 옳다). 치환 시 값은 반드시 **이스케이프**한다 — 회원 이름에 담긴 문자가 문법으로 해석되면 안 된다.

### 7.9 발송 감사 로그 — 법적 증빙 【보안 판정】
마케팅 발송은 분쟁 시 **'누가 · 언제 · 몇 명에게 · 무엇을 · 어떤 동의 근거로' 보냈는가**를 증명해야 한다(정보통신망법상 수신동의 증빙 의무). 현재 화면에는 그 흔적이 없다 — 캠페인 레코드에 작성자·발송 실행자·실행 시각 필드가 아예 없고(`types.ts:8-29`), `stats` 는 성공/실패 숫자뿐이다.

**서버가 남긴다** (프론트 계약이 아니라 서버 의무):
- **발송 실행 감사**(EP-06): 실행 주체(관리자 id) · 실행 시각 · 캠페인 스냅샷(본문·발신번호·세그먼트 구성) · 최종 대상 수 · **동의 검증 결과**(제외된 수신자 수와 사유 — §7.6) · 멱등키.
- **수신자별 전송 기록**: 수신 동의 근거(동의 시각·경로)와 전송 결과. 이것이 §7.6 의 재확인을 **증명 가능하게** 만든다.
- **캠페인 변경 감사**(EP-03/04/05): 누가 언제 무엇을 바꿨는가. 발송된 캠페인의 사후 수정을 §7.2 가 422 로 막지만, 그 시도 자체도 기록 대상이다.
- **보존 기간**: 발송 이력은 법적 증빙이므로 캠페인 삭제(EP-05)와 **독립적으로** 보존한다 — 관리자가 캠페인을 지워도 증빙이 사라지면 안 된다. `sent` 캠페인은 **soft-delete** 를 권고한다(§4 EP-05).

### 7.10 발신번호·세그먼트·템플릿 — 심 없음. 지어내지 않는다
셋 다 프론트가 **동기 픽스처 함수를 직접 호출**한다 — 어댑터도 `// TODO(backend)` 주석도 없다(§6). 그래서 **엔드포인트를 명세하지 않는다**(EP-07/08/09 = 심 없음).

그러나 이 셋은 **다른 심 없음과 무게가 다르다**:
- 발신번호의 `verified` 는 **법적 요건의 판정 근거**다(§7.4).
- 세그먼트 인원은 **수신 동의 상태에서 파생되는 서버 계산값**이다(§7.6).
- 둘 다 브라우저 안 배열에 있다 = **판정 근거가 클라이언트에 있다**.

실연동 시 이 셋에 심을 뚫는 것이 **가장 먼저 와야 한다** — 캠페인 CRUD 만 서버로 옮기고 이 셋을 픽스처로 남기면, 서버가 저장 시점에 `senderId`·`segmentIds` 를 검증할 근거를 갖지 못한다. §7.12 로 이관.

### 7.11 세션 만료와 작성 중 입력
401 은 쿼리 계층 한 곳에서 가로채 재인증 경로로 보낸다(`queryClient.ts:36-38`). **그러나 작성 중이던 폼 입력은 보존되지 않는다** — 앱이 스스로 이동하므로 미저장 가드(FS-034-EL-035)가 발화하지 않는다. 발송 폼은 본문·세그먼트·예약을 모으는 긴 작업이라 손실이 크다. 서버 계약의 문제는 아니나(EXC-19 · 앱 전역), 이 화면이 그 피해자 중 하나임을 기록한다.
