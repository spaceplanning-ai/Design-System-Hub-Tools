# FS-034. SMS 발송 (목록·등록/수정)

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 목적 | SMS/LMS/MMS 발송 캠페인을 발송상태·키워드로 찾고, 발신번호·수신자 세그먼트·본문(바이트 자동판정)·광고성·예약을 지정해 **초안 또는 예약으로 저장**하며, 삭제(단건·일괄)한다 |
| 역할(주 사용자) | 관리자 (구현에 역할 분기 없음 — §4.1) |
| 진입 경로 | 좌측 GNB > 마케팅 관리 > SMS 발송 (`/marketing/sms`) |
| 포함 화면 | 목록 `/marketing/sms` · 등록 `/marketing/sms/new` · 수정 `/marketing/sms/:id/edit`. **상세 조회 라우트는 없다**(행 클릭 = 수정 진입) |
| 구현 경로 | `apps/admin/src/pages/marketing/sms/**` · 공용 `apps/admin/src/pages/marketing/_shared/**` |
| 대응 SCR | SCR-034 (미작성 — §7) |
| 공통 컴포넌트 | `shared/crud/{CrudListShell,CrudTable,useCrudList,useCrudForm,createCrudAdapter,FormServerError,FormConflictDialog,requiredText}` · `shared/ui/{SearchField,SelectField,StatusBadge,Button,Card,CardTitle,FormField,TextareaField,ToggleSwitch,Alert,ConfirmDialog,SelectionBar,RowActions,Empty,ToastProvider,useUnsavedChangesDialog}` · marketing `_shared/{messaging,store,SegmentPicker,VariableInsertBar}` |
| **저장은 발송이 아니다** | 이 화면의 어떤 동작도 문자를 전송하지 않는다. 저장은 캠페인(초안/예약) 레코드를 만들 뿐이며, 실제 전송·예약 실행은 서버 몫이다(§5 '현재 구현 상태'). 화면 문구도 그렇게 고지한다(FS-034-EL-014·EL-031) |

### 4.1 공통 예외 규칙

| 규칙 | 내용 |
|---|---|
| **저장 ≠ 발송** | 이 화면의 어떤 경로도 문자를 보내지 않는다. 목록의 '발송중'·'발송완료'·'취소' 상태는 **픽스처 시드 값**이며, 화면이 만들어 낸 것이 아니다. 실제 전송·예약 실행·통계 누적은 백엔드 몫이다(§5) |
| 검색·필터의 자리 | 검색·상태 필터는 **서버 조회가 아니라 클라이언트 필터**다. 어댑터는 언제나 전량을 돌려주고 화면이 거른다. 그래서 조건을 바꿔도 네트워크가 발생하지 않고 로딩도 없다 |
| 목록 상태 4분기 | 최초 로드(스켈레톤) · 재조회(이전 행 유지 + '새로고침 중…') · 0행(빈 상태 3분기) · 조회 실패(배너)를 정확히 하나만 그린다. 재조회는 표를 비우지 않는다 |
| 네트워크 단절 | 전용 처리가 없다 — 조회는 실패 배너, 쓰기는 각 요소 실패로 떨어진다. 오프라인 감지·배너·복구 refetch 는 구현에 없다 — §7 |
| 세션 만료 | 401 은 **쿼리 계층 한 곳**에서 가로채 재인증 경로로 보낸다(화면별 처리 없음). 다만 **작성 중이던 폼 입력은 보존되지 않는다** — §7 |
| 요청 타임아웃 | 프론트 타임아웃 상한이 없다. abort 는 화면 이탈·언마운트·다이얼로그 닫기 시에만 발생한다 — §7 |
| 중복 제출 | 폼 제출은 `disabled` **와** 동기 제출 락을 함께 쓴다 — 비동기 검증 때문에 버튼이 실제로 잠기기까지의 틈으로 두 번째 요청이 새지 않는다. 삭제·일괄 삭제는 진행 중 확인 버튼 잠금 |
| 취소는 실패가 아니다 | abort 된 요청은 토스트도 배너도 만들지 않고, 일괄 작업의 실패 건수에도 세지 않는다 |
| 실패 통지의 자리 | ① 목록 조회 실패 = 인라인 배너 ② 폼 저장 실패 = 폼 상단 배너(+예상외면 참조 코드) ③ 필드 거절(422) = 그 입력의 인라인 에러 + 포커스 ④ 충돌(409/412) = 전용 다이얼로그 ⑤ 삭제 실패 = 다이얼로그 안 배너 ⑥ 삭제 **성공** = 토스트 |

> ```
> // TODO(backend): GET/POST /api/marketing/sms · GET/PUT/DELETE /api/marketing/sms/:id
> //   발송 실행: POST /api/marketing/sms/:id/send (예약은 스케줄러가 시각에 실행) — 프론트는 트리거만.
> ```
> **`/send` 심은 주석에만 있고 프론트 호출부가 0건이다** — 어댑터 함수도, 발송 버튼도, 발송 확인 다이얼로그도 없다. 즉 **이 화면에서 문자를 보낼 방법이 현재 존재하지 않는다**(FS-034-EL-036). 목록의 '발송중'·'발송완료' 행과 성공률 수치는 픽스처 시드가 만든 것이다. 위 표는 백엔드 연결 후 의도된 동작이며, `/send` 는 **UI 가 함께 구현돼야 성립하는 계약**이다.
