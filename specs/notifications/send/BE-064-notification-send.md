# BE-064. 알림 발송 규칙 백엔드 기능 명세

## 1. 개요

| 항목 | 내용 |
|---|---|
| 대상 화면 | FS-064 알림 발송 (`/notifications/send` · `/new` · `/:id/edit`) |
| 범위 | 발송 규칙 목록 조회 · 상세 조회 · 등록 · 수정(자동 발송 ON/OFF 포함) · 삭제 |
| **범위 밖 — 실 발송** | **발송은 이 계약에 없다.** 프론트에서 나가는 발송 요청이 **0건**이다(`data-source.ts:5-7,17`). 발송 주체는 이벤트를 받은 서버다: `data-source.ts:16-17` 이 그 흐름을 못박는다 — '이벤트 버스가 트리거를 받으면 규칙을 조회해 템플릿을 렌더하고 발송사(SMS/메일)로 넘긴 뒤 retryPolicy 만큼 재시도한다.' 이 화면은 그 서버 동작의 **입력(규칙)** 만 관리한다. **마케팅의 `POST /api/.../send` 같은 발송 트리거를 만들지 않는다**(`data-source.ts:6-7`) |
| **범위 밖 — 템플릿 CRUD** | BE-065(이메일)·BE-066(SMS) 소관. 이 화면은 **소비자**다(§7.3) |
| **범위 밖 — 발송 이력·통계·재시도 로그** | 화면에 표면이 없다. 규칙 목록에 '마지막 발송'·'실패 수' 열이 없고 어댑터에도 그 조회가 없다 — **없는 것을 여기서 만들지 않는다**(§7.9) |
| **범위 밖 — 세그먼트·예약·(광고) 표기·야간 제한·수신거부·알림톡 승인** | 마케팅 관리(BE-036 및 형제) 소관. 법적 근거는 §7.6 |
| 전제 | BE-003 §2·§3 을 상속한다. 인증은 세션 쿠키 기반. 모든 경로는 `/api` 프리픽스. 응답 본문은 `application/json; charset=utf-8`. 시각은 ISO 8601(오프셋 포함) |
| 프론트 어댑터 | `apps/admin/src/pages/notifications/send/data-source.ts` (`ruleAdapter`) — 공용 `createStoreAdapter` 로 배선 |
| 도메인 타입 | `apps/admin/src/pages/notifications/_shared/notification.ts` — **이 화면에 `types.ts` 가 없다.** 알림 도메인 타입·순수 규칙을 세 화면(발송 규칙·이메일 템플릿·SMS 템플릿)이 공유하는 잎 모듈에 모았다 |
| 검증 정본 | zod 스키마 `createRuleSchema(selfId)`(`send/validation.ts:23-55`). **저장소를 읽는 팩토리**다 — '중복 금지'는 한 레코드만 봐서는 판정할 수 없기 때문(`validation.ts:7-9`) |

> **에러 봉투·권한 모델 상속**: BE-003 §2·§3 을 그대로 상속한다(BE-010 §1 · BE-026 §1 과 동일 선언). 아래는 알림 발송 규칙 고유의 차이만 기술한다.

## 2. 공통 (상속)

- **에러 봉투**: BE-003 §2. 공통 에러코드 동일(`VALIDATION_FAILED` · `UNAUTHENTICATED` · `FORBIDDEN` · `CSRF_TOKEN_INVALID` · `NOT_FOUND` · `CONFLICT` · `UNPROCESSABLE` · `RATE_LIMITED` · `INTERNAL_ERROR` · `REQUEST_TIMEOUT`).
- **권한**: `admin` = 전체. `operator` = **조회 계열만**(EP-01·EP-02). **쓰기(EP-03·EP-04·EP-05)는 `admin` 전용**이다 — BE-026(1:1 문의)이 `operator` 에게 쓰기를 연 것과 **반대 판정**이며 근거는 §7.7 이다. 알림 관리 도메인 읽기 권한 없는 주체 → **403**(컬렉션·개별 모두 — §7.8).
- **CSRF**: 쓰기(POST·PUT·DELETE)에 `X-CSRF-Token`.
- **타임아웃**: 조회·쓰기 5초 → 504.
- **프론트 역할 분기 없음**(FS-064 §4.1) — 권한 강제는 서버 책임. 프론트에 쓰기 게이팅이 배선돼 있지 않다(FS-064 §7 #10)는 사실이 이 원칙을 바꾸지 않는다.

## 3. 데이터 계약 (`_shared/notification.ts` 대조)

| 타입 | 필드 | 비고 |
|---|---|---|
| `NotificationRule` | `id` · `trigger` · `channel` · `templateId` · `enabled` · `retryPolicy` · `updatedAt`(ISO) | 목록·상세가 **같은 타입**이다 — 필드가 7개뿐이라 Summary 분리가 필요 없다(§7.9). `updatedAt` 은 **표시에도 쓰이지 않는다**(규칙 목록에 '수정일시' 열이 없다) — 그럼에도 존재한다(§7.1) |
| `NotificationRuleInput` | `Omit<NotificationRule, 'id' \| 'updatedAt'>` | 저장 입력. **id·수정시각은 서버가 정한다** |
| `TriggerId` | `order.placed` \| `order.paid` \| `order.canceled` \| `delivery.started` \| `delivery.completed` \| `account.created` \| `account.password-reset` \| `account.dormant` \| `security.login-new-device` \| `security.verification-code` | 10개 **고정 리터럴 튜플**(`TRIGGER_ID_VALUES` — `notification.ts:77-88`). zod `z.enum` 이 그대로 소비한다 — 목록과 스키마가 단일 원천(`:76`) |
| `TriggerCategory` | `order` \| `delivery` \| `account` \| `security` | 표시 분류. 톤: 주문=info · 배송=success · 계정=neutral · 보안=warning |
| `NotificationChannel` | `email` \| `sms` | **알림톡이 없다** — 사전 승인·광고 심사는 캠페인 규제의 산물이다(`notification.ts:176-179`) |
| `RetryPolicy` | `none` \| `once` \| `thrice` | 재시도 안 함 / 1회 / 3회. **마케팅엔 없는 축**(`notification.ts:334-337`) |
| `templateId` | `string` | 채널에 따라 이메일/SMS 템플릿 저장소를 가리킨다(`notification.ts:401-402`). **외래키인데 계약이 그것을 강제하지 않는다** — §7.4 |

**규칙 유일성 (서버가 정본)**

| 규칙 | 내용 |
|---|---|
| `(trigger, channel)` 유일 | 한 이벤트에 같은 채널 규칙이 둘이면 **수신자가 같은 알림을 두 번 받는다**(`notification.ts:525` · `validation.ts:44`). 수정 중인 자신은 제외한다(`selfId`). **이론상 최대 규칙 수 = 10 트리거 × 2 채널 = 20건** |
| `templateId` 참조 | 채널과 트리거가 **둘 다** 맞는 템플릿이어야 한다(`store.ts:386-394`). 트리거가 다른 템플릿을 걸면 **그 이벤트가 주지 않는 변수가 빈칸으로 나간다**(`store.ts:384-385` · `RuleFormPage.tsx:126-127`) |
| `templateId` 필수 | 빈 문자열 금지 — '문구가 없으면 이벤트가 와도 보낼 것이 없다'(`validation.ts:33`) |

**정렬 (서버가 정본이어야 한다)**: 트리거 정의 순서(주문→배송→계정→보안), 같으면 채널 오름차순(email→sms) — `sortRules`(`notification.ts:496-502`). 프론트도 같은 함수를 갖지만(`store.ts:359-361` 이 이미 정렬해 돌려준다) **서버 순서가 정본**이어야 한다.

**서버가 파생하지 않는 것**: 분류 건수(`countByCategory`) · 켜짐 수(`countEnabled`) · 검색(`searchRules`) — **전부 프론트 순수 함수**다. 규칙이 20건 상한이라 서버 왕복이 필요 없다(§7.9).

## 4. 엔드포인트 명세

### BE-064-EP-01 · 발송 규칙 목록 조회
| 항목 | 내용 |
|---|---|
| 근거 (FS) | FS-064-EL-002.2, EL-004, EL-006, EL-007, EL-009, EL-009.1~.10, EL-010, EL-011, EL-012, EL-017 |
| 메서드·경로 | `GET /api/notifications/rules` |
| 심 | `data-source.ts:14` |
| 권한 | `admin`, `operator` |
| 멱등성 | 멱등(GET) |
| 페이징 | **없음 — 전량 반환이다.** 규칙은 `(trigger, channel)` 유일성이 **20건 상한을 강제**하므로 페이징이 불필요하다(§7.9) |
| 레이트리밋 | 분당 120회 |

**쿼리**: **없다.** 프론트가 분류 필터·검색·정렬을 **전부 클라이언트에서** 수행하므로(`RuleListPage.tsx:92-99`) 어댑터 시그니처 `fetchAll(signal)` 이 파라미터를 받지 않는다.

**응답 200** — `readonly NotificationRule[]`. **트리거 정의 순서 → 채널 순**으로 내려준다(§3).

> **템플릿명 조인 권고(§7.4)**: 현재 프론트는 응답에 없는 템플릿명을 `_shared/store.templateNameOf()` 로 **로컬에서** 만든다(`RuleListPage.tsx:128`). 서버 연결 시 그 조회는 **존재할 수 없다**(브라우저에 템플릿 저장소가 없다). 목록 응답에 `templateName: string`(삭제된 템플릿이면 `''`)을 **조인해 내려주는 것이 옳다** — 그러면 화면 코드가 바뀌지 않고 FS-064-EL-009.6 의 '템플릿 없음' 배지가 그대로 산다.

**에러**: 401 · 403 · 429 · 500 · 504.

### BE-064-EP-03 · 발송 규칙 등록
| 항목 | 내용 |
|---|---|
| 근거 (FS) | FS-064-EL-005, EL-034, EL-035, EL-038, EL-040 |
| 메서드·경로 | `POST /api/notifications/rules` |
| 심 | `data-source.ts:14` |
| 권한 | **`admin` 전용**(§7.7) |
| 멱등성 | **멱등키 필수** — `Idempotency-Key: <uuid>`(§7.10). 프론트가 제출 시도 단위 키를 만들어 `WriteContext.idempotencyKey` 로 싣는다(`useCrudForm.ts:118-123,235` → `crud.ts:197-204`) |
| 레이트리밋 | 분당 60회 |

**바디**(`NotificationRuleInput`): `trigger` · `channel` · `templateId` · `enabled` · `retryPolicy`.

**서버 검증 (요청을 그대로 믿지 않는다)**
1. **`trigger` 는 서버 카탈로그의 값**이어야 한다 — 프론트는 코드 상수 10개를 갖지만(§3) 정본은 서버다(§7.11).
2. **`(trigger, channel)` 유일성** — 이미 있으면 **409**(§7.2). 프론트가 1차 차단하나 그 판정은 **클라이언트 픽스처 기준**이라 신뢰할 수 없다.
3. **`templateId` 참조 검증** — ① 존재하는가 ② `channel` 이 맞는가 ③ **`trigger` 가 맞는가**. 셋 중 하나라도 어긋나면 **422**(§7.4).
4. `enabled` · `retryPolicy` 는 열거 값.

**응답 201/204**. 프론트 `create(input, context): Promise<void>` — 응답 본문을 읽지 않고 목록으로 이동한다.

**에러**: 400 `VALIDATION_FAILED` · 401 · 403 `FORBIDDEN` · 403 `CSRF_TOKEN_INVALID` · **409 `DUPLICATE_RULE`**(§7.2) · **422 `TEMPLATE_NOT_ELIGIBLE`**(§7.4) · 429 · 500 · 504.

### 6.1 어댑터 함수 본문에 요구되는 사항 (시그니처 불변)

- 쓰기(POST·PUT·DELETE)에 `X-CSRF-Token` 헤더.
- `fetchOne` 404 → `HttpError(404, …)` **변환 — 이미 충족**(`crud.ts:192-194`). 백엔드 연결 시 이 동작을 **유지**하라(회귀 금지).
- `update`/`remove` 409 → `HttpError(409, …)` **변환 — 이미 충족**(`crud.ts:219-221,232-234`). 서버 메시지를 그대로 실어야 §7.2 의 세 갈래가 화면에서 갈린다.
- `WriteContext.idempotencyKey` → `Idempotency-Key: <key>` 헤더(`crud.ts:39`).
- **`update` 에 `If-Match: <etag>` 를 추가해야 한다**(§7.1) — 이것만 시그니처 밖의 신규 요구다.

**시그니처를 바꿔야 하는 항목(§7.3)**: `templateOptionsFor()` · `templateNameOf()` · `listRules()`(스키마 안) 은 **동기·비취소·비실패** 함수다. 백엔드 연결 시 `Promise<T>` 로 바뀌며, 그 순간 **호출부가 함께 바뀐다**:
- `RuleFormPage.tsx:128` 의 `useMemo` → `useQuery`(로딩·실패·재시도 UI 가 **신규 요구사항**으로 발생 — FS-064 §4 가 'N/A — 로딩 상태가 없다'로 적어 둔 칸이 살아난다).
- `RuleListPage.tsx:96,128` → **EP-01 조인으로 해소하면 화면이 바뀌지 않는다**(그래서 조인을 권고한다 — §7.4).
- `validation.ts:46` 의 `hasDuplicateRule(listRules(), …)` → **zod 스키마는 동기다.** 비동기 검증으로 바꾸거나 **클라이언트 사전 검사를 버리고 서버 409 에 의존해야 한다**(§7.2 가 후자를 택한다).

**⚠ 연동 산정에 반드시 포함**: 이 화면은 '어댑터 본문만 바꾸면 끝'이 **아니다**. 위 세 호출부가 화면 코드 변경을 강제한다.

## 7. 핵심 판정

### 7.1 낙관적 동시성 — `updatedAt` 이 있는데 쓰지 않는다 【정합 판정】

**현재 상태**: `NotificationRule.updatedAt`(`notification.ts:405`)이 존재한다. 저장소가 매 쓰기마다 갱신한다(`store.ts:371,375`). **그런데 어디에도 실리지 않는다** — 어댑터가 `If-Match`/`version` 을 보내지 않고, `NotificationRuleInput` 이 `updatedAt` 을 `Omit` 한다(`notification.ts:408`). 규칙 목록에 '수정일시' 열조차 없어 **표시에도 쓰이지 않는다**.

**어댑터의 409 가 무엇을 막고 무엇을 못 막는가 — 이 구분을 흐리지 말 것**:

| 상황 | 현재 결과 | 근거 |
|---|---|---|
| A 가 편집 중 B 가 **삭제** → A 저장 | **409 → 충돌 다이얼로그.** 유령 저장 없음 ✔ | `crud.ts:219-221` `if (!exists(id)) throw HttpError(409, '다른 사용자가 먼저 삭제한 항목입니다.')` |
| A 가 편집 중 B 가 **수정** → A 저장 | **A 가 조용히 이긴다 — last-write-wins.** B 의 변경은 흔적 없이 사라진다 ✘ | `crud.ts:219` 의 검사는 **존재 여부**다. 둘 다 존재하므로 통과한다 |

**즉 409 는 '존재 여부' 기반이지 버전 토큰이 아니다.** F3b 가 해소한 것은 **유령 저장**이지 **동시 편집**이 아니다.

**이것이 만드는 사고**: 운영자 A 가 '주문 접수 → 이메일' 규칙의 템플릿을 새 문구로 바꾸는 동안, 운영자 B 가 같은 규칙의 재시도를 3회로 올린다. A 가 나중에 저장하면 **B 의 재시도 설정이 사라진다** — 전체 치환 PUT 이라 서버는 이것이 유실인지 정상 갱신인지 구분할 수 없다. 규칙은 20건뿐이고 운영자 여럿이 같은 것을 만지므로 **충돌 확률이 낮지 않다**.

**판정**: EP-04 는 **`If-Match: <etag>`** 를 요구한다. ETag 는 `updatedAt` 에서 파생하면 족하다(새 필드가 필요 없다). 불일치 시 **409 `STALE_WRITE`**. **프론트는 이미 그 409 를 받을 준비가 돼 있다** — `useCrudForm` 의 충돌 다이얼로그가 입력을 보존한 채 '최신 다시 불러오기 / 닫기'를 준다(`useCrudForm.ts:166-178`). 어댑터가 헤더를 싣기만 하면 **화면 코드 0줄로** 복구 경로가 열린다.

**단, 토글(FS-064-EL-009.8)에는 그 다이얼로그가 없다** — 실패 토스트뿐이다. 토글은 `enabled` 한 필드만 다투므로 **last-write-wins 가 오히려 옳다**(마지막으로 스위치를 누른 사람의 의도가 이긴다). 결론: **`If-Match` 는 폼 저장에만 요구하고 토글은 면제한다.** 그러면 같은 엔드포인트가 헤더 유무로 갈리므로, **헤더가 있으면 검사하고 없으면 통과**시킨다(조건부 요청의 표준 의미와 일치한다).

### 7.2 `(trigger, channel)` 유일성 — 서버가 정본이다 【보안 판정】

**현재 클라이언트가 유일성을 판정한다**: zod 스키마가 `hasDuplicateRule(listRules(), trigger, channel, selfId)` 를 부른다(`validation.ts:46`). `listRules()` 는 **브라우저 안 배열**이다(`store.ts:359-361`).

**왜 이것이 보안 문제인가**: 이 검사는 **끄기 쉽다** — 요청을 직접 만들면 스키마를 거치지 않는다. 그러면 한 이벤트에 같은 채널 규칙이 둘이 되고, **수신자가 같은 알림을 두 번 받는다**(`notification.ts:525`). 인증번호 규칙이 둘이면 **인증번호가 두 개 발송되고**, 그중 어느 것이 유효한지 서버만 안다 — 사용자는 나중 것을 쓰다 실패한다. 주문 규칙이 둘이면 고객이 같은 메일을 두 번 받아 **중복 주문을 의심한다**. 게다가 **발송 비용이 두 배가 된다**(SMS 는 건당 과금 — `SmsTemplateListPage.tsx:57`).

**판정**: 유일성은 **서버가 DB 유니크 제약과 409 로 강제한다.** 클라이언트 검사는 **UX 편의이며 계약이 아니다** — 제거해도 계약은 성립해야 한다.

**⚠ 프론트가 이 409 를 잘못 그린다 — UI 기획 이관**: `useCrudForm` 의 충돌 다이얼로그는 **'최신 다시 불러오기 / 닫기'** 두 선택지를 준다(`useCrudForm.ts:172-176`). 그런데 **중복 규칙 409 는 다시 불러와도 해소되지 않는다** — 사용자가 할 일은 채널을 바꾸거나 기존 규칙을 수정하는 것이다. 서버가 `DUPLICATE_RULE` 을 **409 가 아니라 422 `error.fields.channel`** 로 내면 프론트가 **이미 옳게 그린다**(`useCrudForm.ts:181-192` 가 필드 인라인 + 포커스). 클라이언트 스키마가 같은 메시지를 같은 필드(`channel`)에 꽂고 있어(`validation.ts:47-52`) **표면이 정확히 일치한다.**

**결론: `DUPLICATE_RULE` 은 409 가 아니라 422 로 낸다.** `error.fields = [{ field: 'channel', message: "'<이벤트명>' 이벤트의 <채널명> 규칙이 이미 있습니다. …" }]`. 409 는 **동시성 충돌 전용**으로 남긴다(§7.1). 이것이 프론트 코드 0줄로 옳은 화면을 만드는 유일한 선택이다.

### 7.3 템플릿 후보 조회 — 심이 없고, 연결하려면 화면 코드가 바뀐다 【연동 판정】

`RuleFormPage.tsx:128` 이 `templateOptionsFor(channel, trigger)` 를 **어댑터를 거치지 않고 `_shared/store` 에서 동기 직접 호출**한다. `// TODO(backend)` 주석이 없다 — **연동 심이 아예 없다.**

**이것이 왜 문제인가**: 다른 조회는 어댑터 본문만 fetch 로 바꾸면 연동이 끝난다(시그니처 불변이 이 앱의 연동 전략이다). 그러나 이 호출은 **동기 함수**라 `Promise` 를 반환하는 순간 `useMemo(() => templateOptionsFor(channel, trigger), [channel, trigger])` 가 성립하지 않는다. **화면이 `useQuery` 를 새로 배선해야 하고**, 그러면 FS-064 §4 가 'N/A — 로딩 상태가 없다 / 실패 상태가 없다'로 적어 둔 칸(EL-027·EL-028)이 **전부 새 요구사항으로 살아난다** — 로딩, 실패 배너, 재시도, 취소. 게다가 이 조회는 **`channel`·`trigger` 가 바뀔 때마다 다시 나가므로** 폼 안에서 왕복이 발생한다(현재는 동기라 무료다).

**판정**: 이 문서는 EP-06 의 엔드포인트를 **만들지 않는다** — 이메일 템플릿은 BE-065, SMS 템플릿은 BE-066 이 정본이다. 이 화면은 **소비자**다.

**대안 검토 — 후보 조회를 없앨 수 있다**: EP-01 이 템플릿명을 조인하듯(§7.4), **폼 진입 시 `GET /api/notifications/{email,sms}-templates?trigger=<id>`** 한 번으로 후보를 받는 편이 왕복을 줄인다. 그 쿼리 파라미터는 BE-065/BE-066 이 정의해야 한다 — **현재 그 계약에 `trigger` 쿼리가 없다**(BE-065 §4 · BE-066 §4 는 전량 반환이다). **연동 착수 전 UI 기획 이 이 두 화면의 비동기화를 별도 티켓으로 잡아야 한다 — 어댑터만 바꾸면 되는 작업으로 산정하면 반드시 빠진다.**

### 7.4 `templateId` 는 외래키다 — 서버가 참조를 검증하고, 목록은 이름을 조인한다 【보안·정합 판정】

**참조 검증**: `templateId` 는 채널에 따라 이메일/SMS 템플릿을 가리키는 **외래키**다(`notification.ts:401-402`). 현재 그 무결성을 지키는 것은 **후보를 좁힌 select 하나**뿐이다(`store.ts:386-394`). 요청을 직접 만들면 ① 없는 templateId ② **다른 채널의** 템플릿 ③ **다른 트리거의** 템플릿을 걸 수 있다.

**③이 가장 위험하다**: '배송 출발' 규칙에 '인증번호' 템플릿을 걸면, 그 템플릿의 `#{인증번호}` 를 배송 이벤트가 주지 않으므로 **빈칸으로 발송된다**(`store.ts:384-385` · `notification.ts:266`). 고객은 '인증번호 를 입력해 주세요'라는 문구를 받는다. 더 나쁜 경우 발송사가 그것을 **정상 메시지로 처리해 과금**한다.

**판정**: EP-03·EP-04 는 **`templateId` 의 존재·채널 일치·트리거 일치를 서버가 재판정**한다. 위반 시 **422 `TEMPLATE_NOT_ELIGIBLE`** + `error.fields = [{ field: 'templateId', … }]` → 프론트가 인라인 오류로 그린다(`useCrudForm.ts:181-192`). **404 가 아니다** — 요청 자체는 유효하고 값의 조합이 처리 불가한 것이다.

**목록 조인**: EP-01 의 응답에 **`templateName: string`** 을 조인한다(삭제된 템플릿이면 `''`). 근거: ① 현재 프론트가 `templateNameOf()` 로 만드는 값인데 **서버 연결 후에는 브라우저에 템플릿 저장소가 없다** ② 규칙마다 템플릿을 따로 조회하면 N+1 이다 ③ **`''` 규약을 유지하면 화면이 안 바뀐다** — FS-064-EL-009.6 의 '템플릿 없음 — 발송되지 않습니다' danger 배지가 `name === ''` 로 갈리므로(`RuleListPage.tsx:130`) 서버가 그 규약을 그대로 따르면 된다. 검색도 그 이름을 쓴다(`RuleListPage.tsx:95-97`).

### 7.5 템플릿 삭제가 규칙을 조용히 깨뜨린다 — 막혔다. `trigger` 변경만 남았다 【정합 판정 · 실재 결함(축소)】

> **⚠ 기준 갱신(`4b805ad` → `a5c2639`)으로 이 절의 전제가 뒤집혔다.** 이 화면은 **참조하는 쪽**이라 자기 코드로는 방어할 수 없었고, 이전 판정의 근거는 '`rulesUsingTemplate` 소비처 0건' 이었다. **그 함수는 이제 불린다.**

**① 삭제 — 해소됐다**: `_shared/store.ts:82-103` 의 `remove`(이메일·SMS 템플릿 저장소가 **공유하는 골격**)가 `rulesUsingTemplate(id)`(`:403-405`)를 **호출해** 참조 규칙이 있으면 `HttpError(HTTP_STATUS.conflict, '발송 규칙 N건이 이 템플릿을 쓰고 있어 삭제할 수 없습니다. 규칙에서 먼저 템플릿을 바꾸세요.')` 를 던진다(`:95-101`). 주석 `:84-93` 이 **이 화면의 관점을 그대로 인용한다** — '규칙 화면이 "템플릿 없음" 배지를 띄우긴 하지만 그건 **다른 화면에서, 이미 망가진 뒤의 이야기**다'. 회귀 `store.test.ts:69-98`.

**즉 이 화면의 규칙이 참조하는 템플릿은 이제 지워지지 않는다.** `RuleListPage.tsx:131` 의 '템플릿 없음 — 발송되지 않습니다' danger 배지는 **삭제 경로로는 더 이상 발생하지 않는다**(아래 ②·③ 경로로는 여전히 가능하므로 배지 자체는 유지가 옳다).

**② `trigger` 변경 — 미해소**: 템플릿의 `trigger` 를 바꾸면 참조 규칙은 그대로 그 templateId 를 가리킨다. `patch`(이메일 `store.ts:183-190` · SMS `:253-259`)에 참조 무결성 검사가 **없다** — `remove` 에만 들어왔다. 이 화면의 규칙은 **배지도 뜨지 않은 채**(`templateNameOf` 가 이름을 잘 찾는다) **엉뚱한 문구를 렌더한다** — §7.4 의 '미치환 변수 발송' 과 정확히 같은 결과이고, **삭제보다 나쁘다**(조용히 잘못된 메일이 나간다).

**③ 백엔드 연결 후**: 위 ①은 **프론트 픽스처의 방어**다. 서버가 정본이 되면 사라진다 — 그래서 **계약 판정은 그대로 유지한다**.

**판정(유지)**: 이것은 **참조 무결성 문제이며 서버가 정본**이다. `DELETE /api/notifications/{email,sms}-templates/:id`(BE-065-EP-05 · BE-066-EP-05)는 **그 템플릿을 참조하는 규칙이 있으면 409 `TEMPLATE_IN_USE`** 를 반환하고, 응답에 참조 규칙 수를 실어 화면이 '이 템플릿을 쓰는 발송 규칙 N건이 있습니다'를 말할 수 있게 한다. **프론트 픽스처가 이미 이 모양으로 던지므로 서버는 그것을 재현하면 된다.** `PUT …/:id` 의 `trigger` 변경도 같은 계약이며 **이쪽은 픽스처에도 없다**. 대안(참조 규칙을 함께 비활성화 · `templateId` 를 null 로)은 **운영자가 모르는 사이에 알림이 멎게 하므로 채택하지 않는다**.

**프론트 후속(UI 기획)**: **★ 신규 발현 — 409 는 오는데 화면이 그 문구를 버린다.** 이전엔 삭제가 통과해 버려서 드러나지 않던 결함이다. `useCrudList` 의 삭제 다이얼로그가 실패를 열린 채로 배너에 그리지만(`useCrudList.tsx:110-113,161-163`) **문구가 '삭제하지 못했습니다. 잠시 후 다시 시도해 주세요.' 고정**이라(`:112`) 재시도하면 될 것처럼 말한다 — **재시도해도 영원히 실패한다.** 409 의 메시지를 그대로 보이도록 고쳐야 한다. **또한 삭제 전 경고(`rulesUsingTemplate`)를 살리는 것이 여전히 옳다** — 409 는 최후 방어선이지 첫 안내가 아니다.

### 7.6 보안 알림을 끄는 것은 보안 통제 해제다 — 감사 로그를 남긴다 【보안 판정】

**코드가 말하는 사실**: 트리거 10개 중 2개가 `security` 분류다 — `security.login-new-device`('기존에 없던 기기·위치에서 로그인하면 발생합니다.' — `notification.ts:150-154`) · `security.verification-code`('본인확인 인증번호가 필요할 때 발생합니다.' — `:155-160`). `account.password-reset` 도 인증번호를 나른다(`:236-238`). 픽스처가 이들에 **재시도 3회**를 준 이유를 store 주석이 밝힌다 — **'인증번호가 안 가면 로그인을 못 한다'**(`store.ts:270`). **같은 문장이 `remove` 의 409 참조 무결성 가드의 근거로도 쓰인다**(`store.ts:86-90` — §7.5).

**그런데 이 규칙들은 토글 한 번으로 꺼진다**(FS-064-EL-009.8). 확인 다이얼로그도, 권한 게이팅도, 경고 문구도 없다. `enabled: false` 인 채로 이벤트가 나면 **알림이 나가지 않는다**(`notification.ts:395`).

**그것이 무엇을 뜻하는가**:
- `security.login-new-device` 를 끄면 → **계정 탈취의 유일한 사용자 통지 경로가 사라진다.** 공격자가 로그인해도 주인이 모른다.
- `security.verification-code` 를 끄면 → **본인확인이 멎는다.** 사용자는 로그인·비밀번호 재설정을 할 수 없다(가용성 사고).

즉 **이 토글은 보안 통제의 on/off 스위치**다. 권한을 탈취한 내부자가 가장 먼저 누를 버튼이기도 하다.

**판정 — 세 가지를 계약으로 요구한다**:
1. **감사 로그(필수)**: EP-04 는 `enabled` 가 바뀔 때마다 **누가·언제·어느 규칙을·어느 값으로** 바꿨는지 감사 로그에 남긴다. EP-05(삭제)도 같다 — 삭제는 영구히 끄는 것과 같다. 이 로그는 규칙 레코드가 아니라 **별도 감사 스토어**에 있어야 한다(규칙을 지워도 남아야 한다). 앱에 이미 관리자 로그 화면이 있다(`/logs/admin` — `nav-config.ts:208`) — 그 계약의 소비자로 붙인다.
2. **쓰기 권한을 `admin` 으로 좁힌다**(§7.7).
3. **보안 트리거는 서버가 추가로 방어한다**: `category === 'security'` 규칙의 `enabled: false` 는 **거절하거나(422) 별도 권한을 요구한다.** 판정은 아키텍처 이관 — **화면에 그 구분이 없으므로**(토글이 전부 같다) 지금 계약으로 못박으면 화면이 표현할 수 없는 실패가 생긴다. 그러나 **감사 로그(1)와 권한(2)는 화면 변경 없이 지금 넣을 수 있다.**

**⚠ 지어내지 않기 위한 경계**: 코드에 '보안 알림을 끄면 안 된다'는 규칙은 **없다**. 픽스처는 오히려 `account.dormant` 를 꺼둔 표본으로 두었다(`store.ts:271` 주석 · `ntf-rule-9` 의 `enabled: false` `:351` — '운영자가 휴면 예고를 잠시 중단한 상태'). 즉 **끄는 것 자체는 정상 운영 행위**다. 위 판정은 '끌 수 없게 하라'가 아니라 **'끈 사실이 기록돼야 한다'** 이며, (3)은 미정으로 남긴다.

### 7.7 쓰기는 `admin` 전용 — BE-026 과 반대 판정

BE-026(1:1 문의)은 `operator` 에게 쓰기를 열었다 — '1:1 문의 응대는 운영자의 본업'이기 때문이다. **이 도메인은 반대다.**

**근거**:
1. **발송 규칙은 응대가 아니라 설정이다.** 규칙 하나를 바꾸면 그 이벤트의 **모든 미래 발송**이 바뀐다 — 한 고객에게 보내는 답변과 영향 범위가 다르다.
2. **보안 통제를 포함한다**(§7.6). 인증번호 발송을 끌 수 있는 권한은 상담 권한과 같은 등급일 수 없다.
3. **발송 비용을 직접 만든다.** 규칙을 켜면 그 이벤트마다 SMS 가 나간다 — SMS 는 건당 과금이고 LMS 는 단가가 오른다(`SmsTemplateListPage.tsx:57` · `SmsTemplateFormPage.tsx:181`).
4. **규칙이 20건뿐이고 자주 바뀌지 않는다** — `operator` 를 막아도 운영이 멎지 않는다. 문의 응대와 달리 일상 업무가 아니다.

**결론**: EP-01·EP-02 는 `admin` + `operator`(운영자가 '지금 어떤 알림이 나가는가'를 확인할 이유는 충분하다). **EP-03·EP-04·EP-05 는 `admin` 전용.**

**⚠ 프론트가 이 판정을 표현하지 못한다**: 쓰기 게이팅이 배선돼 있지 않아(`useRouteWritePermissions` 소비처 8곳에 알림 관리 없음 — FS-064 §7 #10) `operator` 도 등록·수정·삭제·토글 컨트롤을 **전부 본다**. 서버 403 이 배너/토스트의 일반 실패 문구로 뭉개진다. **이 판정은 서버 계약으로서 유효하나, 화면이 그것을 미리 말해 주지 못한다** — UI 기획 이관(§7.14 #4).

### 7.8 403 으로 거절한다 — 404 은닉을 하지 않는다 【보안 판정】

BE-003 §3.2 의 원칙을 이 도메인에 적용하면 **BE-026 과 반대 결론**이 나온다.

1. **컬렉션의 존재는 비밀이 아니다** → `GET /api/notifications/rules` 권한 부족 시 **403 `FORBIDDEN`**.
2. **개별 규칙의 존재도 비밀이 아니다** → `GET /api/notifications/rules/:id` 권한 부족 시에도 **403**.

**근거**: BE-026 이 문의를 404 로 은닉한 이유는 `Ticket` 이 `customerName`·`contact`·`body`(고객 원문)를 담는 **개인정보 레코드**이기 때문이었다 — 'id 열거로 이 고객이 무엇을 문의했는가를 알아내는 경로를 닫는다'. **`NotificationRule` 에는 개인정보가 한 글자도 없다**: `id`·`trigger`·`channel`·`templateId`·`enabled`·`retryPolicy`·`updatedAt`(§3) — 전부 **설정 값**이다. 규칙 id 를 열거해 알아낼 수 있는 것은 '이 쇼핑몰이 주문 접수 메일을 보내는가' 정도이며, 그것은 **고객이 주문 한 번으로 알 수 있는 사실**이다. 은닉의 이득이 없다.

**반대로 403 이 옳은 적극적 이유**: 은닉은 **정당한 운영자를 혼란시킨다.** `operator` 가 규칙을 수정하려다 404 를 받으면 '누가 지웠나?' 를 의심하고 찾아 헤맨다 — 실제로는 권한이 없을 뿐이다. 그리고 **화면이 404 와 403 을 다르게 그린다**(`FormPageShell.tsx:116-144`): 404 는 '이미 삭제되었을 수 있습니다' + '다시 시도' 없음. 권한 부족에 그 문구를 주면 **화면이 거짓말을 한다.**

### 7.9 페이징·bulk·Summary 분리를 넣지 않는다 — 20건 상한이 그것을 불필요하게 한다

BE-026 §7.9 는 문의 목록에 페이징과 `TicketSummary` 분리를 요구했다 — 문의는 **상한 없이 매일 쌓이는 컬렉션**이기 때문이다. **이 도메인은 다르다.**

| 축 | 판정 | 근거 |
|---|---|---|
| 페이징 | **넣지 않는다** | `(trigger, channel)` 유일성이 **최대 20건**을 강제한다(§3). 10 트리거 × 2 채널이 이론적 상한이고, 트리거는 코드 상수다. 전량 반환이 옳다 |
| Summary 분리 | **넣지 않는다** | `NotificationRule` 은 필드 7개이고 **개인정보·대용량 텍스트가 없다**(§7.8). 목록이 상세 전문을 담아도 비용이 없다 — BE-026 이 `body`·`timeline` 을 뺀 이유가 여기엔 없다 |
| bulk 삭제 계약 | **넣지 않는다** | 프론트가 id 마다 EP-05 를 병렬 호출한다(`useCrudList.tsx:133-134`). 최대 20요청이며 그 정도는 서버가 감당한다. **단 레이트리밋(분당 60)이 20건 일괄 삭제를 두 번 반복하면 걸린다** — 그때 부분 실패로 떨어진다(§7.14 #7) |
| 서버측 필터·검색 | **넣지 않는다** | 20건을 클라이언트가 거르는 것이 옳다. **단 IA-13 이 요구하는 URL 상태는 이미 충족돼 있다**(`useListState`) — 서버 쿼리 없이도 링크 공유가 된다 |

**이 판정을 재검토할 조건**: 트리거 카탈로그가 **서버 소유가 되어 수십 개로 늘면**(§7.11) 상한 20이 무너진다. 그때 페이징을 다시 논한다.

### 7.10 멱등키 — 폼 저장에만 있고 토글·삭제에는 없다 【정합 판정】

**코드로 확인한 사실**:

| 경로 | 동기 락 | 멱등키 | 근거 |
|---|---|---|---|
| 폼 저장(EP-03·EP-04) | ✔ `submitLockRef` | ✔ `idempotencyKeyRef` | `useCrudForm.ts:103,118-123,228,235` |
| 목록 토글(EP-04) | ✘ | ✘ | `useCrudRowUpdate.ts:44-45` — `{ id, input, signal }` 뿐 |
| 삭제(EP-05) | ✘ | ✘ | `useCrudList.tsx:101-103` — `{ id, signal }` 뿐 |

**프레임워크가 생략을 의도로 선언한다**: `crud.ts:36-39` — '생략 가능하다 — **목록의 단건 삭제처럼 사용자가 매번 확인 다이얼로그를 거치는 조작은 키 없이 온다.** 그때 어댑터는 평소대로 처리한다.'

**판정**:
- **삭제(EP-05)는 그 예외가 성립한다.** 확인 다이얼로그가 매 시도를 게이트하고 `busy` 가 확인 버튼을 잠근다(`ConfirmDialog.tsx:151`). DELETE 는 자연 멱등이라 두 번 도달해도 결과가 같다(두 번째는 404/409). **계약을 바꾸지 않는다.**
- **토글(EP-04)은 그 예외가 성립하지 않는다** — **확인 다이얼로그가 없다.** `busy` 는 상태이므로 리렌더 후에야 잠기고, 그 틈의 두 번째 클릭은 두 번째 요청을 만든다. 완화 요인은 **새 토글이 이전 요청을 abort 한다**는 것뿐인데(`useCrudRowUpdate.ts:39`) **abort 는 클라이언트가 결과를 버리는 것이지 서버 도달을 막지 못한다.** 결과: 서버에 `enabled: false` 와 `enabled: true` 가 **순서 보장 없이** 도착할 수 있다. 마지막에 도착한 것이 이기므로 **스위치가 화면과 반대로 남을 수 있다.**

**결론**: **EP-04 는 `Idempotency-Key` 를 받되 없어도 처리한다**(폼 저장은 싣고 토글은 안 싣는 현실을 계약이 수용한다). 토글의 순서 문제는 **멱등키로 풀리지 않는다** — 멱등키는 '같은 요청 두 번'을 막지 '다른 요청 두 개의 순서'를 정하지 않는다. **UI 기획 이관**: `useCrudRowUpdate` 가 마지막 요청만 반영되도록 보장하려면 abort 만으로 부족하고, 응답 도착 시 자기가 마지막 요청인지 확인해야 한다(§7.14 #3).

### 7.11 트리거 카탈로그의 소유자가 미정이다 【미정 판정】

`TRIGGER_ID_VALUES`(`notification.ts:77-88`)는 **프론트 코드 상수 10개**다. zod `z.enum` 이 그것을 그대로 쓰고(`validation.ts:26`), 분류·설명·표시 순서·변수 목록이 모두 그 상수에 매달려 있다(`:100-161,207-251`).

**서버가 정본이 되어야 하는가?** 코드에 답이 없다:
- **심 없음** — `data-source.ts` 에 트리거 조회 TODO 가 없다.
- 프론트는 **모르는 트리거를 방어한다** — 배지 대신 id 를 그리고(`triggerColumn.tsx:22-25`), `triggerLabel` 이 id 로 폴백하며(`notification.ts:168-170`), 정렬 rank 가 `?? 0` 이다(`:499`). **즉 서버가 새 트리거를 줄 수 있다고 가정한 방어 코드가 이미 있다.**
- 그러나 `z.enum(TRIGGER_ID_VALUES)` 는 **모르는 트리거를 저장 시 거절한다** — 조회는 관대하고 쓰기는 엄격하다. 이 비대칭이 의도인지 코드가 말하지 않는다.

**판정: 미정.** 이 문서는 트리거 카탈로그 엔드포인트를 **만들지 않는다** — 심이 없고, 만들면 발명이 된다. 다만 **결정이 필요하다**: 트리거는 서버 이벤트 버스가 실제로 발생시키는 것이므로(`data-source.ts:16`) **정본이 서버에 있는 것이 자연스럽다.** 서버 소유로 정하면 ① `GET /api/notifications/triggers` 가 생기고 ② `z.enum` 이 동적 검증으로 바뀌며 ③ §7.9 의 20건 상한이 무너지고 ④ 변수 카탈로그(`NOTIFICATION_VARIABLES` — `notification.ts:207-251`)도 함께 서버로 가야 한다(트리거가 주는 변수는 트리거의 속성이다). **아키텍처 이관 — 도메인 경계 결정.**

### 7.12 status 별 surface 가 갈리지 않는다 — 400 은 배너로, 토글 실패는 토스트로 뭉개진다 【정합 판정】

`HttpError` 가 status 를 갖고(`shared/errors/http-error.ts`) `?status=<op>:<code>` 로 9종을 재현할 수 있는데도(`dev.ts:26-37`), 화면이 분기하는 것은 **404 하나뿐**이다(`FormPageShell.tsx:121`).

| status | 프론트 처리 | 판정 |
|---|---|---|
| 404 | ✔ 전용 문구 + '다시 시도' 제거 | 계약대로 |
| 409/412 | ✔ 충돌 다이얼로그(입력 보존) — **폼에서만** | 계약대로. **토글은 토스트로 뭉갠다** |
| 422 | ✔ 필드 인라인 + 포커스 — **`violations` 가 있을 때만**(`useCrudForm.ts:182`) | 계약대로. **서버가 `error.fields` 를 채워야 발현된다** |
| **400** | ✘ **일반 배너로 뭉개진다** — `useCrudForm` 이 400 을 분기하지 않는다(`:194`) | **서버는 필드 거절을 422 로 내라**(§7.14 #5) |
| **403** | ✘ 일반 배너/토스트 | gap |
| **429** | ✘ 일반 배너/토스트 — `Retry-After` 를 쓰지 않는다 | gap |
| 500/504 | ✔ 배너 + **참조 코드**(`useCrudForm.ts:195`) | 계약대로 |

**서버가 할 수 있는 것**: 필드 단위 거절을 **400 이 아니라 422 `error.fields` 로** 내면 프론트가 **이미 옳게 그린다.** BE-003 §2 의 `VALIDATION_FAILED`(400)를 이 도메인에서는 **열거 타입 위반 등 구조적 오류에만** 쓰고, 값 조합 거절(중복 규칙 §7.2 · 템플릿 부적격 §7.4)은 전부 422 로 낸다. **이것이 프론트 코드 0줄로 얻을 수 있는 최대치다.**

### 7.13 삭제의 404 와 409 는 사용자에게 같은 뜻이다

어댑터는 대상 부재 삭제를 **409 '이미 삭제된 항목입니다.'** 로 낸다(`crud.ts:232-234`) — 404 가 아니다. 서버 계약(EP-05)은 404 를 정의한다. **둘이 어긋나는가?**

**판정: 어긋나지 않는다 — 사용자 관점에서 같은 사실이다.** '지우려는 것이 이미 없다.' 화면은 둘 다 삭제 다이얼로그의 error 배너로 받는다(`useCrudList.tsx:110-113`) — **분기가 없으므로 어느 쪽이 와도 같게 보인다.** 서버는 표준을 따라 **404** 를 내고, 어댑터의 409 는 픽스처 구현 상세로 둔다. **단 문구가 '삭제하지 못했습니다. 잠시 후 다시 시도해 주세요.' 고정이라 '다시 시도하면 될 것'처럼 말하는 것은 결함이다**(§7.5 와 같은 뿌리) — UI 기획 이관.
