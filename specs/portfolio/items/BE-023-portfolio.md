# BE-023. 포트폴리오 백엔드 기능 명세

## 1. 개요

| 항목 | 내용 |
|---|---|
| 대상 화면 | FS-023 포트폴리오 (`/portfolio/items` · `/new` · `/:id/edit`) |
| 범위 | 포트폴리오 목록/상세/등록/수정/삭제(단건·일괄), 노출 여부 토글(수정 재사용), 분류 선택지 조회 |
| 범위 밖 | 카테고리 CRUD(BE-024 소유 — 이 문서는 선택지 **읽기**만 다룬다) · 고객 노출 렌더(§7.1 XSS 판정만) · 일괄 전용 엔드포인트(§7.4) · **이미지 파일 업로드(심 없음 — §4 EP-07 · §7.2)** |
| 프론트 어댑터 | `apps/admin/src/pages/portfolio/items/data-source.ts` (`createStoreAdapter` 로 `_shared/store.ts` 에 배선) |
| 도메인 타입 | `apps/admin/src/pages/portfolio/_shared/store.ts` (`PortfolioItem` · `PortfolioItemInput` · `PortfolioCategory`) |
| 검증 정본 | `apps/admin/src/pages/portfolio/items/validation.ts` (`portfolioSchema`) |
| 판정 기준일 | **2026-07-17 · HEAD = `a5c2639`** (PR #22·#24·#26·#28·#30·#32·#34 머지 후). 직전 판정은 `4b805ad` 기준이었고 그때 §7.3 · §7.8 · §6 이 갱신됐다. **이번 기준 갱신으로 이 문서의 판정이 뒤집힌 것은 없다** — PR #26·#30·#32 는 DS·토큰 층 변경이라 이 문서가 다루는 엔드포인트·보안 계약에 닿지 않는다. 관련 프론트 변화는 NFR-023 §2 에 기록했다(A11Y-11 · MOTION-03 이 pass/종속 으로 뒤집혔다) |

> **에러 봉투·권한 모델 상속**: BE-003 §2·§3 을 그대로 상속한다(BE-010 §2 와 동일 선언). 아래는 포트폴리오 고유 차이만 기술한다.

## 2. 공통 (상속)

- **에러 봉투**: BE-003 §2. 공통 에러코드(`VALIDATION_FAILED`·`UNAUTHENTICATED`·`FORBIDDEN`·`CSRF_TOKEN_INVALID`·`NOT_FOUND`·`CONFLICT`·`UNPROCESSABLE`·`RATE_LIMITED`·`INTERNAL_ERROR`·`REQUEST_TIMEOUT`) 동일. `error.traceId` 는 프론트의 **오류 참조 코드**(FS-023-EL-016)와 대응한다 — §7.6.
- **권한**: `admin` = 전체. `operator` = 조회 계열만(목록·상세·분류 선택지), 쓰기(등록·수정·삭제·노출 토글) → 403. 포트폴리오 읽기 권한 없는 관리자 → 컬렉션 403 / 개별 404 은닉(BE-003 §3.2, §7.5).
- **CSRF**: 쓰기(POST·PUT·DELETE)에 `X-CSRF-Token`.
- **타임아웃**: 조회·쓰기 5초 → 504. **프론트에는 타임아웃 상한이 없다**(FS-023 §7 #22) — 서버 상한이 유일한 천장이다.
- **프론트 역할 분기 없음**(FS-023 §4.1) — 이 화면의 쓰기 컨트롤이 권한과 무관하게 렌더되므로(FS-023 §7 #2 — `useRouteWritePermissions` 는 앱에 존재하고 7곳이 쓰지만 `pages/portfolio/**` 는 그 밖이다) **권한 강제는 전적으로 서버 책임**이다. 프론트는 403 을 다른 실패와 같은 문구로 표시한다. **프론트가 게이팅을 붙여도 이 책임은 줄지 않는다** — 클라이언트 게이팅은 UX 이지 보안 경계가 아니다.

## 3. 데이터 계약 (store.ts 대조)

| 타입 | 필드 | 비고 |
|---|---|---|
| `PortfolioItem`(목록 행 = 상세) | `id` · `title` · `categoryId` · `categoryLabel`(**서버 조인 비정규화**) · `client` · `summary` · `coverImageUrl` · `imageUrls`(`readonly string[]`) · `published`(bool) · `date`(`'YYYY-MM-DD'`) | **목록과 상세가 같은 타입이다** — 목록 응답이 이미 전문을 담는다. 목록 표는 그중 분류·제목·고객사·일자·노출만 그린다(이미지 열 없음) |
| `PortfolioItemInput`(쓰기 바디) | `title` · `categoryId` · `client` · `summary` · `coverImageUrl` · `imageUrls` · `published` · `date` | `id`·`categoryLabel` 제외. `categoryLabel` 은 **서버가 `categoryId` 로 채운다**(클라이언트가 보내지 않는다) |
| `PortfolioCategory` | `id` · `label` | 필터·폼 선택지 전용. 정본은 BE-024 |

**제약(`portfolioSchema` 대조)** — `title` 1–120자(trim) · `categoryId` 비어 있지 않음 · `client` 1–60자 · `summary` 1–500자 · `date` `^\d{4}-\d{2}-\d{2}$` · `coverImageUrl` 비어 있지 않음(**스킴 미검사** — §7.2) · `imageUrls` 최대 10장 · `published` boolean.

> **`updatedAt`/`version` 이 없다.** `PortfolioItem` 에 낙관적 동시성 토큰으로 쓸 필드가 존재하지 않는다 — §7.3 이 이를 판정한다.

## 4. 엔드포인트 명세

### BE-023-EP-01 · 포트폴리오 목록 조회
| 항목 | 내용 |
|---|---|
| 근거 (FS) | FS-023-EL-003, EL-004, EL-006, EL-009 |
| 근거 (심) | `// TODO(backend): GET/POST /api/portfolio/items` |
| 메서드·경로 | `GET /api/portfolio/items` |
| 권한 | `admin`, `operator` |
| 멱등성 | 멱등(GET) |
| 페이징 | **없음 — 전량 반환**. 프론트에 페이지네이션이 없다(FS-023 §7 #3) |
| 레이트리밋 | 분당 120회 |

**쿼리**: **없다.** 프론트 어댑터 시그니처는 `fetchAll(signal)` 로 파라미터를 받지 않으며, 분류 필터는 클라이언트가 받은 배열을 거른다(`filterPortfolioItems`). 검색·정렬 파라미터도 없다.

**응답 200** — `readonly PortfolioItem[]`. **정렬은 서버가 확정한다**: `date` 내림차순, 같은 날짜는 `id` 내림차순(안정) — 프론트도 같은 규칙을 `sortPortfolioItems` 로 적용하므로 서버 순서와 일치해야 화면이 흔들리지 않는다. `categoryLabel` 은 조회 시점의 카테고리 표시명을 조인해 채운다.

**에러**: 401 · 403 · 429 · 500 · 504.

> **후속 계약 변경 예고(§7.7)**: 항목 수가 늘면 `page`·`size`·`categoryId` 쿼리와 `{ items, total }` 응답으로 승격해야 한다. 지금 전량 반환으로 확정하는 것은 **프론트에 페이지네이션 표면이 없기 때문**이며, 스키마를 미리 만들지 않는다.

### BE-023-EP-04 · 포트폴리오 수정 (폼 저장 · 노출 토글 공용)
| 항목 | 내용 |
|---|---|
| 근거 (FS) | FS-023-EL-027(수정), **FS-023-EL-006.8(노출 토글)**, FS-023-EL-029(충돌) |
| 근거 (심) | `// TODO(backend): … GET/PUT/DELETE /api/portfolio/items/:id` |
| 메서드·경로 | `PUT /api/portfolio/items/:id` |

| 멱등성 | 멱등(PUT 전체 치환) |

**에러**: 400 · 401 · 403 · 403 CSRF · **404 `PORTFOLIO_NOT_FOUND`**(존재한 적 없는 id) · **409 `CONFLICT`**(§7.3) · 422 `CATEGORY_NOT_FOUND` · 429 · 500 · 504.

### BE-023-EP-07 · 이미지 파일 업로드 — **심 없음 (미정)**

| 항목 | 내용 |
|---|---|
| 근거 (FS) | FS-023-EL-023(대표 이미지), FS-023-EL-024(본문 이미지) |
| 근거 (심) | **없음.** `data-source.ts` 에 업로드 관련 `// TODO(backend)` 주석이 없다 |
| 상태 | **미정 — 계약을 정의하지 않는다** |

`ImageUploadField`/`ImageGalleryField` 는 파일을 서버로 보내지 않는다. `URL.createObjectURL(file)` 로 **`blob:` object URL** 을 만들어 그 문자열을 그대로 `coverImageUrl`·`imageUrls` 에 넣고, EP-03/EP-04 가 그것을 저장한다. 그 URL 은 **생성한 문서(탭)의 수명에만 유효**하므로 저장 후 새로고침하면 이미지가 깨지고, 다른 관리자·고객 화면에서는 처음부터 로드되지 않는다.

**엔드포인트를 지어내지 않는다.** 업로드 심이 없다는 것은 '경로가 무엇인지 아직 정해지지 않았다'는 뜻이며, 그 결정은 §7.2 가 백엔드 명세·UI 기획 로 이관한다. **다만 그때까지도 서버는 방어해야 한다** — §7.2 가 `blob:`/`data:` 거절을 EP-03/EP-04 의 400 조건으로 못 박는다.

## 7. 핵심 판정

### 7.1 본문·제목 XSS — 서버가 저장 시 정제한다 【보안 판정】

포트폴리오의 `title`·`summary`·`client` 는 관리자 입력이며 **고객 포트폴리오 화면에 노출**된다(그것이 이 도메인의 존재 이유다). 관리자 화면은 평문 셀/`TextareaField` 로만 렌더하지만(FS-023-EL-006.5·EL-022), **고객 렌더가 HTML 로 해석하면 저장형 XSS 가 된다** — 관리자 계정 하나가 뚫리거나 내부자가 악의를 가지면 전체 방문자에게 스크립트가 배포된다.

BE-010 §7.2 와 동일 판정: **서버가 저장 시(EP-03·EP-04) 정제**한다 — 허용 태그 화이트리스트 밖의 마크업·`javascript:`/`vbscript:` 스킴·이벤트 핸들러 속성을 제거한다. 계약은 '저장된 `title`·`summary`·`client` 에 실행 가능한 스크립트가 없다'는 **관측 동작만** 정하며 정제 라이브러리를 지정하지 않는다.

**렌더 시점 이스케이프에 의존하지 않는 이유**: 고객 화면은 이 리포 밖이고 렌더러가 여럿일 수 있다(웹·앱 웹뷰·메타 태그·검색 인덱스). 저장 시 정제는 소비자 수와 무관하게 한 번 성립한다.

### 7.2 이미지 URL 스킴 화이트리스트 — 서버가 `blob:`/`data:` 를 거절한다 【보안 판정】

**전제(관측 사실)**: 업로드 심이 없다(EP-07). `ImageUploadField`/`ImageGalleryField` 가 만든 **`blob:` URL 이 그대로 저장 바디에 실린다**. 프론트 검증(`requiredImage`)은 "문자열이 비지 않음"만 보고 스킴을 검사하지 않는다 — 같은 파일의 `optionalHttpUrl` 은 `^https?://` 를 강제하는데 이미지 필드는 그것을 쓰지 않는다.

**판정 두 가지:**

1. **저장 계약 (400)** — EP-03·EP-04 는 `coverImageUrl`·`imageUrls[]` 에 대해 **스킴 화이트리스트**를 강제한다: 자사 업로드 오리진의 `https:` URL(또는 EP-07 이 정해질 때의 업로드 응답 URL)만 허용하고, **`blob:` · `data:` · `javascript:` · `file:` · 그 밖의 스킴은 400 `VALIDATION_FAILED`(`error.fields[].name = coverImageUrl | imageUrls`)로 거절**한다. 이유 세 가지:
   - `blob:`/`data:` 는 **다른 사용자에게 반드시 깨진다** — 저장을 허용하면 DB 에 영구히 죽은 값이 쌓인다.
   - `data:` 는 크기 상한을 우회해 본문에 임의 바이트를 싣는 경로다(저장소 팽창·MIME 스니핑).
   - `javascript:` 는 고객 렌더가 `<a href>`·`<img src>` 로 쓰는 순간 XSS 다(§7.1 과 같은 뿌리).
2. **거절이 곧 신호다** — 400 이 뜨면 프론트는 그 필드에 인라인 오류를 꽂는다(422 라면 `useCrudForm` 이 `violations` 로 자동 매핑한다). 즉 **서버 거절이 업로드 심 부재를 화면에 드러내는 유일한 장치**이므로, 이 검사는 EP-07 이 정해지기 전에 **먼저** 들어가야 한다. 지금 넣지 않으면 백엔드 연결 첫날 `blob:` 문자열이 조용히 저장된다.

**이관**: 업로드 엔드포인트 계약(경로·멀티파트 규약·용량·MIME·바이러스 검사·응답 URL 수명) 자체는 **미정**이며 §7.10 으로 넘긴다.

### 7.3 낙관적 동시성 — 토큰이 없다. 서버가 존재를 정본으로 판정한다 【정합 판정】

**관측 사실 세 가지 (⚠ 2번은 F3b 에서 뒤집혔다 — 정정한다):**

1. `PortfolioItem` 에 `updatedAt`·`version` 이 **없다**(`_shared/store.ts:23-40`) → **If-Match/ETag 로 쓸 값이 없다**. *(변함없음)*
2. ~~`createStoreAdapter` 가 대상 존재를 검사하지 않는다~~ → **정정: 이제 검사한다.** F3b 가 `shared/crud/crud.ts:171` 에 `exists()` 를 넣고 `update` 없는 id → **409 `'다른 사용자가 먼저 삭제한 항목입니다.'`**(`:219-221`), `remove` 없는 id → **409 `'이미 삭제된 항목입니다.'`**(`:232-234`), `fetchOne` 없는 id → **404**(`:192-194`) 를 던지게 했다. **유령 저장·유령 삭제·404 미도달은 전부 해소됐다** — 아래 판정의 '존재 재확인 409' 는 이제 픽스처에서도 관측된다.
3. 화면은 **이미 409/412 를 처리할 준비가 돼 있다** — `useCrudForm.ts:166-179` 가 `isConflict` 로 분기해 입력을 보존한 채 충돌 다이얼로그를 띄운다(FS-023-EL-029). **F3b 이후로는 생산자도 있다** — 소비자·생산자가 모두 성립하므로 이 계약은 프론트에서 검증 가능하다.

**남은 결손은 하나다: 낙관적 동시성 토큰.** 409 는 **'대상이 아직 존재하는가'** 만 판정하므로 **둘 다 존재하는 동시 편집은 last-write-wins** 다. 이 구분을 흐리지 말 것 — 해소된 것은 '유령 저장'이고, 미해소인 것은 '토큰'이다.

**판정:**

- **EP-04·EP-05 는 트랜잭션 안에서 대상 존재를 재확인하고, 없으면 409 `CONFLICT`(`message`: '다른 사용자가 먼저 삭제한 항목입니다.')를 반환한다.** 404 가 아니라 409 인 이유: 사용자는 **존재했던 것을 편집하던 중**이며 할 일은 '없는 페이지'가 아니라 '내 입력을 어떻게 할 것인가'다 — 그 UI 가 이미 충돌 다이얼로그다. (존재한 적 없는 id 로 직접 호출한 경우는 404 다.)
- **EP-04 의 동시 수정은 마지막 쓰기 승리로 확정한다**(BE-010 §7.4 와 같은 결). 토큰이 없고, 프론트에 diverge 필드를 보여 줄 UI 도 없다.
- **그러나 이 화면에서 last-write-wins 의 비용은 다른 화면보다 크다** — 노출 토글(FS-023-EL-006.8)이 **전용 PATCH 가 아니라 PUT 전체 치환**이기 때문이다(§7.4). A 가 목록을 연 뒤 B 가 제목·소개·이미지를 고쳤고, A 가 그 상태에서 노출 스위치만 눌렀다면, A 의 브라우저에 있던 **낡은 전체 항목이 B 의 수정을 통째로 되돌린다**. '노출만 바꿨는데 남의 수정이 사라진다'는 것은 운영자가 예측할 수 없는 손실이다.
- **완화(계약 안에서 가능한 것)**: EP-04 가 `updatedAt` 을 응답에 싣고 요청에 `If-Unmodified-Since`/`If-Match` 를 받도록 승격하면 위 시나리오가 412 → 충돌 다이얼로그로 흡수된다. **필드가 없으므로 이는 계약 변경이며 §7.10 으로 이관한다.** 이 문서는 '토큰 없음 + 존재 재확인 409' 를 현재 계약으로 확정한다.

### 7.4 일괄 삭제·노출 토글 — 전용 엔드포인트를 만들지 않는다

- **일괄 삭제**: 프론트가 `settleAll` 로 단건 DELETE(EP-05)를 병렬 호출하고 부분 실패를 건수로 집계한다(FS-023-EL-011). 단건이 멱등이라 부분 실패 후 재시도가 안전하다. BE-010 §7.1 과 같은 근거로 벌크 계약을 두지 않는다.
- **노출 토글**: 심에 `PATCH /:id/visibility` 가 **없다**. 프론트도 `useCrudRowUpdate` → `adapter.update` 로 **PUT 을 재사용**한다. 심에 없는 엔드포인트를 지어내지 않는다 — BE-010 §EP-08 이 PATCH 를 둔 것은 FAQ 어댑터에 `setFaqVisibility` 라는 **전용 함수가 실재**했기 때문이며, 여기에는 없다.
- 이 선택의 대가(전체 치환 토글이 남의 수정을 되돌린다)는 §7.3 이 판정했다. 전용 PATCH 승격은 §7.10.

### 7.5 403 vs 404 은닉 · 카테고리 참조 무결성 【보안·정합 판정】

**은닉(BE-003 §3.2 상속)**: 컬렉션(EP-01·EP-03·EP-06)은 권한 부족 시 **403**. 개별 리소스(EP-02·EP-04·EP-05)는 **포트폴리오 읽기 권한이 없는 주체에게 404 은닉**, 읽기는 되고 쓰기만 없는 주체(`operator`)에게는 **403** — 이미 존재를 아는 주체에게 존재를 숨기는 것은 무의미하고 정당한 운영자의 권한 문제 인지를 막는다.

**카테고리 참조 무결성 (TOCTOU)**: EP-03·EP-04 는 `categoryId` 가 실재하는 카테고리를 가리키는지 **쓰기 트랜잭션 안에서 확인**하고, 아니면 **422 `CATEGORY_NOT_FOUND`** 로 거절한다(BE-010 §EP-05 선례). 프론트는 폼 진입 시 받은 선택지 목록으로만 고르지만(FS-023-EL-018), **그 목록을 받은 시점과 저장 시점 사이에 다른 관리자가 그 카테고리를 지울 수 있다**. 반대 방향의 무결성(사용 중 카테고리 삭제 차단)은 BE-024 가 소유한다 — 두 문서가 같은 불변식('항목은 실재하는 카테고리만 참조한다')의 양쪽 끝을 막는다.

**400 이 아니라 422 인 이유**: `categoryId` 는 **형식이 틀린 것이 아니라**(문자열이고 비어 있지 않다) 가리키는 대상이 없는 것이다 — 상태 위반이다.

### 7.6 분류 선택지 조회 실패에 표면이 없다 — 서버가 할 수 있는 일 【판정】

`PortfolioListPage.tsx:62`·`PortfolioFormPage.tsx:97` 이 `categoriesQuery.data ?? []` 로만 읽어 EP-06 의 401/403/500/504 가 **화면에 아무 흔적도 남기지 않는다**. 목록에서는 필터가 '전체 분류'만 남고 끝이지만, **폼에서는 필수 필드인 분류를 채울 수 없어 저장이 영구히 불가능해진다** — 사용자는 이유를 알 수 없다.

**서버 측 판정**: EP-06 의 실패는 EP-01 과 **독립적으로** 관측 가능해야 한다(같은 응답에 접어 넣지 않는다) — 프론트가 배너를 붙일 자리를 갖게 하기 위해서다. 다만 **프론트 표면 부재는 서버가 고칠 수 없다** → UI 기획 이관(§7.10). 서버는 `error.traceId` 를 실어 최소한 로그 상관이 가능하게 한다.

### 7.8 멱등키 — 경로가 생겼다. 계약을 확정한다 (F3b — 이전 판정 정정)

~~키가 생성되지만 전달 경로가 없다~~ → **정정**: F3b 가 `CrudAdapter.create/update` 시그니처에 `WriteContext`(`crud.ts:30-42`)를 넣어 **키가 앉을 자리를 만들었다**. 이제 `useCrudForm.ts:118-123` 의 `idempotencyKeyRef` 가 잡은 **제출 시도 단위** 키가 `:211` → `:228`·`:235`(mutation variables) → `crud.ts:288`·`:310`(`adapter.create/update(…, { signal, idempotencyKey })`) 로 **어댑터까지 도달한다**. 키를 mutationFn **밖**(variables)에 두는 이유가 `crud.ts:273-278` 에 적혀 있다 — 안에서 만들면 react-query 재시도마다 새 키가 생겨 서버가 두 요청을 별개 거래로 본다. `useCrudForm.ts:220` 은 **성공했을 때만** 키를 버린다.

픽스처는 그 키로 서버를 흉내 낸다 — `createIdempotencyLedger()`(`crud.ts:62-72`)가 `isReplay`/`record` 를 제공하고 `createStoreAdapter` 가 `create`(`:201-203`)·`update`(`:208`)에서 그것을 통과시킨다. **기록은 적용에 성공한 뒤에만** 한다(`:55-60`: 미리 기록하면 실패한 첫 시도가 키를 태워 재시도가 영원한 no-op 이 되고, 사용자는 '저장했습니다'를 보지만 아무것도 저장되지 않는다).

**계약 판정**: EP-03(등록)은 선택적 **`Idempotency-Key: <uuid>` 요청 헤더**를 받는다 — 심은 `crud.ts:39`(`TODO(backend): 이 값은 Idempotency-Key: <key> 요청 헤더로 나간다 (BE-004-EP-03)`). 같은 키의 재요청은 **재처리 없이 최초 응답을 재생**하고, 키 보존 창은 **24시간**(`pages/members/data-source.ts:243-253` 의 선례 심과 같은 값). 키가 없는 요청(노출 토글 — `useCrudRowUpdate.ts:45` 는 키를 싣지 않는다)은 평소대로 처리한다. 프론트의 동기 제출 락(`submitLockRef`)은 **한 탭 안**의 중복만 막으므로 — 네트워크 재전송·새로고침 후 재제출은 못 막는다 — 서버 측 키 처리가 그 구멍을 닫는다.

### 7.9 제목 유니크 제약을 두지 않는다

BE-010 §7.5 는 **카테고리명**에 유니크를 걸었다 — 필터·select 가 표시명으로 카테고리를 구분하기 때문이다. 포트폴리오 **항목**은 그렇지 않다: 화면이 항목을 `id` 로 구분하고 분류 배지·고객사·일자로 함께 식별한다. 같은 제목의 프로젝트가 실제로 둘일 수 있다(연차별 리뉴얼 등). 제약을 걸면 정당한 등록이 막힌다 → **EP-03 에 409 없음**.
