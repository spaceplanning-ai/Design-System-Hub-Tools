# FS-023. 포트폴리오 (목록·등록/수정)

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 목적 | 포트폴리오(프로젝트 실적)를 분류로 좁혀 보고, 목록에서 바로 노출을 켜고 끄며, 개별 항목을 등록·수정·삭제(단건·일괄)한다 |
| 역할(주 사용자) | 관리자 (구현에 역할 분기 없음 — §4.1) |
| 진입 경로 | 좌측 GNB > 포트폴리오 관리 > 포트폴리오 (`/portfolio/items`) |
| 포함 화면 | 목록 `/portfolio/items` · 등록 `/portfolio/items/new` · 수정 `/portfolio/items/:id/edit`. **상세(`/:id`) 라우트가 없다** — 행 클릭·행 액션의 수정 버튼이 곧바로 수정 폼으로 간다 |
| 구현 경로 | `apps/admin/src/pages/portfolio/items/**` · 섹션 공용 `apps/admin/src/pages/portfolio/_shared/**` |
| 대응 SCR | SCR-023 (미작성 — §7) |
| 공통 컴포넌트 | `shared/crud/{CrudListShell,CrudTable,useCrudList,useCrudRowUpdate,useCrudForm,FormPageShell,FormFeedback,createStoreAdapter,validation.requiredText·requiredImage,dev}` · `shared/ui/{SelectField,StatusBadge,Button,SelectionBar,RowSelectCell,SelectAllHeaderCell,SeqCell,RowActions,ConfirmDialog,ToggleSwitch,FormField,TextareaField,ImageUploadField,ImageGalleryField,Empty,Alert,Card,useToast,useUnsavedChangesDialog,useRowSelection,errorIdOf,controlStyle,pageTitleStyle}` · `shared/useRowNavigation` · `shared/bulk.settleAll` · `shared/async.isAbort` · `shared/errors/http-error` |

> **섹션 공용 모듈**: 포트폴리오(FS-023)·성공 사례(FS-025)가 폼 하단부(`_shared/PortfolioMediaFields.tsx` — 대표 이미지·본문 이미지·노출 토글)와 목록의 노출 토글 열(`_shared/publishColumn.tsx`)을 공유한다. 포트폴리오 항목과 카테고리(FS-024)는 같은 픽스처 저장소(`_shared/store.ts`)를 공유한다 — 항목이 카테고리를 참조하고 카테고리 삭제가 사용 중이면 막히기 때문이다.

> **선언적 CRUD 프레임워크형**: 포트폴리오(FS-023)·성공 사례(FS-025)가 `useCrudList` + `CrudListShell` + `useCrudForm` + `FormPageShell` 이라는 같은 골격을 공유한다. 이 문서가 그 골격의 정본 기술이며, FS-025 는 필드·필터·어댑터 차이만 기술하고 골격 동작은 이 문서를 참조한다.

### 4.1 공통 예외 규칙

| 규칙 | 내용 |
|---|---|
| 네트워크 단절 | 목록 조회 실패는 표를 대체하는 인라인 배너(FS-023-EL-009). 상세 로드 실패는 폼을 대체하는 배너(FS-023-EL-015). 저장 실패는 카드 안 배너(FS-023-EL-016). 노출 토글 실패는 토스트, 삭제 실패는 다이얼로그 안 배너. **오프라인 감지·전용 안내 없음** — §7 #22 |
| 세션 만료 | 앱 전역 401 인터셉터가 쿼리/뮤테이션 캐시에 한 곳(`queryClient.ts`) 있다 — 401 이면 `notifySessionExpired()` 가 재인증 경로로 보낸다. 이 화면의 모든 조회·쓰기가 그 계층을 통과하므로 별도 배선이 없다. **단, 이 화면의 어댑터가 401 을 던지는 실사용 경로는 없다**(백엔드 없음) — `?status=<op>:401` 로만 재현된다 |
| 요청 타임아웃 | **프론트 타임아웃 상한 없음**(`AbortSignal.timeout` 0건). abort 는 화면 이탈 · 다이얼로그 닫기 · 다른 행 토글 시에만 발생 — §7 #22 |
| 중복 제출 | 폼 제출은 **① `disabled={saving \|\| loadingDetail}` + ② 동기 제출 락(`submitLockRef`)** 이중으로 막는다 — RHF 비동기 검증과 disabled 렌더 사이의 틈에서 Enter 를 연타해도 요청은 1건이다. 삭제 확인·일괄 삭제는 요청 중 확인 버튼 비활성. 노출 토글은 그 행의 스위치를 `busy` 로 잠근다(다른 행은 막지 않는다) |
| 실패 통지의 자리 | ① 목록·상세 조회 실패 = 인라인 배너(대체 렌더) ② generic 저장 실패 = 폼 카드 배너(+오류 참조 코드) ③ 필드 단위 거절(422) = **그 입력의 인라인 오류 + 포커스 이동**(배너 아님) ④ 저장 충돌(409/412) = 충돌 다이얼로그(입력 보존) ⑤ 삭제 실패 = 다이얼로그 안 배너 ⑥ 노출 토글 실패 = 토스트(이미 목록을 보고 있으므로) ⑦ 삭제·등록·저장 **성공** = 토스트 ⑧ 다이얼로그 **취소** = '작업이 취소되었습니다' 토스트(discard·'이어서 편집'은 예외 — `suppressCancelToast`) |
| 낙관적 업데이트 | **없다.** 노출 토글(FS-023-EL-006.8)은 `useCrudRowUpdate` 의 비관적 배선이다 — 요청 중 스위치를 잠그고 응답이 와야 값이 바뀐다. 스냅샷·롤백 개념이 없다 — §7 #6 |
| 취소는 실패가 아니다 | abort 는 공유 판정(`isAbort`)으로 걸러진다: 실패 토스트·배너 없음, `mutation.reset()` 으로 버튼 상태 복원, 일괄 삭제의 실패 건수에서도 제외된다 |
| 동시 조회 | 목록 조회는 react-query 가 키 `['portfolio','list']` 로 1건만 유지하고 `placeholderData: (previous) => previous` 로 재조회 중 이전 데이터를 유지한다. 카테고리 선택지는 `['portfolio','category-options']`(placeholderData 없음), 상세는 `['portfolio','detail',id]`. `staleTime` 30초 · `retry: false` · `refetchOnWindowFocus: false` |
| 로딩과 재조회의 구분 | 스켈레톤의 유일한 조건은 **최초 로드**(`isFetching && data === undefined`)다. 데이터가 있는 재조회는 표를 비우지 않고 요약 텍스트에 '· 새로고침 중…' 만 붙는다(`aria-busy`) |
| 권한 없음 | **라우트 read 권한**은 AppShell 이 `RequirePermission` 으로 게이팅한다 — 권한이 없으면 이 화면 대신 403 화면이 뜬다(사이드바·헤더는 남는다). 그러나 **쓰기 컨트롤(등록·수정·삭제·노출 토글)은 권한과 무관하게 항상 렌더된다** — §7 #2. 서버 권한 응답은 조회=인라인 배너, 쓰기=각 요소 실패로 떨어진다. 은닉 정책은 BE-023 이 확정 |
| 행 선택의 수명 | 선택은 화면 로컬 상태(`useRowSelection`). **분류 필터가 바뀌면 전부 해제**된다(`useEffect(() => clear(), [filter, clear])`). 일괄 삭제 전원 성공 시에도 해제된다. 그러나 단건 삭제로 사라진 행의 id 는 해제되지 않는다 — §7 #19 |
| 폼 기준선 | 수정 진입 시 상세가 도착하면 `reset(toValues(item))` 으로 폼을 채우고 그 값이 dirty 기준선이 된다. 저장 성공 시 `reset` 없이 곧바로 목록으로 `replace` 이동한다 |
| 편집 중 원격 변경 | `reset` 은 `detailQuery.data` 참조가 바뀔 때마다 실행된다. 충돌 다이얼로그의 '최신 내용 불러오기'가 상세를 재조회해 이 경로로 폼을 덮는다(의도된 동작 — 내 입력 폐기) |
