# BE-045. 쿠폰 백엔드 기능 명세

## 1. 개요

| 항목 | 내용 |
|---|---|
| 대상 화면 | FS-045 쿠폰 (`/products/coupons` · `/products/coupons/new` · `/products/coupons/:id/edit`) |
| 범위 | 쿠폰 목록 조회, 쿠폰 상세 조회, 쿠폰 등록, 쿠폰 수정(발급 상태 인라인 토글 포함), 쿠폰 삭제(단건·일괄) |
| **범위 밖** | **발급 실행 · 고객 쿠폰함 · 사용(할인 적용)** — 이 계약은 쿠폰의 **정의**만 소유한다. `issuedCount` 를 늘리는 것은 발급 도메인이고, 쿠폰을 쓰는 것은 주문 도메인이다(§7.5). **발급 대상 엔티티 조회**(회원등급·카테고리·상품 목록) — 화면에 그 선택 입력이 아예 없어 **계약이 존재하지 않는다**(§7.1). **쿠폰 상태(예정/진행중/만료/중지)** — 저장 필드가 아니라 `enabled` + 기간에서 파생되는 표시값이다(§7.8) |
| 전제 | BE-003 §2·§3 을 상속한다. 인증은 세션 쿠키 기반. 모든 경로는 `/api` 프리픽스. 응답 본문은 `application/json; charset=utf-8`. 날짜는 `YYYY-MM-DD` |
| 프론트 어댑터 | `apps/admin/src/pages/products/coupons/data-source.ts` (`couponAdapter` = 공용 `createCrudAdapter`) |
| 도메인 타입 | `apps/admin/src/pages/products/coupons/types.ts` |
| 검증 정본 | `apps/admin/src/pages/products/coupons/validation.ts` — `couponSchema`(zod). `validation.ts:1-2` 가 '검증의 정본은 이 zod 스키마다'를 못박는다. **서버는 이 규칙을 재판정한다** — 클라이언트 검증은 UX 다 |

> **에러 봉투·권한 모델 상속**: BE-003 §2·§3 을 그대로 상속한다. 아래는 쿠폰 고유 차이만 기술한다.

## 2. 공통 (상속)

- **에러 봉투**: BE-003 §2. 공통 에러코드 동일(`VALIDATION_FAILED` · `UNAUTHENTICATED` · `FORBIDDEN` · `CSRF_TOKEN_INVALID` · `NOT_FOUND` · `CONFLICT` · `UNPROCESSABLE` · `RATE_LIMITED` · `INTERNAL_ERROR` · `REQUEST_TIMEOUT`).
- **권한**: `admin` = 전체. **`operator` = 조회 전용** — 모든 쓰기(등록·수정·토글·삭제)는 403. **BE-026(1:1 문의)·BE-044(교환/반품)와 반대이고 BE-010(FAQ)과 같다**(§7.7). 상품 도메인 읽기 권한 없는 관리자 → 컬렉션 403 / **개별 쿠폰 403**(§7.4 — 회원·문의와 달리 404 은닉을 **하지 않는다**).
- **CSRF**: 쓰기(POST · PUT · DELETE)에 `X-CSRF-Token`.
- **타임아웃**: 조회 5초 → 504. 쓰기 5초 → 504.
- **프론트 역할 분기 없음** — 이 화면은 `useRouteWritePermissions` 를 소비하지 않는다(FS-045 §7 #4). 권한 강제는 전적으로 서버 책임이며, 그 사실이 이 원칙을 바꾸지 않는다. (형제 화면 `products/items`·`products/returns` 는 배선했다 — 그것도 UX 일 뿐이다.)

## 3. 데이터 계약 (`types.ts` · `validation.ts` 대조)

| 타입 | 필드 | 비고 |
|---|---|---|
| `Coupon` | `id` · `name` · `code` · `issueType` · `discountValue` · `maxDiscount` · `minOrderAmount` · `target` · `totalQuantity` · `issuedCount` · `startAt`(`YYYY-MM-DD`) · `endAt` · `enabled` | 목록·상세가 **같은 타입**이다 — 이 화면에서는 문제가 아니다(모든 필드를 목록이 쓰거나 파생에 필요하다). **상태(status) 필드가 없다** — 파생이다(§7.8) |
| `CouponInput` | `Coupon` 에서 `id` 만 뺀 것(`types.ts:38-51`) | 저장 입력. **`issuedCount` 를 포함한다** — 서버 소유 필드가 요청 바디에 있다(§7.2) |
| `CouponIssueType` | `amount` \| `percent` \| `free_shipping` | `discountValue` 의 단위를 정한다 |
| `CouponTarget` | `all` \| `member_grade` \| `category` \| `product` | **대상 엔티티를 지목할 필드가 없다**(§7.1) |
| 상수 | `COUPON_NAME_MAX = 60` · `COUPON_CODE_MAX = 30` | |

**검증 규칙 (`couponSchema` — `validation.ts:29-97`. 서버가 재판정한다)**

| 필드 | 규칙 | 근거 |
|---|---|---|
| `name` | 필수 · ≤60자 | `requiredText('쿠폰명', COUPON_NAME_MAX)` (`:31`) |
| `code` | 필수 · ≤30자 · **`/^[A-Za-z0-9-]+$/`**(영문·숫자·하이픈) | `:32-40`. **유일성 검사 없음**(§7.3) |
| `issueType` | enum 3종 | `:41` |
| `discountValue` | 무료배송이면 무시 · 그 밖에는 **정수 필수** · **정률이면 1–100** | `:53-75`(교차 검증) |
| `maxDiscount` | **검증 없음**(`z.string()`) — 어떤 문자열이든 통과 | `:43`. §7.9 #6 |
| `minOrderAmount` | 필수 · **정수**. 상한 없음 | `intString('최소 주문 금액')` (`:44`) |
| `target` | enum 4종 | `:45` |
| `totalQuantity` | 필수 · **정수**. 상한 없음. **`issuedCount` 와의 관계 검사 없음** | `intString('발급 수량')` (`:47`) · §7.2 |
| `startAt`·`endAt` | 실재 `YYYY-MM-DD` · **종료 ≥ 시작** | `:76-97`(교차 검증) |
| `enabled` | boolean | `:49` |
| `issuedCount` | `z.number()` — **어떤 수든 통과** | `:50-51`. §7.2 |

**쿠폰 상태 파생 (`couponStatus(coupon, today)` — `types.ts:98-106`. 서버 필드가 아니다)**

| 조건 | 상태 | 톤 |
|---|---|---|
| `!enabled` | `disabled`(중지) | neutral |
| `today < startAt` | `scheduled`(예정) | info |
| `today > endAt` | `expired`(만료) | neutral |
| 그 밖 | `active`(진행중) | success |

> **판정 순서가 계약이다**: `enabled` 가 기간을 이긴다 — 기간 안이라도 꺼져 있으면 '중지'다. 서버가 이 파생을 내려주지 않으므로(§7.8) **프론트의 `today` 는 브라우저 시각이다**(`CouponListPage.tsx:95` `formatDate(new Date())`). 관리자의 브라우저 타임존이 다르면 자정 부근에서 상태가 갈릴 수 있다.

**소진율 파생 (`usageRate` — `types.ts:90-93`)**: `totalQuantity <= 0` 이면 0(무제한), 그 밖에는 `Math.min(100, Math.round((issuedCount / totalQuantity) * 100))`. **`Math.min(100, …)` 이 초과 발급을 숨긴다**(§7.2).

### BE-045-EP-03 · 쿠폰 등록
| 항목 | 내용 |
|---|---|
| 근거 (FS) | FS-045-EL-003, EL-021~EL-030, EL-033, EL-036, EL-037, EL-038, EL-039 |
| 심 | `data-source.ts:74` `POST /api/coupons` |
| 메서드·경로 | `POST /api/coupons` |
| 권한 | **`admin` 만**(§7.7) |
| 멱등성 | **`Idempotency-Key` 헤더로 강제한다.** 프론트가 이미 제출 시도 단위 키를 만들어 전달한다(`useCrudForm.ts:118-123,211,235` → `crud.ts:288`) — 어댑터 본문이 헤더로 옮기기만 하면 된다 |
| 레이트리밋 | 분당 60회 |

**바디**(`CouponInput`): `name` · `code` · `issueType` · `discountValue` · `maxDiscount` · `minOrderAmount` · `target` · `totalQuantity` · **`issuedCount`** · `startAt` · `endAt` · `enabled`.

**서버 검증 (요청을 그대로 믿지 않는다)**
1. **`issuedCount` 무시** — 등록 시 항상 **0** 이다. 요청 값을 받지 않는다(§7.2).
2. §3 의 `couponSchema` 규칙 전부를 **재판정**한다. 위반 시 400 `VALIDATION_FAILED` + `error.fields`.
3. **`code` 유일성** — 대소문자 무시 유일. 프론트가 `toUpperCase()` 로 정규화하지만(`CouponFormPage.tsx:117`) **서버가 정규화의 정본**이다. 중복이면 **409 `COUPON_CODE_TAKEN`**(§7.3).
4. `maxDiscount` — 클라이언트에 검증이 없으므로(§3) **서버가 유일한 방어선**이다: 정수 · 0 이상 · `issueType === 'percent'` 가 아니면 무시.
5. `totalQuantity` · `minOrderAmount` — 상한을 서버가 정한다(클라이언트 상한 없음).

**응답 201/204**. 프론트 `create(input, context?): Promise<void>` — 응답 본문을 읽지 않고 목록을 무효화한 뒤(`crud.ts:290-292`) 목록으로 이동한다.

**에러**: 400 `VALIDATION_FAILED`(`error.fields`: `name`·`code`·`discountValue`·`minOrderAmount`·`totalQuantity`·`startAt`·`endAt`) · 401 · 403 `FORBIDDEN` · 403 `CSRF_TOKEN_INVALID` · **409 `COUPON_CODE_TAKEN`** · 429 · 500 · 504.

### BE-045-EP-04 · 쿠폰 수정 (폼 저장 · 발급 상태 토글 공용)
| 항목 | 내용 |
|---|---|
| 근거 (FS) | FS-045-EL-009.10(토글) · EL-021~EL-030, EL-033, EL-034, EL-038, EL-039(폼) |
| 심 | `data-source.ts:74` `PUT /api/coupons/:id` |
| 메서드·경로 | `PUT /api/coupons/:id` |
| 권한 | **`admin` 만**(§7.7) |
| 멱등성 | **`Idempotency-Key` 헤더.** **폼 저장에는 키가 도달하나 토글에는 도달하지 않는다**(§7.9 #4) |
| 레이트리밋 | 분당 60회 |

**바디**(`CouponInput` — 전체 치환): EP-03 과 같다. **토글도 같은 바디를 보낸다** — `{ ...toCouponInput(item), enabled: next }`(`CouponListPage.tsx:155`)라 조회 시점 스냅샷 전체가 실린다.

**서버 검증**
1. **`issuedCount` 무시** — 저장된 값을 정본으로 유지한다(§7.2 【보안 판정】).
2. §3 규칙 재판정. 위반 시 400.
3. **`code` 유일성** — 자기 자신을 제외하고 검사. 중복이면 409 `COUPON_CODE_TAKEN`.
4. **대상이 없으면 404** — 어댑터가 이미 409 를 던지지만(`crud.ts:126-128`) **서버 계약에서는 404 가 옳다**: '없는 것'과 '충돌한 것'은 다르다. 어댑터의 409 는 픽스처가 유령 저장을 막으려는 근사치다(§7.9 #5).
5. **낙관적 동시성** — `If-Match`/`version` 불일치면 409 `CONFLICT`(§7.2).

**응답 200/204**. 프론트 `update(id, input, context?): Promise<void>` — 목록·상세를 무효화한다(`crud.ts:312-315`).

**에러**: 400 `VALIDATION_FAILED` · 401 · 403 `FORBIDDEN` · 403 `CSRF_TOKEN_INVALID` · **404 `COUPON_NOT_FOUND`** · **409 `COUPON_CODE_TAKEN`** · **409 `CONFLICT`** · 429 · 500 · 504.

### BE-045-EP-05 · 쿠폰 삭제
| 항목 | 내용 |
|---|---|
| 근거 (FS) | FS-045-EL-008, EL-009.11, EL-013, EL-014 |
| 심 | `data-source.ts:74` `DELETE /api/coupons/:id` |

| 레이트리밋 | 분당 60회 |

**일괄 삭제는 별도 엔드포인트가 아니다**: `settleAll(ids, (id) => adapter.remove(id, { signal }))`(`crud.ts:349-350`)가 **id 마다 이 엔드포인트를 병렬 호출**한다. `Promise.allSettled` semantics 라 부분 실패가 가능하고, 프론트는 **실패 건수만** 받는다(FS-045-EL-014).

**서버 검증**
- **발급된 쿠폰의 삭제** — `issuedCount > 0` 이면 어떻게 할 것인가? **미정**(§7.5). 지금은 아무 제약이 없다.

**응답 204**.

**에러**: 400(id 형식) · 401 · 403 `FORBIDDEN` · 403 `CSRF_TOKEN_INVALID` · **404 `COUPON_NOT_FOUND`** · **409 `COUPON_IN_USE`**(§7.5 채택 시) · 429 · 500 · 504.

| 요구 | 내용 |
|---|---|
| CSRF | 쓰기(POST · PUT · DELETE)에 `X-CSRF-Token` 헤더를 싣는다. 시그니처는 바뀌지 않는다(BE-003 §3.3). |

## 7. 핵심 판정

### 7.1 발급 대상이 유형만 있고 대상이 없다 — 계약을 만들 수 없다 【범위 판정 · 미정】

**코드로 확인한 사실**: `Coupon.target` 은 `'all' | 'member_grade' | 'category' | 'product'` 유니온 하나뿐이고(`types.ts:12,26`), **어느 등급·어느 카테고리·어느 상품인지 지목할 필드가 타입에 없다**(`types.ts:14-36` — `targetIds`/`gradeIds`/`categoryIds` 류 grep 0건). 폼도 select 하나만 그린다(`CouponFormPage.tsx:286-294`).

**이것이 뜻하는 것**: 지금 '특정 상품' 쿠폰과 '전체 회원' 쿠폰은 **저장 결과가 라벨 하나만 다르고 동작이 같다.** 미리보기가 조건 줄에 '특정 상품'이라고 쓰지만(`CouponCardPreview.tsx:122`) 그 상품이 무엇인지 아무도 모른다.

**판정**: **이 계약은 발급 대상 엔티티를 정의하지 않는다.** 다음 셋 중 무엇도 코드에 근거가 없으므로 **'미정'으로 남긴다**:
1. `target` 이 애초에 표시용 라벨인가(= 실제 제한이 없는가)?
2. 대상 지정 UI 가 **미구현**인가(= 필드가 곧 붙는가)?
3. 대상 제한을 **주문 도메인**이 다른 경로로 아는가?

**A01 이 확정해야 한다.** 확정 전에 서버가 `targetIds` 를 발명하면 프론트에 그것을 채울 입력이 없어 **항상 빈 배열**이 오고, '특정 카테고리' 쿠폰이 아무 카테고리에도 걸리지 않는다 — 지금과 같은 상태를 더 복잡한 형태로 재현할 뿐이다.

**BE-044 §7.4 와 다르다**: 거기서는 상품 참조가 **실재**하고(`ReturnRequest.productId`) 경계만 다른 문서였다. 여기는 **참조 자체가 없다.**

### 7.2 `issuedCount` 를 요청에서 받지 않는다 — 서버가 소유한다 【보안 판정】

**현재 클라이언트가 `issuedCount` 를 바디에 실어 보낸다**: 스키마가 그것을 폼 값으로 인정하고(`validation.ts:50-51` `issuedCount: z.number()`), `toValues` 가 항목에서 복사하고(`CouponFormPage.tsx:144`), `toInput` 이 그대로 되돌려 보내며(`:124`), **토글도 `toCouponInput(item)` 으로 전체를 실어 보낸다**(`CouponListPage.tsx:155` → `types.ts:136`). 등록 시 초기값은 0(`:110`).

**이것이 만드는 사고**:
1. **소진율 조작.** `issuedCount` 는 목록의 소진율 표시(`usageRate` — `types.ts:90-93`)와 **발급 가능 여부 판정의 근거**다(`totalQuantity` 대비). 조작된 클라이언트가 `issuedCount: 0` 을 보내면 **소진된 쿠폰이 다시 무제한이 된다** — 1,000장짜리 쿠폰을 무한히 발급할 수 있다.
2. **lost update — 조작 없이도 일어난다.** 관리자가 수정 폼을 열고(`issuedCount: 640` 스냅샷) 이름을 고치는 동안 고객 60명이 쿠폰을 받는다(`issuedCount: 700`). 저장하면 **640 이 700 을 덮는다** — 60장의 발급 기록이 사라지고 그만큼 더 발급된다. **발급 상태 토글은 더 잦다** — 목록에서 클릭 한 번이 전체 치환 PUT 이다(`CouponListPage.tsx:155`).
3. **`totalQuantity < issuedCount` 를 막는 것이 없다.** 발급 수량 검증은 `intString` 뿐이고(`validation.ts:47`) 발급된 수와 비교하지 않는다. 그러면 소진율이 100% 를 넘어야 하는데 `usageRate` 가 `Math.min(100, …)` 로 깎아(`types.ts:92`) **초과 발급이 화면에서 100% 로 보인다** — 아무도 모른다.

**판정**: `issuedCount` 는 **서버가 소유한다.** 요청 바디의 이 값은 **무시**한다 — 파싱조차 하지 않는다. 등록 시 항상 0 이고, 이후에는 **발급 도메인만** 증가시킨다(§7.5). 이것은 UX 개선이 아니라 **금액 무결성 요구다**: 쿠폰은 할인이고 할인은 금액이다.

**계약을 둘 중 하나로 바꾼다.**

| 안 | 형태 | 평가 |
|---|---|---|
| **A (권장)** | `CouponInput` 에서 `issuedCount` 를 **제거**한다. 스키마·`toValues`·`toInput`·`toCouponInput` 에서 함께 빠진다 | 위조가 구조적으로 불가능하다. **어댑터 시그니처는 그대로다**(`Input` 타입만 좁아진다). 화면 변경은 4곳(`validation.ts:50-51` · `CouponFormPage.tsx:110,124,144` · `types.ts:136`) |
| **B (차선)** | 필드를 유지하되 서버가 **무시** | 어댑터를 덜 흔든다. 그러나 '무시한다'는 계약은 구현이 한 줄 빠지면 조용히 깨진다 — 안 A 는 타입이 강제한다 |

**추가 판정 — `totalQuantity` 검증**: 서버는 `totalQuantity > 0 && totalQuantity < issuedCount(저장된 값)` 이면 **422 `QUANTITY_BELOW_ISSUED`** 로 거절한다. 발급 수량을 이미 나간 수보다 작게 만드는 것은 데이터를 모순 상태로 만든다.

**낙관적 동시성**: `Coupon` 에 `version`/`updatedAt` 필드가 **없다**(`types.ts:14-36`). 어댑터의 409 는 **'존재 여부' 기반**이다(`crud.ts:126-128`). 즉 **동시 편집(둘 다 존재)은 last-write-wins** 다. 안 A 를 채택하면 다투는 필드가 전부 '관리자가 의도적으로 정하는 값'이 되어 last-write-wins 가 **용인 가능**해진다(나중 사람의 의도가 이긴다). 안 B 를 유지하면 `If-Match`(ETag) 또는 `Coupon.version` 이 **필수**다 — `issuedCount` 를 조용히 되돌리는 경로를 닫아야 한다.

### 7.3 쿠폰 코드 유일성 — 서버만 알 수 있고, 그 실패의 자리가 틀렸다 【보안·정합 판정】

**코드로 확인한 사실**: `couponSchema.code` 는 **형식만** 본다 — 비어 있지 않음 · 30자 이하 · `/^[A-Za-z0-9-]+$/`(`validation.ts:32-40`). **유일성 검사가 클라이언트에도 어댑터에도 없다**(grep 0건). 픽스처 4건은 서로 다른 코드를 갖지만 그것을 강제하는 것이 없다.

**왜 유일성이 필수인가**: `code` 는 **고객이 입력하는 값**이다(`types.ts:17`). 두 쿠폰이 `WELCOME15` 를 공유하면 고객이 그것을 입력했을 때 **어느 쿠폰이 적용되는지 정의되지 않는다** — 할인 금액이 다르면 그것이 곧 금액 사고다. 프론트가 `toUpperCase()` 로 정규화하므로(`CouponFormPage.tsx:117`) `welcome15` 와 `WELCOME15` 는 같은 코드다.

**판정**: **코드는 대소문자 무시 유일**이며 **서버가 정본**이다. 정규화(대문자화·공백 제거)도 서버가 한다 — 프론트의 `toUpperCase()` 는 편의이며 계약이 아니다. 중복이면 거절한다.

**그런데 그 실패가 지금 잘못된 자리로 간다 — 이것이 계약 판정이다**:
- 중복을 **409** 로 내면 `useCrudForm.handleWriteError` 의 `isConflict` 분기가 먼저 잡아(`useCrudForm.ts:166-179`) **충돌 다이얼로그**를 연다. 문구는 '**다른 사용자가 먼저 변경했습니다**' + '최신 내용 불러오기 / 이어서 편집'(`FormFeedback.tsx:58-74`)이다. **틀린 진단이다** — 아무도 이 쿠폰을 변경하지 않았고, '최신 내용 불러오기'는 아무것도 고치지 않는다. 사용자는 코드를 바꿔야 하는데 그것을 알 수 없다.
- **판정: 코드 중복은 `422 COUPON_CODE_TAKEN` + `error.fields: [{ name: 'code', … }]` 로 내려보낸다.** 그러면 `isUnprocessable` 분기가(`useCrudForm.ts:182-192`) `code` 필드에 인라인 오류를 꽂고 **그 필드로 포커스를 옮긴다** — 사용자가 고칠 곳에 정확히 도달한다. **화면 코드 0줄로 성립한다.**
- 이 판정은 §5 매트릭스의 EP-03/EP-04 '409 충돌' 칸을 **바꾼다** — 409 는 **낙관적 동시성 전용**으로 남긴다. HTTP 의미상 중복 생성은 409 가 자연스럽지만, **이 앱에서 409 는 이미 '동시 편집 충돌'이라는 UX 를 뜻한다**(`isConflict` 가 412 와 함께 묶는다 — `http-error.ts:101-107`). 계약이 코드의 의미를 존중해야 한다.

**열거 위험(기록)**: 코드 중복 응답은 **'이 코드가 이미 존재한다'를 알려준다.** 쿠폰 코드는 고객이 입력하는 값이므로 추측당하면 부정 사용이 된다 — 그러나 이 엔드포인트는 **인증된 관리자 전용**(`admin`, §7.7)이라 열거 주체가 이미 전체 목록을 볼 수 있다. **은닉이 무의미하므로 중복을 그대로 알린다.** (고객 대면 '쿠폰 코드 입력' 엔드포인트는 이 계약이 아니며, 거기서는 유효/무효를 구분해 알리면 안 된다 — 그것은 주문 도메인의 판정이다.)

### 7.4 쿠폰은 403 이다 — 404 로 은닉하지 않는다 【보안 판정】

BE-003 §3.2 의 원칙 두 줄을 이 도메인에 적용하면 **회원·문의·교환/반품과 답이 갈린다.**

1. **컬렉션의 존재는 비밀이 아니다** → `GET /api/coupons` 권한 부족 시 **403 `FORBIDDEN`**.
2. **개별 쿠폰 리소스의 존재는 개인정보가 아니다** → **403 `FORBIDDEN`**. **404 은닉을 하지 않는다.**

**근거 — 왜 회원(BE-003 §3.2)·문의(BE-026 §7.6)·교환/반품(BE-044 §7.5)과 다른가**: 그 셋은 **한 사람에 관한 기록**이다 — id 열거가 곧 '누가 무엇을 했는가'의 열거다. **쿠폰은 사람에 관한 기록이 아니다.** `Coupon` 의 12개 필드 중 개인을 지목하는 것이 **하나도 없다**(`types.ts:14-36`) — 이름·코드·할인·기간·수량뿐이다. 쿠폰 id 를 열거해 얻는 것은 **마케팅 계획**이지 개인정보가 아니다.

**그러면 마케팅 계획은 숨길 필요가 없는가?** 경쟁사에 흘리고 싶지 않은 정보이긴 하다. 그러나 **404 은닉은 그것을 지키지 못한다** — 컬렉션이 403 이면 id 를 얻을 경로가 이미 없고, id 를 안다는 것은 다른 경로로 이미 샜다는 뜻이다. 대신 **정당한 운영자가 권한 문제를 인지하지 못하게 만드는 비용**만 확실히 치른다. **은닉의 비용이 이익을 넘는 경우다.**

**이 판정을 뒤집어야 하는 조건**: 쿠폰에 `target` 의 구체 대상이 붙어(§7.1) `member_grade`/특정 회원을 지목하게 되면 **그 순간 개인정보가 된다** — 그때 이 판정을 재검토한다.

### 7.5 발급 실행은 이 계약이 아니다 — 그러나 경계가 비어 있다 【범위 판정】

**코드로 확인한 사실**: `issuedCount` 를 **늘리는 코드가 앱 어디에도 없다.** 픽스처가 640·500·3120·45 라는 값을 갖고 있을 뿐이고(`data-source.ts:22,37,50,65`), 이 화면은 그것을 읽어 소진율만 그린다. `enabled` 토글은 '발급되지 않는다'는 **정의**를 바꿀 뿐 발급 자체를 하지 않는다.

**판정**: 이 계약은 **쿠폰의 정의만 소유한다.** `POST /api/coupons/:id/issue` 류의 발급 엔드포인트를 **만들지 않는다** — 심이 없고, 화면에 진입점이 없으며, 그것은 고객 채널 또는 마케팅 도메인의 계약이다.

**그러나 세 경계가 비어 있다 — 기록만 하고 정하지 않는다**:
1. **발급 조건의 강제 주체가 미정.** `enabled` · `startAt`/`endAt` · `totalQuantity` 는 '언제 누구에게 몇 장까지 발급하는가'의 규칙인데, **그것을 읽어 발급을 막는 코드가 이 앱에 없다.** 발급 도메인이 이 필드들을 읽는다는 계약이 어디에도 없다.
2. **삭제와 발급된 쿠폰의 관계가 미정.** `issuedCount > 0` 인 쿠폰을 지우면 **이미 고객 쿠폰함에 있는 쿠폰이 어떻게 되는가?** EP-05 에 아무 제약이 없다. 상품 카테고리는 '사용 중이면 삭제하지 않는다(서버는 409 로 막는다)'를 코드로 갖고 있다(`_shared/store.ts:690-696`) — 쿠폰에는 그 선례가 적용되지 않았다.
3. **`enabled` 를 끄면 이미 발급된 쿠폰이 못 쓰이는가?** 필드 주석은 '끄면 기간 안이라도 **발급되지 않는다**'(`types.ts:34`)라고 발급만 말한다 — **사용**은 언급하지 않는다.

**A01 이 세 경계를 확정해야 한다.** 확정 전에 서버가 임의로 정하면(예: 삭제 시 발급분 무효화) 운영자가 예상하지 못한 파괴가 일어난다.

### 7.6 목록이 전량을 내려준다 — 페이징을 도입한다

**현재 계약의 문제**: `fetchAll(signal)` 이 파라미터를 받지 않고 전량을 반환하며(`crud.ts:94-98`), 프론트가 발급유형 필터·검색을 전부 클라이언트에서 한다.

**교환/반품(BE-044 §7.9)·문의(BE-026 §7.9)와 다른 점**: 쿠폰은 **매일 쌓이는 컬렉션이 아니다** — 관리자가 의도적으로 만드는 것이라 증가 속도가 훨씬 느리다. 그러나 **상한도 없다**: 시즌마다 쿠폰을 만들면 수년 뒤 수백 건이 된다.

**판정**: EP-01 에 `issue`·`keyword`·`page`·`size`(기본 20 · 상한 100) 쿼리를 도입한다. **목록 전용 표현(`CouponSummary`)은 분리하지 않는다** — `Coupon` 의 12개 필드를 목록이 **전부 쓰거나 파생에 필요하다**(소진율은 `issuedCount`+`totalQuantity`, 상태는 `enabled`+기간, 할인은 `issueType`+`discountValue`). 뺄 것이 없다. **BE-026 §7.9 · BE-044 §7.9 와 달리 여기서는 분리가 이득이 없다.**

**이관**: `filterCoupons`/검색이 서버로 올라가고 페이지네이션 UI(FS-045 §7 #2)·`SeqCell` 오프셋이 함께 붙어야 한다. **URL list state 는 이미 있다**(`useListState` — FS-045-EL-004) — `page` 파라미터를 쓰기만 하면 된다. quality-bar IA-04 P0 가 이 화면을 gap 으로 잡고 있다. 그전까지 현 계약(전량)을 유지한다 — 픽스처 4건에서는 드러나지 않는다.

### 7.7 `operator` 를 조회 전용으로 둔다 — BE-026·BE-044 와 다른 판정

BE-026(1:1 문의)·BE-044(교환/반품)는 `operator` 에게 쓰기를 연다. **이 도메인은 반대이며 BE-010(FAQ)과 같다.**

**근거**: 쿠폰은 **한 고객의 한 건을 처리하는 응대가 아니라 전체 고객에게 나가는 마케팅 자산**이다. 15% 할인 쿠폰 1,000장은 **곧 금액**이고, 그 정의는 응대가 아니라 **결정**이다. BE-010 이 FAQ 를 '전체 고객에게 공표되는 콘텐츠라 신중해야 한다'로 잠근 것과 정확히 같은 이유이며, 쿠폰은 FAQ 보다 무겁다 — FAQ 는 틀리면 고치면 되지만 **발급된 쿠폰은 회수할 수 없다**(§7.5 #2).

**특히 발급 토글이 위험하다**: 목록에서 **클릭 한 번**이 쿠폰을 발급 상태로 만든다(FS-045-EL-009.10) — 확인 다이얼로그도 없다. 그 클릭이 곧 '지금부터 이 할인을 나눠준다'다.

**결론**: EP-01 · EP-02 는 `admin` + `operator`. **EP-03 · EP-04 · EP-05 는 `admin` 만.** `operator` 의 쓰기 시도는 403.

**프론트 영향**: 이 화면에 쓰기 게이팅이 없으므로(FS-045 §7 #4) **`operator` 가 등록 버튼·토글·삭제를 보고 누르며 전부 403 을 받는다** — 그것도 '변경하지 못했습니다'라는 틀린 문구로. 이 판정은 EXC-03 배선을 **더 급하게** 만든다.

### 7.8 쿠폰 상태는 파생값이다 — 서버 필드로 만들지 않는다

상태(예정/진행중/만료/중지)는 **전부 프론트 순수 함수**다(`couponStatus(coupon, today)` — `types.ts:98-106`). 서버는 `enabled`·`startAt`·`endAt` 만 주고 파생하지 않는다.

**판정: 현 상태를 유지한다.** 상태는 **시각의 함수**라 서버가 응답 시점에 계산해 내려주면 **응답이 도착한 순간 이미 낡는다** — 자정을 넘기면 틀린다. 클라이언트가 렌더 시점에 계산하는 것이 옳다.

**단, 두 가지가 계약에 걸린다**:
1. **`today` 가 브라우저 시각이다**(`CouponListPage.tsx:95` `formatDate(new Date())` → `format.ts:161-169`). 관리자가 다른 타임존에 있으면 자정 부근에서 '예정'과 '진행중'이 갈린다. **판정: `startAt`/`endAt` 은 `Asia/Seoul` 기준 영업일**이며, 프론트가 그 타임존으로 `today` 를 계산해야 한다. `shared/format.ts` 가 이미 KST 고정 헬퍼를 갖고 있으나(`seoulDayOf` — `:142`) 이 화면은 `formatDate(new Date())`(브라우저 로컬)를 쓴다 — **불일치를 기록한다**(§7.9 #9).
2. **상태로 필터·정렬하려면** 서버가 계산해야 한다(클라이언트는 현재 페이지만 안다). §7.6 의 페이징이 도입되는 순간 '만료만 보기'는 서버 기능이 된다 — 그때 이 판정을 재검토한다. **지금 화면에 상태 필터가 없으므로**(발급유형 필터뿐) 걸리지 않는다.
