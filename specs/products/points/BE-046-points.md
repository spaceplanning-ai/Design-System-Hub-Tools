## 2. 공통 (상속)

- **에러 봉투**: BE-003 §2. 공통 에러코드 동일(`VALIDATION_FAILED` · `UNAUTHENTICATED` · `FORBIDDEN` · `CSRF_TOKEN_INVALID` · `NOT_FOUND` · `CONFLICT` · `UNPROCESSABLE` · `RATE_LIMITED` · `INTERNAL_ERROR` · `REQUEST_TIMEOUT`).
- **권한**: `admin` = 조회 + 저장. **`operator` = 조회 전용** — 저장은 403. **BE-045(쿠폰) §7.7 과 같고 BE-026·BE-044 와 반대다**(§7.6).
- **CSRF**: 쓰기(PUT)에 `X-CSRF-Token`.
- **타임아웃**: 조회 5초 → 504. 쓰기 5초 → 504.
- **프론트 역할 분기 없음** — 이 화면은 `useRouteWritePermissions` 를 소비하지 않는다(FS-046 §7 #2). 권한 강제는 전적으로 서버 책임이며, 그 사실이 이 원칙을 바꾸지 않는다.

## 3. 데이터 계약 (`validation.ts` · `types.ts` 대조)

**단일 문서다 — id 도 목록도 없다.** `PointsPolicyValues` 는 zod 스키마에서 추론된 타입이며(`validation.ts:50`), **7개 필드가 전부 문자열 또는 enum** 이다.

| 필드 | 폼 타입 | 의미 | 검증(클라이언트) | 기본값 |
|---|---|---|---|---|
| `earnBaseline` | `'payment' \| 'order'` | 적립 기준 — 실결제금액 / 주문금액(할인 전) | `z.enum` (`:42`) | `'payment'` |
| `earnRate` | string | **새 상품의 기본 적립률(%)** | `percentString` — 0–100 정수 (`:41`) | `'1'` |
| `signupBonus` | string | 회원가입 적립금(원) | `intString` — 0 이상 정수 (`:43`) | `'3000'` |
| `minUseAmount` | string | 최소 사용 포인트(P) | `intString` (`:44`) | `'5000'` |
| `useUnit` | string | 사용 단위(P) | `positiveIntString` — **1 이상** (`:45`) | `'100'` |
| `maxUseRate` | string | 1회 사용 한도(%) — 주문금액 대비 | `percentString` (`:46`) | `'50'` |
| `expireMonths` | string | 유효기간(개월) | `positiveIntString` — **1 이상** (`:47`) | `'12'` |

> **왜 숫자가 문자열인가**: 입력 원값 보존이다(`coupons/validation.ts:3` 이 같은 결정을 명시). `type="number"` 는 브라우저마다 입력 중 값이 사라지는 동작이 다르고, 붙여넣은 값을 조용히 삼킨다. **판정: 서버는 `number` 로 받는다** — 문자열은 폼 표현이지 계약이 아니다(§7.7).

**검증기 3종 (`validation.ts:8-38` — 서버가 재판정한다)**

| 검증기 | 규칙 | 오류 문구 |
|---|---|---|
| `intString(label)` | 비어 있지 않음 + `/^\d+$/` | `{label}{을/를} 입력하세요.` · `{label}{은/는} 0 이상의 정수만 입력할 수 있습니다.` |
| `percentString(label)` | 위 + `Number(v) <= 100` | `{label}{은/는} 0% 이상 100% 이하로 입력하세요.` |
| `positiveIntString(label)` | 위 + `Number(v) >= 1` | `{label}{은/는} 1 이상의 정수로 입력하세요.` |

조사는 `objectParticle`/`topicParticle`(`shared/format.ts:306,311`)이 **받침에서 고른다** — 리터럴 폴백 0건.

**교차 검증: 0건.** `z.object({...})` 뒤에 `.check()` 가 없다(`:40-48`). 필드 간 관계를 보지 않는다(§7.3).

**서버 검증 (요청을 그대로 믿지 않는다)**
1. §3 의 클라이언트 규칙 전부를 **재판정**한다.
2. **상한을 더한다** — 클라이언트에 없는 방어선이다(§7.3).
3. **교차 검증을 더한다** — 클라이언트에 0건이다(§7.3).
4. **`If-Match` 불일치면 412** — 저장하지 않는다(§7.4).

**응답 200/204** + 새 `ETag`. 프론트 `save(input, signal): Promise<void>` — 응답 본문을 읽지 않고 문서 쿼리를 무효화한다(`document.ts:60-62`).

### 6.1 어댑터 본문 요구사항 (시그니처 불변)

| 요구 | 내용 |
|---|---|
| CSRF | `save` 에 `X-CSRF-Token` 헤더를 싣는다. 시그니처는 바뀌지 않는다(BE-003 §3.3). |
| **숫자 ↔ 문자열 변환** | `fetch` 는 응답의 `number` 를 `String()` 으로, `save` 는 폼의 문자열을 `Number()` 로 바꾼다. **이 변환은 어댑터 본문에 산다** — 화면은 계속 문자열을 본다(§7.7). `DocumentStore<T>` 의 `T` 는 폼 타입 그대로다. |
| **`ETag` / `If-Match` — 시그니처가 바뀐다** | `DocumentStore<T>`(`document.ts:14-19`)의 `fetch: (signal) => Promise<T>` 는 **ETag 를 돌려줄 자리가 없고**, `save: (input, signal?) => Promise<void>` 는 **그것을 되보낼 자리가 없다.** 낙관적 동시성을 도입하려면 **이 인터페이스가 바뀌어야 한다** — 어댑터 본문만 고쳐 끝나는 다른 연동과 다르다(§7.4). **단일 문서형 5화면(회사 정보·CEO 인사말·비전/미션·오시는 길·적립금 정책 + 배송 정책)이 전부 영향을 받는다** — 이 화면만의 작업이 아니다. |
| 400/422 변환 | 응답 400/422 의 `error.fields` 를 `HttpError.violations` 로 옮긴다. **그러나 이 화면에 그것을 필드로 꽂을 경로가 없다** — `useCrudForm` 을 쓰지 않아 `setError`/`setFocus` 가 없다(§7.8 #6). 배너로 뭉개진다. |
| 412 변환 | 응답 412 를 `HttpError(412, …)` 로 옮긴다 — `isConflict` 가 이미 412 를 인식한다(`http-error.ts:105-107`). **그러나 이 화면에 충돌 다이얼로그가 없다**(§7.4). |
| 멱등키 | **불필요하다**(§7.4) — 이 저장은 거래가 아니라 설정이다. 두 번 적용돼도 최종 상태가 같고 이중 지급이 없다. `If-Match` 가 그 자리를 대신한다. |

## 7. 핵심 판정

### 7.1 이 정책을 읽는 코드가 없다 — 계약을 쓰기 전에 확인해야 한다 【범위 판정 · 미정】

**코드로 확인한 사실**: 7개 필드 **전부**가 이 화면과 그 정의 파일 밖에서 **소비되지 않는다**(grep 0건). 구체적으로:
- `earnBaseline`(실결제/주문금액) — 적립 계산이 앱에 없다.
- `signupBonus` — 회원가입 처리가 앱에 없다(고객은 회원가입으로 유입된다 — BE-003 §1).
- `minUseAmount` · `useUnit` · `maxUseRate` — 주문·결제 처리가 앱에 없다.
- `expireMonths` — 소멸 배치가 앱에 없다.
- `earnRate` — **상품 폼이 읽어야 하는데 읽지 않는다**(§7.2).

**상품의 적립 계산도 이 정책을 보지 않는다**: `earnedPoints(pricing, points)`(`_shared/store.ts:228-232`)의 두 인자가 **전부 상품 것**이다. 그 함수의 주석(`:223-225`)은 '전역 정책의 적립 기준(실결제금액/주문금액)은 **주문 단위 계산에 쓰이고**'라고 적지만 — **그 주문 단위 계산이 존재하지 않는다.** 주석이 미래를 약속하고 있다.

**판정**: 이 화면은 **아무도 읽지 않는 값을 저장한다.** 그것은 결함이 아니라 **순서**다 — 정책이 먼저 있고 소비자가 나중에 붙는 것이 자연스럽다. 그러나 **계약을 쓰는 이 시점에 두 가지가 미정이다**:
1. **누가 이 문서를 읽는가?** 주문 도메인? 회원 도메인? 배치? 각기 다른 시점에 읽으면 **정책 변경의 소급 적용 범위**가 달라진다(§7.5).
2. **읽는 시점이 언제인가?** 적립 시점? 주문 시점? 결제 시점? `earnBaseline` 이 '실결제금액'이면 **환불 후 적립을 회수하는가**? 이 계약은 답하지 않는다.

**A01 이 확정해야 한다.** 확정 전에 서버가 소비 규칙을 발명하면 **운영자가 정책을 바꿨을 때 무엇이 바뀌는지 아무도 모른다** — 그것은 금액 사고의 전형이다.

### 7.2 '기본 적립률'이 화면의 약속을 지키지 못한다 【정합 판정】

**코드로 확인한 사실**: `earnRate` 필드의 hint 는 **'새 상품의 초기 적립률입니다. 상품별 적립 설정이 이 값을 덮어씁니다.'**(`PointsPolicyPage.tsx:50`)이고, 화면 안내문도 '여기서는 **새 상품의 기본 적립률**과 …'(`:120`)이다. `_shared/store.ts:165-169` 의 주석도 그 관계를 명시한다: '새 상품의 적립 기본값 — **전역 정책의 '기본 적립률'(1%)을 그대로 물려받는다**. DEFAULT_SHIPPING 과 같은 자리다'.

**그런데 그 값은 하드코딩이다**: `export const DEFAULT_POINTS: ProductPoints = { mode: 'rate', rate: 1, amount: 0 };`(`_shared/store.ts:169`). **이 상수는 정책 문서를 읽지 않는다** — `pointsPolicyStore` 를 import 하지 않고, 할 수도 없다(`_shared` 는 잎 모듈이고 `points/data-source` 가 그것을 import 하면 순환이다).

**결과**: **정책에서 적립률을 5% 로 바꿔 저장해도 새 상품은 여전히 1% 로 시작한다.** 화면이 사실이 아닌 것을 말한다 — hint 는 계약처럼 읽히는데 그 계약이 코드에 없다.

**판정**: 두 해석 중 하나여야 하며 **현 코드는 어느 쪽도 아니다**.

| 안 | 형태 | 평가 |
|---|---|---|
| **A** | 상품 폼이 **정책을 조회해** 새 상품의 초기 적립률을 채운다 | hint 가 사실이 된다. 상품 폼에 `useDocumentQuery(pointsPolicyKey, …)` 배선이 필요하고, **로딩 중에는 초기값이 없다**는 새 상태가 생긴다. `DEFAULT_SHIPPING` 도 같은 문제를 갖고 있으므로(`:158-163` — 배송 정책과 하드코딩이 갈린다) **한 배치로 묶어야 한다** |
| **B** | hint 를 고쳐 **하드코딩임을 인정**하고, 정책의 `earnRate` 를 **제거**하거나 '참고용 권장값'으로 격하 | 정직하다. 그러나 그러면 이 필드가 무엇에 쓰이는지 알 수 없다 |

**A01 이 확정해야 한다.** 서버 계약 쪽에서는 어느 안이든 `earnRate` 를 문서에 유지하면 되므로 이 판정이 EP-01/EP-02 를 바꾸지 않는다 — **그러나 그 값이 실제로 아무 효과가 없다는 사실은 기록해야 한다.**

### 7.3 클라이언트에 상한도 교차 검증도 없다 — 서버가 유일한 방어선이다 【정합 판정】

**코드로 확인한 사실**: `pointsPolicySchema` 는 `z.object({...})` 뒤에 **`.check()` 가 없다**(`validation.ts:40-48`) — 교차 검증 0건. 게다가 `intString` 은 **상한이 없다**(`:8-16` — `/^\d+$/` 만 본다).

**이것이 통과시키는 것들 — 전부 실재하는 조합이다**:
1. **`signupBonus: '1000000000000'`** — 회원가입 적립금 1조원. `intString` 을 통과한다. 오타 하나(0 하나 더)가 **가입자마다 30만원을 주는 정책**이 된다. `minUseAmount` 도 같다.
2. **`minUseAmount` 가 `useUnit` 의 배수가 아니다** — 픽스처가 정확히 그 상태다(5000 % 100 === 0 이라 지금은 맞지만 강제하는 것이 없다). 최소 5,001P 부터 100P 단위로 쓴다면 **5,001 을 쓸 수 없다** — 사용 가능한 최소가 5,100 이 된다. 화면이 말한 것과 다르다.
3. **`maxUseRate: '0'`** — `percentString` 이 0 을 통과시킨다(`:24` — `Number(v) <= 100` 만 본다). '주문금액의 0% 까지 쓸 수 있다' = **포인트를 아예 못 쓴다.** 그것이 의도인지, 실수인지 알 방법이 없다.
4. **`signupBonus`(3000) < `minUseAmount`(5000)** — **현 픽스처가 이 상태다**(`types.ts:19-27`). 가입 적립금만 받은 회원은 **영원히 그것을 쓸 수 없다.** 두 필드가 서로를 모른다.

**판정: 서버가 유일한 방어선이며, 상한과 교차 검증을 계약에 넣는다.**

| 규칙 | 값 | 위반 시 |
|---|---|---|
| `signupBonus` 상한 | **1,000,000원** — 회원가입 적립금이 100만원을 넘는 정책은 오타일 확률이 실재할 확률보다 높다 | 400 `VALIDATION_FAILED` (`field: signupBonus`) |
| `minUseAmount` 상한 | **1,000,000P** | 400 (`field: minUseAmount`) |
| `useUnit` 상한 | **10,000P** | 400 (`field: useUnit`) |
| `expireMonths` 상한 | **120개월(10년)** | 400 (`field: expireMonths`) |
| `minUseAmount % useUnit === 0` | 교차 | **422 `POLICY_INCONSISTENT`** (`field: minUseAmount`) |
| `maxUseRate >= 1` | 0% 금지 — '포인트 사용 금지'는 별도 스위치여야지 한도 0 으로 표현할 것이 아니다 | 400 (`field: maxUseRate`) |
| `signupBonus === 0 \|\| signupBonus >= minUseAmount` | 교차 — 주는 즉시 쓸 수 없는 적립금은 정책이 아니라 사고다 | **422 `POLICY_INCONSISTENT`** (`field: signupBonus`) |

**위 상한 값들은 제안이며 A01 확정 대기다** — 코드에 근거가 없다. **그러나 '상한과 교차 검증이 서버에 있어야 한다'는 판정 자체는 코드가 근거다**: 클라이언트에 0건이고, 이 값들은 금액이다.

**프론트 후속**: 같은 규칙을 클라이언트 스키마에도 넣는 것이 옳다 — 서버 왕복 없이 즉시 알리는 것이 UX 다. 그러나 **정본은 서버다**(`validation.ts:1` 의 '검증의 정본은 이 zod 스키마다'는 **폼 검증**의 정본을 말하는 것이지 계약의 정본이 아니다).

### 7.4 낙관적 동시성이 아예 없다 — 이 화면이 담당 4화면 중 가장 무방비다 【정합 판정】

**코드로 확인한 사실**: `createDocumentStore.save` 는 **`doc = input` 한 줄**이다(`document.ts:33`). 검증도, 존재 검사도, 버전 비교도 없다 — **항상 성공한다.**

**형제 CRUD 어댑터와 대조하면 격차가 드러난다**: `createCrudAdapter.update` 는 없는 id 에 **`HttpError(409, '다른 사용자가 먼저 삭제한 항목입니다.')`** 를 던지고(`crud.ts:126-128`), `crud.ts:123-125` 주석이 그 이력을 기록한다('예전에는 map 이 **없는 id 를 조용히 지나치고 성공을 반환**했다 … 유령 저장(ghost saved)이다'). **단일 문서에는 그 가드가 원리적으로 성립하지 않는다** — 문서는 언제나 존재하기 때문이다. 그래서 **F3b 가 CRUD 쪽에서 닫은 구멍이 여기서는 열린 채 남았다.**

**이것이 만드는 사고 — 조작 없이도, id 없이도**:
- 관리자 A 가 정책을 열고(적립률 1%) 회원가입 적립금을 3000→5000 으로 고치는 동안, 관리자 B 가 적립률을 1→3% 로 바꿔 저장한다. A 가 저장하면 **A 의 스냅샷 7필드 전체가 B 의 3% 를 덮는다** — **적립률이 1% 로 되돌아간다.** 아무도 모른다. 성공 토스트가 뜬다.
- **전체 치환 PUT 이라 서버가 이것이 유실인지 정상 갱신인지 구분할 수 없다.** BE-026 §7.3 의 타임라인 lost update 와 **정확히 같은 구조**다.

**판정**: EP-02 는 **`If-Match: <etag>` 를 요구**하고, 불일치면 **412 `PRECONDITION_FAILED`** 로 저장하지 않는다. EP-01 이 `ETag` 를 내려준다. 이유:
- `Idempotency-Key` 는 **이 문제를 풀지 않는다** — 키는 '같은 요청의 재시도'를 막지 '낡은 스냅샷의 덮어쓰기'를 막지 못한다. 그래서 §4 는 멱등키를 **불필요**로 판정했다.
- `version` 필드가 아니라 `ETag` 를 쓰는 이유: 문서에 `version` 을 넣으면 **폼 값에 그것이 섞인다**(`PointsPolicyValues` 가 곧 폼 타입이다 — `validation.ts:50`) — `issuedCount` 가 폼 값으로 왕복하는 BE-045 §7.2 와 같은 함정이다. `ETag` 는 **헤더에 살아** 폼을 오염시키지 않는다.

**그러나 이 판정은 프론트 인터페이스를 바꾼다 — 어댑터 본문만으로 끝나지 않는다**: `DocumentStore<T>`(`document.ts:14-19`)의 `fetch: (signal) => Promise<T>` 는 ETag 를 돌려줄 자리가 없고 `save: (input, signal?) => Promise<void>` 는 되보낼 자리가 없다. **`CrudAdapter` 는 `WriteContext`(`crud.ts:30-42`)라는 자리를 이미 갖고 있는데 `DocumentStore` 에는 그 대응물이 없다.** 그리고 이 인터페이스를 **단일 문서형 6화면이 공유한다**(회사 정보·CEO 인사말·비전/미션·오시는 길·배송 정책·적립금 정책) — **이 화면만의 작업이 아니다.**

**프론트 후속**: 412 를 받아도 **해소할 UI 가 없다.** `DocumentFormShell` 은 `serverError: string | null` 만 받아(`:76`) 배너로 뭉갠다 — `useCrudForm` 의 conflict 다이얼로그(`useCrudForm.ts:166-179`)를 상속하지 못한다. `FormConflictDialog`(`FormFeedback.tsx:58-74`)는 이미 공유 컴포넌트이므로 **셸이 그것을 받기만 하면 된다.**

### 7.5 정책은 403 이다 — 404 로 은닉하지 않는다. 그리고 변경 이력이 필요하다 【보안 판정】

**은닉 판정**: BE-003 §3.2 의 원칙을 적용하면 **BE-045(쿠폰) §7.4 와 같은 답**이 나온다.
1. **컬렉션이 없다** — 단일 문서라 열거할 id 가 애초에 없다.
2. **개별 리소스의 존재가 개인정보가 아니다** → **403 `FORBIDDEN`**. 정책 문서 7필드 중 개인을 지목하는 것이 **하나도 없다**. 게다가 **404 은닉이 원리적으로 불가능하다** — 경로에 id 가 없으므로 '이 리소스가 존재하는가'를 물을 수 없고, `/api/points-policy` 가 404 면 그것은 '권한 없음'이 아니라 '엔드포인트 없음'으로 읽힌다.

**변경 이력 판정 — 이것이 이 화면의 진짜 보안 요구다**: 적립금 정책은 **금액 규칙**이다. 적립률 1%→50%, 회원가입 적립금 3,000→300,000 은 **클릭 몇 번**이고, 그 효과는 **모든 신규 주문·모든 신규 가입**에 걸린다. 그런데:
- **누가 바꿨는지 기록되지 않는다.** `save(input)` 이 값만 받는다(`document.ts:30-34`).
- **언제 바꿨는지 기록되지 않는다.**
- **무엇에서 무엇으로 바꿨는지 기록되지 않는다** — 전체 치환이라 이전 값이 사라진다.
- **되돌릴 수 없다.**

**판정**: EP-02 는 **감사 이벤트를 남긴다** — `{ actor: 세션 주체, at: 서버 시각, before: 이전 문서, after: 새 문서 }`. 이것은 UX 개선이 아니라 **감사 요구**다: 적립금 이상이 발견됐을 때 '언제부터 정책이 그랬는가'를 답할 수 있어야 하고, 그 답이 없으면 피해 범위를 산정할 수 없다.

**어디에 남기는가는 이 문서가 정하지 않는다** — 앱에 `pages/logs/admin`(관리자 로그) 화면이 있으나(`nav-config.ts:208`) **이 계약이 그것과 이어져 있는지 코드에 근거가 없다.** `POST /api/admin-logs` 류의 심이 이 화면에 없다. **A63 이 관리자 로그 계약(BE-060 계열)과 함께 봐야 한다** — 여기서만 적으면 로그 쪽이 모른다.

**`operator` 조회 허용의 재검토**: `operator` 가 정책을 **읽을** 수 있게 하는 것은(§7.6) 응대 중 '이 회원이 왜 포인트를 못 쓰나'를 답하기 위해서다. 그 정보는 민감하지 않다 — 고객에게도 공개되는 규칙이다(쇼핑몰 하단의 '적립금 안내'). **읽기를 여는 것이 옳다.**

### 7.6 `operator` 를 조회 전용으로 둔다 — BE-045 와 같은 판정

BE-026(1:1 문의)·BE-044(교환/반품)는 `operator` 에게 쓰기를 연다. **이 도메인은 반대이며 BE-045(쿠폰)·BE-010(FAQ)과 같다.**

**근거**: 적립금 정책은 **한 건의 응대가 아니라 전 고객에게 걸리는 금액 규칙**이다. 교환/반품 완료 처리는 '한 주문의 재고 2개'를 움직이지만, 적립률 변경은 '앞으로의 모든 주문'을 움직인다. **범위가 다르면 권한도 다르다.**

**BE-045 보다도 무거운 이유**: 쿠폰은 적어도 **개별 엔티티**라 하나를 잘못 만들어도 그 하나만 문제다. 정책은 **하나뿐**이라 그것이 곧 전부다. 게다가 **확인 다이얼로그도 없다**(FS-046-EL-015 — 저장 버튼 클릭 한 번).

**결론**: EP-01 은 `admin` + `operator`(§7.5). **EP-02 는 `admin` 만.** `operator` 의 저장 시도는 403.

**프론트 영향**: 이 화면에 쓰기 게이팅이 없으므로(FS-046 §7 #2) **`operator` 가 폼을 편집하고 '저장'을 눌러 403 을 받는다** — 그것도 '저장하지 못했습니다. 잠시 후 다시 시도해 주세요.' 라는 **틀린 문구**로(잠시 후에도 영원히 403 이다). **같은 단일 문서형인 `settings/site`·`settings/languages`·`settings/oauth` 는 전부 `canUpdate` 를 배선했다**(`SiteSettingsPage.tsx:109` · `LanguagesPage.tsx:126` · `OAuthPage.tsx:78`) — 이 화면과 형제 `products/shipping` 만 빠졌다.

### 7.7 폼이 숫자를 문자열로 드는 것은 표현이다 — 계약은 `number` 다

**코드로 확인한 사실**: `PointsPolicyValues` 의 6개 숫자 필드가 전부 `z.string()` 기반이다(`validation.ts:8-38,40-48`). `validation.ts:1` 은 이유를 적지 않지만, 형제 `coupons/validation.ts:3` 이 같은 결정에 근거를 남겼다: '숫자 필드는 **입력 원값 보존**을 위해 문자열로 받고 정수 형식을 판정한다'.

**판정: 서버 계약은 `number` 다.** 문자열 ↔ 숫자 변환은 **어댑터 본문**에 산다(§6.1) — 화면은 계속 문자열을 본다.

**이유**:
- `'007'` · `' 1 '` · `'1.0'` 같은 표현 차이가 계약에 새어 들면 **서버가 정규화를 떠안는다.** `intString` 이 `/^\d+$/` 만 보므로 `'007'` 이 통과한다 — 그것이 7 인지 7 의 다른 표현인지 서버가 판정해야 한다.
- 저장→조회 왕복이 값을 바꾸면 안 된다: `'007'` 을 저장하고 조회하면 `7` 이 와야 하는데, 문자열 계약이면 `'007'` 이 돌아와 **폼이 dirty 가 아닌데 값이 달라 보인다.**
- **`DEFAULT_POINTS_POLICY`(`types.ts:19-27`)는 서버 기본값이 아니다** — 조회 전 폼 초기값이다. 서버는 자기 기본값을 갖고 시딩한다(§4 EP-01). 두 값이 갈리면 **조회 전후로 화면이 다른 정책을 보인다** — 그것은 이미 사실이며(`fetch` 가 400ms 걸린다) 무해하다. 다만 **서버 기본값을 프론트 상수와 맞춰 두는 것이 옳다** — A63 이관.
