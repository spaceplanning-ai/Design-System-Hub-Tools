# BE-047. 리뷰 백엔드 기능 명세

## 1. 개요

| 항목 | 내용 |
|---|---|
| 대상 화면 | FS-047 리뷰 (`/products/reviews` · `/products/reviews/:id`) |
| 범위 | 리뷰 목록 조회, 리뷰 상세 조회, **저장(관리자 답변 · 노출/숨김)**, **삭제(단건)** |
| **범위 밖** | **리뷰 작성** — 리뷰는 고객이 상품 페이지에서 쓴다. 관리자가 리뷰를 만드는 API 는 이 계약에 존재하지 않는다(§7.6). **신고 접수·신고 해소** — `reported`·`reportReason` 은 이 화면에서 **읽기 전용**이고 그것을 세우거나 내리는 진입점이 없다. 신고 접수는 고객 채널의 계약이고, 해소 계약은 **미정**이다(§7.3). **일괄 삭제 엔드포인트** — 화면의 일괄 삭제는 개별 `DELETE` 의 병렬 호출이다(§7.8). **상품·평점 요약** — 상품별 평균 별점은 BE-042(상품) 소관이며 이 화면에 표면이 없다(§7.5) |
| 전제 | BE-003 §2·§3 을 상속한다. 인증은 세션 쿠키 기반. 모든 경로는 `/api` 프리픽스. 응답 본문은 `application/json; charset=utf-8`. 시각은 ISO 8601(오프셋 포함), 날짜는 `YYYY-MM-DD` |
| 프론트 어댑터 | `apps/admin/src/pages/products/reviews/data-source.ts` (`reviewAdapter` = 공용 `createCrudAdapter`, 전용 함수 **0개**) |
| 도메인 타입 | `apps/admin/src/pages/products/reviews/types.ts` — 리뷰 타입 + 순수 뷰/필터 헬퍼. **다른 도메인 타입을 import 하지 않는다** — `pages/products/_shared/store.ts` 와 연결이 없다(§7.5) |
| 검증 정본 | **없다.** zod 스키마가 없고 `useCrudForm` 도 쓰지 않는다. `types.ts` 의 순수 함수는 전부 **표시·필터·정렬**(`starText`·`isPhotoReview`·`filterByRating`·`searchReviews`·`sortReviews`·`averageRating`)이며 **저장을 거절하는 규칙이 하나도 없다**. 어댑터도 재판정하지 않는다 — `patch: (item, input) => ({ ...item, ...input })`(`data-source.ts:78`)가 전부다. **유일한 상한은 `REVIEW_REPLY_MAX = 500` 이고 그것을 강제하는 것은 네이티브 `maxLength` 속성뿐이다**(§7.1) |

> **에러 봉투·권한 모델 상속**: BE-003 §2·§3 을 그대로 상속한다. 아래는 리뷰 고유 차이만 기술한다.

## 2. 공통 (상속)

- **에러 봉투**: BE-003 §2. 공통 에러코드 동일(`VALIDATION_FAILED` · `UNAUTHENTICATED` · `FORBIDDEN` · `CSRF_TOKEN_INVALID` · `NOT_FOUND` · `CONFLICT` · `UNPROCESSABLE` · `RATE_LIMITED` · `INTERNAL_ERROR` · `REQUEST_TIMEOUT`).
- **권한**: `admin` = 전체. `operator` = 조회 계열 + **답변·노출 저장**(EP-03) — 리뷰 응대는 운영자의 본업이다. **그러나 `DELETE`(EP-04)는 `admin` 전용**이다 — 이 도메인의 유일한 비대칭이며 §7.7 이 근거를 댄다. 상품 도메인 읽기 권한 없는 관리자 → 컬렉션 403 / **개별 리뷰 404 은닉**(§7.2).
- **CSRF**: 쓰기(PUT · DELETE)에 `X-CSRF-Token`.
- **타임아웃**: 조회 5초 → 504. 쓰기(저장·삭제) 5초 → 504. **재고 트랜잭션 같은 무거운 부수효과가 없어** BE-044 처럼 상한을 늘릴 이유가 없다.
- **프론트 권한 게이팅이 이 화면에는 아예 없다** — `useRouteWritePermissions` 미소비(FS-047-EL-038). 형제 화면(`products/returns/ReturnDetailPage`)이 배선한 UX 층이 여기 없으므로, **read 전용 역할이 삭제 버튼을 눌러 서버 403 을 받는 경로가 일상적으로 발생한다.** 서버는 그것을 정상 거절로 처리하고 프론트 문구가 뭉개지는 것은 프론트 몫이다(FS-047 §7 #3·#6). 권한 강제는 전적으로 서버 책임이다(`RequirePermission.tsx:8-11`).

## 3. 데이터 계약 (`types.ts` 대조)

| 타입 | 필드 | 비고 |
|---|---|---|
| `Review` | `id` · `productId` · `productName`(비정규화) · `author`(마스킹된 닉네임) · `rating` · `content` · `imageUrls[]` · `createdAt`(`YYYY-MM-DD`) · `visible` · `reported` · `reportReason` · `reply` | 목록·상세가 **같은 타입**이다 — 목록도 `content` 전문·`imageUrls`·`reportReason` 을 받는다(§7.8). **`version`/`updatedAt`/`deletedAt` 이 없다**(§7.1) |
| `ReviewInput` | `Omit<Review, 'id'>` — 11개 필드 **전부** | 저장 입력. **고객 소유 필드(`author`·`rating`·`content`·`imageUrls`·`createdAt`)와 서버 소유 필드(`reported`·`reportReason`)까지 포함한 전체 치환**이다(§7.1 【보안 판정】). 관리자가 실제로 바꾸는 것은 `reply`·`visible` 2개뿐이다 |
| `ReviewRating` | `1 \| 2 \| 3 \| 4 \| 5` | 작성 후 불변. 관리자 변경 UI 없음 |
| 상수 | `REVIEW_REPLY_MAX = 500` | 관리자 답변 상한. **강제는 네이티브 `maxLength` 뿐** — 어댑터도 zod 도 재판정하지 않는다 |

**리뷰에는 lifecycle status 축이 없다**

| 축 | 값 | 표현 | 전이 규칙 |
|---|---|---|---|
| `visible` | `true` \| `false` | 목록 인라인 토글(FS-047-EL-007.10) · 상세 토글(EL-024) · 미리보기 흐림(EL-033.5) | **없다** — 불리언이라 어느 방향으로든 자유롭게 뒤집힌다(§7.3) |

> **BE-044 의 `RETURN_STATUS` 나 BE-026 의 `STATUS_FLOW` 에 해당하는 것이 리뷰에는 없다.** 세 축이 서로 독립한 플래그이며, 그래서 §5 의 **422 상태위반 축이 대부분 비어 있다** — 위반이라 부를 상태가 정의된 적이 없기 때문이다(§7.3).

### BE-047-EP-03 · 저장 (관리자 답변 · 노출/숨김)
| 항목 | 내용 |
|---|---|
| 근거 (FS) | FS-047-EL-007.10, EL-014, EL-024, EL-025, EL-028, EL-029 |
| 심 | `data-source.ts:70` `PUT /api/reviews/:id (관리자 답변·노출 토글)` |
| 메서드·경로 | `PUT /api/reviews/:id` |
| 권한 | `admin`, `operator`(§7.7). 상품 도메인 읽기 권한 없음 → **404 은닉** |
| 멱등성 | **`Idempotency-Key` 헤더로 강제한다**(§7.1). **도메인 멱등 키가 없다** — BE-044 의 `stockAppliedAt` 같은 자연 키가 리뷰에는 존재하지 않으므로 **헤더가 유일한 방어**다 |
| 레이트리밋 | 분당 60회 |

**바디**(현재 `ReviewInput`): `productId` · `productName` · `author` · `rating` · `content` · `imageUrls[]` · `createdAt` · `visible` · `reported` · `reportReason` · `reply`.

**서버 검증 (요청을 그대로 믿지 않는다 — §7.1)**
1. **고객 소유 필드 무시**: `author` · `rating` · `content` · `imageUrls` · `createdAt` 은 **관리자가 바꿀 수 없다.** 프론트가 `toReviewInput(review)`(`types.ts:103-117`)로 되돌려 보내지만 **서버는 저장된 값을 정본으로 유지하고 요청 값을 무시**한다 — 관리자가 고객의 말을 고치는 것은 이 계약의 능력이 아니다(§7.1 【보안 판정】).
2. **서버 소유 필드 무시**: `reported` · `reportReason` 은 신고 채널이 세운다 — **절대 이 요청에서 받지 않는다**(§7.3).
3. **비정규화 참조 무시**: `productId` · `productName` 은 작성 시점에 확정된다 — 요청에서 받지 않는다(§7.5).
4. **`reply`**: 앞뒤 공백 제거 후 0–500자(`REVIEW_REPLY_MAX`). 초과 시 400 `VALIDATION_FAILED`(`error.fields[].name = 'reply'`). **빈 문자열은 유효하다** — 답변 철회가 정상 경로다.
5. **`visible`**: 불리언. 그 밖의 값은 400. **전이 규칙은 없다**(§7.3).

**응답 200/204**. 프론트 `update(id, input, context?): Promise<void>` — 응답 본문을 읽지 않고 상세를 재조회한다(`ReviewDetailPage.tsx:138`) · 목록을 무효화한다(`crud.ts:312-315`).

**에러**: 400 `VALIDATION_FAILED`(`error.fields`: `reply`·`visible`) · 401 · 403 `FORBIDDEN` · 403 `CSRF_TOKEN_INVALID` · **404 `REVIEW_NOT_FOUND`** · **409 `CONFLICT`**(§7.1) · 429 · 500 · 504.

> **422 가 이 엔드포인트에 없다.** 상태 위반이라 부를 상태가 정의된 적이 없기 때문이다(§3 · §7.3). 안 A(§7.1)를 채택하면 바디가 `{ reply, visible }` 2개로 줄어 **거절할 수 있는 것이 400 형식 위반뿐**이 된다 — 그것이 이 도메인의 정확한 모양이다.

## 7. 핵심 판정

### 7.1 관리자가 고객의 말을 되돌려 보낸다 — 전체 치환 PUT 이 UGC 를 위조 가능하게 만든다 【보안 판정】

**이것이 이 화면의 중심 문제다.**

**코드로 확인한 사실**: `toReviewInput(review)`(`types.ts:103-117`)가 `Review` 의 **11개 필드를 전부 복사**해 `ReviewInput` 을 만들고, 화면 두 곳이 그것을 그대로 `update.mutate` 의 `input` 으로 보낸다 — 상세 저장(`ReviewDetailPage.tsx:132` — `{ ...toReviewInput(review), reply: reply.trim(), visible }`)과 **목록의 인라인 토글**(`ReviewListPage.tsx:154` — `{ ...toReviewInput(item), visible: next }`)이다. 어댑터는 그 값을 `{ ...item, ...input }`(`data-source.ts:78`)으로 **통째로 덮는다**. 관리자가 실제로 바꾸는 것은 `reply` 와 `visible` **2개**인데 계약은 11개를 받는다.

**이것이 만드는 사고 — 조작된 클라이언트 하나면 충분하다**:
1. **`content` 를 고쳐 보내면 고객이 쓰지 않은 말이 고객 이름으로 상품 페이지에 걸린다.** 리뷰는 다른 고객의 구매 판단 근거이며, 그 신뢰의 전제는 '판매자가 쓰지 않았다'는 것 하나뿐이다. **이 계약은 그 전제를 지키지 못한다** — 서버가 요청 값을 믿는 순간 판매자가 5점 리뷰의 본문을 대신 쓸 수 있고, **작성자 닉네임(`author`)과 작성일(`createdAt`)이 그대로 붙어 있어 위조를 구분할 수 없다.**
2. **`rating` 을 고쳐 보내면 평점이 조작된다.** 1점 리뷰를 5점으로 바꾸는 데 필요한 것은 요청 바디 한 줄이다. `averageRating`(`types.ts:88-92`)이 소비자 0이라 지금은 드러나지 않지만, 상품별 평균 평점이 붙는 순간 이것이 곧 **표시 가격에 준하는 상거래 정보**가 된다.
3. **`imageUrls` 를 고쳐 보내면 임의 URL 이 고객 페이지의 `<img src>` 가 된다.** 이 값은 `ImageThumb`(`ImageThumb.tsx:52`)를 통해 **실제 네트워크 요청**이 되며, 프론트에 스킴·호스트 검증이 0건이다. 관리자 화면에서는 미리보기(FS-047-EL-033.3)일 뿐이지만 **같은 배열이 고객 상품 페이지로 나간다** — 외부 호스트를 심으면 리뷰를 본 고객의 IP·UA 가 그 호스트로 새고, `javascript:`·`data:` 스킴은 렌더러에 따라 더 나쁘다.
4. **`reported`·`reportReason` 을 고쳐 보내면 신고 사실이 지워진다.** 신고는 **관리자가 아니라 고객·자동 필터가 세우는 사실**인데(§7.3) 관리자 저장 요청이 그것을 덮을 수 있다 — 신고된 리뷰를 노출로 되돌리면서 `reported: false` 를 함께 보내면 **신고가 있었다는 기록 자체가 사라진다.**

**판정**: `author` · `rating` · `content` · `imageUrls` · `createdAt` · `reported` · `reportReason` · `productId` · `productName` 은 **전부 서버가 정본으로 유지한다.** 요청 바디의 이 값들은 **무시**한다 — 파싱조차 하지 않는다. 이것은 UX 개선이 아니라 **UGC 무결성 요구**다: 리뷰의 가치는 그것이 판매자의 말이 아니라는 데 있고, 그 경계가 무너지면 리뷰 기능 전체가 의미를 잃는다.

**계약을 둘 중 하나로 바꾼다.**

| 안 | 형태 | 평가 |
|---|---|---|
| **A (권장)** | `PUT /api/reviews/:id` 는 **관리자가 실제로 바꾸는 2개만** 받는다: `{ reply, visible }` | 위조가 **구조적으로 불가능**하다. 요청 크기가 본문·이미지 수에 비례하지 않는다. 프론트 `toReviewInput` 이 **통째로 사라진다**(`types.ts:103-117` · 테스트 `:82-84`). **어댑터 시그니처는 그대로다**(`CrudAdapter.update(id, input)` — `Input` 타입만 좁아진다) — 화면은 `ReviewDetailPage.tsx:132` 와 `ReviewListPage.tsx:154` **두 곳만** 바뀐다. 인라인 토글은 `{ visible: next }` 한 줄이 된다. **BE-044 §7.3 안 A 와 같은 판정이며 리뷰 쪽이 더 단순하다** |
| **B (차선)** | PUT 을 유지하되 서버가 고객·서버 소유 필드를 **무시** | 어댑터를 덜 흔든다. 그러나 `ReviewInput` 이 실제로 쓰이지 않는 9개 필드를 계속 실어 보내는 기형이 남고, '무시한다'는 계약은 구현이 한 줄 빠지면 조용히 깨진다 — 안 A 는 타입이 강제한다. **리뷰는 안 B 의 비용이 특히 크다** — `content`·`imageUrls` 가 무제한 길이라 요청 크기가 이력에 비례해 커진 BE-044 보다 나쁘다 |

**어느 안이든 서버는 요청의 `content`·`rating`·`author`·`imageUrls`·`reported` 를 절대 신뢰하지 않는다.**

**낙관적 동시성**: `Review` 에 `version`/`updatedAt` 필드가 **없다**(`types.ts:7-27`). 어댑터의 409 는 **'존재 여부' 기반**이다(`crud.ts:126-128` — 없는 id 면 409). 즉 **동시 편집(둘 다 존재)은 last-write-wins** 다 — **그리고 이 화면에서 그것이 특히 나쁘다**: 두 관리자가 같은 리뷰를 열어 하나는 답변을 쓰고 하나는 숨기면, **전체 치환이라 나중 저장이 앞선 답변을 통째로 지운다.** 인라인 토글도 같다 — 목록에서 노출만 껐을 뿐인데 그 사이 다른 관리자가 쓴 답변이 `toReviewInput(item)` 의 낡은 스냅샷으로 덮인다(FS-047-EL-007.10). **판정**: 안 A 채택 시 다투는 필드가 2개로 줄어 `If-Match`(ETag) 또는 `Review.version` 이 **선택**, 안 B 유지 시 **필수**.

### 7.2 리뷰 상세는 404 로 은닉한다 · 작성자 마스킹은 서버가 한다 【보안 판정】

BE-003 §3.2 의 원칙 두 줄을 이 도메인에 적용한다.

1. **컬렉션의 존재는 비밀이 아니다** → `GET /api/reviews` 권한 부족 시 **403 `FORBIDDEN`**.
2. **개별 리뷰 리소스의 존재 자체가 개인정보다** → 상품 도메인 **읽기 권한이 없는** 주체에게는 **404 `REVIEW_NOT_FOUND`** 로 은닉한다.

**근거**: 리뷰 1건은 `author`(작성자 닉네임) · `content`(고객이 쓴 원문 — '생각보다 볼이 좁아요. 반 사이즈 크게 주문하세요.' `data-source.ts:44` 같은 사적 사용 경험) · `imageUrls`(고객이 찍은 사진 — **배경에 집·얼굴·주소가 담길 수 있다**) · `createdAt`(구매 시점의 근사)을 담는다. **리뷰 id 열거는 곧 그 상품을 산 사람들의 목록이다** — `productId` 로 묶으면 특정 상품(예: 의료·성인 카테고리)의 구매자 집합이 드러난다. `Review` 는 회원 레코드보다 민감도가 낮지 않다 — BE-003 §3.2 가 `GET /api/members/:id` 를 404 로 은닉하는 것과 **같은 이유로** 404 여야 한다.

**반대로 읽기 권한이 있는 주체**(`operator`)가 쓰기에서 거절될 때는 **403** 을 준다 — 이미 리뷰의 존재를 아는 주체에게 존재를 숨기는 것은 의미가 없다. 이 도메인에서 그 403 이 실제로 발생하는 자리는 **삭제(EP-04)** 다(§7.7).

**작성자 마스킹**: 픽스처는 `'민**'`·`'지**'`·`'현**'`·`'수**'`(`data-source.ts:14,28,42,56`)처럼 **이미 마스킹된 값**을 담고 타입 주석이 그것을 계약으로 선언한다(`types.ts:11` — `작성자 — 마스킹된 닉네임(실명 아님)`). **판정: 마스킹은 서버가 한다** — 원본 닉네임을 내려보내고 프론트가 가리면 응답 본문에 평문이 남는다(개발자 도구·프록시·로그에 그대로 보인다). **리뷰는 이 위험이 회원 목록보다 크다** — 응답이 `content` 와 함께 오므로 '누가 무엇을 샀고 어떻게 느꼈는가'가 한 객체에 모인다. 리뷰 응대에 실명이 필요한 경우(법적 분쟁·명예훼손 신고 대응)는 **별도 권한의 별도 엔드포인트**로 분리한다 — 목록에 평문을 뿌리지 않는다.

**프론트 영향**: **FS-047-EL-019 가 404 를 '리뷰를 불러오지 못했습니다.' 라는 일반 문구로 그린다** — BE-044 가 '요청을 찾을 수 없습니다. 이미 삭제되었을 수 있습니다.' 로 그리는 것과 다르다. 은닉의 관점에서는 **더 안전하다**(존재 여부에 대한 정보를 아예 주지 않는다). 그러나 정당한 운영자에게도 아무 설명이 없고 재시도 수단조차 없다 — **은닉이 우연히 달성된 것이지 설계된 것이 아니다**(§7.10 #4).

### 7.3 노출/숨김·신고에 전이 규칙이 없다 · 신고를 해소할 계약이 없다 【정합 판정】

**코드로 확인한 사실**: `visible`·`reported` 는 **불리언**이고 `reply` 는 **문자열**이다(`types.ts:20-26`). 세 축 사이에 어떤 판정도 없다 — `canTransition`·`allowedNext` 류 함수가 `grep` 0건이다. 리뷰에는 BE-044 의 `ReturnStatus` 나 BE-026 의 `STATUS_FLOW` 에 해당하는 **lifecycle status 축이 아예 없다**.

**그 자체는 대체로 옳다.** 노출/숨김은 **가역적인 표시 스위치**이고, 숨긴 리뷰를 다시 노출하는 것은 사고가 아니라 정상 조작이다(오판으로 숨겼다면 되돌려야 한다). BE-044 의 완료 전이처럼 **비가역 부수효과가 없으므로** 흐름 상수를 세울 이유가 없다. **`visible` 에는 전이 규칙을 두지 않는다.**

**문제는 `reported` 다.**
- **관리자가 신고를 세우지도 내리지도 못한다.** 화면은 `reported` 를 배너(FS-047-EL-023)와 배지(EL-007.8·EL-032)로 **읽기만** 하고, 그것을 바꾸는 컨트롤이 없다. 관리자가 할 수 있는 것은 **숨기거나 지우는 것**뿐이다.
- **그래서 '검토했고 문제없음'을 표현할 방법이 없다.** 신고를 확인하고 정당한 리뷰라고 판단해도 배너가 영원히 남는다 — 다음 관리자가 같은 리뷰를 다시 검토한다. 신고 트리아지가 업무인데(FS-047 §1) **처리 완료를 기록할 자리가 없다.**
- **동시에 `reported` 가 요청 바디에 있어 조작 가능하다**(§7.1). 즉 **정당한 경로로는 못 내리는데 부당한 경로로는 내릴 수 있다** — 정확히 거꾸로다.

**판정**: `reported`·`reportReason` 은 **서버가 소유한다** — EP-03 이 요청 바디에서 받지 않는다(§7.1). 신고 해소는 **별도 엔드포인트**가 필요하다:

| 필요한 것 | 형태(제안) | 상태 |
|---|---|---|
| 신고 접수 | 고객 채널의 계약 — 이 문서의 범위 밖 | **미정** — 고객 채널 계약 부재 |
| 신고 해소(기각/인정) | `POST /api/reviews/:id/report-resolution` 류 + `resolution`(`dismissed`\|`upheld`) + 처리자·처리 시각 | **미정 — 심이 없다. 발명하지 않는다** |
| 해소 이력 | 누가 언제 어떤 판단을 했는가(감사) | **미정** |

**이 판정은 계약을 만들지 않는다.** `data-source.ts:70` 의 심이 지목하는 것은 `GET /api/reviews · GET/PUT/DELETE /api/reviews/:id` **4개뿐**이고, 신고 해소 엔드포인트의 근거가 코드에 **0건**이다. **A01 이 신고 처리 흐름을 확정해야** 이 자리를 채울 수 있다 — 서버가 임의로 흐름을 정하면 화면에 그것을 부를 컨트롤이 없어 dead endpoint 가 된다. **§5 의 422 상태위반 축이 EP-03·EP-04 에서 비어 있는 이유가 이것이다**: 위반이라 부를 상태가 정의된 적이 없다. §7.3 이 채택되면 **그때 422 `INVALID_REPORT_TRANSITION` 이 §5 에 추가된다**(예: 이미 해소된 신고를 다시 해소).

### 7.4 삭제는 하드 삭제이고 되돌릴 수 없다 · 어댑터의 409 는 계약이 아니다 【정합 판정】

**코드로 확인한 사실**:
- **하드 삭제다** — `crud.ts:143` 이 `items = items.filter((item) => item.id !== id)`. `Review` 에 `deletedAt`·`status`·`archived` 가 **없다**(`types.ts:7-27`). soft-delete 흔적 0건.
- **복원 경로가 없다** — 화면·어댑터에 undo·restore 가 0건. 다이얼로그가 '이 작업은 되돌릴 수 없습니다.'(`ReviewDetailPage.tsx:309` · `useCrudList.tsx:158`)라고 정확히 말한다.
- **어댑터가 없는 id 에 409 를 던진다** — `crud.ts:139-141` `throw new HttpError(409, '이미 삭제된 항목입니다.')`.

**판정 ①: `DELETE` 는 멱등이어야 한다 — 409 를 계약에 넣지 않는다.**
공용 어댑터의 409 는 **`update` 의 유령 저장 방지(`crud.ts:123-125` 주석)를 `remove` 에 대칭으로 복사한 것**이고, 픽스처에서는 합리적이다(개발자가 '지워지지 않았다'를 즉시 본다). **그러나 HTTP 계약으로는 틀리다** — `DELETE` 의 의미는 '이 리소스가 없는 상태로 만들라'이고, 이미 없으면 그 요청은 **성공했다**. 409 로 만들면:
- **일괄 삭제에서 즉시 드러난다**(§7.8) — 다른 관리자가 먼저 지운 건이 '삭제하지 못했습니다' 로 집계된다. **운영자는 지워진 것을 지우지 못했다고 듣는다.**
- **재시도가 영원히 실패한다** — 부분 실패 후 재클릭하면(FS-047-EL-013) 이미 성공한 건이 이번엔 409 를 받아 실패 수가 **늘어난다.**

**계약**: `DELETE /api/reviews/:id` → 있으면 **204**, 없으면 **404 `REVIEW_NOT_FOUND`**(은닉 정책과 일관 — §7.2), **409 는 쓰지 않는다.** 어댑터는 `remove` 의 없는 id 응답을 `HttpError(404)` 로 옮겨야 하며 **이는 `crud.ts` 공용 팩토리의 변경**이라 다른 소비 화면과 함께 봐야 한다(§7.10 #7).

**판정 ②: 삭제 제약을 두지 않는다.**
'답변이 달린 리뷰는 못 지운다' · '신고된 리뷰는 못 지운다' 같은 제약이 코드에 **0건**이고, 계약에도 **두지 않는다.** 근거: 리뷰 삭제의 정당한 사유(욕설·광고·개인정보 노출·명예훼손)는 **답변 여부와 무관**하며, 오히려 답변을 단 뒤 삭제가 필요해지는 경우가 흔하다.

**판정 ③: 그러나 하드 삭제는 위험하고, 그 위험을 기록한다.**
- **오삭제의 복구 경로가 없다.** 고객이 쓴 원문·사진이 사라지고, 그 리뷰가 근거였던 분쟁(반품·환불)에서 증거가 없어진다.
- **`operator` 를 막는 것이 그 유일한 완화다**(§7.7) — 그래서 이 도메인은 삭제만 `admin` 전용이다.
- **soft-delete 는 이 계약이 정하지 않는다** — 심이 `DELETE` 를 지목하고 화면이 '되돌릴 수 없습니다'를 명시하므로, 하드 삭제가 **현재의 의도된 동작**이다. soft-delete 도입은 보존정책·개인정보 파기 요구와 함께 봐야 할 별도 계약이다(§7.10 #8). **개인정보 파기(고객 탈퇴·보존기간 만료)를 하드 삭제 버튼으로 대신하지 말 것** — 그것은 배치의 일이다.

### 7.5 `productId` 가 어디에도 연결되지 않는다 【경계 판정】

**코드로 확인한 사실**: `Review.productId`(`types.ts:9`)는 픽스처에 값이 있고(`data-source.ts:12,26,40,54`) `toReviewInput` 이 복사하며(`types.ts:105`) PUT 바디에 실려 왕복한다. **그런데 그것을 읽는 코드가 0건이다** — grep 히트가 선언·픽스처·복사·테스트뿐이고 **조인·링크·조회가 하나도 없다.** 형제 화면(FS-044 교환/반품)은 같은 자리에서 `fetchReturnProduct(productId)`(`returns/data-source.ts:195-206`)로 상품을 읽는데, **리뷰는 `pages/products/_shared/store.ts` 를 import 조차 하지 않는다**(grep 0건).

**이것이 만드는 것**:
- **상품으로 가는 링크가 없다** — 리뷰를 보다 '이 상품이 뭐지'를 물으면 상품명을 복사해 상품 목록에서 검색해야 한다.
- **`productName` 이 낡는다** — 상품명이 바뀌어도 리뷰의 비정규화 사본은 그대로이고, **조인이 없어 화면이 그 어긋남을 알 수조차 없다.** 목록의 상품명 검색(FS-047-EL-001)이 옛 이름으로만 맞는다.
- **상품이 지워져도 리뷰는 그것을 모른다** — BE-044 는 같은 상황에서 404 배너를 그리는데(그것도 문제이지만 최소한 **드러난다**), 리뷰는 **조용히 유령 상품의 리뷰를 계속 보인다.**
- **`averageRating`(`types.ts:88-92`)이 소비자 0이다** — 테스트만 부른다(`reviews.test.ts:46-54`). **상품별 평점 요약이라는 표면이 계획됐다가 붙지 않은 흔적**으로 읽힌다.

**판정**: **이 계약은 `productId` 로 무엇도 하지 않는다.** `GET /api/reviews/:id` 응답에 상품 객체를 끼워 넣지 않고, `GET /api/products/:id` 를 이 문서가 만들지도 않는다(BE-042 소관 — BE-044 §7.4 와 같은 경계). **현재 화면에 그 표면이 없기 때문이다** — 없는 소비처를 위해 엔드포인트를 발명하지 않는다.

**그러나 이것은 미완의 신호이며 A01 이 확정해야 한다**: `productId` 가 있고 `averageRating` 이 있는데 둘 다 쓰이지 않는다는 것은 **'상품별 리뷰 요약'이라는 화면이 의도됐다는 뜻**이다. 그것이 확정되면 (a) 리뷰 목록에 `?productId=` 필터, (b) 상품 상세에 평점 요약 위젯, (c) 리뷰 상세 → 상품 상세 링크가 함께 붙고 **`averageRating` 이 서버로 올라간다**(전량을 받아 클라이언트에서 평균 내는 것은 페이징과 양립하지 않는다 — §7.8). **그전까지 `productId` 는 응답에 남기되(작성 시점 사실이다) 요청에서는 받지 않는다**(§7.1).

### 7.6 리뷰 작성 — 계약에 존재하지 않는다 【범위 판정】

**코드로 재확인한 사실**:
- `reviewAdapter.create` 는 `createCrudAdapter` 가 **인터페이스로 제공**할 뿐(`crud.ts:110-117`) **호출부가 0건**이다(grep).
- 화면에 등록 버튼·진입점이 **없다** — `CrudListShell` 의 `empty.createAction` 도 넘기지 않아(`ReviewListPage.tsx:205-210`) 빈 상태의 생성 CTA 조차 없다.
- 심(`data-source.ts:70`)이 `GET`·`PUT`·`DELETE` 만 지목하고 **`POST` 를 지목하지 않는다.**
- `build: (input) => { seq += 1; return { id: `rv-${String(seq)}`, ...input }; }`(`data-source.ts:74-77`)가 **id 생성기를 갖고 있으나 도달 불가능하다.**

**판정**: `POST /api/reviews` 를 **만들지 않는다.** 심이 없는 것은 누락이 아니라 **의도된 부재**다 — 리뷰는 상품을 산 고객이 상품 페이지에서 쓰는 것이고, 관리자가 리뷰를 만들 수 있다면 §7.1 이 막으려는 바로 그 위조가 **정문으로 들어온다.** BE-003 §1 이 '**회원 생성** — 고객은 회원가입으로만 유입된다' 로, BE-026 §7.7 이 '**문의 생성** — 문의는 고객 채널이 접수한다' 로, BE-044 §7.6 이 '**요청 등록** — 고객이 접수한다' 로 확정한 것과 같은 결이다. **리뷰는 그중 가장 강한 사례다**: 다른 셋은 '관리자가 대신 접수'가 상상 가능하지만, **관리자가 대신 쓴 리뷰는 리뷰가 아니다.**

**부재의 표현이 약하다는 사실은 남긴다**: BE-026 의 `ticketAdapter` 는 명시적 거절 구현(`Promise.reject`)을 뒀는데 이 어댑터는 `createCrudAdapter` 를 써서 **자리가 자동으로 채워졌고 id 생성기까지 살아 있다** — 실수로 `create` 를 부르면 **리뷰가 실제로 만들어진다.** BE-044 §7.11 #8 과 같은 지적이다(§7.10 #9).

### 7.7 `operator` 에게 저장은 열고 삭제는 막는다 — 이 도메인의 유일한 비대칭

BE-044 §7.8 은 `operator` 에게 교환/반품의 **모든** 쓰기를 열었다. **리뷰는 갈라 놓는다.**

**저장(EP-03)을 여는 근거**: 리뷰 답변은 **운영자의 본업**이다. 낮은 별점을 훑어 답변을 다는 루프가 이 화면의 존재 이유이고(FS-047 §1 · `ReviewListPage.tsx:8-9` 주석), `operator` 가 그것을 못 하면 이 화면을 쓸 사람이 `admin` 뿐이라 역할 구분이 무의미해진다. **노출/숨김도 연다** — 욕설·광고 리뷰를 즉시 내리는 것은 시간이 걸리면 안 되는 조작이고, **가역적**이라 오판의 대가가 작다(§7.3).

**삭제(EP-04)를 막는 근거 — BE-044 와 다른 판정**:
1. **비가역이다.** 하드 삭제라 복원 경로가 없다(§7.4). BE-044 의 완료 처리도 비가역이지만 **거기엔 3중 방어가 있었다**(select disabled · `validateStockPlan` · 어댑터 422). **리뷰 삭제에는 확인 다이얼로그 하나뿐**이고 그 문구가 유일한 게이트다.
2. **대상이 남의 말이다.** 재고는 회사의 것이고 요청은 처리 대상이지만, **리뷰는 고객이 쓴 것**이다. 그것을 영구히 없애는 판단은 '이 리뷰가 정책 위반인가'라는 **해석**을 요구하며, 그 해석의 책임 단위는 `admin` 이다.
3. **일괄 삭제가 있다.** `operator` 에게 열면 **전량 선택 → 삭제**가 클릭 두 번이다(FS-047-EL-011·EL-013). 교환/반품에는 일괄 조작 자체가 없었다.
4. **완화 수단이 이미 충분하다.** `operator` 는 **숨길 수 있다**(가역) — 부적절한 리뷰에 대한 즉각 대응은 숨김으로 완결되고, 영구 삭제는 급하지 않다.

**결론**: EP-01 · EP-02 · EP-03 = `admin` + `operator`. **EP-04 = `admin` 전용**(`operator` → 403). 상품 도메인 권한이 아예 없는 역할은 전부 차단(§7.2 은닉 규칙 적용).

**⚠ 프론트가 이 비대칭을 표현하지 못한다**: 이 화면에 **쓰기 게이팅이 아예 없어**(FS-047-EL-038 — `useRouteWritePermissions` 미소비) `operator` 에게도 삭제 버튼이 그대로 보이고, 누르면 서버 403 이 '삭제하지 못했습니다. 잠시 후 다시 시도해 주세요.' 로 뭉개진다 — **재시도해도 영원히 403 인데 재시도를 권한다.** 형제 `products/returns/ReturnDetailPage.tsx:110,343-345,351` 은 그 훅으로 버튼을 숨기고 사유를 배너로 말한다. **§7.7 의 역할 판정은 그 프론트 작업이 선행돼야 사용자에게 의미가 있다**(§7.10 #1).

### 7.8 목록이 전량·전문을 내려주고, 일괄 삭제가 개별 DELETE 를 병렬로 두드린다

**현재 계약의 세 문제**:
1. **페이징이 없다.** `fetchAll(signal)` 이 파라미터를 받지 않고 전량을 반환하며(`crud.ts:94-98`), 프론트가 별점 필터·검색을 전부 클라이언트에서 한다. **리뷰는 매일 쌓이는 무한 증가 컬렉션이며 상한이 없다** — 회원·FAQ 와 다르고, 교환/반품보다도 빠르게 는다(모든 주문이 반품되지는 않지만 모든 상품이 리뷰를 받는다).
2. **목록이 상세 전문을 담는다.** `Review` 타입 하나를 목록·상세가 공유해 목록 응답에 `content` **전문**과 `imageUrls`·`reportReason` 이 실린다 — 목록은 `content` 를 **ellipsis 로 잘라 보여줄 뿐이고**(`ReviewListPage.tsx:78-84,131`) `imageUrls` 는 **배지 판정(`isPhotoReview`)에만 쓰며**(`:137`) `reportReason` 은 **목록에서 아예 안 쓴다.** 즉 **쓰지 않는 데이터를 받고**, 동시에 **개인정보를 필요 이상으로 넓게 노출한다**(§7.2 의 취지와 충돌).
3. **일괄 삭제가 엔드포인트가 아니다.** `settleAll(ids, (id) => adapter.remove(id, { signal }))`(`crud.ts:349-350`)가 **id 마다 개별 `DELETE` 를 병렬로** 낸다. 100건을 선택하면 **100 요청이 동시에** 나가고, EP-04 의 분당 60 레이트리밋에 **한 번의 클릭으로 닿는다.** 부분 실패는 건수로만 오고(FS-047-EL-013) 어느 건이 실패했는지 화면이 모른다 — 재시도가 전량을 다시 보낸다.

**판정**:
- BE-010 이 `FaqSummary`/`Faq` 로 나눈 것과 같이 **`ReviewSummary` / `Review` 를 분리**하고, EP-01 에 `rating`·`keyword`·`page`·`size`(기본 20 · 상한 100) 쿼리를 도입한다. **`ReviewSummary` 는 `content` 를 `contentPreview`(서버가 자른 고정 길이)로 대체하고 `imageUrls` 를 `photoCount: number` 로, `reportReason` 을 뺀다** — 화면이 실제로 쓰는 것이 정확히 그것이다(ellipsis 미리보기 · 포토 배지 · 신고 배지).
- **일괄 삭제 엔드포인트(`POST /api/reviews/bulk-delete`)는 지금 만들지 않는다** — 심이 없다(엔드포인트 발명 0건 기준). **그러나 §7.4 의 404 판정이 채택되면 현재 구조의 최악(이미 지워진 건을 '실패'로 세는 것)은 해소된다.** 레이트리밋 문제는 남으므로, 페이징 도입 시 **선택 상한(= page size)** 이 자연히 걸려 완화된다. 일괄 엔드포인트는 그 뒤에도 필요하면 별도 계약으로 연다.

**이관**: 이 변경은 **프론트 대공사**다 — `filterByRating`/`searchReviews` 가 서버로 올라가고, 페이지네이션 UI(FS-047 §7 #2)·`SeqCell` 오프셋·스켈레톤 행 수·전체 선택 라벨('이 페이지의')이 함께 붙어야 한다. **URL list state 는 이미 있다**(`useListState` — FS-047-EL-004) — `page` 파라미터와 `clampPage`(`useListState.ts:217-223`)가 **미사용 상태로 대기 중**이라 IA-13 은 선행 조건이 아니다. quality-bar IA-04 P0 가 이 화면을 gap 으로 잡고 있다. **동시에 `searchReviews` 가 `content` 를 보지 않는 문제(FS-047 §7 #10)도 서버 검색으로 옮기며 함께 푼다** — 전문 검색은 원래 서버의 일이다. 그전까지 현 계약(전량)을 유지한다 — 픽스처 4건에서는 드러나지 않는다.
