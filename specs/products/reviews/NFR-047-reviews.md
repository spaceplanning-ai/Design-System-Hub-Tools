### 1.2 이 화면의 성격

**남의 말을 공표하는 화면이다.** 이 화면의 쓰기는 재고나 금액을 움직이지 않는다 — 대신 **고객이 쓴 글이 다른 고객에게 보이는지**를 정한다. 그래서 이 화면의 P0 판정은 세 곳에서 다르게 읽힌다.

1. **비가역 액션이 삭제뿐이고, 그것은 게이트돼 있다.** 노출/숨김은 **가역적**이라 확인 게이트가 필요 없고 실제로 없다. 삭제 3경로(행·일괄·상세)는 전부 `ConfirmDialog` 를 거친다 — **FEEDBACK-02 가 pass 인 드문 화면**이며, 형제 FS-044 가 재고 이동에 게이트를 못 세워 gap 인 것과 대조된다.
2. **그 대신 권한이 무방비다.** 이 화면은 **`useRouteWritePermissions` 를 쓰지 않는다**(FS-047-EL-038) — 소비처 7곳(`products/{items,categories}` · `products/returns/ReturnDetailPage` · `settings/{api-keys,languages,oauth,site}`)에 들어 있지 않다. **read 전용 역할이 삭제 버튼을 보고 누른다.** BE-047 §7.7 이 `operator` 의 삭제를 403 으로 막는데 **그 판정을 표현할 프론트 층이 없다.** EXC-03 이 이 화면 고유의 P0 gap 이다.
3. **개인정보·UGC 가 한 화면에 모인다.** 목록 한 줄에 작성자 닉네임 · 별점 · 본문 · 작성일이 있고, 상세 미리보기에는 고객이 찍은 사진까지 나온다. **그럼에도 이 화면에 CSV·엑셀 내보내기가 없다**(grep 0건) — `shared/download.ts:19-22` `escapeCell` 의 수식 주입 위험(정규식이 `/["\r\n,]/` 뿐이라 `=`/`+`/`-`/`@` 선두를 막지 못한다)은 **이 화면에 표면이 없어 걸리지 않는다.** 내보내기가 붙으면 UGC 가 그대로 들어가므로 그때 최우선 재판정 대상이다(§4.3).

**동시에 이 화면은 공용 프레임워크의 계약을 절반만 소비한다** — 읽기 쪽(`useListState` · `useCrudList` · `CrudListShell` · `isAbort`)은 전부 쓰는데 **실패 쪽(`isNotFound`·`isConflict`·`isForbidden`·`isUnprocessable`·`referenceOf`)은 하나도 쓰지 않는다**(import grep 0건). BE-047 §7.9 가 그 표를 갖고 있으며, 아래 STATE-02 · EXC-04 gap 이 그 한 뿌리에서 나온다.

### 4.3 UGC · 개인정보 노출

리뷰는 **고객이 만든 콘텐츠**이자 **개인정보**이며, 이 화면은 그것을 관리자에게 보이고 다시 서버로 되돌려 보낸다. quality-bar 는 이 축을 다루지 않으므로 여기서 정한다.

| 요구 | 현재 상태 |
|---|---|
| **관리자가 고객의 말을 고칠 수 없다** | **구조적으로 미보장.** `toReviewInput`(`types.ts:103-117`)이 `content`·`rating`·`author`·`createdAt` 을 요청 바디에 실어 **전체 치환 PUT** 으로 보낸다(`ReviewDetailPage.tsx:132` · `ReviewListPage.tsx:154`). 조작된 클라이언트가 5점 리뷰의 본문을 대신 쓸 수 있고 **작성자·작성일이 그대로 붙어 위조를 구분할 수 없다.** **BE-047 §7.1 【보안 판정】 — 최우선** |
| **평점을 조작할 수 없다** | **구조적으로 미보장** — `rating` 이 요청 바디에 있다. `averageRating` 소비자가 0이라 지금은 드러나지 않으나, 상품별 평점이 붙는 순간 **표시 가격에 준하는 상거래 정보**가 된다(BE-047 §7.1 #2 · §7.5) |
| **리뷰 이미지가 신뢰된 호스트에서만 온다** | **미보장.** `imageUrls` 가 요청 바디에 있고(`types.ts:110`) 프론트에 스킴·호스트 검증이 **0건**이다. 그 값이 `ImageThumb`(`ImageThumb.tsx:52`)를 통해 **실제 `<img src>` 네트워크 요청**이 되며 **같은 배열이 고객 상품 페이지로 나간다** — 외부 호스트를 심으면 리뷰를 본 고객의 IP·UA 가 샌다(BE-047 §7.10 #13) |
| **UGC 가 HTML/스크립트로 해석되지 않는다** | **충족.** React 의 기본 이스케이프에 의존한다 — `dangerouslySetInnerHTML` grep **0건**. 본문은 `<p>` 텍스트로만 렌더되고(`ReviewPreview.tsx:149`) 링크로 자동 변환되지 않는다. `overflowWrap: 'anywhere'`(`:70`)가 공백 없는 긴 문자열의 레이아웃 파괴도 막는다 |
| **신고 기록이 지워지지 않는다** | **구조적으로 미보장.** `reported`·`reportReason` 이 요청 바디에 있어(`types.ts:113-114`) 관리자 저장이 그것을 덮을 수 있다 — **신고된 리뷰를 노출로 되돌리면서 `reported: false` 를 함께 보내면 신고가 있었다는 기록 자체가 사라진다**(BE-047 §7.1 #4) |
| **신고를 정당하게 해소할 수 있다** | **미충족 — 수단이 없다.** 화면이 `reported` 를 읽기만 하고(FS-047-EL-023) 내리는 컨트롤이 0건이다. '검토했고 문제없음'을 기록할 자리가 없어 배너가 리뷰를 지울 때까지 남는다. **정당한 경로로는 못 내리는데 부당한 경로(위 항목)로는 내릴 수 있다 — 정확히 거꾸로다**(BE-047 §7.3) |
| **작성자 실명이 응답에 남지 않는다** | **미검증(픽스처 한정).** 픽스처가 `'민**'`(`data-source.ts:14`)처럼 이미 마스킹된 값을 담고 타입 주석이 그것을 계약으로 선언한다(`types.ts:11`). **서버가 마스킹해야 한다** — 프론트가 가리면 응답 본문에 평문이 남는다. **리뷰는 회원 목록보다 위험하다** — `content` 와 함께 와서 '누가 무엇을 샀고 어떻게 느꼈는가'가 한 객체에 모인다(BE-047 §7.2) |

| **목록이 필요 이상의 개인정보를 받지 않는다** | **미충족.** 목록 응답에 `content` 전문 · `imageUrls` · `reportReason` 이 전부 실린다(`Review` 타입을 목록·상세가 공유). 목록은 ellipsis 미리보기·배지 판정에만 쓴다 — **쓰지 않는 개인정보를 건수만큼 받는다.** BE-047 §7.8 이 `ReviewSummary` 분리(`contentPreview` + `photoCount`)를 판정한다 |
| **UGC 가 CSV 수식 주입 벡터가 되지 않는다** | **N/A — 표면이 없다.** **이 화면에 CSV·엑셀 내보내기가 없다**(`download`·`toCsv`·`escapeCell` grep = **0건**). `shared/download.ts:19-22` `escapeCell` 의 정규식이 `/["\r\n,]/` 뿐이라 `=`/`+`/`-`/`@` 선두를 막지 못하는 것은 사실이나, **그 표면이 이 화면에 존재하지 않으므로 판정 대상이 아니다.** **내보내기가 붙으면 이 화면이 앱에서 가장 위험한 소비처가 된다** — 리뷰 본문은 고객이 임의로 쓴 문자열이라 `=cmd\|'/c calc'!A1` 같은 값이 정문으로 들어온다. 도입 시 **최우선 재판정** |
| **삭제가 되돌릴 수 있다** | **미충족(의도된 것).** 하드 삭제이며 복원 경로가 없다(`crud.ts:143`). 다이얼로그가 '되돌릴 수 없습니다'라고 **정확히 고지한다.** 완화는 BE-047 §7.7 의 `operator` 삭제 금지뿐인데 **프론트에 그 게이팅이 없다**(EXC-03 gap). 개인정보 파기 요구는 삭제 버튼이 아니라 보존정책 배치의 일이다(BE-047 §7.4) |

## 6. 측정 도구 · 재현 스위치

> **판정 기준일 2026-07-17 · `HEAD = a5c2639`. E2E 미실행 — 이 문서의 모든 판정 근거는 코드 대조다.** 아래 스위치는 판정을 재현·검증할 때 쓰는 수단이며, 이 문서가 그것을 실행해 얻은 결과를 적은 것이 아니다.

| EXC-10 (일괄 부분 실패) | **`?fail=delete` 는 전 건을 실패시켜 부분 실패를 만들지 못한다.** 부분 실패는 **다른 탭에서 일부를 먼저 지운 뒤**(그 id 가 409 를 받는다) 재현하거나 픽스처를 조작해야 한다 |

**`?fail=save` 의 함정**: 이 화면은 **상세 저장과 인라인 토글이 같은 `save` op** 을 쓴다(`crud.ts:120` — 둘 다 `adapter.update` 를 통과). `?fail=save` 를 켜면 **목록의 토글도 전부 실패**하므로, 상세 저장만 격리해 보려면 상세로 이동한 뒤 그 파라미터를 붙여야 한다.

**`violations` 재현 불가**: `?status=save:422` 는 `violations` 가 **비어 있는** `HttpError` 를 던지고(`dev.ts:84` — `new HttpError(status, STATUS_MESSAGE[status])`), **이 어댑터는 `violations` 를 실은 오류를 던지는 코드가 0건**이다(형제 `returns/data-source.ts:158-163` 과 다르다 — 리뷰에는 도메인 검증이 없다). 따라서 **EXC-07 의 '필드 매핑'은 현재 재현할 대상 자체가 없다** — 백엔드가 400 `error.fields` 를 내려주는 시점에 어댑터 변환(BE-047 §6.1)과 화면 배선이 함께 붙어야 검증 가능하다.

**LATENCY_MS 주의**: `LATENCY_MS = 400`(`dev.ts:12`)은 **개발용 지연이며 성능 예산이 아니다**(§4.1). 실제 네트워크는 0건이므로 이 값으로 응답 시간을 판정하면 안 된다. 다만 **인라인 토글의 체감을 재현하는 데는 유용하다** — 400ms 는 낙관적 갱신 부재(EXC-14)가 실제로 어떻게 느껴지는지 보여준다.

**그 밖의 도구**: `grep`(TOKEN-01 · COMP-01 · A11Y-11 · A11Y-12 · ERP-13 · EXC-03 · EXC-04 · EXC-06 · EXC-08 · EXC-20 판정 — **이 화면의 gap 상당수가 'import 0건'이라 grep 만으로 판정된다**) · RTL(A11Y-11 의 describedby 부재 · `ToggleSwitch` 의 `aria-checked`/`aria-busy` · `Empty` 3분기) · **`reviews.test.ts`**(순수 규칙만 고정한다 — `starText`·`isPhotoReview`·`averageRating`·`filterByRating`·`searchReviews`·`sortReviews`·`toReviewInput`. **어댑터 경계 테스트가 없고**(형제 `returns.test.ts:185-262` 와 다르다) **컴포넌트 테스트도 없다** — 이 화면의 회귀 방어선은 순수 함수뿐이다).

**`http-error.test.ts:92` 의 flaky 주의**: `expect(codes.size).toBe(50)` 이 birthday 충돌로 **~2.6% 확률로 실패한다**(`createErrorReference()` 의 난수 공간 46,656 — 같은 ms 안에서 50개를 뽑을 때 1−e^(−1225/46656)). **이 화면과 무관하나** 같은 배치에서 CI 가 붉어지면 이것을 먼저 의심할 것 — 실재하는 flaky 이며 고치는 것은 이 문서의 경계 밖이다.
