# BE-038. 신청서 관리 백엔드 기능 명세

## 1. 개요

| 항목 | 내용 |
|---|---|
| 대상 화면 | FS-038 신청서 관리 (`/reservations/applications` · `/reservations/applications/:id`) |
| 범위 | 신청서 목록 조회, 신청서 상세 조회, 신청서 처리 저장(상태 전이 + 관리자 메모 + 처리 이력) |
| 범위 밖 | **신청서 생성** — 신청은 고객 채널(웹 폼)이 만든다. 관리자 화면에 등록 UI 가 없고 어댑터의 `create` 를 호출하는 경로가 0건이며 `// TODO(backend)` 심에도 POST 가 없다(§7.5). BE-003 §1 의 '회원 생성 — 고객은 회원가입으로만 유입된다' 와 같은 판정이다. **신청서 삭제** — 감사 성격상 삭제하지 않는다. 심에 DELETE 가 없고 화면에 삭제 경로가 없다(§7.5). **고객 신청 폼(접수) 계약** — 이 문서는 관리자 화면의 계약만 정한다. **보존·파기 정책** — §7.7 이관 |
| 프론트 어댑터 | `apps/admin/src/pages/reservations/applications/data-source.ts` |
| 도메인 타입 | `apps/admin/src/pages/reservations/applications/types.ts` |
| 검증 정본 | **없다** — 이 화면에 `validation.ts`(zod 스키마)가 존재하지 않는다. 클라이언트 검증은 select 후보 축소(`statusChoices`)와 브라우저 `maxLength=500` 뿐이다(§7.4). **따라서 이 계약의 검증은 전부 서버가 정본이다** |

> **에러 봉투·권한 모델 상속**: BE-003 §2·§3 을 그대로 상속한다(BE-010 §2 와 동일 선언). 아래는 신청서 고유 차이만 기술한다.

## 2. 공통 (상속)

- **에러 봉투**: BE-003 §2. 공통 에러코드 동일(`error.code`·`message`·`fields`·`traceId`).
- **권한**: `admin` = 전체. `operator` = 조회 계열만(목록·상세). 처리 저장(EP-03) → **403**. 예약/신청 도메인 읽기 권한 없는 관리자 → 컬렉션 **403** / 개별 **404** 은닉(BE-003 §3.2 · §7.2).
- **CSRF**: 쓰기(PUT)에 `X-CSRF-Token`.
- **타임아웃**: 조회·쓰기 5초 → 504.
- **프론트 역할 분기 없음**(FS-038 §4.1) — 권한 강제는 서버 책임. 프론트는 '처리 저장' 버튼을 권한과 무관하게 렌더하므로 **403 을 반드시 응답으로 돌려야 한다**(§7.6).

## 4. 엔드포인트 명세

### BE-038-EP-01 · 신청서 목록 조회
| 항목 | 내용 |
|---|---|
| 근거 (FS) | FS-038-EL-004, EL-005, EL-005.1~.6, EL-006, EL-007, EL-008 |
| 메서드·경로 | `GET /api/reservations/applications` |
| 권한 | `admin`, `operator` |
| 멱등성 | 멱등(GET) |
| 페이징 | **현재 계약에 없다** — 프론트가 전량을 받아 클라이언트에서 거른다(FS-038 §5). §7.3 판정 |

**쿼리**: **없다.** 프론트는 `applicationAdapter.fetchAll(signal)` 을 파라미터 없이 부르고 상태 필터·키워드 검색·정렬을 **전부 클라이언트에서** 수행한다. 서버에 필터를 넘기는 심이 없으므로 이 계약은 쿼리 파라미터를 정의하지 않는다 — **지어내지 않는다**(§7.3 이 이 결정의 비용과 승격 조건을 적는다).

**응답에 개인정보가 포함된다**: `applicantName` · `applicantContact`. §7.2 참조.

**에러**: 401 · 403 · 429 · 500 · 504.

| 항목 | 내용 |
|---|---|
| 근거 (FS) | FS-038-EL-019, EL-022 |
| 메서드·경로 | `PUT /api/reservations/applications/:id` |
| 권한 | **`admin` 만**. `operator` → 403 |
| 멱등성 | 멱등(PUT 전체 치환). **단 `history` 를 클라이언트가 실어 보내므로 지금 형태로는 재요청이 이력을 중복시킬 수 있다 — §7.1 이 이 문제를 없앤다** |

| 검사 | 위반 시 |
|---|---|
| `status` 가 현재 상태에서 **허용된 전이**인가(§3 표) | **422 `INVALID_STATUS_TRANSITION`** |

심에 **DELETE 가 없다.** 화면에 삭제 경로가 없고 어댑터의 `remove` 를 호출하는 경로가 **0건**이다(`ApplicationDetailPage.tsx:4` — '삭제는 없다(감사 성격)'). 신청서는 감사 기록이므로 삭제하지 않는다. **다만 개인정보를 담으므로 보존 기간 만료 시의 파기 경로는 반드시 정의돼야 한다 — 그것은 이 화면의 API 가 아니라 보존 정책의 문제다**(§7.7 이관).

## 7. 핵심 판정

### 7.1 처리 이력의 작성자·시각은 서버가 정본이다 【보안 판정】

**현재 프론트가 하는 일**(`ApplicationDetailPage.tsx:83-97` `appendHistory` · `:147`):

```
at: new Date().toISOString()   // 브라우저 시각
by: PROCESSED_BY               // '관리자' — :48 의 상수. 로그인 계정이 아니다
id: `${application.id}-h${history.length + 1}`
→ history 배열 **전체**를 update 바디에 실어 보낸다
```

**이것을 서버가 신뢰하면 감사 이력이 위조 가능하다.** 처리 이력은 '누가 언제 이 신청을 반려했는가'를 증명하는 기록인데, 지금 계약대로면 클라이언트가 그 세 값을 전부 정한다 — 요청을 조작하면 **다른 사람 이름으로, 임의의 시각에, 임의의 이력을 통째로 심거나 기존 이력을 삭제**할 수 있다. `by` 가 상수 '관리자'라 지금은 **누가 처리했는지 서버도 화면도 모른다**는 점(이력에 실제 담당자가 남지 않는다)이 이 문제의 다른 얼굴이다.

**계약**:

1. **서버는 요청 바디의 `history`·`by`·`at` 을 무시한다.** 파싱조차 하지 않는다(무시 대신 400 으로 거절해도 좋다 — 어느 쪽이든 클라이언트 값이 저장에 도달해서는 안 된다).
2. `status` 가 실제로 바뀐 경우에만 서버가 이력 한 칸을 **자기가 만들어** 덧붙인다: `at` = 서버 시각(ISO 8601, 오프셋 포함) · `by` = **세션 주체의 표시명**(상수 아님) · `note` = 요청의 `adminNote` · `id` = 서버 채번(길이 기반 채번은 동시 처리 시 충돌한다 — FS-038-EL-022 경합 열).
3. 이력은 **append-only** 다. 어떤 요청도 기존 칸을 수정·삭제할 수 없다.
4. 응답(또는 후속 EP-02)이 서버가 확정한 이력을 돌려주고, 화면은 그것을 그대로 그린다.

**프론트 후속**: `toApplicationInput`(`types.ts:135-146`)에서 `history` 를 빼고 `appendHistory` 를 삭제한다 — 이력 생성은 서버 책임이므로 클라이언트에 그 코드가 남아 있을 이유가 없다. 이 변경 뒤에는 PUT 재요청이 이력을 중복시키지 않으므로 **EP-03 이 진짜로 멱등해진다**.

### 7.2 신청자 개인정보 — 403 vs 404 은닉 + 신청 내용 정제 【보안 판정】

신청서는 `applicantName`(이름) · `applicantContact`(연락처/이메일) · `fields[]`(고객이 자유 입력한 상담 내용·예산·회사 규모)를 담는다. **BE-003 §3.2 의 원칙을 그대로 적용한다**:

1. **컬렉션(`GET /api/reservations/applications`)의 존재는 비밀이 아니다** → 권한 부족 시 **403**.
2. **개별 신청서(`/:id`)의 존재 자체가 개인정보다** — 'APP-20260716-001 이 존재한다'는 사실만으로 '그 날 그 회사가 견적을 문의했다'가 새어 나간다. 예약/신청 도메인 **읽기 권한이 없는** 주체에게는 **404 `APPLICATION_NOT_FOUND`** 로 은닉한다. 반대로 **읽기 권한은 있고 쓰기만 없는** `operator` 가 EP-03 에서 거절될 때는 **403** 이다 — 이미 존재를 아는 주체에게 존재를 숨기는 것은 무의미하고, 정당한 운영자가 권한 문제를 인지하지 못하게 만든다.
3. **id 열거 방지**: `:id` 는 순번(`app-1`·`app-2`)이 아니라 추측 불가능한 값이어야 한다. 픽스처의 `app-N` 형식을 그대로 채택하면 권한 경계를 뚫지 않고도 전체 건수와 접수 순서가 노출된다.

**저장형 XSS**: `fields[].value` 와 `applicantName` 은 **고객이 웹 폼에 자유 입력한 값**이다(관리자 입력이 고객 화면에 나가는 BE-010 §7.2 와 방향이 반대다 — 여기서는 **고객 입력이 관리자 화면에 들어온다**). 관리자 화면은 이 값들을 텍스트로만 렌더하므로(FS-038-EL-013.2, React 가 이스케이프한다) 현재 경로는 안전하다. 그러나 계약은 **저장 시점**에 정한다: 서버는 접수 시 `fields[].value`·`label`·`applicantName`·`applicantContact` 에서 실행 가능한 마크업·`javascript:` 스킴·이벤트 핸들러를 제거하고, '저장된 신청 내용에 실행 가능한 스크립트가 없다'는 관측 동작을 보장한다. 이 값들이 나중에 메일 템플릿·CSV·PDF·고객 회신 화면으로 흐를 때 이스케이프 계층이 React 가 아니게 되기 때문이다.

**연락처 마스킹**: 픽스처는 `010-2222-**33` · `sales@**.co.kr` 처럼 **이미 마스킹된 값**을 담는다(`data-source.ts:16,45`). 이것이 '서버가 마스킹해 내려준다'는 계약인지, 단지 픽스처 편의인지 **코드로는 판정할 수 없다** — 프론트에 마스킹 함수가 없다. **미정으로 남긴다**(§7.7 #5): 서버가 마스킹 정본이라면 처리 담당자가 고객에게 연락할 수 없고(운영 불가), 원문을 내려준다면 목록 조회만으로 전체 연락처가 유출된다. 어느 쪽인지 **정책이 먼저 정해져야** 계약을 쓸 수 있다.

### 7.3 서버가 상태 전이를 강제한다 — 화면의 후보 축소는 UX 편의다

프론트의 전이 방어는 **`statusChoices(from)` 가 select 후보를 좁히는 것 하나**다(`ApplicationDetailPage.tsx:126-129, 241-245`). 순수 판정 함수 `canTransition` 은 **테스트(`applications.test.ts:36-38`)에서만 호출되고 화면 코드 어디에서도 불리지 않는다.** 이 구조에는 두 개의 구멍이 있다:

1. **요청 조작**: select 를 좁히는 것은 DOM 이다. 요청을 직접 만들면 `received → completed`(검토·승인을 건너뛴 완료)나 `rejected → approved`(종료 상태 되살리기)를 보낼 수 있다.
2. **TOCTOU**: 후보는 **로드 시점**의 상태로 계산된다(FS-038-EL-014 경합 열). 상세를 열어 둔 사이 다른 관리자가 `reviewing → rejected` 로 옮겼다면, 내 화면은 여전히 `approved` 를 후보로 그리고 있고 그것을 고르면 **종료 상태에서의 전이**가 서버에 도착한다.

**계약**: 서버는 저장 트랜잭션 안에서 **저장된 현재 상태**를 다시 읽어 §3 표의 전이인지 판정하고, 아니면 **422 `INVALID_STATUS_TRANSITION`** 으로 거절한다. `message` = '이미 처리된 신청서입니다. 최신 상태를 확인해 주세요.' 프론트의 후보 축소는 **UX 편의이고 서버 검사가 정본이다**(BE-010 §7.7 카테고리 참조 무결성과 같은 결).

**페이징·필터를 서버로 올리지 않는 결정**: EP-01 은 쿼리 파라미터가 없고 전량을 내려준다 — 심에 그렇게 적혀 있고 프론트가 그렇게 부른다. 신청서는 **고객이 만드는 무한 증가 컬렉션**이므로 이 계약은 시간이 지나면 반드시 깨진다(응답 크기·첫 렌더 시간·메모리가 신청 수에 비례한다 — NFR-038 §4.1). **지금 지어내지 않는다.** 승격 조건을 남긴다: 프론트에 페이지네이션(IA-04)과 URL list state(IA-13)가 들어오는 배치에서 `status`·`keyword`·`page`·`size` 쿼리와 `total` 을 이 계약에 추가하고, 그때 서버 정렬(`submittedAt` 내림차순)이 정본이 된다.

> **승격 조건 재확인(`HEAD = 4b805ad`) — 절반이 충족됐다.** F3b 가 **URL list state(IA-13)를 이 화면에 배선했다** — `ApplicationListPage.tsx:108` 의 `useListState({ filterDefaults })` 로 `status` 와 `q` 가 이미 **URL 파라미터로 존재한다**(`useListState.ts:26-28,87-99`). 즉 **서버 쿼리로 승격할 때 프론트가 보낼 파라미터 이름이 이미 정해져 있다**(`status`·`q`). **아직 없는 것은 페이지네이션(IA-04)뿐이다** — `useListState` 는 `page` 파라미터와 `clampPage`(`:217-223`)를 이미 갖고 있으나 화면이 쓰지 않는다. **그래도 계약은 지금 바꾸지 않는다**: 심(`data-source.ts:5`)에 쿼리 파라미터가 없고, **프론트가 아직 전량을 받아 클라이언트에서 거르므로**(`:131-134`) 서버 계약을 먼저 정하면 그것은 발명이다. **엔드포인트 발명 0건 원칙 유지.**

### 7.4 전체 치환 PUT 이 접수 원문을 되돌려 보낸다 — 서버가 그 필드들을 거부해야 한다

`toApplicationInput(application)`(`types.ts:135-146`)은 로드한 신청서에서 `id`·`code` 만 빼고 **나머지 전부**를 바디에 담는다. 화면이 실제로 바꾸는 것은 `status` 와 `adminNote` 둘뿐인데, `type`·`applicantName`·`applicantContact`·`submittedAt`·`fields[]` — **고객이 제출한 접수 원문** — 까지 클라이언트를 왕복해 되돌아온다.

이건 두 가지를 뜻한다: ① 관리자가 (또는 조작된 요청이) **접수 원문을 고칠 수 있다** — 신청자 이름이나 신청 내용을 바꿔 놓고 반려할 수 있다는 뜻이고, 그건 감사 기록의 훼손이다. ② 상세를 연 뒤 접수 원문이 서버에서 바뀌었다면(정정 등) 내 저장이 **옛 원문으로 되돌린다**(lost update).

**계약**: 서버는 EP-03 바디에서 `status`·`adminNote` **만** 읽는다. `type`·`applicantName`·`applicantContact`·`submittedAt`·`fields`·`history` 는 무시하거나(권장), 저장된 값과 다르면 **422** 로 거절한다. 어느 쪽이든 **관리자 처리 경로로 접수 원문이 변경돼서는 안 된다.**

**프론트 후속**: `ApplicationInput` 을 `{ status, adminNote }` 로 좁히면 이 문제와 §7.1 이 함께 사라진다. 그때 EP-03 은 `PUT /:id` 보다 `PATCH /api/reservations/applications/:id/processing` 이 정확한 표현이지만, **심에 적힌 것은 PUT** 이므로 이 문서는 PUT 으로 쓴다 — 경로·메서드 변경은 심 변경과 함께 승격한다(§7.7 #6).

### 7.5 생성·삭제 — 범위 밖 (심 없음)

**지어내지 않는다.** 근거 셋이 모두 같은 방향을 가리킨다: ① `data-source.ts:5` 의 유일한 심에 `GET`·`GET·PUT` 만 있다. ② 화면에 등록 버튼도 삭제 버튼도 없다(`ApplicationListPage.tsx` 전수 — 툴바에 검색·필터뿐이고 행 액션은 '상세' 하나다). ③ 어댑터의 `create`/`remove` 를 호출하는 코드가 앱 전체에 0건이다 — `createCrudAdapter` 가 프레임워크 계약상 요구해서 존재할 뿐이고, `data-source.ts:3-4` 가 그 사실을 주석으로 적어 뒀다('build/patch 는 프레임워크 계약상 두되 화면은 fetchAll/fetchOne/update 만 쓴다').

BE-003 §1 이 '회원 생성 — 고객은 회원가입으로만 유입된다'로 같은 판정을 내렸다. **관리자가 신청서를 만들거나 지우는 API 는 이 계약에 존재하지 않는다.**

### 7.6 동시 처리 — 낙관적 동시성 토큰이 없다

`Application` 에 `updatedAt`·`version` 필드가 없고, 앱 전체에서 `If-Match` 는 **회원 메모(BE-004)에만** 존재한다(`shared/api/schema.d.ts` 전수 — 신청서 경로 0건). **`4b805ad` 재확인: F3b 가 `WriteContext`(`crud.ts:30-42`)에 `idempotencyKey` 자리를 만들었으나 `If-Match`/`version` 자리는 만들지 않았다** — 멱등키는 '같은 요청의 재시도'를, 낙관적 동시성은 '다른 사람의 개입'을 다루는 **다른 축**이다. 픽스처 어댑터는 **id 존재 여부**만 검사해 없으면 409 를 던진다(`crud.ts:126-128`) — 이것은 '먼저 삭제된 항목' 을 잡을 뿐 '먼저 **변경**된 항목' 은 잡지 못한다. 그리고 신청서에는 삭제 경로가 없으므로(§7.5) **그 409 는 실질적으로 발화하지 않는다.**

**판정**: 지금 두 관리자가 같은 신청을 동시에 처리하면 **마지막 쓰기가 이긴다** — 먼저 저장한 사람의 메모가 조용히 사라진다. 이를 막는 실질적 장치는 §7.3 의 상태 전이 재판정 하나뿐이고, 그것은 **상태가 바뀌는 경우만** 잡는다(둘 다 메모만 고치면 통과한다).

이 화면에는 충돌을 해소할 UI 가 아예 없다 — 공용 `useCrudForm` 은 409/412 에 충돌 다이얼로그를 띄우지만(`useCrudForm.ts:166-179`), 이 화면은 저수준 `useCrudUpdate` 를 직접 써서 모든 실패를 '저장하지 못했습니다' 배너 하나로 뭉갠다(FS-038-EL-017). **서버가 409 를 올려도 프론트가 그것을 표현하지 못한다.** ⚠ **`4b805ad` 재확인 — 이 판정의 무게가 커졌다**: F3b 이후 `useCrudForm` 위의 화면들은 **어댑터가 409 를 주기만 하면 화면 코드 0줄로** 복구 경로가 열린다. 이 화면만 그 이득을 받지 못한다 — 서버 계약(409)을 지금 정의해도 **이 화면에서는 발현되지 않는다**는 뜻이다(§7.7 #3 과 같은 뿌리).

**계약**: `Application` 에 `updatedAt`(또는 불투명 `version`)을 추가하고 EP-02 응답에 실어 내린 뒤 EP-03 의 `If-Match` 로 되받아, 불일치 시 **409 `CONFLICT`** 로 거절한다. **단 이 계약은 프론트가 충돌 다이얼로그를 갖기 전에는 값이 없다** — 지금 켜면 사용자는 이유를 모른 채 일반 실패 배너만 본다. §7.7 #3 과 **함께** 승격한다.
