# BE-039. 상담 예약 관리 백엔드 기능 명세

## 1. 개요

| 항목 | 내용 |
|---|---|
| 대상 화면 | FS-039 상담 예약 관리 (`/reservations/consultations` · `/new` · `/:id/edit`) |
| 범위 | 상담 예약 목록 조회, 상세 조회, 등록, 수정, 삭제(단건·일괄), 담당자 마스터 조회 |
| 범위 밖 | **일괄 삭제 전용 엔드포인트** — 프론트가 단건 DELETE 를 `settleAll` 로 병렬 호출한다(§7.1). **자원(공간/좌석) 마스터** — `GET /api/reservations/resources` 심은 `_shared/resources.ts:7` 에 있으나 상담 예약은 `resourceId` 를 갖지 않아 소비하지 않는다. 예약(방문)·예약 일정 도메인의 관심사다. **예약(방문) 계약** — 같은 `_shared/booking.ts` 상태 흐름을 공유하지만 별도 화면·별도 문서다. **상담 슬롯 충돌 판정** — **미정**(§7.3). **보존·파기 정책** — §7.8 이관 |
| 프론트 어댑터 | `apps/admin/src/pages/reservations/consultations/data-source.ts` |
| 도메인 타입 | `apps/admin/src/pages/reservations/consultations/types.ts` + 섹션 공용 `pages/reservations/_shared/booking.ts`(상태·전이) |
| 검증 정본 | `apps/admin/src/pages/reservations/consultations/validation.ts` — `consultBookingSchema`(zod/mini). **서버는 이 스키마를 상한이 아니라 하한으로 본다** — 클라이언트 검증은 우회 가능하므로 서버가 같은 규칙을 독립적으로 재검증한다 |

> **에러 봉투·권한 모델 상속**: BE-003 §2·§3 을 그대로 상속한다(BE-010 §2 와 동일 선언). 아래는 상담 예약 고유 차이만 기술한다.

## 2. 공통 (상속)

- **에러 봉투**: BE-003 §2. 공통 에러코드 동일(`error.code`·`message`·`fields`·`traceId`).
- **권한**: `admin` = 전체. `operator` = 조회 계열만(목록·상세·담당자 마스터). 쓰기(등록·수정·삭제) → **403**. 예약/신청 도메인 읽기 권한 없는 관리자 → 컬렉션 **403** / 개별 **404** 은닉(BE-003 §3.2 · §7.2).
- **CSRF**: 쓰기(POST·PUT·DELETE)에 `X-CSRF-Token`.
- **타임아웃**: 조회·쓰기 5초 → 504.
- **프론트 역할 분기 없음**(FS-039 §4.1) — 권한 강제는 서버 책임. 프론트가 등록·수정·삭제 컨트롤을 권한과 무관하게 렌더하므로 **403 을 반드시 응답으로 돌려야 한다**(§7.8 #6).
- **멱등키**: 프론트 `useCrudForm` 이 **제출 시도 단위** 멱등키를 mutationFn 밖(ref)에서 만들어 재시도가 같은 키를 재사용한다(`useCrudForm.ts:112-117`). `TODO(backend): 이 키는 Idempotency-Key 헤더로 나간다` — **EP-02(등록)가 이 헤더를 받아야 한다**(§7.6).

## 3. 데이터 계약 (types.ts 대조)

| 타입 | 필드 | 비고 |
|---|---|---|
| `ConsultChannel` | `'visit' \| 'phone' \| 'video'` | 방문·전화·화상. 표시명은 프론트가 매핑 |
| `BookingStatus` (`_shared/booking.ts`) | `'requested' \| 'confirmed' \| 'visited' \| 'noshow' \| 'cancelled'` | **예약(방문)과 공유하는 상태 집합**이다. `visited` 의 표시명만 도메인별로 다르다('완료' → 상담 예약에서는 '**상담완료**') — 값은 `visited` 그대로다 |
| `ConsultBooking` | `id` · `code`('CSB-YYYYMMDD-NNN') · `customerName` · `customerPhone` · `channel` · `topic` · `preferredDate`('YYYY-MM-DD') · `preferredTime`('HH:mm') · `staffId`(미배정이면 `''`) · `status` · `note` | 목록/폼이 **같은 타입**을 쓴다 — 목록 전용 요약 타입이 없다. **`endTime`·소요시간·`resourceId` 가 없다**(§7.3) |
| `ConsultBookingInput` | `Omit<ConsultBooking, 'id' \| 'code'>` | 등록·수정 바디. `id`·`code` 는 서버가 배정해야 한다(§7.4) |

**검증 규칙** (`validation.ts` `consultBookingSchema` — **서버가 독립 재검증**):

| 필드 | 규칙 | 위반 메시지(프론트) |
|---|---|---|
| `customerName` | `trim() !== ''` · `trim().length <= 40` | '고객명을(를) 입력하세요.' / '고객명은(는) 40자를 넘을 수 없습니다.' |
| `customerPhone` | `trim() !== ''` · `trim().length <= 20` — **형식 검증 없음**(§7.8 #2) | '연락처를(을) 입력하세요.' / '연락처은(는) 20자를 넘을 수 없습니다.' |
| `channel` | `enum(['visit','phone','video'])` | — (위반 불가) |
| `topic` | `trim() !== ''` · `trim().length <= 80` | '상담 주제을(를) 입력하세요.' / '상담 주제은(는) 80자를 넘을 수 없습니다.' |
| `preferredDate` | `isRealDate` — 'YYYY-MM-DD' **그리고 달력상 실재**. **과거를 막지 않는다** | '희망 날짜를 YYYY-MM-DD 형식으로 입력하세요.' |
| `preferredTime` | `/^\d{2}:\d{2}$/` — **영업시간·슬롯 단위 미강제** | '희망 시각을 HH:MM 형식으로 입력하세요.' |
| `staffId` | `z.string()` — **아무 문자열이나 통과. 존재하지 않는 담당자 id 도 통과한다**(§7.7) | — |
| `status` | `enum(['requested','confirmed','visited','noshow','cancelled'])` — **전이 규칙은 검사하지 않는다**(select 후보 축소가 대신한다 — §7.5) | — |
| `note` | `trim().length <= 500` | '메모는 500자를 넘을 수 없습니다.' |

**상태 전이 규칙** (`_shared/booking.ts:44-50` `TRANSITIONS` — 프론트 정본, **서버가 재판정해야 한다** §7.5):

| from | 허용 to | 종료 |
|---|---|---|
| `requested` | `confirmed` · `cancelled` | — |
| `confirmed` | `visited` · `noshow` · `cancelled` | — |
| `visited` · `noshow` · `cancelled` | (없음) | O |

> `occupiesSlot(status)`(`booking.ts:72-74`)는 `requested`·`confirmed`·`visited` 를 '자리를 차지하는 상태' 로 본다(취소·노쇼는 비운다). **예약(방문)의 더블부킹 판정이 이 함수를 쓰지만 상담 예약은 쓰지 않는다** — 겹침 판정 자체가 없기 때문이다(§7.3).

## 4. 엔드포인트 명세

### BE-039-EP-01 · 상담 예약 목록 조회
| 항목 | 내용 |
|---|---|
| 근거 (FS) | FS-039-EL-005, EL-006, EL-008, EL-008.4~.10, EL-009, EL-010, EL-011 |
| 메서드·경로 | `GET /api/reservations/consultations` |
| 권한 | `admin`, `operator` |
| 멱등성 | 멱등(GET) |
| 페이징 | **현재 계약에 없다** — 프론트가 전량을 받아 클라이언트에서 거른다(§7.8 #1) |
| 레이트리밋 | 분당 120회 |

**쿼리**: **없다.** 프론트는 `consultBookingAdapter.fetchAll(signal)` 을 파라미터 없이 부르고 상태 필터·키워드 검색·정렬을 전부 클라이언트에서 수행한다. **지어내지 않는다.**

**응답 200** — `readonly ConsultBooking[]`. **정렬은 희망일시(`preferredDate` → `preferredTime`) 오름차순**(임박한 것이 위, 같은 시각은 `id` 오름차순)으로 서버가 내려준다 — 프론트도 `sortConsultBookings` 로 다시 정렬하므로 계약상 필수는 아니나, 페이징 도입 시 서버 정렬이 정본이 된다.

**응답에 개인정보가 포함된다**: `customerName` · `customerPhone` · `note`(상담 준비·요청사항). §7.2 참조.

**에러**: 401 · 403 · 429 · 500 · 504.

### BE-039-EP-02 · 상담 예약 등록
| 항목 | 내용 |
|---|---|
| 근거 (FS) | FS-039-EL-026(등록), EL-033 |
| 메서드·경로 | `POST /api/reservations/consultations` |
| 권한 | **`admin` 만**. `operator` → 403 |
| 멱등성 | **비멱등 — 그러나 `Idempotency-Key` 헤더로 중복 생성을 막는다**(§7.6). 프론트가 제출 시도 단위 키를 mutationFn 밖에서 만들어 재시도에 재사용한다 |
| 레이트리밋 | 분당 30회 |

**바디**(`ConsultBookingInput`): `customerName` · `customerPhone` · `channel` · `topic` · `preferredDate` · `preferredTime` · `staffId` · `status` · `note`. 문자열은 프론트가 `trim()` 해서 보낸다(`toInput`) — **서버도 독립적으로 정규화·재검증한다**(§3 표).

**서버가 배정하는 것** (§7.4): `id` · `code`. 클라이언트가 보낸 값이 있어도 무시한다.

**등록 시 상태 제약**: 프론트는 등록 모드에서 후보를 '요청·확정·취소'로 좁힌다(`statusChoices('requested')`). 서버는 **신규 등록의 초기 상태를 `requested`·`confirmed` 로만 허용**하고 종료 상태(`visited`·`noshow`·`cancelled`)로의 직접 생성은 **422** 로 거절한다 — 존재한 적 없는 상담을 '노쇼' 로 만들 수 없다. (프론트가 `cancelled` 를 후보에 넣는 것은 `statusChoices` 가 전이 표를 그대로 쓰기 때문이며, 등록 맥락에서는 의미가 없다 — §7.8 #7.)

**응답 201** — 생성된 `ConsultBooking`. 프론트 `create(...): Promise<void>` 로 반환값을 쓰지 않고 목록을 무효화한다.

**에러**: 400 `VALIDATION_FAILED`(`error.fields`: `customerName`·`customerPhone`·`topic`·`preferredDate`·`preferredTime`·`note`) · 401 · 403 `FORBIDDEN` · 403 `CSRF_TOKEN_INVALID` · 422 `STAFF_NOT_FOUND`(존재하지 않는 `staffId` — §7.7) · 422 `INVALID_INITIAL_STATUS` · 429 · 500 · 504.

### BE-039-EP-03 · 상담 예약 상세 조회
| 항목 | 내용 |
|---|---|
| 근거 (FS) | FS-039-EL-023, EL-027, EL-028 + 폼 필드 전체(EL-016~EL-024) |
| 메서드·경로 | `GET /api/reservations/consultations/:id` |
| 권한 | `admin`, `operator`. 예약/신청 도메인 읽기 권한 없음 → **404 은닉**(§7.2) |
| 멱등성 | 멱등(GET) |
| 레이트리밋 | 분당 120회 |

**응답 200** — `ConsultBooking`. 수정 폼이 이 응답으로 필드를 채우고(`toValues`), **상태 전이 후보의 기준(`originalStatus`)도 이 응답의 `status` 다**(FS-039-EL-023).

**404 처리**: 어댑터(`createCrudAdapter.fetchOne`)가 `HttpError(404)` 를 던지고, `useCrudForm` 이 `isNotFound` 로 갈라 `loadFailure = 'not-found'` 를 만들며, `FormPageShell` 이 **재시도 없는** '찾을 수 없습니다 + 목록으로' 를 그린다(FS-039-EL-028). **이 화면은 404 와 5xx 를 제대로 가른다**(신청서와 다른 점).

**요청 횟수 주의**: 폼은 `useCrudForm` 내부 상세와 `useCrudItem`(상태 후보용)을 **둘 다** 호출하지만 **쿼리 키가 같아 react-query 가 1건으로 합친다**(`crud.ts:142` `detailKey`) — 서버는 1회만 받는다.

**에러**: 400(id 형식) · 401 · 404 `CONSULTATION_NOT_FOUND` · 429 · 500 · 504.

### BE-039-EP-04 · 상담 예약 수정
| 항목 | 내용 |
|---|---|
| 근거 (FS) | FS-039-EL-026(수정) |
| 메서드·경로 | `PUT /api/reservations/consultations/:id` |
| 권한 | **`admin` 만**. `operator` → 403 |
| 멱등성 | 멱등(PUT 전체 치환) |
| 레이트리밋 | 분당 60회 |

**바디**: `ConsultBookingInput`(EP-02 동일). `code` 는 바디에 없다 — **수정이 `code` 를 바꾸지 않는다**(희망 날짜를 바꿔도 채번 시점의 번호를 유지한다 — §7.4).

**서버가 강제해야 하는 것**:

| 검사 | 위반 시 |
|---|---|
| `status` 가 **저장된 현재 상태**에서 허용된 전이인가(§3 표) | **422 `INVALID_STATUS_TRANSITION`** (§7.5) |
| `staffId` 가 실재하는 담당자인가(빈 문자열 = 미배정은 허용) | **422 `STAFF_NOT_FOUND`** (§7.7) |
| 낙관적 동시성 — 다른 관리자가 먼저 변경했는가 | **409 `CONFLICT`** (§7.6) |
| §3 표의 필드 규칙 | 400 `VALIDATION_FAILED` + `error.fields` |

**응답 200/204**. **프론트가 409/412 를 전용 충돌 다이얼로그로 처리한다**(FS-039-EL-030 — 입력 보존 + '최신 불러오기'/'닫기', 성공 토스트·이동 없음) — **서버가 409 를 올리면 프론트가 그것을 표현할 수 있다**(신청서와 다른 점, §7.6).

**에러**: 400 · 401 · 403 · 403 CSRF · 404 `CONSULTATION_NOT_FOUND` · 409 `CONFLICT` · 422 `INVALID_STATUS_TRANSITION` · 422 `STAFF_NOT_FOUND` · 429 · 500 · 504.

### BE-039-EP-05 · 상담 예약 삭제 (단건·일괄 공용)
| 항목 | 내용 |
|---|---|
| 근거 (FS) | FS-039-EL-008.11, EL-012, EL-007.2, EL-013 |
| 메서드·경로 | `DELETE /api/reservations/consultations/:id` |
| 권한 | **`admin` 만**. `operator` → 403 |
| 멱등성 | **멱등**. 이미 삭제 재요청 → 204 |
| 레이트리밋 | 분당 60회(일괄이 항목당 개별 요청 — §7.1) |

**응답 204**. 일괄 삭제는 프론트가 `settleAll` 로 항목마다 병렬 DELETE 하고 부분 실패를 건수로 집계한다(§7.1).

**멱등성 주의 — 픽스처와 계약이 갈린다**: 픽스처 어댑터는 없는 id 에 **409('이미 삭제된 항목입니다.')를 던진다**(`crud.ts:82-84`). 이는 EXC-04 의 '유령 삭제 성공' 을 막으려는 **의도된 안전 기본값**이지만, 실제 서버 계약은 **멱등(이미 삭제 → 204)** 이 옳다 — BE-010-EP-07 과 같은 판정이다. 그렇지 않으면 일괄 삭제 재시도가 이미 성공한 항목에서 실패로 집계된다. **404 는 존재한 적 없는 id 에만** 준다. 이 차이는 어댑터를 실제 HTTP 로 바꿀 때 해소된다(§7.8 #5).

**에러**: 400 · 401 · 403 · 403 CSRF · 404 `CONSULTATION_NOT_FOUND`(존재한 적 없는 id만) · 429 · 500 · 504.

### BE-039-EP-06 · 담당자 마스터 조회
| 항목 | 내용 |
|---|---|
| 근거 (FS) | FS-039-EL-019(담당 select), FS-039-EL-008.9(담당 셀) |
| 메서드·경로 | `GET /api/reservations/staff` |
| 권한 | `admin`, `operator` |
| 멱등성 | 멱등(GET) |
| 페이징 | 없음 — 전량 |

**심 근거**: `_shared/resources.ts:7` `// TODO(backend): GET /api/reservations/resources · GET /api/reservations/staff`. 이 화면은 **`/staff` 만** 소비한다 — 상담 예약에 `resourceId` 가 없어 `/resources` 는 쓰지 않는다(§1 범위 밖).

**응답 200** — `readonly { id, name, role }[]`. 예약(방문)·상담 예약이 같은 목록을 쓴다.

> **⚠ 프론트에 어댑터가 없다.** `listStaff()`/`staffName()` 은 **모듈 상수**(`const STAFF: readonly ReservationStaff[]`)를 반환하는 순수 함수이고, 폼은 `const STAFF = listStaff()` 로 **모듈 로드 시 한 번** 읽는다(`ConsultationBookingFormPage.tsx:30`). 즉 이 엔드포인트를 **부르는 코드가 아직 없다** — 로딩·실패 상태도, 재조회도 없다. 백엔드 연결 시 이 상수를 쿼리로 바꾸는 프론트 작업이 선행돼야 하며, 그때 '마스터 조회 실패 시 담당 select 를 어떻게 그릴지'(빈 목록? 배너?)가 새로 정해져야 한다 — §7.8 #3.

**에러**: 401 · 403 · 429 · 500 · 504.

## 7. 핵심 판정

### 7.1 일괄 삭제 — 전용 엔드포인트를 만들지 않는다

프론트가 `settleAll` 로 단건 DELETE(EP-05)를 병렬 호출하고 부분 실패를 건수로 집계한다(`useCrudList.tsx:132-148`). 단건이 멱등이므로 부분 실패 후 재시도가 안전하다. BE-010 §7.1 과 같은 근거로 전용 벌크 계약을 두지 않는다.

**다만 이 결정에는 알려진 비용이 있다**: ① 선택 건수만큼 요청이 나가므로 레이트리밋(분당 60)이 곧 일괄 삭제의 실질 상한이다 — 프론트에 건수 상한이 없어(FS-039 §7 #15) 전체선택 후 삭제하면 429 가 부분 실패로 집계될 수 있다. ② 실패한 것만 골라 재시도할 수 없어 전량이 재실행된다(멱등이라 안전하지만 낭비다). 벌크 계약으로 승격할지는 실제 건수 분포를 보고 정한다 — §7.8 #4.

### 7.2 고객 개인정보 — 403 vs 404 은닉 + 상담 내용 정제 【보안 판정】

상담 예약은 `customerName`(이름) · `customerPhone`(**연락처 — 상담사가 실제로 거는 번호**) · `topic`(상담 분야) · `note`(상담 준비·요청사항)를 담는다. **BE-003 §3.2 의 원칙을 그대로 적용한다**:

1. **컬렉션(`GET /api/reservations/consultations`)의 존재는 비밀이 아니다** → 권한 부족 시 **403**.
2. **개별 상담 예약(`/:id`)의 존재 자체가 개인정보다** — 'CSB-20260717-001 이 존재한다'는 사실만으로 '그 사람이 그 날 그 주제로 상담을 잡았다'가 새어 나간다(`topic` 이 'ERP 도입 컨설팅'·'매장 창업 상담' 처럼 **사업 계획을 드러낸다**). 예약/신청 도메인 **읽기 권한이 없는** 주체에게는 **404 `CONSULTATION_NOT_FOUND`** 로 은닉한다. **읽기 권한은 있고 쓰기만 없는** `operator` 가 EP-04/EP-05 에서 거절될 때는 **403** 이다 — 이미 존재를 아는 주체에게 존재를 숨기는 것은 무의미하다.
3. **id 열거 방지**: `:id` 는 순번(`csb-1`·`csb-2`)이 아니라 추측 불가능한 값이어야 한다. 픽스처의 `csb-N` 을 그대로 채택하면 권한 경계를 뚫지 않고도 전체 건수가 노출된다. **`code`(CSB-YYYYMMDD-**NNN**)는 날짜별 순번을 이미 드러내므로 라우트 키로 쓰면 안 된다** — 지금은 `id` 가 라우트 키이고 `code` 는 표시용이다. 그 분리를 유지한다.

**저장형 XSS**: `topic` · `note` · `customerName` 은 **관리자가 자유 입력**하는 값이다. 관리자 화면은 텍스트로만 렌더하므로(React 이스케이프) 현재 경로는 안전하다. 그러나 상담 예약은 **고객 알림(SMS·메일 확정 안내)으로 흘러나갈 가능성이 큰 데이터**다 — 그 경로에서 이스케이프 계층은 React 가 아니다. BE-010 §7.2 와 같은 판정: 서버가 저장 시 허용 태그 화이트리스트 밖의 마크업·`javascript:` 스킴·이벤트 핸들러를 제거하고, 계약은 '저장된 값에 실행 가능한 스크립트가 없다'는 관측 동작만 정한다.

**연락처 마스킹**: 픽스처는 `010-7777-**88` 처럼 **마스킹된 값**을 담는다(`data-source.ts:14` 등). 그런데 **이 화면은 그 필드를 관리자가 직접 입력·수정한다**(FS-039-EL-017, placeholder '예: 010-1234-5678' — 마스킹되지 않은 형식을 요구한다). **입력은 원문, 표시는 마스킹**이라면 저장·조회 계약이 비대칭이어야 하는데 그 규칙이 어디에도 없다. **미정으로 남긴다**(§7.8 #2): 상담사가 전화를 걸어야 하므로 원문이 필요하고, 목록 조회만으로 전체 연락처가 노출되면 안 된다 — **정책이 먼저 정해져야** 계약을 쓸 수 있다(BE-038 §7.2 와 같은 미결).

### 7.3 상담 슬롯 충돌 — **소요시간 계약이 선행돼야 한다 (미정)** 【판정 보류】

**사실**: 상담 예약에는 시간 겹침 판정이 **없다**. 같은 담당자·같은 시각에 상담을 몇 건이든 만들 수 있고 경고도 차단도 없다(FS-039-EL-035).

**왜 예약(방문)의 판정을 재사용할 수 없는가**: `_shared/reservation.ts` 의 `findConflicts(list, candidate)` 는 ① `candidate.resourceId` 로 같은 자리를 찾고 ② `toMinutes(startTime)`~`toMinutes(endTime)` **구간**을 만들어 `rangesOverlap` 으로 겹침을 본다. `ConsultBooking` 에는 **`resourceId` 도 `endTime` 도 없다** — `preferredTime` 이라는 **한 점**뿐이다. 상담 예약 모듈은 그 함수를 import 조차 하지 않는다(코드 확인). **점은 겹치지 않는다 — 구간이 있어야 겹침이 정의된다.**

**그러므로 이 계약은 충돌 판정을 정의하지 않는다. 지어내지 않는다.** 서버가 무엇을 강제할지 답하려면 **도메인 결정이 먼저** 나와야 한다:

| 미정 질문 | 선택지 | 영향 |
|---|---|---|
| 소요시간을 어떻게 아는가 | (a) 상담유형별 기본값(방문 60분·전화 20분·화상 40분?) (b) `endTime` 을 입력받는다 (c) 담당자·상담유형 조합의 정책 테이블 | (a)는 데이터 모델 무변경 + 서버만 안다 / (b)는 `ConsultBooking` 에 필드 추가 + 폼 변경 + FS 개정 |
| 충돌 축은 무엇인가 | (a) 담당자(`staffId`) (b) 자원(공간) (c) 둘 다 | 방문상담은 공간이 필요하고 전화·화상은 아니다 — **상담유형마다 축이 다를 수 있다** |

**임시 판정**: 위가 정해지기 전까지 **서버는 상담 슬롯 충돌을 검사하지 않는다** — 이것은 '검사하지 않기로 결정했다'가 아니라 **'검사 대상을 정의할 수 없다'** 는 뜻이다. 정의되면 `422 CONSULTATION_SLOT_CONFLICT`(차단) 또는 응답 경고 필드(허용) 중 하나로 승격하고, 그때 프론트도 폼에 충돌 경고 표면을 새로 만들어야 한다(지금은 그 표면이 없다). **§7.8 #1 이관 — A01 도메인 결정이 선행.**

### 7.4 `id`·`code` 는 서버가 채번한다 — 클라이언트 채번은 재사용·충돌한다

**현재 프론트가 하는 일**(`data-source.ts:66-80`):

```
code = 'CSB-' + preferredDate.replaceAll('-','') + '-' + String(같은 날 기존 건수 + 1).padStart(3,'0')
id   = 'csb-' + (++seq)   // 모듈 전역 카운터
```

두 결함이 있다: ① **번호 재사용** — '같은 희망날짜의 **현재** 건수 + 1' 이므로 그 날의 상담을 하나 지우고 새로 만들면 **지워진 번호가 되살아난다**. 상담예약번호는 고객·상담사가 문의 시 대는 식별자라 재사용은 곧 오조회다. ② **동시 등록 충돌** — 두 관리자가 같은 날짜로 동시에 등록하면 둘 다 같은 건수를 읽어 **같은 번호**를 만든다.

**계약**: `id`·`code` 를 **서버가 배정**한다. `code` 는 **날짜별 단조 증가 시퀀스**로 채번하고 **삭제된 번호를 재사용하지 않는다**(그 날 3건을 만들고 2번을 지워도 다음은 4번이다). 클라이언트가 `id`/`code` 를 보내도 무시한다. `code` 는 유니크 제약을 건다.

**수정(EP-04)은 `code` 를 바꾸지 않는다** — 희망 날짜를 옮겨도 채번 시점의 번호를 유지한다(고객에게 이미 안내된 번호다). 지금 프론트도 그렇다(`ConsultBookingInput` 에 `code` 가 없고 `patch` 가 기존 항목을 스프레드한다) — 계약이 그 동작을 확정한다.

### 7.5 서버가 상태 전이를 강제한다 — 화면의 후보 축소는 UX 편의다

프론트의 전이 방어는 **`statusChoices(originalStatus)` 가 select 후보를 좁히는 것 하나**다(`ConsultationBookingFormPage.tsx:112-114, 269-273`). 순수 판정 함수 `canTransition`(`_shared/booking.ts:57`)은 **테스트에서만 호출되고 화면 코드 어디에서도 불리지 않는다.** zod 스키마도 `status` 를 **enum 으로만** 검사하고 전이는 보지 않는다(`validation.ts:21`). 구멍은 둘:

1. **요청 조작**: select 를 좁히는 것은 DOM 이다. 요청을 직접 만들면 `cancelled → confirmed`(취소된 상담 되살리기)나 `requested → visited`(확정 없이 상담완료)를 보낼 수 있다.
2. **TOCTOU**: 후보는 **로드 시점** 상태(`useCrudItem` 의 캐시값)로 계산된다. 폼을 열어 둔 사이 다른 관리자가 `confirmed → cancelled` 로 옮겼다면, 내 화면은 여전히 `visited` 를 후보로 그리고 있고 그것을 고르면 **종료 상태에서의 전이**가 서버에 도착한다.

**계약**: 서버는 저장 트랜잭션 안에서 **저장된 현재 상태**를 다시 읽어 §3 표의 전이인지 판정하고, 아니면 **422 `INVALID_STATUS_TRANSITION`** 으로 거절한다. `message` = '이미 처리된 상담 예약입니다. 최신 상태를 확인해 주세요.' 프론트의 후보 축소는 **UX 편의이고 서버 검사가 정본이다**.

**등록(EP-02)의 초기 상태도 같은 이유로 강제한다** — `requested`·`confirmed` 만 허용하고 종료 상태로의 직접 생성은 422 다.

### 7.6 동시성 — 토큰은 없지만 **충돌 UI 는 있다**

`ConsultBooking` 에 `updatedAt`·`version` 이 없고, 앱 전체에서 `If-Match` 는 회원 메모(BE-004)에만 존재한다(`shared/api/schema.d.ts` — 상담 예약 경로 0건). 그러니 **'다른 관리자가 먼저 **변경**했다'를 지금 감지할 수단이 없다** — 픽스처가 잡는 것은 '먼저 **삭제**했다'(id 부재 → 409, `crud.ts:71-73`)뿐이다.

**그러나 신청서(BE-038 §7.6)와 결정적으로 다른 점이 있다**: 이 화면은 `useCrudForm` 위에 있어 **409/412 를 전용 충돌 다이얼로그로 처리한다**(`useCrudForm.ts:160-173` → `FormConflictDialog`) — 입력을 보존한 채 '최신 불러오기'/'닫기' 를 묻고, **성공 토스트도 목록 이동도 하지 않는다**(유령 저장 금지). 즉 **서버가 409 를 올리면 프론트가 그것을 제대로 표현할 수 있다.**

**계약**: `ConsultBooking` 에 `updatedAt`(또는 불투명 `version`)을 추가해 EP-03 응답에 싣고 EP-04 의 `If-Match` 로 되받아, 불일치 시 **409 `CONFLICT`** 로 거절한다. **이 계약은 프론트 변경 없이 바로 값을 낸다** — 다이얼로그가 이미 있다. 신청서와 달리 여기서는 **지금 승격할 수 있다**(§7.8 #8).

**등록의 중복 방지는 별도 축이다**: `Idempotency-Key`(§2)가 EP-02 의 중복 생성을 막는다 — 프론트가 이미 제출 시도 단위 키를 만들고 재시도에 재사용하므로(`useCrudForm.ts:112-117`), 서버가 그 헤더를 받아 최초 응답을 재생하면 된다.

### 7.7 담당자 참조 무결성 — 서버가 정본 【정합 판정】

`staffId` 검증이 `z.string()` 이다(`validation.ts:20`) — **존재하지 않는 담당자 id 도 클라이언트 검증을 통과한다.** 그리고 표시 쪽 `staffName(id)` 는 못 찾으면 **'미배정'** 으로 폴백한다(`resources.ts` 마지막 함수) — **'배정 해제'와 '삭제된 담당자에게 배정됨'이 화면에서 구분되지 않는다.** 담당자가 퇴사해 마스터에서 빠지면 그에게 배정된 상담들이 조용히 '미배정' 으로 보이고, **아무도 그 상담을 자기 것으로 여기지 않는다.**

**계약**:
1. EP-02·EP-04 는 `staffId` 가 **실재하는 담당자**인지 저장 트랜잭션 안에서 확인하고, 아니면 **422 `STAFF_NOT_FOUND`** 로 거절한다. 빈 문자열(미배정)은 허용한다.
2. 담당자 삭제(이 문서 범위 밖 — 담당자 마스터 도메인)는 **그에게 배정된 상담이 있으면 막거나**(BE-010 §7.7 의 카테고리 참조 무결성과 같은 결) 배정을 명시적으로 이관해야 한다. 조용한 고아 배정을 만들어서는 안 된다.
3. 프론트는 select 를 마스터로 좁혀 UX 편의를 주지만(FS-039-EL-019), **마스터가 모듈 상수라 서버 변경을 모른다** — 그 자체가 TOCTOU 창이다(§7.8 #3).
