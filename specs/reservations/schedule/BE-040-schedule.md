# BE-040. 예약 일정(달력) 백엔드 기능 명세

## 1. 개요

| 항목 | 내용 |
|---|---|
| 대상 화면 | FS-040 예약 일정 (`/reservations/schedule`) |
| 범위 | **없다 — 이 화면은 전용 엔드포인트를 갖지 않는다.** 예약 데이터를 BE-037 의 엔드포인트에서 읽어 클라이언트가 달력으로 재구성한다. 이 문서는 그 재사용의 **조건과 제약**을 계약으로 못 박는다 |
| 범위 밖 | 예약의 조회·등록·수정·삭제 계약 자체 — BE-037 이 정본이다. 이 화면은 **읽기 전용**이며 쓰기 요청이 하나도 없다(FS-040 §4.1) |
| 프론트 어댑터 | **전용 어댑터 없음.** `_shared/reservation-store.ts` 의 `reservationAdapter` 를 그대로 읽는다 |
| 도메인 타입 | `_shared/reservation.ts`(`Reservation`) · `_shared/booking.ts`(`BookingStatus`) · `schedule/schedule-data.ts`(`Slot` — **클라이언트 전용 타입**) |
| 검증 정본 | **해당 없음** — 입력 폼이 없다 |
| 예약 명세 | BE-037 (`specs/reservations/BE-037-reservations.md`) — 이 문서의 모든 엔드포인트가 거기 정의돼 있다 |

> **에러 봉투·권한 모델 상속**: BE-003 §2·§3 을 그대로 상속한다.
>
> **엔드포인트 상속**: **BE-037 을 그대로 재사용한다.** 아래 §4 는 새 계약이 아니라 '이 화면이 BE-037 의 무엇을 어떤 조건으로 쓰는가'의 선언이다.

### 1.1 재사용의 근거 (코드 대조 · `HEAD = a5c2639` 기준 재확인)

이 문서가 전용 엔드포인트를 정의하지 **않는** 이유는 추정이 아니라 코드다. **PR #22·#24·#26·#28·#30·#32·#34 이후에도 아래 다섯 사실을 전부 다시 확인했고 하나도 바뀌지 않았다** — 재사용 선언은 그대로 유효하다.

| 사실 | 근거 |
|---|---|
| **이 화면에 `// TODO(backend)` 심이 0건이다** | `schedule/ScheduleCalendarPage.tsx` · `schedule/components/CalendarGrid.tsx` · `schedule/schedule-data.ts` 전체에 백엔드 주석이 없다(`a5c2639` 전수 재확인 — 심은 `_shared/reservation-store.ts:5` · `_shared/resources.ts:7` 이 갖는다) |
| 데이터를 예약 어댑터에서 읽는다 | `ScheduleCalendarPage.tsx:22` `import { reservationAdapter } from '../_shared/reservation-store'` · `:137` `useCrudListQuery(RESOURCE, reservationAdapter)` (`RESOURCE = 'reservations'` — `:30`) |
| **예약 관리와 같은 쿼리 키를 공유한다** | `crud.ts:244` `listKey(resource) = [resource, 'list']` → 두 화면 다 `['reservations','list']`. 예약 관리에서 저장·삭제하면 **이 화면도 함께 무효화되어 갱신된다**(FS-040 §4.1) |
| 달력 계산이 전부 클라이언트 순수 함수다 | `_shared/calendar.ts`(날짜 산술·겹침) · `_shared/reservation.ts:135-149`(`reservationsInSlot`) · `schedule-data.ts:30-56`(슬롯 정의·셀 집계) — 전부 서버 왕복 없이 배열을 훑는다 |
| 전용 데이터 소스 파일이 없다 | 섹션의 다른 화면들과 달리 `schedule/` 아래에 `data-source.ts` 가 없다(`applications/data-source.ts` · `consultations/data-source.ts` 와 대조) |

**결론**: 예약 일정은 예약 목록의 **다른 표현(projection)** 이지 다른 리소스가 아니다. 심이 없으므로 이 문서는 **엔드포인트를 지어내지 않는다.** 범위 조회(`?from=&to=`)·슬롯 정의·가용량 집계가 계약상 있어야 할 법해 보여도, 심이 없는 것은 **'심 없음(미정)'** 으로 남기고 §7 에서 판정한다.

## 2. 공통 (상속)

- **에러 봉투**: BE-003 §2. 이 화면은 조회 실패를 status 별로 가르지 않고 한 배너로 표시하므로(FS-040-EL-010) 봉투의 `code` 를 소비하지 않는다 — 그래도 `traceId` 는 서버가 실어야 한다(운영자가 신고할 때 필요하다. 단 이 화면은 참조 코드를 표시하지 않는다 — §7.6 #4).
- **권한**: BE-037 §2·§2.1 을 그대로 상속한다 — 이 화면이 부르는 것은 예약 조회뿐이므로 `admin`·`operator` 둘 다 읽는다. **⚠ 다만 프론트 권한 리소스와 서버 권한 리소스가 어긋난다 — §7.1 판정.**
- **CSRF**: **해당 없음** — 쓰기 요청이 없다.
- **타임아웃**: 조회 5초 → 504.
- **프론트 역할 분기 없음**(FS-040 §4.1).

## 3. 데이터 계약

이 화면은 **새 타입을 서버에 요구하지 않는다.** BE-037 §3 의 `Reservation` 을 그대로 받아 클라이언트가 파생한다.

| 화면이 보이는 값 | 어디서 오는가 | 서버 계약인가 |
|---|---|---|
| 셀의 예약 칩(`'<시작시각> <고객명>'`) | `Reservation.startTime` · `Reservation.customerName` | O — BE-037 §3 |
| 셀의 상태 톤 | `Reservation.status` → `bookingStatusTone` | O — 값만. 톤 매핑은 프론트 소유 |
| 셀의 `booked` | `reservationsInSlot(all, date, startMin, endMin).length` — **클라이언트 집계** | ✕ |
| 셀의 `capacity` | `listResources().length` — **클라이언트 상수의 길이** | ✕ (§7.4) |
| 셀의 `full`(마감) | `booked >= capacity` — **클라이언트 판정** | ✕ (§7.4) |
| 슬롯 정의(09~20시 · 11개) | `DAY_SLOTS` — **클라이언트 상수** (`schedule-data.ts:13-14,30-36`) | ✕ (§7.5) |
| 열(날짜) 구성 | `weekDates(anchor)` / `[anchor]` — **클라이언트 계산**, 브라우저 로컬 TZ | ✕ (§7.5) |
| 패널의 자원명 | `resourceName(resourceId)` — **클라이언트 카탈로그** | △ — BE-037-EP-06 이 심이나 동기 함수라 미연결(BE-037 §7.3) |

> **`Slot`(`{ startMin, endMin, label }`)은 서버에 존재하지 않는 타입이다** — 클라이언트가 영업시간 상수에서 만든다. 서버가 이 개념을 알 필요가 없다는 것이 현재 설계의 전제다(§7.5).

### 6.1 어댑터 본문 요구사항

이 화면에는 **자기 어댑터가 없다** — BE-037 §6.1 의 요구사항이 그대로 적용되며 그것으로 충분하다. 이 화면 고유의 추가 요구는 하나뿐이다:

## 7. 핵심 판정

### 7.1 권한 리소스가 프론트와 서버에서 어긋난다 【보안 판정】

**프론트는 이 화면을 `page:/reservations/schedule` 로 게이팅하고, 서버는 예약 도메인 권한으로 게이팅한다 — 두 축이 독립적이다.**

근거: `shared/permissions/route-resource.ts:36-46` 의 `resourceIdForPath` 는 '자기를 감싸는 **가장 구체적인 잎**이 곧 자기 리소스'라는 규칙을 쓴다. `/reservations/schedule` 은 `nav-config.ts:184-189` 에서 **자기 자신이 잎**이므로(`['예약 일정', '/reservations/schedule']`), `/reservations` 잎보다 길어서 이긴다 → 리소스는 `page:/reservations/schedule` 이다. 예약 목록의 리소스(`page:/reservations`)와 **다른 행**이다.

권한 매트릭스가 잎마다 독립이므로(`permissions/resources.ts` 가 nav-config 에서 파생) **'예약 일정 read = ON, 예약 read = OFF'인 역할이 만들어질 수 있다.** 그 역할은:

1. 사이드바에서 '예약 일정'을 본다(예약은 안 보인다).
2. `/reservations/schedule` 에 들어간다 — `RequirePermission`(`AppShell.tsx:490`)이 `page:/reservations/schedule` 의 read 를 확인하고 **통과시킨다**.
3. 화면이 `GET /api/reservations` 를 부른다.
4. 서버가 예약 도메인 읽기 권한을 보고 **403** 을 준다.
5. 화면은 '예약 일정을 불러오지 못했습니다.' 배너만 보인다 — 운영자는 권한 문제인지 서버 장애인지 알 수 없고(FS-040 §4.1: status 를 구분하지 않는다), **관리자는 권한을 켰다고 믿는데 화면이 영원히 실패한다.**

반대 방향은 더 나쁘다. **'예약 일정 read = OFF, 예약 read = ON'** 이면 프론트는 달력을 막지만 **같은 데이터가 예약 목록에 전부 보인다** — 즉 `page:/reservations/schedule` 을 끄는 것은 **보안 통제가 아니라 메뉴 숨김**이다. 달력이 예약 목록에 없는 정보를 하나도 갖지 않기 때문이다(§1.1).

**판정**: 서버 권한의 단위는 **엔드포인트**(`GET /api/reservations`)이고 프론트 권한의 단위는 **화면**이다. 두 화면이 한 엔드포인트를 공유하는 순간 이 불일치는 구조적이다. 계약은 다음을 확정한다:

1. **서버는 `GET /api/reservations` 하나의 권한만 안다** — 예약 도메인 읽기 권한. 달력용 별도 권한을 만들지 않는다(만들면 같은 데이터에 두 개의 진실이 생긴다).
2. **`page:/reservations/schedule` 의 read 는 보안 경계가 아니라 메뉴 가시성이다.** 이 사실을 권한 관리 화면이 관리자에게 알려야 하며(`page:/reservations` 를 끄지 않고 달력만 끄는 것은 아무것도 숨기지 않는다), 그렇지 않으면 **관리자가 숨겼다고 믿는 정보가 숨겨지지 않는다** — 이것이 이 판정을 '보안'으로 분류하는 이유다.
3. 권한 모델이 두 잎을 **함께 켜고 끄도록** 강제하거나(그룹 리소스의 합집합 규칙과 별개의 '종속' 규칙 필요), 최소한 프론트 가드가 **데이터 리소스**(`page:/reservations`)를 함께 확인해야 한다.

**이관**: 백엔드 명세(권한 모델) · UI 기획 — §7.6 #1.

### 7.2 캘린더 범위 조회 — 심이 없으므로 계약을 정의하지 않는다 【정합 판정】

달력은 최대 7일을 보는데 **예약 전량**을 받는다. `?from=&to=` 가 있어야 할 법하다 — 그러나 **심이 없다.**

- `schedule/**` 전체에 `// TODO(backend)` 0건.
- `CrudAdapter.fetchAll(signal)`(`crud.ts:17`)에 파라미터 자리가 없다 — 범위를 넘길 통로 자체가 없다.
- `_shared/reservation-store.ts:5` 의 심은 `GET /api/reservations` 뿐이고 쿼리스트링을 명시하지 않는다.

**따라서 BE-040-EP-03 은 '심 없음(미정)'이다.** 지어내지 않는다.

**다만 이 부재의 결과는 기록한다** — 도입 결정은 백엔드 명세 이 한다:

| 사실 | 결과 |
|---|---|
| 달력이 전량을 읽는다 | 예약이 N건이면 한 주를 보기 위해 N건을 내려받는다. 예약은 무한 증가하는 트랜잭션 데이터다(BE-037-EP-01 도 같은 문제 — BE-037 §7.8 #1) |
| 뷰·기간 조작이 재조회를 유발하지 않는다 | **이것은 현재 설계의 장점이다** — 6개월 전으로 이동해도 요청이 0건이다. 범위 조회를 도입하면 **기간 이동마다 요청이 나가고**, `staleTime` 30초 캐시가 기간별로 쪼개지며, 예약 목록과 공유하던 쿼리 키(`['reservations','list']`)가 깨져 **두 화면의 자동 동기화가 사라진다**(FS-040 §4.1) |
| 클라이언트 집계가 전량에 의존한다 | 범위 조회를 도입하면 `reservationsInSlot` 이 그 범위 안에서만 정확하다 — 지금은 우연히 전량이라 항상 정확하다 |

**도입 시 계약 조건**(지금 정하지 않고 조건만 남긴다): ① 범위는 **닫힌 구간**(`from` ≤ `date` ≤ `to`)이고 날짜 문자열(`YYYY-MM-DD`)이다 — 시각이 아니다(예약의 `date` 가 날짜 문자열이다). ② `from > to` 면 400. ③ 범위 상한(예: 90일)을 두어 `?from=1900-01-01&to=2100-01-01` 이 전량 덤프가 되지 않게 한다. ④ **예약 목록(BE-037-EP-01)과 다른 엔드포인트를 만들지 않는다** — 같은 경로의 선택적 쿼리로 둔다. 그래야 §7.1 의 권한 단위가 하나로 유지된다.

### 7.3 달력은 개인정보 대량 열람 표면이다 【보안 판정】

**한 화면이 일주일치 모든 고객의 이름과 방문 시각을 동시에 렌더한다.** `CalendarGrid.tsx:151-160` 이 셀마다 걸치는 예약 전부를 `StatusBadge label={`${reservation.startTime} ${reservation.customerName}`}` 로 그린다. 주 뷰는 셀이 77개고 셀당 칩 수에 상한이 없다(FS-040 §7 #12).

세 가지가 따라온다.

**(a) 감사 로그가 '무엇을 봤는가'를 기록하지 못한다.** BE-037 §7.4(d)는 예약의 조회·수정을 서버가 기록하도록 요구한다. 그런데 이 화면의 조회는 `GET /api/reservations` **한 번**이고 그 한 번이 곧 '전 기간 모든 고객의 이름과 방문 시각'이다. 로그에는 한 줄이 남지만 운영자가 실제로 본 것은 전량이다 — **접근 범위와 로그 입도가 어긋난다.** 범위 조회(§7.2)를 도입하면 로그가 '이 운영자는 7월 13–19일을 봤다'로 좁혀져 감사 가치가 생긴다. **이것이 §7.2 의 성능 논거보다 강한 도입 근거다.**

**(b) 달력은 이름이 필요하지 않을 수 있다.** 이 화면의 목적은 '언제가 비었나'이고(FS-040 §1), 이름은 슬롯을 고른 **뒤** 패널(FS-040-EL-009.3)에서 보면 된다. 격자 칩이 이름 대신 예약번호나 자원명을 보이면 같은 목적을 달성하면서 노출 면적이 사라진다. **이는 제품 결정이며 계약의 결과는 이렇다**: 칩에서 이름을 빼기로 하면 달력은 `customerName` 을 요구하지 않게 되고, 그때 비로소 **더 마른 projection**(예: `{ id, date, startTime, endTime, resourceId, status }`)을 내려줄 수 있다. 지금은 격자가 이름을 쓰므로 그럴 수 없다. **심이 없으므로 projection 엔드포인트를 정의하지 않는다** — 결정이 나면 §7.2 의 범위 쿼리와 함께 설계한다.

**(c) 집계조차 예약의 존재를 드러낸다.** 칩을 지워도 셀의 `'2/4'`·`'마감'` 은 남는다 — 그것만으로 '그 시간에 누군가 온다'가 노출된다. 따라서 **달력의 어떤 축소도 예약 도메인 읽기 권한을 대체하지 못한다.** 이 화면 전체가 예약 읽기 권한 뒤에 있어야 하며, 그것이 §7.1 이 요구하는 바다.

**판정**: 서버는 `GET /api/reservations` 응답을 **예약 도메인 읽기 권한자에게만** 준다(BE-037 §7.5 상속). 달력이라는 이유로 완화하지 않는다. 감사 로그는 BE-037 §7.4(d)를 상속하되, **범위 조회 도입 시 조회 범위(`from`·`to`)를 로그에 함께 남긴다**(§7.6 #2).

### 7.4 수용량·마감은 서버 계약이 아니다 — 그리고 지금 모델이 갈려 있다 【정합 판정】

셀의 `capacity` 는 **자원의 개수**다: `slotCapacity() = listResources().length`(`schedule-data.ts:39-41`) → 4. `booked` 는 슬롯에 걸치는 유효 예약의 **건수**다. `full = booked >= capacity`(`:56`).

즉 이 화면의 모델은 '**자원 하나에 예약 하나**'(배타 점유)다. 그런데 같은 도메인의 `ReservationResource.capacity` 는 '**동시 수용 인원**'이고(`resources.ts:17-18` 주석: '정원(동시 수용 인원)') 예약 폼이 그것으로 인원을 검증한다(BE-037-EP-02 의 `PARTY_SIZE_EXCEEDS_CAPACITY`). **`slotCapacity()` 는 그 `capacity` 값을 읽지 않고 배열 길이만 센다** — 같은 이름의 두 개념이 한 도메인에 있다.

**정확히는 셋이다**(둘이 아니다): ① 달력 `capacity` = **자원 개수**(4 — `schedule-data.ts:39-41` · 자원 카탈로그 `resources.ts:30-33`, 단위 `건`) · ② 자원 `capacity` = **방별 정원**(room-a 12명 · room-b 4명 · room-c 2명 · seat-1 1명 — `resources.ts:18`, 렌더 `ReservationFormPage.tsx:302` `정원 N명`, 단위 `명`) · ③ `partySize` = **예약 인원**(`_shared/reservation.ts:26-27`, 폼 필드 `ReservationFormPage.tsx:307`, 단위 `명`). 달력의 '수용 M건'은 ① 이고 ②·③ 과 무관하다 — **단위 표기(`건`)는 최소한 그와 일관된다.**

**근인 — 집계가 `resourceId` 를 전혀 보지 않는다.** `reservationsInSlot`(`_shared/reservation.ts:135-149`)은 `reservationsOnDate`(날짜 + `isActive` — `:127-129`) 위에 **시간 겹침만** 얹고 자원을 비교하지 않는다. `slotCell`(`schedule-data.ts:52-57`)이 그 건수를 그대로 `booked` 로 쓴다.

관측되는 결과: **한 슬롯의 예약 4건이 전부 room-b(정원 4명)에 몰려 있어도 '마감'이고, room-a·room-c·seat-1 은 비어 있는데도 그렇다.** 반대로 room-b 에 예약 2건이 겹쳐 있으면(더블부킹) 셀은 '2/4' 로 **여유가 있는 것처럼** 보인다 — 예약 목록은 같은 두 건에 '중복' 배지를 단다(FS-037-EL-007.12). **같은 사실을 두 화면이 반대로 말한다.**

**★ 그 불일치는 추론이 아니라 코드에 있다 — 같은 파일의 두 함수가 서로 반대다.** `findConflicts` 는 자원을 **명시적으로 비교한다**: `_shared/reservation.ts:78` `if (reservation.resourceId !== candidate.resourceId) return false;` — 즉 **더블부킹 판정은 '자원별'이다.** 그런데 바로 아래 `reservationsInSlot`(`:135-149`)은 그 비교를 하지 않는다 — **달력 집계는 '자원 무관'이다.** 같은 4건을 **폼은 room-b 중복예약 4건으로 잡고 달력은 서로 다른 자원 4개를 채운 것으로 취급한다.** 한 모듈 안에서 자원 배정이 한 번은 의미가 있고 한 번은 없다 — **두 모델이 공존하며 서로를 반증한다.**

`schedule.test.ts:51-63` 이 이 동작을 고정한다 — 픽스처 `reservationOf` 의 `resourceId` 기본값이 `'room-b'`(`:16`)이고 `partySize` 기본값이 `2`(`:15`)라, 그 4건은 **전부 같은 방에 인원 8명**인데 테스트는 `full === true` 만 기대한다(`:62`). **테스트가 있다는 것은 의도라는 뜻이 아니라 '현재 동작이 고정됐다'는 뜻이다** — 이 테스트는 세 개념 중 어느 것이 옳은지에 대해 아무 말도 하지 않는다. 모델이 확정되면 **이 테스트가 먼저 바뀌어야 한다.**

**판정**: **이것은 서버가 지금 결정할 수 있는 사안이 아니다** — 도메인 정책(BE-037 §7.8 #4)이 먼저다. 계약이 정하는 것은 다음 세 줄이다:

1. **서버는 `booked`/`capacity`/`full` 을 계산해 주지 않는다** — 심이 없고(BE-040-EP-05 미정), 클라이언트가 이미 계산한다. 그 계산이 틀렸다면 고칠 곳은 클라이언트다. **서버 집계 엔드포인트를 만들어 클라이언트의 모델 혼선을 덮지 않는다** — 그러면 같은 개념이 두 곳에 생긴다.
2. **단, 정책이 '배타 점유'로 확정되면 서버가 그것을 강제해야 한다** — BE-037 §7.1 의 슬롯 충돌 검사(422 `RESERVATION_SLOT_CONFLICT`)가 바로 '자원 하나에 겹치는 예약 하나'다. **그 검사가 서버에 서면 달력의 '2/4'가 거짓말할 수 없게 된다** — 더블부킹이 애초에 저장되지 않으므로 건수 = 점유 자원 수가 성립한다. **즉 BE-037 §7.1 을 구현하는 것이 이 화면의 정합을 회복시키는 방법이다.**
3. 정책이 '인원 수용'으로 확정되면 셀은 `Σ partySize / Σ capacity` 를 세야 하고, 그때 **자원별로 나뉜 격자**(열 = 자원)가 필요해진다 — 지금 구조(열 = 날짜)로는 표현할 수 없다. 그것은 화면 재설계이지 계약 변경이 아니다.

**⚠ EP-02(자원 카탈로그) 연결 시의 위험**: 지금 `slotCapacity()` 는 절대 실패하지 않는다(모듈 상수). 서버 조회로 바꾸면 **실패·로딩 상태가 생기는데 이 화면에 그 표면이 없다** — 응답이 빈 배열이거나 도착 전이면 `capacity = 0` 이 되고 `full = booked >= 0` 이 **항상 참**이라 **격자 전체가 '마감'으로 물든다.** 자원 조회가 500 인데 달력이 "전부 마감"이라고 단정하는 것은 실패보다 나쁘다. 연결 전에 로딩·실패 표면이 선행돼야 한다(§7.6 #3 · BE-037 §7.8 #9).

### 7.5 영업시간·주 시작·'오늘'은 클라이언트가 정한다 — 심 없음 (미정)

| 값 | 어디서 | 심 |
|---|---|---|
| 영업시간 09:00–20:00 | `schedule-data.ts:13-14` (`OPEN_HOUR`·`CLOSE_HOUR`) | **없음** |
| 슬롯 길이 1시간 | `schedule-data.ts:30-36` (`DAY_SLOTS`) | **없음** |
| 주 시작 = 월요일 | `_shared/calendar.ts:58-64` (`startOfWeek`) | **없음** |
| '오늘' | `ScheduleCalendarPage.tsx:111` `toDateString(new Date())` — **브라우저 로컬 TZ** | **없음** |

**BE-040-EP-04 는 '심 없음(미정)'이다.** 지어내지 않는다.

**그러나 부재의 결과 하나는 계약 문제다**: 영업시간이 클라이언트 상수라 **08:00 이나 21:00 예약은 격자의 어느 셀에도 나타나지 않는다**(`DAY_SLOTS` 가 11개뿐이다). 그리고 **예약 폼은 그 시각을 아무 경고 없이 저장한다** — `reservationSchema` 에 영업시간 검증이 없다(BE-037 §3 필드 규약: `startTime` 은 형식 + `endTime > startTime` 만 본다). 즉 **저장은 되는데 달력에서는 영원히 보이지 않는 예약**이 만들어진다. 달력은 예약 일정의 정본 뷰인데 그것이 일부 예약을 조용히 감춘다.

**판정**: 영업시간이 도메인 규칙이라면 **서버가 EP-02/EP-04(BE-037)에서 검증해야 한다**(422 `OUTSIDE_BUSINESS_HOURS`, `error.fields[].name = 'startTime'`). 도메인 규칙이 아니라 '달력의 표시 범위'일 뿐이라면 **달력이 범위 밖 예약을 표시할 방법을 가져야 한다**(슬롯 확장 또는 '영업시간 외 N건' 표시). **둘 중 하나를 고르기 전까지는 조용히 사라지는 예약이 남는다** — 심이 없으므로 여기서 확정하지 않고 이관한다(§7.6 #5 · FS-040 §7 #11).
