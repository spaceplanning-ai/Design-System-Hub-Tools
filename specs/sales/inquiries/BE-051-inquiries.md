# BE-051. 문의 백엔드 기능 명세

## 1. 개요

| 항목 | 내용 |
|---|---|
| 대상 화면 | FS-051 문의 (`/sales/inquiries` · `/sales/inquiries/:id`) |
| 범위 | 문의 목록 조회, 문의 상세 조회, 문의 처리 저장(담당 배정 · 상태 전이 · 답변/내부메모 타임라인 기록), **'견적 발행' 전이의 견적 자동 생성** |
| **범위 밖** | **문의 생성** — 문의는 고객 채널(웹·전화·이메일·방문)이 접수한다. **문의 삭제** — 문의는 감사 대상 기록이다. 두 판정의 근거는 §7.6 이며, BE-003 §1 의 '회원 생성 — 고객은 회원가입으로만 유입된다' 와 같은 결이다. **견적의 조회·편집·삭제** — BE-050 소관. 이 문서는 **견적을 만드는 전이 하나**만 정의하고 그 결과물의 수명은 BE-050 이 소유한다. **고객 노출 렌더** — §7.4 XSS·내부메모 경계 판정만 다룬다 |
| 전제 | BE-003 §2·§3 을 상속한다. 인증은 세션 쿠키 기반. 모든 경로는 `/api` 프리픽스. 응답 본문은 `application/json; charset=utf-8`. 시각은 ISO 8601(오프셋 포함) |

> **에러 봉투·권한 모델 상속**: BE-003 §2·§3 을 그대로 상속한다(BE-010 §1 과 동일 선언). 아래는 문의 고유 차이만 기술한다.

## 2. 공통 (상속)

- **에러 봉투**: BE-003 §2. 공통 에러코드 동일(`VALIDATION_FAILED` · `UNAUTHENTICATED` · `FORBIDDEN` · `CSRF_TOKEN_INVALID` · `NOT_FOUND` · `CONFLICT` · `UNPROCESSABLE` · `RATE_LIMITED` · `INTERNAL_ERROR` · `REQUEST_TIMEOUT`).
- **권한**: `admin` = 전체. `operator` = 조회 계열(목록·상세) + **처리 저장**(EP-03) — 문의 응대는 영업 담당자의 본업이므로 BE-010(콘텐츠)과 달리 쓰기를 연다(§7.6 말미). **단 견적 발행 전이는 예외다** — §7.6 이 별도 권한을 요구한다. 영업 도메인 읽기 권한 없는 관리자 → 컬렉션 403 / **개별 문의 404 은닉**(§7.1).
- **CSRF**: 쓰기(PUT·POST)에 `X-CSRF-Token`.
- **타임아웃**: 조회 5초 → 504. 견적 생성을 동반하는 쓰기는 10초 → 504(§7.5).
- **프론트 역할 분기 없음**(FS-051 §4.1) — 권한 강제는 서버 책임. 프론트에 쓰기 게이팅이 배선돼 있지 않다(FS-051 §7 #27)는 사실이 이 원칙을 바꾸지 않는다.

## 3. 데이터 계약 (`types.ts` 대조)

| 타입 | 필드 | 비고 |
|---|---|---|
| `Inquiry` | `id` · `inquiryNo` · `title` · `type` · `channel` · `customerName` · `company` · `contact` · `assignee`(미배정이면 `''`) · `priority` · `status` · `receivedAt`(ISO) · `body` · `quoteId` · `timeline` | 목록·상세가 **같은 타입**이다 — 목록도 `body`·`timeline` 을 받는다(§7.7) |
| `InquiryEvent` | `id` · `at`(ISO) · `author` · `kind` · `text` | append-only. `kind` = `received`\|`note`\|`reply`\|`status` (`types.ts:15`) |
| `InquiryInput` | `Omit<Inquiry, 'id'>` | 저장 입력. 불변 필드(`inquiryNo`·`receivedAt`·`body`·`customerName`·`company`·`contact`·`channel`·`type`·`priority`)와 **서버 소유 필드 `quoteId`** 까지 포함한 **전체 치환**이다(§7.7) |
| `InquiryStatus` | `received` \| `assigned` \| `in_progress` \| `hold` \| `quote_issued` \| `answered` \| `closed` | 7개. 라벨: 접수·배정·처리중·보류·견적 발행·완료·종결 |
| `InquiryType` | `quote` \| `product` \| `support` \| `partnership` \| `claim` \| `etc` | 접수 후 불변 |
| `InquiryChannel` | `web` \| `phone` \| `email` \| `visit` | 접수 후 불변 |
| `InquiryPriority` | `urgent` \| `high` \| `normal` \| `low` | 접수 후 불변 — **관리자 변경 UI 가 없다** |
| 상수 | `INQUIRY_REPLY_MAX = 1000` (`types.ts:59`) | 답변·메모 본문 상한. **담당(`assignee`) 상한은 코드에 없다 — 이 문서가 정한다**(§4 EP-03) |

**상태 전이 규칙 — 【판정: 계약에 신설한다. 코드에는 없다】**

**코드에 전이 규칙이 존재하지 않는다**(§1.1 대조표). FS-026(1:1 문의)이 `STATUS_FLOW`·`canTransition`·`canSetStatus`·`allowedNextStatuses` 4겹으로 규정하는 자리가 이 도메인에는 **비어 있고**, select 는 7개 상태 전부를 언제나 보여준다(FS-051-EL-021). 따라서 아래 표는 **코드를 서술한 것이 아니라 이 계약이 신설하는 규칙**이며, §7.2 가 그 판정 근거다. **아키텍처 의 도메인 확정이 선행돼야 한다.**

| 현재 | 허용되는 다음 상태 | 비고 |
|---|---|---|
| `received` | `assigned` · `hold` · `closed` | 배정 없이 처리중으로 뛰지 않는다 |
| `assigned` | `in_progress` · `hold` · `closed` | |
| `in_progress` | `quote_issued` · `answered` · `hold` · `closed` | 견적 발행은 처리중에서만 — 배정도 안 된 문의로 견적이 나가지 않게 한다 |
| `hold` | `assigned` · `in_progress` · `closed` | 보류는 어느 진행 단계로도 복귀 가능 |
| `quote_issued` | `answered` · `hold` · `closed` | **`in_progress` 로 되돌아가지 못한다** — §7.2 |
| `answered` | `closed` · `in_progress` | 재오픈 허용 |
| `closed` | **없음** | 종착 |

- 현재 상태 → 자기 자신은 언제나 허용(변경 없음도 유효한 저장).
- **담당 요건**: `in_progress`·`quote_issued`·`answered` 로 가려면 `assignee.trim() !== ''` 이어야 한다. **견적 발행에 담당을 요구하는 것이 이 규칙의 핵심**이다 — 견적서에는 담당자가 찍힌다.
- 위반 시 422(§5).

**견적 발행 전이 (`quote_issued`) — 부수효과 계약**

| 항목 | 내용 |
|---|---|
| 발동 조건 | `requestsQuoteIssue(요청 status)` = `status === 'quote_issued'` **AND** `!hasIssuedQuote(저장된 문의)` = 저장된 `quoteId === ''` (`data-source.ts:126`) |
| 승계 필드 | `QuoteInheritance`(`quotes/types.ts:226-237`): `inquiryId` · `inquiryNo` · `company`→견적 `accountName` · `customerName`→견적 `contactName` · `body`→견적 `inquiryBody` · `issueDate` = **오늘**(문의 접수일이 아니다 — `data-source.ts:135`) |
| 승계하지 않는 것 | 품목·과세유형·금액 — 문의가 그 정보를 갖지 않는다. 견적 상태는 `draft`(작성중) — '품목이 비어 있는 견적을 발송으로 두면 발송되지 않은 견적이 발송으로 보인다'(`quotes/types.ts:242`) |
| 생성물 채번 | 견적번호는 서버가 채번한다(`makeQuoteNo(issueDate, seq)` → `Q-YYYYMMDD-NNN`). **입력이 정하지 않는다**(`quotes/data-source.ts:82-86`) |
| 양방향 링크 | 문의 `quoteId` = 새 견적 id · 견적 `inquiryId`/`inquiryNo` = 문의 값 |
| 타임라인 | `견적 자동 생성 — <견적번호>`, `kind='status'`, `author='시스템'`(`data-source.ts:141-149`) |
| 멱등 | **`quoteId` 가 멱등 키다** — 이미 발행된 문의는 상태를 오가도 저장을 연타해도 견적이 하나다. 견적 저장소가 `findQuoteByInquiry` 로 한 번 더 교차 확인한다(이중 방어) |

### BE-051-EP-03 · 문의 처리 저장 (담당 · 상태 · 타임라인)
| 항목 | 내용 |
|---|---|
| 근거 (FS) | FS-051-EL-020, EL-021, EL-025, EL-027, EL-028, EL-033 |
| 메서드·경로 | `PUT /api/sales/inquiries/:id` |
| 근거 (심) | `data-source.ts:153` `GET/PUT /api/sales/inquiries/:id (답변·상태·담당 저장)` |
| 권한 | `admin`, `operator`. **단 요청 status 가 `quote_issued` 면 견적 create 권한을 함께 요구한다**(§7.6) |
| 멱등성 | `Idempotency-Key` 헤더로 보장한다(§7.8 #1). 형식상 PUT 멱등이지만 **그 멱등성이 곧 lost update 다**(§7.3) |
| 레이트리밋 | 분당 60회 |

**바디**(현재 `InquiryInput`): `inquiryNo` · `title` · `type` · `channel` · `customerName` · `company` · `contact` · `assignee` · `priority` · `status` · `receivedAt` · `body` · `quoteId` · `timeline[]`.

**서버 검증 (요청을 그대로 믿지 않는다)**
1. **불변 필드 무시**: `inquiryNo` · `receivedAt` · `body` · `customerName` · `company` · `contact` · `channel` · `type` · `priority` 는 관리자가 바꿀 수 없다. 프론트가 `toInquiryInput(inquiry)`(`types.ts:195-212`)로 되돌려 보내지만 **서버는 저장된 값을 정본으로 유지하고 요청 값을 무시**한다 — 프론트가 낡은 스냅샷을 되돌려 보내 고객 원문을 덮는 사고를 막는다(§7.7).
2. **`quoteId` 는 서버 소유다**: 요청 바디에 실려 오지만 **절대 신뢰하지 않는다.** 이 값은 발행 전이의 결과로만 바뀐다 — 클라이언트가 `quoteId: ''` 를 보내 **멱등 키를 지우고 재발행을 유도하는 경로를 닫는다**(§7.5 【보안 판정】).
3. **상태 전이**: §3 의 `STATUS_FLOW` + 담당 요건을 **서버가 판정한다.** 프론트에 대응 규칙이 **없으므로**(§1.1) 이 판정은 **1차이자 유일한 방어선**이다. 위반 시 422(§5).
4. **타임라인**: 요청의 `timeline` 을 **신뢰하지 않는다** — §7.2·§7.3 이 계약을 대체한다.
5. `assignee` 는 앞뒤 공백 제거 후 **0–100자**. **프론트에 길이 제약이 없으므로**(§1.1) 이것도 서버가 유일한 방어선이다.
6. 답변/메모 본문은 `INQUIRY_REPLY_MAX = 1000` 자. 프론트는 네이티브 `maxLength` 로 자르지만 **잘림은 검증이 아니다** — 서버가 재확인한다.

**응답 200/204**. 프론트 `update(id, input, context?): Promise<void>` — 응답 본문을 읽지 않고 상세를 재조회한다(`InquiryDetailPage.tsx:202`).

**에러**: 400 `VALIDATION_FAILED`(`error.fields`: `status`·`assignee`) · 401 · 403 `FORBIDDEN` · 403 `CSRF_TOKEN_INVALID` · **404 `INQUIRY_NOT_FOUND`** · **409 `CONFLICT`**(§7.3) · **422 `INVALID_STATUS_TRANSITION`** · **422 `ASSIGNEE_REQUIRED`** · **422 `QUOTE_ISSUE_FAILED`**(§7.5) · 429 · 500 · 504.

## 7. 핵심 판정

### 7.1 문의 상세는 404 로 은닉한다 【보안 판정】

BE-003 §3.2 의 원칙 두 줄을 이 도메인에 적용한다.

1. **컬렉션의 존재는 비밀이 아니다** → `GET /api/sales/inquiries` 권한 부족 시 **403 `FORBIDDEN`**.
2. **개별 문의 리소스의 존재 자체가 개인정보다** → 영업 도메인 **읽기 권한이 없는** 주체에게는 **404 `INQUIRY_NOT_FOUND`** 로 은닉한다.

**근거**: 문의 1건은 `customerName` · `company` · `contact`(이메일·전화번호 — 시드가 `kim@hanbit.example` · `010-5555-6666` 로 그 둘을 다 보여준다) · `body`(고객이 쓴 원문 — 시드의 '100석 기준 ERP 라이선스와 구축 일정')를 담는다. **B2B 영업 문의의 존재 자체가 경쟁 정보다**: '(주)한빛소프트웨어가 ERP 도입 견적을 요청했다'는 사실은 그 거래처에게도, 경쟁사에게도 민감하다. `Inquiry` 는 회원 레코드보다 민감도가 낮지 않다 — BE-003 §3.2 가 `GET /api/members/:id` 를 404 로 은닉하는 것과 **같은 이유로** `GET /api/sales/inquiries/:id` 도 404 여야 한다. 문의 id 열거로 '어느 회사가 무엇을 문의했는가'를 알아내는 경로를 닫는다.

**반대로 읽기 권한이 있는 주체**(`operator`)가 쓰기에서 거절될 때는 **403** 을 준다 — 이미 문의의 존재를 아는 주체에게 존재를 숨기는 것은 의미가 없다.

**프론트 영향**: FS-051-EL-015 가 404 와 5xx 를 같은 문구로 그리므로 **은닉은 성립하지만 정당한 운영자도 원인을 모른다.** 어댑터 쪽 전제(404 → `HttpError`)는 **이미 충족**이므로(§4 EP-02) 화면의 분기만 붙이면 된다.

### 7.2 상태 전이 규칙이 코드에 없다 — 서버가 정본을 신설한다 【정합 판정】

**F3b 코드로 재확인한 사실**: `sales/inquiries/types.ts` 에 상태 전이를 판정하는 함수가 **하나도 없다**. `isInquiryStatus`(`:152-154`)는 문자열이 7개 리터럴 중 하나인지만 본다. 상세의 select 는 `INQUIRY_STATUS_OPTIONS` 7개를 **언제나 전부** 렌더한다(`InquiryDetailPage.tsx:308-312`). `PipelineStepper` 같은 시각적 단계 표현조차 없다.

**이것이 만드는 사고**:
- '종결'된 문의를 '접수'로 되돌릴 수 있다 — 감사 기록이 시간을 거스른다.
- '접수'(담당도 없는) 문의를 곧장 '완료'로 보낼 수 있다 — 아무도 응대하지 않은 문의가 완료로 집계된다.
- **'견적 발행' → '처리중' → '견적 발행'** 을 오갈 수 있는데, 두 번째 발행은 `quoteId` 멱등 키가 막는다(§7.5). 결과: **상태는 '견적 발행'인데 이번 저장은 아무 견적도 만들지 않는** 조합이 정상 경로로 존재하고, 운영자는 FS-051-EL-033 의 '저장했습니다'(발행 문구 없음) 토스트로만 그 차이를 안다.

**판정**: 서버가 §3 의 `STATUS_FLOW` 를 정본으로 신설하고 **422 로 강제한다.** 프론트에 1차 차단이 없으므로 **이 422 는 유일한 방어선**이며, 관용적으로 처리해선 안 된다.

**단, 이 판정에는 선행 조건이 있다** — §3 의 전이표는 **내가 코드에서 읽어낸 것이 아니라 도메인 관례에서 제안한 것**이다. 특히 `quote_issued → in_progress` 금지와 '견적 발행에 담당 필수'는 정책 결정이다. **아키텍처 이 확정하기 전까지 이 표는 제안이다.** 확정 전에 서버가 임의의 표를 강제하면 운영자가 정당한 전이에서 막힌다 — 아키텍처 확정과 서버 구현은 같은 배치여야 한다.

**프론트 후속(UI 기획)**: 확정된 표는 FS-026 처럼 **3중으로** 내려와야 한다 — ① select 선택지를 `allowedNextStatuses` 로 좁힘 ② 저장 직전 `canSetStatus` 재확인 ③ 서버 422. 지금은 ③만 있다.

### 7.3 타임라인 전체 치환 → append-only 이벤트로 계약을 바꾼다 【보안·정합 판정】

**현재 프론트는 조회 시점 타임라인 배열 전체를 클라이언트에서 재조립해 PUT 한다**(`InquiryDetailPage.tsx:159-182` — `let timeline = inquiry.timeline` 에서 시작해 `appendEvent` 로 덧붙인 **배열 전체**를 `InquiryInput.timeline` 에 실어 보낸다).

**이것이 만드는 사고**: 담당 A 가 상세를 열고(타임라인 2건 스냅샷) 답변을 쓰는 동안 담당 B 가 답변을 저장한다(타임라인 3건). A 가 저장하면 **A 의 2건 스냅샷 + A 의 새 이벤트 = 3건**이 서버의 3건을 덮는다 — **B 의 답변이 흔적 없이 사라진다.** 전체 치환 PUT 이라 서버는 이것이 유실인지 정상 갱신인지 구분할 수 없다.

**이 도메인에서 더 나쁜 이유**: 타임라인에는 **`견적 자동 생성 — Q-…` 이벤트가 서버(픽스처에서는 어댑터)에 의해 덧붙는다**(`data-source.ts:141-149`). 발행 직후 상세를 열지 않은 다른 탭이 저장하면 **그 시스템 이벤트가 지워진다** — 견적이 왜 생겼는지 설명하는 유일한 기록이다.

**판정**: 타임라인은 **서버가 소유하는 append-only 컬렉션**이다. 계약을 둘 중 하나로 바꾼다.

| 안 | 형태 | 평가 |
|---|---|---|
| **A (권장)** | `PUT /api/sales/inquiries/:id` 는 **담당·상태만** 받고(`{ assignee, status }`), 답변/메모는 **별도 append 엔드포인트** `POST /api/sales/inquiries/:id/events` `{ kind, text }` 로 분리. 상태변경·배정·견적 생성 이벤트는 **서버가 PUT 의 부수효과로 자동 기록** | 유실이 구조적으로 불가능하다. 프론트의 조립 블록(`:158-182`)이 통째로 사라지고 §7.2 의 `author`/`at` 문제도 자동 해결된다. **어댑터 시그니처가 바뀐다** — `CrudAdapter.update` 로 표현할 수 없어 화면 코드가 함께 바뀐다 |
| **B (차선)** | PUT 을 유지하되 서버가 요청의 `timeline` 을 **무시**하고, 새 이벤트만 별도 필드(`newEvent?: { kind, text }`)로 받아 서버가 append | 어댑터 형태를 덜 흔든다. 그러나 `InquiryInput` 이 실제로 쓰이지 않는 `timeline` 을 계속 실어 보내는 기형이 남고, 요청 크기가 이력에 비례해 계속 커진다 |

**어느 안이든 서버는 요청의 `timeline` 배열을 절대 신뢰하지 않는다.**

**타임라인 작성자·시각 위조 — 함께 판정한다** 【보안 판정】: 현재 클라이언트가 `author`·`at`·`id` 를 전부 정한다 — `author` 는 하드코딩 `'관리자'`(`InquiryDetailPage.tsx:55`) 라 **누가 답변했는지 기록되지 않고**, `at` 은 클라이언트 시각(`:158`)이라 시계가 틀리면 이력의 시간순이 무너지며, `id` 는 `ev-${Date.now()}-s|c`(`:165,175`) 라 **두 담당이 같은 밀리초에 저장하면 충돌한다**(같은 저장 안의 상태·본문 두 이벤트는 접미사 `-s`/`-c` 로 갈리지만, **다른 저장끼리는 갈리지 않는다**). 이 세 값은 **전부 서버가 찍는다** — 요청 바디의 `author`·`at`·`id` 는 **무시**한다. 클라이언트가 보낸 작성자를 신뢰하면 인증된 아무 관리자나 다른 관리자 이름으로 답변을 남길 수 있다(감사 로그 위조). `author` 는 **세션 주체**에서, `at` 은 **서버 시각**에서, `id` 는 **서버 채번**에서 온다. 영업 문의 이력은 견적·계약 분쟁 시 증거로 쓰인다.

**추가 — 담당 변경 이벤트 누락**: 프론트는 상태·본문만 이벤트로 남기고 **담당 변경은 남기지 않는다**(FS-051 §7 #17). 안 A 를 채택하면 서버가 자동 기록하므로 이 구멍도 함께 닫힌다.

**낙관적 동시성 — 정확히 쓴다**: 어댑터의 409(`crud.ts:126-128`)는 **'대상이 존재하는가'** 만 본다 — `If-Match`/`version`/`ETag` 필드가 `Inquiry` 에 **없다.** 따라서 **둘 다 존재하는 문의를 두 관리자가 동시에 편집하면 여전히 last-write-wins** 이며, 409 는 발생하지 않는다. 유령 저장(없는 id 에 성공 반환)은 해소됐지만 **동시 편집 유실은 남아 있다** — 이 둘을 뭉개지 말 것. 서버는 `Inquiry.version`(또는 `If-Match` ETag)을 도입해 409 를 그 경합에도 발생시킨다. **§7.3 안 A 를 채택하면 담당·상태만 다투므로 last-write-wins 로 충분**할 수 있다 — 답변 유실이라는 진짜 위험이 append 로 사라지기 때문이다. **결론: 안 A 채택 시 토큰은 선택, 안 B 유지 시 토큰은 필수.**

### 7.4 답변 본문 XSS · 내부메모 경계 【보안 판정】

타임라인의 `kind='reply'` 이벤트 `text` 는 **관리자가 쓰고 고객에게 전달되는 문구**다(FS-051-EL-025 → EL-028). 관리자 화면은 `Timeline` 이 텍스트 노드로만 렌더하지만(`packages/ui/.../Timeline.tsx:43` — `<p className="tds-timeline__text">{event.text}</p>`), **고객 채널(메일 회신·거래처 포털)이 이를 HTML 로 해석하면 저장형 XSS 가 된다**. BE-010 §7.2 · BE-026 §7.1 과 동일 판정 — 서버가 **저장 시** 허용 태그 화이트리스트 밖의 마크업 · `javascript:`/`data:` 스킴 · 이벤트 핸들러 속성을 제거하고, 계약은 '저장된 답변에 실행 가능한 스크립트가 없다'는 관측 동작만 정한다. 정제는 **저장 시 1회**이며 렌더 시점 이스케이프에 의존하지 않는다.

**고객 문의 원문(`body`)도 같은 처리를 받는다** — 고객이 넣은 스크립트가 관리자 화면(FS-051-EL-019)에 렌더된다. 현재는 텍스트 노드라 안전하지만, 정제 없이 저장하면 향후 어떤 화면이 HTML 로 렌더하는 순간 관리자가 피해자가 된다. **이 도메인에서는 그 '향후'가 이미 예약돼 있다** — `body` 는 견적 발행 시 `quote.inquiryBody` 로 **승계되고**(`quotes/types.ts:260`), 견적은 **PDF/문서 출력이 로드맵에 있는 산출물**이다(quality-bar ERP-10). 문의 원문이 견적서로 흘러 들어간다는 사실이 정제를 저장 시점으로 앞당길 이유다.

**내부메모(`kind='note'`)의 고객 노출 경계**: 내부메모가 고객답변과 **같은 타임라인 배열에 섞여** 배지로만 구분된다(FS-051-EL-029). 시드의 내부메모가 '담당 배정, 견적서 준비 중.'(`data-source.ts:43`)인 것처럼, 이 도메인의 메모에는 **협상 카드가 담긴다** — 서버가 고객 채널로 이력을 내보낼 때 `kind='note'` 를 **반드시 제외**해야 한다. 이것은 렌더 규칙이 아니라 **계약**이다: 고객 대상 응답 스키마에 `note` 이벤트가 들어갈 자리를 만들지 않는다.

### 7.5 견적 발행은 한 트랜잭션이다 — 두 경로를 열지 않는다 【정합·보안 판정】

**코드가 이미 옳은 결정을 내렸고, 그 이유를 남겼다**(`data-source.ts:6-8`):

> '[문의 → 견적 발행] 상태를 「견적 발행」으로 바꾸면 견적이 자동 생성된다(H). 그 부수효과를 어댑터의 patch 안에 둔다 — 화면이 「문의 저장」과 「견적 생성」을 두 번 호출하면 하나만 성공하는 창이 생기고, 그 창에서 새로고침하면 상태는 「견적 발행」인데 견적이 없는 문의가 남는다.'

**판정: 서버도 같은 형태를 유지한다.** `POST /api/sales/inquiries/:id/issue-quote` 심(`quotes/data-source.ts:132-133`)이 존재하지만, 그것을 **공개 엔드포인트로 열면 프론트가 두 번 호출하는 길이 다시 생긴다.** EP-03 이 문의 잠금 → 전이 판정 → 견적 채번·생성 → 역링크 → 타임라인 append 를 **한 트랜잭션**으로 수행하고, 부분 성공을 남기지 않는다. 심의 경로명은 그 내부 트랜잭션의 이름으로 보존한다.

**심과 코드가 어긋나는 지점 — 판정으로 가른다**: 심은 '이미 발행된 문의면 **409 로 거절**한다'고 적혀 있는데, 코드는 **조용히 통과**시킨다(`data-source.ts:126` — 발행 없이 문의만 갱신, `quotes/data-source.ts:135-136` — 기존 견적 반환). 테스트가 코드 쪽을 고정한다(`inquiries.test.ts:140-149` — '이미 발행된 문의를 다시 저장해도 견적을 새로 만들지 않는다').

**판정: 코드가 옳다. 심의 409 를 채택하지 않는다.** 이유 — 발행은 EP-03(문의 저장)의 **부수효과**이지 독립 요청이 아니다. 이미 발행된 문의의 담당을 바꾸려는 정당한 저장이 '이미 발행됨' 409 로 거절되면 **운영자는 그 문의를 영영 편집할 수 없다**. 409 는 '요청 전체가 성립하지 않는다'는 뜻인데, 여기서는 요청의 나머지(담당·상태·답변)가 완전히 성립한다. 대신 **화면이 이미 그것을 정직하게 말한다** — FS-051-EL-022 의 '견적은 다시 생성되지 않습니다' 배너와 FS-051-EL-033 의 토스트 문구 분기(발행 시에만 '견적을 발행했습니다')가 그 역할을 한다. **심의 주석은 갱신 대상이다**(§7.8 #5).

**`quoteId` 는 서버 소유다** 【보안 판정】: `InquiryInput` 이 `quoteId` 를 포함한다(`types.ts:209`). 즉 **클라이언트가 `quoteId: ''` 를 보낼 수 있다.** 서버가 그것을 믿으면 **멱등 키가 지워지고 같은 문의로 견적이 무한히 발행된다** — 견적번호가 채번되고 거래처에 중복 견적이 나간다. 서버는 요청의 `quoteId` 를 **언제나 무시**하고 저장된 값을 정본으로 유지한다. 이 값은 발행 전이의 결과로만, 서버 안에서만 바뀐다. (프론트는 현재 `toInquiryInput` 이 원본을 되돌려 보내므로 사고가 나지 않지만, **그것은 프론트가 착하기 때문이지 계약이 막기 때문이 아니다** — §6.1 이 `quoteId` 를 입력에서 빼도록 이관한다.)

**타임아웃·재시도**: 견적 생성을 동반하는 저장은 문의 갱신보다 무겁다 → §2 가 10초로 둔다. 504 후 운영자가 재시도하면 `Idempotency-Key` 가 재생을 보장하고(§7.8 #1), 그것이 없어도 `quoteId` 멱등 키가 이중 발행을 막는다 — **이중 방어**다.

**끊어진 링크 — 미해결로 남긴다**: 견적이 삭제되면(BE-050 의 `DELETE /api/sales/quotes/:id`) 문의의 `quoteId` 는 남는다. 그러면 ① FS-051-EL-008.9 의 '견적 보기' 가 404 로 가고 ② **그 문의는 `quoteId` 가 비어 있지 않아 재발행도 막힌다** — 견적을 잃은 문의는 영원히 견적을 가질 수 없다. **판정: 견적 삭제가 문의의 `quoteId` 를 비우도록 BE-050 이 역참조를 정리해야 한다**(또는 발행된 견적의 삭제를 금지하고 '발행 취소'로만 다루게 한다). **어느 쪽인지는 BE-050 소관이며 이 문서가 정하지 않는다** — §7.8 #7 로 이관한다.

### 7.6 문의 생성·삭제 — 계약에 존재하지 않는다 (단, 코드는 FS-026 만큼 단단하지 않다) 【범위 판정】

**F3b 코드로 재확인한 사실**:
- 화면에 등록 버튼·행 선택·삭제·일괄 작업이 **없다**(FS-051 §2).
- `InquiryListPage.tsx:3-4` 헤더 주석: '문의는 고객 채널이 만들고 관리자는 처리·답변만 한다. 그래서 CrudListShell(삭제·일괄) 대신 읽기 전용 표를 쓴다'.
- `data-source.ts:3-4` 주석: '문의 생성은 고객 채널이 만든다 — 관리자 생성 UI 없음'.
- `inquiryAdapter.create`/`remove` 는 **호출부가 0건**이다(grep 확인).

**판정**: `POST /api/sales/inquiries` · `DELETE /api/sales/inquiries/:id` 를 **만들지 않는다.** BE-003 §1 이 '회원 생성 — 고객은 회원가입으로만 유입된다' 로 확정한 것과 같은 결이다.

**단, BE-026 과 정확히 갈리는 지점을 기록한다** — 이 판정의 코드 근거는 **BE-026 보다 약하다**:

| | BE-026 (1:1 문의) | BE-051 (이 문서) |
|---|---|---|
| 어댑터 `create`/`remove` | **거절 구현** — `Promise.reject(new Error('문의는 고객 채널에서 접수됩니다.'))` | **동작하는 구현** — `createCrudAdapter` 가 만든 진짜 create/remove |
| 저장소 함수 | `addTicket`/`removeTicket` 이 **아예 없다** | 팩토리 안에 존재 |
| 방어 겹 수 | 4겹(어댑터 거절 · 저장소 부재 · 주석 · UI 부재) | **2겹(주석 · UI 부재)** |

즉 **이 도메인에서 '생성·삭제 없음'을 강제하는 것은 UI 부재와 주석뿐**이며, 코드는 `inquiryAdapter.create(...)` 를 한 줄 부르면 즉시 동작한다. 공용 팩토리를 쓴 대가다 — 팩토리는 5개 함수를 다 만들어 준다. **이것은 결함이 아니라 기록해야 할 사실이다**: 서버가 엔드포인트를 만들지 않으면 이 표면은 도달 불가능해지므로(어댑터 본문이 fetch 로 바뀌는 순간 부를 곳이 사라진다) **계약이 그 강제를 대신한다.** 백엔드 명세 은 연동 시 `create`/`remove` 를 **BE-026 처럼 거절 구현으로 바꾸거나 어댑터에서 제거**해야 한다 — §7.8 #8.

**삭제에 대한 추가 근거**: 영업 문의는 **감사 대상 기록**이다(견적·계약 분쟁 시 증거이며, 발행된 견적이 이 문의를 역참조한다 — `quote.inquiryId`). 개인정보 파기 요구(보존기간 만료)는 관리자 화면의 단건 삭제 버튼이 아니라 **보존정책 배치**로 처리해야 할 별도 계약이며, 그 배치는 **역참조하는 견적을 어떻게 할지 함께 정해야 한다** — 백엔드 명세 이관(§7.8 #9).

**`operator` 에게 쓰기를 연다 — 단 발행은 예외** 【권한 판정】: BE-010(FAQ)은 `operator` 를 조회 전용으로 두지만 이 도메인은 반대다 — 문의 응대는 영업 담당자의 본업이고, 시드의 담당자가 '이영업'·'박계약'인 것이 이 도메인이 담당자 다수를 전제함을 보여준다. **그러나 '견적 발행' 전이는 다르다**: 그것은 문의 처리가 아니라 **거래처에 나갈 산출물(견적서)을 만드는 행위**이며 견적번호를 채번한다. 문의 쓰기 권한만으로 견적을 만들 수 있다면 **견적 리소스의 create 권한이 우회된다.** 판정: `status='quote_issued'` 요청은 **문의 update 권한 AND 견적 create 권한**을 함께 요구하고, 후자가 없으면 **403 `QUOTE_ISSUE_FORBIDDEN`**(404 은닉 대상 아님 — 주체는 이미 문의의 존재를 안다). 프론트에는 이 분기가 **없다**(§7.8 #3).

### 7.7 목록이 전량·전문을 내려준다 — 페이징과 목록 전용 표현을 도입한다

**현재 계약의 두 문제**:
1. **페이징이 없다.** `fetchAll(signal)` 이 파라미터를 받지 않고 전량을 반환하며, 프론트가 필터·검색·정렬을 전부 클라이언트에서 한다. 문의는 **매일 쌓이는 무한 증가 컬렉션**이다 — 거래처·FAQ 와 달리 상한이 없다.
2. **목록이 상세 전문을 담는다.** `Inquiry` 타입 하나를 목록·상세가 공유해 목록 응답에 `body`(고객 원문)와 `timeline`(**내부메모 포함** 전체 이력)이 실린다 — **목록 화면이 쓰지 않는 필드**이며, 동시에 **§7.1·§7.4 의 취지와 정면으로 충돌한다**: 목록 한 번 조회로 **모든 문의의 내부메모가 브라우저에 내려온다.** 목록은 그것을 렌더하지 않지만, 내려보낸 것은 내려보낸 것이다.

**판정**: BE-010 이 `FaqSummary`/`Faq` 로 나눈 것과 같이 **`InquirySummary` / `Inquiry` 를 분리**하고, EP-01 에 `type`·`channel`·`status`·`keyword`·`page`·`size`(기본 20 · 상한 100) 쿼리를 도입한다. `InquirySummary` 는 `body`·`timeline` 을 빼고 `quoteId`·`quoteNo`(역링크 표시용)만 남긴다.

**이관**: 이 변경은 **프론트 대공사**다 — `filterInquiries`/`searchInquiries`/`sortInquiries` 가 서버로 올라가고, 페이지네이션 UI(FS-051 §7 #2)·`SeqCell` 오프셋(#18)이 함께 붙어야 한다. **IA-13 은 이 화면에서 이미 충족돼 있다**(`useListState` 소비 — 다른 화면과 달리 URL 직렬화가 선행 완료됐다) → **페이징 파라미터를 URL 에 얹는 것은 `page` 키 하나 추가로 끝난다**(`useListState.ts:26` `PAGE_PARAM` 이 이미 있고 `clampPage`(`:217-223`)도 준비돼 있다). quality-bar IA-04 P0 · ERP-15 P1 이 이미 이 화면을 gap 으로 잡고 있으므로 **한 배치로 묶는 것이 옳다.** 그전까지 현 계약(전량)을 유지한다 — 픽스처 3건에서는 드러나지 않는다.
