# FS-069. AI 모델 연동 마켓스토어

> **4.0 개정 요지 — 진열대에 이제 무언가를 붙일 수 있다.**
> 3.0 까지 이 화면은 **아무것도 붙일 수 없는 진열대**였다. `settingsPath` 가 13건 전부 `null` 이라
> '앱 설정' 이 전 행 비활성이었고, 그래서 저장된 연동이 0건이고 13/13 이 '연동 해제' 였다
> (3.0 §7.3 #4 — 최우선 미결).
>
> 4.0 은 그 저장 경로를 만든다: **`/settings/api-keys/:providerId`**(`AiConnectionPage.tsx`).
> 목록에서 이름이나 '앱 설정' 을 누르면 주소가 바뀌고, 그 프로바이더가 **실제로 요구하는 칸만**
> 그려지며, 저장하면 **실제로 저장된다** — 목록의 상태가 바뀌고 AI 에이전트 화면의 응답 모드가
> 열린다. 브라우저에서 세 카테고리(모델·클라우드·게이트웨이)와 저장→잠금 해제를 확인했다.
>
> **이 전환은 발명이 아니다.** `/settings/oauth` 가 직전 기준에서 목록/상세로 갈린 그 구현을
> 그대로 따랐다(`OAuthPage`/`OAuthProviderPage` · `listSavePayload`/`providerSavePayload` 로 저장
> 범위를 가르는 것 · `<Link>` 로 미저장 가드를 살리는 것 · 알 수 없는 id 를 빈 화면이 아니라
> 사유와 돌아갈 길로 받는 것). **두 화면이 다른 방식이면 다음 사람이 어느 쪽을 따라야 할지 모른다.**
>
> **저장이 생겼다고 검증까지 생긴 것은 아니다.** '자격증명이 채워짐' 과 '실제로 연결됨' 은 서로
> 다른 사실이며, 후자는 서버가 프로바이더를 실제로 불러 봐야 성립한다. 그래서 상세 화면은 두
> 사실을 **나란히 다른 줄에** 그리고, 검증 쪽은 '확인한 적 없음' 으로 둔다 — 없는 버튼을 그리지
> 않는다(§3.4).
>
> **3.0 개정 요지 — 카탈로그가 통째로 바뀌었다.**
> 2.0 에서 Rest API V2 카드(자격증명 발급·노출·이름변경·폐기)를 걷어내 이 화면을 순수한 진열대로
> 되돌렸다. 3.0 은 그 진열대에 **무엇을 진열하는가**를 바꾼다: 커머스 연동 10종(본인인증·소셜
> 로그인 3종·소셜 공유·PG·로그분석·SMS/알림톡·도메인·SSL)이 전부 빠지고 **AI 모델 프로바이더
> 13종**이 들어왔다.
>
> 그 결과 이 화면의 **근거가 바뀌었다.** 2.0 까지 연동 상태는 OAuth 자격증명에서 해소됐지만,
> 이제는 **저장된 AI 자격증명**에서 해소된다(`ai-connections.ts`). OAuth 결합은 이 화면에서
> 완전히 사라졌다 — 조회도, 실패 배너도, `providerIsUsable` 참조도 없다.
>
> **지워진 표면을 약속으로 남겨 두지 않는다** — 다만 그 표면에 기록돼 있던 결함은 §7.1(키 흐름)과
> §7.2(커머스 카탈로그)에 '삭제로 해소' 로 사유와 함께 남긴다.

### 1.1 이 화면의 설계 원칙 (요구사항 — 명세에 고정한다)

| 원칙 | 내용과 이유 (코드 근거) |
|---|---|
| **연동 상태를 지어내지 않는다** | 상태는 **저장된 자격증명에서만** 나온다(`integrations.ts:resolveIntegrations` ← `data-source.ts:currentIntegrations`). **픽스처로 연동된 척 채우지 않는다** — 그러면 AI 에이전트 화면의 응답 모드가 열리고, 열린 모드가 아무 일도 하지 않는다. ✔ **4.0 에서 초기값은 여전히 빈 목록이고**(13/13 해제) **운영자가 실제로 저장해야 완료가 된다** — 픽스처로 채우는 것과 결정적으로 다른 점이다 |
| **연동은 두 조건이 모두 참일 때만 성립한다** | `connectionIsUsable` = **켜져 있고**(`enabled`) **필수 자격증명이 빠짐없이 저장돼 있다**. 하나만으로는 부족하다: 켜기만 하고 키가 없으면 호출이 401 이고, 키만 있고 꺼져 있으면 운영자가 쓰지 않겠다고 정한 것이다. 테스트가 두 갈래를 각각 못박는다(`api-keys.test.ts`) |
| **자격증명이 '키 하나' 라고 가정하지 않는다** | 각 항목이 자기 요구를 타입으로 들고 다닌다(`credentials: readonly AiCredentialField[]` — `integrations.ts:12-21`). '어차피 API 키 한 칸이면 되겠지' 로 폼을 만들면 **Azure OpenAI·Amazon Bedrock 에서 저장은 되는데 호출이 실패하는** 가장 진단하기 어려운 고장이 난다. 요구가 타입에 있으면 그 폼을 만들 수 없다 — 그것이 이 필드가 존재하는 이유다(§3.1) |
| **평문을 담을 자리를 만들지 않는다 — 단, 비밀이 아닌 값은 예외다** | `AiConnection` 은 **값이 아니라 사실의 집합**이다 — `storedFields` 는 '어느 칸이 채워져 있는가' 만 말한다. ✔ **4.0 에서 저장 문서(`AiConnectionRecord`)가 두 자루로 갈렸다**: `storedSecrets`(비밀 칸의 **이름만**)와 `publicValues`(엔드포인트·배포명·리전·API 버전의 **값**). 후자는 비밀이 아니고 **되읽지 못하면 폼이 쓸모없어진다**(운영자가 배포명을 매번 다시 입력해야 하고 한 글자 틀리면 404 다). 한 자루에 섞지 않는 것이 요점이다 — **비밀이 들어갈 자리가 `publicValues` 에 구조적으로 없다**(§3.4) |
| **검증한 척하지 않는다 — 그리고 '채워짐' 과 구분한다** | `lastVerifiedAt` 은 **서버가 프로바이더를 실제로 불러 본** 시각이다. 브라우저에서 부르면 키가 브라우저로 내려와야 하고 그 순간 '평문을 저장하지 않는다' 가 거짓이 된다. 그래서 **항상 null** 이고 프론트는 검증 시늉을 내지 않는다. ✔ **4.0 에서 이 구분이 화면에 드러난다**: '연결 상태' 카드가 **자격증명(채워짐/채워지지 않음)** 과 **연결 검증(확인한 적 없음)** 을 나란히 다른 줄에 그리고, **부를 곳이 없는 '검증' 버튼을 만들지 않는다**(§3.4) |
| **저장되지 않는데 성공을 알리지 않는다** | ✔ **4.0 에서 저장이 생겼고, 그래서 이 원칙이 더 중요해졌다.** 성공 토스트는 저장이 **실제로 끝난 뒤**(`onSuccess`)에만 뜨고, 그 뒤 목록의 상태와 AI 화면의 잠금이 **실제로 바뀐다** — 화면이 말한 것과 앱이 하는 일이 같다. 이것은 2.0 에서 삭제된 `renameApiKey` 의 결함(없는 id 를 조용히 지나치고 '바꿨습니다' 토스트를 띄웠다)을 되풀이하지 않겠다는 뜻이다(§7.1). **회귀 테스트가 '토스트가 떴다' 가 아니라 '저장소가 실제로 무엇을 들고 있는가' 를 단언한다**(`ai-connection-routing.test.tsx` 머리말) |
| **저장은 화면에 보이는 것만 쓴다** | 상세 화면이 쓰는 것은 **이 프로바이더 하나**다(`connectionSavePayload`). 나머지 연동은 서버가 준 최신 문서 그대로다 — 보이지도 않는 값을 조용히 쓰는 '저장' 은 기능이 아니라 결함이다. `../oauth` 의 `providerSavePayload`/`listSavePayload` 와 **같은 규약**이다 |
| **켜기와 자격증명을 분리해 받지 않는다** | `enabled: true` 로 저장하려면 필수 칸이 다 와야 한다 — 분리하면 '켰는데 안 된다' 가 생긴다. **끄는 것은 언제나 허용한다**: 자격증명이 반쯤 채워져 있어도 끌 수 있어야 하고, 꺼도 저장된 값은 지워지지 않는다(§3.4) |
| **로고를 지어내지 않는다** | **13건 전부 `brand: null` 이다** — OpenAI·Claude·Gemini·Grok 의 공식 마크를 1차 출처에서 확인하지 못했다(§3.2). 확인 못 한 마크는 그리지 않고 짧은 표기 배지로 남긴다(`ServiceGlyph`). 비슷하게 그린 로고는 상표 문제이자 '이게 진짜 로고' 라는 거짓 정보다 |
| **배지 표기는 목록 안에서 유일해야 한다** | 마크가 하나도 없으므로 **배지가 항목을 구분하는 유일한 시각 수단**이다. 그래서 표기를 이름에서 자동으로 따지 않고 **카탈로그가 항목마다 명시한다**(`glyph`). 유일성은 목록 전체를 봐야 판정할 수 있는데 글리프 컴포넌트는 한 항목만 알기 때문이다 — **책임을 목록을 아는 쪽에 둔다**(§3.3) |
| **확인한 주소만 링크한다** | `guideUrl` 은 1차 문서를 **실제로 확인한 것만** 싣는다(오늘은 Azure·Bedrock 둘뿐). 링크는 눌러 봐야 죽은 줄 안다 — 죽은 '연동 방법 안내' 는 없는 것보다 나쁘다(운영자는 우리가 확인했다고 믿고 눌렀다가 404 를 본다) |
| **잠근 것은 지우지 않고, 왜 잠겼는지 말한다** | '앱 설정' 은 갈 곳이 없으면 **비활성으로 남기고 이유를 말한다**(`integrations.ts` — '없는 게 아니라 아직 이라는 사실도 정보다'). 불변식이 테스트로 고정돼 있다: `settingsPath === null` ⟺ `settingsUnavailableReason !== null`. ✔ **4.0 에서 13건 모두 갈 곳이 생겨 사유가 0건이 됐지만, 규약과 필드는 남긴다** — 새 프로바이더가 화면 없이 먼저 카탈로그에 오르는 날이 다시 올 수 있다 |
| **누를 수 없는 것을 링크로 그리지 않는다** | ✔ **4.0 신규.** 갈 곳이 있으면 '앱 설정' 과 이름이 **`<Link>`** 다 — 새 탭·중간 클릭·'링크 주소 복사' 가 앱의 다른 목록(회원·공지·상품)과 똑같이 동작한다(`MembersTable` 의 Link-as-button 선례). 갈 곳이 없으면 비활성 모양의 `<span aria-disabled>` 이고 사유가 옆에 붙는다 — **링크처럼 보이는데 아무 데도 안 가는 것**을 만들지 않는다 |
| **같은 문장을 13번 반복하지 않는다** | ✔ **3.0 신규.** 보이는 행이 **전부 같은 이유로** 잠겨 있으면 그 이유를 표 위에 한 번만 적는다(`IntegrationsCard.tsx:143-156`). 이유를 버튼 옆에 적는 규율은 이유가 **행마다 다를 때**를 위한 것이다 — 13행이 똑같은 문장을 들면 그 문장이 표를 덮고, 정작 행마다 다른 정보(이름·설명)를 밀어낸다. **데이터는 그대로 두고 표시만 접는다** |
| **건수와 행이 같은 필터에서 나온다** | 탭이 두 축(분류·상태)을 섞어 쓰지만 집계는 **한 함수**를 지난다(`filterIntegrations`). 두 축이 각각 목록을 남김없이 나눈다(분류 합 = 상태 합 = 전체) — 테스트가 셋 다 못박는다 |
| **경계를 넘을 때는 합성 지점을 지난다** | AI 에이전트 화면이 연동 상태를 읽지만, `pages/ai → pages/settings` 직접 import 는 축1 blocker 다. 공통 층이 자리를 만들고(`shared/fixtures/ai-providers.ts`) 배선이 구현을 꽂는다(`src/wiring-ai.ts`) — **화면끼리는 끝까지 서로를 모른다**(§5.1) |

### 3.1 자격증명 요구 — 프로바이더별 (요구사항)

**이 표가 이 화면의 새로운 중심이다.** 프론트는 아직 저장하지 않지만, **무엇을 저장해야 하는지는
타입으로 확정돼 있다**(`integrations.ts` 의 `credentials`). 저장 폼은 반드시 이 선언에서 생성돼야
한다 — 손으로 칸을 나열하면 요구와 폼이 갈라진다.

| 프로바이더 | 분류 | 필수 | 선택 | 근거·주의 |
|---|---|---|---|---|
| OpenAI | 모델 | `apiKey` | — | 조직·프로젝트 헤더는 **선택**이다(다중 조직 계정에서만). 필수 칸으로 만들지 않는다 |
| Anthropic Claude | 모델 | `apiKey` | — | ⚠ **인증 방식이 혼자 다르다** — `Authorization: Bearer` 가 아니라 **`x-api-key`** 헤더이고 **`anthropic-version` 헤더가 필수**다. 다만 그것은 **클라이언트가 고정으로 보내는 상수**지 운영자 입력이 아니라 칸으로 만들지 않았다 — 없는 입력을 요구하면 폼이 거짓말을 한다 |
| Google Gemini | 모델 | `apiKey` | — | ⚠ **Vertex AI 와 다른 제품이다** — Vertex 는 프로젝트 id·리전·OAuth 를 요구한다. 여기서 말하는 것은 Gemini Developer API(AI Studio 키)다 |
| Grok | 모델 | `apiKey` | — | OpenAI 호환 형식. **회사명이 아니라 제품명으로 표기한다**(§3.2) |
| Mistral AI · Perplexity · Cohere · DeepSeek · Groq | 모델 | `apiKey` | — | 단일 베어러 키 |
| **Azure OpenAI** | 클라우드 | `apiKey` · **`endpoint`** · **`deployment`** | `apiVersion` | ⚠ **키 하나로 되지 않는다.** 호출 경로에 **모델명이 아니라 배포명**이 들어간다(틀리면 404). 기존(dated) 표면은 `api-version` 이 필수이고, 2025-08 GA 한 v1 표면은 그것을 요구하지 않고 배포명이 본문 `model` 로 옮겨간다 — 그래서 `apiVersion` 은 **선택**이다. 근거: learn.microsoft.com/azure/foundry/openai/api-version-lifecycle |
| **Amazon Bedrock** | 클라우드 | `apiKey` · **`region`** | — | ⚠ **키 하나로 되지 않는다.** 베어러 토큰(Bedrock API key)이 SigV4 와 함께 제공되지만 **리전은 언제나 별도**다 — 엔드포인트 호스트명에 박히고(`bedrock-runtime.<region>.amazonaws.com`), 단기 키는 발급한 리전에서만 쓰인다. 근거: docs.aws.amazon.com/bedrock/latest/userguide/api-keys.html |
| OpenRouter · Together AI | 게이트웨이 | `apiKey` | — | 키 하나로 여러 모델 |

**시크릿 판정**: `apiKey`·`secretAccessKey` 만 `secret: true` 다 — 저장 후 다시 보여줄 수 없고 고정
길이 마스크만 그린다. `endpoint`·`deployment`·`apiVersion`·`region` 은 **설정값이지 비밀이 아니다**
(가리면 운영자가 어느 리소스인지 대조할 방법이 사라진다). 테스트가 두 갈래를 못박는다.

### 3.4 저장의 규칙 — 무엇이 저장되고 무엇이 저장되지 않는가 ✔ 4.0 신규 (요구사항)

**이 절이 4.0 의 중심이다.** 저장이 생기는 순간 '무엇이 어디에 남는가' 가 계약이 된다.

| 규칙 | 내용 | 이유 |
|---|---|---|
| **비밀 칸은 이름만 남는다** | 입력한 평문은 저장 요청에만 존재하고, 문서에는 `storedSecrets: ['apiKey']` 처럼 **칸 이름만** 남는다(`data-source.ts:applyCredentials`). 테스트가 문서를 통째로 직렬화해 평문이 없음을 단언한다 | 실제 백엔드에서는 이 자리가 해시/암호화 보관이 된다. 여기서는 **애초에 값을 버리는 것**으로 같은 성질을 얻는다 |
| **비밀이 아닌 칸은 되읽는다** | `publicValues` 에 값이 남고 폼이 다시 채워진다(배포명·엔드포인트·리전·API 버전) | 되읽지 못하면 운영자가 배포명을 매번 다시 입력해야 하고, 한 글자 틀리면 호출이 404 다 |
| **빈 값은 저장하지 않는다** | 공백뿐인 선택 칸은 `publicValues` 에 키 자체가 들어가지 않는다 — '비어 있음' 과 '없음' 을 구분하지 않고 **없음으로 수렴시킨다** | 빈 문자열을 실제 헤더/파라미터로 보내면 프로바이더가 **401·400 으로 거절한다**. OpenAI 의 조직 헤더가 그 예이고, 그래서 그 칸은 아예 두지 않았다 |
| **비밀 칸을 비우면 기존 값이 유지된다** | 빈 문자열 = '그대로 둔다'. 새 값을 넣었을 때만 교체된다 | 그러지 않으면 배포명 하나 고치려고 저장할 때마다 **콘솔에서 키를 재발급받아야 한다** (우리는 그 키를 돌려줄 수 없다) |
| **부분 저장이 없다** | 연동 하나를 **통째로** 쓴다(그래서 PATCH 가 아니라 PUT 이다 — BE-069 §7.7.2 #3) | Azure 의 4칸 중 3칸만 저장되면 화면은 '연동 해제' 를 그리는데 **운영자는 저장 성공 배너를 봤다** |
| **켜기에는 필수 칸이 다 필요하다** | `enabled: true` + 필수 칸 미충족 → 저장이 막히고 칸마다 오류가 붙는다. **끄기는 언제나 허용** | 분리해 받으면 '켰는데 안 된다' 가 생긴다 |
| **꺼진 연동은 검증하지 않는다** | `enabled: false` 면 필수 칸 검사를 건너뛴다(길이 상한만 본다) | 쓰지 않을 값을 채우라고 요구하지 않는다 |
| **형식은 막지 않고 알린다** | 키의 길이·문자셋·접두어를 **검사하지 않는다**. 엔드포인트·리전은 경고만 | `sk-` 접두어는 관찰이지 규약이 아니다 — 프로바이더가 형식을 바꾸는 날 멀쩡한 값이 거절된다. 13종 전부에 그 위험이 있고 **어느 형식도 문서로 보장받지 못했다** |
| **'채워짐' 은 '연결됨' 이 아니다** | 두 사실을 **다른 줄에** 그린다(EL-016 · EL-017). `enabled === true` 는 전자다 | 한 값으로 뭉치면 **배포명 오타 같은 고장이 '연동 완료' 배지 뒤에 숨는다** |
| **`connectedAt` 은 상태 전이 시점이다** | `해제 → 완료` 로 **처음** 넘어간 순간에 찍히고, 재연동해도 최초 값을 유지한다. 필수 칸이 안 찬 저장에서는 찍히지 않는다 | '저장 시각' 으로 대신하면 열 이름('연동 시작일')이 거짓말이 된다. **화면이 아니라 저장소(서버 자리)가 찍는다** — 페이지는 시각을 만들지 않는다 |

### 3.2 브랜드 마크 — **하나도 확보하지 못했다** (요구사항)

13건 전부 `brand: null` 이고 화면은 머리글자 배지를 쓴다. 이것은 미완성이 아니라 **판단**이다.

| 사실 | 내용 |
|---|---|
| 넘겨받은 자산이 잘못됐다 | Grok·OpenAI 두 SVG 의 **path 데이터가 동일**했다. 두 마크는 기하학적으로 양립 불가다 — OpenAI Blossom 은 6회 회전대칭의 **닫힌** 컴파운드 패스이고, Grok 마크는 대칭이 없는 **분리된 두 열린** 스트로크에 직선 대각 꼬리가 있다. **적어도 하나는 확실히 잘못된 라벨이고, 어느 쪽인지 알 수 없다** |
| 1차 출처 접근 실패 | openai.com/brand(403) · x.ai/legal/brand-guidelines(403) · anthropic.com/company/brand(404). **네 브랜드 모두 색상 hex 를 1차 출처에서 확인하지 못했다**(Claude 는 `#D97757` 과 `#DA7756` 두 값이 돌아다녀 판정 불가) |
| 2차 출처를 베끼지 않는다 | Wikimedia·블로그의 path 는 기하 참고로만 쓰고 산출물에 넣지 않는다 |
| 표기는 제품명 기준 | **xAI 는 2026-02 SpaceX 에 인수돼 2026-07-06 SpaceXAI 로 리브랜딩을 마쳤다.** 회사명을 적으면 낡은 이름이 화면에 박히고 새 이름이 맞는지도 확인되지 않았다 — 제품명 **'Grok'** 은 그 변화와 무관하게 참이다(`integrations.ts` 해당 항목 주석) |
| 기존 마크는 그대로 | Google·Naver·Kakao·Facebook·Apple·LINE 은 검증된 자산이라 유지한다. **`shared/ui/brand-marks.tsx` 는 이번 개정에서 수정하지 않았다** |

### 3.3 배지 표기(`glyph`) — 카탈로그가 정하고, 유일성을 테스트가 고정한다 (요구사항)

마크가 하나도 없으므로(§3.2) **배지가 항목을 구분하는 유일한 시각 수단**이다. 그래서 표기는
이름에서 자동으로 따지 않고 카탈로그 항목마다 명시한다.

| 분류 | 항목 → 표기 |
|---|---|
| 모델 | OpenAI → `OA` · Anthropic Claude → `Cl` · Google Gemini → `Gm` · Grok → `Gk` · Mistral AI → `Mi` · Perplexity → `Px` · Cohere → `Co` · DeepSeek → `DS` · Groq → `Gq` |
| 클라우드 | Azure OpenAI → `Az` · Amazon Bedrock → `Br` |
| 게이트웨이 | OpenRouter → `OR` · Together AI → `Tg` |

**왜 규칙이 아니라 사람이 정하는가 — 실제로 겪은 결함이다.** 3.0 최초 구현은 `ServiceGlyph` 가
`name` 의 첫 글자를 잘랐다. 커머스 3종(사·플·F)에서는 맞았지만 AI 13종에서는 **8종으로 뭉갰다**:

- `G` × 3 — Google Gemini · Grok · Groq
- `A` × 3 — Anthropic Claude · Azure OpenAI · Amazon Bedrock
- `O` × 2 — OpenAI · OpenRouter

기본 '모델' 탭에 똑같은 `G` 배지 셋이 나란히 놓였고, 특히 **Grok/Groq 는 이름이 한 글자 차이라
배지까지 같으면 구분 수단이 아예 없었다.** 앞 두 글자로 늘려도 `Gr`/`Gr` 로 여전히 같다 —
그래서 **끝 글자**를 쓴다(`Gk`/`Gq`). 같은 이유로 Claude 는 회사명(Anthropic)이 아니라 제품명에서,
Bedrock 은 Amazon 이 아니라 제품명에서 딴다(둘 다 `A` 로 Azure 와 겹친다).

| 판정 | 내용 |
|---|---|
| **책임의 위치** | 유일성은 **목록 전체를 봐야** 판정할 수 있는데 `ServiceGlyph` 는 한 항목만 안다. 그래서 표기를 목록을 아는 쪽(카탈로그)이 정해 넘긴다 — 컴포넌트는 받은 것을 그리기만 한다 |
| **필수 필드다** | `glyph` 가 `IntegrationCatalogueEntry` 의 필수 필드라 **프로바이더를 더할 때 결정을 건너뛸 수 없다** |
| **테스트가 고정한다** | 유일성 · 길이(1~2자) · 비어있지 않음 · Grok≠Groq 를 단언한다. **이 결함은 단언 한 줄이면 잡혔을 것이다** — 그래서 그 한 줄을 남긴다 |
| **접근성 대상이 아니다** | `aria-hidden` 이라 스크린리더는 이 표기를 읽지 않는다(이름은 옆 텍스트가 전한다). **순수하게 눈으로 갈리게 하는 것이 목적**이므로 판정 기준도 시각적 구분 하나다 |

> **사람이 직접 받아와야 하는 것** — 아래 셋은 자동 접근이 막혀 브라우저로 열어야 한다:
> `anthropic.com/press-kit`(실제 .zip 존재) · `openai.com/brand` · `x.ai/legal/brand-guidelines`.
> 자산을 확보하면 `brand-marks.tsx` 에 마크를 더하고 카탈로그의 `brand` 를 채우면 된다 —
> 화면은 손대지 않는다.

### 4.1 공통 예외 규칙

| 규칙 | 내용 |
|---|---|
| 네트워크 단절 | ✔ **4.0 에서 호출이 생겼다** — 조회 실패는 목록에서 배너 + 재시도(EL-001.1), 상세에서 폼 대신 배너 + 재시도. 저장 실패는 카드 안 danger 배너이고 **입력을 버리지 않는다** |
| 세션 만료 | 401 은 앱 전역 인터셉터가 받아 `/login?returnUrl=…&reason=session_expired` 로 보낸다. ⚠ **4.0 부터 잃을 입력이 있다** — 상세 폼에 입력한 평문은 사라진다. 재로그인 후 다시 입력해야 하며, **그 편이 평문을 어딘가에 보관하는 것보다 낫다** |
| 요청 타임아웃 | 조회·저장 모두 `AbortController` 를 붙여 화면 언마운트·다이얼로그 취소 시 중단한다. 중단은 실패로 알리지 않는다(EXC-09) |
| 중복 제출 | ✔ **`useSubmitLock` 이 막는다**(EXC-08) — `disabled={saving}` 만으로는 클릭과 리렌더 사이 틈으로 두 번째가 통과한다. 동기 ref 잠금은 렌더를 기다리지 않으므로 그 틈을 닫는다 |
| 실패 통지의 자리 | **인라인이다.** 조회 실패는 배너 + 재시도, 저장 실패는 카드 안 배너(다이얼로그가 떠 있으면 다이얼로그 안). **토스트로 실패를 알리지 않는다**(STATE-02) — 성공만 토스트다 |
| 낙관적 업데이트 | **하지 않는다.** 저장이 성공한 뒤 서버가 돌려준 문서로 기준선을 다시 잡는다 — 검증 결과를 낙관적으로 그리지 않는 것과 같은 규율이다 |
| 동시 조회 | **1건**(`aiConnectionsKey`). 목록과 상세가 **같은 문서 한 벌**을 읽는다 — 그래서 상세에서 저장하면 목록이 곧바로 최신이다 |
| 권한 없음 | **read** — `RequirePermission` 이 `<Outlet>` 을 감싸 403 화면. 리소스는 `page:/settings/api-keys`. `findCoveringLeaf` 가 **상세 라우트를 같은 잎으로 풀어 주므로 두 라우트가 똑같이 덮인다**(OAuth 상세와 같다). ✔ **write** — 상세가 `useRouteWritePermissions().canUpdate` 를 읽어 저장 컨트롤을 **렌더하지 않는다**(EXC-03). 3.0 §7.3 #5('저장 화면이 붙는 순간 곧바로 문제가 된다')를 **화면과 함께 넣었다** |
| 렌더 예외 | AppShell 이 `<Outlet>` 바깥에 `ErrorBoundary`(`resetKey={pathname}`) — 사이드바가 남고 `RouteErrorScreen` |
| 프론트 검증은 보증이 아니다 | 검증 정본은 `validation.ts` 의 zod 스키마이고, **그 스키마는 카탈로그의 `credentials` 에서 만들어진다**(§3.1) — 손으로 칸을 나열하지 않는다. ⚠ **서버도 같은 규칙을 독립적으로 알아야 한다**(BE-069 §7.1) — 프론트 검증은 UX 이지 강제가 아니다 |
| 자격증명의 수명 | 평문은 **저장 요청에만** 실린다. 응답·목록·상태 어디에도 들어가지 않는다 — `AiConnection` 은 `storedFields`(어느 칸이 찼나)만 안다. 검증 시각(`lastVerifiedAt`)은 **서버가 실제로 불러 본** 뒤에만 채워진다 |

### 5.1 도메인 경계 — AI 화면이 읽는 연동 상태

AI 에이전트 화면(`pages/ai`)의 응답 모드 잠금이 이 화면의 연동 상태에서 파생된다. 그런데
`pages/ai → pages/settings` 직접 import 는 **code-quality 축1 blocker** 다.

| 층 | 것 | 역할 |
|---|---|---|
| 공통 | `shared/fixtures/ai-providers.ts` | **자리만** 만든다 — `registerAiProviderLookup` · `enabledAiProviders()` · `hasEnabledAiProvider()` · `AI_PROVIDER_CATALOG`(표시명) |
| 설정 | **`data-source.ts` 의 `aiProviderStatuses()`** | 구현 — 저장된 연동에서 `{ id, label, enabled }` 를 만든다. ✔ **4.0 에서 `integrations.ts` 에서 옮겨왔다**: 이제 저장소를 읽어야 하는데 `integrations.ts` 가 저장소를 읽으면 `data-source ↔ integrations` 순환이 된다(저장소는 `connectedAt` 판정에 카탈로그의 `required` 를 쓴다). **순수한 쪽(카탈로그·필터·집계)이 남고 저장소를 아는 쪽이 옮겨간 것**이다 |
| 배선 | `src/wiring-ai.ts` 의 `wireAiProviders()` | 둘을 잇는다. **화면끼리는 끝까지 서로를 모른다** |

- **fail-closed**: 배선되지 않았거나 조회기가 던지면 '연동 없음' 으로 수렴한다. 미배선을 '전부 사용
  가능' 으로 읽으면 연동이 없는데 모드가 열리고, 고른 모드가 아무 일도 하지 않는다(FEEDBACK-03).
- **노출 범위를 좁힌다**: 카탈로그는 13건이지만 공통 층에 넘기는 것은 **핵심 4종**(openai · claude ·
  gemini · grok)뿐이다. 응답 모드 잠금이 게이트웨이·클라우드까지 알 이유가 없고, 유니온을 넓히면
  소비자 쪽 분기가 프로바이더를 하나 더할 때마다 깨진다.
- ⚠ **`enabled === true` 는 '자격증명이 갖춰졌다' 이지 '방금 확인했다' 가 아니다** — 실제 호출 검증은
  아직 없다(§3.4). 소비자 문구가 '사용 가능' 을 단언하지 않아야 한다.
- ✔ **4.0 에서 이 잠금이 실제로 풀린다.** 상세 화면에서 OpenAI 를 켜고 키를 저장하면
  `aiProviderStatuses()` 가 곧바로 `enabled: true` 를 돌려주고 `/ai/chat` 의 '빠른 · 전문가 · 헤비'
  가 열린다 — 모드가 **조회 시점에** 해소되기 때문이다(`pages/ai/_shared/modes.ts`
  `resolveResponseModes`). **브라우저에서 확인했다**: 저장 전 세 모드에 '미연결' 알약과
  'OpenAI · Claude · Gemini · Grok 중 하나를 연동하면 열립니다' 가 붙어 있고, 저장 후 그것이 사라지고
  'OpenAI 연동됨 — 모든 응답 모드를 고를 수 있습니다' 로 바뀐다.
- **조회기는 한 벌뿐이다.** 두 번째 조회기를 만들지 않았다 — 계약은 `shared/fixtures/ai-providers.ts`
  가 정의하고, 이 화면은 그 자리에 구현을 꽂을 뿐이다.
- ⚠ **`peek()` 은 픽스처의 성질이지 계약이 아니다.** 조회기 계약이 동기(`() => Status[]`)라 저장소의
  동기 읽기가 필요했고, 그래서 `_shared/store.ts` 에 `peek()` 을 더했다. **백엔드가 붙으면 이 표면은
  사라진다** — 그때 조회기는 react-query 캐시를 읽어야 한다(§7.3 #16).

> **현재 구현 상태**: 백엔드는 없다. 저장소는 `_shared/store.ts` 의 **리비전 붙은 픽스처 저장소**이며
> 지연·실패·409 재현만 한다(`?fail=load` · `?fail=save` · `?fail=conflict`). **초기값은 빈 목록**이고
> 운영자가 저장해야 채워진다. **TODO(backend) 심 5곳이 실재한다** — 조회·상세 조회·저장·검증
> (`data-source.ts`) · 연동 목록(`integrations.ts`). BE-069 는 이 심들을 계약으로 옮긴 것이며
> **발명한 엔드포인트가 없다.** ⚠ **인메모리라 새로고침하면 초기화된다** — 백엔드 부재의 결과이며

## 7. 미결 사항

### 7.1 2.0 에서 **키 흐름 삭제로 해소된** 항목 (기록으로만 남긴다)

Rest API V2 카드와 그 하부 시스템이 삭제되면서, 그 표면의 결함들은 **고쳐진 것이 아니라 대상이
사라져 성립하지 않게** 됐다. 되살릴 일이 생기면 이 표가 그때 다시 살아난다.

| 결함 | 어떻게 해소됐나 |
|---|---|
| **유령 저장 + 유령 폐기** — `renameApiKey` 가 `keys.map` 이라 **없는 id 를 조용히 지나치고 성공을 반환**했다(사실상 no-op 인데 '키 이름을 바꿨습니다' 토스트가 떴다). **이미 폐기된 키의 이름도 그대로 바꿨다.** `revokeApiKey` 는 재폐기 시 **`revokedAt` 을 새 시각으로 덮어 감사 기록을 훼손**했다 | **삭제로 해소.** `data-source.ts` 가 세 함수와 함께 삭제됐다. 고치지 않고 지웠다 — 이 화면에 자격증명 수명주기가 더는 없다. ✔ **3.0 의 설계 원칙이 이 교훈을 명시적으로 계승한다**(§1.1 '저장되지 않는데 성공을 알리지 않는다') |
| **발급 모달의 안내 문구가 거짓이었다** — '이후에는 **마지막 4자리만 표시**되며' 라고 약속했는데 모델은 last4 를 **의도적으로 거부**했고 표는 고정 12글리프를 그렸다 | **삭제로 해소.** `CreateApiKeyModal.tsx` 가 삭제됐다 |
| **`issuedByKey` 가 평문을 세션 내내 들고 있었다** — 멱등 재생을 위해 모듈 수준 `Map` 에 평문을 담아, '평문은 모달 지역 state 로만 산다' 는 원칙과 어댑터 층에서 어긋나 있었다. **재생이 요청 바디를 보지도 않아** 같은 멱등키로 다른 바디를 보내면 조용히 옛 응답을 돌려줬다 | **삭제로 해소** |
| **브라우저가 시크릿을 만들었다** — `createDummyPlaintextKey` 가 클라이언트에서 평문을 생성해, '평문은 서버가 만들어 1회만 돌려준다' 는 중심 계약이 아직 참이 아니었다 | **삭제로 해소.** 마지막 소비자가 사라져 `_shared/secret.ts` 에서 함수와 테스트 3건도 삭제됐다. ⚠ **`MASKED_SECRET_TEXT`·`copyToClipboard` 는 남겼다** — 소셜 로그인 설정이 여전히 쓴다 |
| **발급 주체가 클라이언트 값이었다** — `createdBy: '김운영'` 하드코딩 | **삭제로 해소.** 다만 요구 자체가 사라지진 않았다 — 저장 주체 기록은 AI 연동에도 필요하다 |
| **조회 실패가 화면 전체를 가렸다** — early return 이 연동 목록과 사이드바까지 지웠다 | **삭제로 해소 — 옳은 쪽으로 수렴했다.** 화면을 대체하는 실패 경로가 하나도 남지 않았다 |
| 읽기 전용 안내가 3중 AND 였고 문구가 이름 변경 권한을 빠뜨렸다 · 폐기 중 전 행이 잠기고 `aria-busy` 가 사라졌다 · 제출 실패 직후 포커스가 이름 입력으로 튀었다 · 이름 중복 검사가 경합에 취약했다 · 폐기된 키의 이름도 중복 판정에 들어갔다 · 수동 복사가 `copied` 를 못 켰다 · 노출 모달에 이탈 가드가 없었다 · 모델은 아는데 화면이 안 그리는 값이 셋이었다 · 페이지네이션·스코프 오류 연결·raw `length`·`describedBy`·abort 잔여 경로 | **전부 삭제로 해소** — 해당 컴포넌트·모델과 함께 사라졌다 |
