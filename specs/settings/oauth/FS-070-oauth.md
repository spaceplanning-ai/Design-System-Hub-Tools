# FS-070. OAuth 설정

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 목적 | 소셜 로그인 제공자(Google · 카카오 · 네이버)의 **사용 여부**와 **자격증명**(Client ID · Client Secret · 카카오 네이티브 앱 키) · **Redirect URI** 를 한 폼에서 정하고 저장하며, **로그인 화면의 표시 정책** 1건을 함께 둔다 |
| 역할(주 사용자) | 관리자 (구현에 역할 분기 없음 — 권한은 `useRouteWritePermissions` 로 게이팅. §4.1) |
| 진입 경로 | 좌측 GNB > 시스템 > 시스템 설정 > OAuth 설정 (`shared/layout/nav-config.ts:216` `['OAuth 설정', '/settings/oauth']`) |
| 포함 화면 | 단일 라우트 `/settings/oauth` — **하위 라우트가 없다(잎)** |
| **범위 밖** | **연결 테스트** — 버튼이 **비활성**이다. '프론트가 흉내 낼 수 없다(시크릿은 서버에만 있고 CORS 도 막혔다)'(`OAuthProviderCard.tsx:502-505`). **제공자 추가** — 3종이 코드 상수(`validation.ts:35`)이며 화면에서 만들 수 없다. **카카오 어드민 키** — 의도적으로 **받지 않는다**(`validation.ts:27-32` '로그인에는 필요하지 않은 마스터 키다'). **카카오싱크 동의항목·약관 연결** — 콘솔로 보낸다(EL-010.2). **실제 소셜 로그인 동작** — 이 화면은 자격증명을 저장할 뿐 인증 흐름의 주체가 아니다(백엔드 소관). **저장된 시크릿 조회** — 기능이 아니라 **불가능**하다(§1.1) |
| 구현 경로 | `apps/admin/src/pages/settings/oauth/**` (`OAuthPage.tsx` · `data-source.ts` · `validation.ts` · `oauth.test.ts` · `components/{OAuthProviderCard.tsx,OAuthProviderCard.test.tsx,ProviderBrandIcon.tsx}`) · 공유 `pages/settings/_shared/**` |
| 대응 SCR | SCR-070 (미작성 — §7 #1) |
| **하류 소비자** | **이 화면의 값이 API Key 화면의 연동 목록을 결정한다** — `api-keys/integrations.ts:222` 가 `providerIsUsable`(`oauth/validation.ts:429-431`)를 불러 '연동 완료' 를 판정한다. **판정을 복제하지 않는다**(`integrations.ts:17` 이 근거를 적는다). 즉 여기서 제공자를 끄거나 Client ID 를 비우면 **다른 화면의 상태 표시가 함께 바뀐다**(§5) |
| 공통 컴포넌트 | `settings/_shared/{SettingsFormShell,ConflictDialog,AuditNote,MASKED_SECRET_TEXT,copyToClipboard,useSettingsQuery,useSaveSettings,useSubmitLock,createRevisionedStore,formatAuditAt}` · `shared/ui/{Alert,Button,Card,CardTitle,ConfirmDialog,FormField,Icon,ToggleSwitch,controlStyle,errorIdOf,useToast,useUnsavedChangesDialog}` · `@tds/ui{Checkbox}` · `shared/permissions/RequirePermission` · `shared/form/zodResolver` · `shared/async(isAbort)` |

> **이 화면은 사이트 설정과 같은 셸·저장소를 쓴다**(`SettingsFormShell` + `createRevisionedStore`) — 단일 문서 폼이다. **다른 점은 문서가 중첩이라는 것**(`{ providers: [...], display: {...} }` — `validation.ts:342-343`)이며 그로 인해 충돌 표시(EL-021.1)·시크릿 취급(§1.1)이 갈린다.
>
> ⚠ **이전 판(1.0)이 비교 대상으로 삼던 '언어 설정'(`settings/languages/**` · FS/BE/NFR-068)은 이 세션에서 기능째 삭제됐다.** 그 화면과 대조하던 판정은 **역사적 기록으로만** 남기고, 살아 있는 형제는 **사이트 설정**(`site/SiteSettingsPage.tsx`)과 **API Key**(`api-keys/**`) 둘이다.

### 1.1 이 화면의 설계 원칙 (요구사항 — 명세에 고정한다)

| 원칙 | 내용과 이유 (코드 근거) |
|---|---|
| **저장된 시크릿은 폼에 채워지지 않는다** | `validation.ts:3-11` 이 선언한다: '폼의 `secret` 필드는 **저장된 시크릿이 아니라 "새로 넣을 값"** 이다. 빈 문자열 = 그대로 둔다(기존 유지) / 값이 있음 = 이 값으로 교체한다.' **채우면 DOM 에 평문이 살고, 그 순간 마스킹은 눈속임이 된다** |
| **모델은 `hasSecret` 불리언만 안다** | 화면·저장소가 저장된 시크릿의 값을 갖지 않는다(`data-source.ts:3-5` '저장소는 **평문 시크릿을 갖지 않는다** — `hasSecret` 불리언만 안다. 조회 응답에도 시크릿은 실리지 않는다') |
| **마스킹은 자릿수도 흘리지 않는다** | `MASKED_SECRET_TEXT = '••••••••••••'` — 고정 글리프. **last4 도 남기지 않는다**(`_shared/secret.ts:36-37` '식별이 이름으로 충분하다'). API Key 화면(`sk_test_••••0001` — `_shared/secret.ts:32-34`)과 다른 판정이며 근거가 코드에 있다 |
| **저장은 시크릿을 화면에서 지우는 시점이다** | `normalizeAfterSave`(`data-source.ts:73-83`)가 새로 넣은 평문을 비우고 `hasSecret: true` 로 바꾼다. '이걸 하지 않으면 저장한 뒤에도 입력칸에 평문이 남아 있고, 그 상태가 곧 새 기준선이 되어 **DOM 에 평문이 계속 산다**'(`:70-71`) |
| **비밀이 아닌 값을 비밀처럼 다루지 않는다** | 카카오 **네이티브 앱 키는 시크릿이 아니다** — '모바일 앱에 심겨 배포되므로 애초에 비밀이 될 수 없다. 그래서 평문으로 왕복하고 마스킹하지 않는다 — **비밀이 아닌 값을 비밀처럼 다루면 진짜 비밀의 취급이 헐거워진다**'(`validation.ts:317-324`). `normalizeAfterSave` 도 이 값과 표시 정책은 지우지 않는다(`data-source.ts:80-81`) |
| **필요 없는 최고 권한 자격증명을 받지 않는다** | 카카오 **어드민 키 입력란을 두지 않는다** — '로그인은 REST API 키 + 시크릿으로 끝난다. 필요 없는 최고 권한 자격증명을 받아 두면 유출 시 피해만 커진다'(`validation.ts:27-32`). **누가 다시 추가하려 하면 그 주석이 이유다** |
| **꺼진 제공자는 검증하지 않는다** | '쓰지 않을 값을 채우라고 요구하지 않는다'(`validation.ts:346-347`) |
| **형식으로 막지 않는다 — 알려 주기만 한다** | '시크릿·키의 길이나 문자셋은 검사하지 않는다. Google 은 `GOCSPX-` 접두어를 어디에서도 보장하지 않고, 카카오·네이버 키 형식은 문서화된 적이 없다 — **오늘 맞는 정규식이 내일 틀린다**'(`validation.ts:13-25`). 그래서 길이 상한만 막고 형식은 **경고**로만 말한다(EL-007.1). **예외는 Google Client ID 접미어 하나**(`.apps.googleusercontent.com`) — 문서화된 규약이라 막는다(`:21-24`) |
| **어긋날 수 없는 값은 입력받지 않는다** | iOS URL 스키마는 **입력이 아니라 Client ID 의 파생값**이다 — '자유 입력이면 예전 값을 남겨 둘 수 있고 그러면 Client ID 와 **조용히 어긋난다** — 어긋난 순간 iOS 로그인만 실패하고 그 실패는 이 화면 어디에도 드러나지 않는다'(`validation.ts:187-189`) |
| **Redirect URI 를 저장 시점에 막는다** | '이걸 저장 시점에 막지 않으면 실패는 **로그인 순간에, 사용자 앞에서** 난다'(`validation.ts:266`) |
| **Redirect URI 를 정규화하지 않는다** | '제공자의 매칭은 **대소문자와 끝 슬래시까지 포함한 정확한 문자열 일치**다. 친절하게 고쳐 주면 **오히려 로그인이 깨진다**'(`validation.ts:259-264`). 판정만 하고 값은 손대지 않는다 |
| **가짜 성공을 보여주지 않는다** | 연결 테스트가 **비활성** + 이유 문구. '눌러서 아무 일도 없거나 가짜 성공을 보여주는 것보다, 왜 못 쓰는지 적어 두는 편이 정직하다 (FEEDBACK-03: no-op 금지)'(`OAuthProviderCard.tsx:15-16`) |
| **자격증명과 화면 규칙을 섞지 않는다** | `kakaoTalkInAppLoginOnly` 는 **카카오가 주는 값이 아니라 우리 로그인 화면의 정책**이라 `providers[]` 밖 `display` 에 살고 별도 카드로 그린다 — '자격증명 옆에 두면 언젠가 **카카오 콘솔 어디에서 이 값을 받나요** 라는 질문으로 돌아온다'(`validation.ts:328-337` · `OAuthPage.tsx:335-339`) |
| **동시 편집을 덮어쓰지 않는다** | revision 토큰 기반 409 → 충돌 다이얼로그(`_shared/store.ts:124-126`). 사이트 설정과 같은 계약 |
