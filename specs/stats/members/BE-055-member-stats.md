# BE-055. 회원 통계 백엔드 기능 명세

## 1. 개요

| 항목 | 내용 |
|---|---|
| 대상 화면 | FS-055 회원 통계 (`/stats/members`) |
| 범위 | 회원 가입/탈퇴/누적 **집계 조회**와 등급 구성 스냅샷. 심(`// TODO(backend)`)이 있는 만큼만 계약으로 세운다 |
| 범위 밖 | **회원 개별 리소스 전체** — 이 화면은 `pages/members` 를 import 하지 않고(`MemberStatsPage.tsx:7-8`) 개별 회원을 열 수 없다(FS-055 §1.1). 회원 CRUD 는 **BE-003/BE-004 가 정본**이고 이 문서는 그것을 재정의하지 않는다. CSV 내보내기 — **클라이언트가 만든다**(§7.3) |
| 프론트 어댑터 | **없다.** `CrudAdapter` 계열을 쓰지 않는다 — `fetchMemberStats(query, signal)`(`data-source.ts:145-161`) 하나가 `_shared/mock.ts:45-54` 의 `loadStats` 를 부른다 |
| 도메인 타입 | `members/types.ts` — `MemberRow` · `MemberTierRow` · `MemberByChannel<T>` · `MemberStats` (**전부 클라이언트 선언**) |
| 검증 정본 | **해당 없음** — 입력 폼이 없다. 조회 조건 검증은 `_shared/period.ts:200-206` 이 클라이언트에서 한다 |
| 쓰기 | **0건.** 이 화면은 서버 상태를 바꾸지 않는다(FS-055 §4.1) |

> **에러 봉투·권한 모델 상속**: BE-003 §2·§2.1·§3 을 그대로 상속한다. 여기서 재정의하지 않는다.
>
> **⚠ 이 문서의 성격**: 심은 있으나 **프론트가 그 심대로 부르지 않는다**. 그 어긋남을 덮지 않고 §6.1 에 그대로 적고 §7 에서 판정한다. **심을 넘어서는 엔드포인트는 만들지 않는다.**

### 1.1 코드 대조 근거표

| 사실 | 근거 |
|---|---|
| **심이 2건 있다** | `members/data-source.ts:6-7` — `GET /api/stats/members?start&end&channel` · `GET /api/stats/members/tiers?start&end&channel` |
| **그런데 프론트 호출은 1건이다** | `MemberStatsPage.tsx:115-121` 이 `useStatsQuery([...], fetchMemberStats)` 하나만 건다. `fetchMemberStats`(`data-source.ts:145`)는 `loadStats` 를 **한 번** 부르고 `{daily, compareDaily, tiers, compareTiers}` 를 **함께** 돌려준다(`:149-157`) |
| **등급 구성이 서버에서 오지 않는다** | `data-source.ts:155-156` — `tiers: tiersOf(daily)` · `compareTiers: tiersOf(compareDaily)`. `tiersOf`(`:130-138`) → `tierRowsOf`(`:121-128`)는 **일자별 배열의 마지막 누적에 고정 비중을 곱하는 클라이언트 순수 함수**다 |
| **`channel` 이 서버로 가지 않는다** | `MemberStatsPage.tsx:118` 이 `fetchMemberStats({period, comparePeriod}, signal)` 로 부른다 — `channel` 이 인자에 없다. 쿼리 키에도 없다(`:116`). 응답이 **5경로를 모두** 채워 오고(`MemberByChannel` — `types.ts:76-79`) 화면이 하나를 고른다(`:124-125`) |
| **비교 기간을 실을 파라미터가 심에 없다** | 심은 `?start&end&channel` 뿐이다. 그런데 프론트는 `comparePeriod` 를 인자로 넘기고(`:118`) 응답에 `compareDaily`·`compareTiers` 를 기대한다(`types.ts:85,87`) |
| **데이터가 목(mock)에서 파생된다** | `mock.ts:45-54` `loadStats` = `wait(delay) → failIfRequested → build()`. `build()` 는 `dailyOf`(`data-source.ts:112-118`)이고 그 값은 `seededSeries`(`mock.ts:87-100`)의 시드 난수다. **서버 왕복이 하나도 없다** |
| **`'all'` 은 저장되지 않는다** | `types.ts:32-33` (`MemberJoinChannel = Exclude<MemberChannel,'all'>`) · `data-source.ts:90-117` (`totalRowsOf` 가 넷을 더한다) — '따로 만들면 합계와 경로별 값이 어긋나 관리자가 둘 다 못 믿는다' |
| 전용 어댑터 파일이 없다 | `stats/**` 에 `adapter.ts` 가 없다. `shared/crud` 에서 가져오는 것은 `parseFilter`·`useDebouncedSearch`·`failIfRequested`/`LATENCY_MS` 뿐이다 |

**결론**: **서버 집계 계약이 존재하지 않는다.** 심 2건은 표식이고, 현재 프론트의 실제 모양(**호출 1건 · 응답 4벌 × 5경로 · 등급/경로/비교가 전부 클라이언트 파생**)과 **맞지 않는다**. 이 문서는 심에 있는 것만 §4 에 세우고 어긋남은 §6.1 에 남긴 뒤 §7.1 에서 판정한다. **집계 엔드포인트를 지어내지 않는다.**

## 2. 공통 (상속)

- **에러 봉투**: BE-003 §2. ⚠ 이 화면은 봉투의 `code`·`fields`·`traceId` 를 **전부 버린다** — `queries.ts:47` 이 `isError` 하나로 한 문구를 만든다(§7.5).
- **권한**: BE-003 §3 상속. 조회뿐이므로 `admin`·`operator` 둘 다 읽는다. **⚠ 프론트 권한 리소스와 서버 권한이 다른 축이다 — §7.1.**
- **403 vs 404 은닉**: BE-003 §3.2 상속. **⚠ 이 화면은 그 정책의 경계에 선다** — BE-003 §3.2 는 '**개별 회원 리소스의 존재 자체가 개인정보**'라 404 로 은닉하라 하고 '**컬렉션 리소스의 존재는 비밀이 아니다**'라 403 을 주라 한다. **이 화면의 리소스는 집계이지 개별 회원이 아니다** → **403** 이 맞다(§7.4).
- **CSRF**: **해당 없음** — 쓰기 요청이 0건이다.
- **타임아웃**: 조회 5초 → 504 `REQUEST_TIMEOUT`. ⚠ 프론트에 클라이언트 상한이 없다(§7.6 #5).
- **프론트 역할 분기 없음**(FS-055 §4.1). 단 **`export` 액션은 프론트가 갈라 버튼을 숨긴다** — 서버 계약이 아니라 UX 다(§7.3).

### 3.1 서버가 줄 것 — `MemberRow` (심이 있는 만큼)

`members/types.ts:56-66`. **이 타입은 클라이언트가 선언했고 서버는 아직 모른다.**

| 필드 | 타입 | 의미 (`types.ts:3-7` 어휘) |
|---|---|---|
| `id` | string | 구간 식별자 — 일자 `'2026-07-16'` |
| `label` | string | 표시명 `'2026.07.16'` — ⚠ **표시 문자열이 데이터에 섞여 있다**(§7.4) |
| `joins` | number | 그 날 **가입을 완료한** 회원 수 |
| `withdrawals` | number | 그 날 탈퇴 처리된 회원 수 (**관리자 강제 탈퇴 포함**) |
| `cumulative` | number | 그 날 **마감 기준** 남아 있는 전체 회원 수. **날짜별 값을 더하면 안 된다**(`types.ts:64`) |

### 3.2 서버가 줄 것 — `MemberTierRow`

`members/types.ts:68-73`. `{ id: 'basic'|'silver'|'gold'|'vip', label, members }`. **기간 말 시점의 스냅샷이지 일별 흐름이 아니다**(`:68`).

> **불변식 세 개가 계약이어야 한다** — 픽스처는 지키지만 아무도 강제하지 않는다: ① `cumulative[n] === cumulative[n-1] + joins[n] - withdrawals[n]`(`data-source.ts:79`) ② `all` 의 각 필드 === 네 경로의 같은 필드의 합(`:96-108`) ③ `Σ tiers[].members === cumulative` (기간 말). 서버가 ①을 깨면 표의 '순증'과 '누적'이 서로를 부정하고, ②를 깨면 '전체'와 경로별 값이 어긋나며(그 순간 **관리자는 둘 다 믿지 않는다** — `:90`), ③을 깨면 등급 구성비의 합이 100% 가 아니게 된다. **화면은 셋 중 어느 것도 검증하지 않는다**(§7.6 #2).

## 4. 엔드포인트 명세

### BE-055-EP-01 · 회원 통계 조회

| 항목 | 내용 |
|---|---|
| 근거 (FS) | FS-055-EL-003 · EL-004 · EL-005.2 · EL-006 |
| 심 | `data-source.ts:6` — `// TODO(backend): GET /api/stats/members?start&end&channel` |
| 메서드·경로 | `GET /api/stats/members` |
| 권한 | 회원 통계 도메인 읽기(`admin`·`operator`) — BE-003 §3 상속. **⚠ §7.1 · §7.4** |
| 멱등성 | 멱등(GET) |
| 페이징 | **없음 — 기간 전량.** 페이지는 클라이언트가 자른다(`table.ts:54-57`) |
| 레이트리밋 | **미정** — 심에 명시가 없다 |

**쿼리 파라미터** (심이 이름만 정했다):

| 이름 | 타입 | 필수 | 근거 | 프론트가 보내는가 |
|---|---|---|---|---|
| `start` | `YYYY-MM-DD` (서울 달력일, inclusive) | O | `period.ts:19-22` | O |
| `end` | `YYYY-MM-DD` (서울 달력일, inclusive) | O | 같음 | O |
| `channel` | `'all'\|'email'\|'naver'\|'kakao'\|'google'` | ? | `types.ts:18-26` | **✕ — 보내지 않는다**(§6.1 · §7.2) |

**응답 200** — **미정.** 심은 응답 모양을 말하지 않는다. 프론트가 기대하는 것은 `MemberStats`(`types.ts:81-88`) = `{daily, compareDaily, tiers, compareTiers}` 이고 각각이 **경로 5벌**(`MemberByChannel`)인데, **그중 셋은 이 엔드포인트의 쿼리로 요청할 수단이 없다**(§6.1). 계약을 여기서 확정하지 않고 §7.2 로 이관한다.

**에러**: BE-003 §2.1 상속 — 400 `VALIDATION_FAILED`(§7.4) · 401 `UNAUTHENTICATED` · **403 `FORBIDDEN`**(컬렉션 — §7.4) · 429 `RATE_LIMITED` · 500 `INTERNAL_ERROR` · 504 `REQUEST_TIMEOUT`. **전부 FS-055-EL-006 한 문구로 수렴한다**(§7.5).

### BE-055-EP-02 · 등급 구성 조회 — **심은 있으나 프론트가 부르지 않는다**

| 항목 | 내용 |
|---|---|
| 심 | `data-source.ts:7` — `// TODO(backend): GET /api/stats/members/tiers?start&end&channel` |
| 프론트 호출 | **없다.** `data-source.ts:155-156` 이 `tiersOf(daily)` 로 **클라이언트에서 만든다** |

`tierRowsOf`(`:121-128`)는 **기간 말 누적**(`rows[rows.length-1]?.cumulative`)에 `TIER_WEIGHTS`(`:48-53` — basic 0.62 · silver 0.24 · gold 0.11 · vip 0.03)를 곱한다. 즉 **이것은 집계가 아니라 상수 분할이다** — 어느 경로를 고르든 어느 기간을 보든 **비율이 언제나 62/24/11/3** 이다(§7.2). 서버가 붙으면 이 함수는 **사라져야 하고** 그때 비로소 이 심이 실호출이 된다. **지금 계약을 확정하지 않는다.**

⚠ **심의 `?start&end` 가 이 축에 맞는지도 미정이다.** 등급 구성은 `types.ts:68` 이 말하듯 **'기간 말 시점의 스냅샷'** 이다 — 스냅샷에 `start` 가 왜 필요한가? '기간 말'만 필요하면 `?asOf=YYYY-MM-DD` 가 맞고, '기간 중 등급 변동'을 뜻하면 지금의 화면(막대 4줄)이 표현할 수 없는 것을 요구하는 셈이다. **축의 의미를 서버가 정해야 한다** — §7.2.

### BE-055-EP-03 · 비교 기간 조회 — **심 없음 (미정)**

화면은 비교 기간을 **1급 기능**으로 갖는다(기본값이 '직전 기간' — `period.ts:38`). 그런데 EP-01·EP-02 의 심에 `compareStart`/`compareEnd` 자리가 없고, 프론트는 픽스처가 같은 함수를 두 번 돌려 만든다(`data-source.ts:151,156`). **심이 없으므로 지어내지 않는다** — §7.2.

> ⚠ **이 화면에서 비교는 픽스처의 특별 취급을 받았다.** `data-source.ts:36-60` 이 `ANCHOR_DATE`·`BASE_MEMBERS`·`DAILY_NET_DRIFT` 로 누적 곡선을 기준일에서 흘려 만드는데, 주석(`:40-45`)이 그 이유를 밝힌다 — **'이 보정이 없으면 비교 기간(더 이른 기간)의 누적이 현재 기간과 같은 값에서 출발해 누적 회원 수의 증감이 언제나 0으로 뜬다. 비교가 본체인 화면에서 그것은 버그다.'** 즉 **누적의 비교는 '기간 밖의 상태'를 알아야 성립한다** — 서버 계약이 그것을 어떻게 줄지는 미정이다.

### BE-055-EP-04 · 기간 프리셋 · 영업 달력 — **심 없음 (미정)**

'최근 7일 = 오늘 포함 7일'(`period.ts:141-146`) · 월 말일 보정(`:100-106`) · '오늘'의 기준 시계가 전부 클라이언트다. §7.2.

### BE-055-EP-05 · CSV 내보내기 — **심 없음 (미정) · 서버 계약이 아니다**

`useCsvExport`(`useCsvExport.ts:39-93`)가 **이미 받은 배열**로 파일을 만든다. 요청이 나가지 않는다. §7.3.

## 5. 예외 매트릭스

**이 화면은 읽기 전용이라 쓰기 축(409·422)이 원리적으로 성립하지 않는다.** 프론트가 실제로 거는 조회는 EP-01 하나이므로 그 한 행을 9축 전수로 채우고, 심만 있는 EP-02 를 '도입 시' 관점으로 함께 채운다.

## 7. 핵심 판정

### 7.1 프론트 권한 리소스와 서버 권한이 다른 축이다 【보안 판정】

**프론트는 이 화면을 `page:/stats/members` 로 게이팅하고, 서버는 회원 통계 도메인 권한으로 게이팅한다.**

근거: `nav-config.ts:199` 에서 `/stats/members` 는 **자기 자신이 잎**이다. `permissions/resources.ts:84,92` 가 잎마다 `page:<to>` 리소스를 만들고 `AppShell.tsx:490-492` 의 `RequirePermission` → `resourceIdForPath` 가 그 잎의 read 를 확인한다. 프론트 권한의 단위는 **화면**, 서버는 **엔드포인트**다.

**⚠ 이 화면에는 그 불일치의 더 날카로운 버전이 있다 — 같은 '회원' 도메인에 두 개의 잎이 있다.**

`page:/users/members`(회원 목록)와 `page:/stats/members`(이 화면)는 **권한 매트릭스의 서로 다른 행**이다. 그래서 **'회원 목록 read = OFF, 회원 통계 read = ON'인 역할이 만들어질 수 있다.** 그 역할은 개별 회원을 못 보지만 **가입/탈퇴/누적/등급 구성을 본다.** 이것이 의도라면 좋은 설계다 — 마케터에게 개인정보 없이 지표만 주는 것이 정확히 그 그림이다. **그리고 이 화면은 그 분리를 실제로 지탱한다**: 개별 회원으로 내려갈 경로가 없고(FS-055 §1.1) `pages/members` 를 import 하지 않는다(`MemberStatsPage.tsx:7-8`) — **즉 이 잎을 켜는 것으로 개인정보가 새지 않는다.** 이 점에서 **예약 일정(BE-040 §7.1)과 정반대다**: 거기서는 달력 잎을 끄는 것이 아무것도 숨기지 못했다(같은 데이터가 예약 목록에 있었으므로). **여기서는 두 잎이 진짜로 다른 것을 보여준다.**

**판정 세 줄로 확정한다.**

1. **서버는 `GET /api/stats/members` 하나의 권한만 안다** — 회원 **통계** 읽기 권한. **회원 개별 읽기 권한(BE-003 §3)과 별개의 권한이어야 한다** — 그래야 프론트의 두 잎 분리가 서버에서도 성립한다. 이것을 하나로 합치면 '지표만 보는 역할'이 원리적으로 불가능해지고, 프론트의 `page:/stats/members` 는 다시 메뉴 숨김으로 전락한다.
2. **`page:/stats/members` 의 read 는 여전히 보안 경계가 아니다** — `RequirePermission.tsx:8-11` 이 명문화했듯 위조된 localStorage 로 우회된다(권한 모델이 `permission-store.ts` 의 localStorage 에 산다). 실제 차단은 서버의 몫이다.
3. **그래서 프론트 가드를 통과한 뒤 서버 403 을 받는 경로가 실재한다** — 그때 화면은 '통계를 불러오지 못했습니다.' 한 줄만 보이고(§7.5) **운영자는 권한 문제인지 서버 장애인지 알 수 없다.**

**⚠ 추가로 이 화면에는 게이트가 하나 더 있고 그것은 죽어 있다.** `StatsPageShell.tsx:95-104` 가 `can(navPageResourceId(route),'read')` 로 자기 403 을 그리는데 `AppShell` 이 **같은 리소스·같은 액션**을 먼저 막으므로 실앱에서 도달 불가다. 게다가 공유 `ForbiddenScreen`(`ErrorScreens.tsx:103-119`)과 달리 `role="alert"` 도 '대시보드로' 탈출 버튼도 없다. **설정 섹션은 이미 수렴했다**(`App.tsx:335-336`). **이관**: §7.6 #1.

### 7.2 서버 집계 계약이 없다 — 심을 넘어 지어내지 않는다 【정합 판정】

**화면이 보이는 값의 거의 전부가 클라이언트 파생이다**(§3). 서버가 줄 것은 **`daily` 행 배열**뿐이고 나머지는 그 위에서 계산된다. 심은 2건인데 실제 호출은 1건이고 파라미터도 어긋난다(§6.1).

**따라서 BE-055-EP-03(비교) · EP-04(프리셋) · EP-05(CSV)는 '심 없음(미정)'이고, EP-01 의 응답 계약과 EP-02 의 축 의미(`asOf` vs `start&end`)도 '미정'이다. 지어내지 않는다.**

**다만 부재의 결과는 기록한다** — 도입 결정은 백엔드 명세 이 한다:

| 사실 | 결과 |
|---|---|
| **누적 파생이 옳다 — 그리고 그것이 서버 계약을 요구한다** | `cumulativeOf`(`types.ts:114-120`)가 마지막 날의 마감값을 취한다(합이 **아니다**). ✔ 이 파생은 정확하다 — **형제 화면 FS-054 는 같은 자리에서 UV 를 더해 틀렸다**(BE-054 §7.6 #2). **그러나 옳은 파생이라도 원천이 필요하다**: `cumulative` 는 **기간 안의 데이터만으로 만들 수 없다**(기간 시작 시점의 누적을 알아야 한다). 픽스처는 `ANCHOR_DATE`/`BASE_MEMBERS`/`DAILY_NET_DRIFT` 로 그것을 지어낸다(`data-source.ts:36-60`) — **서버는 그 '기간 밖의 상태'를 진짜로 알아야 한다** |
| **`'all'` 이 누적까지 더한다** | `totalRowsOf`(`data-source.ts:96-109`)가 `cumulative` 를 네 경로에서 **합산한다**(`:107`). **회원 하나가 한 경로에만 속한다면 옳다** — 그리고 `types.ts:32-33` 의 모델(`MemberJoinChannel` = 가입 경로)이 그것을 전제한다. **하지만 그 전제가 계약으로 적혀 있지 않다.** 소셜 계정 연동으로 한 회원이 두 경로를 갖게 되면 '전체' 누적이 **중복 계상**된다 — FS-054 의 UV 합산과 같은 종류의 오류가 된다. §7.6 #7 |
| **등급 구성이 집계가 아니라 상수 분할이다** | `tierRowsOf`(`data-source.ts:121-128`)가 기간 말 누적에 `TIER_WEIGHTS`(62/24/11/3)를 곱한다 → **어느 경로·어느 기간이든 비율이 같다.** 등급별 증감(`ShareBarList.tsx:131`)은 누적 변화율을 네 번 반복할 뿐이고 '카카오 가입자는 실버가 유난히 많다'에 답할 수 없다. **픽스처의 성질이며 EP-02 가 실호출이 되면 해소된다** |
| 기간 상한이 없다 | `?start=1900-01-01&end=2100-01-01` → `eachDay`(`period.ts:59-65`)가 73,000 날짜 × **경로 4벌 + 합성 1벌** × **비교 2벌** = 73만 행. 지금은 클라이언트가 멈추고 **서버가 붙으면 서버가 멈춘다.** 도입 시 **범위 상한(예: 366일)이 400 `VALIDATION_FAILED` 로 강제돼야 한다** |
| 경로/축 전환에 재조회가 0이다 | **현재 설계의 장점이다** — 응답 한 벌에 4벌 × 5경로가 다 있어(`types.ts:81-88`) 토글이 요청을 만들지 않는다. 심대로 쪼개면 이 성질을 잃고 축별 로딩·부분 실패 표면이 새로 필요해진다(§7.6 #3) |

**도입 시 계약 조건**(지금 정하지 않고 조건만 남긴다): ① 기간은 **닫힌 구간**의 서울 달력일 문자열(`YYYY-MM-DD`). ② `start > end` 면 400 `VALIDATION_FAILED` + `fields[].name='end'`. ③ **범위 상한**을 두고 초과 시 400. ④ **`cumulative` 는 서버가 계산해 준다** — 기간 밖의 상태가 필요하므로 클라이언트가 만들 수 없다. ⑤ **회원이 여러 가입 경로를 가질 수 있는지 확정한다** — '아니오'가 계약이면 `'all'` 합산이 옳고, '예'면 `'all'` 을 **서버가 별도로 계산해 줘야 한다**(§7.6 #7). ⑥ 비교 기간은 **같은 엔드포인트의 선택적 쿼리**로 둔다 — 별 엔드포인트를 만들면 §7.1 의 권한 단위가 갈라진다. ⑦ 등급 축을 서버로 옮기면 **`tierRowsOf`·`TIER_WEIGHTS` 를 지운다**.

### 7.3 CSV 내보내기는 서버 계약이 아니다 — 그리고 이 화면에서는 그래도 된다 【보안 판정】

`useCsvExport`(`useCsvExport.ts:39-93`)가 **이미 받은 배열**로 파일을 만든다. **요청이 나가지 않는다.**

**(a) 내보내기 범위는 '현재 조건 전체'가 맞다 — 그리고 실제로 그렇다.** `MemberStatsPage.tsx:250-261` 이 축에 따라 `tierRows`(4행) 또는 `rows`(**페이지 자르기 전** 기간 일수)를 넘기고 `exportCount`(`:238`)가 그 수를 정직하게 말한다. **UTF-8 BOM** 이 붙고(`download.ts:16,38`) **값은 화면에 보이는 그대로**다 — `StatsColumn.csv` 가 `render` 와 같은 원천을 쓰도록 타입이 묶는다(`_shared/types.ts:44-53`). 등급별 CSV 는 **구성비까지 담는다**(`MemberStatsPage.tsx:194,216-221`) — 화면(막대)이 보여주는 값과 엑셀의 값이 같아야 하기 때문이다.

**(b) 그것은 '현재 조건 전체'가 이미 브라우저에 있기 때문이다.** 서버 페이징이 없어 전량이 메모리에 있으니 클라이언트 생성이 성립한다. **서버 페이징이 도입되는 순간 이 등식이 깨진다** — 그때 '현재 페이지만' 내보내게 되고 그것은 ERP-12 위반이다. 즉 **§7.2 의 페이징 도입은 내보내기 엔드포인트를 함께 요구한다.**

**(c) 개인정보 판정: 이 화면의 CSV 에 개인정보가 없다 — 그리고 그것은 우연이 아니라 설계다.** 일자별 CSV 5컬럼(일자·신규 가입·탈퇴·순증·누적)과 등급별 CSV 3컬럼(등급·회원 수·구성비)이 **전부 집계 수치**다. 개별 회원을 식별할 값이 하나도 없고, **애초에 이 화면이 개별 회원을 갖고 있지 않다** — `pages/members` 를 import 하지 않고(`MemberStatsPage.tsx:7-8`) 행 클릭 목적지가 없다(FS-055 §1.1). **회원 도메인의 화면인데 개인정보 표면이 0인 것이 이 화면의 핵심 성질이고, §7.1 의 두 잎 분리를 성립시키는 근거다.**

**(d) 그럼에도 권한 게이트는 옳다 — 그러나 실효는 없다.** `StatsPageShell.tsx:119` 가 `can(resourceId,'export')` 로 버튼을 **그리지 않는다**(비활성이 아니라 부재). `export` 는 실재하는 액션이다(`permissions/resources.ts:31,46`). **다만 이것은 서버 강제가 아니다** — 클라이언트 생성이라 **서버가 막을 지점이 없고** read 권한만 있으면 데이터는 이미 브라우저에 있다. **판정: 통계 CSV 에서 `export` 권한은 read 권한과 실질적으로 같다.** 둘을 다르게 설정할 수 있는 권한 화면은 **관리자에게 있지도 않은 통제를 있다고 말한다.** 이 화면의 데이터가 집계뿐이라 실피해는 없으나(c), **매출 통계(BE-056 §7.3)에서는 같은 구조가 재무 데이터에 걸린다** — §7.6 #8.

### 7.4 403 이지 404 가 아니다 — 회원 도메인의 은닉 정책이 여기엔 걸리지 않는다 【보안 판정】

**BE-003 §3.2 는 회원 도메인에 두 줄의 원칙을 세웠다**: ① 컬렉션의 존재는 비밀이 아니다 → **403** ② **개별 회원 리소스의 존재 자체가 개인정보다** → 회원 읽기 권한이 없는 주체에게 **404 `MEMBER_NOT_FOUND`** 로 은닉한다.

**이 화면은 그 경계에 선다** — 이름에 '회원'이 들어가고 리소스도 회원 도메인이지만, **다루는 것은 개별 회원이 아니라 기간의 집계다.** 그래서:

**판정**: `GET /api/stats/members` 는 **403 `FORBIDDEN`** 이다. 404 은닉을 적용하지 않는다. 근거 셋:

1. **은닉할 id 가 없다.** BE-003 §3.2 의 404 는 '회원 id 열거를 차단'하기 위한 것이다(`GET /api/members/:id` 표 행). 이 엔드포인트의 응답에는 **회원 id 가 하나도 없다** — 날짜와 수뿐이다(§3.1). 숨길 것이 없는 리소스를 숨기는 시늉은 비용만 남긴다.
2. **404 로 답하면 '없음'이 '실패'로 보인다.** 이 화면은 200 + 빈 배열을 `StatsEmpty`('집계된 가입 기록이 없습니다')로 정확히 그린다(`StatsTable.tsx:187`). 권한 부족에 404 를 주면 그 경로가 **'조회 실패' 배너**로 떨어지는데(`queries.ts:47` 이 status 를 안 본다 — §7.5), 그러면 **'권한 없음'과 '서버 장애'와 '데이터 없음'이 한 문구로 뭉개진다.**
3. **집계의 존재는 비밀이 아니다.** '이 쇼핑몰에 회원 통계라는 것이 있다'는 사실은 사이드바에 이미 적혀 있다.

> ⚠ **다만 집계도 정보를 흘린다.** '누적 회원 수 48,200명'은 그 자체로 영업 정보다. **그래서 403 은 '권한 없이 볼 수 있다'는 뜻이 아니다** — 이 화면 전체가 회원 통계 읽기 권한 뒤에 있어야 하며, 그것이 §7.1 이 요구하는 바다. 403 이냐 404 냐는 **거절의 형식**에 관한 판정이지 거절 여부가 아니다.

**표시 문자열 혼입**: `MemberRow.label`(`types.ts:61`)이 `'2026.07.16'` 같은 **표시 문자열**이고 `MemberTierRow.label`(`:71`)이 `'일반'`·`'VIP'` 다. 이것을 서버가 주면 **표시 규칙이 서버로 새어 나간다** — 날짜 표기나 등급 이름을 바꾸는 일이 백엔드 배포가 된다. **판정: 서버는 `id` 만 준다.** `label` 은 응답 필드가 아니다 — 클라이언트가 `id` 와 `MEMBER_TIERS`(`types.ts:49-54`)에서 만든다. (같은 이유로 **`400 VALIDATION_FAILED` 의 `fields[].name` 은 쿼리 파라미터명(`start`·`end`)이지 화면 라벨('조회 기간')이 아니다** — BE-003 §2 규약 그대로.)

### 7.5 실패의 입도가 봉투보다 거칠다 【정합 판정】

BE-003 §2 는 `code`·`fields`·`traceId` 를 **필수로** 싣게 한다. 그런데 `queries.ts:46-47` 은 그것을 **전부 버린다** — `query.isError` 하나로 '통계를 불러오지 못했습니다.' 가 된다. 주석은 '원문 에러 메시지를 그대로 노출하지 않는다(EXC-20)' 라 적었고 **그 절반은 옳다**(never-leak). 그러나 EXC-20 의 나머지 절반(**복사 가능한 참조 코드**)과 EXC-06(**status 별 뚜렷한 UX**)이 함께 사라졌다. 결과: 400·401·403·429·500·504 가 **같은 한 줄**이 되고, §7.1 이 지적한 '프론트 통과 후 서버 403' 과 §7.4 가 구분하려 한 '권한 없음 vs 데이터 없음'이 **화면에서 다시 뭉개진다.**

**판정**: 서버는 봉투를 규약대로 싣는다(BE-003 §2 — 재정의하지 않는다). **버리는 쪽은 프론트이고 고칠 곳도 프론트다** — `useStatsQuery` 가 status·`code`·`traceId` 를 함께 내면 화면이 분기할 수 있다. §7.6 #4.
