# BE-057. 주문 통계 백엔드 기능 명세

## 1. 개요

| 항목 | 내용 |
|---|---|
| 대상 화면 | FS-057 주문 통계 (`/stats/orders`) |
| 범위 | 주문 집계 조회 **1건**(일자별 주문·취소·반품·교환 + 상태 분포). 심에 있는 상태별 집계(EP-02)는 **화면이 부르지 않는다** — §7.2 |
| 범위 밖 | 주문의 조회·수정·상태 전이 계약 — 주문 관리의 정본이 갖는다. 이 화면은 **읽기 전용**이며 서버 쓰기 요청이 0건이다(FS-057 §4.1) |
| 프론트 어댑터 | `orders/data-source.ts` `fetchOrderStats(query, signal)` — **픽스처**. `_shared/mock.ts` `loadStats` 골격 위에 있다 |
| 도메인 타입 | `orders/types.ts` — `OrderStatus` · `OrderRow` · `OrderStats` · `OrderSegment`. `_shared/period.ts` — `StatsPeriod` |
| 검증 정본 | **해당 없음** — 입력 폼이 없다. 유일한 클라이언트 검증은 **기간 순서**(`period.ts:200-206` `periodErrorOf`)이며 서버에 나가지 않는다 |
| 백엔드 현황 | **없다.** 데이터는 결정론적 픽스처다(`mock.ts:56-100`). 이 문서는 **심 2건**을 근거로 계약을 서술하고, 심이 없는 것은 **'심 없음(미정)'** 으로 남긴다 |

>

| 사실 | 근거 |

**결론**: 주문 통계의 서버 계약은 **엔드포인트 1건**(EP-01)이다. 심에 적힌 EP-02 는 **화면이 소비하지 않으므로 계약으로 확정하지 않는다**(§7.2). 그 밖에 있어야 할 법해 보이는 것(집계 캐시·세그먼트 서버 필터·내보내기 엔드포인트)은 **심이 없으므로 지어내지 않는다** — §7.3 · §7.4.

## 2. 공통 (상속)

- **에러 봉투**: BE-003 §2 · §2.1 상속. 이 화면은 `code` 를 읽지 않고 status 도 가르지 않는다 — 모든 실패가 FS-057-EL-014 한 배너로 수렴한다(§7.5).
- **권한**: BE-003 §3 상속. 이 화면이 부르는 것은 집계 조회뿐이므로 **주문 도메인 읽기 권한**이 기준이다. **⚠ 프론트 권한 리소스(`page:/stats/orders`)와 서버 권한 축이 어긋난다 — §7.1 판정.**
- **403 vs 404**: BE-003 §3.2 상속. 이 엔드포인트는 **컬렉션 집계**이므로 권한 부족 시 **403**이다 — 은닉할 개별 리소스 id 가 응답에 없다.
- **CSRF**: **해당 없음** — 쓰기 요청이 0건이다(BE-003 §3.3 의 적용 대상이 아니다).
- **타임아웃**: 조회 5초 → 504. **⚠ 프론트에 클라이언트 타임아웃 상한이 없다**(`AbortSignal.timeout` 앱 전체 0건) — 서버가 응답하지 않으면 화면이 무한 대기한다(§7.6 #4).
- **프론트 역할 분기 없음**(FS-057 §4.1) — 단 `read`/`export` 권한 게이트는 있다(§7.1).

## 3. 데이터 계약

이 화면이 서버에 요구하는 것은 **일자별 주문 지표 한 벌**이다. 나머지는 전부 클라이언트 파생이다.

| 화면이 보이는 값 | 어디서 오는가 | 서버 계약인가 |
|---|---|---|
| 일자별 행(`OrderRow`) | EP-01 응답 | **O** — §3.1 |
| 비교 기간 일자별(`compareDaily`) | EP-01 응답(같은 호출) | **O** |
| KPI '주문 건수' | `sumOf(daily, ordersOfSegment(row, segment))` — **클라이언트 합계 + 클라이언트 세그먼트 필터** | ✕ |
| KPI '취소율'·'반품률' | `cancelRate`/`returnRate`(`types.ts:106-119`) — **클라이언트 계산** | ✕ (§7.4) |
| KPI 증감(▲12.3%) | `deltaOf(value, compareValue)`(`format.ts:91-101`) — **클라이언트 계산** | ✕ |
| 추이 차트 계열 | `stats.daily.map(pick)`(`OrderStatsPage.tsx:213`) — **클라이언트 사영** | ✕ |
| 상태별 구성비 | `statusTotal(stats.daily, status)`(`types.ts:101-103`) — **클라이언트 집계** | ✕ (§7.2) |
| 구성비 %(14.2%) | `shareOf(value, total)`(`format.ts:123-125`) — **클라이언트 계산** | ✕ |
| 표의 정렬·페이지 | `sortRows`/`pageSlice`(`table.ts:15-57`) — **클라이언트** | ✕ (§7.3) |
| 기간(프리셋 계산) | `resolvePreset(preset, today)`(`period.ts:175-177`) · `today = formatDate(new Date())` KST | ✕ — **클라이언트가 '오늘'을 정한다**(§7.7) |
| CSV 파일 | `toCsvText` + `downloadCsv` — **브라우저 생성** | ✕ (§7.3) |

### 3.1 EP-01 응답 타입 — `OrderStats`

`types.ts:58-79` 가 정본이다. 서버는 이 모양을 그대로 낸다.

| 필드 | 타입 | 필수 | 규약 |
|---|---|---|---|
| `daily[]` | `OrderRow[]` | O | 요청 기간의 **모든 날**이 하루도 빠짐없이 들어야 한다 — §3.2 |
| `daily[].id` | string | O | 구간 식별자 = 일자 `'YYYY-MM-DD'`(`types.ts:59-60`). 표의 기본 정렬 키다(`OrderStatsPage.tsx:68,132`) |
| `daily[].label` | string | O | 표시용 `'2026.07.16'`. **서버가 낸다** — 클라이언트가 `formatDayLabel(day)` 로 만들고 있으나(`data-source.ts:73`) 그것은 픽스처의 편의다. §7.8 #1 |
| `daily[].orders` | number | O | 그날 접수된 주문 건수. **`statusCounts` 의 합과 같아야 한다** — §3.2 |
| `daily[].canceled` | number | O | **배송중 이전에** 멈춘 건수 |
| `daily[].returned` | number | O | **배송중 이후에** 되돌아온 건수 |
| `daily[].exchanged` | number | O | 교환 건수. 표에만 뜨고 KPI 에는 없다 |
| `daily[].statusCounts` | `Record<OrderStatus, number>` | O | 상태 7종 **전부** 키가 있어야 한다(`types.ts:71`). 0 이어도 키를 빼지 않는다 — `statusCounts[segment]` 가 `undefined` 를 만나면 KPI 가 `NaN` 이 된다(`types.ts:97`) |
| `compareDaily[]` | `OrderRow[] \| null` | O | 비교 기간의 같은 모양. **비교 안 함이면 `null`**(빈 배열이 아니다 — 카드가 증감을 그리지 않는 신호다: `types.ts:15-16`) |

**`OrderStatus`** = `'pending'|'preparing'|'holding'|'waiting'|'shipping'|'delivered'|'confirmed'`(`types.ts:20-21`). **카페24 어휘 고정이다** — 서버가 다른 이름을 쓰면 같은 주문이 두 이름을 갖는다(`types.ts:3-6`).

### 3.2 서버가 반드시 지켜야 하는 불변식

| # | 불변식 | 깨지면 무엇이 보이는가 |
|---|---|---|
| 1 | **`Σ statusCounts = orders`** (행마다) | 구성비 막대의 합계가 100% 가 아니게 된다. 픽스처는 마지막 상태(구매확정)로 반올림 오차를 흡수해 이를 지킨다(`data-source.ts:31-53`) — **서버도 같은 보정을 해야 한다.** 화면은 검증하지 않는다 |
| 2 | **`daily` 에 기간의 모든 날이 있다** | 차트 x축이 날짜를 건너뛴다. `labels` 가 `daily.map(row=>row.label)`(`OrderStatsPage.tsx:207`)이라 **빠진 날은 축에서 사라질 뿐 눈에 띄지 않는다** — 7월 13일이 없으면 12일 옆에 14일이 붙는다. 0건인 날도 `orders:0` 행으로 채워야 한다 |
| 3 | **`compareDaily` 의 길이 = `daily` 의 길이** | 두 계열이 같은 x축에 겹쳐 그려진다(`StatsTrendCard.tsx:92-93`). 길이가 다르면 비교선이 축을 벗어나거나 짧게 끊긴다 — 화면은 검증하지 않는다 |
| 4 | **`statusCounts` 에 상태 7종 키가 전부 있다** | `ordersOfSegment` 가 `undefined` 를 반환 → KPI 가 `NaN건` 으로 렌더된다(`types.ts:97` 은 인덱스 접근을 가드하지 않는다) |
| 5 | **음수를 내지 않는다** | 취소율이 음수가 되고 증감 색이 뒤집힌다. `rateOf`(`types.ts:91-93`)는 분모 0 만 가드한다 |
| 6 | **`canceled`/`returned` 가 '배송중' 경계로 갈린다** | KPI 힌트가 그 경계를 운영자에게 약속한다(`OrderStatsPage.tsx:48-49`). 서버가 다른 기준으로 세면 **화면의 설명이 거짓말이 된다** — 이것이 §7.4 가 '지표 정의는 서버가 소유해야 한다'고 판정하는 이유다 |

## 4. 엔드포인트 명세

### BE-057-EP-01 · 주문 집계 조회

| 항목 | 내용 |
|---|---|
| 근거 (FS) | FS-057-EL-009, EL-010, EL-012, EL-013, EL-014 |
| 근거 (심) | `orders/data-source.ts:6` `GET /api/stats/orders?start&end&status` |
| 메서드·경로 | `GET /api/stats/orders` |
| 권한 | 주문 도메인 읽기 (BE-003 §3 상속) — **⚠ §7.1** |
| 멱등성 | 멱등(GET) |
| 페이징 | **없음** — 기간의 전체 일자를 낸다. 표의 페이지는 클라이언트가 자른다(§7.3) |
| 레이트리밋 | **미정** — 심에 없다. 화면의 재조회 빈도는 §7.6 #5 참조 |

**쿼리 파라미터**

| 이름 | 타입 | 필수 | 규약 | 현재 프론트 |
|---|---|---|---|---|
| `start` | `YYYY-MM-DD` | O | **서울 기준 달력 날짜**, 포함 | `params.period.start` |
| `end` | `YYYY-MM-DD` | O | 포함. `end >= start`(클라이언트가 먼저 막는다 — `period.ts:200-206`) | `params.period.end` |
| `status` | `OrderStatus` | X | **심에 있으나 프론트가 보내지 않는다** — §7.3 판정 | 미사용 |

> **비교 기간은 쿼리에 없다.** 프론트가 `comparePeriod` 를 계산해(`period.ts:186-193`) 한 호출에 넘기지만, **심에는 그 파라미터가 없다**(`?start&end&status` 뿐). 즉 심대로면 비교 기간을 받으려면 **두 번 부르거나** 심에 없는 파라미터를 더해야 한다. **지어내지 않는다** — §7.8 #2 로 이관한다.

**응답 200** — `OrderStats`(§3.1). 불변식 §3.2 를 지킨다.

**에러**: 400(`VALIDATION_FAILED` — `start`/`end` 형식·순서) · 401 · **403**(컬렉션) · 429 · 500 · 504. **전부 FS-057-EL-014 한 배너로 수렴한다** — 화면이 status 를 구분하지 않는다(§7.5).

### BE-057-EP-02 · 상태별 집계 조회 — **심은 있으나 화면이 부르지 않는다**

| 항목 | 내용 |
|---|---|
| 근거 (심) | `orders/data-source.ts:7` `GET /api/stats/orders/status?start&end` |
| 화면 소비 | **없다** — `OrderStats`(`types.ts:74-79`)에 대응 필드가 없고, 상태별 구성비는 `statusTotal(stats.daily, …)`(`OrderStatsPage.tsx:230`)로 EP-01 응답에서 파생된다 |

**판정은 §7.2** — 심이 있다는 이유로 계약을 확정하지 않는다. 화면이 부르지 않는 엔드포인트를 계약으로 굳히면 **같은 값의 두 번째 원천**이 생긴다.

---

### BE-057-EP-03 · 세그먼트(상태) 서버 필터 — **심의 파라미터가 미사용 (미정)**

`?status=` 가 심에 있으나 프론트가 보내지 않는다(`queryKey` 에 `segment` 없음 — `OrderStatsPage.tsx:142`). 세그먼트를 서버로 옮길지는 §7.3 판정.

### BE-057-EP-04 · 내보내기 엔드포인트 — **심 없음 (미정)**

CSV 를 브라우저가 만든다(`useCsvExport.ts:59-76`). 서버 내보내기 심이 없다 — §7.3 판정.

### BE-057-EP-05 · 지표 정의(상태 목록·취소/반품 경계) 조회 — **심 없음 (미정)**

상태 7종과 취소/반품 경계가 클라이언트 상수다(`types.ts:29-37`, `:13-16`). 서버가 이를 내려주는 심이 없다 — §7.4 판정.

### 6.1 어댑터 본문 요구사항

`fetchOrderStats`(`data-source.ts:82-92`)의 **안쪽만** 교체한다. 골격은 그대로 둔다.

4. **`compareDaily: null` 과 `[]` 를 구분해 낸다.** `null` 은 '비교 안 함'이고 `[]` 는 '비교 기간에 데이터가 없음'이다 — 화면이 `null` 로 증감 렌더 여부를 가른다(`OrderStatsPage.tsx:163`).

## 7. 핵심 판정

### 7.1 권한 리소스가 프론트와 서버에서 어긋난다 【보안 판정】

**프론트는 이 화면을 `page:/stats/orders` 로 게이팅하고, 서버는 주문 도메인 권한으로 게이팅한다 — 두 축이 독립적이다.**

근거: `StatsPageShell.tsx:92` `navPageResourceId(route)` → `resources.ts:66-68` `page:${to}` → **`page:/stats/orders`**. 이 리소스는 `nav-config.ts:201` 의 잎에서 파생된다(`resources.ts:88-95` 가 nav-config 를 훑어 만든다). 주문 **관리**(주문 도메인의 정본 화면)는 **다른 행**이다.

권한 매트릭스가 잎마다 독립이므로 **'주문 통계 read = ON, 주문 관리 read = OFF'인 역할이 만들어질 수 있다.** 그 역할은:

1. 사이드바에서 '주문 통계'를 본다.
2. `/stats/orders` 에 들어간다 — `RequirePermission` 도 `StatsPageShell.tsx:95` 도 `page:/stats/orders` 만 보고 **통과시킨다**.
3. 화면이 `GET /api/stats/orders` 를 부른다.
4. 서버가 주문 도메인 읽기 권한을 보고 **403** 을 준다.
5. 화면은 '통계를 불러오지 못했습니다.' 만 보인다 — **운영자는 권한 문제인지 서버 장애인지 알 수 없고**(§7.5), 관리자는 권한을 켰다고 믿는데 화면이 영원히 실패한다.

반대 방향이 **더 중요하다**. **'주문 통계 read = OFF, 주문 관리 read = ON'** 이면 프론트는 통계를 막지만 — **막은 것이 무엇인지 따져야 한다.** 이 화면은 주문 관리에 없는 정보를 갖는다: **취소율·반품률의 추세**와 **상태별 구성비**다. 주문 관리는 주문 하나하나를 보여줄 뿐 '이번 주 반품률이 지난주의 두 배'를 말하지 않는다. 즉 **BE-040 §7.1(달력)과 달리, 통계를 끄는 것은 실제로 무언가를 숨긴다** — 개별 주문이 아니라 **영업 성과의 요약**을.

**그러나 그 요약은 개별 주문에서 재구성 가능하다.** 주문 관리에서 전량을 내려받아 세면 같은 숫자가 나온다(그것이 정확히 이 화면이 하는 일이다 — 클라이언트 집계). 따라서:

**판정**: 계약은 다음을 확정한다.

1. **서버는 `GET /api/stats/orders` 를 주문 도메인 읽기 권한으로 게이팅한다.** 통계용 별도 권한을 만들지 않는다 — 만들면 같은 사실에 두 개의 진실이 생기고, '통계는 되는데 주문은 안 되는' 역할이 서버에서도 가능해진다.
2. **`page:/stats/orders` 의 read 는 보안 경계가 아니라 메뉴 가시성이다.** 데이터의 경계는 주문 도메인 권한 하나다. 권한 관리 화면이 이 사실을 관리자에게 알려야 한다 — 그렇지 않으면 **관리자가 '통계를 껐으니 성과가 안 보인다'고 믿는데, 주문 관리 권한이 있는 사람은 여전히 셀 수 있다.**
3. **프론트 가드가 데이터 리소스를 함께 확인하거나**, 권한 모델이 두 잎의 종속 규칙을 갖거나 — 둘 중 하나가 필요하다. 지금은 어느 쪽도 없다.

**⚠ `export` 권한은 더 약하다.** `StatsPageShell.tsx:119` 가 `can(resourceId,'export')` 로 내보내기 버튼을 숨기지만, **CSV 는 이미 브라우저에 있는 데이터로 만든다**(§7.3) — 서버 요청이 없다. 즉 **`export` 를 끄는 것은 버튼을 숨길 뿐 데이터 반출을 막지 못한다.** 화면을 볼 수 있는 사람은 개발자 도구·복사·스크린샷으로 같은 것을 가져간다. **`export` 권한이 통제라고 믿어서는 안 된다** — 그것이 통제가 되려면 내보내기가 서버 엔드포인트여야 한다(§7.3 · §7.8 #5).

**이관**: A63(권한 모델) · A11 — §7.8 #4 · #5.

### 7.2 상태별 집계 심을 계약으로 확정하지 않는다 【정합 판정】

`data-source.ts:7` 에 `GET /api/stats/orders/status?start&end` 심이 있다. **그러나 화면이 부르지 않는다.**

- `OrderStats`(`types.ts:74-79`)에 대응 필드가 없다.
- `OrderStatsPage.tsx:225-235` 가 `statusTotal(stats.daily, status.id)` 로 **EP-01 응답에서 합쳐 낸다.**
- `:224` 주석이 결정을 명시한다: *'상태별은 기간 전체를 상태로 합친 것이다 — 일자별 원천에서 낸다(같은 값의 두 번째 원천을 두지 않는다)'*.

**판정**: **EP-02 를 계약으로 확정하지 않는다.** 근거는 심의 부재가 아니라 **소비의 부재**이며, 그 소비 부재가 **의도된 설계**라는 것이 코드에 적혀 있다.

이 결정이 옳은 이유를 계약으로 못 박는다:

1. **두 원천은 반드시 갈라진다.** `daily[].statusCounts` 와 `/status` 응답이 같은 기간을 각자 세면, 경계(자정·타임존·집계 시점)에서 어긋나는 날이 온다. 그러면 **같은 화면의 표와 막대가 다른 말을 한다** — 운영자는 어느 쪽을 믿을지 알 수 없다.
2. **불변식 §3.2 #1(`Σ statusCounts = orders`)이 이미 그 일을 한다.** 상태 분포가 일자별 응답 안에 있으므로 합이 맞는지 **한 응답 안에서** 검증 가능하다. 분리하면 그 검증이 불가능해진다.
3. **성능 논거가 약하다.** 기간이 90일이어도 `daily` 는 90행 × 상태 7개 = 630 숫자다. 별도 왕복을 정당화하지 못한다.

**따라서 심 `:7` 은 삭제 대상이다** — 코드에 남은 심이 '연결해야 할 것'으로 읽혀 미래의 누군가가 EP-02 를 붙이면 위 세 문제가 그대로 발생한다. §7.8 #3.

**단, 규모가 바뀌면 판정도 바뀐다**: 기간 상한이 없어(§7.6 #3) 1년(365행)을 고를 수 있고, 상태가 늘면(카페24는 7종이나 도메인이 확장되면) `daily` 가 두꺼워진다. **그때는 EP-02 가 아니라 EP-01 의 사영(projection) 파라미터**(예: `?include=status-only`)가 답이다 — 원천을 쪼개지 않고 응답만 좁힌다.

**이 화면의 거의 모든 숫자가 클라이언트에서 만들어진다**(§3 표). 서버가 내는 것은 일자별 원시 행뿐이고, KPI·비율·구성비·증감·정렬·페이지·CSV 가 전부 브라우저에서 계산된다.

|---|---|---|
| 주문 상태 7종·순서 | `types.ts:29-37` | 표시 순서 = 주문이 흐르는 순서 |
| **취소 vs 반품 = '배송중' 경계** | `types.ts:13-16`(주석) | KPI 힌트가 **화면에 그대로 적는다**: '배송중 이전에 멈춘 주문입니다. 배송이 시작된 뒤로는 반품으로 잡힙니다.'(`OrderStatsPage.tsx:48`) |
| 취소율 = 취소 ÷ 주문 | `types.ts:106-111` | KPI 힌트: '취소 건수 ÷ 주문 건수. 배송중 이전에 멈춘 주문만 셉니다.'(`:192`) |
| 반품률 = 반품 ÷ 주문 | `types.ts:114-119` | KPI 힌트(`:201`) |

**문제**: 화면은 `canceled` 필드가 '배송중 이전에 멈춘 건수'라고 **운영자에게 문구로 약속한다**. 그런데 그 판정은 **서버가 한다**(필드를 채우는 쪽). 서버가 다른 기준으로 세면 — 예컨대 '취소 요청 시각'이 아니라 '취소 승인 시각'으로 가르거나, 배송중 이후 취소를 `canceled` 에 넣으면 — **화면의 힌트가 거짓말이 되고, 아무도 알아채지 못한다.** 화면에는 검증할 수단이 없고 서버에는 그 문구를 지킬 의무가 계약으로 적혀 있지 않다.

**판정**: **지표 정의의 소유자는 서버다. 계약이 다음을 확정한다.**

1. **§3.2 #6 을 계약 불변식으로 승격한다** — `canceled` 는 배송 시작(상태가 `shipping` 이 되는 시점) **이전에** 종료된 주문만, `returned` 는 **이후에** 되돌아온 주문만 센다. 이 경계는 카페24의 처리 기준이며 **협상 대상이 아니다**(`types.ts:13-16`).
2. **상태 어휘를 서버가 고정한다** — `OrderStatus` 7종의 id 는 카페24 어휘에 고정된다. 서버가 상태를 추가·개명하면 **클라이언트의 `Record<OrderStatus, number>` 접근이 `undefined` 를 만나 KPI 가 `NaN` 이 된다**(§3.2 #4). 상태 추가는 **깨는 변경**이다.
3. **비율은 서버가 내지 않는다** — 취소율·반품률은 클라이언트가 계산한다(`types.ts:106-119`). 서버가 별도로 내면 반올림이 갈려 **표의 '취소율' 열과 KPI '취소율'이 다른 값을 보인다**(둘 다 클라이언트 계산인 지금은 갈릴 수 없다). 서버는 **원시 건수만** 낸다.
4. **⚠ 그러나 정의가 코드 주석에만 있는 것은 계약이 아니다.** '배송중 경계'는 지금 `types.ts` 주석과 화면 힌트 문구에만 존재한다 — 서버 팀이 읽을 곳이 없다. **이 문서 §3.2 #6 이 그 자리다.** 도메인이 이 경계를 확정하기 전까지 KPI 힌트는 **검증되지 않은 약속**이다 — §7.8 #6.

### 7.5 실패가 status 를 구분하지 않고 참조 코드도 없다 【보안·운영 판정】

`queries.ts:46-47`:
```
// 원문 에러 메시지를 그대로 노출하지 않는다 — 서버 내부를 흘리지 않는다 (EXC-20)
error: query.isError ? '통계를 불러오지 못했습니다.' : '',
```

**잘한 절반**: 서버 원문·스택을 노출하지 않는다. BE-003 §2 의 '`message` 에 내부 정보 노출 금지'를 프론트에서 한 번 더 막는다 — 서버가 실수로 흘려도 화면에 닿지 않는다. **이 판정은 유지한다.**

**나머지 절반이 비었다**:

1. **401/403/429/500/504 가 한 문구다.** 에러 타입은 status 를 지니는데(`shared/errors/http-error.ts`) 이 화면이 읽지 않는다. §7.1 이 그린 시나리오 — 프론트 가드를 통과한 뒤 서버 403 — 에서 운영자는 **원인을 알 방법이 없다.** 권한 문제와 서버 장애가 같은 화면이다.
2. **`traceId` 를 표시하지 않는다.** BE-003 §2 가 `traceId` 를 필수로 두고 '500 문의 시 사용자가 그대로 전달한다'고 규정하는데, **이 화면에는 전달할 코드가 없다.** 봉투가 실어 보낸 것을 화면이 버린다.
3. **429 의 `Retry-After` 를 읽지 않는다.** '다시 시도'를 즉시 다시 누를 수 있어 레이트리밋을 더 때린다.

**판정**: 서버는 BE-003 §2 를 그대로 지킨다(`code` · `traceId` 필수 · `Retry-After`). **소비하지 않는 것은 프론트의 결함이며 서버 계약을 바꿀 이유가 아니다** — 봉투를 줄이면 나중에 프론트가 고쳐질 때 되돌려야 한다. §7.8 #7.

### 7.7 '오늘'을 클라이언트가 정한다 — 그러나 이번엔 옳게 되어 있다 【정합 판정】

프리셋('최근 7일')은 **클라이언트가 계산한다**: `useStatsParams.ts:162` `formatDate(new Date())` → `resolvePreset(preset, today)`(`period.ts:175-177`) → `start`/`end`.

**이것이 FS-040(예약 달력)의 ERP-09 gap 과 다른 점을 명확히 한다**:

| 축 | 예약 달력(FS-040) | 이 화면 |
|---|---|---|
| '오늘'의 기준 | **브라우저 로컬 TZ**(`toDateString(new Date())` + `calendar.ts` 의 로컬 getter) | **KST 고정** — `formatDate` 가 `Asia/Seoul` Intl 포매터다(`shared/format.ts:63`) |
| 달력 산술 앵커 | 브라우저 로컬 | **UTC 정오** 한 벌(`period.ts:73,77,97,104`) |
| 자정 넘김 | 첫 렌더에 고정 → '오늘'이 어제가 된다 | **마운트 시 고정**(`useStatsParams.ts:162` `useState(() => …)`) — 같은 성질이다 |

즉 **타임존 축은 수렴됐고**(도쿄에서 봐도 서울의 '오늘'이다), **자정 넘김만 남는다**: 화면을 열어 둔 채 자정을 넘기면 `today` 가 어제를 가리켜 '최근 7일'이 하루 밀린다. 다만 `:159-161` 주석이 그 선택의 근거를 밝힌다 — 매 렌더 `new Date()` 를 부르면 **의존성이 매번 달라져 무한 재조회**가 된다. **고정이 옳고, 자정 넘김은 별도 해소가 필요한 잔여다**(예: 날짜 변경 감지 후 갱신).

**판정**: **서버는 '오늘'을 정하지 않는다.** 프리셋은 클라이언트 관심사이고, 서버는 `start`/`end` 를 **받아서** 처리한다.

**단 계약이 확정할 것**: **`start`/`end` 는 Asia/Seoul 기준 달력 날짜다.** 서버가 이를 UTC 로 해석하면 하루가 밀린다 — 한국 시각 7월 17일 08:00 의 주문은 UTC 로 7월 16일 23:00 이다. `?start=2026-07-17&end=2026-07-17` 은 **KST 07-17 00:00:00 ~ 07-17 23:59:59.999** 를 뜻한다. 이 한 줄이 없으면 통계 전체가 하루씩 어긋난다. §7.8 #9.
