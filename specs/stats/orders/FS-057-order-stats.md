### 1.1 이 화면의 지표 모델

| 항목 | 정의 | 코드 |
|---|---|---|
| 주문 상태 | 카페24 어휘 그대로 **7종** — 입금전·배송준비중·배송보류·배송대기·배송중·배송완료·구매확정. 표시 순서 = 주문이 흐르는 순서 | `types.ts:29-37` (`ORDER_STATUSES`) |
| 취소·반품·교환 | **상태가 아니라 별도 CS 흐름**이다. 한 주문이 상태 하나만 갖는다는 전제를 지키려고 상태 목록에 끼우지 않고 각자 건수로 센다 | `types.ts:8-11` |
| 취소 vs 반품의 경계 | **'배송중' 시점**. 배송중 이전에 멈추면 취소, 이후에 돌아오면 반품 — 카페24의 처리 기준 | `types.ts:13-16` · KPI 힌트 `OrderStatsPage.tsx:48-49` |
| 세그먼트 | 전체 + 상태 7종 = 8. KPI '주문 건수'와 추이의 '주문' 계열만 좁힌다 | `types.ts:42-45` (`ORDER_SEGMENTS`) |
| 드릴다운 축 | 일자별(표) · 상태별(구성비 막대) 2종 | `types.ts:52-55` (`ORDER_BREAKDOWNS`) |
| 취소율 · 반품률 | 취소(반품) 건수 ÷ 주문 건수 × 100. **분모 0 이면 0**(0으로 나누지 않는다) | `types.ts:91-93,106-119` |
| 상태 분포의 합 | `statusCounts` 의 합 = `orders` 여야 구성비 합계가 100% 가 된다. 픽스처가 마지막 상태(구매확정)로 반올림 오차를 흡수해 이를 지킨다 | `types.ts:70-71` · `data-source.ts:31-53` |
| 기간 | 서울 기준 달력 날짜 구간(양 끝 포함). 프리셋 7종 + 직접 입력 | `_shared/period.ts:18-22,134-166` |
| 비교 기간 | 없음 / 직전 기간 / 전년 동기. **기본값 = 직전 기간** | `_shared/period.ts:32-38,186-193` |

> **세그먼트는 KPI·추이만 좁히고 표·구성비 막대는 좁히지 않는다.** `OrderStatsPage.tsx:155,218` 이 `ordersOfSegment(row, segment)` 로 KPI '주문 건수'와 추이 '주문' 계열에만 세그먼트를 먹이고, 일자별 표(`:337` `rows={stats.daily}`)와 상태별 막대(`:227-233`)는 **세그먼트와 무관하게 전량**을 그린다. 즉 '배송보류'를 고른 채 표를 보면 표의 '주문 건수' 열은 전체 주문이다 — 화면에 그 차이를 알리는 문구가 없다. §7 #2
