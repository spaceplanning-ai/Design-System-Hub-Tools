# BE-056. 매출 통계 백엔드 기능 명세

## 1. 개요

| 항목 | 내용 |
|---|---|
| 대상 화면 | FS-056 매출 통계 (`/stats/revenue`) |
| 범위 | 결제·환불·순매출·과세 구분의 **집계 조회**. 심(`// TODO(backend)`)이 있는 만큼만 계약으로 세운다 |
| 범위 밖 | 주문·결제 원장(ledger) 자체 · 정산 · PG 수수료 · 환불 사유 — 심이 없고 화면도 요구하지 않는다. CSV 내보내기 — **클라이언트가 만든다**(§7.3) |
| 프론트 어댑터 | **없다.** `CrudAdapter` 계열을 쓰지 않는다 — `fetchRevenueStats(query, signal)`(`data-source.ts:144-158`) 하나가 `_shared/mock.ts:45-54` 의 `loadStats` 를 부른다 |
| 도메인 타입 | `revenue/types.ts` — `RevenueRow` · `RevenueByMethod` · `RevenueStats` · `PayMethod` (**전부 클라이언트 선언**) |
| 검증 정본 | **해당 없음** — 입력 폼이 없다. 조회 조건 검증은 `_shared/period.ts:200-206` 이 클라이언트에서 한다 |
| 쓰기 | **0건.** 이 화면은 서버 상태를 바꾸지 않는다(FS-056 §4.1) |
| **이 문서의 무게** | **이 화면의 데이터는 돈이다.** 다른 통계와 같은 구조인데 **결과가 다르다** — CSV 가 재무 데이터 전량이고(§7.3), 지표 정의가 세무와 부딪히며(§7.4), 반올림이 검산을 깨뜨린다(§7.6 #9) |

> **에러 봉투·권한 모델 상속**: BE-003 §2·§2.1·§3 을 그대로 상속한다. 여기서 재정의하지 않는다.
>
> **⚠ 이 문서의 성격**: 심은 있으나 **프론트가 그 심대로 부르지 않는다**. 그 어긋남을 덮지 않고 §6.1 에 그대로 적고 §7 에서 판정한다. **심을 넘어서는 엔드포인트는 만들지 않는다.**

### 1.1 코드 대조 근거표

| 사실 | 근거 |
|---|---|
| **심이 2건 있다** | `revenue/data-source.ts:10-11` — `GET /api/stats/revenue?start&end&method` · `GET /api/stats/revenue/methods?start&end` |
| **그런데 프론트 호출은 1건이다** | `RevenueStatsPage.tsx:181-187` 이 `useStatsQuery([...], fetchRevenueStats)` 하나만 건다. `fetchRevenueStats`(`data-source.ts:144`)는 `loadStats` 를 **한 번** 부르고 `{daily, compareDaily, byMethod}` 를 **함께** 돌려준다(`:148-154`) |
| **결제수단별 합계가 서버에서 오지 않는다** | `data-source.ts:153` — `byMethod: methodTotalsOf(daily)`. `methodTotalsOf`(`:126-137`)는 **일자별 배열을 수단마다 접는 클라이언트 순수 함수**다 |
| **`method` 가 서버로 가지 않는다** | `RevenueStatsPage.tsx:184` 이 `fetchRevenueStats({period, comparePeriod}, signal)` 로 부른다 — `method` 가 인자에 없다. 쿼리 키에도 없다(`:182`). 응답이 **5수단을 모두** 채워 오고(`RevenueByMethod` — `types.ts:69-73`) 화면이 하나를 고른다(`:190`) |
| **비교 기간을 실을 파라미터가 심에 없다** | 심은 `?start&end&method` 뿐이다. 그런데 프론트는 `comparePeriod` 를 인자로 넘기고(`:184`) 응답에 `compareDaily` 를 기대한다(`types.ts:79`) |
| **데이터가 목(mock)에서 파생된다** | `mock.ts:45-54` `loadStats` = `wait(delay) → failIfRequested → build()`. `build()` 는 `dailyOf`(`data-source.ts:117-123`)이고 그 값은 `seededSeries`(`mock.ts:87-100`)의 시드 난수다. **서버 왕복이 하나도 없다** |
| **`'all'` 은 저장되지 않는다** | `types.ts:33-34` (`RevenuePayMethod = Exclude<PayMethod,'all'>`) · `data-source.ts:95-115` (`totalRowsOf` 가 넷을 더한다) — '따로 만들면 합계와 수단별 값이 어긋나 관리자가 둘 다 못 믿는다' |
| **순매출의 정의가 코드 한 곳에만 있다** | `data-source.ts:47-71` `rowOf` — 순매출·과세 구분을 만드는 **유일한 자리**다(`:6-8` 이 그 이유를 기록: '두 표의 합계가 어긋나서야 발견된다') |
| **부가세 고지가 클라이언트 상수다** | `RevenueStatsPage.tsx:48-54` `VAT_NOTICE` — JSX 상수. 서버가 정의를 소유하지 않는다(§7.4) |
| 전용 어댑터 파일이 없다 | `stats/**` 에 `adapter.ts` 가 없다 |

**결론**: **서버 집계 계약이 존재하지 않는다.** 심 2건은 표식이고, 현재 프론트의 실제 모양(**호출 1건 · 응답 3벌 × 5수단 · 수단/비교/과세 구분이 전부 클라이언트 파생**)과 **맞지 않는다**. 이 문서는 심에 있는 것만 §4 에 세우고 어긋남은 §6.1 에 남긴 뒤 §7.2 에서 판정한다. **집계 엔드포인트를 지어내지 않는다.**

## 2. 공통 (상속)

- **에러 봉투**: BE-003 §2. ⚠ 이 화면은 봉투의 `code`·`fields`·`traceId` 를 **전부 버린다**(`queries.ts:47` — §7.5).
- **권한**: BE-003 §3 상속. 조회뿐이므로 `admin`·`operator` 둘 다 읽는다. **⚠ 프론트 권한 리소스와 서버 권한이 다른 축이고, 이 화면에서는 그 불일치가 재무 데이터에 걸린다 — §7.1.**
- **403 vs 404 은닉**: BE-003 §3.2 상속. 이 화면의 리소스는 **집계 컬렉션**이므로 **403** 이다 — 은닉할 개별 id 가 없다(§7.4).
- **CSRF**: **해당 없음** — 쓰기 요청이 0건이다.
- **타임아웃**: 조회 5초 → 504 `REQUEST_TIMEOUT`. ⚠ 프론트에 클라이언트 상한이 없다(§7.6 #5).
- **프론트 역할 분기 없음**(FS-056 §4.1). 단 **`export` 액션은 프론트가 갈라 버튼을 숨긴다** — 서버 계약이 아니라 UX 다(§7.3).

### 3.1 서버가 줄 것 — `RevenueRow` (심이 있는 만큼)

`revenue/types.ts:47-67`. **이 타입은 클라이언트가 선언했고 서버는 아직 모른다.**

| 필드 | 타입 | 의미 (`types.ts:3-8` 어휘 — 카페24 정의) |
|---|---|---|
| `id` | string | 구간 식별자 — 일자 `'2026-07-16'` 또는 결제수단 `'card'` |
| `label` | string | 표시명 `'2026.07.16'` · `'신용카드'` — ⚠ **표시 문자열이 데이터에 섞여 있다**(§7.4) |
| `paymentTotal` | number(원) | 기간 안에 결제가 **완료된** 금액의 합 |
| `refundTotal` | number(원) | 기간 안에 환불 **처리된** 금액의 합 |
| `netRevenue` | number(원) | **결제합계 − 환불합계.** 음수가 될 수 있다(§7.6 #3) |
| `orderCount` | number | 결제가 완료된 주문 건수 |
| `taxable` / `taxFree` / `zeroRated` | number(원) | 순매출의 과세 / 면세 / 영세 구분. **셋의 합 === `netRevenue`** |

> **불변식 넷이 계약이어야 한다** — 픽스처는 지키지만 아무도 강제하지 않는다: ① `netRevenue === paymentTotal - refundTotal` ② `taxable + taxFree + zeroRated === netRevenue`(`data-source.ts:57-59` 가 과세를 '나머지 전부'로 두어 반올림 오차를 흡수한다) ③ `all` 의 각 필드 === 네 수단의 같은 필드의 합(`:96-114`) ④ `byMethod[m]` === 그 수단 일자별의 합(`:126-137`). **화면은 넷 중 어느 것도 검증하지 않는다** — 깨지면 표의 과세+면세+영세가 순매출과 어긋나고 '전체'와 수단별 값이 갈라지는데 **화면은 아무 경고 없이 그린다**(§7.6 #2). **이 화면의 숫자는 돈이고 관리자가 검산한다** — 검산이 안 맞으면 신뢰가 깨진다.

## 4. 엔드포인트 명세

### BE-056-EP-01 · 매출 통계 조회

| 항목 | 내용 |
|---|---|
| 근거 (FS) | FS-056-EL-004 · EL-005 · EL-006 · EL-007 |
| 심 | `data-source.ts:10` — `// TODO(backend): GET /api/stats/revenue?start&end&method` |
| 메서드·경로 | `GET /api/stats/revenue` |
| 권한 | 매출 통계 도메인 읽기(`admin`·`operator`) — BE-003 §3 상속. **⚠ §7.1** |
| 멱등성 | 멱등(GET) |
| 페이징 | **없음 — 기간 전량.** 페이지는 클라이언트가 자른다(`table.ts:54-57`) |
| 레이트리밋 | **미정** — 심에 명시가 없다 |

**쿼리 파라미터** (심이 이름만 정했다):

| 이름 | 타입 | 필수 | 근거 | 프론트가 보내는가 |
|---|---|---|---|---|
| `start` | `YYYY-MM-DD` (서울 달력일, inclusive) | O | `period.ts:19-22` | O |
| `end` | `YYYY-MM-DD` (서울 달력일, inclusive) | O | 같음 | O |
| `method` | `'all'\|'card'\|'transfer'\|'vbank'\|'easy'` | ? | `types.ts:19-27` | **✕ — 보내지 않는다**(§6.1 · §7.2) |

**응답 200** — **미정.** 심은 응답 모양을 말하지 않는다. 프론트가 기대하는 것은 `RevenueStats`(`types.ts:75-82`) = `{daily, compareDaily, byMethod}` 이고 앞 둘이 **수단 5벌**(`RevenueByMethod`)인데, **그중 둘은 이 엔드포인트의 쿼리로 요청할 수단이 없다**(§6.1). 계약을 여기서 확정하지 않고 §7.2 로 이관한다.

**에러**: BE-003 §2.1 상속 — 400 `VALIDATION_FAILED`(§7.4) · 401 `UNAUTHENTICATED` · **403 `FORBIDDEN`**(컬렉션) · 429 `RATE_LIMITED` · 500 `INTERNAL_ERROR` · 504 `REQUEST_TIMEOUT`. **전부 FS-056-EL-007 한 문구로 수렴한다**(§7.5).

### BE-056-EP-02 · 결제수단별 합계 조회 — **심은 있으나 프론트가 부르지 않는다**

| 항목 | 내용 |
|---|---|
| 심 | `data-source.ts:11` — `// TODO(backend): GET /api/stats/revenue/methods?start&end` |
| 프론트 호출 | **없다.** `data-source.ts:153` 이 `methodTotalsOf(daily)` 로 **클라이언트에서 만든다** |

`methodTotalsOf`(`:126-137`)는 각 수단의 일자별 배열을 접어 `rowOf(method, payMethodLabel(method), Σpayment, Σrefund, Σorders)` 를 만든다. **즉 이 축은 EP-01 응답의 순수한 재집계이지 새 데이터가 아니다** — 서버가 EP-01 을 제대로 주면 **EP-02 는 필요 없을 수도 있다**(클라이언트 재집계가 정확하다). **심이 있다는 것이 필요하다는 뜻은 아니다** — §7.2.

⚠ **심의 파라미터에 `method` 가 없다**(`?start&end` 뿐) — EP-01 에는 있는데. 수단별 합계를 수단으로 좁히는 것이 이상하므로 **일관돼 보이나**, 화면은 `RevenueStatsPage.tsx:278-281` 에서 **세그먼트로 그 한 줄만 남기는 필터를 클라이언트에서 한다**. 서버가 그것을 해야 하는지는 **미정**.

---

### BE-056-EP-03 · 비교 기간 조회 — **심 없음 (미정)**

화면은 비교 기간을 **1급 기능**으로 갖는다(기본값이 '직전 기간' — `period.ts:38`). 그런데 EP-01·EP-02 의 심에 `compareStart`/`compareEnd` 자리가 없고, 프론트는 픽스처가 같은 함수를 두 번 돌려 만든다(`data-source.ts:152`). **심이 없으므로 지어내지 않는다** — §7.2.

### BE-056-EP-04 · 기간 프리셋 · 영업 달력 · 마감 — **심 없음 (미정)**

'최근 7일 = 오늘 포함 7일'(`period.ts:141-146`) · 월 말일 보정(`:100-106`) · '오늘'의 기준 시계가 전부 클라이언트다. ⚠ **매출에는 '마감'이 있다** — 그 날의 결제가 언제 확정되는가(자정? 정산일?)는 도메인 규칙인데 **심도 없고 화면도 모른다**. §7.4.

### BE-056-EP-05 · 과세 구분 판정 — **심 없음 (미정) · 그런데 클라이언트가 하고 있다**

`data-source.ts:39-41,55-59` 가 `TAX_FREE_SHARE = 0.12` · `ZERO_RATED_SHARE = 0.03` 을 순매출에 곱해 과세/면세/영세를 만든다. **이것은 세무 판정이지 표시 포맷이 아니다** — §7.4 판정.

### BE-056-EP-06 · CSV 내보내기 — **심 없음 (미정) · 서버 계약이 아니다**

`useCsvExport`(`useCsvExport.ts:39-93`)가 **이미 받은 배열**로 파일을 만든다. 요청이 나가지 않는다. §7.3.

## 5. 예외 매트릭스

**이 화면은 읽기 전용이라 쓰기 축(409·422)이 원리적으로 성립하지 않는다.** 프론트가 실제로 거는 조회는 EP-01 하나이므로 그 한 행을 9축 전수로 채우고, 심만 있는 EP-02 를 '도입 시' 관점으로 함께 채운다.

### 6.2 어댑터 본문 요구사항

교체 지점은 **`data-source.ts:148-154` 의 `build` 콜백** 하나다. 서버를 붙일 때 지켜야 할 것:

> ① **`signal` 을 반드시 흘려보낼 것**(`loadStats(scope, signal, ...)` → `wait(delay, signal)` — `mock.ts:51`). 빠뜨리면 이탈한 화면의 실패 배너가 뜬다.
> ② **`?fail=`·`?empty=`·`?delay=` 재현 경로를 살릴 것**(`mock.ts:24-34,52-53`) — STATE-01/02 회귀 테스트(`stats-screens.test.tsx`)의 유일한 손잡이다.
> ③ **응답 범위를 좁히지 말 것.** KPI 가 전부 `daily` 전량 위에서 계산된다 — 응답이 잘리면 **결제합계·순매출이 조용히 과소 집계된다.** 표에는 페이지가 있지만 **KPI 는 없다**(기간 전체의 합이다). **금액에서 과소 집계는 '조금 틀린 것'이 아니라 '틀린 것'이다.**
> ④ **§3.1 의 네 불변식을 지킬 것.** 화면은 하나도 검증하지 않는다.
> ⑤ **`netRevenue` 와 과세 구분을 서버가 줄 것**(§7.2 · §7.4) — 클라이언트 뺄셈·상수 비율을 남겨 두면 정의가 두 곳에 생긴다.
> ⑥ **`'all'`·`byMethod` 를 서버가 주기로 하면 `totalRowsOf`·`methodTotalsOf` 를 지울 것** — 남겨 두면 합계가 두 곳에서 나온다.

## 7. 핵심 판정

### 7.1 프론트 권한 리소스와 서버 권한이 다른 축이다 — 그리고 여기서는 돈이 걸린다 【보안 판정】

**프론트는 이 화면을 `page:/stats/revenue` 로 게이팅하고, 서버는 매출 통계 도메인 권한으로 게이팅한다.**

근거: `nav-config.ts:200` 에서 `/stats/revenue` 는 **자기 자신이 잎**이다. `permissions/resources.ts:84,92` 가 잎마다 `page:<to>` 리소스를 만들고 `AppShell.tsx:490-492` 의 `RequirePermission` → `resourceIdForPath` 가 그 잎의 read 를 확인한다. 프론트 권한의 단위는 **화면**, 서버는 **엔드포인트**다.

**판정 세 줄로 확정한다.**

1. **서버는 `GET /api/stats/revenue` 하나의 권한만 안다** — 매출 통계 읽기 권한. 화면별 권한을 만들지 않는다.
2. **`page:/stats/revenue` 의 read 는 보안 경계가 아니다** — `RequirePermission.tsx:8-11` 이 명문화했듯 **위조된 localStorage 로 우회된다**(권한 모델이 `permission-store.ts` 의 localStorage 에 산다). **이 화면에서 그것은 다른 통계보다 무겁다**: 우회의 대가가 '방문자 수를 봤다'가 아니라 **'회사의 매출·환불·객단가 전량을 봤다'** 이다. 실제 차단은 전적으로 서버의 몫이며, **서버가 이 엔드포인트를 열어 두면 프론트 가드는 아무것도 지키지 못한다.**
3. **그래서 프론트 가드를 통과한 뒤 서버 403 을 받는 경로가 실재한다** — 그때 화면은 '통계를 불러오지 못했습니다.' 한 줄만 보이고(§7.5) **운영자는 권한 문제인지 서버 장애인지 알 수 없다.**

**⚠ 추가로 이 화면에는 게이트가 하나 더 있고 그것은 죽어 있다.** `StatsPageShell.tsx:95-104` 가 `can(navPageResourceId(route),'read')` 로 자기 403 을 그리는데 `AppShell` 이 **같은 리소스·같은 액션**을 먼저 막으므로 실앱에서 도달 불가다. 게다가 공유 `ForbiddenScreen`(`ErrorScreens.tsx:103-119`)과 달리 `role="alert"` 도 '대시보드로' 탈출 버튼도 없다. **설정 섹션은 이미 수렴했다**(`App.tsx:335-336`). **이관**: §7.6 #1.

### 7.2 서버 집계 계약이 없다 — 심을 넘어 지어내지 않는다 【정합 판정】

**화면이 보이는 값의 거의 전부가 클라이언트 파생이다**(§3). 서버가 줄 것은 **`daily` 행 배열**뿐이고 나머지는 그 위에서 계산된다. 심은 2건인데 실제 호출은 1건이고 파라미터도 어긋난다(§6.1).

**따라서 BE-056-EP-03(비교) · EP-04(프리셋/마감) · EP-05(과세 구분) · EP-06(CSV)은 '심 없음(미정)'이고, EP-01 의 응답 계약과 EP-02 의 필요성도 '미정'이다. 지어내지 않는다.**

**다만 부재의 결과는 기록한다** — 도입 결정은 A63 이 한다:

| 사실 | 결과 |
|---|---|
| **객단가 파생이 옳다** | `averageOrderValue`(`types.ts:107-116`)가 **합계끼리 나눈다** — 날짜별 객단가를 단순 평균하면 '**결제 1건뿐인 날이 100건인 날과 같은 무게**'를 가져 값이 조용히 틀어진다(주석이 그것을 밝힌다). ✔ **형제 화면 FS-054 는 같은 자리에서 UV 를 더해 틀렸다**(BE-054 §7.6 #2) — 이 화면은 함정을 피했다 |

| **그러나 정의가 프론트 코드에만 있다** | '순매출 = 결제합계 − 환불합계'가 `data-source.ts:54` 한 줄이고 `types.ts:3-8` 이 그 어휘를 소유한다. **같은 지표를 다른 소비자(정산·리포트·세무 대리인 파일)가 요구하면 정의가 두 벌이 되고, 두 벌은 반드시 갈라진다** — 그것이 `types.ts:10-12` 가 경고한 바로 그 사고다('같은 기간의 매출이 두 값으로 보고된다') |

| **기간 상한이 없다** | `?start=1900-01-01&end=2100-01-01` → `eachDay`(`period.ts:59-65`)가 73,000 날짜 × **수단 4벌 + 합성 1벌** × **비교 2벌** = 73만 행. 지금은 클라이언트가 멈추고 **서버가 붙으면 서버가 멈춘다.** 도입 시 **범위 상한(예: 366일)이 400 `VALIDATION_FAILED` 로 강제돼야 한다** |

### 7.3 CSV 내보내기는 서버 계약이 아니다 — 그리고 **이 화면에서는 그것이 문제다** 【보안 판정】

**이 판정은 BE-054 §7.3 · BE-055 §7.3 과 구조가 같고 결론이 다르다.** 방문자·회원 통계에서는 '집계뿐이라 실피해가 없다'로 끝났다. **여기서는 끝나지 않는다.**

`useCsvExport`(`useCsvExport.ts:39-93`)가 **이미 받은 배열**로 파일을 만든다. **요청이 나가지 않는다.** 따라서:

**(a) 내보내기 범위는 '현재 조건 전체'가 맞고 실제로 그렇다.** `RevenueStatsPage.tsx:300-303` 이 `rows`(= **페이지 자르기 전** 배열)를 넘기고 `exportCount` 가 그 수를 정직하게 말한다. UTF-8 BOM(`download.ts:16,38`) · 값은 화면에 보이는 그대로(`_shared/types.ts:44-53` 이 `csv` 와 `render` 를 같은 원천에 묶는다) · 진행률·취소(`useCsvExport.ts:66-69`). **ERP-12 의 문자를 전부 지킨다.**

**(b) 그런데 그 파일의 내용이 무엇인가.** 일자별 9컬럼 — 일자 · **결제합계 · 환불합계 · 순매출 · 결제건수 · 객단가 · 과세 · 면세 · 영세**. **이것은 회사의 재무 데이터 전량이다.** 개인정보는 없다(개별 주문·고객이 없다 — FS-056 §7 #19 가 '주문 상세로 내려갈 경로가 없다'고 기록). **그러나 개인정보만이 보호 대상은 아니다** — 매출·환불·객단가·과세 구분은 **영업비밀**이고, 유출 시 손해가 개인정보와 다른 방식으로 크다.

**(c) 그래서 이 화면에서는 세 가지가 따라온다.**

**① 내보내기 감사가 원리적으로 불가능하다.** CSV 가 **클라이언트에서** 생성되므로 **서버에 '누가 언제 무엇을 내보냈는가'가 남지 않는다.** 서버 로그에는 `GET /api/stats/revenue` 한 줄만 있고, 그 운영자가 화면만 봤는지 전 기간 매출을 엑셀로 떨어뜨려 나갔는지 **구별할 수 없다.** BE-040 §7.3a 가 예약 달력에서 지적한 '접근 범위와 로그 입도의 어긋남'과 같은 종류인데, **여기서는 export 라는 별개의 행위가 통째로 로그 밖에 있다.**

**② `export` 권한이 실질 통제가 아니다.** `StatsPageShell.tsx:119` 가 `can(resourceId,'export')` 로 버튼을 **그리지 않는다**(비활성이 아니라 부재 — 셸 주석 `:11-12`). `export` 는 실재하는 액션이다(`permissions/resources.ts:31,46` — 'CSV/엑셀 다운로드'). **그러나 클라이언트 생성이라 서버가 막을 지점이 없다** — read 권한만 있으면 데이터는 이미 브라우저에 있고, 버튼을 숨기는 것은 **콘솔에서 우회된다**(더 간단히는 화면을 복사하면 된다). **판정: 통계 CSV 에서 `export` 권한은 read 권한과 실질적으로 같다.** 그런데 권한 화면은 둘을 **따로 켜게 해 놓았다** — **관리자가 '이 역할은 매출을 볼 수는 있지만 내려받을 수는 없다'고 믿게 만든다. 그 믿음은 틀렸다.** 방문자·회원 통계에서는 이것이 이론적 결함이었지만 **여기서는 실질 통제의 부재를 통제로 표시하는 것**이다.

**③ 부가세 고지가 파일에 따라가지 않는다.** 화면의 상주 Alert(`RevenueStatsPage.tsx:48-54`)는 '**이 통계는 쇼핑몰 운영 참고용이며, 국세청 신고 등 제출용 자료로는 사용하실 수 없습니다**' 라 말한다. **그런데 `toCsvText`(`shared/download.ts:31-33`)는 헤더 + 행만 쓴다** — 고지가 들어가지 않는다. **화면을 떠난 숫자는 경고 없이 돌아다니고, 그 숫자가 신고에 옮겨 적히는 것이 정확히 고지가 막으려던 사고다.** 파일명(`stats-revenue-daily-2026-07-16.csv`)도 그 사실을 말하지 않는다.

**판정**: 서버는 `GET /api/stats/revenue` 응답을 **매출 통계 읽기 권한자에게만** 준다(BE-003 §3 상속). 통계라는 이유로 완화하지 않는다. **그리고 위 세 가지는 클라이언트 CSV 를 유지하는 한 해소되지 않는다** — 감사·권한 강제·고지 첨부는 전부 **서버가 파일을 만들 때에만** 가능하다. **심이 없으므로 export 엔드포인트를 정의하지 않는다** — 결정이 나면 §7.2 의 페이징·범위와 함께 설계한다(§7.6 #7).

### 7.4 과세 구분은 세무 판정이지 표시 포맷이 아니다 【보안·정합 판정】

`data-source.ts:39-41,55-59`:

```
TAX_FREE_SHARE = 0.12 · ZERO_RATED_SHARE = 0.03
taxFree = round(netRevenue × 0.12) · zeroRated = round(netRevenue × 0.03) · taxable = netRevenue − taxFree − zeroRated
```

**클라이언트가 상수 비율로 세 구분을 만든다.** 픽스처로서는 그럴듯한 모양을 낼 뿐이지만, **화면은 그것을 '과세 (원)'·'면세 (원)'·'영세 (원)' 라는 이름으로 표에 세우고 CSV 로 떨어뜨린다**(`RevenueStatsPage.tsx:106-131,143`). **즉 지어낸 수가 세무 용어를 달고 파일로 나간다.**

**세 가지가 따라온다.**

**(a) 이것은 상품 속성에서 나와야 한다.** 면세(도서·농축수산물)와 영세(수출)는 **상품마다 정해져 있고**(`types.ts:14-15` 가 그 어휘를 적는다) 주문의 품목 구성에 따라 달라진다. **비율이 아니라 합계다** — 어떤 날은 면세가 0일 수 있고 어떤 날은 절반일 수 있다. **지금 화면은 어느 날이든 12%/3% 를 보인다.**

**(b) 그래서 이 축은 실호출 없이는 의미가 없다.** 방문자 통계의 시간대별(BE-054-EP-02)도 지어낸 곡선이지만 **그것은 '몇 명'이라 틀려도 운영 판단이 흔들리는 정도다.** 여기서는 **'과세 매출이 얼마인가'** 이고, 그 수는 **부가세와 직결된다.** 고지(`RevenueStatsPage.tsx:52-53`)가 '제출용으로 쓸 수 없다'고 말하는 것이 이 화면을 방어하지만, **고지는 CSV 에 따라가지 않는다**(§7.3c③).

**(c) 판정**: **서버가 `taxable`/`taxFree`/`zeroRated` 를 계산해 준다.** 클라이언트 비율(`TAX_FREE_SHARE`·`ZERO_RATED_SHARE`)은 **픽스처와 함께 지워져야 한다** — 남겨 두면 '세무 판정이 프론트 상수에 있다'가 계약이 된다. 서버가 주기 전까지 **이 세 컬럼은 픽스처의 장식이며, 그 사실이 문서(여기)와 화면(고지)에 함께 적혀 있어야 한다**. §7.6 #8.

> ⚠ **부가세 고지 자체도 클라이언트 상수다**(`RevenueStatsPage.tsx:48-54` — JSX). 세법이 바뀌거나 통계 정의가 바뀌면 **프론트 배포가 필요하다.** 고지의 내용은 **서버가 소유한 지표 정의에서 나오는 것이 맞을 수 있으나 심이 없다** — §7.6 #10.

**표시 문자열 혼입**: `RevenueRow.label`(`types.ts:52`)이 `'2026.07.16'`·`'신용카드'` 다. 이것을 서버가 주면 **표시 규칙이 서버로 새어 나간다** — 날짜 표기나 결제수단 이름을 바꾸는 일이 백엔드 배포가 된다. **판정: 서버는 `id` 만 준다.** `label` 은 응답 필드가 아니다 — 클라이언트가 `id` 와 `PAY_METHODS`(`types.ts:21-27`)에서 만든다(`payMethodLabel` — `:37-39` 이 이미 그 함수다). (같은 이유로 **`400 VALIDATION_FAILED` 의 `fields[].name` 은 쿼리 파라미터명(`start`·`end`)이지 화면 라벨이 아니다** — BE-003 §2 규약 그대로.)

**마감이 계약에 없다**: 매출에는 '그 날이 언제 닫히는가'가 있다(자정? 정산일?). `useStatsParams.ts:162` 의 '오늘'은 **브라우저 시계**이고 마감을 모른다. **오늘이 진행 중인 날이면 '오늘 매출'은 계속 자란다** — 화면은 그것을 확정된 수처럼 보인다. **심이 없으므로 여기서 확정하지 않는다** — §7.6 #11.

### 7.5 실패의 입도가 봉투보다 거칠다 — 매출에서는 그 대가가 다르다 【정합 판정】

BE-003 §2 는 `code`·`fields`·`traceId` 를 **필수로** 싣게 한다. 그런데 `queries.ts:46-47` 은 그것을 **전부 버린다** — `query.isError` 하나로 '통계를 불러오지 못했습니다.' 가 된다. 주석은 '원문 에러 메시지를 그대로 노출하지 않는다(EXC-20)' 라 적었고 **그 절반은 옳다**(never-leak — 매출 화면에서 서버 내부를 흘리는 것은 특히 나쁘다). 그러나 EXC-20 의 나머지 절반(**복사 가능한 참조 코드**)과 EXC-06(**status 별 뚜렷한 UX**)이 함께 사라졌다.

**이 화면에서 그 대가가 특히 크다**: 매출이 안 보이면 운영자는 **즉시 에스컬레이션한다**('매출이 안 나와요'). 그때 붙일 `traceId` 가 없으면 개발자가 로그와 상관지을 수 없고, 401/403/429/500/504 가 한 문구라 **'권한이 빠졌나 서버가 죽었나'조차 전화로 물어야 한다.**

**판정**: 서버는 봉투를 규약대로 싣는다(BE-003 §2 — 재정의하지 않는다). **버리는 쪽은 프론트이고 고칠 곳도 프론트다** — `useStatsQuery` 가 status·`code`·`traceId` 를 함께 내면 화면이 분기할 수 있다. §7.6 #4.
