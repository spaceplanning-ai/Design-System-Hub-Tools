# BE-058. 유입 분석 백엔드 기능 명세

## 1. 개요

| 항목 | 내용 |
|---|---|
| 대상 화면 | FS-058 유입 분석 (`/stats/traffic`) |
| 범위 | 유입 집계 조회 **1건**(일자별 유입·구매·매출 + 채널 5종 분해 + 검색엔진별 + 랜딩페이지별). 심은 **3건**이나 프론트는 **한 번만 부른다** — §7.2 |
| 범위 밖 | 주문·매출의 정본 계약(주문 통계 BE-057 · 매출 통계 BE-056). 이 화면은 **읽기 전용**이며 서버 쓰기 요청이 0건이다(FS-058 §4.1) |
| 프론트 어댑터 | `traffic/data-source.ts` `fetchTrafficStats(query, signal)` — **픽스처**. `_shared/mock.ts` `loadStats` 골격 위에 있다 |
| 도메인 타입 | `traffic/types.ts` — `TrafficChannel` · `TrafficMetrics` · `TrafficRow` · `TrafficDayRow` · `TrafficStats` · `TrafficSegment`. `_shared/period.ts` — `StatsPeriod` |
| 검증 정본 | **해당 없음** — 입력 폼이 없다. 유일한 클라이언트 검증은 **기간 순서**(`period.ts:200-206`)이며 서버에 나가지 않는다 |
| 백엔드 현황 | **없다.** 데이터는 결정론적 픽스처다(`mock.ts:56-100`). 이 문서는 **심 3건**을 근거로 계약을 서술하고, 심이 없는 것은 **'심 없음(미정)'** 으로 남긴다 |

>

### 1.1 코드 대조 근거표 (HEAD = `a5c2639` 기준)

**결론**: 유입 분석의 서버 계약은 **엔드포인트 1건**(EP-01)이다. 심이 3건인 것은 **응답의 세 부분을 가리킬 뿐 요청 3회를 뜻하지 않는다** — 프론트가 한 번 부르고 한 객체를 받는다. 심대로 쪼개면 **축 전환마다 왕복이 생긴다**(지금은 0회) — §7.2. 그 밖의 것(세그먼트 서버 필터·서버 집계·내보내기 엔드포인트)은 **심이 없거나 미사용이므로 지어내지 않는다** — §7.3.

## 2. 공통 (상속)

- **에러 봉투**: BE-003 §2 · §2.1 상속. 이 화면은 `code` 를 읽지 않고 status 도 가르지 않는다 — 모든 실패가 FS-058-EL-017 한 배너로 수렴한다(§7.6).
- **권한**: BE-003 §3 상속. **⚠ 이 화면의 권한 축이 특히 복잡하다** — 유입 데이터는 **웹 애널리틱스**이나 응답에 **매출액**이 실린다. 즉 두 도메인이 한 응답에 있다 — §7.1 판정.
- **403 vs 404**: BE-003 §3.2 상속. 이 엔드포인트는 **컬렉션 집계**이므로 권한 부족 시 **403**이다 — 은닉할 개별 리소스 id 가 응답에 없다.
- **CSRF**: **해당 없음** — 쓰기 요청이 0건이다.
- **타임아웃**: 조회 5초 → 504. **⚠ 프론트에 클라이언트 타임아웃 상한이 없다** — 서버가 침묵하면 화면이 무한 대기한다(§7.7 #3).
- **프론트 역할 분기 없음**(FS-058 §4.1) — 단 `read`/`export` 권한 게이트는 있다(§7.1).

### 3.1 EP-01 응답 타입 — `TrafficStats`

`types.ts:55-84` 가 정본이다. 서버는 이 모양을 그대로 낸다.

| 필드 | 타입 | 필수 | 규약 |
|---|---|---|---|
| `daily[]` | `TrafficDayRow[]` | O | 요청 기간의 **모든 날**이 하루도 빠짐없이 들어야 한다 — §3.2 #2 |
| `daily[].id` | string | O | 일자 `'YYYY-MM-DD'` |
| `daily[].label` | string | O | 표시용 `'2026.07.16'`. §7.9 #1 |
| `daily[].visits` | number | O | 그날 총 유입수. **`Σ channels[*].visits` 와 같아야 한다** — §3.2 #1 |
| `daily[].orders` | number | O | 그날 총 구매건수. **`Σ channels[*].orders` 와 같아야 한다** |
| `daily[].revenue` | number | O | 그날 총 매출액(원). **`Σ channels[*].revenue` 와 같아야 한다** |
| `daily[].channels` | `Record<TrafficChannel, TrafficMetrics>` | O | 채널 5종 **전부** 키가 있어야 한다(`types.ts:73`). 0 이어도 키를 빼지 않는다 — `row.channels[segment]` 가 `undefined` 를 만나면 KPI 가 `NaN` 이 된다(`types.ts:95`) |
| `compareDaily[]` | `TrafficDayRow[] \| null` | O | 비교 기간의 같은 모양. **비교 안 함이면 `null`**(빈 배열이 아니다 — 카드가 증감을 그리지 않는 신호다) |
| `engines[]` | `TrafficRow[]` | O | **검색엔진 채널의 유입만** 쪼갠 것 — §3.2 #4 |
| `landings[]` | `TrafficRow[]` | O | **전체 유입**을 쪼갠 것 — §3.2 #5 |
| `TrafficRow.id` / `.label` | string | O | `label` 이 표의 첫 칸이다. **랜딩페이지는 URL 경로를 담는다** — §7.4 |
| `TrafficRow.visits`/`.orders`/`.revenue` | number | O | `TrafficMetrics` 상속(`types.ts:64`) |

**`TrafficChannel`** = `'search'|'bookmark'|'external'|'sns'|'ad'`(`types.ts:17`). **카페24 애널리틱스 어휘 고정이다** — 서버가 다른 이름을 쓰면 같은 값이 두 이름을 갖는다(`types.ts:3-10`). **특히 `'bookmark'` 는 '즐겨찾기'가 아니라 '리퍼러가 없는 유입 전부'다**(`types.ts:6-10`) — 서버가 이름대로 즐겨찾기만 세면 **주소창 직접입력·앱 접속이 어느 채널에도 안 잡혀 채널 합이 전체와 어긋난다**(§3.2 #1 위반).

### 3.2 서버가 반드시 지켜야 하는 불변식

| # | 불변식 | 깨지면 무엇이 보이는가 |
|---|---|---|
| 1 | **`daily[].visits/orders/revenue = Σ channels[*]`** (행마다) | KPI '전체'와 채널별 구성비가 **서로 다른 말을 한다**. 픽스처는 하루 전체를 **채널 합으로 계산해** 이를 구조적으로 보장한다(`data-source.ts:114-121`) — **서버도 같은 방식이어야 한다.** 화면은 검증하지 않는다. ⚠ **'북마크'가 리퍼러 없는 유입 전부를 담지 않으면 이 불변식이 깨진다**(§3.1) |
| 2 | **`daily` 에 기간의 모든 날이 있다** | 차트 x축이 날짜를 건너뛴다. `labels` 가 `daily.map(row=>row.label)`(`TrafficStatsPage.tsx:186`)이라 **빠진 날은 축에서 사라질 뿐 눈에 띄지 않는다.** 0건인 날도 행으로 채워야 한다 |
| 3 | **`compareDaily` 의 길이 = `daily` 의 길이** | 두 계열이 같은 x축에 겹쳐 그려진다(`StatsTrendCard.tsx:96-97`). 길이가 다르면 비교선이 축을 벗어난다 — 화면은 검증하지 않는다 |
| 4 | **`engines` 의 모수는 `Σ daily[].channels.search.visits` 다** | `data-source.ts:156-157` 이 그렇게 만든다. **서버가 전체 유입으로 쪼개면 검색엔진별 유입수 합이 채널별 '검색엔진' 값보다 커진다** — 같은 화면의 두 축이 어긋난다. ⚠ **이 규약은 코드 주석 한 줄에만 있다**(`:156`) — §7.9 #2 |
| 5 | **`landings` 의 모수는 `Σ daily[].visits`(전체)다** | `data-source.ts:158`. #4 와 **모수가 다르다는 것**이 요점이다 — 두 표가 같은 컬럼(`columnsOf`)을 쓰면서 다른 전체를 쪼갠다 |
| 6 | **`channels` 에 채널 5종 키가 전부 있다** | `metricsOfSegment` 가 `undefined` 를 반환 → KPI 가 `NaN` 으로 렌더된다(`types.ts:95` 는 인덱스 접근을 가드하지 않는다) |
| 7 | **음수를 내지 않는다** | 전환율이 음수가 되고 증감 색이 뒤집힌다. `conversionRate`(`types.ts:113-115`)는 분모 0 만 가드한다 |
| 8 | **`orders` 는 `visits` 를 넘지 않는다** | 넘으면 **구매전환율이 100% 를 넘는다.** `types.ts:12-13` 이 그 위험을 명시한다('분모를 주문으로 잡으면 100%를 넘는 값이 나온다'). 화면은 상한을 두지 않는다 — `formatPercentValue(137.4)` → '137.4%' 를 그대로 그린다 |
| 9 | **`revenue` 는 원 단위 정수다** | `formatWonValue`(`format.ts:36-38`)가 `Math.round` 하나 소수가 오면 표시만 반올림되고 합계는 어긋난다 |

### BE-058-EP-02 · 검색엔진별 집계 — **심은 있으나 별도 호출이 아니다**

### 6.1 어댑터 본문 요구사항

`fetchTrafficStats`(`data-source.ts:150-169`)의 **안쪽만** 교체한다. 골격은 그대로 둔다.

1. **`loadStats` 골격을 지킨다.** `mock.ts:45-54` 가 `wait(delay, signal)` → `failIfRequested(scope,'list')` → `build()` 순서다. 실제 fetch 는 `build()` 자리에 들어간다 — **`signal` 을 반드시 흘려보낸다**.
2. **한 응답을 유지한다.** 심대로 3회 부르면 `useStatsQuery` 가 셋으로 갈려 **`isFirstLoad` 판정이 세 벌**이 되고(하나만 도착한 중간 상태가 생긴다) 축 전환마다 왕복이 생긴다 — §7.2.
3. **`?fail=`·`?empty=`·`?delay=` 재현 경로를 지운다면 그 사실을 문서화한다.** `stats-screens.test.tsx` 5개 시나리오 × 6화면이 이 스위치 위에 서 있다.
4. **응답을 그대로 믿지 않는다.** §3.2 의 불변식 9개 중 **화면이 검증하는 것은 0개다.** 특히 #6(채널 키 누락 → `NaN`) · #1(채널 합 불일치 → KPI 와 구성비가 다른 말) · #8(전환율 >100%)은 조용히 틀린 숫자를 낸다.
5. **`compareDaily: null` 과 `[]` 를 구분해 낸다.** `null` 은 '비교 안 함'이고 `[]` 는 '비교 기간에 데이터 없음'이다 — 화면이 `null` 로 증감 렌더 여부를 가른다(`TrafficStatsPage.tsx:155`).
6. **`engines`/`landings` 의 모수 규약(§3.2 #4·#5)을 지킨다** — 지금 코드 주석 한 줄(`data-source.ts:156`)에만 있는 규칙이다.

## 7. 핵심 판정

### 7.1 한 응답에 두 도메인이 있다 — 권한 축이 셋으로 갈린다 【보안 판정】

**이 화면의 권한 문제는 BE-057(주문 통계)보다 한 겹 깊다.**

| 축 | 무엇 | 근거 |
|---|---|---|
| **프론트 화면 권한** | `page:/stats/traffic` | `StatsPageShell.tsx:92` `navPageResourceId('/stats/traffic')` → `resources.ts:66-68`. `nav-config.ts:202` 의 잎에서 파생 |
| **서버 애널리틱스 권한** | 유입 데이터(방문·리퍼러·랜딩) | 미정 — 심에 없다 |
| **서버 매출 권한** | `revenue` 필드 | 미정 — 심에 없다 |

**응답 하나에 애널리틱스와 매출이 섞여 있다.** `TrafficRow`(`types.ts:64-67`)가 `visits`·`orders`·**`revenue`** 를 함께 담고, `TrafficDayRow`(`:70-74`)도 그렇다. 즉:

> **'유입은 봐도 되지만 매출은 안 되는' 역할이 이 엔드포인트를 부를 수 없다.** 유입수를 보려면 매출액이 함께 온다.

이것이 실무에서 흔한 요구다 — 마케팅 담당자에게 채널 성과(유입·전환율)는 주되 **매출 절대액**은 주지 않는 경우. 지금 계약으로는 불가능하다: 응답이 통짜다.

**판정**: 계약이 다음을 확정한다.

1. **`GET /api/stats/traffic` 의 권한 단위는 '유입 분석 읽기'이며, 그것은 매출 열람을 포함한다.** 응답에 `revenue` 가 있는 한 이 엔드포인트를 부를 수 있는 사람은 매출을 본다. **분리하려면 응답을 사영해야 한다**(예: 매출 권한이 없으면 `revenue` 를 뺀 응답) — **그러나 그것은 심에 없고 화면도 그 상태를 표현하지 못한다**(`revenue` 가 `TrafficMetrics` 의 필수 필드다 — `types.ts:60`). **지금 확정하지 않고 이관한다** — §7.9 #4.
2. **`page:/stats/traffic` 의 read 는 보안 경계가 아니라 메뉴 가시성이다.** 데이터 경계는 서버 권한이다. **⚠ 그런데 이 화면을 끄는 것은 실제로 무언가를 숨긴다** — 채널별 성과는 다른 화면에 없다(주문 통계는 상태, 매출 통계는 금액, 방문자 통계는 사람만 센다). **BE-040(달력, 아무것도 숨기지 못함) · BE-057(재구성 가능) 과 다르다: 이 화면의 정보는 재구성이 어렵다.** 그럼에도 프론트 게이트는 **통제가 아니다** — 서버가 응답을 주면 브라우저는 받는다.
3. **`export` 권한은 반출 통제가 아니다** — BE-057 §7.1 과 같다. CSV 를 브라우저가 만든다(`useCsvExport.ts:59-76`) → 요청이 없다 → **서버 로그에 '누가 매출 데이터를 내보냈나'가 남지 않는다.** 이 화면은 **매출액을 담은 CSV** 를 만들 수 있으므로 그 부재가 주문 통계보다 무겁다 — §7.9 #5.

**이관**: 백엔드 명세(권한 모델) · UI 기획 — §7.9 #4 · #5.

### 7.2 심이 3건이라고 엔드포인트 3개로 확정하지 않는다 【정합 판정】

`data-source.ts:6-8` 에 심이 셋이다. **그러나 프론트는 한 번만 부르고 한 객체를 받는다** — `fetchTrafficStats`(`:150-169`)가 `{daily, compareDaily, engines, landings}` 를 낸다.

**판정**: **EP-01 하나로 확정하고, EP-02·EP-03 은 '심은 있으나 별도 호출이 아님'으로 남긴다.** 심대로 쪼개는 것이 나빠 보이는 이유를 계약으로 못 박는다:

1. **축 전환이 재조회를 유발하게 된다.** 지금 `queryKey` 에 `view` 가 없어(`TrafficStatsPage.tsx:128`) 채널별↔검색엔진별↔랜딩페이지별을 오가도 **요청이 0건**이다. 쪼개면 축마다 왕복이 생긴다. **이것은 명백한 퇴보다.**
2. **`isFirstLoad` 판정이 세 벌이 된다.** `useStatsQuery` 를 셋 걸면 하나만 도착한 중간 상태가 생기고, 화면은 그것을 표현할 수단이 없다(`StatsPageShell` 이 `error` 문자열 하나와 `loading` 불리언 하나만 안다).
3. **모수 규약(§3.2 #4·#5)이 무너진다.** `engines` 의 모수는 `Σ daily[].channels.search.visits` 다 — **`daily` 와 같은 트랜잭션에서 계산돼야** 두 축이 어긋나지 않는다. 별도 엔드포인트로 쪼개면 두 요청 사이에 데이터가 갱신되어 **검색엔진별 합 ≠ 채널별 '검색엔진'** 이 되는 순간이 생긴다.

**따라서 심 `:7`·`:8` 은 삭제 또는 재작성 대상이다** — 지금 형태로 남으면 미래의 누군가가 3회 호출로 구현해 위 셋이 발생한다. §7.9 #6.

**단, 규모가 바뀌면 판정도 바뀐다**: 랜딩페이지가 실제로는 수백~수천 개일 수 있다(지금은 **클라이언트 상수 7개** — `data-source.ts:71-79`). 그때는 `landings` 를 응답에서 빼는 것이 아니라 **서버가 상위 N 개로 자르고 나머지를 '기타'로 합치는 것**이 답이다(검색엔진이 이미 그 모양이다 — `:67` `{ id:'etc', label:'기타' }`). 원천을 쪼개지 않고 응답을 좁힌다 — §7.9 #7.

**판정**: **엔드포인트를 지어내지 않는다.** EP-01 하나가 이 화면의 서버 계약 전부다.

### 7.4 지표 정의를 서버가 소유하지 않는다 — 그리고 전환율이 특히 위험하다 【정합 판정】

| 정의 | 어디에 | 화면이 운영자에게 하는 약속 |
|---|---|---|
| 유입 채널 5종 | `types.ts:24-30` | 카페24 화면과 같은 이름 |
| **'북마크' = 리퍼러 없는 유입 전부** | `types.ts:6-10`(주석) | **화면 힌트가 그대로 적는다**: '북마크에는 즐겨찾기뿐 아니라 주소창 직접입력·앱 접속처럼 유입 경로가 남지 않은 방문이 함께 담깁니다.'(`TrafficStatsPage.tsx:43-44`) |
| **구매전환율 = 구매건수 ÷ 방문수 × 100** | `types.ts:12-13,113-115` | **화면 힌트가 공식을 적는다**: '구매건수 ÷ 방문수 × 100 으로 계산합니다.'(`:180`) |
| 매출액 = '해당 유입에서 발생한 결제 금액의 합' | — | 힌트(`:172`). **정의가 코드 어디에도 없다** — 픽스처가 `orders × 객단가`(`data-source.ts:85`)로 만들 뿐이다 |

**문제 1 — '북마크'.** 화면은 `bookmark` 가 '리퍼러 없는 유입 전부'라고 **운영자에게 문구로 약속한다.** 그 판정은 서버가 한다. **서버가 이름대로 즐겨찾기만 세면** ① 힌트가 거짓말이 되고 ② **주소창 직접입력·앱 접속이 어느 채널에도 안 잡혀 §3.2 #1(채널 합 = 전체)이 깨진다** ③ 그러면 KPI '총 유입수'와 채널별 구성비 합계가 어긋난다. **이름이 오해를 부르는 필드일수록 계약이 명시적이어야 한다.**

**문제 2 — 전환율의 분모.** `types.ts:12-13` 이 **명시적으로 경고한다**: '분모는 구매건수가 아니라 **방문(유입)**이다. 분모를 주문으로 잡으면 100%를 넘는 값이 나온다'. 그런데 **화면에 상한이 없다** — `formatPercentValue(137.4)`(`format.ts:41-43`)가 '137.4%' 를 그대로 그린다. 즉 서버가 `orders > visits` 를 내면 **전환율 137.4% 가 아무 경고 없이 표에 뜬다.** 그것이 §3.2 #8 을 불변식으로 세운 이유다.

**판정**: **지표 정의의 소유자는 서버다.** 계약이 확정한다:

1. **§3.2 #1 · #8 을 계약 불변식으로 승격한다** — 채널 합 = 전체(북마크가 리퍼러 없는 유입 전부를 담는다는 뜻이다) · `orders ≤ visits`.
2. **`bookmark` 의 정의를 서버 계약에 적는다** — '리퍼러가 없거나 판별 불가한 모든 유입'. 이름이 'bookmark' 인 것은 카페24 어휘를 따르기 위함이며 **의미는 이름과 다르다**(`types.ts:6-10`).
3. **전환율은 서버가 내지 않는다** — 클라이언트가 계산한다(`types.ts:113-115`). 서버가 별도로 내면 반올림이 갈려 **표의 '구매전환율' 열과 KPI '구매전환율'이 다른 값을 보인다**(둘 다 클라이언트 계산인 지금은 갈릴 수 없다). 서버는 **원시 건수만** 낸다.
4. **매출액의 정의가 어디에도 없다** — '해당 유입에서 발생한 결제 금액'이 **무엇을 기준으로 유입에 귀속되는지**(라스트 클릭? 세션 내? 기간 내?) 코드에도 문서에도 없다. **어트리뷰션 모델이 정해지지 않은 채 매출액을 채널에 붙이고 있다** — 이것이 이 화면에서 가장 큰 미정이다. §7.9 #8.

### 7.5 구글 참조검색어 고지가 클라이언트 상수다 【정합 판정】

`TrafficStatsPage.tsx:291-302` 이 검색엔진별 축에서 `<Alert tone="info">` 로 고지한다 — 구글이 개인정보 보호정책상 검색어를 리퍼러로 넘기지 않아 카페24가 '참조검색어 없음'으로 집계한다는 내용이다. `data-source.ts:56-61` 이 같은 사실을 주석으로 적는다.

**세 가지가 따라온다.**

1. **도메인 지식이 화면에 하드코딩됐다.** 서버가 '이 검색엔진은 검색어를 주지 않는다'를 알리는 필드가 없다. 구글의 정책이 바뀌거나 다른 엔진(빙·네이버 앱)이 같은 상태가 되면 **화면 코드를 고쳐야 한다.** 그것이 옳은지는 판단이 필요하다 — 정책은 자주 바뀌지 않으므로 하드코딩이 실용적일 수 있다.
2. **고지가 여기 없는 것을 설명한다.** 이 표에는 **검색어 컬럼이 아예 없다**(`columnsOf` — 유입수·구매건수·매출액·전환율뿐). 고지는 '검색어를 확인할 수 없다'고 말하는데 **운영자는 애초에 여기서 검색어를 기대하지 않는다** — 검색어는 `/stats/keywords` 의 일이다(`data-source.ts:60-61`). FS-058 §7 #8.
3. **조회 실패 시에도 남는다.** `notice` 슬롯이 `StatsPageShell.tsx:123` 에서 배너보다 위에 렌더되고 조건이 `params.view` 뿐이라, 실패해도 '아래 유입수·매출액은 정상으로 잡히지만…' 이라 말한다 — **아래에 아무것도 없는데.** FS-058 §7 #9.

**판정**: **서버가 참조검색어 가용성을 알리는 계약을 지금 만들지 않는다** — 심이 없고, 이 표에 검색어가 없어 **서버가 알려도 화면이 쓸 자리가 없다.** 고지의 자리는 **검색어 분석(FS-059)** 이며 거기서는 `참조검색어 없음` 이 **실제 행으로 존재한다**(BE-059 §3). §7.9 #9.

### 7.6 실패가 status 를 구분하지 않고 참조 코드도 없다 【보안·운영 판정】

BE-057 §7.5 와 **같은 사안이다**(`_shared/queries.ts` 한 벌을 공유한다). 이 화면 고유의 근거만 적는다:

- `queries.ts:46-47` 이 원문을 감추는 것은 옳다 — **이 화면의 응답에는 매출액이 있다.** 서버 원문이 새면 내부 스키마가 노출된다.
- **그러나 §7.1 의 시나리오에서 원인 파악이 불가능하다**: '유입은 되는데 매출 권한이 없어 403' 인지 '서버 장애' 인지 같은 화면이다.
- `traceId` 미표시 · `Retry-After` 미소비도 동일.

**판정**: 서버는 BE-003 §2 를 그대로 지킨다. **소비하지 않는 것은 프론트의 결함이며 서버 계약을 바꿀 이유가 아니다.** §7.9 #10.

### 7.8 '오늘'을 클라이언트가 정한다 — TZ 축은 수렴됐다 【정합 판정】

BE-057 §7.7 과 **같은 사안이다**(`_shared/period.ts` · `shared/format.ts` 를 공유한다). 판정도 같다:

- '오늘'은 **KST 고정**(`useStatsParams.ts:162` `formatDate(new Date())` → `shared/format.ts:63` `Asia/Seoul`).
- 달력 산술 앵커는 **UTC 정오** 한 벌(`period.ts:73,77,97,104`).
- **서버는 '오늘'을 정하지 않는다** — `start`/`end` 를 받아서 처리한다.
- **계약이 확정할 것**: **`start`/`end` 는 Asia/Seoul 기준 달력 날짜다.** 서버가 UTC 로 해석하면 하루가 밀린다.

**이 화면 고유의 위험**: 유입은 **밤에도 들어온다.** KST 00:00~09:00 의 유입은 UTC 로 전날이다 — 서버가 UTC 로 자르면 **매일 아침 9시간치 유입이 전날로 밀린다.** 주문(결제 시각이 낮에 몰림)보다 경계 오차가 크다. §7.9 #12.
