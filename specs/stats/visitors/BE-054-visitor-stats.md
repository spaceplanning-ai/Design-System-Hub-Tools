# BE-054. 방문자 통계 백엔드 기능 명세

## 1. 개요

| 항목 | 내용 |
|---|---|
| 대상 화면 | FS-054 방문자 통계 (`/stats/visitors`) |
| 범위 | 방문 지표 **집계 조회** 하나. 심(`// TODO(backend)`)이 있는 만큼만 계약으로 세운다 |
| 범위 밖 | 방문 로그 원본(raw event) 수집·적재·보존 — 심이 없고 화면도 요구하지 않는다. CSV 내보내기 — **클라이언트가 만든다**(§7.3) |
| 프론트 어댑터 | **없다.** `useCrudListQuery`/`CrudAdapter` 계열을 쓰지 않는다 — `fetchVisitorStats(query, signal)`(`data-source.ts:89-104`) 하나가 `_shared/mock.ts:45-54` 의 `loadStats` 를 부른다 |
| 도메인 타입 | `visitors/types.ts` — `VisitorRow` · `VisitorStats` · `VisitorSegment` (**전부 클라이언트 선언**) |
| 검증 정본 | **해당 없음** — 입력 폼이 없다. 조회 조건의 검증은 `_shared/period.ts:200-206`(`periodErrorOf`)이 클라이언트에서 한다 |
| 쓰기 | **0건.** 이 화면은 서버 상태를 바꾸지 않는다(FS-054 §4.1) |

> **에러 봉투·권한 모델 상속**: BE-003 §2·§2.1·§3 을 그대로 상속한다. 여기서 재정의하지 않는다.
>
> **⚠ 이 문서의 성격**: 심은 있으나 **프론트가 그 심대로 부르지 않는다**. 그 어긋남을 덮지 않고 §6.1 에 그대로 적고 §7 에서 판정한다. **심을 넘어서는 엔드포인트는 만들지 않는다.**

### 1.1 코드 대조 근거표

| 사실 | 근거 |
|---|---|
| **심이 3건 있다** | `visitors/data-source.ts:6-8` — `GET /api/stats/visitors?start&end&segment` · `GET /api/stats/visitors/hourly?start&end` · `GET /api/stats/visitors/weekday?start&end` |
| **그런데 프론트 호출은 1건이다** | `VisitorStatsPage.tsx:144-150` 이 `useStatsQuery([...], fetchVisitorStats)` 하나만 건다. `fetchVisitorStats`(`data-source.ts:89`)는 `loadStats` 를 **한 번** 부르고 `{daily, compareDaily, hourly, weekday}` 를 **함께** 돌려준다(`:94-101`) |
| **시간대별·요일별이 서버에서 오지 않는다** | `data-source.ts:98-99` — `hourly: hourlyRowsOf(daily)` · `weekday: weekdayRowsOf(daily)`. 둘 다 **일자별 배열을 접어 만드는 클라이언트 순수 함수**다(`:61-82`) |
| **세그먼트가 서버로 가지 않는다** | `VisitorStatsPage.tsx:147` 이 `fetchVisitorStats({period, comparePeriod}, signal)` 로 부른다 — `segment` 가 인자에 없다. 쿼리 키에도 없다(`:145`). 유형별 값은 응답 행에 이미 있고 화면이 고른다(`types.ts:89-93` `visitsOfSegment`) |
| **비교 기간을 실을 파라미터가 심에 없다** | 심은 `?start&end&segment` 뿐이다. 그런데 프론트는 `comparePeriod` 를 인자로 넘기고(`:147`) 응답에 `compareDaily` 를 기대한다(`types.ts:57`) |
| **데이터가 목(mock)에서 파생된다** | `mock.ts:45-54` `loadStats` = `wait(delay) → failIfRequested → build()`. `build()` 는 `dailyRowsOf`(`data-source.ts:55-59`)이고 그 값은 `seededSeries`(`mock.ts:87-100`)의 시드 난수다. **서버 왕복이 하나도 없다** |
| 전용 어댑터 파일이 없다 | `stats/**` 에 `adapter.ts` 가 없다. `shared/crud` 에서 가져오는 것은 `parseFilter`(`useStatsParams.ts:37`) · `useDebouncedSearch`(`StatsFilterBar.tsx:19`) · `failIfRequested`/`LATENCY_MS`(`mock.ts:13`) 뿐이다 |

**결론**: **서버 집계 계약이 존재하지 않는다.** 심 3건은 '언젠가 이렇게 부르겠다'는 표식이고, 현재 프론트의 실제 모양(**호출 1건 · 응답 4벌 · 세그먼트/비교/축이 전부 클라이언트 파생**)과 **맞지 않는다**. 이 문서는 심에 있는 것만 §4 에 세우고, 심과 프론트가 어긋나는 지점은 §6.1 에 표로 남긴 뒤 §7.1 에서 판정한다. **집계 엔드포인트를 지어내지 않는다.**

## 2. 공통 (상속)

- **에러 봉투**: BE-003 §2. ⚠ 이 화면은 봉투의 `code` 를 **소비하지 않는다** — `queries.ts:47` 이 `isError` 하나로 '통계를 불러오지 못했습니다.' 를 만든다(§7.5). `traceId` 는 서버가 반드시 실어야 하나 **화면이 표시하지 않는다**(§7.6 #4).
- **권한**: BE-003 §3 상속. 이 화면이 부르는 것은 조회뿐이므로 `admin`·`operator` 둘 다 읽는다. **⚠ 다만 프론트 권한 리소스와 서버 권한 리소스가 다른 축이다 — §7.1.**
- **403 vs 404 은닉**: BE-003 §3.2 상속. 이 화면의 리소스는 **컬렉션(집계)** 이므로 **403** 이다 — 개별 회원 리소스가 아니라 은닉할 id 가 없다(§7.4).
- **CSRF**: **해당 없음** — 쓰기 요청이 0건이다(BE-003 §3.3 은 상태 변경 요청에만 걸린다).
- **타임아웃**: 조회 5초 → 504 `REQUEST_TIMEOUT`. ⚠ **프론트에 클라이언트 타임아웃 상한이 없다**(`AbortSignal.timeout` 미사용) — 서버가 응답하지 않으면 화면은 무한히 스켈레톤이다(§7.6 #5).
- **프론트 역할 분기 없음**(FS-054 §4.1). 단 **`export` 액션은 프론트가 갈라 버튼을 숨긴다** — 서버 계약이 아니라 UX 다(§7.3).

## 3. 데이터 계약

이 화면이 보이는 값이 어디서 오는가. **'서버 계약인가' 열이 이 문서의 요점이다.**

| 화면이 보이는 값 | 어디서 오는가 | 서버 계약인가 |
|---|---|---|
| KPI '전체 방문 수' | `sumOf(daily, visitsOfSegment(row, segment))` — **클라이언트 합산** | ✕ — 행의 `visits`/`newVisitors`/`returningVisitors` 만 서버 몫 |
| KPI '순 방문자 수' | `sumOf(daily, row.uniqueVisitors)` | ✕ — **⚠ 합산이 틀릴 수 있다(§7.2)** |
| KPI '전체 페이지뷰' | `sumOf(daily, row.pageViews)` | ✕ |
| KPI '재방문율' | `revisitRate(daily)`(`types.ts:71-74`) — 클라이언트 파생 | ✕ |
| KPI '평균 체류시간' | `averageDuration(daily)`(`types.ts:77-82`) — **방문 수 가중평균** | ✕ |
| KPI 증감(▲12.3%) | `deltaOf(value, compareValue)`(`format.ts:91-101`) | ✕ |
| 추이 차트 계열 | `stats.daily.map(pick)` / `stats.compareDaily.map(pick)` | ✕ — 행 배열만 서버 몫 |
| 표 '일자별' 행 | `stats.daily` | **O** — 이것만이 서버가 줄 것이다 |
| 표 '시간대별' 행 | `hourlyRowsOf(daily)`(`data-source.ts:61-68`) — **클라이언트가 일자별을 시간 가중치로 쪼갠다** | ✕ (§7.2 — 심은 있는데 안 쓴다) |
| 표 '요일별' 행 | `weekdayRowsOf(daily)`(`data-source.ts:70-82`) — **클라이언트가 같은 요일을 평균한다** | ✕ (§7.2) |
| 표 '재방문율(%)' 칸 | `rowRevisitRate(row)`(`VisitorStatsPage.tsx:52-54`) | ✕ |
| 비교 기간 자체 | `comparePeriodOf(period, mode)`(`period.ts:186-193`) — **클라이언트 산술** | ✕ (§7.2) |
| 기간 프리셋('최근 7일') | `PERIOD_PRESETS`(`period.ts:134-166`) — **클라이언트 상수** | ✕ (§7.2) |
| '오늘' | `formatDate(new Date())`(`useStatsParams.ts:162`) — **브라우저 시계 · 서울 고정 표기** | ✕ (§7.2) |
| CSV 파일 | `toCsvText` + `downloadCsv`(`shared/download.ts`) — **브라우저가 Blob 으로 만든다** | ✕ (§7.3) |

### 3.1 서버가 줄 것 — `VisitorRow` (심이 있는 만큼)

`visitors/types.ts:40-52`. **이 타입은 클라이언트가 선언했고 서버는 아직 모른다.**

| 필드 | 타입 | 의미 (`types.ts:1-11` 어휘) |
|---|---|---|
| `id` | string | 구간 식별자 — 일자 `'2026-07-16'` |
| `label` | string | 표시명 `'2026.07.16'` — ⚠ **표시 문자열이 데이터에 섞여 있다**(§7.4) |
| `visits` | number | 전체 방문 수(중복 포함) |
| `uniqueVisitors` | number | 순 방문자 수(UV) |
| `pageViews` | number | 전체 페이지뷰(PV) |
| `newVisitors` | number | 처음 온 방문자 |
| `returningVisitors` | number | 다시 온 방문자 |
| `durationSeconds` | number | 그 구간의 평균 체류시간(초) |

> **불변식 두 개가 계약이어야 한다** — 픽스처는 지키지만 아무도 강제하지 않는다: ① `newVisitors + returningVisitors === visits`(`data-source.ts:39-40` 이 뺄셈으로 만든다) ② `uniqueVisitors <= visits`(`:42` 가 `Math.max` 로 누른다). 서버가 이것을 깨면 재방문율이 100% 를 넘고 화면은 아무 경고 없이 그린다(§7.6 #2).

## 4. 엔드포인트 명세

### BE-054-EP-01 · 방문자 통계 조회

| 항목 | 내용 |
|---|---|
| 근거 (FS) | FS-054-EL-003 · EL-004 · EL-005 · EL-006 |
| 심 | `data-source.ts:6` — `// TODO(backend): GET /api/stats/visitors?start&end&segment` |
| 메서드·경로 | `GET /api/stats/visitors` |
| 권한 | 방문자 통계 도메인 읽기(`admin`·`operator`) — BE-003 §3 상속. **⚠ §7.1** |
| 멱등성 | 멱등(GET) |
| 페이징 | **없음 — 기간 전량.** 페이지는 클라이언트가 자른다(`table.ts:54-57`) |
| 레이트리밋 | **미정** — 심에 명시가 없다 |

**쿼리 파라미터** (심이 이름만 정했다 — 타입·규약은 프론트 코드에서 읽는다):

| 이름 | 타입 | 필수 | 근거 | 프론트가 보내는가 |
|---|---|---|---|---|
| `start` | `YYYY-MM-DD` (서울 달력일, inclusive) | O | `period.ts:19-22` (`StatsPeriod`) | O — `params.period.start` |
| `end` | `YYYY-MM-DD` (서울 달력일, inclusive) | O | 같음 | O — `params.period.end` |
| `segment` | `'all'\|'new'\|'returning'` | ? | `types.ts:15-21` | **✕ — 보내지 않는다**(§6.1 · §7.2) |

**응답 200** — **미정.** 심은 응답 모양을 말하지 않는다. 프론트가 기대하는 것은 `VisitorStats`(`types.ts:54-61`) = `{daily, compareDaily, hourly, weekday}` 인데, **그중 셋(`compareDaily`·`hourly`·`weekday`)은 이 엔드포인트의 쿼리로 요청할 수단이 없다**(§6.1). 계약을 여기서 확정하지 않고 §7.2 로 이관한다.

**에러**: BE-003 §2.1 상속 — 400 `VALIDATION_FAILED`(§7.4) · 401 `UNAUTHENTICATED` · **403 `FORBIDDEN`**(컬렉션) · 429 `RATE_LIMITED` · 500 `INTERNAL_ERROR` · 504 `REQUEST_TIMEOUT`. **전부 FS-054-EL-006 한 문구로 수렴한다** — 이 화면은 status 를 구분하지 않는다(§7.5).

### BE-054-EP-02 · 시간대별 조회 — **심은 있으나 프론트가 부르지 않는다**

| 항목 | 내용 |
|---|---|
| 심 | `data-source.ts:7` — `// TODO(backend): GET /api/stats/visitors/hourly?start&end` |
| 프론트 호출 | **없다.** `data-source.ts:98` 이 `hourlyRowsOf(daily)` 로 **클라이언트에서 만든다** |

`hourlyRowsOf`(`:61-68`)는 기간 전체의 방문 합을 **고정 시간 가중치**(`HOUR_WEIGHTS` — `:31-34`, 새벽 바닥·점심/저녁 봉우리)로 24등분한다. 즉 **이것은 집계가 아니라 모양 흉내다** — 실제 시간대 분포를 알 수 없으므로 픽스처가 그럴듯한 곡선을 지어낸다. 서버가 붙으면 이 함수는 **사라져야 하고**, 그때 비로소 이 심이 실호출이 된다. **지금 계약을 확정하지 않는다** — §7.2.

---

### BE-054-EP-03 · 요일별 조회 — **심은 있으나 프론트가 부르지 않는다**

| 항목 | 내용 |
|---|---|
| 심 | `data-source.ts:8` — `// TODO(backend): GET /api/stats/visitors/weekday?start&end` |
| 프론트 호출 | **없다.** `data-source.ts:99` 이 `weekdayRowsOf(daily)` 로 **클라이언트에서 만든다** |

`weekdayRowsOf`(`:70-82`)는 같은 요일의 행을 모아 **방문 수만 평균**한다. ⚠ **그 평균을 `rowOf(id, label, visits, seed)` 에 넣어 나머지 지표(PV·UV·체류시간)를 다시 시드 난수로 만든다**(`:80`) — 즉 요일별 행의 PV/UV/체류시간은 **그 요일 실제 값의 평균이 아니라 새로 지어낸 수**다. 픽스처로서는 무해하나 **이 축의 의미는 서버가 정해야 한다**(그 요일의 합인가 평균인가 — §7.2). **지금 확정하지 않는다.**

---

### BE-054-EP-04 · 비교 기간 조회 — **심 없음 (미정)**

화면은 비교 기간을 **1급 기능**으로 갖는다(기본값이 '직전 기간'이다 — `period.ts:38`). 그런데 EP-01 의 심에는 `compareStart`/`compareEnd` 자리가 없고, 프론트는 `comparePeriod` 를 `fetchVisitorStats` 의 인자로만 넘긴 뒤 픽스처가 같은 함수를 두 번 돌려 만든다(`data-source.ts:97`). **심이 없으므로 파라미터를 지어내지 않는다** — §7.2 판정.

### BE-054-EP-05 · 기간 프리셋 · 영업 달력 — **심 없음 (미정)**

'최근 7일 = 오늘 포함 7일'(`period.ts:141-146`) · '이번 달 = 1일~오늘' · 월 말일 보정(`:100-106`) · '오늘'의 기준 시계가 전부 클라이언트다. 서버가 이 규칙을 알 필요가 있는지는 **미정** — §7.2.

### BE-054-EP-06 · CSV 내보내기 — **심 없음 (미정) · 서버 계약이 아니다**

`useCsvExport`(`useCsvExport.ts:39-93`)가 **이미 받은 배열**로 파일을 만든다. 요청이 나가지 않는다. §7.3 판정.

## 5. 예외 매트릭스

**이 화면은 읽기 전용이라 쓰기 축(409·422)이 원리적으로 성립하지 않는다.** 프론트가 실제로 거는 조회는 EP-01 하나이므로 그 한 행을 9축 전수로 채우고, 심만 있는 EP-02/03 을 '도입 시' 관점으로 함께 채운다.

### 6.2 어댑터 본문 요구사항

이 화면에는 `CrudAdapter` 가 없다 — 교체 지점은 **`data-source.ts:93-101` 의 `build` 콜백** 하나다. 서버를 붙일 때 지켜야 할 것:

> ① **`signal` 을 반드시 흘려보낼 것.** 지금 `loadStats(scope, signal, ...)`(`mock.ts:45-54`)가 `wait(delay, signal)` 로 abort 를 성립시킨다 — 화면 이탈 시 요청이 취소되고 **abort 는 에러로 노출되지 않는다**(FS-054 §4.1). fetch 로 바꿀 때 `signal` 을 빠뜨리면 이탈한 화면의 실패 배너가 뜬다.
> ② **`?fail=`·`?empty=`·`?delay=` 재현 경로를 살릴 것**(`mock.ts:24-34,52-53`). 이 세 스위치가 STATE-01/02 회귀 테스트(`stats-screens.test.tsx`)의 유일한 손잡이다.
> ③ **응답 범위를 좁히지 말 것.** KPI·차트·표가 전부 `daily` 전량 위에서 계산된다 — 응답이 잘리면 합계·평균·재방문율이 **조용히 과소 집계된다**. 표는 페이지가 있지만 **KPI 는 없다**(기간 전체의 합이다).
> ④ **§3.1 의 두 불변식을 지킬 것**(`new + returning === visits` · `unique <= visits`). 화면은 검증하지 않는다.

## 7. 핵심 판정

### 7.1 프론트 권한 리소스와 서버 권한이 다른 축이다 【보안 판정】

**프론트는 이 화면을 `page:/stats/visitors` 로 게이팅하고, 서버는 방문자 통계 도메인 권한으로 게이팅한다.**

근거: `nav-config.ts:198` 에서 `/stats/visitors` 는 **자기 자신이 잎**이다(`['방문자 통계','/stats/visitors']`). `permissions/resources.ts:84,92` 가 잎마다 `page:<to>` 리소스를 만들고, `AppShell.tsx:490-492` 의 `RequirePermission` → `useRouteCan('read')` → `resourceIdForPath(pathname)` 가 그 잎을 찾아 read 를 확인한다. 즉 프론트 권한의 단위는 **화면**이고 서버 권한의 단위는 **엔드포인트**다.

**판정 세 줄로 확정한다.**

1. **서버는 `GET /api/stats/visitors` 하나의 권한만 안다** — 방문자 통계 읽기 권한. 화면별 권한을 만들지 않는다(만들면 같은 데이터에 두 개의 진실이 생긴다).
2. **`page:/stats/visitors` 의 read 는 보안 경계가 아니라 메뉴 가시성이다.** `RequirePermission.tsx:8-11` 이 이 사실을 이미 명문화했다 — '실제 차단은 여전히 서버의 몫이며 위조된 localStorage 로 이 가드는 우회된다'. 권한 모델(`permission-store.ts`)이 **localStorage** 에 산다는 것이 그 근거다.
3. **그래서 프론트 가드를 통과한 뒤 서버 403 을 받는 경로가 실재한다** — '방문자 통계 read = ON'이지만 서버의 방문자 통계 권한이 없는 역할이 만들어질 수 있다. 그때 화면은 '통계를 불러오지 못했습니다.' 한 줄만 보이고(§7.5), **운영자는 권한 문제인지 서버 장애인지 알 수 없다.** 관리자는 권한을 켰다고 믿는데 화면이 영원히 실패한다.

**⚠ 그런데 이 화면에는 추가로 게이트가 하나 더 있고, 그것은 죽어 있다.** `StatsPageShell.tsx:95-104` 가 `can(navPageResourceId(route),'read')` 를 **직접** 확인해 자기 403 을 그린다. `AppShell` 의 가드가 **같은 리소스·같은 액션**을 먼저 막으므로 이 분기는 실앱에서 도달할 수 없다. 그리고 둘의 UI 가 다르다 — 공유 `ForbiddenScreen`(`ErrorScreens.tsx:103-119`)은 `role="alert"` 로 알리고 '대시보드로' 탈출로를 주는데, 통계의 것은 **둘 다 없다**. **설정 섹션은 이미 공유 가드로 수렴했다**(`App.tsx:335-336`) — 통계만 남았다. **이관**: §7.6 #1.

### 7.2 서버 집계 계약이 없다 — 심을 넘어 지어내지 않는다 【정합 판정】

**화면이 보이는 값의 거의 전부가 클라이언트 파생이다**(§3). 서버가 줄 것은 **`daily` 행 배열 하나**뿐이고, 나머지는 전부 그 위에서 계산된다. 심은 3건인데 실제 호출은 1건이고 파라미터도 어긋난다(§6.1).

**따라서 BE-054-EP-04(비교 기간) · EP-05(프리셋/영업 달력) · EP-06(CSV)은 '심 없음(미정)'이고, EP-01 의 응답 계약과 EP-02·EP-03 의 축 의미도 '미정'이다. 지어내지 않는다.**

**다만 부재의 결과는 기록한다** — 도입 결정은 A63 이 한다:

| 사실 | 결과 |
|---|---|
| **집계가 클라이언트에 있다** | 재방문율·평균 체류시간(가중평균)·증감률의 **정의가 프론트 코드에만 있다**(`types.ts:71-82` · `format.ts:91-101`). 같은 지표를 다른 소비자(리포트·API·엑셀 배치)가 요구하면 정의가 두 벌이 되고, **두 벌은 반드시 갈라진다** |
| **`uniqueVisitors` 를 클라이언트가 더한다** | ⚠ **이것은 정의상 틀릴 수 있다.** UV 는 중복을 제거한 사람 수라 **날짜별 UV 를 더하면 UV 가 아니다** — 7일 연속 방문한 한 사람이 7명으로 세어진다. `VisitorStatsPage.tsx:173` 이 `sumOf(daily, row.uniqueVisitors)` 로 KPI '순 방문자 수'를 만들고, 카드 힌트는 '중복을 제거한 실제 방문자 수입니다'(`:176`)라고 **단정한다**. 픽스처에서는 드러나지 않는다(숫자가 지어낸 것이라 검증할 원본이 없다). **기간 UV 는 서버만 계산할 수 있다** — 일자별 합으로는 원리적으로 구할 수 없다. §7.6 #2 |
| 기간 상한이 없다 | `?start=1900-01-01&end=2100-01-01` → `eachDay` 가 73,000 날짜(`period.ts:59-65`). 지금은 클라이언트가 멈추고(FS-054 §7 #9), **서버가 붙으면 서버가 멈춘다.** 도입 시 **범위 상한(예: 366일)이 400 `VALIDATION_FAILED` 로 강제돼야 한다** |
| 세그먼트/축 전환에 재조회가 0이다 | **현재 설계의 장점이다** — 응답 한 벌에 4축이 다 들어 있어(`types.ts:54-61`) 토글이 요청을 만들지 않는다. 심대로 축을 3개 엔드포인트로 쪼개면 **이 성질을 잃고** 축별 로딩·부분 실패 표면이 새로 필요해진다(§7.6 #3) |

**도입 시 계약 조건**(지금 정하지 않고 조건만 남긴다): ① 기간은 **닫힌 구간**의 서울 달력일 문자열(`YYYY-MM-DD`) — 시각이 아니다(`shared/format.ts` ③ 규약과 같다). ② `start > end` 면 400 `VALIDATION_FAILED` + `fields[].name='end'`. ③ **범위 상한**을 두고 초과 시 400. ④ **기간 UV 는 서버가 계산해 별도 필드로 준다** — 일자별 행의 합으로 만들 수 없다(§7.6 #2). ⑤ 비교 기간은 **같은 엔드포인트의 선택적 쿼리**(`compareStart`·`compareEnd`)로 둔다 — 별 엔드포인트를 만들면 §7.1 의 권한 단위가 갈라진다. ⑥ 시간대별·요일별을 서버로 옮기면 **`hourlyRowsOf`·`weekdayRowsOf` 를 지운다** — 남겨 두면 같은 축의 값이 두 곳에서 나온다.

### 7.3 CSV 내보내기는 서버 계약이 아니다 — 그리고 이 화면에서는 그래도 된다 【보안 판정】

`useCsvExport`(`useCsvExport.ts:39-93`)는 **이미 받은 배열**을 돌며 `toCsvText` 로 문자열을 만들고 `downloadCsv` 가 Blob 으로 떨군다(`shared/download.ts:36-49`). **요청이 나가지 않는다.** 따라서:

**(a) 내보내기 범위는 '현재 조건 전체'가 맞다 — 그리고 실제로 그렇다.** `VisitorStatsPage.tsx:250-253` 이 `rows`(= 현재 축의 **페이지 자르기 전** 배열)를 넘기고 `exportCount={rows.length}` 로 '엑셀 내보내기 (N건)' 을 정직하게 말한다. 페이지가 아니라 조건 전체다(ERP-12 의 핵심 요구). **UTF-8 BOM** 도 붙는다(`download.ts:16,38`) — 없으면 엑셀이 한글을 깨뜨린다. **값은 화면에 보이는 그대로다** — `StatsColumn.csv` 가 `render` 와 같은 원천을 쓰도록 타입이 강제한다(`types.ts:44-53`).

**(b) 그런데 그것은 '현재 조건 전체'가 이미 브라우저에 있기 때문이다.** 서버 페이징이 없어 전량이 메모리에 있으니 클라이언트 생성이 성립한다. **서버 페이징이 도입되는 순간 이 등식이 깨진다** — 그때 '현재 페이지만' 내보내게 되고 그것은 ERP-12 위반이다. 즉 **§7.2 의 페이징 도입은 내보내기 엔드포인트를 함께 요구한다.** 지금은 심이 없으므로 정의하지 않는다.

**(c) 개인정보 판정: 이 화면의 CSV 에 개인정보가 없다.** 내보내는 컬럼 8개는 전부 **집계 수치**다(`VisitorStatsPage.tsx:56-121` — 구간·방문 수·UV·PV·신규·재방문·재방문율·체류시간). 개별 방문자를 식별할 값이 하나도 없다. **이 점에서 이 화면은 예약 일정(BE-040 §7.3)과 정반대다** — 거기서는 격자가 고객 이름을 렌더해 대량 열람 표면이 됐다. 여기서는 **원본 이벤트로 내려갈 경로 자체가 없다**(FS-054 §7 #14).

**(d) 그럼에도 권한 게이트는 옳다.** `StatsPageShell.tsx:119` 가 `can(resourceId,'export')` 로 버튼을 **그리지 않는다**(비활성이 아니라 부재 — `StatsFilterBar.tsx:223`). `export` 는 실재하는 액션이다(`permissions/resources.ts:31,46` — 'CSV/엑셀 다운로드'). **다만 이것은 서버 강제가 아니다** — 클라이언트 생성이라 **서버가 막을 지점이 없다.** read 권한만 있으면 데이터는 이미 브라우저에 있고, 버튼을 숨기는 것은 콘솔에서 우회된다. **판정: 통계 CSV 에서 `export` 권한은 read 권한과 실질적으로 같다.** 둘을 다르게 설정할 수 있는 권한 화면은 **관리자에게 있지도 않은 통제를 있다고 말한다.** 이 화면의 데이터가 집계뿐이라 실피해는 없으나(c), **매출 통계(BE-056 §7.3)에서는 같은 구조가 재무 데이터에 걸린다** — §7.6 #7.

### 7.4 표시 문자열이 데이터 계약에 섞여 있다 【정합 판정】

`VisitorRow.label`(`types.ts:44`)은 `'2026.07.16'`·`'14시'`·`'월요일'` 같은 **표시 문자열**이고 `id` 와 별도로 전송된다. 픽스처는 `formatDayLabel(day)`(`period.ts:211-213` — `'-'` 를 `'.'` 로 바꾼다)로 만든다.

이것을 서버가 주면 **표시 규칙이 서버로 새어 나간다** — 날짜 표기를 `2026.07.16` 에서 `2026-07-16` 으로 바꾸는 일이 백엔드 배포가 된다. 반대로 **클라이언트가 만들면** `id` 만 있으면 되고 `label` 은 계약에서 사라진다.

**판정**: **서버는 `id` 만 준다.** `label` 은 응답 필드가 아니다 — 클라이언트가 `id` 와 축(`view`)에서 만든다. 지금 픽스처가 둘 다 만드는 것은 픽스처가 곧 서버 역할을 겸하기 때문이며, 계약으로 굳히면 안 된다. (같은 이유로 **`400 VALIDATION_FAILED` 의 `fields[].name` 은 쿼리 파라미터명(`start`·`end`)이지 화면 라벨('조회 기간')이 아니다** — BE-003 §2 표의 규약 그대로.)

### 7.5 실패의 입도가 봉투보다 거칠다 【정합 판정】

BE-003 §2 는 `code`·`fields`·`traceId` 를 **필수로** 싣게 한다. 그런데 `queries.ts:46-47` 은 그것을 **전부 버린다**:

```
error: query.isError ? '통계를 불러오지 못했습니다.' : '',
```

주석은 '원문 에러 메시지를 그대로 노출하지 않는다 — 서버 내부를 흘리지 않는다 (EXC-20)' 라 적었고 **그 절반은 옳다**(never-leak). 그러나 EXC-20 의 나머지 절반(**복사 가능한 참조 코드**)과 EXC-06(**status 별 뚜렷한 UX**)이 함께 사라졌다. 결과: 401·403·429·500·504 가 **같은 한 줄**이 되고, §7.1 이 지적한 '프론트 통과 후 서버 403' 상황에서 운영자는 원인을 알 수 없으며, 429 의 `Retry-After` 도 읽히지 않는다.

**판정**: 서버는 봉투를 규약대로 싣는다(BE-003 §2 — 재정의하지 않는다). **버리는 쪽은 프론트이고, 고칠 곳도 프론트다** — `useStatsQuery` 가 `code`/status/`traceId` 를 함께 내면 화면이 분기할 수 있다. §7.6 #4.
