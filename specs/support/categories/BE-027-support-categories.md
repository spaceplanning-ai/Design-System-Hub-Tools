# BE-027. 문의 유형 관리 백엔드 기능 명세

## 1. 개요

| 항목 | 내용 |
|---|---|
| 대상 화면 | FS-027 문의 유형 관리 (`/support/categories` · 등록/수정 모달 · 삭제 확인 다이얼로그). 하위 라우트 없음 |
| 범위 | 문의 유형 목록 조회(참조 사용량 포함), 등록, 수정(이름·사용여부), 삭제(참조 중 차단) |
| 범위 밖 | 1:1 문의(티켓) CRUD(`/support/tickets`) · 답변 템플릿 CRUD(`/support/replies`) — 별도 FS/BE. **활성 유형 목록 조회**(`listActiveCategories` — 이 화면이 부르지 않고 위 두 화면이 소비한다: FS-027-EL-013·§7.6 에서 판정만 한다). 고객 노출 렌더(§7.8 XSS 판정만) |
| 프론트 어댑터 | `apps/admin/src/pages/support/categories/data-source.ts` (`createStoreAdapter` · scope `support-categories`) |
| 도메인 타입 | `apps/admin/src/pages/support/_shared/domain.ts` — `SupportCategory` · `SupportCategoryUsage` · `SupportCategoryInput` · `categoryInUse` · `CATEGORY_LABEL_MAX` |
| 사용량 계산 정본 | `apps/admin/src/pages/support/_shared/usage.ts` — `countCategoryUsage(categoryId, tickets, templates)` |
| 검증 정본 | `apps/admin/src/pages/support/categories/validation.ts` — `supportCategorySchema` (`requiredText('유형 이름', 30)` + `z.boolean()`) |

> **에러 봉투·권한 모델 상속**: BE-003 §2·§3 을 그대로 상속한다(BE-010 §2 와 동일 선언). 아래는 문의 유형 고유 차이만 기술한다.

## 2. 공통 (상속)

- **에러 봉투**: BE-003 §2. 공통 에러코드 동일. 5xx 는 `traceId` 를 실어 프론트가 참조 코드로 보일 수 있게 한다(EXC-20).
- **권한**: `admin` = 전체. `operator` = 조회 계열만(목록). 쓰기(등록·수정·삭제) → 403. 고객센터 읽기 권한 없는 관리자 → 컬렉션 403 / 개별 404 은닉(BE-003 §3.2).
- **CSRF**: 쓰기(POST·PUT·DELETE)에 `X-CSRF-Token`.
- **타임아웃**: 조회·쓰기 5초 → 504.
- **프론트 역할 분기 없음**(FS-027 §4.1) — 권한 강제는 **서버 책임**이다. 프론트는 쓰기 버튼조차 가리지 않으므로(§7.9) `operator` 는 버튼을 누르고 403 을 받는다.

## 3. 데이터 계약 (domain.ts 대조)

| 타입 | 필드 | 비고 |
|---|---|---|
| `SupportCategory` | `id`(string) · `label`(string) · `active`(bool) | `active` = 사용여부. **끄면 신규 문의/템플릿의 유형 선택에서 숨는다 — 기존 참조는 유지**(소프트 비활성 · §7.6). **`updatedAt`/`version` 없음** — 낙관적 동시성 토큰을 실을 자리가 없다(§7.3) |
| `SupportCategoryUsage` | `SupportCategory` + `ticketCount`(number) + `templateCount`(number) | 목록 행. **참조를 두 축으로 분해해 내려준다** — 프론트가 `티켓 N · 템플릿 M` 로 표시하고(`categoryUsageLabel`) 합이 0 보다 크면 삭제를 잠근다(`categoryInUse`) |
| `SupportCategoryInput` | `label`(string) · `active`(bool) | 등록·수정 공용 입력 |

**목록 응답은 `SupportCategoryUsage[]` 다** — 프론트 어댑터의 `list: listCategoryUsage` 가 두 참조 수를 **같은 응답에** 요구한다. 별도 사용량 엔드포인트를 두지 않는다(BE-010 EP-03 과 다른 선택 — §7.5).

**두 축은 독립이다**: `categoryInUse`(`domain.ts:36-38`)는 `ticketCount + templateCount` 만 보고 **`active` 를 보지 않는다.** 즉 사용여부를 꺼도 참조가 있으면 삭제되지 않는다. 서버 계약도 이 독립성을 그대로 지킨다(§7.1 · §7.6).

## 4. 엔드포인트 명세

### BE-027-EP-01 · 유형 목록 조회 (사용량 포함)
| 항목 | 내용 |
|---|---|
| 근거 (FS) | FS-027-EL-001, EL-003, EL-003.1, EL-003.2, EL-003.3, EL-004, EL-006 |
| 메서드·경로 | `GET /api/support/categories` |
| 권한 | `admin`, `operator` |
| 멱등성 | 멱등(GET) |
| 페이징 | **없음 — 전량.** 프론트가 페이지네이션·검색·필터를 갖지 않는다(FS-027-EL-003). 유형 수 상한 100개(§7.5) |
| 레이트리밋 | 분당 120회 |

**쿼리**: 없음. **`active` 필터도 없다** — 이 화면은 비활성 유형까지 전부 보여야 한다(FS-027-EL-003). 활성만 걸러 내려주는 표현이 필요한 곳은 문의 폼·템플릿 폼이며 **범위 밖**이다(§7.6).

**응답 200** — `readonly SupportCategoryUsage[]` (`{ id, label, active, ticketCount, templateCount }[]`). 두 참조 수는 **삭제되지 않은** 티켓·템플릿 기준. 프론트는 합이 0보다 크면 삭제 버튼을 잠근다(FS-027-EL-003.5). **정렬**: 저장소 배열 순서(추가 순)를 그대로 쓴다 — 프론트에 정렬 UI 가 없으므로 **서버가 안정된 순서를 보장**하면 된다(생성 순 오름차순 권고).

**성능 주의**: 픽스처는 매 조회마다 티켓·템플릿 전량을 순회해 센다(`usage.ts:12-21` — 유형마다 `filter` 2회 → 유형 수 × 참조 수). 티켓이 쌓이면 이 계산은 **O(유형 수 × 티켓 수)** 다. 서버는 집계 쿼리(`GROUP BY category_id`) 또는 카운터 컬럼으로 상수화한다 — 계약(응답 모양)은 바뀌지 않는다.

**에러**: 401 · 403 · 429 · 500 · 504.

### BE-027-EP-03 · 유형 수정 (이름 · 사용여부)
| 항목 | 내용 |
|---|---|
| 근거 (FS) | FS-027-EL-003.4, EL-007.2, EL-007.6(수정), EL-012 |
| 메서드·경로 | `PUT /api/support/categories/:id` |
| 권한 | `admin` 만. `operator` → 403 |
| 멱등성 | 멱등(PUT 전체 치환). 같은 바디 재요청은 무변화 |
| 레이트리밋 | 분당 10회 |

**바디**: `SupportCategoryInput`(EP-02 동일). **응답 200/204**.

**사용여부(`active`) 전환은 참조를 건드리지 않는다 (§7.6)**: `active: false` 로 바꿔도 그 유형을 참조하는 티켓·템플릿은 **그대로 유지된다.** 서버는 비활성화를 이유로 참조를 끊거나 마이그레이션하지 않는다 — 소프트 비활성의 정의가 '신규 선택지에서만 숨긴다'이기 때문이다(FS-027-EL-013).

**라벨 전파 (§7.4)**: 티켓(`Ticket.categoryLabel`)과 템플릿(`ReplyTemplate.categoryLabel`)이 라벨을 비정규화 보관한다. 서버는 **같은 트랜잭션 안에서** 참조 항목의 표시 라벨을 정합시키거나, 조회 시 조인해 내려준다. 둘 중 하나를 택해 계약으로 고정한다 — 프론트 픽스처는 전자를 흉내 낸다(`store.ts:250-262` 가 `tickets`·`templates` 를 함께 갱신).

**에러**: 400 `VALIDATION_FAILED` · 401 · 403 · 403 CSRF · **404 `CATEGORY_NOT_FOUND`**(§7.3) · **409 `CATEGORY_NAME_DUPLICATED`**(다른 유형과 이름 충돌 — §7.2) · **422 `LAST_ACTIVE_CATEGORY`**(마지막 활성 유형의 사용여부를 끄려는 시도 — §7.6) · 429 · 500 · 504.

**어댑터 본문 요구사항(시그니처 불변)**: 쓰기 함수 전부 `X-CSRF-Token` 헤더 · `fetchAll` 은 두 참조 수(`ticketCount`·`templateCount`)가 실린 표현을 그대로 받는다 · 현재 `wait(LATENCY_MS=400)`+`failIfRequested(scope, op)` 자리가 실제 `fetch` 로 바뀐다. **`createStoreAdapter` 는 존재 검사를 하지 않으므로(§7.3) 실연동 시 서버 404/422 를 그대로 던지는 경로가 반드시 생겨야 한다.**

## 7. 핵심 판정

### 7.1 유형 참조 무결성 — 서버가 정본, 참조 중은 422 【보안·정합 판정】

참조 중 유형 삭제 차단을 **프론트 버튼 잠금(FS-027-EL-003.5)만으로 두는 것은 불충분하다.** 프론트는 목록 조회 시점의 `ticketCount + templateCount` 로 버튼을 잠그는데(`CategoriesPage.tsx:115` `categoryInUse(category)`), 그 값은 **조회 시점의 스냅샷**이다.

**이 화면에서 TOCTOU 는 특히 현실적이다.** 포트폴리오 카테고리(BE-024 §7.1)의 참조는 다른 **관리자**가 만들지만, **문의 유형의 참조는 고객이 만든다** — 관리자가 목록을 보는 동안에도 고객이 그 유형으로 1:1 문의를 접수하면 `ticketCount` 가 0에서 1이 된다. 관리자는 그 사실을 알 수 없고(`staleTime` 30초 · `refetchOnWindowFocus: false`), '사용 안 함' 배지를 보고 삭제를 누른다. 프론트가 이 경합을 **원리적으로 이길 수 없는** 구조다.

**서버가 삭제 트랜잭션 안에서 두 참조 축(티켓·템플릿)을 재확인**하고 1건이라도 있으면 거절한다. 프론트 잠금은 **UX 편의**(왜 못 지우는지 미리 알려 주는 것)이고, **서버 검사가 정본**이다. BE-010 §7.7 과 같은 논증 구조다.

**상태코드는 422 `CATEGORY_IN_USE` 로 정한다 — 심 주석(`409`)과 다르다.** 근거 셋:

1. **409 는 이 화면에서 이미 임자가 있다.** §7.2 가 유형 이름 중복을 409 `CATEGORY_NAME_DUPLICATED` 로 막는다. 참조 중 삭제까지 409 로 두면 한 리소스에서 409 가 **복구 행동이 정반대인 두 의미**(이름을 바꿔라 / 참조를 먼저 끊어라)를 갖는다.
2. **semantic 이 다르다.** 409 Conflict = '요청이 리소스의 현재 표현과 충돌한다'(같은 이름이 이미 존재). 422 Unprocessable Content = '요청은 형식상 유효하나 도메인 상태 규칙을 위반한다'(참조가 남아 있어 지울 수 없다). 삭제 요청에 충돌하는 표현이 있는 게 아니라 도메인 불변식이 삭제를 거부하는 것이다.
3. **선례가 확정돼 있다.** BE-010 §7.7 이 같은 문제(사용 중 FAQ 카테고리 삭제 차단)를 422 `CATEGORY_IN_USE` 로 정했고 BE-024 §7.1(포트폴리오 카테고리)이 이를 따랐다. 같은 앱의 세 카테고리 화면이 화면마다 다른 상태코드를 쓰면 에러 계약이 화면 지식이 된다.

심 주석의 `409` 는 코드 작성 시점의 약칭으로 보이며, 저장소 구현(`removeCategory` 가 던지는 것)은 상태코드를 갖지 않는 generic `Error`(`store.ts:268`)라 **실제로 409 를 반환하는 코드는 없다**. 계약을 이 명세로 고정하고 주석을 갱신한다(§7.10 #5).

### 7.2 유형 이름 중복 — 정규화 이름 유니크(409)

`addCategory`(`_shared/store.ts:242-248`)는 **중복 검사 없이 배열에 덧붙인다.** 프론트도 검사하지 않는다(FS-027-EL-007.1 경합 열). 그래서 지금은 '배송' 유형이 둘 만들어질 수 있다.

유형은 **표시명(label)이 유일한 식별 단서**다 — 티켓 목록의 유형 컬럼, 티켓 필터, 문의 폼의 선택지가 label 로만 구분된다. 동명 유형 둘은 관리자가 어느 쪽을 고르는지 알 수 없게 만들고, 참조가 둘로 갈려 삭제 차단 판단과 SLA 통계까지 흐린다.

**앞뒤 공백 제거 + 대소문자 무시**로 정규화한 이름의 중복을 **409 `CATEGORY_NAME_DUPLICATED`** 로 막는다(EP-02 등록 · EP-03 수정 모두). 유니크 제약은 DB 레벨에 둬 동시 생성 경합에서도 성립하게 한다. BE-010 §7.5 · BE-024 §7.2 와 같은 판정이다.

**비활성 유형도 유니크 검사에 포함한다** — `active: false` 인 '배송'이 있는데 새 '배송'을 만들 수 있으면 목록에 동명이 둘 나타난다(이 화면은 비활성도 보여준다 — FS-027-EL-003).

### 7.3 유령 저장·유령 삭제 — 존재 검사가 서버 책임 【정합 판정】

`createStoreAdapter`(`shared/crud/crud.ts:107-137`)는 `update`/`remove` 에서 **존재 검사를 하지 않고** store 함수에 그대로 위임한다. 그리고 store 는:

- `updateCategory`(`_shared/store.ts:250-262`) — `categories.map(...)` 이라 **없는 id 는 조용히 지나가고 성공을 반환**한다.
- `removeCategory`(`_shared/store.ts:265-271`) — 없는 id 면 `countCategoryUsage` 가 `{tickets: 0, templates: 0}` 을 반환해 차단을 통과하고, `filter` 가 아무것도 지우지 않은 채 성공을 반환한다.

결과: 다른 관리자가 방금 지운 유형을 수정·삭제하면 **'저장했습니다'/'삭제했습니다' 토스트가 뜨지만 실제로 일어난 일은 없다.** quality-bar EXC-04 가 이름 붙인 유령 저장(ghost saved)이다.

**같은 파일의 `createCrudAdapter` 는 이미 이것을 고쳤다** — `update`(crud.ts:71-73)·`remove`(crud.ts:82-84)가 `items.some(...)` 로 확인하고 없으면 409 를 던진다. `createStoreAdapter` 만 그 검사를 물려받지 못했다(비대칭 — FS-027 §7 #5).

**판정**: 서버가 정본이다. EP-03·EP-04 는 대상이 없으면 **404 `CATEGORY_NOT_FOUND`** 로 답한다(409 가 아니다 — 충돌이 아니라 대상 부재다). **낙관적 동시성 토큰(If-Match/ETag)은 도입하지 않는다** — `SupportCategory` 에 `updatedAt`/`version` 이 없고(§3), 필드가 `label`·`active` 둘뿐이라 병합할 것이 없다. 마지막 쓰기가 이긴다(BE-010 §7.4 와 같은 결). 단 **대상 부재는 반드시 404 로 알린다**.

### 7.4 라벨 비정규화 전파 — 서버 트랜잭션이 정합

티켓(`Ticket.categoryLabel` — `domain.ts:70-71`)과 템플릿(`ReplyTemplate.categoryLabel` — `domain.ts:368-369`)이 라벨을 비정규화 보관하고, 픽스처의 `updateCategory` 가 라벨 변경 시 **두 컬렉션을 모두** 갱신한다(`store.ts:256-261`). 백엔드 연결 후 이 정합은 **서버 트랜잭션 책임**이다: EP-03 이 라벨을 바꾸면 같은 트랜잭션 안에서 참조 티켓·템플릿의 표시 라벨을 정합시키거나, 애초에 조인으로 내려준다. **'라벨을 바꿨는데 티켓 목록에는 옛 라벨이 남는' 상태가 관측되면 안 된다**는 것이 계약이다.

**프론트에 남는 구멍**: `useCrudUpdate` 는 `['support-categories', …]` 키만 무효화한다(`crud.ts:198-201`). 티켓·템플릿 목록 캐시는 무효화되지 않아 **서버가 정합해도 프론트는 `staleTime` 30초까지 옛 라벨을 보인다**(FS-027 §7 #12 — STATE-06 P1). 서버 계약이 아니라 프론트 무효화 범위 문제이므로 §7.10 #6 으로 이관한다.

### 7.5 유형 수 상한 — 100개

프론트가 목록을 **전량 렌더**하고(FS-027-EL-003) 페이지네이션이 없다. 문의 폼·템플릿 폼의 유형 선택지와 티켓 목록의 유형 필터도 이 목록에 비례해 늘어난다.

**상한 100개.** 도달하면 EP-02 가 **422 `CATEGORY_LIMIT_EXCEEDED`** 로 거절한다. BE-010 §7.3 · BE-024 §7.5 와 같은 값·같은 근거다.

### 7.6 사용여부(active) — 소프트 비활성의 계약과 '마지막 활성 유형' 차단 【정합 판정】

`active` 는 **삭제 차단과 독립된 두 번째 축**이다(§3). 계약은 셋이다:

1. **비활성화는 참조를 끊지 않는다.** `active: false` 로 바꿔도 그 유형을 참조하는 티켓·템플릿은 유지되고 계속 그 라벨로 보인다. 서버는 비활성화를 이유로 참조를 마이그레이션하지 않는다.
2. **비활성 유형은 신규 선택지에서만 숨는다.** `listActiveCategories()`(`store.ts:238-240`)가 하는 일이며, 그 소비자는 1:1 문의·문의 답변 화면이다 — **이 명세의 범위 밖**이다. 그 화면들의 BE 가 '활성만 내려주는 표현'을 정의한다. 여기서는 **`active` 를 저장·반환하는 것까지**가 계약이다.
3. **마지막 활성 유형은 끌 수 없다 — 422 `LAST_ACTIVE_CATEGORY`.**

세 번째가 새 판정이다. 근거: 모든 유형의 `active` 를 끄면 `listActiveCategories()` 가 **빈 배열**을 반환하고, 신규 문의의 유형 선택지가 사라진다 — 고객이 1:1 문의를 넣지 못하거나 유형 없는 티켓이 생긴다. **이 화면에는 그것을 막는 장치가 없다**(FS-027 §7 #11 — 모달의 토글은 다른 유형의 상태를 알지 못한다). 관리자 한 명이 유형을 하나씩 끄다가 고객 문의 접수를 통째로 막을 수 있고, 그 사실은 고객이 문의를 못 넣을 때에야 드러난다.

프론트가 이 검사를 하려면 '지금 활성이 몇 개인가'를 알아야 하는데 모달은 단건만 안다. **서버는 항상 안다** — 그래서 서버가 막는다. 활성 유형이 정확히 1개이고 그것을 끄려는 `PUT` 이면 422 로 거절하고 `message` = '마지막 활성 유형은 사용여부를 끌 수 없습니다.' 로 답한다. 프론트에 전용 UI 가 없어 generic 배너로 떨어지는 것은 §7.10 #7 로 이관한다.

**삭제와는 별개다**: 마지막 유형을 **삭제**하는 것은 §7.1 이 이미 다룬다(참조가 있으면 차단, 없으면 허용) — 참조 없는 유형이 하나뿐일 때 지우면 유형이 0개가 되지만, 그건 '카테고리를 다 지웠다'는 명시적 의도이고 목록이 빈 상태로 그것을 보여준다(FS-027-EL-004). 반면 비활성화는 **목록에 그대로 보이면서** 선택지만 조용히 비우므로 관측되지 않는다 — 그래서 비활성화만 막는다.

### 7.7 단건 조회 — 지금은 계약에 넣지 않는다

어댑터가 `fetchOne` 을 배선했지만 심 주석에 없고 소비자도 없다(§4 EP-05). **심 없는 엔드포인트를 계약에 지어 넣지 않는다.** 수정 모달이 목록 행 데이터를 그대로 쓰는 현재 설계에서는 필요하지 않다. 모달이 열린 사이 다른 관리자가 그 유형을 바꿔도 모달은 열릴 때의 값으로 덮는다 — §7.3 의 '마지막 쓰기 승리'와 일관된다.

### 7.8 유형 이름 XSS — 서버가 저장 시 정제한다 【보안 판정】

유형 이름은 **관리자 입력**이며 **고객 화면에 노출된다** — 고객이 1:1 문의를 넣을 때 유형 선택지로 그대로 나가고, 고객이 자기 문의 내역을 조회하면 유형 라벨이 표시된다. 관리자 화면은 JSX 텍스트 보간(`{category.label}`)이라 자동 이스케이프되지만, **고객 렌더가 HTML 로 해석하면 저장형 XSS** 가 된다. 관리자 화면이 안전하다는 것은 계약이 아니다 — 저장된 값이 안전해야 계약이다.

**전파 경로가 특히 넓다**: 라벨은 티켓·템플릿의 `categoryLabel` 로 **복제**되므로(§7.4), 오염된 이름 하나가 정제되지 않은 채 여러 레코드로 퍼진다. 저장 시점 정제가 전파 시점 정제보다 확실한 이유다.

BE-010 §7.2 와 동일 판정 — 서버가 저장 시 마크업·`javascript:` 스킴·이벤트 핸들러를 제거하고, 계약은 '저장된 유형 이름에 실행 가능한 스크립트가 없다'는 관측 동작만 정한다. 유형 이름은 **서식이 필요 없는 짧은 라벨**이므로 허용 태그 화이트리스트는 **빈 집합**이다(모든 마크업 제거).

정제는 **정규화(§7.2)보다 먼저** 적용한다 — 정제 후 이름으로 중복을 판정해야 `<b>배송</b>` 과 `배송` 이 같은 이름으로 수렴한다.

### 7.9 등록 중복 제출 — 서버가 방어선을 겸한다

EP-02 는 비멱등이고 멱등키가 없다. 그런데 프론트에 **동기 제출 락이 없다**(FS-027 §7 #3 — 모달이 `useCrudCreate` 저수준 훅을 직접 쓰고 `disabled={saving}` 렌더 가드에만 의존한다). RHF 의 zod resolver 가 비동기라 `isPending` 이 true 로 렌더되기 전 빠른 두 번째 Enter 가 **두 번째 POST 를 만든다.**

프론트 수정이 정답이고 그것을 이관했다(§7.10 #2). **서버도 방어선을 하나 갖는다**: §7.2 의 이름 유니크 제약이 중복 생성을 자동으로 막는다 — 두 번째 요청은 409 를 받는다. 우연한 방어이지 설계가 아니며 사용자는 성공 대신 실패 배너를 보게 되므로 프론트 락이 여전히 필요하다. **멱등키는 도입하지 않는다** — 유형 생성은 금액·발송이 아니고 유니크 제약이 결과적 멱등성을 준다.
