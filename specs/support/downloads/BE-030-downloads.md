# BE-030. 자료실 백엔드 기능 명세

## 1. 개요

| 항목 | 내용 |
|---|---|
| 대상 화면 | FS-030 자료실 (`/support/downloads` · `/new` · `/:id/edit`) |
| 범위 | 자료 목록 조회, 자료 상세 조회, 자료 등록, 자료 수정(노출 토글 포함), 자료 삭제(단건·일괄), **파일 업로드(`POST /api/uploads`)**, 업로드된 파일의 다운로드 제공 |
| **범위 밖** | **자료 카테고리 관리** — 카테고리가 클라이언트 상수(`DOWNLOAD_CATEGORY_OPTIONS`)이며 조회·생성·삭제 심이 없다(§7.6 에서 정본을 판정). **다운로드 수 집계 로직** — 관리자는 읽기만 하고 증가 심이 없다(§7.7). **일괄 전용 엔드포인트**(§7.1). **고객센터 자료실 렌더** 자체 |
| 프론트 어댑터 | `apps/admin/src/pages/support/downloads/data-source.ts` (`createCrudAdapter` + 자체 시드 — 존재 검사·멱등 ledger 포함) |
| 판정 기준일 | **2026-07-17 · HEAD = `4b805ad`** (F3a·F3b·통합 머지 후). 이전 판정은 F2(`3cd3078`) 기준이었다 |
| 도메인 타입 | `apps/admin/src/pages/support/downloads/types.ts` |
| 검증 정본 | `apps/admin/src/pages/support/downloads/validation.ts` — `downloadSchema`(zod/mini). **파일 규칙은 `types.ts` 의 `downloadFileError`·`MAX_FILE_SIZE_MB`** |

> **에러 봉투·권한 모델 상속**: BE-003 §2·§3 을 그대로 상속한다. 아래는 자료실 고유 차이만 기술한다.

## 2. 공통 (상속)

- **에러 봉투**: BE-003 §2. 공통 에러코드 동일(`error.code`·`message`·`fields`·`traceId`).
- **권한**: `admin` = 전체. `operator` = 조회 계열만(목록·상세), 쓰기(등록·수정·노출 토글·삭제·**업로드**) → **403**. 고객센터 읽기 권한 없는 관리자 → 컬렉션 **403** / 개별 자료도 **403**(§7.5 — 자료는 개인정보가 아니라 은닉 대상이 아니다).
- **CSRF**: 쓰기(POST·PUT·DELETE)에 `X-CSRF-Token`. **업로드(EP-06)도 포함**한다. 어댑터 **함수 시그니처는 바뀌지 않는다** — 본문이 헤더를 싣는다.
- **타임아웃**: 조회·쓰기 5초 → 504. **업로드는 예외 — 60초**(20MB 상한 × 저속 회선, §4 EP-06).
- **프론트 역할 분기 없음**(FS-030 §4.1) — 권한 강제는 전적으로 서버 책임.

## 3. 데이터 계약 (`types.ts` 대조)

| 타입 | 필드 | 비고 |
|---|---|---|
| `DownloadItem` | `id`(string) · `title`(string) · `categoryLabel`(string) · `version`(string) · `fileName`(string) · `fileSize`(number, 바이트) · `fileKind`(`'document'\|'image'\|'archive'\|'etc'`) · `downloadCount`(number) · `visible`(boolean) · `createdAt`(string, ISO) | **`updatedAt`/version 토큰 없음**(§7.4). **파일 URL 필드 없음**(§7.2 — 계약 확장 필요) |
| `DownloadInput` | `title` · `categoryLabel` · `version` · `fileName` · `fileSize` · `fileKind` · `visible` | `id`·`downloadCount`·`createdAt` 제외 — **서버가 소유**한다(`downloads.test.ts:99-104` 가 이 계약을 회귀 고정) |

- **`version` 은 문서의 판**('v2.1')이지 **낙관적 동시성 토큰이 아니다**(§7.4). 이름이 같아 혼동하기 쉬우므로 명시한다.
- **`fileKind` 는 파일명 확장자에서 파생된 값**(`fileKindOf`)이며 클라이언트가 계산해 보낸다 — **서버는 이를 신뢰하지 않는다**(§7.2).
- `downloadCount`·`createdAt` 은 **응답 전용**이다. 등록 시 서버가 `0`·`now` 로 채우고, 수정 시 **보존**한다(`build`/`patch` 규약). 노출 토글이 전체 입력을 다시 보내도 이 둘은 입력에 없어 유지된다.
- 정렬 규칙(프론트 재현): `createdAt` 내림차순, 동률은 `id`(`sortDownloads`). 서버도 같은 순서로 내려야 한다.

### BE-030-EP-03 · 자료 등록

| 항목 | 내용 |
|---|---|
| 근거 (FS) | FS-030-EL-019(등록) |
| 메서드·경로 | `POST /api/support/downloads` |
| 권한 | `admin` 만. `operator` → 403 |
| 멱등성 | **비멱등. 단 `Idempotency-Key` 헤더를 받는다** — `useCrudForm` 이 제출 시도 단위로 키를 만들어(`useCrudForm.ts:112-117`) 재시도가 같은 키를 재사용한다. 서버는 같은 키의 재요청에 **최초 응답을 재생**한다(중복 생성 금지) |
| 레이트리밋 | 분당 60회 |

**바디**(`DownloadInput`): `title`(1–100자) · `categoryLabel`(1–30자, §7.6) · `version`(0–20자, 빈 문자열 허용) · `fileName`(1–260자) · `fileSize`(정수 ≥ 0) · `fileKind`(열거) · `visible`(bool).

**서버가 채우는 값**: `id` · `downloadCount = 0` · `createdAt = now`. 클라이언트가 이 셋을 보내면 **무시**한다(신뢰 경계).

**응답 201** — 생성 `DownloadItem`. 프론트 `create(...): Promise<void>`.

**에러**: 400 `VALIDATION_FAILED`(`error.fields`: `title`·`categoryLabel`·`version`·`fileName`·`fileSize`) · 401 · 403 · 403 CSRF · 422 `CATEGORY_NOT_FOUND`(§7.6 채택 시) · **422 `FILE_NOT_UPLOADED`**(§7.2 — 참조하는 파일이 업로드되지 않았음) · 429 · 500 · 504.

### BE-030-EP-04 · 자료 수정 (노출 토글 공용)

| 항목 | 내용 |
|---|---|
| 근거 (FS) | FS-030-EL-019(수정), **FS-030-EL-008.9(노출 토글)** |
| 메서드·경로 | `PUT /api/support/downloads/:id` |
| 권한 | `admin` 만. `operator` → 403 |
| 멱등성 | **멱등**(PUT 전체 치환) |
| 레이트리밋 | 분당 120회(목록의 노출 토글이 이 경로를 쓴다) |

**바디**: `DownloadInput`(EP-03 동일 — **전체 치환**).

> **노출 토글이 전용 경로를 쓰지 않는다**: 목록의 노출 스위치가 `PATCH .../visibility` 가 아니라 **이 PUT 에 전체 입력을 다시 실어** 보낸다(`DownloadListPage.tsx:86-95` 가 `toDownloadInput(item)` + `visible: next`). 심(`data-source.ts:64`)에 PATCH 가 없으므로 **PATCH 경로를 지어내지 않는다.** 대가는 §7.3 에서 판정한다.

**보존 규칙**: `downloadCount`·`createdAt` 은 바디에 없으므로 **서버가 기존 값을 유지**한다(`patch` 규약). 클라이언트가 보내더라도 무시한다.

**응답 200/204**.

**에러**: 400 · 401 · 403 · 403 CSRF · **404 `DOWNLOAD_NOT_FOUND`** · **409 `CONFLICT`**(다른 관리자가 먼저 삭제한 항목 — §7.4) · 412 `PRECONDITION_FAILED`(낙관적 잠금 도입 시 — §7.4) · 422 `CATEGORY_NOT_FOUND` · 422 `FILE_NOT_UPLOADED` · 429 · 500 · 504.

### BE-030-EP-05 · 자료 삭제 (단건·일괄 공용)

| 항목 | 내용 |
|---|---|
| 근거 (FS) | FS-030-EL-008.11, EL-015, EL-007.1, EL-016 |
| 메서드·경로 | `DELETE /api/support/downloads/:id` |
| 권한 | `admin` 만. `operator` → 403 |
| 멱등성 | **멱등**. 이미 삭제 재요청 → 204 |
| 레이트리밋 | 분당 60회(일괄이 항목당 개별 요청) |

**응답 204**. 일괄 삭제는 프론트가 `settleAll` 로 항목마다 병렬 DELETE 한다(§7.1).

> **프론트 픽스처와의 차이 (유효 — 라인만 갱신)**: `createCrudAdapter.remove`(`crud.ts:139-141`)는 이미 삭제된 id 에 **409 를 던진다**(조용한 통과를 막으려는 픽스처의 안전 기본값). **계약은 멱등(204)이 정본**이다 — 일괄 삭제의 부분 재시도가 안전해야 하고, 이미 지워진 것을 지우라는 요청은 목표 상태가 이미 달성된 것이다. 백엔드 연결 시 어댑터의 409 는 사라진다.
>
> ⚠ **같은 불일치가 이제 `createStoreAdapter` 에도 있다** — F3b 가 그 팩토리에도 같은 409 를 넣었다(`crud.ts:232-234`). 즉 **앱의 모든 CRUD 어댑터가 '이미 삭제 = 409'** 이고 계약만 204 다. 이 결정은 자매 문서(BE-023 §7.10 #9 · BE-028 §7.5)와 함께 한 번에 확정해야 한다 — **일괄 삭제의 부분 실패 건수가 여기 걸린다**(`useCrudList.tsx:136-143` 이 `settleAll` 의 실패 건수를 그대로 사용자에게 보인다: 이미 삭제된 항목이 '실패 1건'으로 세어지면 거짓 실패다).

**첨부 파일 처리**: 자료 삭제 시 업로드된 파일도 함께 정리한다(고아 파일 방지 — §7.2).

**에러**: 400 · 401 · 403 · 403 CSRF · 404 `DOWNLOAD_NOT_FOUND`(존재한 적 없는 id만) · 429 · 500 · 504.

### BE-030-EP-06 · 파일 업로드 ★ **앱에서 유일한 업로드 엔드포인트**

| 항목 | 내용 |
|---|---|
| 근거 (FS) | FS-030-EL-014.4, EL-014.5, EL-014.6 · §1.1 |
| 메서드·경로 | **`POST /api/uploads`** |
| 권한 | `admin` 만. `operator` → 403 |
| 멱등성 | 비멱등(매 요청이 새 파일 객체를 만든다). `Idempotency-Key` 를 받으면 재시도가 중복 파일을 만들지 않는다(권고) |
| 타임아웃 | **60초**(다른 계열 5초와 다르다 — 20MB × 저속 회선) |
| 레이트리밋 | 분당 20회 |

> **경로가 `/api/support/downloads/...` 아래가 아닌 이유**: 심이 `POST /api/uploads`(`FileUploadField.tsx:8` · `data-source.ts:64`)를 **도메인 밖 공용 경로**로 선언한다. 자료실이 현재 유일한 소비자지만 계약상 범용 업로드다 — 다른 화면(공용 `ImageUploadField`)이 나중에 같은 경로를 쓸 수 있다. **심에 있는 경로를 그대로 따른다.**

**요청**: `multipart/form-data`, 필드 `file`(단일). **다중 업로드 없음**(FS-030-EL-014.4 — 파일 1개).

**응답 201**:
```
{ "fileId": "<불투명 id>", "fileName": "<정규화된 저장 파일명>", "fileSize": <바이트>, "fileKind": "document|image|archive", "url": "<다운로드 URL>" }
```

**서버 검증(정본 — §7.2)**: 확장자 **및 MIME** 화이트리스트 · 용량 상한 **20MB** · 파일명 정규화. **클라이언트 검증(`downloadFileError`)은 UX 편의이며 서버가 정본이다.**

**에러**: 400 `VALIDATION_FAILED`(`file` 누락·비어있음) · 401 · 403 · 403 CSRF · **413 `FILE_TOO_LARGE`**(20MB 초과) · **415 `UNSUPPORTED_FILE_TYPE`**(확장자/MIME 화이트리스트 밖) · 429 · 500 · 504.

> **⚠ 프론트 미연동**: 이 엔드포인트를 **호출하는 코드가 현재 없다**(FS-030 §1.1). `FileUploadField` 가 `File` 을 버리고 이름·용량만 폼에 넘긴다. 연동 시 필요한 계약 확장은 §7.2 가 판정한다.

### BE-030-EP-07 · 파일 다운로드 제공

| 항목 | 내용 |
|---|---|
| 근거 (FS) | (관리자 화면에 다운로드 UI 가 없다 — **고객센터 소비 + 관리자 검증용**) |
| 메서드·경로 | `GET <EP-06 이 반환한 url>` |
| 권한 | **미정 — §7.5 판정 참조** |
| 멱등성 | 멱등(GET) |

> **심 없음 (계약 필수)**: 다운로드 URL 자체는 어댑터 심에 없다. **그러나 파일을 업로드하는 계약이 있으면 그것을 내려주는 계약이 반드시 있어야 한다** — 없으면 EP-06 이 만든 파일에 도달할 수 없다. **경로·권한은 미정**이며 §7.2·§7.5 가 **보안 요구사항만** 확정한다(`Content-Type` 강제 · `Content-Disposition: attachment`). 경로 확정은 A63 후속.

### BE-030-EP-NN · 자료 카테고리 관리 — **심 없음 (미정)**

카테고리가 클라이언트 상수(`types.ts:52-58`)이며 조회·생성·삭제 심이 없다. **정본이 어디인지 미정** — §7.6 판정.

### BE-030-EP-NN · 다운로드 수 증가 — **심 없음 (미정)**

`downloadCount` 는 응답 전용이고 증가시키는 심이 없다(관리자는 읽기만). 고객이 파일을 내려받을 때 증가하는 것이 자연스러우나 **그 지점이 EP-07 인지 별도 이벤트인지 미정** — §7.7 판정.

## 7. 핵심 판정

### 7.1 일괄 삭제 — 전용 엔드포인트를 만들지 않는다

프론트가 `settleAll` 로 단건 DELETE(EP-05)를 병렬 호출하고 부분 실패를 건수로 집계한다(FS-030-EL-016). 단건이 멱등이므로 부분 실패 후 재시도가 안전하다. BE-009 §7.1 · BE-010 §7.1 과 같은 근거로 전용 벌크 계약을 두지 않는다.

### 7.2 【보안 판정 · 최우선】 파일 업로드 — 서버가 정본이고, `.svg`/`.html` 이 저장형 XSS 표면이다

**이 배치 최대의 보안 표면이다.** 자료실은 앱에서 유일하게 파일을 받는 화면이며, 받은 파일은 **고객센터에서 공개 배포**된다.

**관측된 클라이언트 현황(코드 확인)**:
- 검증은 `downloadFileError`(`types.ts:113-121`) — **확장자만 본다**(`fileKindOf` → 파일명의 마지막 `.` 뒤 문자열). **MIME 를 보지 않는다.**
- 허용 확장자에 **`.svg` 가 포함**된다(`types.ts:75` `IMAGE_EXT`). `.html` 은 어느 목록에도 없어 `etc` → 거절된다.
- `<input type="file">` 에 **`accept` 속성이 없다**(`FileUploadField.tsx:249-258`).
- **파일이 전송되지 않는다** — `onSelect(file.name, file.size)` 로 메타만 넘긴다.

**판정**:

1. **서버 검증이 정본이다.** 클라이언트의 `downloadFileError`·`accept` 는 **UX 편의**다 — 잘못 고른 파일을 20MB 올린 뒤 거절당하지 않게 해준다. 서버는 클라이언트를 신뢰하지 않는다:
   - **확장자 화이트리스트** — `types.ts` 의 세 목록(`DOCUMENT_EXT`·`IMAGE_EXT`·`ARCHIVE_EXT`)이 계약의 출발점이다. 목록 **밖은 전부 거절**(415).
   - **MIME 화이트리스트 + 실제 내용 검사(magic number)** — 확장자와 선언 `Content-Type` 과 **실제 바이트가 일치**해야 한다. `payload.exe` 를 `payload.pdf` 로 바꾼 파일이 통과하면 안 된다. 클라이언트는 이 검사를 **할 수 없다**(확장자만 본다).
   - **`fileKind` 를 신뢰하지 않는다** — 클라이언트가 보낸 파생 값이다. 서버가 실제 파일에서 다시 판정하고 불일치면 서버 값이 이긴다.
   - **용량 상한 20MB** — `MAX_FILE_SIZE_MB`(`types.ts:44`)와 일치. 초과는 **413**. 스트리밍 중 상한을 넘으면 즉시 끊는다(전량 버퍼링 금지 — DoS).

2. **`.svg` 저장형 XSS — 허용하되 실행을 막는다.** SVG 는 이미지이면서 **스크립트를 실행할 수 있는 XML 문서**다. `<svg><script>` 나 `onload=` 를 품은 파일이 **원본 오리진에서 렌더되면 세션 탈취**로 이어진다. 셋 중 하나를 택한다:
   - **(a) 권고 — 실행 불가능하게 제공한다**: EP-07 이 **`Content-Type: application/octet-stream`** + **`Content-Disposition: attachment; filename="..."`** 을 강제한다. 브라우저가 렌더하지 않고 내려받는다. **자료실의 목적이 '배포용 파일 다운로드' 이므로 인라인 렌더가 애초에 필요 없다** — 이것이 가장 자연스럽다.
   - **(b) 별도 오리진**: 파일을 앱과 다른 오리진(예: `files.example.com`)에서 제공해 XSS 가 앱 세션에 닿지 못하게 한다. (a)와 병행 가능.
   - **(c) `.svg` 를 화이트리스트에서 제거**: 가장 단순하나 정당한 용도(로고·도면)를 막는다.
   - **결정: (a) + (b) 를 채택한다.** `Content-Disposition: attachment` 는 `.svg` 뿐 아니라 **모든 자료 파일**에 적용한다 — `.html` 이 나중에 화이트리스트에 들어와도 안전하고, `.pdf` 의 인라인 렌더러 취약점도 함께 닫힌다. 추가로 **`Content-Security-Policy: sandbox`** 와 **`X-Content-Type-Options: nosniff`** 를 응답에 싣는다(nosniff 가 없으면 브라우저가 `octet-stream` 을 무시하고 내용을 스니핑해 SVG 로 렌더할 수 있다 — **이 헤더가 (a)의 전제다**).

3. **저장 파일명 정규화 — path traversal 차단.** 업로드 파일명은 **사용자 입력**이다. `../../etc/passwd` · `..\\` · 널 바이트 · 유니코드 제어문자 · 예약어(`CON`·`NUL`)가 올 수 있다. **저장 경로에 원본 파일명을 쓰지 않는다** — 서버가 생성한 불투명 id 로 저장하고, 원본 파일명은 **메타데이터로만** 보관해 `Content-Disposition` 의 `filename*`(RFC 5987)에 인코딩해 돌려준다. `DownloadItem.fileName`(260자)은 **표시용**이며 저장 경로가 아니다.

4. **계약 확장 필요 — `DownloadItem` 에 파일 참조가 없다.** 현재 타입에 `fileId`/`url` 필드가 없어(§3) **업로드된 파일과 자료 레코드를 잇는 수단이 계약에 존재하지 않는다.** 연동 시:
   - EP-06 이 `fileId` 를 반환 → 폼이 보관 → EP-03/04 의 `DownloadInput` 에 **`fileId` 추가**.
   - EP-03/04 는 그 `fileId` 가 **존재하고 · 미참조이고 · 같은 관리자가 올린 것**인지 확인한다. 아니면 **422 `FILE_NOT_UPLOADED`**.
   - `DownloadItem` 응답에 **`url` 추가**(고객센터·관리자 검증용).
   - **고아 파일 정리**: 업로드됐으나 일정 시간(예: 24시간) 안에 자료에 참조되지 않은 파일은 삭제한다. 자료 삭제(EP-05) 시에도 파일을 함께 정리한다.

5. **`fileSize` 를 신뢰하지 않는다.** 클라이언트가 보낸 숫자다. 서버는 **실제 저장된 바이트**로 덮어쓴다 — 목록의 용량 표시가 거짓이 되지 않게.

### 7.3 노출 토글이 PUT 전체 치환을 쓴다 — 심을 따르되 대가를 명시한다

목록의 노출 스위치가 `PATCH .../visibility`(BE-010-EP-08 의 방식)가 아니라 **EP-04 PUT 에 전체 입력을 다시 실어** 보낸다(`DownloadListPage.tsx:86-95`). 심에 PATCH 가 없으므로 **PATCH 를 지어내지 않는다.**

**대가(계약이 감수하는 것)**:
- **덮어쓰기 범위가 넓다** — 다른 관리자가 그 사이에 제목을 고쳤다면, 노출 토글 한 번이 **화면에 캐시된 옛 제목으로 되돌린다**(lost update). PATCH 였다면 `visible` 만 바뀌었을 일이다.
- **`downloadCount`·`createdAt` 은 안전하다** — `DownloadInput` 에 없어 서버가 보존한다(§3).
- **완화**: `staleTime: 30초`라 캐시가 최대 30초 낡을 수 있다. §7.4 의 낙관적 잠금이 도입되면 이 경로가 412 를 받아 사용자가 인지하게 된다 — **§7.4 가 이 문제의 실질적 해법이다.**
- **권고**: 프론트가 부분 수정을 원하면 `PATCH /api/support/downloads/:id` 를 심에 추가한 뒤 계약을 확장한다. **심 없이 계약만 만들지 않는다.**

### 7.4 낙관적 동시성 — 현재 미도입, 409 는 '삭제된 항목' 에서만 난다

`DownloadItem` 에 `updatedAt`/version 필드가 **없고**(§3) 어댑터가 `If-Match` 를 싣지 않는다(앱 전역에서 `If-Match` 는 `shared/api/schema.d.ts` 의 회원 메모 계약에만 존재 — 코드 확인). **`version` 필드는 문서의 판이지 동시성 토큰이 아니다.**

**현재 상태**: 프론트에 충돌 UI 는 **이미 있다**(`FormConflictDialog` — 입력 보존 + '최신 불러오기'/'계속 편집'). 그런데 **그것을 띄울 409 가 '다른 관리자가 먼저 삭제한 항목' 에서만 발생**한다(`crud.ts:71-73`). **'다른 관리자가 먼저 수정' 은 탐지되지 않고 마지막 쓰기가 조용히 이긴다.**

**판정**: BE-010 §7.4 와 달리 **이 화면은 낙관적 잠금을 도입할 준비가 돼 있다** — 충돌 UI 가 이미 배선돼 있기 때문이다(BE-010 은 UI 가 없어서 미도입을 택했다). 다음을 **묶어서** 도입할 것을 권고한다(셋 중 하나만 하면 프론트가 처리하지 못하는 응답이 생긴다):
1. `DownloadItem` 에 `updatedAt`(또는 불투명 `ETag`) 추가 → EP-01/02 응답에 포함.
2. EP-04 가 **`If-Match` 필수** — 누락 시 400(`error.fields[].name = "If-Match"`), 불일치 시 **412 `PRECONDITION_FAILED`**.
3. 어댑터가 `fetchOne`/`fetchAll` 의 토큰을 보관해 `update` 에 싣는다(시그니처 불변 — 본문이 처리).

`useCrudForm.handleWriteError` 가 **이미 `isConflict` 로 409·412 를 함께 잡으므로**(`useCrudForm.ts:159-172`) 프론트 변경 없이 412 가 충돌 다이얼로그로 흐른다. **§7.3 의 lost update 도 이것으로 닫힌다.**

### 7.5 【보안 판정】 403 vs 404 은닉 · 다운로드 URL 접근 권한

**관리자 API(EP-01~06)**: BE-003 §3.2 의 두 원칙을 적용한다.
1. **컬렉션의 존재는 비밀이 아니다** → EP-01·EP-03·EP-06 은 권한 부족 시 **403**.
2. **개별 자료가 은닉 대상인가?** — 회원(BE-003)은 '존재 자체가 개인정보' 라 404 로 은닉했다. **자료는 고객센터에 공개 배포되는 콘텐츠**이며 id 열거로 얻을 비밀이 없다. 따라서 EP-02·EP-04·EP-05 는 권한 부족 시 **403**. 404 는 **실제로 없는 자료**에만 쓴다.

**다운로드 URL(EP-07) — 미정이나 요구사항은 확정한다**: 관리자 화면에 다운로드 UI 가 없어 **코드로는 판단할 수 없다.** 자료실의 목적이 '고객에게 배포' 이므로 **공개(비인증)가 자연스러우나** 다음이 미정이다:
- **`visible=false` 인 자료의 파일이 URL 로 접근 가능한가?** — 숨긴 자료의 파일이 URL 만 알면 받아진다면 '숨김' 이 보안 통제가 아니라 UI 필터일 뿐이다. **판정: `visible=false` 인 자료의 파일은 비인증 요청에 404 를 준다**(관리자는 미리보기 위해 인증 상태로 접근 가능). 그렇지 않으면 '아직 공개하면 안 되는 가격표' 를 숨겨도 유출된다.
- **URL 이 추측 가능한가?** — 순차 id(`dl-5`)면 열거로 미공개 자료를 훑을 수 있다. **판정: 불투명·비순차 id**(§7.2-3 의 저장 id 를 그대로 쓴다).
- **경로·인증 방식**(서명 URL vs 세션 쿠키 vs 공개)은 **미정 — A63 후속**. 위 두 요구사항은 어느 방식을 택하든 성립해야 한다.

**모든 다운로드 응답에 필수(§7.2-2)**: `Content-Disposition: attachment` · `X-Content-Type-Options: nosniff` · `Content-Type: application/octet-stream`.

### 7.6 자료 카테고리의 정본 — 미정

카테고리가 **클라이언트 상수**다(`types.ts:52-58` — 카탈로그·제품 매뉴얼·양식/서식·브로슈어·기타 5종 하드코딩). 조회·생성·삭제 심이 없다. `DownloadItem.categoryLabel` 은 id 가 아니라 **라벨 문자열**이다.

**두 갈래 — A11 이 결정한다**:
- **(a) 클라이언트 상수 유지**(현 구현): 서버는 `categoryLabel` 을 **자유 문자열(1–30자)로 받는다.** 검증하지 않으므로 422 `CATEGORY_NOT_FOUND` 가 없다. 단순하나 카테고리를 늘리려면 배포해야 하고, 오타가 그대로 저장돼 필터가 조용히 0건이 된다.
- **(b) 서버 정본**(권고): 카테고리 조회 엔드포인트를 추가하고 `DownloadInput` 이 `categoryId` 를 보낸다. 서버가 존재를 검증해 **422 `CATEGORY_NOT_FOUND`**. FAQ(BE-010-EP-02)가 이미 이 패턴이라 **앱 전체가 일관**해진다. **문의 유형(`support/categories`)과는 별개 분류다**(`types.ts:51` 주석).
- **결정 전까지 이 계약은 (a)를 기준으로 쓴다** — 심에 카테고리 엔드포인트가 없기 때문이다. §4 의 422 `CATEGORY_NOT_FOUND` 는 **(b) 채택 시에만** 유효하다.

### 7.7 다운로드 수 증가 지점 — 미정

`downloadCount` 는 응답 전용이며 증가 심이 없다. **고객이 파일을 내려받을 때(EP-07) 증가하는 것이 자연스럽다.** 다만:
- **봇·프리페치·재개(Range 요청)를 어떻게 세는가** — 한 번의 다운로드가 여러 요청일 수 있다.
- **관리자의 검증용 다운로드도 세는가** — 세면 통계가 오염된다.
- **판정: 미정 — A63 후속.** 관리자 화면은 이 값을 **읽기만** 하므로(FS-030-EL-008.8) 이 계약의 관측 동작은 '서버가 관리하는 값이며 EP-03 이 0으로 시작하고 EP-04 가 보존한다' 까지다.

### 7.8 목록 상한 — 전량 반환이며 상한 미정

EP-01 이 전량을 반환하고 **필터·검색까지 클라이언트가 한다**(§4). 자료가 수백 건이 되면 응답·DOM 이 함께 커진다(FS-030 §7 #11). BE-010-EP-01 이 `page`/`size`(상한 100) + 서버 필터로 푼 문제를 이 화면은 전량으로 넘겼다. **현재 계약은 상한을 두지 않는다 — 미정.** 페이징을 도입하면 클라이언트 필터를 서버 쿼리로 옮겨야 하므로(부분집합에 클라이언트 필터를 걸면 결과가 거짓이 된다) **프론트·계약을 함께 바꿔야 한다** — A11·A41 합의.

### 7.9 제목 중복 — 제약을 두지 않는다

같은 제목의 자료 둘이 공존할 수 있다(예: 개정 전후 카탈로그). **`version` 필드가 그 구분을 담당**하므로(§3) 제목 유니크 제약은 오히려 정당한 운영을 막는다. BE-010 §7.5 가 **카테고리명**에 유니크를 건 것과 대비된다 — 카테고리는 필터의 식별자라 동명이 혼란이지만, 자료 제목은 콘텐츠다.
