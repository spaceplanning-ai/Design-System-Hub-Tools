# BE-029. 고객노출 FAQ 큐레이션 백엔드 기능 명세

## 1. 개요

| 항목 | 내용 |
|---|---|
| 대상 화면 | FS-029 고객노출 FAQ 큐레이션 (`/support/faq` — 단일 라우트, 하위 라우트 없음) |
| 범위 | 고객노출 FAQ 목록 조회, 표시 순서 재정렬, 노출 여부 부분 수정, BEST 고정 부분 수정 |
| **범위 밖** | **FAQ 등록·수정·삭제** — 이 화면에 폼이 없고 어댑터에 POST·DELETE 심이 없다. FAQ 작성 계약은 **BE-010(`/content/faq`)** 이 소유한다. **카테고리 관리**(생성·삭제·사용량) — BE-010 소관. **답변(`answer`) 본문** — 이 화면의 도메인 타입에 `answer` 가 없어 읽지도 쓰지도 않는다(§7.2 XSS 판정에서 경계를 명시한다). **고객센터 렌더** 자체 |
| 프론트 어댑터 | `apps/admin/src/pages/support/faq/data-source.ts` |
| 도메인 타입 | `apps/admin/src/pages/support/faq/types.ts` |
| 검증 정본 | **없다** — 이 화면에 폼·zod 스키마가 없다. 클라이언트 순수 규칙은 `types.ts`(`sortCustomerFaqs`·`applyFaqOrder`·`countVisible`)뿐이며 전부 표시·낙관 반영용이다 |
| 판정 기준일 | **2026-07-17 · HEAD = `4b805ad`** (F3a·F3b·통합 머지 후). 이전 판정은 F2(`3cd3078`) 기준이었다 |

> **에러 봉투·권한 모델 상속**: BE-003 §2·§3 을 그대로 상속한다. 아래는 고객노출 FAQ 고유 차이만 기술한다.

## 2. 공통 (상속)

- **에러 봉투**: BE-003 §2. 공통 에러코드 동일(`error.code`·`message`·`fields`·`traceId`).
- **권한**: `admin` = 전체. `operator` = 조회 계열만(목록), 쓰기(재정렬·노출 토글·BEST 고정) → **403**. 고객센터 읽기 권한 없는 관리자 → 컬렉션 **403**(BE-003 §3.2 — 이 화면의 리소스는 전부 컬렉션이며 은닉할 개별 리소스가 없다. §7.5 판정).
- **CSRF**: 쓰기(PUT·PATCH)에 `X-CSRF-Token`. 어댑터 **함수 시그니처는 바뀌지 않는다** — 본문이 헤더를 싣는다.
- **타임아웃**: 조회·쓰기 5초 → 504.
- **프론트 역할 분기 없음**(FS-029 §4.1) — 권한 강제는 전적으로 서버 책임이며, 프론트는 403 을 다른 실패와 같은 문구로 표시한다.

## 3. 데이터 계약 (`types.ts` 대조)

| 타입 | 필드 | 비고 |
|---|---|---|
| `CustomerFaq` | `id`(string) · `question`(string) · `categoryLabel`(string) · `visible`(boolean) · `pinned`(boolean) · `order`(number) | **이것이 전부다.** `answer` 없음 · `categoryId` 없음(라벨만) · `updatedAt`/version 없음(§7.4) |

- `categoryLabel` 은 **표시용 라벨**이다 — 이 화면은 카테고리 id 를 모르고 필터도 하지 않는다. 서버가 조인해 내려준다.
- `order` 는 **고객센터 표시 순서**다(작을수록 위). BE-010 의 `order`(콘텐츠 관리 정렬 순서)와 **같은 값인지 별개 축인지가 §7.1 의 판정 대상**이다.
- `pinned` 는 **BEST/인기 고정** — BE-010 계약에 존재하지 않는 축이다.
- 정렬 규칙(프론트 재현): `order` 오름차순, 동률은 `id` 안정 정렬(`sortCustomerFaqs`). 서버도 같은 순서로 내려야 한다 — 프론트가 재정렬하지 않고 그대로 그린다.

### BE-029-EP-02 · 표시 순서 재정렬

| 항목 | 내용 |
|---|---|
| 근거 (FS) | FS-029-EL-003.3, EL-003.10, EL-003.11 |
| 메서드·경로 | `PUT /api/support/faq/order` |
| 권한 | `admin` 만. `operator` → 403 |
| 멱등성 | **멱등** — 같은 `orderedIds` 재요청은 같은 최종 순서를 만든다. 프론트의 재시도(FS-029-EL-003.11)가 이에 의존한다 |

**바디**: `{ orderedIds: string[] }` — 고객노출 FAQ **전량**의 새 순서. 프론트는 전량 목록을 들고 있으므로 부분집합을 보내지 않는다(`applyFaqOrder` 가 길이·구성 불일치를 낙관 반영 단계에서 이미 거른다).

**서버 동작**: 받은 순서대로 `order` 를 **1..n 으로 다시 매긴다**. `order` 는 유니크 제약이 아니라 서버가 재정렬로 정합하는 파생 값이다.

**응답 200/204**. 프론트 `reorderCustomerFaqs(orderedIds, signal): Promise<void>`.

**에러**: 400 `VALIDATION_FAILED`(`orderedIds` 에 존재하지 않는 id · 중복 id · 전량 구성 불일치) · 401 · 403 · 403 CSRF · 429 · 500 · 504.

### BE-029-EP-03 · 큐레이션 플래그 부분 수정 (노출 · BEST 공용)

| 항목 | 내용 |
|---|---|
| 근거 (FS) | FS-029-EL-003.7(노출), FS-029-EL-003.8(BEST) |
| 메서드·경로 | `PATCH /api/support/faq/:id` |
| 권한 | `admin` 만. `operator` → 403 |
| 멱등성 | **멱등** — 바디의 목표 상태를 그대로 세팅하므로 같은 값 재요청은 무변화. 프론트의 낙관적 토글·재시도(FS-029 §4.1 중복 제출)와 일치하며, **동기 락 부재로 같은 틱에 중복 요청이 나가도 피해가 없는 근거**다(§7.6) |
| 레이트리밋 | 분당 120회 |

**바디**: `{ visible?: boolean }` 또는 `{ pinned?: boolean }` — **PATCH 이므로 보낸 필드만 바꾼다.** 프론트는 항상 한 번에 **한 필드만** 보낸다(`setCustomerFaqVisible` 은 `visible` 만, `setCustomerFaqPinned` 는 `pinned` 만). 두 필드를 함께 보내는 호출부는 없다.

> **왜 `/visibility` 전용 경로(BE-010-EP-08)가 아닌가**: 이 화면은 큐레이션 축이 **둘**(`visible`·`pinned`)이고 앞으로 늘 수 있다(예: 노출 기간). 축마다 경로를 파면 `PATCH /faq/:id/visibility` · `/pinned` 로 갈라진다. 심(`data-source.ts:63`)이 `PATCH /api/support/faq/:id` **하나**를 선언하므로 그 계약을 따른다 — 부분 수정의 대상은 '큐레이션 플래그'이지 개별 필드가 아니다.

**응답 200/204**. 프론트 `setCustomerFaqVisible(id, visible)` · `setCustomerFaqPinned(id, pinned)` — 둘 다 `Promise<void>`.

**에러**: 400 `VALIDATION_FAILED`(`visible`/`pinned` 형식 · 알 수 없는 필드) · 401 · 403 · 403 CSRF · 404 `FAQ_NOT_FOUND` · 422 `PINNED_LIMIT_EXCEEDED`(BEST 상한 도입 시 — §7.3) · 429 · 500 · 504.

## 7. 핵심 판정

### 7.1 【정합 판정 · 최우선】 `/support/faq` 와 `/content/faq` 가 같은 FAQ 를 보지 않는다 — 백엔드가 정본을 통일해야 한다

**관측된 사실(코드 대조)**:
- `pages/support/faq/data-source.ts:12-61` 이 자체 mutable 배열을 들고 시드 id 가 `cfaq-1`…`cfaq-6` 이다.
- `pages/content/faq/` 의 픽스처는 `FAQ-*` id 를 쓴다.
- 두 모듈 사이에 import 가 없고 공유 store 도 없다(`CustomerFaqPage.tsx:5` 가 '페이지 간 직접 import 없이 라우트 링크로 연결' 이라고 명시).

**결과**: 콘텐츠 관리에서 FAQ 를 등록해도 이 화면에 나타나지 않고, 여기서 숨겨도 그쪽 `visible` 과 무관하며, 양쪽의 `order` 가 서로를 모른다. **UI 는 '발행된 FAQ 를 큐레이션한다'고 말하지만(FS-029-EL-001) 데이터가 그 말을 뒷받침하지 않는다.** 픽스처 단계에서는 두 화면을 독립적으로 시연하려는 선택으로 볼 수 있으나, **백엔드 계약에서는 선택의 여지가 없다** — 관리자가 콘텐츠 관리에서 만든 FAQ 가 고객센터 큐레이션 목록에 없으면 이 화면은 거짓말을 한다.

**판정 — 하나의 FAQ 엔티티, 두 개의 관심사**:
1. **FAQ 정본은 하나다.** `GET /api/support/faq`(EP-01)와 `GET /api/faqs`(BE-010-EP-01)는 **같은 테이블**을 읽는다. EP-01 은 그중 **발행된(고객 노출 대상) 것만** 큐레이션 뷰로 투영한다.
2. **`visible` 은 공유 필드다.** BE-010-EP-08(`PATCH /api/faqs/:id/visibility`)과 BE-029-EP-03(`{ visible }`)이 **같은 값을 쓴다**. 두 화면 중 어디서 꺼도 고객센터에서 숨어야 한다. 두 엔드포인트를 유지하되 **같은 필드를 갱신**한다(경로가 둘인 것은 화면 경계의 문제이지 데이터 경계의 문제가 아니다).
3. **`pinned` 는 이 화면 전용 필드다.** BE-010 계약에 없다. 고객센터 큐레이션 축이므로 여기서만 쓴다.
4. **`order` 는 판정이 필요하다.** 두 가지 중 하나를 UI 기획·백엔드 명세 이 확정해야 한다:
   - **(a) 단일 축** — 콘텐츠 관리의 정렬 순서와 고객센터 표시 순서가 같은 값. 구현이 단순하나 두 화면의 재정렬이 서로를 덮는다.
   - **(b) 별도 축**(권고) — `order`(콘텐츠 관리 관리용)와 `displayOrder`(고객센터 표시용)를 분리한다. FS-029 가 '고객센터에서 **어떤 순서로 보여줄지**' 를 이 화면의 존재 이유로 선언하므로(§1.1), 콘텐츠 관리의 정렬과 고객 노출 순서는 다른 관심사다. 이 경우 EP-02 는 `displayOrder` 만 재매김하고 BE-010-EP-09 는 `order` 만 건드린다.
   - **결정 전까지 이 계약은 `order` 를 '고객센터 표시 순서'로만 정의한다**(§3).

**이관**: 백엔드 명세(BE-010 과의 계약 통합) · UI 기획(제품 결정 — `order` 단일/분리, 두 화면의 역할 경계 확정).

### 7.2 【보안 판정】 저장형 XSS — 이 화면은 본문을 편집하지 않으나 표시 문자열은 렌더한다

**범위 경계**: 이 화면은 `answer` 를 **읽지도 쓰지도 않는다**(`types.ts:9-20` 에 `answer` 필드가 없다). 따라서 **FAQ 답변의 XSS 정제 책임은 BE-010 §7.2 에 있고 이 계약은 그것을 재정의하지 않는다** — 여기서 다시 규정하면 같은 필드에 두 벌의 정제 규칙이 생긴다.

**그러나 이 화면에도 XSS 표면이 있다**: `question` 과 `categoryLabel` 을 서버에서 받아 렌더한다(FS-029-EL-003.5/.6). 두 문자열은 **관리자 입력에서 유래**하며(콘텐츠 관리 FAQ 에서 작성) **고객센터에도 노출된다**.

**판정**:
1. **서버가 저장 시 정제한다** — `question`·카테고리명은 BE-010-EP-05/06/11 의 저장 경로에서 이미 정제된다(BE-010 §7.2). 이 화면은 그 결과를 소비할 뿐이므로 **정제 지점을 여기에 두지 않는다**(정제는 저장 경계에서 한 번).
2. **이 화면의 렌더는 안전하다** — React 가 `{faq.question}`·`{faq.categoryLabel}` 을 텍스트 노드로 이스케이프한다. `dangerouslySetInnerHTML` 사용 0건(전 파일 확인). **관리자 화면 쪽에는 XSS 경로가 없다.**
3. **계약의 관측 동작**: EP-01 이 내려주는 `question`·`categoryLabel` 에 실행 가능한 스크립트가 없다 — 이는 저장 경계(BE-010)의 보증을 이 응답이 상속한다는 뜻이다.
4. **토스트 문구 주의**: 성공 토스트가 `'<질문>' 를 노출합니다.` 처럼 `question` 을 문자열 보간한다(FS-029-EL-003.7 · `CustomerFaqPage.tsx:134`·`:165`). 이 역시 React 텍스트 렌더라 안전하나, **향후 이 문구를 HTML 로 렌더하는 토스트 변형을 도입하면 즉시 XSS 경로가 된다** — 금지한다. (별개로 그 문구는 조사를 리터럴 '를' 로 고정해 받침 있는 질문에서 어색하다 — FS-029 §7 #10 · ERP-13. 보간 자체는 그대로 두고 조사만 `shared/format.ts:306` `objectParticle` 로 라우팅하면 되므로 이 보안 판정과 충돌하지 않는다.)

### 7.3 목록 상한 · BEST 고정 상한 — 둘 다 미정

- **목록 상한**: EP-01 이 전량을 반환하고 프론트에 페이지네이션이 없다(FS-029 §7 #3). 발행 FAQ 가 수백 건이 되면 응답과 DOM 이 함께 커지고 재정렬 바디(`orderedIds`)도 그만큼 길어진다. **재정렬이 전량 순서를 다루므로 페이징과 재정렬은 양립하기 어렵다** — BE-010-EP-09 가 '현재 페이지 슬롯 안에서 재배치' 로 푼 문제를 이 화면은 '전량' 으로 피했다. 상한(예: 200건)을 두고 초과 시 큐레이션 대상을 좁히는 축(카테고리 선택 등)을 도입할지 UI 기획 이 결정해야 한다. **현재 계약은 상한을 두지 않는다 — 미정.**
- **BEST 고정 상한**: `pinned` 를 전 건에 켤 수 있고 그러면 상단 강조가 무의미해진다(FS-029 §7 #4). 상한을 도입하면 EP-03 이 **422 `PINNED_LIMIT_EXCEEDED`** 로 거절하고, 프론트는 롤백 + 실패 토스트로 표시한다(현재 문구가 코드별로 갈리지 않으므로 'BEST 고정을 변경하지 못했습니다.' 로 뭉개진다 — §7.6 #2). **현재 상한 미정 — 서버는 상한을 강제하지 않는다.**

### 7.4 재정렬·토글 동시성 — 마지막 쓰기 승리 (낙관적 잠금 미도입)

`CustomerFaq` 에 `updatedAt`/version 필드가 없고(§3) 어댑터가 `If-Match` 를 싣지 않는다(앱 전역에서 `If-Match` 는 `shared/api/schema.d.ts` 의 회원 메모 계약에만 존재한다 — 코드 확인). 프론트가 낙관 반영 후 실패 시 롤백하고 성공 시 재조회로 정합하는 구조라(FS-029-EL-003.7/.8/.11), **서버가 충돌을 409 로 올려도 프론트에 이를 해소할 UI 가 없다**(충돌 다이얼로그 없음).

**판정**: BE-010 §7.4 와 동일 — EP-02·EP-03 에 낙관적 잠금을 두지 않는다. 두 관리자가 동시에 재정렬하면 마지막 요청이 이기고, 동시에 반대로 토글하면 마지막 요청이 이긴다. **EP-03 이 멱등(목표 상태 세팅)이라 토글의 last-write-wins 는 데이터 손상이 아니라 관리자 의도의 경합일 뿐이다.** 동시 편집이 실제 문제가 되면 `updatedAt` 추가 + 409 + 충돌 UI 를 **묶어서** 별도 계약으로 승격한다(셋 중 하나만 도입하면 프론트가 처리하지 못하는 응답이 생긴다).

### 7.5 【보안 판정】 403 vs 404 은닉 — 이 도메인은 전부 403

BE-003 §3.2 의 두 원칙을 이 화면에 적용한다:
1. **컬렉션 리소스의 존재는 비밀이 아니다** → EP-01(목록)·EP-02(재정렬)는 권한 부족 시 **403**.
2. **개별 리소스 은닉이 필요한가?** — 회원(BE-003)은 '개별 리소스의 존재 자체가 개인정보' 라 404 로 은닉했다. **FAQ 는 그 반대다** — 고객센터에 공개 노출되는 콘텐츠이며 id 열거로 얻을 비밀이 없다. 따라서 EP-03(`PATCH /:id`)은 권한 부족 시 **403** 을 준다. 404 는 **실제로 없는 FAQ** 에만 쓴다(콘텐츠 관리에서 방금 삭제된 경우).
3. 프론트가 403 과 500 을 구분하지 않으므로(FS-029 §7 #6) 이 판정은 **당장 화면에 보이지 않는다** — 그러나 계약은 정확해야 한다. 프론트가 status 별 분기를 도입하면(EXC-06) 그때 이 표가 정본이 된다.
