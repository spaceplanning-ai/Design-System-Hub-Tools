# BE-028. 문의 답변 — 답변 템플릿 관리 백엔드 기능 명세

## 1. 개요

| 항목 | 내용 |
|---|---|
| 대상 화면 | FS-028 문의 답변 — 답변 템플릿 관리 (`/support/replies` · `/new` · `/:id/edit`) |
| **관리 대상** | **답변 템플릿(canned response·매크로)** 이다 — 개별 고객 답변이 아니다. 화면명은 '문의 답변'이지만 코드가 일관되게 템플릿을 말한다(FS-028 §1): `ENTITY_LABEL = '답변 템플릿'` · 타입 `ReplyTemplate` · 리소스 키 `'support-templates'` · **심 경로 `/api/support/reply-templates`**. 개별 답변은 BE-026(1:1 문의)의 타임라인 이벤트(`kind='reply'`)가 소유한다 |
| 범위 | 답변 템플릿 목록 조회 · 상세 조회 · 등록 · 수정 · 삭제(단건·일괄) |
| 범위 밖 | **문의 유형 CRUD** — BE-027 소관(이 화면은 유형 태그 선택지의 소비자, §7.3). **템플릿 치환·삽입** — 클라이언트 순수 함수(`applyTemplate`)이며 서버 계약이 아니다(§7.4). **고객 노출 렌더** — §7.1 XSS 판정만. **일괄 전용 엔드포인트** — §7.6 |
| 전제 | BE-003 §2·§3 을 상속한다. 인증은 세션 쿠키 기반. 모든 경로는 `/api` 프리픽스. 응답 본문은 `application/json; charset=utf-8` |
| 프론트 어댑터 | `apps/admin/src/pages/support/replies/data-source.ts` (`replyTemplateAdapter` — **`createStoreAdapter` 로 조립**. F3b 이후 `createCrudAdapter` 와 같은 404/409 계약을 갖는다 — §7.5) |
| 판정 기준일 | **2026-07-17 · HEAD = `4b805ad`** (F3a·F3b·통합 머지 후). 이전 판정은 F2(`3cd3078`) 기준이었다 — §7.5 · §6 · §7.9 가 그 사이 바뀐 프론트 사실을 반영해 갱신됐다 |
| 도메인 타입 | **`types.ts` 가 없다.** `ReplyTemplate` · `ReplyTemplateInput` · 상수가 `apps/admin/src/pages/support/_shared/domain.ts` 에 있다 — 1:1 문의(BE-026)·문의 유형(BE-027)과 도메인을 공유하기 때문이다 |
| 검증 정본 | `apps/admin/src/pages/support/replies/validation.ts` 의 `replyTemplateSchema`(zod) |

> **에러 봉투·권한 모델 상속**: BE-003 §2·§3 을 그대로 상속한다(BE-010 §1 과 동일 선언). 아래는 답변 템플릿 고유 차이만 기술한다.

## 2. 공통 (상속)

- **에러 봉투**: BE-003 §2. 공통 에러코드 동일.
- **권한**: `admin` = 전체. `operator` = **조회 계열만**(목록·상세). 쓰기(등록·수정·삭제) → **403**. **BE-026 과 다른 판정이다** — 근거는 §7.7. 고객센터 도메인 읽기 권한 없는 관리자 → 컬렉션 403 / 개별 404(§7.8).
- **CSRF**: 쓰기(POST·PUT·DELETE)에 `X-CSRF-Token`.
- **타임아웃**: 조회·쓰기 5초 → 504.
- **프론트 역할 분기 없음**(FS-028 §4.1) — 권한 강제는 서버 책임.

## 3. 데이터 계약 (`_shared/domain.ts` 대조)

| 타입 | 필드 | 비고 |
|---|---|---|
| `ReplyTemplate` | `id` · `title` · `categoryId`(유형 태그 id — **공용이면 `''`**) · `categoryLabel`(서버 조인·비정규화. 공용이면 `'전체'`) · `body` | 목록·상세가 **같은 타입**이다 — 목록도 `body` 전문을 받는다(§7.9) |
| `ReplyTemplateInput` | `Omit<ReplyTemplate, 'id' \| 'categoryLabel'>` = `{ title, categoryId, body }` | 라벨은 서버가 조인하므로 제외 |
| 상수 | `TEMPLATE_TITLE_MAX = 60` · `TEMPLATE_BODY_MAX = 1000` · `TEMPLATE_ALL_LABEL = '전체'` | 검증 상한의 정본 |

**검증 정본 (`replies/validation.ts` — 서버가 재현해야 하는 규칙)**

| 필드 | 규칙 | 위반 문구(프론트) |
|---|---|---|
| `title` | `requiredText('템플릿 제목', 60)` — 필수, 앞뒤 공백 제거 후 1–60자 | '템플릿 제목을 입력하세요.' / '템플릿 제목은 60자를 넘을 수 없습니다.' |
| `categoryId` | **`z.string()` — 제약이 없다.** 빈 문자열이면 공용, 아니면 유형 id | 없음 — **어떤 문자열도 통과한다**(§7.2) |
| `body` | `requiredText('본문', 1000)` — 필수, 앞뒤 공백 제거 후 1–1000자 | '본문을 입력하세요.' / '본문은 1000자를 넘을 수 없습니다.' |

**공용 태그 규약**: `categoryId === ''` 는 '전체 유형에 노출되는 공용 템플릿'이다. `null` 이 아니라 **빈 문자열**이며 `categoryLabel` 은 `'전체'` 로 조인된다(`_shared/store.ts` `withTemplateLabel`). 서버 계약도 이 표현을 따른다 — `null`/`undefined` 로 바꾸면 프론트 `templatesForCategory`(`template.categoryId === ''` 비교)와 select 의 `<option value="">` 가 함께 깨진다.

**치환 변수**: 본문은 `{{고객명}}` · `{{문의번호}}` · `{{담당자}}` 를 담을 수 있다. **서버는 이를 해석하지 않는다** — 저장·반환 시 원문 그대로다. 치환은 1:1 문의 상세가 삽입 시점에 클라이언트에서 수행한다(§7.4).

### BE-028-EP-05 · 답변 템플릿 삭제 (단건 · 일괄 공용)
| 항목 | 내용 |
|---|---|
| 근거 (FS) | FS-028-EL-006.7, EL-010, EL-005.2, EL-011 |
| 메서드·경로 | `DELETE /api/support/reply-templates/:id` |
| 권한 | `admin` 만. `operator` → 403 |
| 멱등성 | **멱등.** 이미 삭제 재요청 → 204 |

**응답 204**. 일괄 삭제는 프론트가 `settleAll` 로 항목마다 병렬 DELETE 한다(§7.6).

> **참조 무결성 판정 — 템플릿 삭제는 막지 않는다**: 문의 유형은 참조되면 삭제가 차단되지만(`removeCategory` 가 티켓·템플릿 수를 세어 던진다 — BE-027), **템플릿은 그런 제약이 없다.** 근거: 템플릿은 티켓이 참조하는 것이 아니라 **티켓 답변 작성 시 본문이 복사되는 원본**이다(`applyTemplate` 이 문자열을 만들어 `composer` 에 넣고, 저장되는 것은 타임라인 이벤트의 `text` 다 — 템플릿 id 가 티켓에 남지 않는다). 템플릿을 지워도 이미 보낸 답변은 영향받지 않는다 — **고아가 생기지 않는다.**

## 7. 핵심 판정

### 7.1 템플릿 본문 XSS — 서버가 저장 시 정제한다 【보안 판정】

템플릿 `body` 는 **관리자 입력이며 최종적으로 고객에게 전달된다.** 경로가 한 단계 우회할 뿐이다: 관리자가 템플릿을 저장 → 1:1 문의 상세가 삽입(`applyTemplate` — FS-026-EL-025) → 답변 이벤트 `text` 로 저장(BE-026) → **고객 채널로 발송**. 관리자 화면은 텍스트 노드로만 렌더하지만, 고객 채널(웹 마이페이지·카카오톡 알림톡·이메일)이 HTML 로 해석하면 **저장형 XSS** 다.

BE-010 §7.2 · BE-026 §7.1 과 동일 판정 — 서버가 **저장 시** 허용 태그 화이트리스트 밖의 마크업 · `javascript:`/`data:` 스킴 · 이벤트 핸들러 속성을 제거하고, 계약은 '저장된 본문에 실행 가능한 스크립트가 없다'는 관측 동작만 정한다.

**이 화면 고유의 위험 — 정제는 두 곳에서 각각 해야 한다**: 템플릿 정제(EP-03/04)만으로는 부족하다. 템플릿을 삽입한 뒤 **관리자가 본문을 자유롭게 고칠 수 있으므로**(FS-026-EL-026 의 textarea) 최종 답변에 새 스크립트가 들어갈 수 있다. 반대로 답변 정제(BE-026)만 하고 템플릿을 정제하지 않으면 **오염된 템플릿이 저장소에 남아 매번 정제에 의존**하게 된다. **두 지점 모두 정제한다** — 템플릿은 오염 저장을 막고, 답변은 최종 산출물을 막는다.

`title` 도 정제 대상이다 — 티켓 상세의 삽입 드롭다운 옵션 라벨(`[<유형라벨>] <제목>`)로 렌더된다.

### 7.2 유형 태그 참조 무결성 — 서버가 판정한다 【정합 판정】

`replyTemplateSchema` 의 `categoryId` 가 **`z.string()` 이다**(`validation.ts:10`) — 제약이 없다. 프론트는 select 로 선택지를 좁히지만 그것은 UI 편의일 뿐, **없는 유형 id 를 담은 요청을 막는 것이 아무것도 없다.**

**TOCTOU**: 폼이 유형 목록을 **마운트 시 1회 고정**하므로(`ReplyFormPage.tsx:77` `useMemo(..., [])`) 폼을 열어 둔 사이 다른 관리자가 그 유형을 지우면, 프론트 select 에는 남아 있는 유형 id 로 저장 요청이 나간다.

**판정**: 서버가 저장 트랜잭션 안에서 `categoryId` 참조를 재확인하고 **존재하지 않으면 422 `CATEGORY_NOT_FOUND`** 로 거절한다(빈 문자열 = 공용은 통과). BE-010 §7.7(카테고리 참조 무결성)과 같은 결이다 — **프론트 select 는 UX 편의이고 서버 검사가 정본이다.**

**다만 유형 삭제 자체가 이미 차단된다**: `removeCategory`(`_shared/store.ts:265-271`)가 그 유형을 쓰는 티켓·템플릿 수를 세어 1건이라도 있으면 던진다. 따라서 이 TOCTOU 는 **정상 경로에서는 발생하기 어렵다** — 그러나 서버 검사를 생략할 근거는 되지 못한다(위조 요청·비활성화 후 삭제 등 경로가 남는다).

### 7.3 유형 목록 조회 — 심이 없고, 연결하려면 화면 코드가 바뀐다 【연동 판정】

`ReplyFormPage.tsx:77` 이 `listCategoryUsage()` 를 **어댑터를 거치지 않고 `_shared/store` 에서 동기 직접 호출**한다. `// TODO(backend)` 주석이 없다 — **연동 심이 아예 없다.**

**두 가지가 문제다**:
1. **동기 함수라 어댑터 본문만 바꿔서는 연결되지 않는다.** `Promise` 를 반환하는 순간 `useMemo(() => listCategoryUsage(), [])` 가 성립하지 않아 **화면이 `useQuery` 를 새로 배선해야 하고**, 그러면 FS-028 §4 가 'N/A — 로딩/실패 상태가 없다'로 적어 둔 칸들이 **전부 새 요구사항으로 살아난다**(로딩 스켈레톤·실패 배너·재시도·취소). BE-026 §7.4 와 같은 사안이며 **세 화면(티켓 목록·티켓 상세·템플릿 폼)이 같은 함정을 공유**한다.
2. **필요보다 무거운 조회다.** `listCategoryUsage()` 는 유형마다 **티켓·템플릿 사용 건수를 집계**하는데(`countCategoryUsage` — 전체 티켓·템플릿을 순회), 이 폼은 **`label` 과 `active` 만 쓴다**(`ReplyFormPage.tsx:119-123`). 백엔드가 붙으면 이것은 '유형 목록을 얻으려고 사용량 집계 API 를 부르는' 꼴이 된다. **연결 시 `GET /api/support/categories`(사용량 없이)로 갈아타야 한다** — BE-027 이 두 표현을 모두 제공한다면 가벼운 쪽을 쓴다.

**판정**: 이 문서는 EP-06 의 엔드포인트를 **만들지 않는다** — 문의 유형의 정본은 BE-027 이다. 이 화면은 소비자이며, 연동 시 ① 비동기화 ② 가벼운 표현으로 교체 두 가지를 함께 해야 한다. **어댑터만 바꾸면 되는 작업으로 산정하면 반드시 빠진다.**

### 7.4 치환 변수는 클라이언트가 처리한다 — 서버 계약에 넣지 않는다

`applyTemplate`(`_shared/domain.ts:387-395`)이 `{{고객명}}`·`{{문의번호}}`·`{{담당자}}` 를 `replaceAll` 로 바꾼다. 서버는 템플릿 본문을 **원문 그대로** 저장·반환한다.

**판정: 현 상태를 유지한다.** 치환은 '이 티켓의 값으로 이 템플릿을 채운다'는 **표시 시점 연산**이며, 결과물(관리자가 편집한 최종 답변)만 저장된다 — 서버 왕복이 필요 없다. 순수 함수라 테스트도 있다(`_shared/domain.test.ts` 'applyTemplate 은 치환 변수를 실제 값으로 바꾼다').

**단, 계약에 걸리는 것 하나 — 치환 연쇄 【보안 판정(경미)】**: `replaceAll` 이 **순차 실행**된다(고객명 → 문의번호 → 담당자). 고객명이 문자열 `'{{담당자}}'` 이면, 1단계에서 그것이 본문에 삽입되고 3단계가 **그것을 다시 치환**해 담당자 이름으로 바꾼다. `customerName` 은 **고객이 정하는 값**이므로 이론상 고객이 치환 토큰을 주입할 수 있다.

**영향은 낮다** — 치환되는 값이 전부 이 티켓의 값이라(고객명·문의번호·담당자) 유출되는 정보가 없다. 그러나 **담당자 이름이 예기치 않게 노출**되고, 향후 치환 변수가 늘면(예: `{{내부메모}}`) 실제 유출 경로가 된다.

**판정**: 서버 계약 변경은 불필요하다. 프론트가 **한 번의 패스로 치환**(정규식 하나로 토큰을 찾아 맵에서 값을 꺼내는 방식)하도록 고치면 연쇄가 구조적으로 사라진다 — A11 이관.

### 7.5 대상 부재 · 낙관적 동시성 — 서버가 정본이다 【정합 판정】 *(F3b 로 프론트 전제가 바뀌어 정정)*

**정정 — 두 팩토리가 이제 같은 가드를 갖는다.** F2 때는 `createCrudAdapter` 에만 존재 확인 가드가 있고 `createStoreAdapter` 는 위임만 했으나, F3b 가 후자에도 넣었다:

| 팩토리 | 없는 id 로 `fetchOne` | 없는 id 로 `update` | 없는 id 로 `remove` |
|---|---|---|---|
| `createCrudAdapter` (`crud.ts:86-147`) | **404** (`:105-107`) | **409** `'다른 사용자가 먼저 삭제한 항목입니다.'`(`:126-128`) | **409** `'이미 삭제된 항목입니다.'`(`:139-141`) |
| **`createStoreAdapter`** (`crud.ts:165-240`) | **404** (`:192-194`) | **409** (`:219-221`) | **409** (`:232-234`) |

`replyTemplateAdapter` 는 **`createStoreAdapter`** 로 조립된다(`data-source.ts:19-26`). 위임 대상인 `updateTemplate`(`map`)·`removeTemplate`(`filter`)이 여전히 없는 id 를 조용히 지나치지만, **`:171` 의 `exists()` 가 그 앞을 막는다**(`:210-218` 주석이 '형제 팩토리는 같은 자리에서 409 로 막는데 여기만 뚫려 있었다'고 전환을 밝힌다).

**결과(정정)**: 다른 관리자가 방금 지운 템플릿을 편집하면 → **어댑터가 409 를 던지고** → `useCrudForm` 의 충돌 다이얼로그(FS-028-EL-027)가 **입력을 보존한 채** 열린다 → 성공 토스트도 목록 이동도 없다. **유령 저장은 해소됐고, 이 화면은 화면 코드 0줄 변경으로 그 이득을 받았다**(`:216-217` 주석이 그렇게 설계했음을 명시한다). F2 서술('저장된 것은 아무것도 없는데 성공 토스트가 뜬다')은 더 이상 사실이 아니다.

### 7.6 일괄 삭제 — 전용 엔드포인트를 만들지 않는다

프론트가 `settleAll` 로 단건 DELETE(EP-05)를 항목마다 병렬 호출하고 부분 실패를 건수로 집계한다(FS-028-EL-011). **단건이 멱등(이미 삭제 → 204)이므로 부분 실패 후 재시도가 안전하다** — 성공분을 다시 요청해도 204 다. BE-010 §7.1 과 같은 근거로 전용 벌크 계약을 두지 않는다.

**레이트리밋 주의**: 선택 수만큼 동시 발사되므로 EP-05 의 분당 60회를 큰 일괄(60건 초과)이 스스로 넘길 수 있다. 프론트에 선택 상한이 없다(FS-028 §7 #18). 429 는 부분 실패 건수로 집계돼 '재시도' 안내가 뜨지만, **재시도도 같은 429 를 맞는다** — `Retry-After` 를 존중하는 백오프가 없다. A11 이관.

### 7.7 `operator` 는 조회만 — BE-026 과 다른 판정

**BE-026(1:1 문의)은 `operator` 에게 쓰기를 열고, 이 문서는 닫는다.** 같은 고객센터 도메인인데 판정이 갈리는 이유:

| | 1:1 문의(BE-026) | 답변 템플릿(BE-028) |
|---|---|---|
| 쓰기의 성격 | **한 고객에게 보내는 응대** — 운영자의 본업 | **전 상담원이 공유하는 표준 문구** — 조직의 목소리 |
| 잘못 쓰면 | 그 문의 1건이 영향 | **그 템플릿을 쓰는 모든 미래 답변**이 영향 |
| 되돌리기 | 타임라인에 정정 답변 추가 | 이미 나간 답변은 되돌릴 수 없다 |

템플릿은 **콘텐츠에 가깝다** — BE-010(FAQ)이 `operator` 를 조회 전용으로 두는 것과 같은 이유다. 한 사람이 사과 문구를 잘못 고치면 그 뒤 모든 사과가 잘못된 문구로 나간다. **`operator` 는 템플릿을 읽고 쓰되(EP-01·EP-02 → 티켓 답변에 삽입), 만들고 고치고 지우는 것은 `admin` 이다.**

### 7.8 개별 템플릿은 404 로 은닉한다 【보안 판정】

BE-003 §3.2 의 원칙을 적용한다.

1. **컬렉션의 존재는 비밀이 아니다** → `GET /api/support/reply-templates` 권한 부족 시 **403 `FORBIDDEN`**.
2. **개별 템플릿** → 고객센터 도메인 **읽기 권한이 없는** 주체에게는 **404 `TEMPLATE_NOT_FOUND`**.

**근거 — BE-026 과 민감도가 다르다**: 템플릿에는 **고객 개인정보가 없다**(제목·유형·문구뿐). 그래서 티켓 상세(BE-026 §7.6 — '문의의 존재 자체가 개인정보')만큼 강한 은닉 필요는 없다. **그럼에도 404 를 쓰는 이유는 일관성이다** — 같은 도메인의 개별 리소스가 어떤 것은 403, 어떤 것은 404 를 주면 프론트가 두 규칙을 기억해야 하고, 그 분기가 어긋나는 순간(BE-026 의 404 를 403 으로 잘못 처리) 은닉이 깨진다. **개별 리소스는 404, 컬렉션은 403** 이라는 한 줄이 도메인 전체에 서야 한다.

**읽기 권한이 있는 주체가 쓰기에서 거절될 때는 403**(§7.7 의 `operator` 가 정확히 이 경우다) — 이미 존재를 아는 주체에게 존재를 숨기는 것은 무의미하고, 정당한 운영자가 '권한이 없다'는 사실을 인지하지 못하게 만든다.

### 7.9 목록이 전량·전문을 내려준다 — 페이징을 도입한다

**두 문제 — BE-026 §7.9 와 같은 뿌리다**:
1. **페이징이 없다.** `fetchAll(signal)` 이 파라미터 없이 전량을 반환하고 프론트가 검색을 클라이언트에서 한다. 템플릿은 운영 기간에 비례해 늘어난다(티켓만큼 빠르지는 않다).
2. **목록이 `body` 전문을 담는다.** `ReplyTemplate` 하나를 목록·상세가 공유해 목록 응답에 1000자 본문이 전부 실린다 — 목록은 **60자 미리보기만 쓴다**(`bodyPreview` — FS-028-EL-006.6).

**판정**: BE-010 이 `FaqSummary`/`Faq` 를 나눈 것과 같이 **`ReplyTemplateSummary`(`body` 대신 `bodyPreview`) / `ReplyTemplate`(전문)** 을 분리하고, EP-01 에 `keyword`·`categoryId`·`page`·`size`(기본 20 · 상한 100) 쿼리를 도입한다.

**단, 우선순위는 BE-026 보다 낮다**: ① 템플릿 증가 속도가 티켓보다 훨씬 느리다(수십~수백 건 규모) ② **1:1 문의 상세가 이 목록의 전량·전문을 필요로 한다**(FS-026-EL-025 가 `templatesForCategory` 로 클라이언트 필터 후 본문을 삽입한다) — 페이징된 목록으로는 그 화면이 동작하지 않는다. **두 소비자의 요구가 다르므로** 페이징 도입 시 티켓 상세용 표현(`?categoryId=<id>&include=body`, 페이징 없음)을 별도로 열거나 EP-01 을 두 표현으로 가른다.

**이관**: FS-028 §7 #2(IA-04 P0) · #24(유형 필터 부재)와 한 배치로. `?categoryId=` 쿼리는 두 요구(목록 필터 + 티켓 상세 선택지)를 함께 만족시킨다.

### 7.10 템플릿 제목 중복 — 유니크로 막되 409 UX 를 고쳐야 한다

**프론트에 중복 검사가 없다**(`replyTemplateSchema` 는 `title` 의 길이만 본다). 같은 이름 템플릿 둘이 만들어지면 **1:1 문의 상세의 삽입 드롭다운에서 구분할 수 없다** — 옵션 라벨이 `[<유형라벨>] <제목>` 이라(FS-026-EL-025) 유형까지 같으면 완전히 동일한 두 옵션이 보인다. BE-010 §7.5(카테고리명 중복)와 같은 결이다.

**판정**: 앞뒤 공백 제거 + 대소문자 무시로 정규화한 `title` 의 중복을 **409 `TEMPLATE_TITLE_DUPLICATED`** 로 막는다.

**그러나 프론트 UX 가 부적절하다 — 이관 필수**: `useCrudForm` 의 `handleWriteError` 는 `isConflict(cause)`(409/412)를 **무조건 '다른 사용자가 먼저 변경했습니다' 충돌 다이얼로그**로 띄운다(`useCrudForm.ts:158-172`). 제목 중복은 **동시성 충돌이 아니라 필드 검증 실패**인데, 사용자에게 '최신 내용 불러오기 / 계속 편집' 을 묻게 된다 — 어느 쪽도 답이 아니다(고칠 것은 제목이다).

**해소 안 (택1)**:
- **(A · 권장)** 제목 중복을 **422 `VALIDATION_FAILED`** + `error.fields: [{ name: 'title', ... }]` 로 반환한다. 그러면 `useCrudForm` 의 422 경로가 **제목 입력에 인라인 에러 + 포커스**를 준다(`:176-186`) — 화면 코드 변경 0줄. 다만 '중복'을 422 로 표현하는 것은 HTTP 관례에서 벗어난다(409 가 정석).
- **(B)** 409 를 유지하고 프론트가 `error.code` 로 갈라 `TEMPLATE_TITLE_DUPLICATED` 는 필드 에러로, `CONFLICT` 는 다이얼로그로 보낸다 — `useCrudForm` 수정 필요(공용 훅이라 영향 범위가 넓다).

**A 를 권장한다** — 관례를 조금 굽히는 대가로 공용 훅을 건드리지 않고 올바른 UX 를 얻는다. BE-010 §7.6 이 '카테고리 생성 실패가 코드별로 갈리지 않는다'를 A11 로 이관한 것과 같은 사안이며, **이 문서는 계약 단계에서 그것을 피한다.**
