# BE-005. 관리자(운영진) 관리 목록 백엔드 기능 명세

## 1. 개요

| 항목 | 내용 |
|---|---|
| 대상 화면 | FS-005 관리자(운영진) 관리 목록 (`/users/admins`) |
| 범위 | 운영자 목록 조회(그룹 필터 · 키워드 검색 · 페이징 · 전체/그룹별 건수), 운영진 그룹 목록 조회, 운영자 상세 조회(회원 상세 화면 재사용), **운영진 그룹 생성 · 삭제 · 삭제 전 참조 현황 조회**, **사전등록 발신번호 목록 조회** |
| 범위 밖 | **운영자 삭제 · 알림 발송 · 운영자(사람)에 대한 쓰기 전반** — 목록에 행 액션이 없다. **운영자 계정 생성** — 이 계약에 존재하지 않는다. **발신번호 사전등록·해지** — 통신사·문자 대행사 등록 대장이 정본이며 설정 도메인의 관심사다. 이 계약은 등록이 끝난 목록을 읽기만 한다 |
| 전제 | 인증은 세션 쿠키 기반. 모든 경로는 `/api` 프리픽스. 응답 본문은 `application/json; charset=utf-8`. 시각은 ISO 8601(오프셋 포함) |
| 프론트 어댑터 | `apps/admin/src/pages/admins/data-source.ts` |
| 도메인 타입 | `apps/admin/src/pages/admins/types.ts` |
| 상세 명세 재사용 | 운영자 상세(BE-005-EP-03)는 **회원 상세 계약(BE-004)** 을 그대로 따른다 — `fetchAdminDetail` 이 `MemberDetail` 을 돌려준다. §2 에러 봉투 · §3 권한 모델은 BE-003 을 상속한다 |

> **에러 봉투·권한 모델은 회원 도메인(BE-003 §2·§3)을 상속한다.** 운영자 도메인이 다른 형식을 만들면 백엔드가 두 벌을 구현하게 된다. 아래 §2·§3 은 BE-003 과 동일한 봉투·역할·은닉 원칙을 이 도메인의 리소스에 적용한 것이다.

## 2. 공통 에러 봉투

모든 4xx / 5xx 응답은 아래 봉투를 따른다(BE-003 §2 와 동일). 각 엔드포인트는 **에러코드 목록만** 나열한다.

```json
{
  "error": {
    "code": "ADMIN_NOT_FOUND",
    "message": "회원을 찾을 수 없습니다.",
    "fields": null,
    "traceId": "01J8X4K2M9P7Q3R5S6T8V0W2Y4"
  }
}
```

| 필드 | 타입 | 필수 | 설명 |
|---|---|---|---|
| `error.code` | string | O | 기계 판정용 에러코드(SCREAMING_SNAKE_CASE) |
| `error.message` | string | O | 사용자에게 그대로 보여줄 수 있는 문구. **내부 정보 노출 금지** |
| `error.fields` | array \| null | X | 400 검증 실패 시에만. `name` 은 쿼리 파라미터명 |
| `error.traceId` | string | O | 로그 상관관계 ID |

### 2.1 공통 에러코드

| HTTP | 에러코드 | 발생 조건 | 프론트 처리 |
|---|---|---|---|
| 400 | `VALIDATION_FAILED` | 쿼리 파라미터 제약 위반 | `error.fields` 를 매핑. 프론트에 매핑 UI 가 없어 조회 실패로 표시된다 |
| 401 | `UNAUTHENTICATED` | 세션 없음·만료 | `/login?returnUrl=<현재경로>` 로 이동 (현재 프론트 미구현 — §7) |
| 403 | `FORBIDDEN` | 인증됐으나 운영진 도메인 조회 권한 부족 | FS-005-EL-010 조회 실패 배너 |
| 404 | `ADMIN_NOT_FOUND` | 운영자 id 없음·삭제됨, 또는 운영진 도메인 읽기 권한 없음(§3.2 은닉) | 어댑터가 `Error('회원을 찾을 수 없습니다')` 로 변환 → FS-004-EL-012 |
| 409 | `ADMIN_GROUP_NAME_DUPLICATED` | 같은 이름의 운영진 그룹이 이미 있다 (BE-005-EP-04) | 모달의 그룹명 칸 아래에 `error.message` 를 그대로 표시 |
| 422 | `ADMIN_GROUP_IN_USE` | 그룹에 운영자가 남아 있거나, 그 그룹을 발신 프로필로 쓰는 메시지 템플릿이 있다 (BE-005-EP-06) | 확인 다이얼로그 안 danger 배너에 `error.message` 를 그대로 표시 |
| 429 | `RATE_LIMITED` | 레이트리밋 초과. `Retry-After`(초) 헤더 동반 | 재시도 안내 |
| 500 | `INTERNAL_ERROR` | 서버 내부 오류 | 일반 문구 + `traceId`. **원인 노출 금지** |
| 504 | `REQUEST_TIMEOUT` | 서버 처리 상한 초과 | 500 과 동일한 일반 문구 + `traceId` |

## 3. 인증 · 권한 모델

### 3.1 역할

| 역할 | 설명 | 비고 |
|---|---|---|
| `admin` | 운영진 도메인 전체 권한 — 운영자 목록 · 그룹 목록 · 운영자 상세 조회 | 이 문서의 모든 엔드포인트를 호출할 수 있다 |
| 운영진 도메인 조회 권한이 없는 인증 관리자 | 운영자 명부에 접근할 수 없는 그 밖의 역할 | 컬렉션은 **403**, 개별 운영자는 **404** 은닉 (§3.2) |

> **운영자 명부 열람 자체가 민감하다.** 운영진 관리 화면은 '누가 관리 권한을 가졌는가' 를 드러낸다. 회원 목록보다 좁은 권한으로 보호해야 하며, 열람 권한 정책(어떤 역할이 운영자 목록을 볼 수 있는가)은 아키텍처 판정 사안으로 §7 에 남긴다. **프론트에는 역할 분기가 없다**(FS-005 §4.1) — 권한 강제는 전적으로 서버의 책임이다.

### 3.2 403 vs 404 은닉 정책

BE-003 §3.2 의 원칙을 이 도메인에 적용한다.

| 리소스 | 권한 부족 시 응답 | 근거 |
|---|---|---|
| `GET /api/admins` (컬렉션) | 403 `FORBIDDEN` | 운영자 컬렉션의 존재는 비밀이 아니다. 열거할 개별 id 가 노출되지 않는다 |
| `GET /api/admin-groups` (컬렉션) | 403 `FORBIDDEN` | 그룹 컬렉션의 존재는 비밀이 아니다 |
| `GET /api/admins/:id` (개별) | 운영진 도메인 읽기 권한 없음 → 404 `ADMIN_NOT_FOUND` | 운영자 id 열거를 막는다. 존재 여부 자체가 정보다. 어댑터가 이 코드를 '회원을 찾을 수 없습니다' 로 변환해 FS-004-EL-012 를 띄운다 |

### 3.3 CSRF · 3.4 타임아웃 상한

- **CSRF**: 그룹 생성·삭제가 상태 변경 요청이므로 `POST /api/admin-groups` · `DELETE /api/admin-groups/:id` 는 **`X-CSRF-Token` 헤더를 요구한다**(BE-003 §3.3 과 동일한 규약). 조회 계열은 요구 대상이 아니다.
- **타임아웃**: 조회·쓰기 모두 서버 처리 상한 5초 → 초과 시 504 `REQUEST_TIMEOUT`. 프론트 권고 상한 10초(현재 프론트에 상한 없음 — §7).

## 4. 엔드포인트 명세

### BE-005-EP-01 · 운영자 목록 조회

| 항목 | 내용 |
|---|---|
| 근거 (FS) | FS-005-EL-001.1, FS-005-EL-001.2, FS-005-EL-004, FS-005-EL-005, FS-005-EL-006, FS-005-EL-009, FS-005-EL-010, FS-005-EL-010.1 |
| 메서드 · 경로 | `GET /api/admins` |
| 권한 | `admin` (운영진 도메인 조회 권한) |
| 멱등성 | 멱등 (GET). 재시도 안전. FS-005-EL-010.1('다시 시도')은 같은 조건으로 같은 요청을 다시 보낸다 |
| 페이징 | offset 방식. `page` 1-base, `size` 기본 10 · 상한 100 (프론트 `PAGE_SIZE = 10`) |
| 레이트리밋 | 관리자당 분당 120회 |

**요청 — 쿼리**

| 이름 | 타입 | 필수 | 제약 | 설명 |
|---|---|---|---|---|
| `groupId` | string | X | 그룹 id 또는 `all` (기본 `all`). 1–64자 | 그룹 필터. `all` 은 그룹 조건 없음 (프론트 상수 `GROUP_ALL = 'all'`) |
| `keyword` | string | X | 0–50자. 서버가 앞뒤 공백을 제거한다. 빈 문자열이면 조건 없음 | 닉네임 **또는** 계정에 대한 대소문자 무시 부분 일치 |
| `page` | number | X | 정수 ≥ 1 (기본 1) | 페이지 번호 |
| `size` | number | X | 정수 1–100 (기본 10) | 페이지 크기 |

- `page` 가 총 페이지 수를 넘으면 `admins` 는 빈 배열, `total` 은 조건에 걸린 실제 건수를 반환한다. 서버는 페이지를 자동 보정하지 않는다(FS-005 §7 #8 — 프론트가 빈 상태 FS-005-EL-008 을 표시한다).
- `keyword` 가 50자를 넘으면 400. 프론트에 길이 제약이 없으므로(FS-005-EL-004) 어댑터가 그대로 보내며, 400 은 조회 실패로 표시된다 — §7 에 변경 요청 로 기록한다.

**응답 200** — `AdminListResult`

```json
{
  "admins": [
    {
      "id": "A-00003",
      "nickname": "박백민",
      "account": "whitepark@example.com",
      "groupId": "admin-2",
      "group": "운영팀 admin_2",
      "joinedAt": "2024-06-26",
      "department": "운영본부",
      "position": "이사",
      "phone": "010-7173-7804",
      "memo": ""
    }
  ],
  "totalAll": 5,
  "groupCounts": { "admin": 3, "admin-2": 2 },
  "total": 2
}
```

| 필드 | 타입 (types.ts) | 설명 |
|---|---|---|
| `admins` | `readonly AdminUser[]` | 현재 페이지의 운영자. 필드명·타입이 `AdminUser` 와 일치한다 |
| `totalAll` | number | **검색·그룹 필터와 무관한 전체 운영자 수** (FS-005-EL-001.1 배지 — 좌측 배지가 검색할 때마다 흔들리면 안 된다) |
| `groupCounts` | `AdminGroupCounts` = `Record<string, number>` (키 = 그룹 id) | 그룹별 **전체** 운영자 수. `GET /api/admin-groups` 의 모든 그룹 id 를 키로 포함한다 |
| `total` | number | 필터·검색을 적용한 뒤의 전체 건수. 페이지네이션(FS-005-EL-009)과 조회 요약(FS-005-EL-005)이 쓴다 |

- `AdminUser.department` · `AdminUser.position` 은 **빈 문자열일 수 있다**(FS-005-EL-006.7 · 006.8 — 값을 지어내지 않는다). 서버는 값이 없으면 `""` 를 보낸다.
- `AdminUser.joinedAt` 은 `YYYY-MM-DD` 다. 목록은 상대 시각을 쓰지 않으므로(FS-005-EL-006.6) ISO date-time 필드가 없다 — 회원 목록(`joinedAtIso`)과 다른 지점이다.

**에러 목록**

| HTTP | 에러코드 | 조건 |
|---|---|---|
| 400 | `VALIDATION_FAILED` | `groupId` 형식·`keyword`(50자)·`page`(<1)·`size`(범위 밖) 위반 |
| 401 | `UNAUTHENTICATED` | 세션 없음·만료 |
| 403 | `FORBIDDEN` | 운영진 도메인 조회 권한 부족 (컬렉션은 은닉하지 않는다 — §3.2) |
| 429 | `RATE_LIMITED` | 분당 120회 초과 |
| 500 | `INTERNAL_ERROR` | 내부 오류 |
| 504 | `REQUEST_TIMEOUT` | 서버 처리 5초 초과 |

### BE-005-EP-02 · 운영진 그룹 목록 조회

| 항목 | 내용 |
|---|---|
| 근거 (FS) | FS-005-EL-001 |
| 메서드 · 경로 | `GET /api/admin-groups` |
| 권한 | `admin` (운영진 도메인 조회 권한) |
| 멱등성 | 멱등 (GET). 재시도 안전 |
| 페이징 | 없음 — 전량 반환. **그룹 수 상한 200개** (BE-003 §7.5 와 동일 근거). 상한 강제는 생성 엔드포인트(BE-005-EP-04)의 책임이다 |
| 레이트리밋 | 관리자당 분당 60회 |

**요청** — 파라미터 없음.

**응답 200** — `readonly AdminGroup[]`

```json
[
  { "id": "admin", "name": "운영팀 admin", "phoneNumbers": [], "emails": [], "usableAsSender": false },
  {
    "id": "sp-brand",
    "name": "스페이스플래닝 대표",
    "phoneNumbers": ["1588-1234", "02-577-1000"],
    "emails": ["news@spaceplanning.ai", "hello@spaceplanning.ai"],
    "usableAsSender": true
  }
]
```

| 필드 | 타입 (`shared/domain/admin-group.ts`) | 설명 |
|---|---|---|
| `[].id` | string | 그룹 id. `fetchAdmins` 의 `groupId` 쿼리와 `groupCounts` 의 키로 그대로 쓰인다 |
| `[].name` | string | 그룹 표시명. 좌측 필터는 `'운영 - <name>'` 로, 표와 메시지 템플릿의 발신 프로필 셀렉트는 `name` 을 그대로 쓴다 |
| `[].phoneNumbers` | `readonly string[]` | 이 그룹으로 발송할 수 있는 발신번호들. **사전등록이 끝난 번호만** 들어온다. 0개일 수 있다 |
| `[].emails` | `readonly string[]` | 이 그룹으로 발송할 수 있는 발신 주소들. 0개일 수 있다 |
| `[].usableAsSender` | boolean | 메시지 템플릿의 발신 프로필로 고를 수 있는가. `false` 면 그 셀렉트에 나타나지 않는다 |

- **v1.1 변경 — `label` → `name`, 그리고 필드 3개 추가.** 운영진 그룹과 메시지 템플릿의 '발신 프로필' 이 한 실체로 통합됐다(FS-005 §1.1). 통합 전에는 같은 값을 운영진 도메인이 `label`, 마케팅 도메인이 `name` 으로 불렀다 — 합친 뒤에도 둘을 남기면 어느 쪽이 정본인지 화면마다 갈린다.
- **1:N 을 서버가 지킨다.** 한 그룹은 발신번호·발신 주소를 여러 개 갖는다. 편집기의 둘째 셀렉트가 '고른 프로필이 쓸 수 있는 번호/주소' 를 나열하므로(FS 참조), 배열을 단일 값으로 접으면 안 된다. 생성 폼이 대표값 1개씩만 받는 것은 **입력의 편의**일 뿐 모델의 제약이 아니다.
- **`usableAsSender` 로 목록과 자격을 가른다.** 조회·권한 필터 전용 그룹까지 발신 프로필 셀렉트에 뜨면, 고르면 발신번호 후보가 0개라 저장도 못 하는 항목이 드롭다운을 채운다. **목록은 하나, 자격은 플래그다.**
- **이 엔드포인트는 운영진(staff) 그룹만 반환한다.** 회원 그룹(`type=member`)은 회원 목록(BE-003-EP-02)의 관심사이며 여기 섞이지 않는다 — BE-003 §7.4 판정은 그대로 유효하다.

**에러 목록**

| HTTP | 에러코드 | 조건 |
|---|---|---|
| 401 | `UNAUTHENTICATED` | 세션 없음·만료 |
| 403 | `FORBIDDEN` | 운영진 도메인 조회 권한 부족 (컬렉션은 은닉하지 않는다 — §3.2) |
| 429 | `RATE_LIMITED` | 분당 60회 초과 |
| 500 | `INTERNAL_ERROR` | 내부 오류 |
| 504 | `REQUEST_TIMEOUT` | 서버 처리 5초 초과 |

### BE-005-EP-03 · 운영자 상세 조회

| 항목 | 내용 |
|---|---|
| 근거 (FS) | FS-005-EL-006.3, FS-005-EL-006.10, FS-005-EL-006.11 (→ FS-004 회원 상세 재사용) |
| 메서드 · 경로 | `GET /api/admins/:id` |

### BE-005-EP-04 · 운영진 그룹 생성

| 항목 | 내용 |
|---|---|
| 근거 (FS) | FS-005-EL-001.3, FS-005-EL-011, FS-005-EL-011.1 ~ FS-005-EL-011.5 |
| 메서드 · 경로 | `POST /api/admin-groups` |
| 권한 | `admin` (운영진 도메인 쓰기 권한) |
| 멱등성 | **비멱등**. 다만 이름 유일 제약(409)이 사실상 중복 생성을 막는다 — 응답이 유실돼 재시도하면 두 번째는 409 다. 멱등키를 두지 않는 근거가 이것이다(적립금 지급과 달리 중복이 조용히 성립하지 않는다) |

| 레이트리밋 | 관리자당 분당 10회 |

**요청 — 본문** `application/json`

```json

```

| 이름 | 타입 | 필수 | 제약 | 설명 |
|---|---|---|---|---|

- **서버가 사전등록 여부를 다시 검증한다.** 화면은 셀렉트로만 고르게 하지만(FS-005-EL-011.3), 요청 본문은 화면을 거치지 않고도 만들 수 있다. 미등록 번호를 통과시키면 저장은 되고 발송만 실패하는 그룹이 생긴다 → 400 `VALIDATION_FAILED`(`error.fields[].name = "phoneNumbers"`).

**응답 201** — 만들어진 `AdminGroup` (BE-005-EP-02 와 같은 스키마). 프론트는 이 응답으로 목록·건수 쿼리를 무효화한다.

### BE-005-EP-06 · 운영진 그룹 삭제

| 항목 | 내용 |
|---|---|
| 근거 (FS) | FS-005-EL-012, FS-005-EL-013 |
| 메서드 · 경로 | `DELETE /api/admin-groups/:id` |
| 권한 | `admin` (운영진 도메인 쓰기 권한) |
| 멱등성 | **멱등**. 이미 삭제된 그룹에 재요청하면 상태를 바꾸지 않고 204 를 반환한다(BE-003-EP-05 회원 삭제와 같은 규약) — 프론트의 재시도가 404 를 만나 실패로 보이지 않게 한다 |
| CSRF | `X-CSRF-Token` 필수 (§3.3) |
| 레이트리밋 | 관리자당 분당 10회 |

**요청 — 경로**: `id` (string, 필수, 1–64자)

**응답 204** — 본문 없음.

**삭제 가드 — 422 `ADMIN_GROUP_IN_USE`**

그룹을 지우는 것은 **그 그룹을 가리키는 다른 것들을 고아로 만드는** 행위다. 두 경우 모두 **막고**, `error.message` 에 사람이 읽을 수 있는 사유와 해결 방법을 담아 프론트가 그대로 표시하게 한다.

| # | 조건 | `error.message` 형태 | 판정 근거 |
|---|---|---|---|
| 1 | 그룹에 운영자가 1명 이상 남아 있다 | `'<그룹명>' 그룹에는 운영자 N명이 속해 있어 삭제할 수 없습니다. 이 운영자들을 다른 그룹으로 옮긴 뒤 다시 삭제해 주세요.` | 운영자를 '그룹 없음' 으로 옮기는 선택지를 두지 않았다. `AdminUser.groupId` 는 null 을 허용하지 않고, 무엇보다 **운영자에게 '그룹 없음' 은 뜻이 있는 상태가 아니다** — 그룹이 곧 권한 묶음이므로 그룹 없는 운영자는 아무것도 할 수 없는 계정이다. 쓸 곳이 없는 상태를 표현하려고 모델을 넓히지 않는다 |
| 2 | 그 그룹을 발신 프로필로 쓰는 메시지 템플릿이 1건 이상 있다 | `'<그룹명>' 그룹을 발신 프로필로 쓰는 메시지 템플릿이 있어 삭제할 수 없습니다 ('A', 'B'). 해당 템플릿의 발신 프로필을 먼저 바꾼 뒤 다시 삭제해 주세요.` | 지우면 그 템플릿의 `senderProfileId` 가 존재하지 않는 id 를 가리킨다 — 목록·상세는 '—' 로 넘어가지만 발송 시점에는 발신자가 없는 템플릿이다. **걸린 템플릿 이름을 전부 나열한다**: 어느 것을 고쳐야 하는지가 그 목록이기 때문이며, 잘라내면 운영자가 남은 것을 찾아 헤맨다 |

- 두 조건이 함께 성립하면 **①을 먼저** 알린다. 운영자를 옮기는 편이 템플릿을 고치는 것보다 먼저 해결돼야 하는 일이고, 두 사유를 한 문장에 겹쳐 쓰면 무엇부터 해야 하는지가 흐려진다.
- 그룹을 지워도 **사전등록 발신번호 자체는 사라지지 않는다.** 번호의 정본은 통신사 등록 대장이며(§1 범위 밖), 그룹은 그중 어떤 것을 쓸지에 대한 참조를 들 뿐이다.

**에러 목록**

| HTTP | 에러코드 | 조건 |
|---|---|---|
| 400 | `VALIDATION_FAILED` | 경로 `id` 형식 위반 |
| 401 | `UNAUTHENTICATED` | 세션 없음·만료 |
| 403 | `FORBIDDEN` | 운영진 도메인 쓰기 권한 부족 |
| 404 | `ADMIN_GROUP_NOT_FOUND` | 존재한 적 없는 id (이미 삭제된 그룹은 204 — 멱등) |
| 422 | `ADMIN_GROUP_IN_USE` | 위 삭제 가드 ① 또는 ② |
| 429 | `RATE_LIMITED` | 분당 10회 초과 |

- **등록·해지 흐름은 이 계약에 없다**(§1 범위 밖). 이 엔드포인트가 있는 이유는 하나다: 그룹 생성 폼이 자유 입력 대신 **고르게** 하기 위해서다.

## 7. 핵심 판정 (근거를 남긴다)

### 7.1 쓰기 엔드포인트는 **그룹에 대해서만** 있다 (v1.1 개정)

v1.0 은 '이 도메인에 생성·수정·삭제 엔드포인트가 하나도 없다' 를 계약으로 못박았다. 근거는 FS-005 §1.1 이 그룹 생성 진입점을 두지 않았다는 것이었다. **그 전제가 바뀌었다** — 운영진 그룹과 메시지 템플릿의 '발신 프로필' 이 한 실체로 통합되면서(FS-005 §1.1 v1.1), 그 실체를 만들 화면이 필요해졌다. 통합된 실체를 만들 수단이 어디에도 없으면 운영자는 발신 프로필을 만들 방법 자체를 갖지 못한다.

바뀐 것과 바뀌지 않은 것을 분명히 한다:

- **그룹에 대한 쓰기가 생겼다** — BE-005-EP-04(생성) · BE-005-EP-06(삭제). 수정(PATCH)은 아직 없다: 화면에 편집 진입점이 없다. **없는 프론트 호출에 대응하는 엔드포인트를 발명하지 않는다**는 원칙은 그대로다(발신번호를 나중에 덧붙이는 흐름이 생기면 그때 승격한다 — §7.4).
- **운영자(사람)에 대한 쓰기는 여전히 없다.** 목록에 행 액션·일괄 액션이 없다. 운영자 계정 생성·삭제·권한 변경은 별도 화면(권한 관리 FS-006 및 미작성 운영자 관리 흐름)의 관심사다.

### 7.5 그룹 삭제 — 막는다. 재배치도 소프트 삭제도 두지 않는다

삭제 가드(BE-005-EP-06)는 두 경우를 **모두 막는 쪽**으로 판정했다. 대안을 검토한 기록을 남긴다.

- **'그룹 없음' 으로 옮기고 지운다** — 채택하지 않았다. `AdminUser.groupId` 를 nullable 로 넓혀야 하고, 그 변경은 표의 그룹 셀·좌측 배지 집계·필터 조회 조건까지 번진다. 무엇보다 **운영자에게 '그룹 없음' 은 뜻이 있는 상태가 아니다** — 그룹이 곧 권한 묶음이므로 그룹 없는 운영자는 아무것도 할 수 없는 계정이다. 쓸 곳이 없는 상태를 표현하려고 모델을 넓히지 않는다.
- **다른 그룹으로 일괄 재배치한 뒤 지운다** — 채택하지 않았다. '어느 그룹으로 옮길지' 는 권한을 옮기는 결정이고, 삭제 다이얼로그 안에서 곁다리로 내릴 결정이 아니다. 필요해지면 별도 흐름으로 승격한다(§7.4).
- **소프트 삭제(회원과 같은 방식)** — 채택하지 않았다. 회원 삭제를 소프트로 둔 이유는 주문·이력의 참조 무결성이었다(§7.1 회원). 그룹은 참조가 남아 있으면 **애초에 지우지 못하게** 하므로 고아 참조가 생기지 않는다 — 지워진 그룹을 남겨 둘 이유가 없다.
- **발신 프로필로 쓰이는 그룹**은 지우면 템플릿이 존재하지 않는 발신자를 가리킨다. 목록·상세는 '—' 로 넘어가지만 발송 시점에는 발신자가 없는 템플릿이다. `senderProfileId` 를 null 로 밀어 넣는 것(= 조용한 파손)보다 **막고 이름을 대는 편**이 복구가 빠르다.

### 7.2 운영자 상세는 회원 상세 계약(BE-004)을 상속한다

`/users/admins/:id` 가 `MemberDetailPage` 를 재사용하므로, 상세의 요청/응답/에러/권한은 BE-004 를 그대로 쓴다. 다만 **운영자↔회원 데이터 매핑이 부적합**하다(FS-005 §7 #3): 운영자에게는 회원 등급·동의정보·쿠폰·적립금이 없는데 현재 어댑터가 빈 값으로 채운다. 그리고 상세의 ⋯ 메뉴(삭제·알림 발송)가 **회원 도메인 어댑터**를 호출한다(FS-005 §7 #4) — 운영자 삭제·알림이 회원 엔드포인트로 나간다. 두 문제는 운영자 전용 상세 계약이 정해지기 전까지 이 문서가 해결하지 않고 아키텍처 판정으로 남긴다.

### 7.3 운영자 명부 열람 권한 — 아키텍처 판정 사안

운영자 명부는 '누가 관리 권한을 가졌는가' 를 드러내므로 회원 목록보다 좁은 권한으로 보호해야 한다. 어떤 역할이 `GET /api/admins` 를 볼 수 있는지(예: `admin` 만, 또는 권한 관리 read 를 가진 역할)는 권한 매트릭스(FS-006/BE-006)와 함께 확정해야 한다 — §7.4 로 이관한다.
