# FS-005. 관리자(운영진) 관리 목록

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 목적 | 사이트를 함께 관리하는 운영진(관리자 계정)을 그룹·키워드로 좁혀 찾고, 개별 운영자의 상세로 이동한다 |
| 역할(주 사용자) | 관리자 (구현에 역할 분기 없음 — §4.1 참조) |
| 진입 경로 | 좌측 GNB > 사용자 관리 > 관리자 관리 (`nav-config.ts`, `to: '/users/admins'`) |
| 관련 화면 | FS-004 회원 상세 (**운영자 상세가 이 구현을 재사용한다** — §1.1) / FS-007 고객 설정(사이드바 안내문 링크) / 로그인 이력(FS-008) |
| 대응 SCR | SCR-005 (미작성 — §7) |
| 구현 경로 | `apps/admin/src/pages/admins/AdminsPage.tsx` 및 `apps/admin/src/pages/admins/components/**` |

### 1.1 이 화면의 설계 원칙 (요구사항 — 명세에 고정한다)

| 원칙 | 내용과 이유 |
|---|---|
| **회원 관리 목록과 구조를 공유한다** | 좌측 그룹 패널 + 우측 검색·표·페이지네이션 배치는 FS-003 회원 관리 목록과 같은 한 벌을 쓴다(`AdminsPage.tsx` 머리말). 필터 축·행 이동 규칙·스켈레톤·조회 실패 배너 규칙이 동일하다 |
| **운영진 그룹을 이 화면에서 만들고 지운다** | 좌측 패널 하단에 '+ 새 그룹 만들기' 와 '<그룹명> 그룹 삭제' 가 있다(FS-005-EL-001.3 · FS-005-EL-001.4). **왜 바뀌었나 — 관리자 그룹과 발신 프로필의 통합**: 운영진 그룹(`pages/admins`)과 메시지 템플릿의 '발신 프로필'(`marketing/message-templates`)은 실무에서 같은 것을 가리키는 두 이름이었고, 구현에서도 서로를 모르는 **두 벌의 하드코딩 목록**으로 존재했다. 둘을 한 실체로 합쳐 `shared/domain/admin-group.ts` + `shared/fixtures/admin-groups.ts` 를 정본으로 두었고, 그 실체를 만드는 화면이 필요해졌다. 합친 항목은 발신번호·발신 이메일을 **여러 개**(1:N) 가지며, `usableAsSender` 플래그가 켜진 그룹만 템플릿 편집기의 발신 프로필 목록에 나타난다 |
| **상세는 회원 상세(MemberDetailPage)를 재사용한다** | `/users/admins/:id` 는 `MemberDetailPage` 에 `fetchDetail=fetchAdminDetail` · `listPath="/users/admins"` 를 주입해 그대로 렌더한다(`App.tsx`). 따라서 운영자 상세의 요소·예외는 **FS-004 를 그대로 참조**하며 이 문서에서 중복 명세하지 않는다. 운영자↔회원의 데이터 차이는 §7 미결로 남긴다 |
| **운영자 행에 대한 액션은 없다** | 회원 관리의 행 ⋯ 메뉴(삭제·알림 발송)·일괄 액션·내보내기가 이 목록에는 **없다**. 운영자(사람)에 대한 쓰기는 이 화면에 없다 — `data-source.ts` 의 쓰기 함수는 **그룹**에 대한 것(`createAdminGroup` · `deleteAdminGroup`)뿐이다 |

> **v1.1 개정 근거 — 위 두 원칙은 v1.0 의 확정(`gate: G9` · `confirmedAt: 2026-07-15`)을 대체한다.**
> v1.0 은 '운영진 그룹을 만드는 기능이 없다' 와 '행 액션·쓰기가 없다(목록은 조회 전용)' 를 확정 사항으로 고정했다. 관리자 그룹과 발신 프로필을 한 실체로 통합하기로 하면서 그 전제가 사라졌다 — 통합된 실체를 만드는 화면이 어디에도 없으면, 운영자는 발신 프로필을 만들 수단 자체를 갖지 못한다. **되살리려면 통합 판정을 먼저 뒤집어야 한다.** 조회 전용이라는 성질은 '운영자(사람)에 대해서' 만 유지된다.

| 중복 제출 | 그룹 만들기·지우기의 확인 버튼은 요청 중 비활성(`aria-busy`)되어 요청을 1건만 만든다. 삭제 버튼(FS-005-EL-001.4)은 참조 현황 조회 중 비활성이다. 조회는 조건이 바뀌면 진행 중이던 요청을 abort 하고 최신 조건만 반영한다 |
| 요청 취소 | 그룹 만들기·지우기는 모달·다이얼로그를 닫으면 진행 중이던 요청을 abort 한다. **취소된 요청은 성공도 실패도 알리지 않는다** — 화면을 떠난 요청의 결과가 뒤늦게 토스트로 튀어나오지 않게 한다 |
| 쓰기 후 목록 정합 | 그룹을 만들거나 지우면 **그룹 목록과 운영자 목록 조회를 함께** 무효화한다. 그룹별 건수(`groupCounts`)는 운영자 목록 응답이 싣기 때문에, 그룹 목록만 갱신하면 새 항목의 배지가 조회되지 않은 값으로 남는다 |
| 권한 없음 | **이 화면의 구현에는 역할 기반 분기가 없다.** 요소가 역할에 따라 숨거나 비활성되지 않으며, 전용 403 화면도 없다. 서버가 권한 부족을 응답하면 조회는 FS-005-EL-010 으로 떨어진다. 은닉 정책(403 vs 404)은 백엔드 명세(BE-005)가 확정한다 |
| 행 선택의 수명 | 선택(FS-005-EL-006.1 · FS-005-EL-006.2)은 화면 로컬 상태이며 서버로 전송되지 않는다. 그룹·키워드·페이지 중 하나라도 바뀌면 전부 해제된다. **선택을 소비하는 액션이 없다** — §7 미결 사항 |
| 동시 조회 | 목록 조회는 동시에 1건만 유지된다. 조건이 바뀌면 진행 중이던 요청을 abort 하므로 늦게 도착한 이전 응답이 최신 결과를 덮어쓰지 않는다 |
