# BE-008. 로그인 이력 (감사 로그) 백엔드 기능 명세

## 1. 개요

| 항목 | 내용 |
|---|---|
| 대상 화면 | FS-008 로그인 이력 (`/users/login-history`) |
| 범위 | 로그인 시도 이력 조회(결과·계정 유형·기간·키워드 필터 · 페이징 · 기간 안 건수), 이력 내보내기 조회. **그리고 이 도메인의 핵심 — 감사 불변성과 열거 오라클 방지의 서버 강제**(§3·§7) |
| 범위 밖 | **이력의 생성** — 로그인 시도는 로그인 도메인(BE-001)이 인증 처리 중에 기록한다. 이 화면은 그 기록을 **읽기만** 한다. **이력의 삭제·수정** — 감사 불변성 원칙상 그런 엔드포인트를 만들지 않는다(§7.1). **CSV 직렬화** — 프론트 `toCsv` 가 브라우저에서 한다(§7.4) |
| 전제 | 인증은 세션 쿠키 기반. 모든 경로는 `/api` 프리픽스. **읽기 전용이므로 CSRF 대상 쓰기가 없다.** 응답 본문은 `application/json; charset=utf-8`. 시각은 ISO 8601(오프셋 포함) |
| 프론트 어댑터 | `apps/admin/src/pages/login-history/data-source.ts` |
| 도메인 타입 | `apps/admin/src/pages/login-history/types.ts` (`LoginHistoryEntry` · `LoginHistoryResult` · `LoginFailureReason` 등) |
| 실패 사유 정의 | `LoginFailureReason` 은 **BE-001 이 정의한 사유만** 쓴다(§7.2). 화면이 사유를 지어내지 않는다 |

> **감사 로그는 이 앱이 바꾸지 않는다 — 오직 읽는다.** 이 문서에 쓰기 엔드포인트가 없는 것은 누락이 아니라 **설계**다(§7.1). 그리고 이 도메인에는 로그인 도메인(BE-001)의 열거 오라클 방지 원칙이 이어진다(§3.2·§7.2).

## 2. 공통 에러 봉투

모든 4xx / 5xx 응답은 BE-003 §2 와 동일한 봉투를 따른다. 각 엔드포인트는 **에러코드 목록만** 나열한다.

```json
{
  "error": {
    "code": "INVALID_DATE_RANGE",
    "message": "조회 기간이 올바르지 않습니다.",
    "fields": [{ "name": "to", "code": "FUTURE_DATE", "message": "미래 날짜는 조회할 수 없습니다." }],
    "traceId": "01J8X4K2M9P7Q3R5S6T8V0W2Y4"
  }
}
```

| 필드 | 타입 | 필수 | 설명 |
|---|---|---|---|
| `error.code` | string | O | 기계 판정용 에러코드 |
| `error.message` | string | O | 사용자 노출 가능 문구. **내부 정보 노출 금지** |
| `error.fields` | array \| null | X | 400 검증 실패 시. `name` = `from` \| `to` \| `outcome` \| `accountKind` \| `page` \| `size` |
| `error.traceId` | string | O | 로그 상관관계 ID |

### 2.1 공통 에러코드

| HTTP | 에러코드 | 발생 조건 | 프론트 처리 |
|---|---|---|---|
| 400 | `VALIDATION_FAILED` | 쿼리 파라미터 형식 위반 | 조회 실패(프론트 선검증이 대부분 막는다) |
| 400 | `INVALID_DATE_RANGE` | 기간 형식·미래·역전·90일 초과 | 프론트가 선검증하므로 정상 흐름에서 미발생(§7.3) |
| 401 | `UNAUTHENTICATED` | 세션 없음·만료 | `/login?returnUrl=` (현재 프론트 미구현 — §7.5) |
| 403 | `FORBIDDEN` | 감사 로그 조회 권한 부족 (§3.2) | 조회는 FS-008-EL-012, 내보내기는 FS-008-EL-013 |
| 429 | `RATE_LIMITED` | 레이트리밋 초과. `Retry-After` 동반 | 재시도 안내 |
| 500 | `INTERNAL_ERROR` | 서버 내부 오류 | 일반 문구 + `traceId` |
| 504 | `REQUEST_TIMEOUT` | 서버 처리 상한 초과 | 500 과 동일 |

## 3. 인증 · 권한 모델 · 감사 원칙

### 3.1 역할

| 역할 | 설명 | 비고 |
|---|---|---|
| 로그인 이력 리소스 `read` 권한 보유 | 이력을 조회·내보낼 수 있다 | 로그인 이력은 회원·운영자의 IP·기기·실패 사유를 담아 **민감하다** |
| 로그인 이력 read 권한 없음 | `/users/login-history` 리소스 접근 불가 | `GET /api/login-history` → 403 (§3.2) |

> 감사 로그 열람은 개인정보(IP·기기·마스킹 전 이름)에 닿으므로 좁은 권한으로 보호한다. **프론트에는 역할 분기가 없다**(FS-008 §4.1) — 권한 강제는 전적으로 서버의 책임이다. 이력 자체는 이 도메인이 쓰지 않으므로 쓰기 권한 개념이 없다(§7.1).

### 3.2 403 vs 404 은닉 · ⚠ 열거 오라클 방지

| 리소스 | 권한 부족 시 응답 | 근거 |
|---|---|---|
| `GET /api/login-history` (컬렉션) | 403 `FORBIDDEN` | 이력 컬렉션의 존재는 비밀이 아니다. 열거할 개별 id 가 노출되지 않는다 |
| `GET /api/login-history/export` (컬렉션) | 403 `FORBIDDEN` | 위와 동일 |

**⚠ 열거 오라클 방지 (BE-001 §3.2·§4 를 이 도메인이 잇는다)**: 미등록 계정의 로그인 시도도 **동일하게 이력에 기록된다.** 로그인 도메인은 미등록 이메일을 오답 비밀번호와 완전히 같은 응답(같은 에러코드·문구·실패 카운트)으로 처리하고, 연속 실패 5회에 미등록 이메일도 동일하게 잠근다 — 그렇지 않으면 '5회 넘겨도 안 잠기는 이메일 = 미등록' 이라는 열거 오라클이 생긴다(BE-001 §4 잠금 규칙). 이 감사 이력은 그 시도를 `failureReason='unknown_account'` 로 기록해 **관리자에게는 진짜 사유를 보여주되**(FS-008 §1.1), 사용자에게는 로그인 화면이 계정 존재를 노출하지 않는다(FS-001-EL-019). 두 층(사용자 응답 vs 감사 기록)이 분리돼야 한다.

### 3.3 감사 불변성 (append-only)

- 이력은 **추가만 되고 수정·삭제되지 않는다.** 이 계약에 이력을 바꾸는 엔드포인트가 없고(§7.1), 저장소도 append-only 로 다뤄야 한다(권한 있는 운영자도 개별 이력을 지울 수 없다).
- 이력 열람 자체도 감사 대상이다 — 누가 언제 어떤 조건으로 이력을 조회·내보냈는지 별도 감사 로그에 남긴다(§7.6 이관).

### 3.4 타임아웃 상한

| 계열 | 서버 처리 상한 | 초과 시 | 프론트 상한(권고) |
|---|---|---|---|
| 조회 | 5초 | 504 `REQUEST_TIMEOUT` | 10초 (현재 상한 없음 — §7.5) |
| 내보내기 조회 | 30초 | 504 `REQUEST_TIMEOUT` | 60초 (현재 상한 없음 — §7.5) |

## 4. 엔드포인트 명세

### BE-008-EP-01 · 로그인 이력 조회

| 항목 | 내용 |
|---|---|
| 근거 (FS) | FS-008-EL-001, FS-008-EL-002, FS-008-EL-003, FS-008-EL-003.1, FS-008-EL-003.2, FS-008-EL-005, FS-008-EL-007, FS-008-EL-007.1, FS-008-EL-008, FS-008-EL-008.6, FS-008-EL-011, FS-008-EL-012, FS-008-EL-012.1 |
| 메서드 · 경로 | `GET /api/login-history` |
| 권한 | 로그인 이력 리소스 `read` |
| 멱등성 | 멱등 (GET). 재시도 안전 |
| 페이징 | offset 방식. `page` 1-base, `size` 기본 10 · 상한 100 (프론트 `PAGE_SIZE = 10`) |
| 레이트리밋 | 관리자당 분당 120회 |

**요청 — 쿼리**

| 이름 | 타입 | 필수 | 제약 | 설명 |
|---|---|---|---|---|
| `outcome` | string | X | `all` \| `success` \| `failure` (기본 `all`) | 결과 필터 |
| `accountKind` | string | X | `all` \| `member` \| `admin` (기본 `all`) | 계정 유형 필터 |
| `from` | string | O | `YYYY-MM-DD`. 실재 날짜 | 조회 구간 시작(양 끝 포함) |
| `to` | string | O | `YYYY-MM-DD`. 실재 날짜. `to >= from`, `to <= 오늘`, 구간 ≤ 90일 | 조회 구간 끝(양 끝 포함) |
| `keyword` | string | X | 0–100자. 앞뒤 공백 제거 | 계정·이름·IP 부분 일치(대소문자 무시) |
| `page` | number | X | 정수 ≥ 1 (기본 1) | 페이지 번호 |
| `size` | number | X | 정수 1–100 (기본 10) | 페이지 크기 |

- `outcome`·`accountKind`·`keyword`·기간은 **AND** 로 결합한다.
- 기간 검증(형식·미래·역전·90일)은 프론트가 선검증하므로(FS-008-EL-003.1~.4) 정상 흐름에서 400 이 나오지 않는다. 우회 호출에 대비해 서버도 같은 규칙을 강제한다(§7.3).

**응답 200** — `LoginHistoryResult`

```json
{
  "entries": [
    {
      "id": "lh-00001",
      "occurredAtIso": "2026-07-14T02:13:05+09:00",
      "account": "user1@naver.com",
      "name": "명재***",
      "accountKind": "member",
      "outcome": "failure",
      "failureReason": "invalid_password",
      "consecutiveFailures": 3,
      "ip": "203.0.113.7",
      "browser": "Chrome 126",
      "os": "Windows 11",
      "subjectId": "M-00001"
    }
  ],
  "outcomeCounts": { "all": 128, "success": 96, "failure": 32 },
  "kindCounts": { "all": 128, "member": 110, "admin": 18 },
  "total": 42
}
```

| 필드 | 타입 (types.ts) | 설명 |
|---|---|---|
| `entries` | `readonly LoginHistoryEntry[]` | 현재 페이지의 시도. **응답은 이미 최신순 정렬**이어야 한다 |
| `entries[].name` | string | **서버가 마스킹한** 이름('명재***'). 미등록 계정은 `""`(프론트가 '—' 표시) |
| `entries[].failureReason` | `LoginFailureReason \| null` | 성공이면 null. 실패면 `invalid_password` \| `account_locked` \| `unknown_account` \| `session_expired`(BE-001 정의) |
| `entries[].consecutiveFailures` | number | **서버가 계산한** 그 계정의 연속 실패 수(성공이면 0). 프론트가 목록을 훑어 세지 않는다(§7.2) |
| `entries[].subjectId` | `string \| null` | 이동할 계정 레코드 id. **미등록 계정은 null**(§7.2 — 없는 계정의 상세는 없다) |
| `outcomeCounts` · `kindCounts` | `OutcomeCounts` · `AccountKindCounts` | **기간 안에서만** 센 결과별/유형별 건수. 결과·유형·검색 필터와 무관(기간이 스코프 — FS-008 좌측 배지) |
| `total` | number | 필터·검색·기간을 전부 적용한 뒤의 건수. 페이지네이션(FS-008-EL-011)과 요약(FS-008-EL-007)이 쓴다 |

**에러 목록**

| HTTP | 에러코드 | 조건 |
|---|---|---|
| 400 | `VALIDATION_FAILED` | `outcome`·`accountKind` enum 밖, `keyword` > 100자, `page`·`size` 범위 밖 |
| 400 | `INVALID_DATE_RANGE` | `from`·`to` 형식·미래·역전·90일 초과 (§7.3) |
| 401 | `UNAUTHENTICATED` | 세션 없음·만료 |
| 403 | `FORBIDDEN` | 감사 로그 read 권한 부족 (컬렉션은 은닉하지 않는다 — §3.2) |
| 429 | `RATE_LIMITED` | 분당 120회 초과 |
| 500 | `INTERNAL_ERROR` | 내부 오류 |
| 504 | `REQUEST_TIMEOUT` | 서버 처리 5초 초과 |

### BE-008-EP-02 · 로그인 이력 내보내기 조회

| 항목 | 내용 |
|---|---|
| 근거 (FS) | FS-008-EL-006, FS-008-EL-013 |
| 메서드 · 경로 | `GET /api/login-history/export` |
| 권한 | 로그인 이력 리소스 `read` (내보내기는 별도 액션이 아니라 read 로 조회한 것을 파일로 받을 뿐) |
| 멱등성 | 멱등 (GET). 재시도 안전. 응답은 요청 시점 스냅샷 |
| 페이징 | **없음 — 조건에 걸린 전량을 1회 응답으로 반환한다.** 90일 상한이 규모를 제한하지만, 동기 응답 상한을 별도로 두고 초과 시 422 `EXPORT_TOO_LARGE` 로 거절한다(§7.4) |
| 레이트리밋 | 관리자당 **분당 3회** (전량 스캔이라 조회보다 강하게 제한) |

**요청 — 쿼리**: `outcome` · `accountKind` · `from` · `to` · `keyword` (BE-008-EP-01 과 동일 제약). `page`·`size` 를 받지 않는다 — 내보내기는 **현재 페이지가 아니라 조건 전체**다(FS-008-EL-006).

**응답 200** — `readonly LoginHistoryEntry[]` (EP-01 의 `entries` 와 동일한 스키마. 봉투 없이 배열 반환)

- **CSV 는 서버가 만들지 않는다.** 프론트 `toCsv(entries)` 가 브라우저에서 10열(시각·계정·이름·유형·결과·실패 사유·연속 실패·IP·브라우저·OS)로 직렬화하고 Blob 다운로드를 한다(FS-008-EL-006 — 파일명 `login-history-<YYYY-MM-DD>.csv`). 판정은 §7.4.
- 조건에 걸린 이력이 0건이면 빈 배열을 반환한다(프론트는 헤더만 든 CSV 를 받는다).

**에러 목록**

| HTTP | 에러코드 | 조건 |
|---|---|---|
| 400 | `VALIDATION_FAILED` / `INVALID_DATE_RANGE` | EP-01 과 동일 |
| 401 | `UNAUTHENTICATED` | 세션 없음·만료 |
| 403 | `FORBIDDEN` | 감사 로그 read 권한 부족 (§3.2) |
| 422 | `EXPORT_TOO_LARGE` | 조건에 걸린 이력이 동기 응답 상한을 넘는다(§7.4). `message` = '조회 결과가 너무 많습니다. 기간·조건을 좁혀 주세요.' |
| 429 | `RATE_LIMITED` | 분당 3회 초과 |
| 500 | `INTERNAL_ERROR` | 내부 오류 |
| 504 | `REQUEST_TIMEOUT` | 서버 처리 30초 초과 |

## 5. 예외 매트릭스

| 엔드포인트 | 400 검증 | 401 인증 | 403 vs 404 | 404 대상없음 | 409 충돌 | 422 상태위반 | 429 과부하 | 500 오류 | 타임아웃 |

## 7. 핵심 판정 (⚠ 보안·감사 — 근거를 남긴다)

### 7.1 쓰기 엔드포인트가 없다 — 감사 불변성의 계약

이 도메인에 이력을 **생성·수정·삭제하는 엔드포인트를 만들지 않는다.** 근거: 지울 수 있는 감사 로그는 감사 로그가 아니다(침입자가 가장 먼저 지우는 것이 자기 흔적이다). 이력의 **생성**은 로그인 도메인(BE-001)이 인증 처리 중에 append-only 로 남기고, 이 화면은 읽기만 한다. **삭제·수정 API 는 권한 있는 운영자에게도 제공하지 않는다** — FS-008-EL-004 안내문('관리자도 수정하거나 삭제할 수 없습니다')이 이 계약을 사용자에게 고지한다. 저장소도 append-only 로 다뤄 물리 삭제를 막는다.

### 7.2 연속 실패·마스킹·subjectId 는 서버가 정한다 — 프론트가 지어내지 않는다

- **연속 실패(`consecutiveFailures`)는 서버가 계산한다.** 페이지네이션된 목록을 프론트가 훑어 세면 페이지 경계에서 값이 왜곡된다(1페이지의 3연속이 2페이지에서 1연속). 계정 탈취 시도의 신호를 페이지 나누기가 지우면 안 되므로, 서버가 시도 시점 기준의 정확한 연속 실패 수를 각 행에 실어 내려준다.
- **이름은 서버가 마스킹한다.** 개인정보이므로 '명재***' 처럼 마스킹된 채로 내려온다. 미등록 계정은 이름이 없어 `""` 다.
- **`subjectId` 는 미등록 계정에서 null 이다.** 없는 계정의 상세는 없으므로 프론트가 링크·행 이동을 만들지 않는다(FS-008-EL-008.2 · 008.10). 억지로 경로를 만들면 죽은 링크가 되고 없는 계정을 있는 것처럼 보이게 한다.

### 7.3 미등록 계정 열거 오라클 방지 (BE-001 §3.2·§4 상속)

미등록 계정의 시도도 **동일하게 기록·잠금·집계된다.** 로그인 도메인이 미등록 이메일을 오답 비밀번호와 완전히 같은 응답으로 처리하고 동일 임계값(연속 5회)에서 잠그므로(BE-001 §4), 감사 이력에도 그 시도가 `unknown_account` 사유로 남는다. **관리자용 이력은 진짜 사유를 보여주되**(FS-008 §1.1), 그 정보가 로그인 화면의 응답으로 새 나가지 않도록 두 층을 분리한다 — 감사 열람 권한이 있는 관리자만 진짜 사유를 본다. 기간 검증(형식·미래·역전·90일)도 서버가 강제해 우회 호출을 막는다.

### 7.4 내보내기 — 동기 응답 + 상한. CSV 는 프론트에 남긴다

- 프론트는 `fetchLoginHistoryForExport(...): Promise<readonly LoginHistoryEntry[]>` 로 **배열을 직접 받아** `toCsv` 로 파일을 만든다. 90일 기간 상한이 규모를 1차로 제한한다. 그럼에도 90일 안에서 이력이 매우 많을 수 있으므로 **동기 응답 상한을 두고 초과 시 422 `EXPORT_TOO_LARGE`** 로 거절해 범위를 좁히게 한다(잘라서 일부만 주지 않는다 — 조용히 잘린 감사 CSV 는 잘못된 판단으로 이어진다).
- **CSV 직렬화는 프론트에 남긴다.** 서버가 CSV 를 내려주면 `toCsv` 가 사라지고 Content-Type·다운로드 코드가 바뀐다. 열 구성이 이미 FS-008-EL-006 에 고정돼 있고 프론트에 구현돼 있어 서버로 옮길 이득이 없다. 대량이 실제 문제가 되면 비동기 스트리밍 내보내기를 별도 계약으로 승격한다(FS-008 §7 #2).

### 7.5 후속 이관 (UI 기획 · 프론트 구현 · 백엔드 명세 · 아키텍처)

| # | 내용 | 이관 대상 |

### 7.6 열람 감사 — 감사 로그를 여는 것도 감사한다

로그인 이력은 회원·운영자의 IP·기기·실패 사유를 담는다. 그것을 **누가 언제 어떤 조건으로 조회·내보냈는지**도 별도 감사 로그에 남겨야 한다 — 감사 로그의 열람이 추적되지 않으면 내부자 오남용을 잡을 수 없다. 이 열람 감사의 계약은 §7.5 #5 로 이관한다(이 화면 자체의 범위 밖).
