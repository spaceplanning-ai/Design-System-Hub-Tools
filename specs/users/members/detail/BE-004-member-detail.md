# BE-004. 회원 상세 백엔드 기능 명세

## 1. 개요

| 항목 | 내용 |
|---|---|
| 대상 화면 | FS-004 회원 상세 (`/users/members/:id`) |
| 범위 | 회원 상세 조회(회원 정보 · 동의 정보 · 활동 정보 · 적립금 잔액/내역 · 보유 쿠폰 · 관리자 메모), 비밀번호 변경, 적립금 지급/차감, 적립금 내역 취소, 관리자 메모 저장 |
| 범위 밖 | **회원 정보 수정** — 회원 정보는 전부 읽기 전용이며 관리자가 바꿀 수 있는 값은 비밀번호뿐이다(FS-004 §1.1). 이름·연락처·주소·등급을 바꾸는 엔드포인트는 이 계약에 존재하지 않는다. **동의 정보 변경** — 관리자가 회원의 동의를 대신 바꿀 수 없다(FS-004-EL-005.4). **쿠폰 발급·회수** — 상세 화면에 컨트롤이 없다. **회원 삭제 · 알림 발송** — BE-003 에 정의됐다(§4.6, 중복 정의하지 않는다) |
| 전제 | 인증은 세션 쿠키 기반. 모든 경로는 `/api` 프리픽스. 응답 본문은 `application/json; charset=utf-8`. 시각은 ISO 8601(오프셋 포함) |
| 프론트 어댑터 | `apps/admin/src/pages/members/data-source.ts` |
| 도메인 타입 | `apps/admin/src/pages/members/types.ts` |
| 목록 명세 | BE-003 (`specs/users/members/BE-003-members.md`) — §2 에러 봉투 · §3 권한 모델이 **동일하다** |

> **§2 · §3 은 BE-003 과 형식이 동일하다.** 도메인마다 다른 에러 형식·권한 모델을 만들면 백엔드가 두 벌을 구현하게 된다. 아래 정의를 바꾸려면 BE-003 을 함께 개정한다. BE-001(로그인 백엔드 명세)은 아직 없으며(FS-001 만 존재), 작성될 때 이 봉투·권한 모델을 재사용한다.

## 2. 공통 에러 봉투

모든 4xx / 5xx 응답은 아래 봉투를 따른다. 예외 없다. 각 엔드포인트(§4)는 **에러코드 목록만** 나열하며 자체 응답 형식을 정의하지 않는다.

```json
{
  "error": {
    "code": "INSUFFICIENT_POINTS",
    "message": "보유 적립금보다 많이 차감할 수 없습니다.",
    "fields": [
      { "name": "amount", "code": "OUT_OF_RANGE", "message": "1 이상 1,000,000 이하여야 합니다." }
    ],
    "traceId": "01J8X4K2M9P7Q3R5S6T8V0W2Y4"
  }
}
```

| 필드 | 타입 | 필수 | 설명 |
|---|---|---|---|
| `error.code` | string | O | 기계 판정용 에러코드(SCREAMING_SNAKE_CASE). 프론트 분기의 유일한 근거 |
| `error.message` | string | O | 사용자에게 그대로 보여줄 수 있는 문구. **내부 정보(스택·쿼리·경로·저장소 구조) 노출 금지** |
| `error.fields` | array \| null | X | 400 / 422 검증 실패 시에만. 프론트가 입력 필드에 매핑한다. `name` 은 요청 파라미터명 또는 헤더명 |
| `error.traceId` | string | O | 로그 상관관계 ID. 500 문의 시 사용자가 그대로 전달한다 |

### 2.1 공통 에러코드

| HTTP | 에러코드 | 발생 조건 | 프론트 처리 |
|---|---|---|---|
| 400 | `VALIDATION_FAILED` | 요청 파라미터·헤더가 제약 위반 | `error.fields` 를 입력 필드에 매핑해 표시 |
| 401 | `UNAUTHENTICATED` | 세션 없음·만료 | `/login?returnUrl=<현재경로>` 로 이동 (BE-003 §7.7 — 현재 프론트에 미구현) |
| 403 | `FORBIDDEN` | 인증됐으나 역할 부족 (존재 은닉이 무의미한 리소스 — §3.2) | 조회는 FS-004-EL-011 배너, 쓰기는 각 요소의 실패 처리 |
| 403 | `CSRF_TOKEN_INVALID` | 쓰기 요청의 `X-CSRF-Token` 헤더 누락·불일치 (§3.3) | 쓰기 실패 처리와 동일 |
| 404 | `MEMBER_NOT_FOUND` | 회원 id 가 없거나 삭제(탈퇴)됨, 또는 회원 도메인 읽기 권한 없음 (§3.2 은닉) | 어댑터는 `Error('회원을 찾을 수 없습니다')` 로 변환한다 → FS-004-EL-012 (§6.1) |
| 404 | `POINT_ENTRY_NOT_FOUND` | 적립금 내역 id 가 이 회원의 내역이 아니다 | 내역 삭제 실패 처리 |
| 409 | `CONFLICT` | 동시 수정·중복 요청 | 최신 상태 재조회 후 재시도 안내 |
| 422 | `UNPROCESSABLE` | 형식은 맞으나 상태 위반 | 상태별 안내 문구 표시 |
| 429 | `RATE_LIMITED` | 레이트리밋 초과. `Retry-After`(초) 헤더를 함께 반환한다 | `Retry-After` 초 뒤 재시도 안내 |
| 500 | `INTERNAL_ERROR` | 서버 내부 오류 | 일반 문구 + `traceId` 표시. **원인 노출 금지** |
| 504 | `REQUEST_TIMEOUT` | 서버 처리 상한 초과 (§3.4) | 500 과 동일한 일반 문구 + `traceId` |

## 3. 인증 · 권한 모델

### 3.1 역할

| 역할 | 설명 | 비고 |
|---|---|---|
| `admin` | 회원 도메인 전체 권한 — 상세 조회 · 비밀번호 변경 · 적립금 지급/차감 · 적립금 내역 취소 · 메모 저장 · 회원 삭제(BE-003-EP-05) · 알림 발송(BE-003-EP-06) | 이 문서의 모든 엔드포인트를 호출할 수 있다 |
| `operator` | 상세 조회 + **관리자 메모 저장**. 그 외 쓰기(비밀번호 변경 · 적립금 조정 · 내역 취소 · 회원 삭제 · 알림 발송)는 **403** | 메모는 운영 주석이므로 조회 권한자에게 허용한다. 적립금은 돈이고 비밀번호는 계정 통제권이므로 허용하지 않는다 |
| 회원 도메인 권한이 없는 인증 관리자 | 회원 리소스에 접근할 수 없는 그 밖의 역할 | 개별 회원 리소스는 **404** 은닉 (§3.2) |

> **프론트에는 역할 분기가 없다** (FS-004 §4.1). 요소가 역할에 따라 숨거나 비활성되지 않는다. 권한 강제는 전적으로 서버의 책임이며, 프론트는 403 을 다른 실패와 같은 문구로 표시한다.

### 3.2 403 vs 404 은닉 정책

BE-003 §3.2 와 동일한 두 원칙을 적용한다.

1. **컬렉션 리소스의 존재는 비밀이 아니다** → 권한 부족 시 `403 FORBIDDEN`.
2. **개별 회원 리소스의 존재 자체가 개인정보다** → 회원 도메인 **읽기 권한이 없는** 주체에게는 `404 MEMBER_NOT_FOUND` 로 은닉한다. **읽기 권한이 있는 주체**(예: `operator`)가 쓰기 권한만 없어 거절될 때는 `403 FORBIDDEN` 을 준다 — 이미 회원의 존재를 아는 주체에게 존재를 숨기는 것은 의미가 없고, 정당한 운영자가 권한 문제를 인지하지 못하게 만든다.

| 리소스 | 권한 부족 시 응답 | 근거 |
|---|---|---|
| `GET /api/members/:id` | **404 `MEMBER_NOT_FOUND`** | 회원 id 열거를 막는다. 존재 여부 자체가 개인정보다. 프론트는 이 코드를 FS-004-EL-012('회원을 찾을 수 없습니다.')로 표시한다 — **404 은닉이 곧 이 문구다** |
| `PATCH /api/members/:id/password` | `operator` → **403** / 회원 도메인 읽기 권한 없음 → **404** | `operator` 는 상세에서 이미 이 회원을 본다. 읽기 권한이 없는 주체에게는 id 열거를 차단한다 |
| `POST /api/members/:id/points` | `operator` → **403** / 회원 도메인 읽기 권한 없음 → **404** | 위와 동일 |
| `DELETE /api/members/:id/points/:entryId` | `operator` → **403** / 회원 도메인 읽기 권한 없음 → **404** | 위와 동일. `entryId` 가 다른 회원의 내역이면 404 `POINT_ENTRY_NOT_FOUND` (내역 id 열거 차단) |
| `PUT /api/members/:id/memo` | `operator` → **허용(403 아님)** / 회원 도메인 읽기 권한 없음 → **404** | `operator` 에게 메모 저장 권한이 있다 (§3.1) |
| `DELETE /api/members/:id` · `POST /api/members/:id/notifications` | BE-003 §3.2 참조 | 중복 정의하지 않는다 |

> **FS-004 §7 #7 미결 사항을 이 표로 확정한다.** FS-004-EL-012('회원을 찾을 수 없습니다.')는 **권한 부족의 은닉 응답을 겸한다**.

### 3.3 CSRF

세션 쿠키 기반 인증이므로 상태 변경 요청(POST · PUT · PATCH · DELETE)은 `X-CSRF-Token` 헤더를 요구한다. 누락·불일치 시 `403 CSRF_TOKEN_INVALID`. 프론트 어댑터의 **함수 시그니처는 바뀌지 않는다** — 어댑터 함수 본문이 헤더를 싣는다(§6.1).

### 3.4 타임아웃 상한

| 계열 | 서버 처리 상한 | 초과 시 | 프론트 상한(권고) |
|---|---|---|---|
| 상세 조회 | 5초 | 504 `REQUEST_TIMEOUT` | 10초 (현재 프론트에 상한 없음 — §7.6) |
| 쓰기(비밀번호 · 적립금 · 내역 취소 · 메모) | 5초 | 504 `REQUEST_TIMEOUT` | 10초 (현재 프론트에 상한 없음 — §7.6) |

서버 상한이 프론트 권고 상한보다 항상 작다 — 서버가 먼저 응답을 확정하므로 프론트가 결과를 모른 채 끊는 구간이 없다.

### BE-004-EP-03 · 적립금 지급 / 차감

| 항목 | 내용 |

**요청 — 경로 / 헤더 / 바디** (`PointAdjustInput`)

| 위치 | 이름 | 타입 (types.ts) | 필수 | 제약 |

**응답 200** — 생성된 내역 1행 (`PointEntry`)

```json

```

| 필드 | 타입 | 설명 |
|---|---|---|
| `id` | string | 서버가 생성한 내역 id |
| `date` | string | `YYYY-MM-DD` (서버 시각 기준) |
| `reason` | string | 앞뒤 공백을 제거한 요청 `reason` |
| `orderNo` | `string \| null` | **수기 조정이므로 항상 `null`** (FS-004-EL-007.7 — 관련 주문번호가 없으면 프론트가 `'—'` 로 표시한다) |

### BE-004-EP-04 · 적립금 내역 취소

| 항목 | 내용 |
|---|---|
| 근거 (FS) | FS-004-EL-007.8 |
| 메서드 · 경로 | `DELETE /api/members/:id/points/:entryId` |
| 권한 | **`admin` 만.** `operator` → 403 (§3.2). 회원 도메인 읽기 권한 없음 → 404 `MEMBER_NOT_FOUND` |
| 멱등성 | **멱등**. 이미 취소된 내역에 재요청하면 **잔액을 다시 되돌리지 않고** 204 를 반환한다 — 프론트가 경합을 성공으로 처리하는 동작(FS-004-EL-007.8 경합 열)과 일치한다. 멱등키가 필요 없다 |
| 페이징 | N/A — 단건 처리 |
| 레이트리밋 | 관리자당 분당 30회 |

**요청 — 경로**

| 위치 | 이름 | 타입 | 필수 | 제약 |
|---|---|---|---|---|
| 경로 | `id` | string | O | 회원 id. 1–64자 |
| 경로 | `entryId` | string | O | 적립금 내역 id (`PointEntry.id`). 1–64자. **이 회원의 내역이 아니면 404** |

**응답 204** — 본문 없음. 프론트 `removePointHistory(id, entryId): Promise<void>` 가 본문을 쓰지 않는다.

**처리 방식 — 이력을 지우지 않고 상쇄 거래로 취소한다 (§7.2 판정)**

- 서버는 원장에서 대상 행을 **물리적으로 지우지 않는다.** 대상 내역을 '취소됨' 으로 표시하고, 같은 금액의 **반대 부호 상쇄 거래(reversal)** 를 원장에 기록한다. 잔액은 상쇄 거래만큼 되돌아간다.
- **조회 응답(BE-004-EP-01 의 `pointHistory`)에서는 취소된 내역과 그 상쇄 거래를 둘 다 제외한다.** 그 결과 화면에 보이는 표는 프론트가 낙관적으로 행을 제거한 결과와 **정확히 일치**한다 — 새로고침해도 표가 달라지지 않는다.
- **`orderNo` 가 `null` 인 내역만 취소할 수 있다.** 주문에 연동된 적립(`orderNo !== null`)을 회원 화면에서 지우면 주문 도메인의 적립 기록과 어긋난다 — 회수는 주문 취소 흐름의 책임이다. 시도하면 422 `POINT_ENTRY_NOT_REVERSIBLE`.
- 취소 결과 잔액이 음수가 되면 422 `INSUFFICIENT_POINTS` 로 거절한다 (음수 잔액 금지 — §7.1).
- 취소 행위는 감사 로그에 남긴다 (행위자 관리자 id · 대상 회원 id · 내역 id · 금액 · 시각 · `traceId`).

**에러 목록**

| HTTP | 에러코드 | 조건 |
|---|---|---|
| 400 | `VALIDATION_FAILED` | `id` · `entryId` 형식 위반 |
| 401 | `UNAUTHENTICATED` | 세션 없음·만료 |
| 403 | `FORBIDDEN` | `operator` 등 쓰기 권한 부족 |
| 403 | `CSRF_TOKEN_INVALID` | `X-CSRF-Token` 누락·불일치 |
| 404 | `MEMBER_NOT_FOUND` | 없는 회원 id · 삭제(탈퇴)된 회원 · 권한 은닉 |
| 404 | `POINT_ENTRY_NOT_FOUND` | `entryId` 가 존재하지 않거나 **이 회원의 내역이 아니다** (내역 id 열거 차단) |
| 422 | `POINT_ENTRY_NOT_REVERSIBLE` | 주문 연동 적립(`orderNo !== null`)이다. `message` = '주문에 연동된 적립 내역은 취소할 수 없습니다.' |
| 422 | `INSUFFICIENT_POINTS` | 취소하면 잔액이 음수가 된다. `message` = '이미 사용된 적립금이라 취소할 수 없습니다.' |
| 429 | `RATE_LIMITED` | 분당 30회 초과 |
| 500 | `INTERNAL_ERROR` | 내부 오류 |
| 504 | `REQUEST_TIMEOUT` | 서버 처리 5초 초과 |

프론트는 모든 행에 삭제 버튼을 렌더한다(FS-004-EL-007.8) — 주문 연동 행에서도 버튼이 보이고, 누르면 422 가 돌아오지만 사용자에게 문구가 표시되지 않는다(FS-004 §7 #2). §7.6 에 변경 요청 로 기록한다.

| 403 | `CSRF_TOKEN_INVALID` | `X-CSRF-Token` 누락·불일치 |
| 404 | `MEMBER_NOT_FOUND` | 없는 id · 삭제(탈퇴)된 회원 · 권한 은닉 |
| 409 | `MEMO_CONFLICT` | `If-Match` 가 서버의 현재 메모 버전과 다르다 — **다른 관리자가 먼저 저장했다**. `message` = '다른 관리자가 메모를 먼저 저장했습니다. 새로고침 후 다시 시도해 주세요.' |
| 429 | `RATE_LIMITED` | 분당 30회 초과 |
| 500 | `INTERNAL_ERROR` | 내부 오류 |
| 504 | `REQUEST_TIMEOUT` | 서버 처리 5초 초과 |

403 `FORBIDDEN` 이 이 엔드포인트에 없다 — `admin` 과 `operator` 가 모두 저장할 수 있고, 그 밖의 역할은 404 로 은닉된다 (§3.2).

프론트는 409 를 다른 실패와 같은 문구('저장에 실패했습니다. 잠시 후 다시 시도해 주세요.' — FS-004-EL-009.4)로 표시한다. 재시도해도 같은 409 가 반복되므로 문구 분기가 필요하다 — §7.6 에 변경 요청 로 기록한다.

### 4.6 참조 엔드포인트 (BE-003 에 정의됨 — 중복 정의하지 않는다)

이 화면의 ⋯ 액션 메뉴 2개는 목록 화면과 **같은 엔드포인트**를 호출한다. 두 벌로 정의하면 백엔드가 두 벌을 구현하게 된다.

| FS 요소 | 이름 | 엔드포인트 | 정의 위치 |
|---|---|---|---|
| FS-004-EL-002.1, FS-004-EL-014.4 | 회원 삭제 (⋯ 메뉴 → 확인 다이얼로그) | `DELETE /api/members/:id` | **BE-003-EP-05** (근거 FS 에 FS-004-EL-002.1 · FS-004-EL-014.4 가 포함돼 있다) |
| FS-004-EL-002.2 | ⋯ 메뉴 '알림 발송' | `POST /api/members/:id/notifications` | **BE-003-EP-06** (근거 FS 에 FS-004-EL-002.2 가 포함돼 있다) |

두 엔드포인트의 요청·응답·에러·**예외 9축**은 BE-003 §4 · §5 에 있다. 이 문서의 §5 매트릭스는 **BE-004 가 정의한 5개 엔드포인트**만 다룬다.

삭제 성공(204) 후 프론트는 `/users/members` 로 이동하며 브라우저 히스토리에서 이 상세 화면을 대체한다 (FS-004-EL-014.4) — 삭제된 회원의 상세는 404 `MEMBER_NOT_FOUND` 이므로 뒤로 가기로 돌아올 수 없어야 한다.

### 6.1 어댑터 함수 본문에 요구되는 사항 (시그니처 불변)

`data-source.ts` 머리말은 '백엔드가 준비되면 **이 파일의 함수 본문만** 실제 HTTP 호출로 교체하면 되고, 화면 코드는 한 줄도 바뀌지 않는다' 고 규정한다. 아래는 그 본문 교체에 포함되는 요구사항이다.

| # | 요구사항 | 대상 함수 | 내용 |
|---|---|---|---|
| 1 | **CSRF 헤더** | `changePassword` · `addPointHistory` · `removePointHistory` · `saveMemo` | `X-CSRF-Token` 헤더를 싣는다 (§3.3) |
| 2 | **404 → 문구 변환** | `fetchMemberDetail` | 404 `MEMBER_NOT_FOUND` 를 받으면 **`new Error('회원을 찾을 수 없습니다')`** 로 변환해 던진다. `MemberDetailPage.tsx` 가 `error.message === '회원을 찾을 수 없습니다'` 로 FS-004-EL-012 를 분기한다 — **이 문자열은 계약의 일부다** |
| 3 | **멱등키 생성** | `addPointHistory` | 호출마다 UUID v4 를 만들어 `Idempotency-Key` 헤더에 싣는다. 이것으로 **전송 계층 재시도(자동 재전송·연결 복구)** 로 인한 중복 지급이 막힌다. 시그니처에 키 인자를 추가하지 않는다 |
| 4 | **ETag 보관** | `fetchMemberDetail` → `saveMemo` | `fetchMemberDetail` 응답의 `ETag` 헤더를 **회원 id 별로 어댑터 모듈 안에 보관**하고, `saveMemo` 가 그 값을 `If-Match` 헤더로 싣는다. `saveMemo` 의 204 응답이 돌려주는 새 `ETag` 로 보관값을 갱신한다. 두 함수의 시그니처는 바뀌지 않는다 (§7.4) |

**#3 의 한계 (프론트 구현 쪽 변경 요청 — §7.6 #1)**: 호출마다 새 키를 만들면 **사용자가 실패 후 버튼을 다시 누르는 경우**는 새 키가 되어 새 거래로 처리된다. 요청이 서버에서 성공했으나 응답이 유실된 상황에서 사용자가 재클릭하면 **두 번 지급된다.** 이를 막으려면 키를 **제출 시도 단위**로 생성해 입력값(`kind` · `amount` · `reason`)이 바뀌지 않은 재시도에서 같은 키를 재사용해야 하며, 이는 어댑터가 화면 상태를 알아야 하므로 프론트 구현 의 결정이 필요하다. **서버 계약은 어느 쪽이든 지원한다** — 키가 같으면 24시간 안에서 최초 응답을 재생한다.

## 7. 핵심 판정 (근거를 남긴다)

### 7.1 적립금은 돈이다 — 멱등키 필수, 음수 잔액 금지

- **`Idempotency-Key` 헤더를 필수로 요구한다** (UUID v4, 24시간 보존). 같은 키 + 같은 바디는 최초 응답을 재생하고 **두 번 지급하지 않는다**. 같은 키 + 다른 바디는 409 `IDEMPOTENCY_CONFLICT` — 키를 재사용해 다른 금액을 밀어넣는 것을 막는다. 키가 없으면 400 이다: 키를 선택 사항으로 두면 실수로 누락된 요청이 정확히 위험한 요청(재시도 가능한 지급)이 된다.
- **음수 잔액을 허용하지 않는다.** 차감액이 서버 잔액을 넘으면 422 `INSUFFICIENT_POINTS` 로 **거절**하며, 잔액만큼만 부분 차감하는 동작을 서버가 임의로 수행하지 않는다. 근거: 음수 잔액은 주문·결제 도메인이 해석할 수 없는 상태다('사용 가능한 적립금이 -3,000' 이 무엇을 뜻하는지 정의돼 있지 않다). 부분 차감은 관리자가 의도하지 않은 금액을 조용히 만들어낸다.
- **잔액 검사와 차감은 같은 원자 연산이다.** 프론트의 사전 검증(FS-004-EL-007.5 ③ '보유 적립금보다 많이 차감할 수 없습니다.')은 **화면이 들고 있는 잔액** 기준이라 신뢰할 수 없다 — 다른 관리자가 동시에 차감했다면 통과한 값이 서버에서 실패한다. 서버가 최종 판정한다.
- **낙관적 잠금(버전/ETag)은 불필요하다.** 적립금 조정은 잔액을 읽어 고쳐 쓰는 것이 아니라 **증분 연산**(잔액에 더하거나 뺀다)이다. 두 관리자가 동시에 +5,000 과 -1,000 을 요청하면 순서와 무관하게 둘 다 유효하고 결과는 +4,000 이다 — 잃어버리는 의도가 없다. 잠금을 걸면 정당한 동시 조정이 실패한다.
- 지급 후 잔액 상한은 **100,000,000 포인트** 다 (422 `BALANCE_LIMIT_EXCEEDED`). 오타(0 하나 더 입력)로 만들어지는 비정상 잔액을 막는다.

### 7.2 적립금 내역 삭제 — 이력을 지우지 않는다. 상쇄 거래(reversal)로 처리한다

**회계 관점에서 이력의 물리 삭제를 허용하지 않는다.** 적립금은 회사가 회원에게 지고 있는 부채다. 원장 행을 지우면 '누가 언제 얼마를 지급했다가 되돌렸는가' 를 재구성할 수 없고, 부정 지급 후 흔적을 지우는 경로가 열린다.

동시에 **프론트가 기대하는 화면을 바꾸지 않는다.** FS-004-EL-007.8 은 '성공하면 그 행을 표에서 제거하고, 잔액에서 해당 행의 증감분을 되돌린다' 고 규정한다. 두 요구를 함께 만족시키는 방식이 상쇄 거래다.

| 계층 | 동작 |
|---|---|
| 원장(감사 추적) | 대상 내역을 **'취소됨'으로 표시**하고, 같은 금액의 **반대 부호 상쇄 거래를 추가 기록**한다. 원본 행은 남는다 |
| 잔액 | 상쇄 거래만큼 되돌아간다 — 프론트의 낙관적 갱신과 결과가 같다 |
| 조회 응답 (`pointHistory`) | **취소된 내역과 그 상쇄 거래를 둘 다 제외한다** — 화면의 표는 프론트가 행을 제거한 결과와 정확히 일치하며, 새로고침해도 달라지지 않는다 |
| 관리자 화면 | 원장(취소 이력)을 노출하지 않는다. 감사 원장의 열람은 이 도메인의 관심사가 아니다 |

**취소할 수 있는 내역을 제한한다**: `orderNo` 가 `null` 인 내역(수기 조정)만 취소할 수 있다. 주문 연동 적립(`orderNo !== null`)을 회원 화면에서 되돌리면 주문 도메인의 적립 기록과 어긋난다 — 그 회수는 주문 취소 흐름의 책임이다. 시도하면 422 `POINT_ENTRY_NOT_REVERSIBLE` 이며, 프론트가 그 행에도 삭제 버튼을 보여주고 있으므로 §7.6 에 변경 요청 를 기록한다.

**이미 사용된 적립금의 지급을 취소하면** 잔액이 음수가 된다 → 422 `INSUFFICIENT_POINTS` 로 거절한다(§7.1 음수 잔액 금지와 같은 근거). 회수가 필요하면 관리자가 **가능한 만큼 차감(EP-03)** 하고 사유를 남긴다 — 그 편이 이력에 정확히 기록된다.

### 7.3 관리자의 회원 비밀번호 변경 — 계정 탈취 벡터다. 부수 효과 3종을 필수로 확정한다

이 엔드포인트는 **관리자가 회원의 계정 통제권을 가져갈 수 있는 경로**다. 악의적·탈취된 관리자 계정이 회원 비밀번호를 바꾸고 그 계정으로 로그인할 수 있다. 아래 셋을 **선택 사항이 아니라 계약의 일부**로 확정한다.

| # | 확정 | 근거 |
|---|---|---|
| 1 | **감사 로그 필수** | 누가·언제·어느 회원의 비밀번호를 바꿨는지 남지 않으면 사후에 탈취를 식별할 수 없다. 비밀번호 값·해시는 로그에 남기지 않는다. 이 로그는 삭제할 수 없다 |
| 2 | **회원 통보(메일) 필수** | 회원이 자기 계정에 무슨 일이 있었는지 알 수 있는 유일한 창구다. 통보가 없으면 탈취가 조용히 성공한다. **거래성 알림이므로 마케팅 수신 미동의 회원에게도 발송한다.** 통보 실패가 비밀번호 변경을 되돌리지 않는다(큐가 최대 3회 재시도) — 되돌리면 계정 복구가 메일 서버 가용성에 묶인다 |
| 3 | **기존 세션·리프레시 토큰 전면 무효화(강제 재로그인) 필수** | 이 기능의 목적은 **계정 복구**다(FS-004 §1.1). 계정이 이미 탈취된 상황에서 비밀번호만 바꾸고 공격자의 살아 있는 세션을 남기면 복구가 되지 않는다 |

**레이트리밋을 이 엔드포인트에만 특별히 강하게 건다** — 관리자당 분당 5회, **회원당 시간당 3회**. 회원당 상한이 있어야 여러 관리자 계정을 동원한 표적 공격도 제한된다.

**관리자 재인증(step-up)은 v1.0 계약에 넣지 않는다.** 비밀번호 변경 직전에 관리자 본인의 비밀번호·OTP 를 다시 요구하는 것이 보안상 더 낫지만, 프론트에 재인증 UI 가 없고(FS-004-SEC-10 은 새 비밀번호 입력 2개뿐) 도입하면 모달 흐름과 어댑터 시그니처가 함께 바뀐다. **프론트가 이미 기대하는 모양이 우선**이라는 원칙에 따라 이 문서는 도입하지 않고 §7.6 #7 로 아키텍처 에 이관한다.

**낙관적 잠금은 불필요하다.** 새 비밀번호는 이전 값에 의존하지 않는 전면 대체 쓰기다. 두 관리자가 동시에 바꾸면 마지막 값이 남고 **두 값 모두 유효한 비밀번호**이며 잃어버리는 의도가 없다. 누가 어떤 값으로 바꿨는지는 감사 로그가 남긴다.

### 7.4 409 경합 — 낙관적 잠금이 필요한 엔드포인트는 **메모 저장 하나뿐**이다

FS-004 §7 #6('경합 처리 정책이 없다 — 메모와 비밀번호는 마지막 저장이 이긴다')을 확정한다.

| 엔드포인트 | 낙관적 잠금 | 근거 |
|---|---|---|
| **BE-004-EP-05 메모 저장** | **필요 — `If-Match` 필수, 불일치 시 409 `MEMO_CONFLICT`** | 메모는 **이전 값을 읽고 고쳐 쓰는** 값이다. A 가 연 화면의 메모와 B 가 저장한 메모가 다르면, A 가 저장하는 순간 **B 의 문장이 통째로 사라진다.** 잃어버리는 의도가 있으므로 충돌을 감지해야 한다 |
| BE-004-EP-02 비밀번호 변경 | 불필요 | 전면 대체 쓰기 — 이전 값에 의존하지 않는다 (§7.3) |
| BE-004-EP-03 적립금 지급/차감 | 불필요 | 증분 연산 — 두 조정이 순서와 무관하게 둘 다 유효하다 (§7.1). 중복 방지는 멱등키가 담당한다 |
| BE-004-EP-04 적립금 내역 취소 | 불필요 | 내역 단위 멱등 연산 — 이미 취소된 내역 재요청은 잔액 변경 없이 204 다 |
| BE-004-EP-01 상세 조회 | 불필요 | 읽기 전용 |
| BE-003-EP-05 회원 삭제 | 불필요 | 멱등 — 이미 삭제된 회원 재요청은 204 다 (BE-003 §5) |

**ETag 는 메모 버전만 반영한다.** `GET /api/members/:id` 의 `ETag` 헤더는 `memo` 필드의 버전만 반영하며, **적립금 조정·활동 기록 변경은 ETag 를 바꾸지 않는다.** 응답 전체의 해시를 ETag 로 쓰면 다른 관리자가 적립금만 조정해도 메모 저장이 409 로 실패하는 **가짜 충돌**이 발생한다. 이 화면에서 낙관적 잠금이 필요한 값은 메모뿐이므로 ETag 의 의미를 메모로 한정한다.

프론트 시그니처는 바뀌지 않는다 — 어댑터가 ETag 를 회원 id 별로 보관해 `If-Match` 로 싣는다 (§6.1 #4).

### 7.5 쓰기 후 재조회 — 하지 않는다 (FS-004 §7 #5 확정)

프론트는 적립금 조정·내역 취소·메모 저장 뒤에 **재조회하지 않고 화면 상태만 갱신한다**(FS-004 §4.1 '쓰기 후 갱신'). 이 명세는 그 동작을 **유지한다**:

- 적립금 조정(EP-03)은 응답으로 **생성된 내역 행**(`PointEntry`)을 돌려주고, 프론트가 그 행을 표 맨 위에 붙이고 잔액을 증감분만큼 갱신한다 — 서버 계산과 결과가 같다.
- 내역 취소(EP-04)는 상쇄 거래 방식이라(§7.2) 프론트가 행을 지운 결과와 서버 재조회 결과가 **정확히 일치**한다.
- 메모 저장(EP-05)은 `If-Match` 낙관적 잠금이 있어(§7.4) 화면 값과 서버 값이 어긋난 채 저장되지 않는다.

**어긋날 수 있는 유일한 지점은 잔액이다** — 다른 관리자가 동시에 조정하면 화면 잔액이 서버 잔액과 달라진다(FS-004-EL-007.1 경합 열). 이를 없애려면 EP-03 · EP-04 의 응답에 `balance` 를 실어야 하고, 그러면 어댑터 반환 타입(`Promise<PointEntry>` → `Promise<{ entry, balance }>`)과 화면 갱신 코드가 바뀐다 — **프론트가 이미 기대하는 모양이 우선**이므로 v1.0 에서는 바꾸지 않고 §7.6 #2 로 이관한다. 새로고침하면 서버 값으로 되돌아간다.
