# FS-004. 회원 상세

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 목적 | 회원 1명이 제출한 정보와 운영 기록(활동·적립금·쿠폰·관리자 메모)을 한 화면에서 확인하고, 계정 복구를 위한 비밀번호 변경·적립금 조정·메모 저장·회원 삭제·알림 발송을 수행한다 |
| 역할(주 사용자) | 관리자 (구현에 역할 분기 없음 — §4.1 참조) |
| 진입 경로 | FS-003 회원 관리 목록에서 행 클릭(FS-003-EL-011.13) · 닉네임 링크(FS-003-EL-011.3) · 메모 아이콘(FS-003-EL-011.11) |
| 관련 화면 | FS-003 회원 관리 목록 (← 돌아가기 · 삭제 후 복귀 대상) / `/users/admins` (운영진 — 이 화면의 관심사가 아니다) |
| 대응 SCR | SCR-004 (미작성 — §7) |
| 구현 경로 | `apps/admin/src/pages/members/MemberDetailPage.tsx` 및 `apps/admin/src/pages/members/components/**` |

### 1.1 이 화면의 설계 원칙 (요구사항 — 명세에 고정한다)

| 원칙 | 내용과 이유 |
|---|---|
| **회원 정보는 전부 읽기 전용** | 좌측 '회원 정보' 카드(FS-004-EL-004)의 값(추천인 코드·회원 유형·계정·이름·연락처·국가·주소·상세주소·생년월일·소셜 로그인·추천인)은 **회원 본인이 제출한 데이터**다. 관리자가 대신 고치면 회원의 제출 기록과 실제 값이 어긋난다. 따라서 이 값들은 입력 컨트롤(`input`/`select`)이 아니라 `dl`/`dt`/`dd` 텍스트로만 렌더한다. 구현: `components/MemberInfoCard.tsx` |
| **변경 가능한 유일한 값은 비밀번호** | 계정 복구(회원이 로그인하지 못하는 상황)를 위해 비밀번호만 관리자가 바꿀 수 있다. 진입점은 FS-004-EL-004.8 버튼 하나뿐이며, 실제 값은 서버에서 내려오지 않고 자리표시자(`••••••••`)만 표시된다 |
| **'관리자 메모'와 '적립금 지급/차감'은 위 원칙의 예외가 아니다** | 이 둘은 회원이 제출한 정보가 아니라 **관리자가 붙이는 운영 기록**이다. 회원 정보 자체는 여전히 읽기 전용이다 |
| **운영진 그룹 섹션이 없다** | 운영진(스태프) 그룹과 권한은 **별도 뷰 `/users/admins`** 의 관심사다. 이 화면은 일반 회원 1명의 정보만 다룬다. 그룹 유형 `staff` 는 FS-003-EL-016.4(새 그룹 만들기 모달)에서 만들 수 있지만, 이 상세 화면에는 노출되지 않는다. 구현 근거: `types.ts` `GroupType` 주석, `MemberInfoCard.tsx` 헤더 주석 |
| **⋯ 메뉴는 '회원 삭제' / '알림 발송' 두 개뿐** | 우측 상단 액션 메뉴(FS-004-EL-002)의 항목은 2개로 고정이다. 등급 변경·회원 추가 같은 항목은 이 화면의 책임이 아니다 |
| **회원 추가 경로가 없다** | 고객은 회원가입으로만 유입된다. 이 화면에도, 목록 화면에도 회원 생성 진입점이 없다 |

| FS-004-SEC-12 | 적립금 내역 삭제 확인 다이얼로그 (비표시 기본) | FS-004-EL-007.8 로만 열린다. **돈이 움직이는 파괴적 액션은 클릭 한 번으로 실행되지 않는다** |

> **이 화면에 존재하지 않는 영역**: 운영진 그룹 섹션 (사유는 §1.1). 회원 정보 편집 폼 (사유는 §1.1).

### 4.1 공통 예외 규칙

| 규칙 | 내용 |
|---|---|
| 네트워크 단절 | 상세 조회는 실패로 처리되어 FS-004-EL-011(조회 실패 배너)로 대체 렌더된다. 쓰기 계열(비밀번호 변경·적립금 조정·내역 삭제·메모 저장·회원 삭제·알림 발송)은 각 요소의 실패 처리를 따른다. 서버 오류·네트워크 단절·타임아웃을 구분하는 문구가 구현에 없다 |
| 세션 만료 | 세션 만료를 감지해 로그인으로 보내는 처리가 이 화면의 구현에 없다. 조회는 FS-004-EL-011 로, 쓰기는 각 요소의 실패 처리로 떨어진다 — §7 미결 사항 |
| 요청 타임아웃 | 프론트 타임아웃 상한이 구현에 없다 — §7 미결 사항(상한 자체는 미정). 요청은 응답 또는 abort 까지 대기한다. **abort 신호는 조회와 쓰기에 모두 연결되어 있다** — 회원 id 변경·화면 이탈(조회), 모달·다이얼로그 닫기(비밀번호 변경·회원 삭제·내역 삭제)가 abort 지점이다 |
| 중복 제출 | 비밀번호 저장(FS-004-EL-013.9) · 적립금 조정(FS-004-EL-007.5) · 메모 저장(FS-004-EL-009.3) · 회원 삭제(FS-004-EL-014.4) · 내역 삭제(FS-004-EL-007.12) · 알림 발송(FS-004-EL-002.2)은 요청 중 해당 버튼(또는 메뉴 항목)이 비활성되어 차단된다. **적립금 지급/차감은 버튼 비활성만으로 방어하지 않는다** — 같은 태스크 안의 이중 클릭이나 응답 유실 후 재클릭에 대비해 **제출 시도 단위의 멱등키**를 실어 보내므로, 요청이 두 번 나가도 서버가 최초 응답을 재생한다(BE-004 §7.1). 키는 성공해야 버려진다 |
| 실패 통지의 자리 | 실패를 어디에 띄우는지는 **사용자가 지금 무엇을 보고 있는가**로 정한다(`shared/ui/README.md`) — ① 폼 안의 실패는 **폼 하단 인라인 문구**(FS-004-EL-007.6 · FS-004-EL-009.4) ② 모달·다이얼로그가 떠 있는 동안의 실패는 **그 안의 배너**(FS-004-EL-013.10 · FS-004-EL-014.5 · FS-004-EL-007.13) — 토스트는 모달 뒤로 밀려 시선 밖이고 모달이 닫히면 사라진다 ③ 그 외(알림 발송)는 **토스트**(FS-004-EL-003): 자동으로 사라지지 않고 '다시 시도'가 붙는다 ④ **조회** 실패는 인라인 배너(FS-004-EL-011) |
| 동시 조회 | 상세 조회는 동시에 1건만 유지된다. 회원 id 가 바뀌거나 화면을 벗어나면 진행 중이던 요청을 abort 하므로, 늦게 도착한 이전 응답이 최신 결과를 덮어쓰지 않는다 |
| 권한 없음 | **이 화면의 구현에는 역할 기반 분기가 없다.** 요소가 역할에 따라 숨거나 비활성되지 않으며, 전용 403 화면도 없다. 서버가 권한 부족을 응답하면 조회는 FS-004-EL-011 로, 쓰기는 각 요소의 실패 처리로 떨어진다. 은닉 정책(403 vs 404)은 백엔드 명세가 확정한다 — §7 미결 사항 |
| 쓰기 후 갱신 | 적립금 조정·내역 삭제·메모 저장은 **재조회하지 않고 화면 상태만 갱신한다**. 브라우저를 새로고침하면 서버 값으로 되돌아간다. 회원 삭제만 화면을 목록으로 이동시킨다 |
| 모달 공통 동작 | 두 모달(FS-004-EL-013 · FS-004-EL-014)은 같은 껍데기 — **`@tds/ui` 의 `Modal`** — 를 쓴다(FS-004-EL-014 는 `ConfirmDialog` 를 경유). `members/components/Modal.tsx` 사본은 존재하지 않는다. 공통 동작 — ① `role="dialog"` + `aria-modal="true"` + 제목과 `aria-labelledby` 연결 ② **포커스 트랩**: Tab·Shift+Tab 이 모달 안 요소를 순환한다 ③ **Esc 키로 닫힘**(이벤트 전파를 멈춘다) ④ 딤 배경 클릭으로 닫힘 ⑤ 열릴 때 지정된 요소 또는 첫 포커스 가능 요소로 포커스 이동 ⑥ 닫힐 때 모달을 연 요소로 포커스 복귀 ⑦ 열려 있는 동안 배경 스크롤 잠금 ⑧ `document.body` 로 portal 렌더. **Esc·딤·× 3경로는 `Modal` 이 소유하며 전부 `onClose` 한 곳으로 모인다**(`Modal.tsx:122-126` `requestClose`) — 그래서 미저장 가드가 `onClose` 하나만 감싸도 3경로가 함께 덮이고, 나머지 하나(취소 버튼)는 호출부가 같은 `requestClose` 를 넘겨 덮는다. 푸터 버튼은 `Modal` 이 감싸지 않는 불투명 슬롯이다(`Modal.tsx:27-31`) |
| 모달 닫힘 애니메이션 | `Modal` 은 `onClose()` 를 **퇴장 애니메이션이 끝난 뒤**에 부른다(`Modal.tsx:216-218` — exit keyframe 이름 대조). reduced-motion 환경에서는 애니메이션이 없으므로 즉시 부른다(`Modal.tsx:129-132`). 어느 쪽이든 **`onClose()` 가 곧 언마운트라는 것이 설계 전제**이며(`Modal.tsx:19-25`), 미저장 가드가 그 전제를 깨뜨릴 때 §7 미결 #11 의 latch 가 발생한다 |
