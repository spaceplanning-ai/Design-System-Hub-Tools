# FS-006. 권한 관리 (역할 · 권한 매트릭스)

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 목적 | 관리자 앱의 **역할(Role)** 을 만들고, 각 역할이 어떤 메뉴 리소스에 어떤 액션(조회·등록·수정·삭제·내보내기)을 갖는지, 어떤 대시보드 위젯을 보는지, 데이터 접근 범위를 어디까지 갖는지 정한다. 한 역할을 '적용 중' 으로 만들면 사이드바·대시보드가 그 역할을 따른다 |
| 역할(주 사용자) | 관리자 (구현에 역할 분기 없음 — §4.1 참조) |
| 진입 경로 | 좌측 GNB > 사용자 관리 > 권한 관리 (`nav-config.ts`, `to: '/users/roles'`) |
| 관련 화면 | FS-005 관리자 관리(누가 어떤 역할을 갖는지는 그쪽 관심사) / 이 역할의 read 가 꺼지면 사이드바에서 사라지는 모든 화면(`AppShell`) |
| 대응 SCR | SCR-006 (미작성 — §7) |
| 구현 경로 | `apps/admin/src/pages/permissions/PermissionsPage.tsx` 및 `components/**`. 권한 모델·저장은 `apps/admin/src/shared/permissions/**` (`PermissionProvider` · `resources.ts` · `roles.ts` · `feature-registry.ts`) |

### 1.1 이 화면의 설계 원칙 (요구사항 — 명세에 고정한다)

| 원칙 | 내용과 이유 |
|---|---|
| **⚠ 프론트 권한은 보안 경계가 아니다** | 이 화면이 켜고 끄는 체크박스는 **UI 를 그리는 규칙**일 뿐, 서버 접근을 막지 않는다. 실제 권한 강제는 전적으로 서버의 책임이며(백엔드 명세 BE-006 §3·§7 이 확정), 이 화면의 상태는 현재 브라우저 `localStorage` 에만 있다. **보안 판정은 BE-006 이 정본이다** |
| **권한의 단위는 역할이다** | 역할 = 리소스×액션 매트릭스 + 대시보드 위젯 + 데이터 접근 범위(scope). 누가 어떤 역할을 갖는지(운영자 배정)는 이 모델에서 제거됐다 — 관리자 관리(FS-005)의 몫이다(`roles.ts`) |
| **모델이 규칙을 강제한다 (UI 가 아니라)** | ① 조회(read)를 끄면 나머지 4개 액션이 함께 꺼진다 ② 다른 액션을 켜면 조회가 함께 켜진다 ③ 그룹 리소스의 권한 = 자식(페이지)들의 합집합. 이 세 규칙은 화면이 아니라 모델(`resources.ts` `enforceMatrix`)이 강제하므로, 저장값이 모순 상태(read=false 인데 update=true)로 남는 경로가 없다. **화면은 규칙을 보여줄 뿐 흉내 내지 않는다** |
| **저장 버튼이 없다 (즉시 저장)** | 체크를 바꾸면 그 자리에서 저장된다. 그 역할이 '적용 중' 이면 사이드바·대시보드가 리로드 없이 따라 바뀐다(`PermissionProvider` 구독) |
| **시스템 역할(슈퍼어드민)은 잠겨 있다** | '슈퍼어드민(전체권한)' 은 이름 변경·삭제·권한 수정이 불가하며 항상 전 권한 ON 이다. 잠금 사유는 색·비활성만이 아니라 `aria-describedby` 로 연결된 문구로도 전달된다 |

### 4.1 공통 예외 규칙

| 규칙 | 내용 |
|---|---|
| 저장 위치 (현재) | 모든 편집(역할 생성·이름·삭제·적용·매트릭스·위젯·scope)은 **`localStorage`(`PermissionProvider`)에만 저장된다.** 실제 네트워크 호출이 없어 로딩·실패·중복 제출 경로가 대부분 성립하지 않는다. 백엔드 연동(BE-006)이 붙으면 이 규칙이 바뀌고 로딩·실패·경합 처리가 필요해진다 — §7 미결 사항 |
| 다른 탭 동기화 | 다른 탭에서 바꾼 역할/권한은 `storage` 이벤트로 이 화면에 실시간 반영된다(`PermissionProvider`) |
| 저장소 접근 불가 | 프라이빗 모드 등으로 `localStorage` 를 못 쓰면 저장은 무시되고 현재 세션 동안 메모리 상태로 동작한다. 로드 실패 시 기본 역할 3종으로 폴백한다 |
| 시스템 역할 잠금 | 슈퍼어드민(시스템 역할)은 이름 변경·삭제·권한 수정·scope 변경이 전부 비활성된다. 저장값이 조작돼도 로드 시 항상 전 권한 ON 으로 정규화된다(`normalizeRole`) |
| 권한 없음 | **이 화면의 구현에는 역할 기반 분기가 없다** — 화면을 연 관리자 자신의 권한으로 요소가 숨거나 비활성되지 않는다(시스템 역할 잠금은 '보고 있는 대상 역할' 의 속성이지 사용자 권한이 아니다). 서버가 이 화면 자체에 대한 접근·저장 권한을 강제해야 한다(BE-006) — §7 미결 사항 |
| ⚠ 보안 경계 | 이 화면의 어떤 체크·저장도 서버 접근을 막지 못한다. 유효 권한 강제는 서버가 매 요청마다 판정한다(BE-006 §3·§7). 프론트 상태는 UI 편의일 뿐이다 |
